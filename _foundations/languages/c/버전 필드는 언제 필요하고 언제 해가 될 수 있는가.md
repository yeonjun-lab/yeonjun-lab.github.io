## 1. 왜 이 주제를 깊게 봐야 하는가

schema evolution을 이해하면 아주 자연스럽게 다음 질문이 나온다.

- 포맷 앞에 `version` 필드 하나 넣으면 끝 아닌가
- 버전만 올리면 새 포맷과 옛 포맷을 구분할 수 있지 않나
- 모든 파일 포맷과 프로토콜은 버전 번호를 가져야 하지 않나
- TLV 같은 확장 가능한 구조가 있다면 굳이 버전 필드가 필요한가

초보자는 종종 이렇게 생각한다.

- 버전 필드는 무조건 좋은 것
- 버전 번호가 있으면 호환성 문제가 해결됨
- 새 필드가 생기면 버전만 올리면 됨

하지만 실제로는 그렇게 단순하지 않다.  
버전 필드는 강력한 도구일 수 있지만, 잘못 쓰면 오히려 evolution을 더 어렵게 만들 수 있다.

예를 들어 다음 같은 문제가 생길 수 있다.

- 버전 분기문이 계속 늘어나 parser가 복잡해짐
- “v1이면 이렇게, v2면 저렇게, v3면 또 다르게” 식으로 코드가 폭발함
- 버전 번호는 올렸지만 실제로 어떤 필드가 어떻게 달라졌는지 불명확함
- 일부 필드만 추가됐는데도 전체 포맷을 통째로 다른 버전처럼 취급하게 됨
- forward compatibility를 버리고 “버전 다르면 거부” 방식으로 굳어짐

즉, 버전 필드는 만능 해법이 아니다.  
정확히는 **포맷 진화를 통제하기 위한 한 가지 전략**일 뿐이다.

어떤 시스템은 explicit version field가 매우 유용하다.  
반면 어떤 시스템은 TLV, optional field, unknown field skip 같은 확장성 규칙만으로 더 유연하게 진화할 수 있다.

즉, 핵심 질문은 이것이다.

- 버전 필드가 필요한가
- 필요하다면 어디에 두어야 하는가
- 어떤 변화는 버전으로 처리하고 어떤 변화는 field-level evolution으로 처리할 것인가

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- schema evolution을 실제 설계 규칙으로 끌어내릴 수 있다
- versioning이 “번호 붙이기”가 아니라 compatibility strategy라는 점이 분명해진다
- parser 분기 폭발을 피하는 감각을 준다
- explicit version과 implicit extensibility의 trade-off를 이해하게 해 준다
- 파일 포맷, 네트워크 프로토콜, 저장 포맷 설계에서 더 장기적인 판단 기준을 준다

이 문서는 버전 필드가 무엇인지, 언제 유용한지, 언제 오히려 해가 될 수 있는지,  
그리고 실무에서 어떻게 versioning 전략을 세워야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

버전 필드 문제를 큰 구조로 보면 다음과 같다.

### 2.1 explicit versioning은 포맷이 몇 번째 규칙 집합인지 직접 적는 방식이다

예를 들어 메시지 시작에 이렇게 둘 수 있다.

    [version=1][payload...]

혹은 파일 헤더에:

    magic
    version
    body

형태를 둘 수 있다.

즉, reader는 먼저 버전을 보고 어떤 규칙으로 나머지를 해석할지 결정한다.

### 2.2 implicit extensibility는 버전 번호 없이 구조 자체가 확장을 견디게 만드는 방식이다

예를 들어 TLV처럼 unknown field를 skip할 수 있는 구조에서는  
명시적 버전 번호가 없어도 새 필드를 추가하며 진화할 수 있다.

즉, 버전 필드 없이도 evolution이 가능하다.  
핵심은 **포맷 구조 자체가 변화에 견디는가**다.

### 2.3 버전 필드는 큰 구조 전환에는 유용할 수 있다

포맷 전체 의미가 근본적으로 달라지거나,  
헤더 구조가 완전히 바뀌거나,  
old parser가 더는 안전하게 처리할 수 없는 경우에는  
명시적 버전이 매우 유용하다.

즉, 버전 필드는 “부분 확장”보다 “규칙 집합 전환”에 더 잘 맞을 수 있다.

### 2.4 하지만 사소한 필드 추가마다 버전을 올리면 복잡성이 커진다

optional field 하나 추가할 때마다 version 분기를 만들면  
reader/writer 조합이 급격히 복잡해진다.

즉, 버전 필드는 필요할 때만 쓰고,  
field-level extensibility와 역할을 분리해야 한다.

## 3. 내부 동작 원리

### 3.1 버전 필드는 parser entry point를 바꾼다

explicit version field가 있으면 parser는 보통 이렇게 동작한다.

1. 헤더를 읽는다
2. version 값을 본다
3. version에 맞는 해석 규칙을 선택한다
4. 나머지를 파싱한다

즉, 버전은 파서의 최상위 분기 기준이 된다.

### 3.2 작은 변화까지 version branch로 처리하면 코드가 분기 지옥이 된다

예를 들어:

- v1: `id`, `name`
- v2: `id`, `name`, `created_at`
- v3: `id`, `name`, `created_at`, `flags`
- v4: `id`, `nickname`, `created_at`, `flags`

이걸 매번 “if version == ...”로 나누기 시작하면  
parser는 점점 “버전별 별도 포맷 해석기”가 된다.

즉, 버전 필드를 지나치게 세밀하게 쓰면  
schema evolution이 아니라 parser fragmentation이 된다.

### 3.3 unknown field skip 가능한 구조에서는 version 없이도 많은 변화가 가능하다

예를 들어 TLV라면 새 필드를 추가해도 old reader는 다음처럼 할 수 있다.

- 아는 field는 읽음
- 모르는 field는 skip
- 메시지 나머지는 계속 처리

이 경우 단순 optional field 추가만으로는  
굳이 전체 version을 올릴 필요가 없을 수 있다.

즉, 포맷 구조가 self-describing하고 extensible하면  
version field 의존도를 줄일 수 있다.

### 3.4 버전 필드는 “이 변화는 old parser가 안전하게 처리할 수 없는가”를 판단 기준으로 삼아야 한다

좋은 기준은 이것이다.

- old parser가 unknown field를 skip하고 계속 처리할 수 있는가
- 아니면 새 구조 때문에 전체 해석 규칙이 깨지는가

후자라면 버전 상승이 필요할 수 있다.  
전자라면 field-level extension으로 충분할 수 있다.

즉, 버전은 “변화가 있었다”는 이유만으로 올리는 것이 아니라  
**해석 규칙 단위의 단절이 생겼는가**를 기준으로 판단해야 한다.

### 3.5 파일 포맷과 스트리밍 프로토콜에서 버전 전략이 다를 수 있다

파일 포맷은 보통 처음에 헤더가 있고,  
reader가 전체 포맷 버전을 보고 한 번에 전략을 정하기 쉽다.

반면 스트리밍 프로토콜은 메시지 단위 확장과 optional field가 더 중요할 수 있다.  
즉, 버전 하나보다 per-field extensibility가 더 실용적일 수 있다.

### 3.6 major/minor 전략도 결국 compatibility 정책이다

명시적 버전 필드를 쓴다면 흔히 이런 생각을 한다.

- major = 호환성 깨짐
- minor = 호환성 유지 확장

이건 번호 체계가 아니라 정책 체계다.  
즉, version number 의미를 팀과 시스템이 정확히 공유해야 한다.

## 4. 핵심 구성 요소

### 4.1 explicit version field

메시지나 파일 안에 명시적으로 넣는 버전 번호다.  
최상위 해석 규칙을 선택하는 기준이 된다.

### 4.2 implicit extensibility

버전 번호 없이 optional field, TLV, unknown skip 같은 구조로  
포맷 자체가 진화를 견디게 하는 방식이다.

### 4.3 major change

old parser가 더는 안전하게 해석할 수 없는 수준의 큰 변화다.  
명시적 버전 전환이 필요한 후보가 된다.

### 4.4 minor extension

기존 parser가 무시하거나 기본값 처리로 견딜 수 있는 수준의 변화다.  
field-level evolution으로 처리 가능한 경우가 많다.

### 4.5 parser branching cost

버전 분기 때문에 parser 코드가 복잡해지고 유지보수 비용이 증가하는 문제다.

### 4.6 compatibility policy

어떤 변화에 버전을 올리고, 어떤 변화는 확장 필드로 처리할지에 대한 설계 규칙이다.

## 5. 실행 흐름 또는 처리 순서

두 가지 전략을 비교해 보자.

### 5.1 explicit version 중심 전략

파일 헤더:

    [magic][version][body]

reader 흐름:

1. magic 확인
2. version 확인
3. version 1 parser 또는 version 2 parser 선택
4. body 파싱

이 방식은 전체 규칙 집합이 크게 달라질 때 강하다.

### 5.2 field extensibility 중심 전략

메시지 구조:

    [TLV field 1][TLV field 2][TLV field 3]...

reader 흐름:

1. field 읽기
2. 아는 type이면 해석
3. 모르는 type이면 length만큼 skip
4. 다음 field 진행

이 방식은 작은 확장을 연속적으로 처리하기 좋다.

### 5.3 언제 explicit version이 필요한가

예를 들어 body 자체 인코딩 방식이 바뀐다고 하자.

- v1: fixed record layout
- v2: TLV layout

이 경우 old parser는 field skip 정도로 해결할 수 없다.  
전체 body 규칙이 달라졌기 때문이다.

즉, 이런 경우는 explicit version이 매우 유용하다.

### 5.4 언제 explicit version이 과한가

반대로 단순히 optional field 하나 추가하는 경우라면  
버전 필드 분기를 만들기보다 unknown skip 구조로 처리하는 편이 더 낫다.

즉, 작은 진화는 필드 확장으로,  
큰 규칙 전환은 버전으로 나누는 것이 자연스럽다.

## 6. 성능 / 최적화 관점

### 6.1 explicit version은 파서 진입을 단순하게 할 수 있다

헤더에서 버전을 보고 바로 해석 전략을 정하면  
포맷 전체가 완전히 다를 때는 오히려 구현이 깔끔해질 수 있다.

즉, 큰 format family 분리에는 도움이 된다.

### 6.2 하지만 세분화된 버전 분기는 유지보수 비용을 폭증시킨다

사소한 차이를 모두 version switch로 처리하면  
코드가 중복되고 테스트 조합이 늘어나며 parser 복잡도가 크게 증가한다.

즉, 단기 단순성이 장기 복잡성으로 바뀔 수 있다.

### 6.3 extensible format은 런타임 파싱 비용이 조금 늘 수 있다

TLV나 self-describing field 구조는  
고정 포맷보다 type/length 해석 비용이 약간 더 들 수 있다.

하지만 그 대가로 얻는 확장성과 compatibility가 크다.

### 6.4 전체 비용은 parsing speed보다 운영 안정성까지 포함해야 한다

포맷 설계는 단순 CPU 비용 문제가 아니다.  
버전 필드 전략이 잘못되면 운영 중 구버전/신버전 공존이 어려워지고  
배포 비용이 더 커질 수 있다.

즉, 성능보다 total lifecycle cost로 봐야 한다.

---

## 7. 실무에서 중요한 판단 기준

### 7.1 “변화가 있다”와 “버전을 올려야 한다”를 동일시하지 않는다

모든 변화에 version increment가 필요한 것은 아니다.  
핵심은 old parser가 그 변화를 안전하게 견딜 수 있는가다.

### 7.2 큰 규칙 전환에만 explicit version을 쓰는 편이 좋다

예를 들어:

- 헤더 구조 완전 변경
- 인코딩 방식 전환
- 메시지 framing 규칙 변경
- 기존 parser가 절대 오해할 큰 의미 변경

이런 경우엔 explicit version이 적절하다.

### 7.3 optional field 추가는 가능한 한 extensible format으로 처리한다

작은 확장은 field-level evolution으로 해결하는 편이 parser 구조를 덜 망친다.

### 7.4 version 필드 의미를 문서화한다

version이 있다고 끝이 아니다.  
다음이 명확해야 한다.

- major/minor 기준
- unknown version 처리 정책
- downgrade 가능성
- 최소 지원 버전
- 호환성 보장 범위

즉, 번호보다 정책 문서가 더 중요하다.

### 7.5 “지원 중인 버전 조합” 테스트 전략이 필요하다

버전이 늘어나면 테스트 조합도 늘어난다.  
따라서 어떤 writer/reader 조합을 보장할지 명확히 정해야 한다.

즉, versioning은 구현뿐 아니라 테스트 전략 문제이기도 하다.

---

## 8. 더 깊게 볼 포인트

### 8.1 version negotiation

클라이언트/서버가 서로 지원 가능한 버전을 협상하는 방식으로 확장할 수 있다.

### 8.2 feature flags vs schema version

버전 번호 대신 기능 비트/능력 목록으로 협상하는 전략과 비교할 수 있다.

### 8.3 rolling upgrade

분산 시스템에서 구버전과 신버전이 동시에 떠 있을 때 버전 전략이 어떻게 작동해야 하는지로 이어질 수 있다.

### 8.4 canonical format vs legacy support

내부적으로는 새 포맷 하나만 유지하고, 경계에서만 옛 포맷을 지원하는 설계로 확장할 수 있다.

### 8.5 parser architecture

버전별 parser를 완전히 분리할지, 공통 core + adapter 방식으로 설계할지 더 깊게 볼 수 있다.

---

## 정리

버전 필드는 강력한 도구지만 만능은 아니다.

핵심은 다음과 같다.

- explicit version field는 포맷 규칙 집합을 명시적으로 구분하는 도구다.
- implicit extensibility가 가능한 포맷에서는 작은 확장을 버전 없이 처리할 수 있다.
- 버전 필드는 큰 구조 전환에는 유용하지만, 작은 변화마다 남발하면 parser 분기 비용이 커진다.
- backward/forward compatibility를 먼저 판단한 뒤 version 전략을 정해야 한다.
- optional field 추가와 unknown field skip은 field-level evolution으로 처리하는 편이 더 나을 수 있다.
- version number보다 더 중요한 것은 compatibility policy와 문서화다.

즉, 버전 필드를 이해한다는 것은  
단순히 “숫자 하나 붙인다”를 아는 것이 아니라  
**데이터 포맷의 변화 중 어떤 것은 field-level 확장으로 처리하고, 어떤 것은 규칙 집합 전환으로 다뤄야 하는지 구분하면서, parser 복잡도와 장기 호환성을 함께 고려하는 versioning 전략을 세울 수 있게 되는 것**을 뜻한다.

이 관점이 잡혀야 파일 포맷, 네트워크 프로토콜, 저장 데이터 구조를 훨씬 더 오래 살아남는 형태로 설계할 수 있다.
