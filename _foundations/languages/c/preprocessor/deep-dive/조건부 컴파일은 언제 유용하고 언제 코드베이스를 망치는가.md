## 1. 왜 이 주제를 깊게 봐야 하는가

조건부 컴파일은 C 코드베이스에서 매우 강력한 도구다.  
동시에 매우 쉽게 남용되는 도구이기도 하다.

초급 단계에서는 보통 이렇게 배운다.
```c
    #ifdef DEBUG
    printf("debug\n");
    #endif
```
이 정도 예제만 보면 조건부 컴파일은 단순한 “디버그 코드 켜기/끄기” 기능처럼 보인다.  
하지만 실무에서는 곧바로 훨씬 큰 문제와 연결된다.

- 운영체제별 분기
- 컴파일러별 속성 분기
- 아키텍처별 코드 경로
- 라이브러리/기능 존재 여부 분기
- ABI 차이 흡수
- 디버그/릴리스 정책 분기
- 실험용 기능 토글
- 장기 유지되는 제품 라인별 차등 빌드

즉, 조건부 컴파일은 단순 편의 문법이 아니라 **하나의 소스 트리 안에 여러 개의 프로그램 변형을 공존시키는 메커니즘**이다.

바로 그래서 위험하다.

실무에서 흔히 다음과 같은 오해가 생긴다.

- `#ifdef`는 가볍게 붙여도 된다
- 빌드만 되면 관리 가능한 코드다
- 런타임 분기보다 빠르니 전처리기로 처리하는 게 낫다
- 기능이 다르면 그냥 매크로로 잘라 내면 된다
- 플랫폼 차이가 있으면 파일 곳곳에서 `#ifdef`로 감싸면 된다

이 직관들은 단기적으로는 맞아 보일 수 있다.  
하지만 장기적으로는 코드베이스를 여러 조각의 “조건부 존재 코드”로 찢어 놓기 쉽다.

왜 이 주제를 깊게 봐야 하는가?

첫째, 조건부 컴파일은 단순 분기가 아니라 **존재 여부 분기**다.  
일반 `if`는 코드가 존재한 채 실행 경로만 갈라지지만, 조건부 컴파일은 어떤 코드가 아예 컴파일 단위에서 사라질 수 있다. 이 차이는 매우 크다.

둘째, 조건부 컴파일의 비용은 런타임 비용이 아니라 **조합 폭발과 추론 비용**에 있다.  
매크로 3개만 있어도 코드베이스는 8가지 변형을 가질 수 있다. 이때 실제로 빌드되고 테스트되는 조합은 일부뿐일 가능성이 높다. 즉, 코드 리뷰로는 멀쩡해 보이지만 죽은 경로, 오래된 경로, 테스트되지 않는 경로가 쌓이기 쉽다.

셋째, 조건부 컴파일은 잘 쓰면 매우 유용하다.  
플랫폼 차이, 컴파일러 확장 차이, 헤더/기능 탐지, ABI 보호, 저수준 성능 최적화 분기처럼 **컴파일 타임에만 결정해야 하는 차이**에는 오히려 적합하다.

즉, 중요한 것은 “써도 되느냐”가 아니라 **어떤 종류의 차이를 컴파일 타임에 둘 것인가**다.

이 문서의 핵심 목표는 조건부 컴파일을 “가끔 필요한 문법”으로 소개하는 데 있지 않다.  
핵심은 다음을 분리하는 데 있다.

- 조건부 컴파일이 정말 필요한 문제와, 런타임 분기나 구조 분리가 더 나은 문제
- 존재 여부 분기와 실행 경로 분기의 차이
- 왜 소스 곳곳의 `#ifdef`가 코드베이스를 찢는가
- 빌드 시스템, 헤더, 플랫폼 추상화 계층과 어떤 관계가 있는가
- 실무에서 어떻게 제한하고 어떤 곳에 몰아넣어야 하는가

즉, 이 문서는 조건부 컴파일을 “편리한 토글”로 설명하지 않는다.  
그 대신 **코드 존재를 빌드 변형별로 재구성하는 강력하지만 고비용인 도구**로 설명한다.

## 2. 전체 구조

조건부 컴파일을 정확히 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 전처리기 층위

조건부 컴파일은 컴파일 이전 단계에서 작동한다.

- 어떤 코드를 남길지
- 어떤 코드를 제거할지
- 어떤 선언을 노출할지
- 어떤 매크로 정의를 활성화할지

즉, 이것은 실행 중 분기가 아니라 **번역 단위 자체를 바꾸는 소스 재작성 규칙**이다.

### 2.2 빌드 변형 층위

조건부 컴파일 매크로 하나는 단순 플래그가 아니라 하나의 빌드 변형 축이 된다.

- DEBUG / RELEASE
- LINUX / BSD / WINDOWS
- USE_SSL / NO_SSL
- FEATURE_X_ENABLED / DISABLED

즉, `#ifdef` 하나를 추가하는 것은 곧 “코드베이스의 변형 수”를 하나 늘리는 일이다.

### 2.3 인터페이스 층위

조건부 컴파일은 단순 구현 분기뿐 아니라 다음을 바꿀 수 있다.

- 함수 선언 존재 여부
- 구조체 필드 존재 여부
- 타입 정의 차이
- 헤더 공개 계약
- ABI 차이

즉, 조건부 컴파일은 구현 세부가 아니라 **인터페이스와 바이너리 경계 자체를 바꿀 수 있다**.

### 2.4 테스트 가능성 층위

조건부 컴파일이 늘어날수록 모든 조합을 빌드하고 테스트하기 어려워진다.  
그 결과 다음 문제가 생긴다.

- 죽은 분기
- 실제로는 아무도 빌드하지 않는 경로
- 한 플랫폼에서만 돌아가는 코드
- 오래된 컴파일러 분기가 계속 남아 있는 상태
- 조건 조합 특정 시점에만 터지는 오류

즉, 조건부 컴파일의 진짜 비용은 “읽기 어렵다”보다 **검증 공간이 폭발한다**는 데 있다.

### 2.5 실무 구조화 층위

좋은 코드베이스는 조건부 컴파일을 분산시키지 않는다.  
가능하면 다음 위치로 몰아넣는다.

- 플랫폼 추상화 헤더
- config 헤더
- 빌드 시스템 생성 헤더
- 작은 어댑터 레이어
- 선언 속성 매크로 모음

즉, 조건부 컴파일의 핵심 관리 전략은 **흩뿌리지 않고 경계에 집중시키는 것**이다.

## 3. 내부 동작 원리

### 3.1 조건부 컴파일은 “코드를 실행할지”가 아니라 “코드를 존재하게 할지”를 결정한다

이 차이가 가장 중요하다.

런타임 분기:
```c
    if (debug_enabled) {
        log_debug(...);
    }
```
이 경우 코드는 항상 컴파일된다.  
다만 실행 시점에 분기만 달라진다.

조건부 컴파일:
```c
    #ifdef DEBUG
    log_debug(...);
    #endif
```
이 경우 `DEBUG`가 없으면 해당 코드는 아예 번역 단위에서 사라진다.

왜 이 차이가 중요한가?

- 런타임 분기 코드는 항상 타입 검사와 파싱과 이름 해석을 거친다.
- 조건부 컴파일로 제거된 코드는 해당 빌드에서는 아예 검증 대상이 아니다.
- 따라서 조건부 컴파일은 단순 최적화가 아니라 **존재하는 프로그램 자체를 바꾸는 행위**다.

이 지점을 놓치면 `#ifdef`를 `if`처럼 가볍게 쓰게 된다.  
하지만 둘은 비용 구조가 다르다.

### 3.2 `#ifdef` 하나는 코드 조합 공간을 하나 더 늘린다

다음처럼 생각하면 위험이 잘 보인다.

- 매크로 `A`가 있으면 두 상태: 켜짐 / 꺼짐
- 매크로 `B`가 있으면 두 상태
- 매크로 `C`가 있으면 두 상태

조합 수는 2 × 2 × 2 = 8이다.

즉, 소스 한 파일에

- 플랫폼 매크로
- 디버그 매크로
- 기능 매크로
- 실험 매크로

가 섞이기 시작하면, 코드가 아니라 **코드군(code family)** 을 관리하게 된다.

왜 이 직관이 중요한가?

많은 사람은 `#ifdef`를 한 줄짜리 토글처럼 본다.  
하지만 실무적으로는 그것이 새로운 빌드 변형 축을 만드는 일이다.

즉, 조건부 컴파일의 비용은 한 줄의 간결함이 아니라  
**검증해야 할 프로그램 수를 늘리는 것**에 있다.

### 3.3 좋은 조건부 컴파일과 나쁜 조건부 컴파일의 차이는 “결정 시점”에 있다

조건부 컴파일이 적절한 경우는 대개 다음과 같다.

- 컴파일러 속성 존재 여부는 빌드 시점에만 안다
- 플랫폼별 헤더 이름 차이는 런타임에 바꿀 수 없다
- ABI가 달라 구조체 레이아웃을 강제로 맞춰야 한다
- 특정 명령어 집합 최적화 경로는 컴파일 타임에 분리해야 한다
- 기능 자체가 라이브러리/시스템 제공 여부에 달려 있다

이 경우는 **컴파일 시점에만 결정 가능한 차이**다.

반대로 조건부 컴파일이 부적절한 경우는 자주 다음과 같다.

- 디버그 로깅을 켜고 끄는 정책
- 제품 기능 활성/비활성
- 실험 토글
- 운영 정책 차이
- 사용자 옵션 차이

이것들은 자주 런타임 설정이나 빌드 상위 계층의 모듈 조합으로 더 잘 다뤄질 수 있다.

즉, 조건부 컴파일이 유용한지 판단하려면 먼저 물어야 한다.

- 이 차이는 정말 컴파일 시점에만 결정 가능한가
- 아니면 실행 정책 차이일 뿐인가

### 3.4 가장 위험한 형태는 “본문 내부를 잘게 찢는 `#ifdef`”다

예를 들어 이런 코드를 생각해 보자.
```c
    void process(struct ctx *ctx) {
    #ifdef PLATFORM_A
        pre_a(ctx);
    #endif

        step1(ctx);

    #ifdef FEATURE_X
        step_feature_x(ctx);
    #else
        step_feature_default(ctx);
    #endif

        step2(ctx);

    #ifdef DEBUG
        log_state(ctx);
    #endif

    #ifdef PLATFORM_B
        post_b(ctx);
    #endif
    }
```
겉보기에는 합리적으로 보일 수 있다.  
하지만 시간이 지나면 함수 하나가 여러 개의 부분 프로그램으로 찢어진다.

문제는 다음과 같다.

- 읽는 사람이 현재 어떤 빌드 변형을 상상해야 하는지 모른다
- 특정 플랫폼 + 특정 기능 + 특정 디버그 조합에서만 존재하는 실행 흐름이 생긴다
- 함수의 핵심 제어 흐름이 조건부 존재 코드에 가려진다
- 테스트 조합이 급증한다

즉, 조건부 컴파일이 본문 깊숙이 섞일수록 “한 함수”라는 추상화가 깨진다.  
이것은 단순 가독성 문제가 아니라 **함수의 의미가 빌드 조합별로 쪼개진다**는 문제다.

### 3.5 더 나은 구조는 보통 “분기를 본문이 아니라 경계로 밀어내는 것”이다

위 문제를 줄이는 더 나은 방향은 자주 다음과 같다.

#### 1) 플랫폼별 구현 파일 분리

공용 헤더:
```c
    void platform_pre(struct ctx *ctx);
    void platform_post(struct ctx *ctx);
```
플랫폼 A 구현 파일:
```c
    void platform_pre(struct ctx *ctx) { pre_a(ctx); }
    void platform_post(struct ctx *ctx) { }
```
플랫폼 B 구현 파일:
```c
    void platform_pre(struct ctx *ctx) { }
    void platform_post(struct ctx *ctx) { post_b(ctx); }
```
공용 로직:
```c
    void process(struct ctx *ctx) {
        platform_pre(ctx);
        step1(ctx);
        feature_step(ctx);
        step2(ctx);
        platform_post(ctx);
    }
```
이제 공용 흐름은 본문 안에서 찢기지 않는다.

#### 2) 설정 값을 런타임 정책으로 분리

디버그 로그가 단지 정책 차이라면:
```c
    if (ctx->debug_enabled) {
        log_state(ctx);
    }
```
이 구조는 코드 존재를 바꾸지 않고 실행 경로만 바꾼다.  
즉, 검증 공간이 줄어든다.

핵심은 `#ifdef`를 없애는 것이 아니라, **필요한 차이를 더 바깥 경계에서 한 번만 선택하게 만드는 것**이다.

### 3.6 선언부의 조건부 컴파일은 구현부보다 더 신중해야 한다

구현부 내부 분기도 위험하지만, 선언부 분기는 더 큰 문제를 만들 수 있다.

예를 들어:
```c
    #ifdef USE_FAST_MODE
    struct item {
        int a;
        int b;
    };
    #else
    struct item {
        int a;
    };
    #endif
```
이 코드는 구조체 레이아웃 자체를 바꾼다.  
이때 같은 헤더를 보는 모든 번역 단위가 정확히 동일한 매크로 상태를 공유하지 않으면 매우 위험하다.

즉, 선언/타입/ABI 경계에서의 조건부 컴파일은 구현부의 단순 분기보다 훨씬 더 강한 관리가 필요하다.

왜 이 직관이 중요할까?

많은 사람은 “어차피 같은 매크로로 같이 컴파일될 텐데”라고 생각한다.  
하지만 빌드 시스템, 외부 라이브러리, 생성 헤더, 개별 번역 단위 옵션이 엇갈리면 이 전제가 깨질 수 있다.

즉, 인터페이스를 바꾸는 조건부 컴파일은 **빌드 일관성 전체를 전제하는 고위험 선택**이다.

### 3.7 조건부 컴파일은 죽은 코드를 만들기 쉽다

다음과 같은 상황은 매우 흔하다.

- 예전 플랫폼 대응용 분기
- 지금은 쓰지 않는 컴파일러 대응 분기
- 특정 제품 라인 전용 기능 분기
- 과거 실험용 feature macro
- 임시 디버그 코드가 굳어진 분기

이 코드들은 존재하지만 실제로는 아무도 빌드하지 않을 수 있다.  
즉, “죽은 함수”보다 더 위험한 **죽은 빌드 변형 코드**가 된다.

왜 더 위험한가?

- 컴파일도 안 해 봤을 수 있다
- 리뷰어도 현재 빌드 기준으로만 본다
- 정적 분석도 활성 조합만 볼 수 있다
- 리팩터링 시 함께 깨지기 쉽다

즉, 조건부 컴파일의 큰 위험 중 하나는 코드가 “존재하지만 검증되지 않는 상태”로 오래 남을 수 있다는 점이다.

---

## 4. 핵심 구성 요소

### 4.1 설정 매크로(configuration macro)

조건부 컴파일의 출발점이다.

예:

- `DEBUG`
- `HAVE_EPOLL`
- `USE_OPENSSL`
- `COMPILER_HAS_BUILTIN_X`
- `TARGET_POSIX`

중요한 것은 매크로 이름 자체가 아니라, 그것이 어떤 종류의 차이를 표현하는가다.

- 컴파일 환경 차이
- 플랫폼 차이
- 기능 존재 여부
- 정책 차이
- 실험 차이

이 분류가 흐리면 `#ifdef`도 흐려진다.

### 4.2 config 헤더

조건부 컴파일을 흩뿌리지 않기 위한 핵심 장치다.  
빌드 시스템이나 탐지 스크립트가 생성하는 `config.h`류 파일이 대표적이다.

예:
```c
    #define HAVE_EPOLL 1
    #define HAVE_CLOCK_GETTIME 1
    #define HAVE___BUILTIN_EXPECT 1
```
이 구조의 장점은 다음과 같다.

- 매크로 출처를 한 곳에 모은다
- 빌드 환경 탐지와 코드 사용 지점을 분리한다
- “누가 이 매크로를 정의했는가”를 명확히 한다

즉, 좋은 조건부 컴파일은 보통 **매크로 결정과 매크로 소비를 분리**한다.

### 4.3 플랫폼 추상화 경계

다음과 같은 래퍼 계층이 중요하다.

- `platform_file.h`
- `os_socket.h`
- `compiler_attrs.h`
- `arch_barrier.h`

이런 계층의 목적은 공용 로직 본문에서 `#ifdef`를 제거하는 것이다.  
즉, 조건부 컴파일의 이상적 위치는 본문 내부가 아니라 **경계 어댑터 층**이다.

### 4.4 런타임 분기와의 구분

조건부 컴파일은 존재 여부 분기다.  
런타임 분기는 실행 경로 분기다.

실무적으로는 먼저 다음을 물어야 한다.

- 이 차이는 정말 코드 존재를 바꿔야 하는가
- 아니면 실행 정책만 바꾸면 되는가

이 질문을 건너뛰면 `#ifdef` 남용이 시작된다.

### 4.5 조합 공간(combination space)

조건부 컴파일을 볼 때는 항상 “현재 분기 하나”가 아니라 전체 조합 공간을 봐야 한다.

- 매크로 수
- 독립성 여부
- 상호 배타성 여부
- 실제 빌드 매트릭스
- 테스트 매트릭스

즉, `#ifdef` 리뷰는 문법 리뷰가 아니라 **변형 공간 관리 리뷰**여야 한다.

## 5. 실행 흐름 또는 처리 순서

### 5.1 유용한 경우: 컴파일러 속성 래핑
```c
    #if defined(__GNUC__) || defined(__clang__)
    #define NORETURN __attribute__((noreturn))
    #else
    #define NORETURN
    #endif

    NORETURN void fatal(const char *msg);
```
이 경우 조건부 컴파일은 적절하다.  
왜냐하면 속성 문법은 컴파일러별 차이이며, 런타임에 바꿀 수 없기 때문이다.

핵심은 다음이다.

- 차이가 컴파일 시점에만 결정 가능
- 공용 인터페이스 품질 개선
- 분기 범위가 작고 집중됨

### 5.2 유용한 경우: 시스템 기능 존재 여부
```c
    #if HAVE_EPOLL
        return event_loop_epoll_run(loop);
    #else
        return event_loop_poll_run(loop);
    #endif
```
이것도 상황에 따라 적절할 수 있다.  
특히 시스템이 해당 기능을 아예 제공하지 않으면 런타임 분기로는 해결되지 않는다.

다만 이 경우에도 더 나은 구조는 자주 다음과 같다.
```c
    int event_loop_run(struct loop *loop) {
        return event_loop_backend_run(loop);
    }
```
그리고 backend 선택을 별도 구현 파일/초기화 단계로 분리하는 것이다.  
즉, 유용하더라도 분기 범위를 줄이는 것이 중요하다.

### 5.3 나쁜 경우: 디버그 정책을 본문 곳곳에 흩뿌리기
```c
    void handle_request(struct req *r) {
    #ifdef DEBUG
        dump_req(r);
    #endif

        parse_req(r);

    #ifdef DEBUG
        dump_state();
    #endif

        execute_req(r);

    #ifdef DEBUG
        dump_resp();
    #endif
    }
```
이 구조는 처음에는 쉬워 보인다.  
하지만 디버그 정책이 코드 흐름 곳곳에 섞이면서 본문이 찢어진다.

더 나은 방향은 자주 다음과 같다.
```c
    if (debug_enabled) {
        dump_req(r);
    }
```
또는 logging API 내부에서 레벨 정책을 숨기는 것이다.

즉, 디버그 출력의 존재 자체를 없애야 하는 특별한 이유가 없다면, 런타임 정책이 더 낫다.

### 5.4 나쁜 경우: 기능 제품군을 `#ifdef`로 한 함수 안에 누적
```c
    int process_order(struct order *o) {
    #ifdef PRODUCT_A
        apply_a_discount(o);
    #endif

    #ifdef PRODUCT_B
        apply_b_discount(o);
    #endif

    #ifdef PRODUCT_C
        apply_c_rule(o);
    #endif

        finalize(o);
        return 0;
    }
```
이 구조는 제품 라인이 늘수록 본문이 파편화된다.  
실제로는 “하나의 함수”가 아니라 “여러 제품별 함수 조각의 겹침”이 된다.

더 나은 방향은 보통 전략 객체, 설정 테이블, 함수 포인터, 모듈 분리다.  
즉, 제품 정책 차이를 컴파일 타임 존재 차이로 바꾸지 않는 것이 중요하다.

### 5.5 고위험 경우: 공개 타입 정의를 조건부로 바꾸기
```c
    struct config {
        int timeout;
    #ifdef ENABLE_STATS
        int stats_level;
    #endif
    };
```
이 구조는 같은 헤더를 보는 모든 코드가 동일한 매크로 상태를 공유한다는 강한 전제를 둔다.  
이 전제가 깨지면 구조체 레이아웃 불일치와 ABI 문제로 이어질 수 있다.

즉, 인터페이스 타입 자체를 조건부로 바꾸는 것은 매우 신중해야 한다.  
가능하면 숨겨진 구현체 포인터, 별도 확장 구조, 명시적 버전 구조를 고려하는 편이 낫다.

## 6. 성능 / 최적화 관점

### 6.1 조건부 컴파일은 런타임 비용을 줄일 수 있지만, 유지비용을 크게 늘릴 수 있다

많은 사람이 `#ifdef`를 성능 최적화로 정당화한다.

- 디버그 코드 제거
- 특정 경로 완전 제거
- 플랫폼별 빠른 경로 선택
- 불필요한 분기 제거

이 논리는 때때로 맞다.  
하지만 성능 이득이 있다는 사실이 곧바로 좋은 구조라는 뜻은 아니다.

실무적으로 더 큰 비용은 다음이다.

- 조합별 빌드 테스트 필요
- 죽은 코드 축적
- 리뷰 난이도 상승
- 리팩터링 시 비활성 분기 파손
- 특정 조합에서만 나는 빌드/링크/ABI 오류

즉, 조건부 컴파일은 런타임 분기 비용을 줄이는 대신 **코드베이스 관리 비용을 앞으로 당겨서 지불**하게 만든다.

### 6.2 “런타임 분기보다 빠르다”는 이유는 자주 과장된다

디버그 플래그, 옵션 플래그, 정책 플래그 정도라면 런타임 분기 비용은 대개 관리 가능한 수준일 수 있다.  
그런데 이 차이를 무조건 `#ifdef`로 바꾸면 얻는 것은 1~2개의 branch 제거일 수 있지만, 잃는 것은 코드 조합 폭발이다.

즉, 성능 이유가 실제 hot path 측정에 근거한 것인지, 단순 직관인지 먼저 구분해야 한다.

### 6.3 조건부 컴파일의 진짜 최적화 포인트는 “컴파일 타임에만 결정 가능한 차이”다

다음은 조건부 컴파일이 특히 정당화되는 경우다.

- 해당 심볼/헤더/시스템 콜이 아예 존재하지 않음
- 특정 명령어 집합 코드를 다른 컴파일 옵션으로 빌드해야 함
- 컴파일러 속성/내장 함수 사용 가능 여부가 다름
- ABI 호환을 위해 특정 구조가 필요함

이 경우는 런타임 분기로 대체할 수 없거나, 대체 비용이 훨씬 크다.  
즉, 조건부 컴파일은 성능 도구라기보다 **컴파일 타임 환경 차이 흡수 도구**일 때 가장 설득력 있다.

### 6.4 잘못된 조건부 컴파일은 성능 최적화가 아니라 검증 회피가 될 수 있다

실무에서 종종 이런 일이 벌어진다.

- 복잡한 경로를 `#ifdef`로 빼서 일부 빌드에서는 아예 안 보이게 함
- 실험 코드가 활성 조합에서만 돌아감
- 문제 있는 플랫폼 코드를 비활성 상태로 방치
- “안 쓰는 옵션이니 괜찮다”가 누적됨

이 경우 조건부 컴파일은 최적화가 아니라 **검증되지 않은 코드를 숨기는 장치**가 된다.  
즉, 빠름보다 더 큰 구조적 부채를 만든다.

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
조건부 컴파일에 대한 오해는 주로 “분기”라는 단어 하나로 런타임 분기와 전처리 분기를 같은 종류로 취급할 때 생긴다.

### 7.1 실행 경로 분기와 존재 여부 분기가 갈리는 지점

- 런타임 `if`: 코드가 존재한 채 실행 경로만 갈린다
- `#if` / `#ifdef`: 코드 존재 자체가 빌드별로 달라진다

이 차이는 매우 크다.  
전자는 항상 파싱/타입검사/빌드 검증을 받지만, 후자는 특정 조합에서만 존재한다.

즉, 조건부 컴파일은 단순 “빠른 if”가 아니다.

### 7.2 구현 분기와 인터페이스 분기가 갈리는 지점

구현부 내부 분기:

- 함수 내부 알고리즘 차이
- 플랫폼별 helper 선택

인터페이스 분기:

- 타입 정의 차이
- 구조체 필드 차이
- 함수 선언 존재 차이
- ABI 차이

후자는 훨씬 더 위험하다.  
즉, 조건부 컴파일이 인터페이스까지 침범하는 순간 관리 강도가 높아져야 한다.

### 7.3 환경 차이와 정책 차이가 갈리는 지점

이 경계가 실무적으로 가장 중요하다.

- 환경 차이  
  컴파일러, 플랫폼, 시스템 기능, ABI
- 정책 차이  
  로그 레벨, 제품 기능, 실험 옵션, 운영 모드

전자는 조건부 컴파일 정당성이 크고,  
후자는 런타임 구성이나 모듈 선택이 더 나을 가능성이 높다.

즉, `#ifdef`는 환경 차이를 다루는 데 강하고, 정책 차이를 다루는 데는 남용되기 쉽다.

### 7.4 config 결정과 코드 소비가 갈리는 지점

좋은 구조는 보통 다음을 분리한다.

- 빌드 시스템/탐지 스크립트가 매크로를 결정
- `config.h`나 래퍼 헤더가 그 결과를 노출
- 구현 코드는 그 매크로를 제한적으로 소비

나쁜 구조는 곳곳에서 임의로 `-D` 플래그를 기대하거나, 파일마다 서로 다른 의미로 매크로를 해석하는 경우다.

즉, 조건부 컴파일도 **입력 결정 계층과 사용 계층 분리**가 중요하다.

### 7.5 경계 집중화와 본문 파편화가 갈리는 지점

- 좋은 구조: 헤더/어댑터/플랫폼 계층에 집중
- 나쁜 구조: 함수 본문, 알고리즘 본문, 비즈니스 로직 본문 깊숙이 분산

즉, `#ifdef`의 유용성은 종종 “있느냐 없느냐”보다 **어디에 배치했느냐**에서 갈린다.

## 8. 실무에서 중요한 판단 기준

### 8.1 먼저 “이 차이는 컴파일 시점에만 결정 가능한가”를 물어라

이 질문이 가장 중요하다.

- 헤더 존재 여부
- 시스템 기능 존재 여부
- 컴파일러 속성 지원 여부
- ABI 차이

이런 것은 조건부 컴파일이 정당화되기 쉽다.

반면 다음은 대개 다시 생각해야 한다.

- 디버그 출력
- 제품 정책
- 기능 토글
- 실험 옵션
- 운영 모드

이들은 종종 런타임 정책이 더 낫다.

### 8.2 조건부 컴파일은 가능한 한 경계로 몰아라

좋은 위치:

- `config.h`
- `compiler_attrs.h`
- `platform_*.h`
- 어댑터 구현 파일
- backend 선택 초기화 코드

나쁜 위치:

- 핵심 알고리즘 본문 군데군데
- 비즈니스 로직 조건 분기
- 자료구조 조작 함수 내부 곳곳
- 공용 타입 선언 전체를 뒤흔드는 곳

즉, 본문 안이 아니라 **경계에서 한 번만 선택**하게 만들어야 한다.

### 8.3 공용 타입/ABI를 바꾸는 분기는 극도로 신중하라

다음은 강한 경고 신호다.

- 구조체 필드 조건부 포함
- 공개 enum/typedef 조건부 변경
- 함수 프로토타입 존재 여부 조건부 변경
- 서로 다른 번역 단위에서 다른 정의 가능성

이 경우는 단순 구현 선택보다 위험도가 훨씬 높다.

### 8.4 매크로 축 수를 관리하라

실무적으로는 다음을 추적해야 한다.

- 활성 build macro 축이 몇 개인가
- 어떤 축은 서로 배타적인가
- 어떤 축은 조합 가능한가
- 실제 CI는 몇 조합을 빌드하는가
- 지원 조합과 미지원 조합이 문서화되어 있는가

즉, `#ifdef`를 리뷰할 때는 한 줄이 아니라 **조합 공간 증가량**을 봐야 한다.

### 8.5 죽은 분기를 주기적으로 없애라

다음은 제거 후보다.

- 더 이상 지원하지 않는 플랫폼 분기
- 오래된 컴파일러 우회 코드
- 사용 중지된 제품 라인 분기
- 임시 실험 매크로
- 한 번도 CI에 포함되지 않는 조합

조건부 컴파일은 쌓일수록 악화된다.  
즉, 유지 정책에는 추가뿐 아니라 **정기적 제거**가 포함되어야 한다.

### 8.6 코드베이스 규약을 정하라

예를 들어 다음 같은 규약이 유효할 수 있다.

- 플랫폼 차이는 플랫폼 추상화 계층에서만
- 공용 로직 본문에 `#ifdef` 최소화
- 정책 차이는 런타임 구성 우선
- 공개 타입 조건부 변경 금지 또는 엄격 제한
- 모든 지원 조합은 CI 빌드 대상
- 매크로 출처는 `config.h` 또는 지정 헤더에 한정

즉, 조건부 컴파일은 문법 문제가 아니라 **코드베이스 운영 규칙 문제**다.

## 9. 판단 체크리스트

다음 질문에 답하지 못하면, 그 조건부 컴파일은 위험할 가능성이 높다.

- 이 차이는 정말 컴파일 시점에만 결정 가능한가
- 런타임 분기나 모듈 분리로 해결할 수 없는가
- 이 `#ifdef`는 코드 조합 수를 얼마나 늘리는가
- 실제로 이 조합은 CI에서 빌드/테스트되는가
- 이 분기는 구현 세부인가, 공용 인터페이스/ABI까지 바꾸는가
- 분기를 함수 본문이 아니라 경계 헤더/어댑터로 밀어낼 수 없는가
- 매크로 출처와 의미가 문서화되어 있는가
- 오래된 플랫폼/기능/실험 분기를 계속 누적하고 있지 않은가
- 이 분기는 환경 차이를 표현하는가, 정책 차이를 표현하는가
- 1년 뒤 누가 이 조합을 실제로 유지할지 설명할 수 있는가

## 정리

조건부 컴파일은 유용하다.  
하지만 단순한 토글 문법이 아니다. 더 정확히 말하면, 조건부 컴파일은 **하나의 소스 트리에서 서로 다른 프로그램 변형을 만들어 내는 전처리기 수준의 존재 여부 분기 도구**다.

반드시 기억해야 할 핵심은 다음이다.

- `#ifdef`는 런타임 `if`와 다르다.
- 조건부 컴파일은 코드 존재 자체를 바꾸므로 검증 공간을 늘린다.
- 진짜 정당한 사용처는 컴파일 시점에만 결정 가능한 환경 차이 쪽이다.
- 정책 차이, 기능 토글, 디버그 출력은 자주 런타임 분기나 구조 분리가 더 낫다.
- 본문 깊숙이 흩어진 `#ifdef`는 함수를 빌드 조합별 조각으로 찢는다.
- 구현 분기보다 인터페이스/ABI 분기는 훨씬 더 신중해야 한다.
- 좋은 전략은 조건부 컴파일을 경계 헤더, config 헤더, 플랫폼 어댑터로 집중시키는 것이다.
- 조건부 컴파일을 추가하는 일은 새로운 조합 축을 추가하는 일과 같다.

즉, 조건부 컴파일을 잘 쓴다는 것은  
“필요할 때 `#ifdef`를 넣는다”가 아니라,  
**어떤 차이만 컴파일 타임에 두고, 그 차이를 코드베이스 어디에 집중시킬지 설계하는 것**에 가깝다.

## 10. 더 깊게 볼 포인트

### 10.1 include guard와 once는 어떻게 이해해야 하는가

다음 문서로 자연스럽다.  
조건부 컴파일과 같은 전처리기 층위에서, “중복 포함 제어”가 어떤 문제를 해결하고 어디까지 믿을 수 있는지로 이어지는 흐름이 맞다.

### 10.2 config 헤더는 어떻게 설계해야 조건부 컴파일이 흩어지지 않는가

빌드 시스템, feature detection, platform abstraction 사이의 역할 분리를 더 깊게 볼 수 있다.

### 10.3 공개 구조체 레이아웃을 조건부로 바꾸는 것은 왜 ABI 관점에서 특히 위험한가

전처리기와 ABI 경계가 만나는 고위험 사례를 더 엄밀하게 다룰 수 있다.

### 10.4 런타임 feature flag와 컴파일 타임 feature macro는 어떤 기준으로 나눠야 하는가

정책 차이와 환경 차이의 경계를 더 체계적으로 정리할 수 있다.

### 10.5 지원 조합 매트릭스를 CI에서 어떻게 강제해야 죽은 분기를 줄일 수 있는가

조건부 컴파일을 운영 관점에서 관리하는 방법으로 이어질 수 있다.
