## 1. 왜 이 주제를 깊게 봐야 하는가

C를 배우다 보면 함수형 매크로(function-like macro)는 매우 일찍 등장한다.
```c
    #define MAX(a, b) ((a) > (b) ? (a) : (b))
```
겉보기에는 함수와 비슷하다.  
이름이 있고, 인자를 받고, 값을 계산하는 것처럼 보인다. 그래서 많은 초보자는 자연스럽게 이렇게 생각한다.

- 함수 호출보다 가볍다
- 인라인 함수와 거의 같다
- 잘 괄호만 치면 안전하다
- 간단한 유틸리티는 매크로로 쓰면 된다
- 컴파일러가 알아서 최적화해 줄 테니 문제 없다

이 직관은 일부 경우에만 맞다.  
그리고 바로 그 점이 위험하다.

함수형 매크로의 진짜 문제는 “문법이 낡았다”거나 “느리다/빠르다”의 문제가 아니다.  
핵심은 **겉보기 호출 형태와 실제 언어 의미가 다르다**는 점이다. 즉, 함수처럼 보이지만 함수의 의미론을 갖지 않는다.

실무에서 이 차이가 문제를 만드는 이유는 다음과 같다.

- 인자가 한 번만 평가된다고 착각한다.
- 타입 검사가 함수처럼 이뤄진다고 착각한다.
- 디버깅 시 스택 프레임과 호출 경계를 기대한다.
- 연산자 우선순위가 자연스럽게 처리된다고 믿는다.
- 지역 변수/스코프/수명 규칙을 함수처럼 상상한다.
- 인자에 부작용이 있어도 함수 호출처럼 안전하다고 착각한다.

이 주제를 깊게 봐야 하는 이유는 세 가지다.

첫째, 함수형 매크로는 전처리기 단계에서 동작한다.  
즉, 이것은 C 언어의 함수 호출 메커니즘이 아니라, **토큰 치환 규칙**이다. 이 차이를 놓치면 이후 `inline`, `_Generic`, 조건부 컴파일, 헤더 설계까지 모두 흐려진다.

둘째, 함수형 매크로의 위험은 “실수하기 쉽다” 수준이 아니라, **언어가 보장하는 경계와 개발자가 기대하는 경계가 어긋난다**는 데 있다.  
그래서 테스트에서는 돌아가도, 사용 방식이 바뀌거나 인자 형태가 달라지면 갑자기 위험해진다.

셋째, 함수형 매크로는 무조건 나쁜 것이 아니다.  
상수 표현식 생성, 타입 일반화 전의 관용 패턴, 컴파일 타임 분기, 선언/속성 캡슐화 같은 영역에서는 여전히 유용할 수 있다. 즉, 중요한 것은 “금지”가 아니라 **적용 경계**다.

이 문서의 핵심 목표는 함수형 매크로를 “괄호 잘 치면 된다” 수준으로 설명하는 데 있지 않다.  
핵심은 다음을 분리하는 데 있다.

- 함수형 매크로는 언어 차원에서 정확히 무엇인가
- 함수처럼 보이지만 함수가 아닌 지점은 어디인가
- 왜 side effect, 평가 횟수, 우선순위, 타입, 디버깅에서 문제가 생기는가
- 어떤 문제는 괄호로 막을 수 있고, 어떤 문제는 구조적으로 못 막는가
- 실무에서는 언제 매크로를 피하고 언제 제한적으로 써야 하는가

즉, 이 문서는 함수형 매크로를 “짧은 문법 설탕”으로 설명하지 않는다.  
그 대신 **토큰 치환이 함수 호출처럼 오해될 때 생기는 구조적 위험**으로 설명한다.

## 2. 전체 구조

함수형 매크로를 정확히 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 전처리기 층위

함수형 매크로는 컴파일 이전, 정확히는 전처리 단계에서 토큰 치환으로 확장된다.  
즉, 이 시점에서는 아직 함수 호출도, 타입 검사도, 평가 순서도 없다.

따라서 함수형 매크로를 이해하려면 먼저 “이것은 실행 단위가 아니라 소스 재작성 규칙”이라는 점을 분명히 해야 한다.

### 2.2 언어 의미론 층위

매크로가 확장된 뒤에야 비로소 C 컴파일러는 표현식을 본다.  
즉, 최종적으로 해석되는 것은 매크로 이름이 아니라 “치환된 결과 코드”다.

이 때문에 매크로 사용자는 보통 매크로의 의도가 아니라 확장 결과의 의미를 책임져야 한다.

### 2.3 평가와 부작용 층위

함수 호출에서는 인자가 함수 본문과 분리되어 있고, 본문 안에서 매개변수는 하나의 이름으로 취급된다.  
반면 함수형 매크로는 인자 토큰이 본문 여러 곳에 복제될 수 있다.

그래서 다음 문제가 생긴다.

- 인자 다중 평가
- side effect 중복 수행
- 예상과 다른 실행 횟수
- 평가 순서와 결합된 버그

### 2.4 타입 / 스코프 / 디버깅 층위

함수형 매크로는 함수가 아니므로 다음을 자연스럽게 제공하지 않는다.

- 타입 서명
- 단일 호출 경계
- 디버깅 시 명확한 프레임
- 지역 스코프
- 심볼 수준의 독립 존재

즉, 함수형 매크로는 호출 문법만 함수와 닮았을 뿐, 함수 인터페이스의 핵심 계약들은 제공하지 않는다.

### 2.5 실무 설계 층위

실무에서는 함수형 매크로를 다음과 구분해서 판단해야 한다.

- `static inline` 함수
- 일반 함수
- `_Generic`
- enum / const / static const
- 선언용 매크로
- 조건부 컴파일용 매크로

즉, 함수형 매크로는 “가벼운 함수”가 아니라, 특정 문제에만 맞는 별도 도구다.

## 3. 내부 동작 원리

### 3.1 함수형 매크로는 함수 호출이 아니라 토큰 재작성이다

가장 먼저 깨야 할 오해는 이것이다.

- `MAX(a, b)`처럼 쓰면 함수가 호출된다

실제로는 함수 호출이 아니다.  
전처리기는 `MAX(x, y)`를 보고 단지 등록된 치환 규칙대로 토큰을 펼친다.

예를 들어 다음이 있다고 하자.
```c
    #define MAX(a, b) ((a) > (b) ? (a) : (b))
```
그렇다면
```c
    int m = MAX(x, y);
```
는 전처리 후 대략 다음처럼 된다.
```c
    int m = ((x) > (y) ? (x) : (y));
```
중요한 것은 이 시점에 `a`, `b`가 “변수”나 “매개변수 객체”로 존재하지 않는다는 점이다.  
그것들은 단지 치환 자리표시자였고, 확장 후에는 사라진다.

왜 이 직관이 중요한가?

많은 사람이 함수형 매크로를 함수처럼 상상하면서 다음을 기대한다.

- 인자가 먼저 계산된다
- 매개변수에 값이 들어간다
- 본문이 한 번 실행된다

하지만 매크로에는 그런 실행 경계가 없다.  
있었던 것은 오직 **텍스트가 아니라 토큰 수준의 코드 복제**다.

### 3.2 괄호는 일부 문제만 막아 준다

매크로 문서에서 가장 자주 보는 조언은 괄호다.

- 인자마다 괄호를 쳐라
- 전체 식도 괄호를 쳐라

이 조언은 맞다.  
예를 들어 다음은 위험하다.
```c
    #define SQR(x) x * x

    int v = SQR(1 + 2);
```
확장 결과는 다음처럼 된다.
```c
    int v = 1 + 2 * 1 + 2;
```
즉, 의도와 다른 결과가 나온다.

괄호를 넣으면 그 문제는 줄어든다.
```c
    #define SQR(x) ((x) * (x))
```
이제
```c
    int v = SQR(1 + 2);
```
는 다음처럼 된다.
```c
    int v = ((1 + 2) * (1 + 2));
```
이 문제는 해결된다.

하지만 여기서 중요한 오해가 하나 생긴다.

- 괄호를 잘 치면 함수형 매크로는 안전하다

이건 틀리다.  
괄호는 **우선순위와 결합 방식** 문제는 줄여 주지만, **다중 평가** 문제는 해결하지 못한다.

예:
```c
    int i = 3;
    int v = SQR(i++);
```
확장 결과는 다음과 비슷하다.
```c
    int v = ((i++) * (i++));
```
이건 함수 호출처럼 “인자 한 번 평가”가 아니다.  
즉, 괄호는 중요하지만 만능이 아니다.

### 3.3 가장 본질적인 위험은 “인자가 복제된다”는 점이다

함수형 매크로의 핵심 위험은 평가가 아니라 **토큰 복제**다.

다음 매크로를 보자.
```c
    #define MIN(a, b) ((a) < (b) ? (a) : (b))
```
그리고 다음 코드를 보자.
```c
    int x = 10;
    int y = 20;
    int m = MIN(x++, y);
```
확장 결과는 대략 다음이다.
```c
    int m = ((x++) < (y) ? (x++) : (y));
```
이제 `x++`는 한 번이 아니라 두 번 등장할 수 있다.  
그리고 조건 분기에 따라 실제 증가 횟수도 달라질 수 있다.

이 점이 왜 함수와 다른가?

함수였다면:
```c
    static inline int min_int(int a, int b) {
        return a < b ? a : b;
    }

    int m = min_int(x++, y);
```
이 경우 인자 `x++`는 호출 시 한 번만 평가된다.  
그 뒤 함수 본문 안에서는 `a`라는 값이 사용될 뿐이다.

즉, 함수형 매크로는 “본문 안에서 매개변수를 여러 번 쓴다”가 아니라,  
정확히는 **인자 표현식 자체가 여러 위치에 복제된다**는 점이 핵심이다.

### 3.4 side effect가 있는 인자는 매크로와 결합될 때 위험해진다

실무에서 특히 중요한 규칙은 다음이다.

- side effect가 있는 표현식을 함수형 매크로 인자로 넘기는 것은 매우 조심해야 한다

side effect 예:

- `i++`
- `--n`
- `*p++`
- `arr[idx++]`
- `f()` 호출
- 락 획득/해제 성격의 함수
- 상태를 바꾸는 getter처럼 보이는 함수

예를 들어:
```c
    #define ABS(x) ((x) < 0 ? -(x) : (x))

    int v = ABS(read_sensor());
```
겉보기에는 괜찮아 보일 수 있다.  
하지만 확장 결과는 `read_sensor()`가 두 번 나타날 수 있다.

즉, “인자를 함수처럼 넘긴다”는 직관이 틀린다.  
실제로는 “인자 표현식 조각을 본문 여러 곳에 붙여 넣는다”에 더 가깝다.

이 때문에 side effect 인자는 함수형 매크로와 결합될 때 구조적으로 위험하다.  
그리고 이 문제는 괄호로 해결되지 않는다.

### 3.5 타입 검사는 함수처럼 되지 않는다

함수에는 시그니처가 있다.
```c
    int foo(int x, int y);
```
이 구조에서는 호출 시 타입 불일치, 변환, 경고, 프로토타입 검사가 작동한다.

반면 함수형 매크로는 그런 시그니처가 없다.
```c
    #define ADD(a, b) ((a) + (b))
```
이 매크로는 정수끼리도, 포인터와 정수도, 부동소수끼리도, 구조체에 대해 잘못 쓰는 시도도 일단 확장 자체는 한다.  
문제가 드러나는 것은 확장 뒤, 최종 식을 컴파일할 때다.

즉, 매크로는 “이런 타입을 받는 인터페이스”를 직접 선언하지 않는다.  
그 결과 사용자는 매크로 이름에서 타입 계약을 읽기 어렵다.

왜 이 직관이 틀리는가?

많은 사람이 `ADD(x, y)`를 보며 함수처럼 생각한다.  
하지만 실제로는 타입 검사 대상이 `ADD`가 아니라 **치환된 식 전체**다.  
즉, 매크로는 함수형 인터페이스처럼 보이지만, 인터페이스 경계를 강하게 만들지 못한다.

### 3.6 디버깅과 진단 경계도 함수와 다르다

함수는 디버깅할 때 다음 장점을 가진다.

- 호출 스택에 이름이 보인다
- 브레이크포인트를 걸기 쉽다
- 인자와 지역 변수를 구분해 본다
- 단일 호출 경계가 있다

반면 함수형 매크로는 전처리 후 사라진다.  
디버거는 최종 확장된 코드만 본다. 컴파일러 경고도 종종 “매크로 확장 결과의 어느 부분”에서 터진다.

즉, 매크로 버그는 종종 원래 호출문이 아니라 확장 결과를 봐야 이해된다.  
이 점은 작은 유틸리티에서는 사소해 보여도, 대형 코드베이스에서는 진단 비용을 크게 높인다.

### 3.7 함수형 매크로는 스코프와 이름 공간도 함수처럼 갖지 않는다

함수는 독립된 정의 단위다.  
하지만 매크로는 전처리기 이름 공간에 들어간다.

- 동일 이름 충돌 가능
- 헤더 포함 순서 영향
- `#undef` 필요성
- 의도치 않은 이름 재정의
- 지역 스코프 개념 부재

즉, 함수형 매크로는 호출 문법은 함수처럼 보이지만, 선언/정의/스코프 관리 측면에서는 함수가 아니다.  
이 때문에 헤더 설계와 라이브러리 인터페이스에서 더 조심해야 한다.

## 4. 핵심 구성 요소

### 4.1 매크로 이름

함수처럼 보이는 진입점이다.  
하지만 이 이름은 심볼이 아니라 전처리기 치환 트리거다.

즉, `MAX(...)`라는 형태는 호출 표현식처럼 보이지만, 실제로는 “이 이름을 보면 치환하라”는 규칙에 가깝다.

### 4.2 매크로 매개변수

함수 매개변수처럼 보이지만, 실제로는 자리표시 토큰이다.  
확장 후에는 독립된 객체나 값으로 존재하지 않는다.

이 차이가 중요한 이유는 다음과 같다.

- 한 번 평가된 값을 저장해 두는 개념이 없다
- 본문에서 여러 번 등장하면 인자 표현식도 여러 번 복제된다

### 4.3 매크로 본문

함수 본문처럼 보여도 실행 블록이 아니다.  
그것은 단지 최종 C 식/문장으로 펼쳐질 토큰 조합이다.

즉, 본문 품질은 “읽기 좋은가”보다 “확장 후 어떤 C 코드가 되느냐”로 평가해야 한다.

### 4.4 괄호 전략

함수형 매크로에서 기본적인 방어 수단이다.

- 각 인자를 괄호로 감싼다
- 전체 결과도 괄호로 감싼다

이것은 우선순위 오류를 줄이는 데 유효하다.  
하지만 다중 평가, 타입 계약 부재, 디버깅 어려움은 해결하지 못한다.

### 4.5 side effect 인자

함수형 매크로 위험도의 핵심 판단 기준이다.

- 증가/감소 연산
- 대입 포함 식
- 상태 변경 함수 호출
- 숨은 부작용이 있는 getter류 함수
- volatile/MMIO 접근 식

이런 인자가 매크로 본문에 여러 번 복제되면 의미가 크게 달라질 수 있다.

### 4.6 대체 수단

실무 판단에서 함께 고려해야 할 도구들이다.

- `static inline`
- 일반 함수
- `_Generic`
- enum / const / static const
- do-while 문 매크로
- 선언 전용 매크로

즉, 함수형 매크로는 항상 첫 선택지가 아니다.

## 5. 실행 흐름 또는 처리 순서

### 5.1 연산자 우선순위 문제

잘못된 매크로:
```c
    #define ADD1(x) x + 1

    int v = 2 * ADD1(3);
```
확장 결과:
```c
    int v = 2 * 3 + 1;
```
의도는 `2 * (3 + 1)`일 수 있지만 실제는 다르다.

보완:
```c
    #define ADD1(x) ((x) + 1)
```
이제 확장 결과는 의도에 더 가깝다.

이 예는 함수형 매크로가 왜 “본문 전체 괄호”와 “인자 괄호”를 둘 다 필요로 하는지 보여 준다.

### 5.2 다중 평가 문제

고전적인 예:
```c
    #define MAX(a, b) ((a) > (b) ? (a) : (b))

    int i = 1;
    int j = 2;
    int m = MAX(i++, j);
```
확장 결과:
```c
    int m = ((i++) > (j) ? (i++) : (j));
```
이제 `i++`는 조건식과 결과식에 모두 나타날 수 있다.

핵심은 다음이다.

- 함수였다면 인자 평가 1회
- 매크로는 표현식 복제
- 부작용이 있는 인자에서 의미 붕괴

### 5.3 타입 계약 착시

다음 매크로는 범용처럼 보인다.
```c
    #define ABS(x) ((x) < 0 ? -(x) : (x))
```
겉보기에는 어떤 숫자 타입에도 쓸 수 있어 보인다.  
하지만 실제로는 부호 없는 타입, 오버플로우 가능성, 사용자 정의 의미를 가진 표현식 등에 대해 함수처럼 명확한 계약이 없다.

즉, 매크로는 “타입 제약을 가진 API”처럼 보일 수 있지만, 실제로는 치환 뒤 식의 의미에 문제를 떠넘긴다.

### 5.4 디버깅 시 확장 결과 추적

다음 호출이 있다고 하자.
```c
    value = CLAMP(read_temp(), low, high);
```
매크로가 복잡하면 디버깅 시 개발자는 종종 호출문만 보고 추론하려 한다.  
하지만 실제 문제는 확장된 표현식의 평가 횟수, 연산자 결합, 타입 변환에서 생길 수 있다.

즉, 함수형 매크로 문제는 소스의 호출 위치보다 **전처리 후 코드**를 봐야 드러나는 경우가 많다.

### 5.5 문장형 사용에서의 함정

함수형 매크로라 해도 문장처럼 쓰이는 경우가 많다.
```c
    #define SET_FLAG(x) x |= FLAG

    if (cond)
        SET_FLAG(mask);
    else
        do_other();
```
이런 구조는 괄호나 블록 처리 방식에 따라 예상 밖 해석을 만들 수 있다.  
즉, 함수처럼 보이는 매크로라도 식(expression)인지 문(statement)인지, 단일 문장처럼 안전한지 따져야 한다.

이 지점은 문장형 매크로와 연결되지만, 함수형 매크로도 호출 문법 때문에 같은 오해를 자주 낳는다.

## 6. 성능 / 최적화 관점

### 6.1 “매크로가 더 빠르다”는 직관은 현대 기준에서 단순화다

과거에는 함수 호출 비용을 피하기 위해 매크로가 자주 쓰였다.  
하지만 오늘날에는 `static inline`이 매우 강력한 대안이다.

즉, “호출 오버헤드가 싫으니 매크로”라는 판단은 더 이상 자동으로 맞지 않는다.

왜 이 직관이 틀리기 쉬운가?

- 컴파일러는 작은 함수를 인라인할 수 있다
- 함수는 타입 검사와 디버깅 경계를 유지한다
- 매크로는 빠를 수 있어도 의미론적 비용이 크다

따라서 성능만을 이유로 함수형 매크로를 고르는 것은 점점 설득력이 약하다.

### 6.2 함수형 매크로의 비용은 런타임보다 의미론적 비용에 있다

실무에서 더 큰 비용은 보통 다음이다.

- 다중 평가 버그
- 디버깅 난이도 상승
- 진단 메시지 복잡화
- 타입 계약 불명확
- 코드 리뷰 추론 비용 증가
- 헤더 오염 및 이름 충돌

즉, 함수형 매크로의 핵심 비용은 CPU 사이클보다 **사람이 의미를 추적하는 비용**에 가깝다.

### 6.3 side effect 인자에서는 성능이 아니라 correctness가 먼저 무너진다

부작용 있는 인자를 매크로에 넣으면 문제는 “조금 느리다”가 아니다.  
아예 수행 횟수와 의미가 달라질 수 있다.

예:

- I/O 레지스터 읽기 두 번 발생
- 락 획득 함수 두 번 호출
- 카운터 증가 두 번
- 상태 변경 함수 중복 실행

즉, 이런 경우 매크로의 문제는 최적화 문제가 아니라 correctness 붕괴다.

### 6.4 헤더 수준 범용성은 얻지만 인터페이스 정밀도는 잃기 쉽다

매크로는 헤더에서 쉽게 배포되고, 타입 일반화 비슷한 효과를 낼 수 있다.  
하지만 그 대가로 다음을 잃기 쉽다.

- 명시적 타입 서명
- 심볼 기반 디버깅
- 안전한 호출 경계
- 오용 시 조기 진단

즉, 함수형 매크로는 “유연성”을 얻는 대신 “정밀한 인터페이스”를 희생하는 경향이 있다.

---

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
함수형 매크로에 대한 오해는 대개 “함수처럼 보이는 표면”과 “토큰 치환이라는 실제 동작”을 같은 것으로 취급할 때 생긴다.

### 7.1 호출 문법과 호출 의미가 갈리는 지점

다음은 겉보기에는 같다.
```c
    foo(x, y)
    MAX(x, y)
```
하지만 실질은 다르다.

- 함수 호출: 인자 평가 + 매개변수 바인딩 + 함수 본문 실행
- 매크로 호출 형태: 인자 토큰 치환 + 확장 결과 컴파일

즉, 함수형 매크로는 **호출처럼 보이는 치환**이지 호출 그 자체가 아니다.

### 7.2 괄호로 막을 수 있는 문제와 못 막는 문제가 갈리는 지점

괄호는 다음을 줄인다.

- 우선순위 오류
- 결합 방식 오류
- 일부 문맥 결합 문제

하지만 괄호로 못 막는 것은 다음이다.

- 다중 평가
- side effect 중복
- 타입 계약 부재
- 디버깅 경계 부재
- 이름 공간 오염

즉, “괄호를 잘 쳤으니 안전하다”는 결론은 틀리다.

### 7.3 inline 함수와 매크로가 갈리는 지점

둘 다 겉보기에는 “호출 비용이 적은 작은 로직”에 쓰일 수 있다.  
하지만 보장 수준은 다르다.

- `inline` 함수: 함수 의미론 유지
- 매크로: 전처리 치환

즉, 인자 평가 횟수, 타입 검사, 디버깅 경계, 스코프 측면에서 둘은 같은 계열 도구가 아니다.

### 7.4 범용성 착시와 타입 계약이 갈리는 지점

매크로는 여러 타입에서 쓸 수 있어 “범용”처럼 보인다.  
하지만 이것은 종종 “명시적 계약이 없다”는 뜻이기도 하다.

즉, 매크로가 다양한 식에 확장된다는 사실은 강점일 수 있지만, 동시에 **무엇이 허용되는지 인터페이스가 흐리다**는 뜻일 수 있다.

### 7.5 언어 규칙과 전처리기 규칙이 갈리는 지점

이것이 가장 큰 경계다.

- 함수는 C 언어 의미론 안에 있다
- 매크로는 전처리기 규칙에서 먼저 작동한다

따라서 함수형 매크로 문제를 언어 규칙만으로 이해하려 하면 항상 뒤늦게 이해하게 된다.  
먼저 “확장 결과가 무엇인가”를 보고, 그다음 그 코드가 어떤 C 의미를 갖는지 봐야 한다.

## 8. 실무에서 중요한 판단 기준

### 8.1 side effect 인자를 받는 매크로는 먼저 의심하라

다음 인자를 매크로에 넘긴다면 바로 경계해야 한다.

- `i++`, `--j`
- 함수 호출
- MMIO/volatile 접근
- 락/참조 카운트/상태 변경 함수
- 대입 포함 식

이런 인자를 허용하는 함수형 매크로는 구조적으로 위험하다.

### 8.2 단순 계산이라도 먼저 `static inline` 가능성을 보라

다음 질문을 먼저 해야 한다.

- 이걸 함수로 표현하면 안 되는가
- 타입 계약이 필요한가
- 디버깅 경계가 필요한가
- 다중 평가 위험을 없애야 하는가

이 질문에 하나라도 “예”라면 함수형 매크로보다 `static inline`이 더 나은 경우가 많다.

### 8.3 함수형 매크로가 적합한 경우를 좁게 잡아라

다음은 상대적으로 매크로가 설득력 있는 경우다.

- 타입 일반화가 꼭 필요한 작은 표현식
- 컴파일 타임 상수 조합
- 조건부 컴파일과 결합된 래핑
- 속성/선언/어노테이션 캡슐화
- 매우 제한된 사용 규약이 문서화된 경우

즉, “작고 자주 쓰는 계산”이라는 이유만으로 매크로를 고르면 안 된다.

### 8.4 매크로는 호출자 계약을 더 강하게 문서화해야 한다

함수보다 안전 장치가 약하므로, 오히려 더 강한 문서화가 필요하다.

- side effect 인자를 넘기지 말 것
- 인자는 순수한 식일 것
- 특정 타입만 의도함
- 결과를 lvalue로 쓰지 말 것
- 문장/식 문맥 제약

즉, 매크로 인터페이스는 종종 함수보다 더 엄격한 사용 규약을 필요로 한다.

### 8.5 헤더 공용 API에서는 이름 오염과 추론 비용을 같이 보라

함수형 매크로는 한 번 헤더에 들어가면 광범위하게 퍼진다.  
따라서 다음을 같이 봐야 한다.

- 이름 충돌 가능성
- include 순서 영향
- 디버깅 비용
- 코드 리뷰에서의 오해 가능성
- 외부 사용자의 예상 사용 방식

즉, 공용 헤더일수록 함수형 매크로는 더 신중해야 한다.

### 8.6 “겉보기 호출” 때문에 더 위험하다는 점을 잊지 마라

일반 상수 매크로보다 함수형 매크로가 더 위험한 이유는,  
개발자가 너무 쉽게 함수와 같은 정신 모델을 덧씌우기 때문이다.

즉, 함수형 매크로의 위험성은 문법 복잡성보다 **잘못된 직관을 유발하는 표면 형태**에 있다.

---

## 9. 판단 체크리스트

다음 질문에 명확히 답하지 못하면, 그 함수형 매크로는 위험할 가능성이 높다.

- 이 매크로는 정말 함수가 아니라 전처리 치환이어도 되는가
- 동일 기능을 `static inline`으로 표현할 수 없는가
- 인자가 본문에서 여러 번 나타나지 않는가
- side effect 인자를 넘겨도 의미가 깨지지 않는가
- 괄호는 인자와 전체 식에 모두 적용되었는가
- 타입 계약이 불명확해서 오용 가능성이 크지 않은가
- 디버깅/진단 비용을 감수할 이유가 있는가
- 헤더에 퍼질 때 이름 오염 문제가 없는가
- 호출 문법 때문에 함수처럼 오해될 여지는 없는가
- 문서에서 사용 제약을 명확히 적고 있는가

## 정리

함수형 매크로는 함수처럼 보이지만 함수가 아니다.  
더 정확히 말하면, 함수형 매크로는 **인자를 받아 실행되는 호출 단위가 아니라, 인자 토큰을 본문에 치환해 새로운 C 코드를 만들어 내는 전처리 규칙**이다.

반드시 기억해야 할 핵심은 다음이다.

- 함수형 매크로의 본질은 호출이 아니라 치환이다.
- 괄호는 중요하지만 일부 문제만 막는다.
- 가장 큰 구조적 위험은 인자 표현식의 다중 복제다.
- side effect 인자는 함수형 매크로와 결합될 때 매우 위험하다.
- 타입 검사, 디버깅 경계, 스코프 의미는 함수와 다르다.
- 현대 C에서는 많은 경우 `static inline`이 더 나은 대안이다.
- 함수형 매크로는 금지 대상이 아니라 적용 경계를 좁게 잡아야 하는 도구다.

즉, 함수형 매크로를 이해한다는 것은  
“짧게 쓰는 함수 비슷한 문법”을 안다는 것이 아니라,  
**전처리기가 만든 겉보기 호출과 실제 언어 의미가 어디서 갈라지는지 이해하는 것**에 가깝다.

## 10. 더 깊게 볼 포인트

### 10.1 매크로와 inline 함수는 언제 다르게 선택해야 하는가

다음 문서로 가장 자연스럽다.  
함수형 매크로의 위험을 봤다면, 실제 실무에서 언제 매크로를 유지하고 언제 `static inline`으로 바꿔야 하는지로 바로 이어지는 흐름이 맞다.

### 10.2 문장형 매크로는 왜 `do { ... } while (0)` 패턴을 쓰는가

함수형 매크로와는 다르지만, “겉보기 한 문장”과 “실제 확장 결과”의 차이를 이해하는 데 직접 연결된다.

### 10.3 `_Generic`은 함수형 매크로의 어떤 문제를 줄이고 어떤 문제는 그대로 남기는가

타입 일반화와 인터페이스 정밀도를 함께 다룰 수 있는 다음 확장 주제다.

### 10.4 volatile/MMIO 접근을 매크로로 감쌀 때는 왜 더 조심해야 하는가

다중 평가가 correctness 붕괴로 바로 이어지는 대표 사례다.

### 10.5 공용 헤더에서 매크로 기반 API를 설계할 때 문서화해야 할 사용 제약은 무엇인가

이름 공간, 사용 문맥, side effect 금지, 타입 제약을 어떻게 명시할지 더 깊게 다룰 수 있다.
