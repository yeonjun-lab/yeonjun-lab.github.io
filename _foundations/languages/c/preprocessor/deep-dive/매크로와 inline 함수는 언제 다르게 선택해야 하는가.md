## 1. 왜 이 주제를 깊게 봐야 하는가

C 코드베이스를 읽다 보면 같은 목적처럼 보이는 두 도구가 계속 등장한다.

- 함수형 매크로
- `inline` 또는 `static inline` 함수

겉보기에는 둘 다 비슷해 보인다.

- 짧은 로직을 감싼다
- 헤더에 넣어 배포할 수 있다
- 호출 오버헤드를 줄이려는 의도로 보인다
- 유틸리티처럼 사용된다

그래서 많은 개발자가 초기에 다음처럼 생각한다.

- 매크로는 옛날 방식이고 inline 함수가 최신 방식이다
- 성능이 중요하면 매크로를 쓰고, 안전성이 중요하면 inline 함수를 쓴다
- 컴파일러가 어차피 인라인해 줄 테니 둘은 거의 같다
- 함수처럼 쓰고 싶으면 inline, 타입을 가리지 않으려면 매크로다
- 작은 계산은 아무거나 써도 차이가 없다

이 직관들은 일부만 맞는다.  
그리고 바로 그 부분적 맞음이 위험하다.

실무에서 중요한 것은 “둘 다 짧은 코드를 감싼다”는 표면 유사성이 아니라, **둘이 속한 층위가 다르다**는 점이다.

- 매크로는 전처리기 치환 규칙이다.
- inline 함수는 여전히 C 함수다.
- 매크로는 호출처럼 보이지만 호출이 아니다.
- inline은 호출 비용을 줄일 수 있어도 함수 의미론을 유지한다.

즉, 이 둘의 차이는 최적화 정도가 아니라 **언어 경계와 인터페이스 정밀도**의 차이다.

이 주제를 깊게 봐야 하는 이유는 세 가지다.

첫째, 실무에서 “작은 유틸리티니까 매크로로 해도 된다”라는 판단은 자주 잘못된다.  
작다고 해서 타입 계약, 디버깅 경계, side effect 안전성, 스코프 의미가 사라지는 것은 아니다. 오히려 작은 코드일수록 무심코 널리 퍼지고, 널리 퍼질수록 잘못된 선택의 비용이 커진다.

둘째, 반대로 “무조건 inline 함수가 낫다”도 틀릴 수 있다.  
조건부 컴파일, 선언 조합, 타입 일반성, 컴파일 타임 상수 조합, 토큰 결합 같은 문제는 inline 함수가 원천적으로 해결할 수 없다. 즉, 매크로는 단지 낡은 대체물이 아니라, 다른 문제를 푸는 도구이기도 하다.

셋째, 이 선택은 헤더 설계, ABI 경계, 빌드 구조, 공용 API 품질과 직접 연결된다.  
무엇을 매크로로 공개하고 무엇을 함수로 공개할지에 따라, 사용자가 보게 되는 계약의 강도와 오용 가능성이 달라진다.

이 문서의 핵심 목표는 “매크로 vs inline”을 장단점 표 하나로 끝내는 데 있지 않다.  
핵심은 다음을 분리하는 것이다.

- 둘은 어떤 층위에서 다르게 동작하는가
- 어떤 문제는 inline 함수가 구조적으로 더 낫고, 어떤 문제는 매크로만 가능한가
- 성능, 타입, 디버깅, 헤더 배포, 조건부 컴파일, 상수 표현식 관점에서 어떻게 갈리는가
- 왜 “가볍다”는 이유만으로 매크로를 고르면 안 되는가
- 실무에서 선택 기준을 어떻게 문서화하고 일관되게 유지해야 하는가

즉, 이 문서는 매크로와 inline 함수를 “비슷한 도구의 취향 차이”로 다루지 않는다.  
그 대신 **전처리기 도구와 언어 도구를 어떤 경계에서 구분해 써야 하는가**의 문제로 다룬다.

## 2. 전체 구조

이 주제를 정확히 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 전처리기와 언어 의미론 층위

가장 먼저 갈라지는 지점이다.

- 매크로는 전처리기에서 토큰 치환으로 확장된다.
- inline 함수는 C 언어의 함수 규칙 안에서 해석된다.

즉, 둘은 같은 “짧은 코드 캡슐화 도구”처럼 보여도, 애초에 작동하는 층위가 다르다.

### 2.2 인터페이스 계약 층위

inline 함수는 함수 시그니처를 가진다.

- 타입 계약
- 반환형
- 인자 평가 1회
- 디버깅 가능한 호출 경계

반면 매크로는 이런 계약을 직접 제공하지 않는다.  
즉, 같은 이름 뒤에 괄호가 붙어 있어도 사용자에게 제공되는 인터페이스 정밀도는 다르다.

### 2.3 최적화 / 코드 생성 층위

많은 사람이 이 문제를 “호출 오버헤드” 하나로 축소한다.  
하지만 실제로는 다음이 함께 걸린다.

- 인라인 가능성
- 중복 코드 생성
- 디버깅 품질
- 번역 단위별 처리
- 헤더 포함 확산
- 코드 크기 증가

즉, 성능만으로 판단하면 거의 반드시 단순화된다.

### 2.4 헤더 설계 층위

둘 다 헤더에 넣기 쉬워 보이지만 의미는 다르다.

- 매크로는 include 순간 이름 공간을 오염시킬 수 있다.
- `static inline`은 함수 정의를 각 번역 단위에 제공한다.
- 공용 API에서 무엇을 노출할지에 따라 사용자가 기대하는 계약이 달라진다.

### 2.5 실무 선택 층위

결국 중요한 것은 이 질문이다.

- 지금 내가 풀려는 문제가 “계산 로직 캡슐화”인가
- “조건부 토큰 생성/선언 조합”인가
- “타입 안전한 작은 함수”인가
- “상수 표현식이 꼭 필요한가”인가

즉, 선택 기준은 스타일이 아니라 문제 유형이어야 한다.

## 3. 내부 동작 원리

### 3.1 inline 함수는 함수 의미론을 유지한 채 인라인될 수 있는 함수다

가장 먼저 분명히 해야 할 점은 이것이다.

`inline`은 “함수가 아니다”를 뜻하지 않는다.  
오히려 반대다. `inline`은 **여전히 함수이되, 컴파일러에게 인라인 가능성을 열어 둔 함수**다.

예:

    static inline int max_int(int a, int b) {
        return a > b ? a : b;
    }

이 함수는 다음 성질을 가진다.

- 인자 `a`, `b`는 함수 호출 규칙에 따라 한 번 평가된다.
- 타입이 시그니처에 의해 결정된다.
- 디버깅 시 함수 경계로 다룰 수 있다.
- 컴파일러가 상황에 따라 실제 호출로 둘 수도 있고 인라인할 수도 있다.

즉, inline 함수의 본질은 “호출 오버헤드 제거”보다 **함수 의미론을 보존하면서 코드 삽입 최적화를 허용한다**는 데 있다.

### 3.2 매크로는 함수 의미론을 유지하지 않는다

반면 매크로는 애초에 함수가 아니다.

    #define MAX(a, b) ((a) > (b) ? (a) : (b))

이것은 인자를 받고 값을 반환하는 것처럼 보이지만, 실제로는 단지 토큰 치환 규칙이다.  
따라서 다음이 다르다.

- 인자 평가 1회 보장 없음
- 시그니처 없음
- 디버깅 경계 없음
- 스코프 없음
- 타입 계약 없음

즉, inline 함수와 매크로의 차이는 “컴파일러가 인라인하느냐” 이전에 이미 갈려 있다.

왜 이 직관이 중요한가?

많은 사람이 이렇게 생각한다.

- inline 함수는 함수지만 결국 인라인되면 매크로와 비슷하다

이 말은 코드 생성 결과 일부에서는 비슷해 보일 수 있어도,  
언어 계약과 오용 가능성 측면에서는 전혀 다르다.

### 3.3 인자 평가 모델이 구조적으로 다르다

가장 분명한 차이 중 하나는 인자 평가다.

inline 함수:

    static inline int square_int(int x) {
        return x * x;
    }

    int v = square_int(i++);

여기서 `i++`는 함수 호출 시 한 번 평가되고, 그 값이 `x`에 들어간다.

매크로:

    #define SQR(x) ((x) * (x))

    int v = SQR(i++);

확장 결과는 대략 다음이다.

    int v = ((i++) * (i++));

즉, 이 차이는 우연한 버그 가능성이 아니라 **구조적 의미 차이**다.  
함수는 인자 값을 받지만, 매크로는 인자 표현식을 복제한다.

따라서 side effect가 있는 인자를 받을 가능성이 있다면, inline 함수가 구조적으로 훨씬 안전하다.

### 3.4 타입 계약이 필요한 순간 inline 함수 쪽으로 기운다

다음과 같은 작은 유틸리티를 생각해 보자.

    static inline size_t min_size(size_t a, size_t b) {
        return a < b ? a : b;
    }

이 함수는 단순하지만 장점이 분명하다.

- `size_t`라는 계약이 있다
- 부호 혼합 문제를 줄인다
- 경고가 더 명확해진다
- 호출자가 어떤 타입을 넘겨야 하는지 읽을 수 있다

같은 로직을 매크로로 만들면:

    #define MIN(a, b) ((a) < (b) ? (a) : (b))

사용 범위가 넓어지는 대신 계약은 흐려진다.  
즉, “범용처럼 보인다”는 장점은 자주 “타입 경계가 약하다”는 비용을 동반한다.

실무에서는 이것이 매우 중요하다.  
작은 도우미일수록 오히려 여러 문맥에서 재사용되므로, 타입 계약이 흐린 도구는 나중에 더 큰 비용을 만든다.

### 3.5 그렇다고 inline 함수가 매크로를 완전히 대체하지는 못한다

여기서 자주 나오는 과잉 단순화는 이것이다.

- inline 함수가 있으니 매크로는 필요 없다

이건 틀리다.  
매크로만 가능한 문제들이 분명히 있다.

예:

- 조건부 컴파일에 따른 토큰 포함/제외
- 속성/선언/스토리지 클래스 조합
- 식별자 결합
- 문자열화
- 특정 컴파일러 확장 캡슐화
- 상수 표현식이나 정적 초기화에 필요한 조합
- 타입 일반화 전처리 패턴

즉, inline 함수는 “함수 의미론을 가진 계산 로직”에 강하고,  
매크로는 “언어 코드 조각 생성”에 강하다.

이 차이를 놓치면 두 극단 중 하나로 흐른다.

- 계산 로직도 매크로로 남용
- 전처리 문제까지 함수로 억지 해결 시도

### 3.6 성능 이유만으로 매크로를 고르는 것은 점점 설득력이 약하다

과거에는 함수 호출 비용을 피하려고 매크로를 쓰는 관행이 강했다.  
하지만 현대 C 컴파일러는 작은 `static inline` 함수를 매우 잘 다룬다.

즉, 다음 판단은 점점 약해졌다.

- 성능이 중요하니 매크로
- 작은 함수는 어차피 매크로가 낫다

왜 이 직관이 틀리기 쉬운가?

- inline 함수도 충분히 호출 오버헤드를 줄일 수 있다
- 컴파일러는 타입과 문맥을 더 잘 이해한 채 최적화할 수 있다
- 매크로는 성능과 무관한 의미론적 위험이 크다
- 실제 비용은 CPU보다 디버깅과 오용에 있는 경우가 많다

즉, 성능이 목적이라면 먼저 inline 함수가 가능한지 보는 편이 더 현대적이고 보수적인 판단이다.

### 3.7 공용 헤더에서의 선택은 단지 구현 문제가 아니라 API 계약 문제다

헤더에 매크로를 넣는 순간 그것은 전처리기 이름 공간에 전파된다.

- 이름 충돌 가능성
- `#undef` 필요성
- include 순서 영향
- 호출자가 함수처럼 오해할 가능성

반면 `static inline` 함수는 함수 정의를 헤더에 제공하더라도, 여전히 언어 차원의 인터페이스를 유지한다.

즉, 공용 API 설계에서 다음이 중요하다.

- 사용자가 이것을 “함수 계약”으로 이해해야 하는가
- 아니면 “전처리 규칙”으로 이해해야 하는가

이 질문에 전자가 답이면, 매크로보다 inline 함수가 훨씬 낫다.

## 4. 핵심 구성 요소

### 4.1 함수 의미론(function semantics)

inline 함수의 핵심 자산이다.

- 인자 평가 1회
- 시그니처 존재
- 타입 검사
- 지역 변수/스코프
- 디버깅 가능한 경계

즉, inline 함수는 “최적화 가능한 함수”이지, “함수 비슷한 것”이 아니다.

### 4.2 토큰 치환(token substitution)

매크로의 본질이다.

- 전처리 단계 동작
- 본문 토큰 확장
- 언어 의미론 이전에 적용
- 사용 위치에 따라 전혀 다른 식으로 해석될 수 있음

즉, 매크로는 코드를 계산하는 도구가 아니라 코드를 만들어 내는 도구에 더 가깝다.

### 4.3 타입 계약(type contract)

선택 기준의 핵심 요소다.

- 특정 타입만 의도하는가
- 경고와 오류를 조기에 받고 싶은가
- 호출자 오용을 줄여야 하는가

이 질문에 “예”가 많을수록 inline 함수 쪽이 유리하다.

### 4.4 상수 표현식 / 전처리 능력

매크로 쪽이 강한 영역이다.

- `#if`와 결합
- 토큰 결합
- 속성 캡슐화
- 선언 조합
- 식별자 생성
- 문자열화

즉, “언어 실행 의미”가 아니라 “소스 구조 생성”이 목적이면 매크로가 더 적합할 수 있다.

### 4.5 헤더 배포 방식

둘 다 헤더에 들어갈 수 있지만 의미는 다르다.

- 매크로: 전처리기 이름 공간 오염 가능
- `static inline`: 번역 단위마다 정의 제공, 그러나 함수 계약 유지

즉, 헤더 친화성만으로 둘을 같은 계열 도구로 보면 안 된다.

---

## 5. 실행 흐름 또는 처리 순서

### 5.1 side effect 인자를 받을 수 있는 작은 계산

잘못된 선택:

    #define MAX(a, b) ((a) > (b) ? (a) : (b))

    int m = MAX(i++, j);

이 경우 매크로는 구조적으로 위험하다.

더 나은 선택:

    static inline int max_int(int a, int b) {
        return a > b ? a : b;
    }

    int m = max_int(i++, j);

이 경우 인자 평가가 한 번으로 고정된다.

핵심은 다음이다.

- 계산이 작다는 사실보다
- 인자 의미가 안전해야 한다는 사실이 먼저다

### 5.2 특정 타입 전용 유틸리티

예:

    static inline uint32_t align_up_u32(uint32_t x, uint32_t a) {
        return (x + (a - 1u)) & ~(a - 1u);
    }

이 로직을 매크로로도 만들 수 있다.  
하지만 특정 정수 폭과 의미를 전제로 한다면, 함수 시그니처가 더 강한 문서화가 된다.

즉, “재사용성”보다 “의도한 타입 계약”이 더 중요할 때는 inline 함수가 자연스럽다.

### 5.3 조건부 속성 또는 선언 조합

예를 들어 특정 컴파일러에서만 속성을 붙여야 하는 경우:

    #if defined(__GNUC__)
    #define NORETURN __attribute__((noreturn))
    #else
    #define NORETURN
    #endif

이런 문제는 inline 함수가 할 수 없다.  
왜냐하면 이것은 계산 로직이 아니라 선언 조합이기 때문이다.

즉, 전처리기 수준의 선언 생성 문제는 매크로 영역이다.

### 5.4 타입 일반화가 필요한 작은 선택자

예를 들어 여러 숫자 타입에서 동일 이름을 쓰고 싶다면, 단순 inline 함수 하나로는 부족할 수 있다.  
이 경우 `_Generic`과 매크로를 조합하는 식이 등장할 수 있다.

즉, “타입별로 다른 함수를 고르는 이름 디스패치”는 함수보다 매크로 쪽이 자연스러운 문제일 수 있다.

### 5.5 디버깅이 중요한 공용 유틸리티

공용 헤더에 널리 퍼질 작은 도우미라면, 매크로보다 inline 함수가 대체로 낫다.

이유:

- 호출 경계가 더 잘 드러남
- 타입 오류가 더 명확함
- 매크로 오용 비용 감소
- IDE/디버거/정적 분석 도구 친화성 증가

즉, 널리 배포될수록 “가벼움”보다 “계약 명확성”이 더 중요해진다.

---

## 6. 성능 / 최적화 관점

### 6.1 매크로가 더 빠르다고 일반화하면 안 된다

가장 흔한 단순화는 이것이다.

- 매크로는 치환이니까 무조건 빠르다
- 함수는 호출이니까 느리다
- inline 함수도 결국 함수라서 불리할 수 있다

이 직관은 현대 컴파일 환경에서는 과장되기 쉽다.

- 작은 `static inline`은 충분히 인라인될 수 있다
- 함수 시그니처 정보가 최적화에 오히려 도움이 될 수 있다
- 매크로는 코드 팽창과 진단 비용을 늘릴 수 있다

즉, 성능 차이를 추상적으로 상상해 매크로를 고르는 것은 근거가 약하다.

### 6.2 inline 함수의 장점은 성능과 의미론을 함께 잡을 수 있다는 점이다

작은 계산 로직에 대해 inline 함수가 강한 이유는 다음 조합 때문이다.

- 인자 평가 1회
- 타입 검사
- 디버깅 가능성
- 여전히 인라인 최적화 가능

즉, “매크로처럼 빠르고 함수처럼 안전하다”가 아니라,  
정확히는 **함수 계약을 유지하면서도 매크로가 예전에 가졌던 성능상의 이유 일부를 상당 부분 대체한다**는 뜻이다.

### 6.3 매크로의 진짜 비용은 의미론적 비용이다

매크로를 성능 도구로만 보면 다음 비용을 놓친다.

- side effect 버그
- 타입 계약 약화
- 헤더 오염
- 디버깅 난이도
- 정적 분석 품질 저하
- 코드 리뷰 비용 증가

즉, 매크로 선택은 CPU 사이클 문제가 아니라 **언어 경계 희생 비용** 문제로 봐야 한다.

### 6.4 공용 유틸리티는 실행 비용보다 오용 비용이 더 크다

널리 쓰이는 헤더 유틸리티에서는 다음이 더 중요하다.

- 잘못된 인자를 넣어도 쉽게 깨지지 않는가
- IDE와 분석 도구가 잘 이해하는가
- 디버깅 시 추적하기 쉬운가
- 사용자가 함수처럼 오해해도 큰 사고가 적은가

이 질문에 inline 함수가 더 좋은 답을 주는 경우가 많다.

---

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
매크로와 inline 함수 선택에서 생기는 오해는 대개 “둘 다 짧은 코드를 감싼다”는 표면 공통점을 실제 보장 수준까지 확장할 때 생긴다.

### 7.1 전처리기 규칙과 함수 의미론이 갈리는 지점

가장 큰 경계다.

- 매크로: 전처리기 치환 규칙
- inline 함수: 함수 의미론 유지

이 차이는 호출 오버헤드보다 훨씬 중요하다.  
즉, inline 함수는 언어 내부 도구이고, 매크로는 언어 외곽 도구다.

### 7.2 인라인 가능성과 함수 계약이 갈리는 지점

많은 사람이 “인라인되면 결국 매크로와 비슷하다”고 생각한다.  
하지만 틀리다.

- 인라인 여부는 코드 생성 문제
- 함수 계약은 언어 의미 문제

즉, 실제 기계어가 비슷해질 수 있어도, 인터페이스 보장 수준은 여전히 다르다.

### 7.3 타입 일반성과 타입 무계약이 갈리는 지점

매크로는 여러 타입에서 “작동할 수” 있다.  
하지만 그것이 곧바로 “좋은 타입 일반성”을 뜻하지는 않는다.

- 의도된 타입 분기와 계약이 있는 일반성  
  `_Generic` 또는 명시적 함수군이 더 적절할 수 있음
- 그냥 아무 식이나 넣을 수 있음  
  계약 부재일 수 있음

즉, “매크로라서 범용”이라는 말은 때로 장점이 아니라 위험 신호다.

### 7.4 계산 로직과 코드 생성 로직이 갈리는 지점

이 질문이 실무적으로 가장 중요하다.

- 지금 필요한 것이 “값 계산”인가  
  inline 함수 후보
- 아니면 “선언/토큰/속성/조건부 코드 생성”인가  
  매크로 후보

즉, 계산 문제에 매크로를 쓰기 시작하면 매크로 남용이 되기 쉽다.

### 7.5 공용 API와 내부 구현 편의가 갈리는 지점

내부에서 잠깐 쓰는 매크로와 공용 헤더에 노출되는 매크로는 비용이 다르다.  
공용 API는 사용자의 정신 모델에 직접 영향을 준다.

- 함수처럼 이해되길 원한다  
  inline 함수가 더 적합
- 전처리 규칙임을 노출해도 된다  
  매크로 가능

즉, 공용 인터페이스일수록 보장 강도가 높은 도구를 우선해야 한다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 기본값은 먼저 inline 함수를 검토하라

다음 질문에 하나라도 “예”면 기본적으로 inline 함수부터 보라.

- 계산 로직인가
- 타입 계약이 필요한가
- side effect 인자를 안전하게 처리해야 하는가
- 디버깅 경계가 필요한가
- 공용 헤더에 노출되는가

즉, 작은 계산 유틸리티의 기본 선택은 매크로보다 `static inline`이 더 안전한 경우가 많다.

### 8.2 매크로는 “함수로는 풀 수 없는 문제”에 우선 배치하라

다음은 매크로가 더 자연스러운 영역이다.

- 조건부 컴파일
- 토큰 결합
- 문자열화
- 속성/선언 조합
- 컴파일러별 어노테이션 감싸기
- 상수 표현식용 조합
- 타입 디스패치 전처리 래퍼

즉, 매크로는 계산보다 코드 생성 문제에서 더 정당화된다.

### 8.3 side effect 가능성이 있으면 매크로를 매우 엄격히 제한하라

다음 인자가 들어갈 가능성이 있는 API라면 매크로를 재검토해야 한다.

- 증가/감소 연산
- 함수 호출
- MMIO/volatile 읽기
- 상태 변경 getter
- 락/참조 카운트 조작

이 경우 inline 함수가 사실상 기본값이다.

### 8.4 공용 헤더일수록 함수 계약을 선호하라

공용 API에서 중요한 것은 빠름보다 예측 가능성이다.

- 타입이 보인다
- IDE가 이해한다
- 경고가 명확하다
- 디버깅이 쉽다
- 오용을 줄인다

이 다섯 가지는 매크로보다 inline 함수가 더 잘 제공하는 경우가 많다.

### 8.5 성능 이유만으로 매크로를 선택하지 마라

선택 전에 먼저 물어야 한다.

- 실제 측정이 있는가
- inline 함수가 충분히 인라인되지 않는 근거가 있는가
- 매크로가 줄이는 비용보다 의미론적 비용이 더 크지 않은가

즉, 성능은 추상적 감각이 아니라 근거가 필요하다.

### 8.6 팀 차원의 선택 기준을 문서화하라

이 문제는 개인 취향으로 남기면 코드베이스가 금방 불균질해진다.  
예를 들어 다음 기준을 둘 수 있다.

- 계산 로직은 기본적으로 `static inline`
- 매크로는 전처리기 기능이 꼭 필요할 때만
- side effect 인자를 받을 수 있는 매크로 금지
- 공용 헤더 매크로는 naming과 문서화 강화
- `_Generic` 도입 가능 영역은 별도 검토

즉, 매크로 vs inline 선택은 스타일이 아니라 코드베이스 규약이어야 한다.

## 9. 판단 체크리스트

다음 질문에 답하지 못하면, 그 선택은 아직 불안정하다.

- 이 로직은 값 계산인가, 코드 생성인가
- 함수 의미론이 필요한가
- 인자 평가 1회 보장이 필요한가
- 타입 계약이 사용자에게 드러나야 하는가
- 디버깅 경계가 필요한가
- side effect 인자가 들어와도 안전해야 하는가
- 공용 헤더에서 널리 노출되는가
- 상수 표현식/토큰 결합/조건부 컴파일 같은 전처리기 능력이 꼭 필요한가
- 성능 이유가 실제 측정 기반인가
- 이 선택이 코드베이스 전체 규약과 일관되는가

## 정리

매크로와 inline 함수는 둘 다 짧은 로직을 감싸는 도구처럼 보이지만, 같은 도구가 아니다.  
더 정확히 말하면,

- 매크로는 전처리기 단계의 코드 생성 도구이고
- inline 함수는 함수 의미론을 유지한 채 인라인 최적화를 허용하는 언어 도구다.

반드시 기억해야 할 핵심은 다음이다.

- 계산 로직이라면 기본적으로 inline 함수를 먼저 검토하는 편이 안전하다.
- 타입 계약, 인자 평가 1회, 디버깅 경계가 중요하면 inline 함수가 유리하다.
- 조건부 컴파일, 토큰 결합, 선언 조합처럼 코드 생성이 목적이면 매크로가 필요할 수 있다.
- 성능만을 이유로 매크로를 선택하는 것은 현대 기준에서 점점 설득력이 약하다.
- 공용 헤더일수록 함수 계약의 이점이 커진다.
- 매크로는 금지 대상이 아니라, 전처리기 기능이 정말 필요한 좁은 경계에서 써야 하는 도구다.

즉, 이 선택의 본질은  
“어느 쪽이 더 짧고 빠른가”가 아니라,  
**내가 지금 필요한 것이 함수 의미론인가, 전처리기 코드 생성 능력인가**를 구분하는 데 있다.

## 10. 더 깊게 볼 포인트

### 10.1 조건부 컴파일은 언제 유용하고 언제 코드베이스를 망치는가

다음 문서로 자연스럽다.  
매크로가 진짜 필요한 대표 영역이 바로 조건부 컴파일이기 때문이다. 전처리기 기능의 정당한 사용과 남용 경계를 이어서 볼 수 있다.

### 10.2 `_Generic`은 매크로와 inline 함수 사이의 어떤 틈을 메우는가

타입 일반성과 함수 계약 사이의 절충 지점을 더 깊게 볼 수 있다.

### 10.3 공용 헤더에서 `static inline`을 배포할 때의 ODR 유사 문제, 코드 크기, 디버깅 품질은 어떻게 균형 잡아야 하는가

헤더 설계와 빌드 구조 관점에서 더 세밀한 논의가 가능하다.

### 10.4 volatile/MMIO 접근 래퍼는 왜 매크로보다 inline 함수가 더 안전한 경우가 많은가

다중 평가와 의미론적 경계가 correctness에 직접 영향을 주는 사례다.

### 10.5 선언용 매크로와 계산용 매크로를 코드베이스 규약에서 어떻게 분리해야 하는가

전처리기 사용 정책을 더 정교하게 다룰 수 있다.
