## 정의

`extern`과 `static`은 C에서 이름의 연결 범위(linkage)와 저장 기간, 가시성을 이해할 때 핵심이 되는 키워드다.  
겉보기에는 단순한 선언 수식어처럼 보이지만, 실제로는 **이 이름이 다른 파일에서 보이는가**, **어느 저장공간에 존재하는가**, **링커가 이 이름을 어떻게 다루는가**와 직접 연결된다.

핵심은 다음이다.

1. `extern`은 보통 "이 이름의 정의가 다른 곳에도 연결될 수 있다"는 뜻과 연결된다
2. `static`은 문맥에 따라 의미가 달라지지만, 파일 범위에서는 보통 "이 번역 단위 내부로 가시성을 제한한다"는 뜻과 연결된다
3. 함수 내부의 `static`은 또 다른 의미를 가지며, 수명은 길고 스코프는 지역적이다
4. 이 둘을 구분하지 못하면 중복 정의, 링크 실패, 잘못된 전역 노출 문제가 쉽게 생긴다

즉 `extern`과 `static`은 단순 문법이 아니라  
**심볼을 외부에 공개할지 숨길지, 객체를 언제까지 유지할지 결정하는 도구**다.

---

## 먼저 구분해야 할 세 가지 축

`extern`과 `static`을 정확히 이해하려면 아래 세 축을 먼저 분리해야 한다.

### 1. 스코프(scope)

이 이름을 소스 코드 어디에서 사용할 수 있는가

### 2. 저장 기간(storage duration)

이 객체가 언제 생성되고 언제까지 살아 있는가

### 3. 연결 범위(linkage)

이 이름이 다른 번역 단위와 연결될 수 있는가

많은 혼동은 이 세 가지를 섞어서 보기 때문에 생긴다.  
특히 `static`은 문맥에 따라 저장 기간과 연결 범위를 함께 바꾸는 것처럼 보여 더 헷갈린다.

---

## 핵심 개념

### 1. 파일 범위의 `static`은 외부 연결을 끊는다

예시:
```
    static int counter = 0;

    static void helper(void) {
        counter++;
    }
```
이 코드에서 `counter`와 `helper`는 현재 `.c` 파일 안에서만 보이는 이름으로 취급된다.  
즉 다른 `.c` 파일에서 같은 이름으로 참조할 수 없다.

핵심은 다음이다.

- 객체 자체는 정적 저장 기간을 가진다
- 하지만 이름의 연결은 내부 링크(internal linkage)로 제한된다
- 결과적으로 심볼이 현재 번역 단위 내부 구현으로 숨겨진다

즉 파일 범위의 `static`은 "전역처럼 오래 살지만, 외부에는 숨긴다"로 이해하면 좋다.

---

### 2. 파일 범위의 `extern`은 외부 정의와 연결될 수 있음을 나타낸다

예시:
```
`config.h`

    extern int global_limit;
```
```
`config.c`

    int global_limit = 100;
```
여기서 헤더의 `extern int global_limit;`은  
"이 이름의 실제 정의는 다른 번역 단위 어딘가에 있다"는 의미다.

즉 `extern` 선언은 보통 다음을 뜻한다.

- 저장공간을 새로 만드는 정의가 아님
- 이 이름을 외부 심볼로 참조하겠다는 선언
- 실제 정의는 다른 파일에 있어야 함

따라서 여러 파일이 같은 전역 객체를 함께 참조할 수 있다.

---

### 3. `extern`은 "참조 선언", `static`은 "내부 숨김" 쪽으로 기억하는 것이 좋다

물론 세부 규칙은 더 있지만, 기초 단계에서는 아래처럼 잡는 것이 가장 유용하다.

- `extern` → 외부에 있는 정의를 참조하거나 외부 연결을 전제로 하는 이름
- `static` → 현재 번역 단위 내부로 이름을 숨김

이 관점이 빌드/링킹 문맥에서는 가장 실용적이다.

---

## 파일 범위 변수에서 어떻게 다르게 동작하는가

### 1. 외부 연결을 가진 전역 변수

예시:
```
`a.c`

    int shared = 10;
```
```
`b.c`

    extern int shared;

    int get_value(void) {
        return shared;
    }
```
이 경우 `shared`는 외부 연결을 가지므로 `b.c`에서도 참조 가능하다.

즉:

- `a.c`는 정의를 제공
- `b.c`는 선언을 통해 참조
- 링커가 두 파일을 연결

---

### 2. 내부 연결을 가진 파일 전용 변수

예시:
```
`a.c`

    static int shared = 10;
```
```
`b.c`

    extern int shared;
```
이 경우는 연결되지 않는다.  
왜냐하면 `a.c`의 `shared`는 외부에 공개되지 않기 때문이다.

즉 이름은 같아 보여도:

- `static int shared`는 외부 심볼이 아님
- `extern int shared`는 외부 심볼을 찾으려 함
- 따라서 링크가 실패할 수 있다

이 차이가 매우 중요하다.

---

## 함수에서도 같은 원리가 적용된다

### 1. 일반 함수 정의

예시:
```
    int add(int a, int b) {
        return a + b;
    }
```
이 함수는 기본적으로 외부에서 참조 가능한 함수 심볼이 될 수 있다.

---

### 2. `static` 함수

예시:
```
    static int add(int a, int b) {
        return a + b;
    }
```
이 함수는 현재 `.c` 파일 내부에서만 사용하도록 제한된다.

즉 `static` 함수는 다음 용도로 매우 중요하다.

- 내부 헬퍼 숨기기
- API 표면 축소
- 이름 충돌 방지
- 구현 세부사항 은닉

실무적으로는 "외부로 공개하지 않을 함수는 `static`을 검토한다"가 좋은 습관이다.

---

## 함수 내부의 `static`은 왜 또 다르게 느껴지는가

여기서 `static`은 의미가 달라진다.

예시:
```
    void f(void) {
        static int count = 0;
        count++;
    }
```
이 경우 `count`는 함수 안에서만 이름이 보인다.  
즉 스코프는 지역적이다.

하지만 수명은 지역변수처럼 함수 호출 때마다 생겼다 사라지는 것이 아니다.  
프로그램 전체 실행 동안 유지된다.

즉 함수 내부 `static`은 다음 특성을 가진다.

- 스코프: 블록 내부
- 저장 기간: 정적 저장 기간
- 값 유지: 함수 호출 사이에 유지됨

이것이 일반 지역변수와의 핵심 차이다.

---

## 지역변수와 함수 내부 `static` 비교

### 일반 지역변수

예시:
```
    void f(void) {
        int x = 0;
        x++;
    }
```
`f`를 호출할 때마다 `x`는 새로 만들어지고 초기화된다.  
호출이 끝나면 사라진다.

### 함수 내부 `static` 변수

예시:
```
    void f(void) {
        static int x = 0;
        x++;
    }
```
여기서 `x`는 처음 한 번 초기화된 뒤 계속 유지된다.  
다음 호출에서도 이전 값이 남아 있다.

즉 같은 함수 내부 선언이어도 저장 기간이 완전히 다르다.

---

## 왜 `extern` 선언이 헤더에 자주 들어가는가

헤더는 여러 `.c` 파일이 공통으로 볼 선언을 두는 곳이다.  
따라서 외부에 정의된 전역 변수나 함수 선언을 헤더에 두는 패턴이 자주 나온다.

예시:
```
`counter.h`

    extern int g_counter;
    void increment(void);
```
```
`counter.c`

    int g_counter = 0;

    void increment(void) {
        g_counter++;
    }
```
이 구조의 장점은 다음과 같다.

- 정의는 한 곳에만 둔다
- 선언은 여러 곳에서 공유한다
- 링커가 하나의 정의와 여러 참조를 연결한다

즉 `extern`은 선언/정의 분리 패턴의 핵심 도구다.

---

## 왜 전역 변수 정의를 헤더에 두면 문제가 생기기 쉬운가

예시:
```
`bad.h`

    int g_value = 10;
```
이 헤더를 여러 `.c` 파일이 포함하면, 각 번역 단위마다 `g_value` 정의가 생길 수 있다.  
그 결과 링크 단계에서 multiple definition 문제가 생길 수 있다.

올바른 패턴은 보통 다음이다.
```
`good.h`

    extern int g_value;
```
```
`good.c`

    int g_value = 10;
```
즉 헤더에는 선언, 소스 파일에는 정의를 둔다.

이 패턴은 `extern`의 의미를 정확히 이해해야 자연스럽다.

---

## `extern`은 함수 선언에도 사실상 깔려 있다고 볼 수 있다

예시:
```
    int add(int, int);
```
이 함수 선언은 보통 외부 연결을 전제로 하는 선언으로 생각해도 된다.  
즉 다른 파일에서 정의된 함수일 수도 있고, 같은 파일 뒤쪽에서 정의될 수도 있다.

따라서 함수 선언에서는 `extern`을 명시적으로 잘 쓰지 않아도,  
개념적으로 외부 연결 가능성을 전제하는 경우가 많다.

반면:
```
    static int add(int, int);
```
이렇게 쓰면 현재 번역 단위 내부로 제한된다.

즉 함수 선언에서도 `static`은 가시성 제어 도구다.

---

## 심볼 관점에서 다시 보면

이 주제는 결국 심볼 가시성 이야기다.

### 외부 연결(external linkage)

다른 번역 단위에서도 같은 이름으로 연결될 수 있음

예:
- 일반 전역 함수
- 일반 파일 범위 전역 변수
- `extern`으로 선언된 외부 객체 참조

### 내부 연결(internal linkage)

현재 번역 단위 내부에서만 이름이 유효함

예:
- 파일 범위 `static` 변수
- 파일 범위 `static` 함수

### 무연결(no linkage)에 가까운 지역 이름

예:
- 일반 지역변수
- 함수 매개변수

즉 `extern`과 `static`은 스코프가 아니라  
**링커가 이 이름을 어떻게 볼지**를 조정하는 핵심 장치다.

---

## 자주 하는 오해

### 오해 1. `static`은 무조건 "고정된 값"이라는 뜻이다

아니다.  
C에서 `static`은 값이 안 바뀐다는 뜻이 아니다.  
문맥에 따라 저장 기간 또는 연결 범위를 조정한다.

---

### 오해 2. `extern`을 쓰면 메모리가 새로 생긴다

아니다.  
보통 `extern` 선언은 새로운 저장공간 정의가 아니라, 외부 정의를 참조하기 위한 선언이다.

---

### 오해 3. 전역 변수는 다 다른 파일에서도 보인다

항상 아니다.  
파일 범위 `static`이면 현재 번역 단위 내부에서만 보인다.

---

### 오해 4. 함수 내부 `static`은 전역 변수와 같다

완전히 같지는 않다.

- 저장 기간은 길다
- 하지만 이름 스코프는 함수 내부에 제한된다

즉 "전역처럼 오래 살지만 지역 이름으로 숨겨진 객체"에 가깝다.

---

### 오해 5. 헤더에 전역 변수 정의를 넣어도 괜찮다

대체로 위험하다.  
여러 소스 파일이 그 헤더를 포함하면 중복 정의가 생길 수 있다.

보통은 헤더에 `extern` 선언, `.c` 파일에 정의를 둔다.

---

## 실무 관점

### 1. 공개 API와 내부 구현을 나누는 핵심 도구다

라이브러리나 모듈을 만들 때 모든 함수를 외부로 노출할 필요는 없다.  
외부에서 쓸 함수만 공개하고, 내부 헬퍼는 `static`으로 숨기는 것이 좋다.

예시:

- 공개 함수: 헤더에 선언
- 내부 함수: `.c` 내부 `static`

이렇게 하면 모듈 경계가 명확해진다.

---

### 2. 이름 충돌을 줄인다

대규모 프로젝트에서는 흔한 이름이 충돌하기 쉽다.

예를 들어 여러 파일에 `helper`, `init`, `parse` 같은 이름이 있을 수 있다.  
이때 내부 헬퍼를 `static`으로 두면 외부 심볼 충돌을 줄일 수 있다.

즉 `static`은 단순 숨김이 아니라 심볼 공간 관리 도구다.

---

### 3. 상태 유지 로직에서 함수 내부 `static`이 유용할 수 있다

예시:

- 호출 횟수 카운터
- 간단한 캐시
- 한 번만 초기화할 상태

하지만 이 방식은 전역적 숨은 상태를 만들 수 있으므로, 멀티스레드/재진입성 측면에서는 신중해야 한다.

즉 편리하지만 비용도 있다.

---

### 4. 링크 오류를 해석할 수 있게 된다

예를 들어:

- 선언은 `extern`인데 실제 정의가 없음
- 정의는 `static`이라 외부에서 못 찾음
- 헤더에 정의를 넣어 중복 심볼이 생김

이런 문제는 `extern`/`static` 의미를 알면 바로 해석할 수 있다.

---

### 5. 다음 문서들과 직접 연결된다

이 문서는 다음 주제와 강하게 연결된다.

- 헤더 파일과 선언/정의 분리는 왜 필요한가
- 정적 라이브러리와 동적 라이브러리는 무엇이 다른가
- 오브젝트 파일과 심볼이란 무엇인가
- 링크 에러는 왜 발생하는가

즉 `extern`과 `static`은 빌드/링킹 축에서 가장 실전적인 개념 중 하나다.

---

## 정리

`extern`과 `static`은 이름의 연결 범위와 객체의 수명을 이해할 때 핵심 키워드다.

핵심은 다음과 같다.

- 파일 범위 `static`은 이름을 현재 번역 단위 내부로 숨긴다
- `extern`은 외부에 있는 정의와 연결될 수 있음을 나타낸다
- 함수 내부 `static`은 지역 스코프를 유지하면서 값은 계속 살아 있게 한다
- 헤더에는 보통 `extern` 선언을 두고, 정의는 `.c` 파일에 둔다
- 심볼 가시성을 잘 설계하면 모듈화와 링크 안정성이 좋아진다

즉 `extern`과 `static`을 이해한다는 것은  
단순 키워드 암기가 아니라 **어떤 이름을 외부에 공개하고 어떤 상태를 어디까지 유지할지 설계하는 것**이다.

이 관점이 잡히면 헤더 설계, 전역 변수 관리, 내부 헬퍼 숨김, 링크 오류 해석이 훨씬 명확해진다.
