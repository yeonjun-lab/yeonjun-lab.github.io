## 1. 왜 이 주제를 공부했는가

C를 처음 배울 때는 대부분 하나의 `.c` 파일 안에서 모든 코드를 작성한다.  
이 단계에서는 변수, 함수, 배열, 포인터, 동적 메모리 같은 개념을 익히는 데 집중할 수 있다.  
하지만 실제 프로그램이 조금만 커져도 곧 다음 문제가 생긴다.

- 함수가 많아져서 한 파일이 너무 커진다.
- 여러 기능을 분리하고 싶다.
- 다른 파일에서도 같은 함수나 변수를 써야 한다.
- 공통 인터페이스와 내부 구현을 나누고 싶다.
- 빌드 시간이 길어지거나 구조가 복잡해진다.
- 팀 단위로 작업할 때 경계를 명확히 해야 한다.

이 시점부터는 더 이상 “C 문법”만으로는 부족하다.  
프로그램이 **여러 파일로 어떻게 나뉘고, 서로 어떻게 연결되며, 컴파일러와 링커가 그 관계를 어떻게 해석하는지**를 이해해야 한다.

바로 여기서 등장하는 것이 다음 개념들이다.

- 헤더 파일
- 선언(declaration)
- 정의(definition)
- `extern`
- 중복 정의
- 심볼(symbol)
- 링크(link)

입문 단계에서는 보통 이렇게 외운다.

- `.h`에는 선언을 넣는다.
- `.c`에는 정의를 넣는다.

이 설명은 방향은 맞지만 충분하지 않다.  
왜 그렇게 해야 하는지 이해하지 못하면, 실제 프로젝트에서 곧 다음 문제를 만나게 된다.

- 왜 같은 함수를 여러 파일에서 쓸 수 있는가?
- 왜 함수 선언만 있어도 컴파일이 되는가?
- 왜 정의를 헤더에 잘못 넣으면 중복 정의 에러가 나는가?
- 왜 `#include`는 마법이 아니라 단순 치환처럼 동작하는가?
- 왜 전역 변수는 더 조심해서 다뤄야 하는가?

즉, 헤더와 선언/정의 분리는 스타일 문제가 아니라 **프로그램 구조와 빌드 모델**의 문제다.

예를 들어 다음처럼 생각해보자.

```c
// math_utils.h
int add(int a, int b);
```

```c
// math_utils.c
int add(int a, int b) {
    return a + b;
}
```

```c
// main.c
#include "math_utils.h"
```

```c
int main(void) {
    return add(1, 2);
}
```

이 구조가 가능한 이유는 단순히 관습이 있어서가 아니다.  
컴파일 단계와 링크 단계에서 각각 필요한 정보가 다르기 때문이다.

이 문서는 헤더 파일이 왜 필요한지, 선언과 정의는 왜 분리하는지, `.h`와 `.c`의 역할이 무엇인지, 그리고 이 구조가 실무에서 왜 필수적인지를 정리하기 위해 작성한다.

## 2. 정의

### 2.1 declaration은 존재와 형태를 알려주는 것이다

declaration은 어떤 이름이 존재하며, 그 이름이 어떤 타입과 형태를 가지는지 알려주는 것이다.

예를 들어:

```c
int add(int a, int b);
```

이 문장은 `add`라는 함수가 존재하고,  
두 개의 `int`를 받아 `int`를 반환한다는 사실을 알려준다.

중요한 점은, declaration은 **그 대상을 실제로 만드는 것**과는 다를 수 있다는 것이다.  
즉, “이런 것이 있다”고 알려주는 것이 declaration이다.

### 2.2 definition은 실제 구현 또는 저장 공간을 제공하는 것이다

definition은 그 이름에 대해 실제 실체를 제공하는 것이다.

예를 들어 함수라면:

```c
int add(int a, int b) {
    return a + b;
}
```

이 문장은 `add`의 실제 구현을 제공한다.  
즉, declaration이 “있다”고 알려주는 것이라면, definition은 “이게 그것이다”를 제공하는 것이다.

변수도 마찬가지다.  
정의는 실제 저장 공간까지 포함하는 경우가 있다.

### 2.3 header file은 인터페이스를 공유하기 위한 파일이다

헤더 파일(`.h`)은 보통 여러 소스 파일이 공통으로 알아야 하는 선언을 담는 파일이다.

즉, 헤더 파일의 핵심 역할은 **공유 인터페이스 제공**이다.

예를 들어 함수 선언, 구조체 선언, 상수 매크로, 타입 별칭 등이 들어갈 수 있다.

### 2.4 source file은 구현을 담는 파일이다

소스 파일(`.c`)은 실제 구현을 담는 파일이다.

즉, `.c` 파일은 다음과 같은 것을 포함한다.

- 함수 정의
- 내부 전용 helper 함수
- 실제 알고리즘 구현
- 내부 전역 상태

정리하면 다음과 같다.

- `.h` → 다른 파일이 알아야 하는 인터페이스
- `.c` → 실제 구현

### 2.5 선언과 정의를 분리하는 이유는 역할이 다르기 때문이다

모든 코드를 한 파일에 몰아넣으면 선언/정의 구분이 덜 중요해 보일 수 있다.  
하지만 여러 파일이 하나의 프로그램을 이루는 순간, “무엇을 알려야 하는가”와 “무엇을 실제로 제공하는가”가 달라진다.

즉, 선언과 정의 분리는 단순 스타일이 아니라 **정보 공개 범위와 빌드 구조 분리**다.

## 3. 핵심 개념

### 3.1 컴파일러는 현재 번역 단위 안에서만 먼저 본다

C 소스 파일은 보통 개별적으로 컴파일된다.  
즉, 한 `.c` 파일을 컴파일할 때 컴파일러는 우선 그 파일 안에서 필요한 정보를 확인한다.

이때 컴파일러가 필요한 것은 다음이다.

- 함수 이름이 있는가
- 그 함수의 인자 타입과 반환 타입은 무엇인가
- 변수와 타입 이름이 어떤 의미인가

즉, 어떤 함수의 실제 구현이 당장 다른 파일에 있어도,  
현재 파일에서 그 함수의 형태를 알고 있으면 일단 컴파일은 가능하다.  
이 역할을 declaration이 한다.

### 3.2 링커는 나중에 실제 정의를 연결한다

컴파일이 끝나면 각 파일은 개별 결과물을 가진다.  
그 다음 단계에서는 여러 파일 사이에 흩어진 함수와 변수 정의를 맞춰서 연결해야 한다.

즉, 컴파일러는 “형태를 아는지”를 먼저 보고,  
링커는 “실제 구현이 존재하는지”를 나중에 본다.

이 때문에 declaration과 definition이 분리될 수 있다.

### 3.3 헤더 파일은 공통 선언을 반복 없이 공유하게 해준다

만약 헤더 파일이 없다면, 여러 `.c` 파일에서 같은 선언을 계속 직접 적어야 한다.  
그렇게 되면 선언이 조금만 달라져도 불일치 버그가 생길 수 있다.

헤더 파일은 이런 문제를 줄인다.

즉, 하나의 선언을 헤더에 두고, 필요한 파일들이 `#include`해서 같은 인터페이스를 공유하게 만드는 것이다.

### 3.4 `#include`는 마법이 아니라 텍스트 포함이다

초보자는 `#include`를 어떤 특별한 연결 기능처럼 느낄 수 있다.  
하지만 개념적으로는 현재 파일 안에 다른 파일 내용을 가져오는 것에 가깝다.

예를 들어:

```c
#include "math_utils.h"
```

이 문장은 개념적으로 `math_utils.h` 내용을 이 자리에 펼쳐 넣는 효과를 낸다.

즉, 헤더는 “링크용 특수 파일”이라기보다 **컴파일 전에 공통 선언을 삽입하는 파일**로 보는 것이 맞다.

### 3.5 정의를 헤더에 무심코 넣으면 중복 정의 문제가 생길 수 있다

헤더는 여러 `.c` 파일에서 포함될 수 있다.  
따라서 실제 정의를 헤더에 무심코 넣으면, 그 정의가 여러 번 나타난 것처럼 보일 수 있다.

예를 들어 함수 정의나 전역 변수 정의를 잘못 헤더에 넣으면  
여러 번역 단위에서 같은 정의가 생겨 중복 정의 문제가 생길 수 있다.

즉, 헤더는 공유 선언에 적합하고,  
실제 구현과 저장 공간 제공은 보통 `.c`에 두는 것이 안전하다.

### 3.6 인터페이스와 구현을 분리하면 구조가 안정된다

선언/정의 분리의 큰 장점은 다음과 같다.

- 외부에 공개할 것과 내부 구현을 나눌 수 있다.
- 내부 구현을 바꿔도 인터페이스가 유지되면 다른 파일 영향이 줄어든다.
- 팀 개발에서 경계가 명확해진다.
- 테스트와 모듈화가 쉬워진다.

즉, 선언과 정의 분리는 단순 문법 규칙이 아니라 **소프트웨어 구조화 원리**다.

## 4. 예시

### 4.1 함수 선언과 정의 분리 예시

다음 구조를 보자.

```c
// math_utils.h
int add(int a, int b);
```

```c
// math_utils.c
int add(int a, int b) {
    return a + b;
}
```

```c
// main.c
#include "math_utils.h"
```

```c
int main(void) {
    int result = add(1, 2);
    return result;
}
```

여기서 핵심은 다음이다.

- `main.c`는 `add`의 구현을 직접 보지 않아도 된다.
- `main.c`는 `add`의 declaration만 알면 된다.
- 실제 구현은 나중에 링크 단계에서 연결된다.

즉, declaration은 사용을 가능하게 하고, definition은 실제 실행을 가능하게 한다.

### 4.2 선언만 있고 정의가 없으면 어떤 문제가 생기는가

다음처럼 선언만 있다고 하자.

```c
int add(int a, int b);
```

그리고 `main`에서 사용한다.

```c
int main(void) {
    return add(1, 2);
}
```

컴파일 시점에는 형태를 알고 있으므로 일단 진행될 수 있다.  
하지만 실제 `add` 정의가 어디에도 없다면 최종적으로 연결할 수 없다.

즉, declaration만으로는 충분하지 않고 결국 definition이 필요하다.

### 4.3 정의를 헤더에 잘못 넣는 예시

다음처럼 헤더에 실제 정의를 넣는다고 생각해보자.

```c
// bad.h
int add(int a, int b) {
    return a + b;
}
```

이 헤더를 여러 `.c` 파일이 포함하면, 각 파일마다 같은 정의가 들어간 것처럼 될 수 있다.

즉, “공유를 위해 헤더에 넣는다”와 “실제 구현까지 헤더에 둔다”는 다른 문제다.  
헤더는 반복 삽입될 수 있기 때문에 정의를 조심해서 다뤄야 한다.

### 4.4 헤더가 없으면 선언이 흩어지는 예시

헤더 없이 여러 파일이 같은 선언을 각자 적는다고 해보자.

```c
// a.c
int add(int a, int b);
```

```c
// b.c
int add(int x, int y);
```

겉으로는 비슷해 보여도, 선언이 커지거나 구조체, 타입 별칭이 엮이면 금방 유지보수가 어려워진다.  
헤더는 이런 반복과 불일치를 줄여준다.

### 4.5 전역 변수와 선언/정의 차이 예시

전역 변수는 함수보다 더 조심해야 한다.  
어떤 파일에서는 “이 전역 변수가 어딘가에 있다”고 알리기만 해야 하고,  
실제 저장 공간은 한 곳에서만 제공해야 한다.

즉, 전역 변수는 declaration과 definition의 차이를 더 민감하게 체감하게 해준다.

## 5. 헷갈리기 쉬운 부분

### 5.1 선언과 정의를 같은 것으로 보는 문제

입문 단계에서는 선언과 정의가 한 줄 안에 같이 보이는 경우가 많다.  
그래서 둘이 같은 것처럼 느껴질 수 있다.

하지만 둘은 다르다.

- declaration → 존재와 형태를 알림
- definition → 실제 구현 또는 저장 공간 제공

이 차이를 놓치면 다중 파일 구조에서 바로 혼란이 생긴다.

### 5.2 헤더는 무조건 다 넣는 파일이라고 생각하는 문제

헤더는 “아무거나 넣는 공용 창고”가 아니다.  
헤더는 다른 파일이 알아야 하는 인터페이스만 넣는 것이 원칙에 가깝다.

내부 전용 구현, 외부에 공개할 필요 없는 helper 함수, 모듈 내부 상태를 무분별하게 헤더에 넣으면 구조가 금방 흐려진다.

### 5.3 `#include`를 링크라고 착각하는 문제

`#include`는 컴파일 전에 텍스트를 포함하는 것에 가깝다.  
실제 여러 파일의 정의를 연결하는 것은 링크 단계다.

즉, 포함(include)과 연결(link)은 다른 단계다.

### 5.4 헤더에 정의를 넣어도 “작동해 보일 때”가 있어 더 헷갈리는 문제

작은 예제에서는 정의를 헤더에 넣어도 겉으로는 되는 것처럼 보일 수 있다.  
하지만 파일 수가 늘어나면 중복 정의 문제나 구조적 혼란이 생기기 쉽다.

즉, 작은 예제에서의 우연한 동작을 일반 원칙으로 착각하면 안 된다.

### 5.5 인터페이스와 구현 경계를 의식하지 않는 문제

초보자는 보통 “돌아가면 됐다”에 집중한다.  
하지만 프로그램이 커지면 더 중요한 것은 “어디까지를 외부에 보여주고 어디까지를 내부에 숨길 것인가”다.

이 경계를 만드는 핵심 도구가 바로 header / source 분리다.

## 6. 어디서부터 인터페이스 계약과 구현 책임이 갈리는가

### 6.1 선언이 있다는 것과 구현이 있다는 것은 다른 약속이다

초보자는 함수 원형을 봤을 때 이미 “그 기능이 존재한다”고 느끼기 쉽다.  
하지만 선언은 현재 번역 단위가 그 이름과 형태를 알 수 있게 해 줄 뿐이고,  
실제 동작 보장은 정의가 제공한다.

즉, 선언은 사용 가능성의 약속이고  
정의는 실제 제공 책임이다.  
이 둘을 섞으면 컴파일 성공과 링크 성공을 혼동하게 된다.

### 6.2 헤더는 텍스트 포함 수단이지만 역할은 계약 문서에 가깝다

`#include`는 기술적으로 텍스트 포함에 가깝다.  
하지만 구조적으로는 “다른 파일이 알아야 하는 인터페이스를 공유한다”는 의미를 가진다.

즉, 헤더를 단순 복붙 도구처럼 보면  
왜 내부 구현을 헤더에 무분별하게 넣으면 안 되는지가 잘 안 보인다.

### 6.3 `.c` 파일은 코드 저장소가 아니라 정의 책임이 있는 번역 단위다

소스 파일을 그냥 구현을 넣는 장소로만 보면 부족하다.  
각 `.c` 파일은 독립적으로 컴파일되어 오브젝트 파일이 되고,  
링커는 그 안의 정의와 참조를 기준으로 프로그램 전체를 조립한다.

즉, `.c` 파일에 정의를 둔다는 것은  
구현을 기록하는 것과 동시에 링크 책임을 배치하는 일이기도 하다.

### 6.4 헤더에 정의를 넣으면 “공유”가 아니라 “복제”가 일어날 수 있다

초보자가 가장 많이 놓치는 지점이 여기다.  
헤더를 여러 파일이 include하면 그 내용은 각 translation unit에 들어간다.

따라서 정의를 무분별하게 헤더에 넣으면  
하나의 구현을 공유하는 것이 아니라 여러 번 복제하는 결과가 될 수 있다.

즉, 헤더 정의 문제는 스타일 취향이 아니라  
번역 단위 경계와 링크 규칙의 문제다.

### 6.5 인터페이스를 줄인다는 것은 공개 심볼을 줄인다는 뜻이기도 하다

어떤 선언을 헤더에 넣는 순간, 그 이름은 외부 사용자에게 보이는 계약 후보가 된다.  
반대로 내부 구현을 `.c` 내부에 남기면 외부 의존성을 줄일 수 있다.

즉, 헤더 설계는 문서 정리가 아니라  
모듈 경계와 공개 심볼 범위를 설계하는 일이다.

## 7. 실무 관점

### 7.1 모듈화의 출발점이다

실무에서 코드를 잘 나눈다는 것은 단순히 파일 개수를 늘리는 것이 아니다.  
각 모듈이 어떤 인터페이스를 제공하고, 어떤 구현을 내부에 숨길지를 정하는 것이다.

헤더 파일은 이 인터페이스 경계를 표현하는 핵심 수단이다.

### 7.2 팀 개발에서 계약 역할을 한다

헤더는 일종의 계약 문서 역할을 한다.  
다른 팀원은 헤더만 보고 이 모듈을 어떻게 사용할지 알 수 있어야 한다.

즉, 헤더가 깔끔하면 모듈 사용이 쉬워지고,  
헤더가 혼란스러우면 팀 전체 생산성이 떨어진다.

### 7.3 구현 변경의 영향 범위를 줄여준다

인터페이스가 안정적으로 유지된다면, `.c` 내부 구현을 바꿔도 외부 사용 코드는 크게 손댈 필요가 없을 수 있다.

즉, 선언/정의 분리는 변경 격리에도 도움이 된다.

### 7.4 빌드 구조와도 직접 연결된다

헤더와 소스 분리를 이해하면 다음이 자연스럽게 이어진다.

- 개별 파일 컴파일
- 오브젝트 파일 생성
- 링크 단계
- 심볼 해석
- 라이브러리 구성

즉, 이 문서는 단순 파일 분리 팁이 아니라 빌드 시스템 이해의 입구다.

### 7.5 전역 상태와 API 설계를 더 신중하게 하게 만든다

헤더를 설계하다 보면 자연스럽게 이런 질문을 하게 된다.

- 이 함수를 외부에 공개해야 하는가?
- 이 타입을 외부에 노출해야 하는가?
- 전역 변수를 정말 공개해야 하는가?
- 함수 선언을 더 명확하게 만들 수 있는가?

즉, 헤더 설계는 API 설계 능력과 직접 연결된다.

## 판단 체크리스트

- 선언과 정의를 같은 “존재”로 뭉뚱그리지 않고 계약과 구현 책임으로 구분하고 있는가
- `#include`를 링크가 아니라 translation unit 형성 단계의 텍스트 포함으로 이해하고 있는가
- `.c` 파일에 정의를 둔다는 것이 링크 책임 배치와도 연결된다는 점을 이해하고 있는가
- 헤더에 정의를 넣는 일이 공유가 아니라 복제로 이어질 수 있음을 설명할 수 있는가
- 헤더 설계를 공개 인터페이스와 심볼 범위 설계 문제로 보고 있는가

## 정리

헤더 파일과 선언/정의 분리는 C 프로그램 구조의 핵심이다.

핵심은 다음과 같다.

- declaration은 존재와 형태를 알린다.
- definition은 실제 구현 또는 저장 공간을 제공한다.
- 헤더 파일은 여러 파일이 공유해야 하는 인터페이스를 담는다.
- 소스 파일은 실제 구현을 담는다.
- `#include`는 헤더 내용을 현재 파일에 포함시키는 역할을 한다.
- 컴파일은 선언 정보를 보고 진행될 수 있지만, 실행 가능한 프로그램을 만들려면 실제 정의가 필요하다.
- 정의를 헤더에 무분별하게 넣으면 중복 정의 문제가 생길 수 있다.
- 선언/정의 분리는 곧 인터페이스/구현 분리이기도 하다.

즉, 헤더 파일과 선언/정의 분리를 이해한다는 것은  
“`.h`와 `.c`를 나눠 쓴다” 수준을 넘어서  
**여러 파일이 하나의 프로그램으로 결합되는 방식과, 인터페이스와 구현을 분리해 구조화하는 원리**를 이해한다는 뜻이다.

이 관점이 잡혀야 C 프로젝트를 단일 파일 예제 수준을 넘어 실제 프로그램 구조로 확장할 수 있다.
