## 정의

정적 라이브러리와 동적 라이브러리는 둘 다 재사용 가능한 코드 묶음이지만, 프로그램에 결합되는 시점과 방식이 다르다.  
정적 라이브러리는 링크 시점에 실행 파일 안으로 복사되어 들어가고, 동적 라이브러리는 실행 파일과 분리된 채로 남아 있다가 링크 단계와 실행 단계의 협력을 통해 사용된다.

핵심은 다음이다.

1. 정적 라이브러리는 보통 최종 실행 파일에 코드가 포함된다
2. 동적 라이브러리는 보통 별도 파일로 존재하며 실행 시 로드된다
3. 두 방식은 배포 방식, 메모리 사용, 업데이트, 의존성 관리에서 차이를 만든다
4. 어느 쪽이 무조건 더 좋다기보다, 프로젝트의 목적과 운영 방식에 따라 선택 기준이 달라진다

즉 라이브러리 방식의 차이는 단순 파일 확장자 차이가 아니라,  
**코드 재사용을 어떤 시점과 책임 구조로 결합할 것인가**의 차이다.

---

## 먼저 큰 그림부터

라이브러리는 "여러 프로그램이 공통으로 사용할 수 있는 함수/데이터 묶음"이라고 생각하면 된다.

예를 들어 아래 같은 함수들이 있다고 하자.
```c
    int add(int a, int b);
    int sub(int a, int b);
    int mul(int a, int b);
```
이 코드를 매 프로젝트마다 복붙하는 대신, 라이브러리로 만들어 재사용할 수 있다.

그런데 이 라이브러리를 프로그램에 붙이는 방식이 두 가지다.

- 정적 라이브러리
- 동적 라이브러리

차이는 "최종 실행 파일 안에 들어가느냐"와 "밖에 남느냐"로 잡으면 된다.

---

## 핵심 개념

### 1. 정적 라이브러리는 링크 시점에 코드가 합쳐진다

정적 라이브러리는 보통 오브젝트 파일들을 묶은 아카이브다.  
링커는 최종 실행 파일을 만들 때 이 안에서 필요한 코드 조각을 꺼내 실행 파일 안에 포함시킨다.

즉 개념적으로는 다음과 같다.

- `main.o`
- `util.o`
- `libmath.a`

링커가 이들을 합쳐:

- `program`

을 만든다.

이후 실행할 때는 라이브러리 파일이 별도로 없어도 된다.  
필요한 코드가 이미 실행 파일 안에 들어 있기 때문이다.

즉 정적 라이브러리는 **빌드 시점 결합**이라고 보면 된다.

---

### 2. 동적 라이브러리는 실행 파일과 분리된 채 유지된다

동적 라이브러리는 보통 공유 가능한 별도 바이너리 파일이다.  
실행 파일은 이 라이브러리의 코드를 직접 전부 품고 있지 않고, 나중에 해당 라이브러리를 사용하겠다는 정보와 연결 지점을 가진다.

즉 개념적으로는 다음과 같다.

- `program`
- `libmath.so` 또는 플랫폼별 공유 라이브러리

실행 시에는 로더/동적 링커가 이 라이브러리를 메모리에 적재하고, 필요한 심볼을 연결해 프로그램이 동작하게 만든다.

즉 동적 라이브러리는 **실행 시점까지 이어지는 결합 구조**라고 보면 된다.

---

## 정적 라이브러리는 실제로 무엇인가

정적 라이브러리는 보통 여러 오브젝트 파일을 묶은 파일이다.

즉 내부적으로는 대략 아래처럼 생각하면 된다.

- `add.o`
- `sub.o`
- `mul.o`

이것들을 하나로 모아:

- `libmath.a`

같은 정적 라이브러리를 만든다.

중요한 점은 이것이 "이미 완성된 실행 코드 한 덩어리"가 아니라,  
링커가 필요한 조각을 꺼내 쓸 수 있는 오브젝트 파일 묶음이라는 점이다.

즉 정적 라이브러리는 **오브젝트 파일 보관소**에 가깝다.

---

## 동적 라이브러리는 실제로 무엇인가

동적 라이브러리는 실행 파일과 별도로 존재하면서, 여러 프로그램이 공유해서 사용할 수 있는 코드/데이터 묶음이다.

즉 개념적으로 다음 성격을 가진다.

- 독립된 바이너리 파일
- 외부에 공개할 심볼을 가짐
- 프로그램이 이 파일에 의존함
- 실행 시 동적 로딩 또는 동적 심볼 해결이 일어남

즉 동적 라이브러리는 단순 보관소가 아니라,  
**실행 시 연결 대상으로 남아 있는 별도 모듈**이다.

---

## 정적 링크와 동적 링크의 차이

### 1. 정적 링크

정적 링크에서는 링커가 라이브러리 코드를 실행 파일 안에 포함한다.

결과:

- 실행 파일 하나만 있어도 동작하기 쉬움
- 파일 크기가 커질 수 있음
- 같은 라이브러리를 여러 프로그램이 각각 복사해 가질 수 있음

---

### 2. 동적 링크

동적 링크에서는 실행 파일이 라이브러리 자체를 포함하지 않고,  
해당 라이브러리를 실행 시 사용한다는 관계를 유지한다.

결과:

- 실행 파일 자체는 더 작을 수 있음
- 여러 프로그램이 하나의 라이브러리를 공유할 수 있음
- 실행 환경에 해당 라이브러리가 있어야 함
- 버전 불일치 문제 가능

즉 차이는 단순 저장 위치가 아니라 운영 책임 구조의 차이다.

---

## 왜 정적 라이브러리를 쓰는가

### 1. 배포가 단순해질 수 있다

정적 링크된 프로그램은 필요한 코드가 실행 파일 안에 포함되어 있으므로,  
별도 라이브러리 설치 문제를 줄일 수 있다.

즉 하나의 바이너리로 배포하기 쉬운 경우가 있다.

---

### 2. 실행 환경 의존성을 줄일 수 있다

대상 시스템에 특정 버전의 공유 라이브러리가 없어서 실행이 실패하는 문제를 피하기 쉽다.

즉 운영 환경이 제각각일 때 안정성을 얻기 좋다.

---

### 3. 임베디드/제한 환경에서 유리할 수 있다

런타임 로더, 공유 라이브러리 탐색, 시스템 의존성을 최소화해야 하는 환경에서는 정적 링크가 더 단순할 수 있다.

즉 환경 통제가 중요할수록 정적 방식이 매력적일 수 있다.

---

## 왜 동적 라이브러리를 쓰는가

### 1. 여러 프로그램이 같은 라이브러리를 공유할 수 있다

예를 들어 많은 프로그램이 같은 C 런타임이나 그래픽 라이브러리를 사용한다면,  
각 실행 파일마다 같은 코드를 중복 포함하는 대신 공유 라이브러리를 함께 쓰는 편이 효율적일 수 있다.

즉 코드 공유가 가능하다.

---

### 2. 라이브러리 업데이트가 유연할 수 있다

공유 라이브러리를 교체함으로써 관련 프로그램들이 새 구현을 사용할 수 있다.  
물론 ABI 호환성 문제가 없다는 전제가 중요하다.

즉 업데이트 전략이 더 유연해질 수 있다.

---

### 3. 플러그인 구조와도 잘 맞는다

동적 라이브러리는 나중에 로드하는 구조와도 잘 맞기 때문에,  
플러그인, 모듈 확장, 런타임 기능 추가에 자주 쓰인다.

즉 동적 라이브러리는 단순 재사용뿐 아니라 **확장 구조**와도 잘 맞는다.

---

## 실행 파일 크기와 메모리 사용 측면

### 정적 라이브러리

- 실행 파일 크기가 커질 수 있다
- 동일 라이브러리 코드가 여러 프로그램에 중복 포함될 수 있다

### 동적 라이브러리

- 개별 실행 파일 크기는 줄어들 수 있다
- 여러 프로세스가 공용 라이브러리 코드를 공유할 수 있다

다만 이것을 너무 단순화하면 안 된다.  
실제 크기와 메모리 이점은 운영체제, 로더, 최적화 방식, 링크 설정에 따라 달라질 수 있다.

그래도 큰 경향은 다음처럼 이해하면 된다.

- 정적: 각 실행물에 포함
- 동적: 공용 모듈 공유 가능

---

## 배포와 운영 측면에서의 차이

### 정적 라이브러리 방식

장점:
- 실행 환경 단순화
- 의존성 누락 문제 감소
- 특정 버전 라이브러리를 함께 묶어 안정성 확보 가능

단점:
- 패치 시 프로그램 재배포가 필요할 수 있음
- 여러 프로그램이 같은 코드를 각각 품을 수 있음
- 보안 업데이트 전파가 느릴 수 있음

### 동적 라이브러리 방식

장점:
- 라이브러리 단독 업데이트 가능
- 여러 프로그램이 공유 가능
- 플러그인/모듈형 구조와 궁합이 좋음

단점:
- 실행 시 라이브러리 의존성 문제 발생 가능
- 버전 충돌, 로딩 실패, ABI 불일치 가능
- 배포가 더 복잡할 수 있음

즉 정적은 단순성과 독립성,  
동적은 공유성과 유연성이 강점이다.

---

## ABI와 왜 연결되는가

동적 라이브러리는 특히 ABI(Application Binary Interface)와 깊게 연결된다.

왜냐하면 실행 파일은 라이브러리의 함수 시그니처 수준뿐 아니라,  
호출 규약, 심볼 이름, 구조체 레이아웃, 바이너리 호환성까지 기대하고 연결되기 때문이다.

즉 소스 수준 API가 같아 보여도 ABI가 달라지면 동적 링크된 프로그램이 깨질 수 있다.

정적 라이브러리도 ABI와 완전히 무관한 것은 아니지만,  
동적 라이브러리는 운영 중 별도 교체/공유가 일어나기 때문에 ABI 안정성이 특히 중요하다.

---

## 헤더와 라이브러리는 왜 같이 다녀야 하는가

라이브러리를 쓸 때는 보통 두 가지가 필요하다.

- 선언이 들어 있는 헤더
- 실제 코드가 들어 있는 라이브러리

예를 들어 헤더는 컴파일 단계에서 필요하다.
```c
    int add(int, int);
```
컴파일러는 이 선언을 보고 코드가 타입상 맞는지 확인한다.

반면 실제 `add`의 구현은 링크 단계 또는 실행 단계에서 라이브러리에서 온다.

즉 헤더와 라이브러리는 역할이 다르다.

- 헤더: 컴파일러에게 인터페이스 제공
- 라이브러리: 링커/로더에게 실제 구현 제공

이 차이를 이해하면 "헤더는 있는데 왜 링크가 실패하지?" 같은 질문이 풀린다.

---

## 정적 라이브러리와 동적 라이브러리는 어느 단계에서 관여하는가

### 정적 라이브러리

주로 링크 단계에서 관여한다.  
필요한 코드가 실행 파일에 포함되면, 런타임에는 원본 정적 라이브러리 파일이 따로 필요하지 않을 수 있다.

### 동적 라이브러리

링크 단계에서도 심볼 참조 정보가 필요하고,  
실행 단계에서도 실제 라이브러리 로드와 심볼 해결이 중요하다.

즉 동적 라이브러리는 빌드뿐 아니라 런타임까지 책임이 이어진다.

---

## 자주 하는 오해

### 오해 1. 정적 라이브러리는 그냥 큰 실행 파일이다

아니다.  
정적 라이브러리는 보통 오브젝트 파일 묶음이며, 최종 실행 파일은 아니다.

---

### 오해 2. 동적 라이브러리는 링크를 안 해도 된다

아니다.  
동적 라이브러리도 심볼 연결과 의존성 설정이 필요하다.  
다만 최종 코드 결합 방식이 정적과 다를 뿐이다.

---

### 오해 3. 동적 라이브러리가 항상 더 좋다

아니다.  
배포 단순성, 환경 제어, 독립 실행성 측면에서는 정적 방식이 더 적합한 경우가 많다.

---

### 오해 4. 정적 라이브러리는 무조건 실행이 빠르다

항상 그렇지 않다.  
성능은 단순 링크 방식보다 코드 배치, 최적화, 호출 패턴, 로더 비용 등 여러 요소에 좌우된다.

---

### 오해 5. 헤더만 있으면 라이브러리 없이도 된다

아니다.  
헤더는 선언만 제공한다.  
실제 구현이 링크되거나 로드되지 않으면 프로그램은 완성되지 않는다.

---

## 실무 관점

### 1. 배포 전략 선택 문제다

CLI 도구, 임베디드 바이너리, 단일 배포 파일을 원하면 정적 링크가 유리할 수 있다.  
반면 데스크톱 환경, 서버 공용 라이브러리, 플러그인 구조라면 동적 라이브러리가 더 적합할 수 있다.

즉 라이브러리 선택은 기술 취향보다 운영 전략 문제다.

---

### 2. 디버깅과 장애 대응 방식이 달라진다

정적 링크는 실행 파일 하나 기준으로 추적하기가 쉬운 편이다.  
동적 링크는 라이브러리 버전, 로딩 경로, 심볼 충돌, 런타임 의존성 문제를 함께 봐야 할 수 있다.

즉 동적 방식은 유연하지만 운영 관찰 포인트가 늘어난다.

---

### 3. 보안 업데이트 정책과도 연결된다

공유 라이브러리 취약점이 발견되었을 때:

- 동적 방식은 라이브러리 갱신으로 여러 프로그램에 영향을 줄 수 있음
- 정적 방식은 각 프로그램을 다시 빌드/배포해야 할 수 있음

즉 보안 운영 정책도 선택 기준이 된다.

---

### 4. 다음 문서들과 직접 연결된다

이 문서는 아래 주제와 자연스럽게 이어진다.

- 헤더 파일과 선언/정의 분리는 왜 필요한가
- 오브젝트 파일과 심볼이란 무엇인가
- extern과 static은 무엇이 다르고 왜 중요한가
- 표준 라이브러리와 시스템 콜의 차이

즉 라이브러리 문서는 빌드/링킹 축과 시스템 인터페이스 축을 이어 주는 다리 역할을 한다.

---

## 정리

정적 라이브러리와 동적 라이브러리는 둘 다 재사용 가능한 코드 묶음이지만, 프로그램과 결합되는 방식이 다르다.

핵심은 다음과 같다.

- 정적 라이브러리는 링크 시 실행 파일 안으로 포함된다
- 동적 라이브러리는 별도 파일로 남아 실행 시 로드된다
- 정적 방식은 배포 단순성과 독립성이 강점이다
- 동적 방식은 코드 공유와 업데이트 유연성이 강점이다
- 선택 기준은 빌드 방식이 아니라 운영 전략, 배포 환경, ABI 안정성까지 포함한다

즉 라이브러리 방식을 이해한다는 것은  
"코드를 재사용한다"를 넘어서 **그 재사용 코드를 언제, 어디서, 누구 책임으로 결합할 것인가**를 이해하는 것이다.
