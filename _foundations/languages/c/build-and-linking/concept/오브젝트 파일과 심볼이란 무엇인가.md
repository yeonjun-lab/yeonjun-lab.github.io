## 정의

오브젝트 파일(object file)은 컴파일과 어셈블 단계를 거쳐 만들어진, 아직 완전히 연결되지 않은 기계어 조각이다.  
심볼(symbol)은 그 조각 안에서 함수, 전역 변수, 외부 참조 같은 이름과 위치 정보를 표현하는 단위다.

즉 오브젝트 파일은 단순한 "중간 파일"이 아니라,  
링커가 프로그램 전체를 조립하기 위해 읽는 **코드/데이터/심볼/재배치 정보의 묶음**이다.

핵심은 다음이다.

1. `.c` 파일 하나는 보통 하나의 오브젝트 파일로 이어진다
2. 오브젝트 파일 안에는 기계어만 있는 것이 아니다
3. 함수 이름, 전역 변수 이름, 외부 참조 정보가 심볼로 기록된다
4. 링커는 심볼을 보고 여러 오브젝트 파일을 연결한다

즉 오브젝트 파일과 심볼을 이해한다는 것은  
C 프로그램이 "파일 단위로 나뉜 소스"에서 "하나의 실행물"로 합쳐지는 구조를 이해하는 것이다.

---

## 핵심 개념

### 1. 오브젝트 파일은 아직 완성되지 않은 프로그램 조각이다

예를 들어 다음 파일이 있다고 하자.
```
`add.c`

    int add(int a, int b) {
        return a + b;
    }
```
이 파일을 컴파일하고 어셈블하면 `add.o` 같은 오브젝트 파일이 만들어진다.

이 파일 안에는 대략 다음 정보가 들어 있다.

- `add` 함수의 기계어 코드
- 함수 이름 `add`에 대한 심볼 정보
- 필요한 섹션 정보
- 나중에 주소를 맞추기 위한 재배치 정보가 있을 수 있음

즉 오브젝트 파일은 단순 텍스트가 아니라,  
링커가 읽을 수 있는 구조화된 바이너리다.

---

### 2. 심볼은 "이 이름이 무엇을 가리키는가"를 나타낸다

예시:
```
    int global_value = 10;

    int add(int a, int b) {
        return a + b + global_value;
    }
```
이 코드에는 적어도 다음 같은 이름이 등장한다.

- `global_value`
- `add`

이 이름들은 컴파일 후 오브젝트 파일 안에서 심볼 형태로 기록될 수 있다.

심볼은 대략 아래 정보를 담는다고 이해하면 된다.

- 이름
- 종류(함수, 변수 등)
- 정의인지 참조인지
- 어느 섹션에 속하는지
- 오프셋/주소 정보
- 외부 가시성 여부

즉 심볼은 링커가 "이 이름을 어디에 연결해야 하는가"를 판단하는 기준이다.

---

### 3. 정의된 심볼과 참조된 심볼이 구분된다

예시:
```
`main.c`

    int add(int, int);

    int main(void) {
        return add(1, 2);
    }
```
```
`add.c`

    int add(int a, int b) {
        return a + b;
    }
```
여기서 `main.c`는 `add`를 사용하지만 정의하지 않는다.  
반면 `add.c`는 `add`를 정의한다.

따라서:

- `main.o`에는 `add`에 대한 **참조 심볼**이 생길 수 있다
- `add.o`에는 `add`에 대한 **정의 심볼**이 생길 수 있다

링커는 이 둘을 연결한다.

즉 프로그램 전체 연결은 결국  
"어디서 정의됐고 어디서 참조됐는가"를 심볼 수준에서 맞추는 일이다.

---

## 오브젝트 파일 안에는 무엇이 들어 있는가

구체적 포맷은 플랫폼마다 다를 수 있지만, 개념적으로는 아래가 중요하다.

### 1. 코드 섹션

함수 본문 같은 실행 코드가 들어간다.

예:
- `.text`

---

### 2. 데이터 섹션

초기화된 전역 변수, 정적 변수 등의 데이터가 들어갈 수 있다.

예:
- `.data`

---

### 3. BSS 성격의 영역

초기값이 0이거나 명시적 초기화가 없는 정적 저장 기간 객체 정보가 표현될 수 있다.

예:
- `.bss`

---

### 4. 심볼 테이블

이름과 관련된 정보가 들어 있다.

예:
- 어떤 함수가 정의되었는가
- 어떤 전역 변수가 외부로 공개되는가
- 어떤 이름이 아직 외부에서 해결되어야 하는가

---

### 5. 재배치 정보

현재 오브젝트 파일만으로는 최종 주소를 모르는 부분에 대해  
링커가 나중에 수정해야 할 위치 정보가 들어 있다.

즉 오브젝트 파일은 "거의 완성된 코드"이지만,  
여전히 전체 프로그램 맥락에서 조정이 필요하다.

---

## 왜 재배치 정보가 필요한가

예시:
```
`main.c`

    int add(int, int);

    int main(void) {
        return add(1, 2);
    }
```
`main.o`가 만들어질 때, `add`의 최종 메모리 주소는 아직 모를 수 있다.  
왜냐하면 `add`가 다른 파일에 있고, 전체 링크가 끝나야 최종 배치가 정해질 수 있기 때문이다.

따라서 오브젝트 파일은 "여기에서 `add`를 호출해야 하는데, 최종 주소는 나중에 채워 넣어야 함" 같은 정보를 기록해 둔다.

이것이 재배치(relocation)의 핵심 아이디어다.

즉 링커는 단순히 조각을 나열하는 것이 아니라,  
**미완성 주소 참조를 실제 주소로 패치하는 작업**도 한다.

---

## 로컬 심볼과 외부 심볼

모든 이름이 동일한 방식으로 보이지는 않는다.

### 1. 로컬 심볼

현재 오브젝트 파일 내부에서만 의미가 있는 이름이다.

예를 들어 `static` 함수나 `static` 전역 변수는 외부 파일에서 직접 접근되지 않도록 내부 가시성을 가질 수 있다.

예시:
```
    static int helper(int x) {
        return x + 1;
    }
```
이 경우 `helper`는 현재 번역 단위 내부에서만 보이도록 처리될 수 있다.

---

### 2. 외부 심볼

다른 오브젝트 파일과 연결될 수 있는 이름이다.

예시:
```
    int add(int a, int b) {
        return a + b;
    }
```
이 함수는 외부 링크 가시성을 가질 수 있고,  
다른 파일에서 선언 후 참조할 수 있다.

즉 심볼은 단순 이름표가 아니라,  
**링커 관점의 가시성과 연결 가능성 정보**를 함께 가진다.

---

## 왜 지역 변수는 심볼 논의에서 중심이 아닌가

초보자는 "그럼 지역 변수도 심볼인가?"라고 묻기 쉽다.  
개념적으로 이름이 붙어 있으니 그렇게 느껴질 수 있다.

하지만 빌드/링킹 관점에서 가장 중요한 심볼은 보통 아래다.

- 함수
- 전역 변수
- 정적 저장 기간 객체
- 외부 참조 이름

지역 변수는 함수 내부 스택 프레임이나 최적화 결과에 따라  
링커가 다룰 이름 수준 개체로 오래 유지되지 않을 수 있다.

즉 오브젝트 파일/심볼 문맥에서 핵심은  
파일 간 연결과 주소 해석에 필요한 이름들이다.

---

## 링크는 심볼 해석 과정이다

링크를 단순히 "오브젝트 파일을 합친다"라고 생각하면 부족하다.  
실제 핵심은 심볼 해석이다.

예시:

- `main.o`는 `add`가 필요하다고 말함
- `add.o`는 `add`를 정의하고 있음
- 링커는 이를 매칭함
- 호출 위치를 실제 주소로 재배치함

이 과정에서 다음 문제가 생길 수 있다.

### 1. undefined reference

참조는 있는데 정의가 없음

### 2. multiple definition

같은 외부 심볼 정의가 여러 개 있음

### 3. 잘못된 가시성

`static` 때문에 외부에서 못 보거나, 선언과 정의가 어긋남

즉 링크 오류의 본질은 대부분 심볼 문제다.

---

## `static`과 `extern`이 왜 여기서 중요해지는가

`static`과 `extern`은 심볼의 가시성과 연결 범위를 바꾸는 핵심 키워드다.

예를 들어:

- `static` 함수: 현재 번역 단위 내부 심볼로 제한
- `extern` 선언: 다른 곳에 정의된 심볼을 참조한다는 의미

즉 다음 문서인 `extern과 static은 무엇이 다르고 왜 중요한가`는  
사실상 심볼 가시성 문서라고 봐도 된다.

오브젝트 파일과 심볼을 이해한 다음에 `static`/`extern`으로 가야 자연스럽다.

---

## 심볼이 없으면 링커는 무엇을 못 하는가

링커는 이름 없이 마법처럼 코드를 맞추지 않는다.  
어떤 코드 조각이 무엇을 필요로 하는지, 무엇을 제공하는지를 알아야 한다.

즉 심볼이 없다면 링커는 아래를 할 수 없다.

- 어떤 함수 호출이 어디를 가리키는지 결정
- 어떤 전역 변수가 실제로 어디에 있는지 연결
- 같은 이름 충돌 감지
- 외부 라이브러리 함수 연결

즉 심볼은 분리 컴파일을 가능하게 하는 핵심 장치다.

---

## 라이브러리와도 어떻게 연결되는가

정적 라이브러리나 동적 라이브러리도 결국 심볼 관점에서 연결된다.

예를 들어 `printf`를 쓰면,  
링커 또는 동적 로더는 해당 심볼을 적절한 라이브러리 정의와 연결해야 한다.

즉 라이브러리는 단순한 "기능 묶음"이 아니라,  
심볼을 제공하는 코드 집합으로도 볼 수 있다.

이 때문에 라이브러리 연결 문제도 결국 심볼 해석 문제로 돌아온다.

---

## 자주 하는 오해

### 오해 1. 오브젝트 파일은 그냥 실행 파일 직전 단계일 뿐이다

반만 맞다.  
오브젝트 파일은 단순 중간 부산물이 아니라,  
코드/데이터/심볼/재배치 정보가 담긴 중요한 연결 단위다.

---

### 오해 2. 심볼은 변수 이름 목록 같은 것이다

부족한 이해다.  
심볼은 단순 이름이 아니라,  
정의/참조/가시성/섹션/주소 정보를 포함하는 연결 정보다.

---

### 오해 3. 링크는 파일을 그냥 순서대로 붙이는 것이다

아니다.  
링커는 심볼을 해석하고, 재배치하고, 충돌을 검출하고, 주소를 정한다.

---

### 오해 4. 선언만 있으면 링크도 성공한다

아니다.  
선언은 참조를 가능하게 할 뿐이다.  
실제 정의 심볼이 어딘가 존재해야 링크가 성공한다.

---

### 오해 5. 같은 이름 함수가 여러 파일에 있어도 상관없다

외부 가시성 심볼이라면 문제될 수 있다.  
같은 심볼 정의가 여러 개 있으면 multiple definition 오류가 날 수 있다.

---

## 실무 관점

### 1. 링크 에러를 읽는 핵심 기준이 된다

실무에서 자주 보는 오류는 대부분 심볼 문제다.

- undefined reference to `foo`
- multiple definition of `bar`

이 메시지를 읽으려면 "어느 파일이 참조했고, 어느 파일이 정의해야 하는가"를 심볼 관점으로 봐야 한다.

---

### 2. 모듈 분리와 내부 구현 은닉에 직접 연결된다

어떤 함수는 외부 공개 API로 두고,  
어떤 함수는 `static`으로 파일 내부에 숨길 수 있다.

즉 심볼 가시성 제어는 모듈 경계를 만드는 수단이다.

---

### 3. 라이브러리 설계와 연결된다

라이브러리를 만든다는 것은 결국  
어떤 심볼을 외부에 제공할지 결정하는 일이기도 하다.

즉 공개 함수, 내부 헬퍼, 전역 변수 노출 여부는 전부 심볼 설계 문제다.

---

### 4. 디버깅과 도구 사용에도 중요하다

`nm`, `objdump`, `readelf` 같은 도구를 쓰면  
오브젝트 파일과 실행 파일 안의 심볼을 직접 볼 수 있다.

즉 이 주제를 이해하면 "빌드가 안 된다"를 막연히 보는 것이 아니라,  
실제로 파일 내부 구조를 추적할 수 있게 된다.

---

### 5. 다음 문서들과 직접 연결된다

이 문서는 아래 주제로 자연스럽게 이어진다.

- extern과 static은 무엇이 다르고 왜 중요한가
- 헤더 파일과 선언/정의 분리는 왜 필요한가
- 정적 라이브러리와 동적 라이브러리는 무엇이 다른가
- 컴파일, 어셈블, 링크의 전체 흐름

즉 오브젝트 파일과 심볼은 빌드/링킹 축의 중심 개념이다.

---

## 정리

오브젝트 파일은 컴파일과 어셈블을 거쳐 만들어진,  
아직 최종 연결 전 상태의 코드 조각이다.  
심볼은 그 안에서 함수, 전역 변수, 외부 참조 같은 이름과 연결 정보를 표현하는 단위다.

핵심은 다음과 같다.

- 오브젝트 파일에는 코드만 있는 것이 아니다
- 심볼은 정의와 참조를 연결하는 기준이다
- 링커는 심볼을 해석해 프로그램 전체를 조립한다
- 재배치 정보는 최종 주소 확정을 가능하게 한다
- 링크 오류의 대부분은 심볼 수준 문제다

즉 C 프로그램이 여러 파일로 나뉘어도 하나의 실행물로 합쳐질 수 있는 이유는  
오브젝트 파일과 심볼이라는 구조가 있기 때문이다.

이 관점이 잡히면 다음의 `extern`, `static`, 선언/정의 분리, 라이브러리 연결도 훨씬 선명하게 이해된다.
