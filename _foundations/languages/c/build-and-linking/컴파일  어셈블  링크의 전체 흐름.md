## 정의

C 프로그램은 소스 파일을 바로 실행하는 언어가 아니다.  
작성한 `.c` 파일은 전처리, 컴파일, 어셈블, 링크 단계를 거쳐 실행 파일 또는 라이브러리 형태가 되어야 한다.

즉 C의 빌드 과정은 하나의 마법 같은 "컴파일"이 아니라, 실제로는 여러 단계로 나뉜 파이프라인이다.

핵심은 다음이다.

1. 전처리기는 컴파일러가 읽을 소스를 준비한다
2. 컴파일러는 C 코드를 어셈블리 수준 표현으로 변환한다
3. 어셈블러는 이를 기계어가 담긴 오브젝트 파일로 바꾼다
4. 링커는 여러 오브젝트 파일과 라이브러리를 연결해 최종 실행물을 만든다

즉 C를 제대로 이해하려면 "컴파일한다"는 말을 하나의 덩어리로 보지 말고,  
**각 단계가 무엇을 입력으로 받고 무엇을 출력하는지** 구분해서 봐야 한다.

---

## 전체 흐름 한눈에 보기

가장 단순화하면 흐름은 다음과 같다.

1. `.c` 소스 작성
2. 전처리 결과 생성
3. 컴파일 결과인 어셈블리 생성
4. 어셈블 결과인 오브젝트 파일 생성
5. 여러 오브젝트 파일과 라이브러리를 링크
6. 실행 파일 또는 공유 라이브러리 생성

즉 개념적으로는 아래처럼 이해하면 된다.
```
    source.c
      -> preprocessed source
      -> assembly
      -> object file
      -> executable
```
실제 도구는 이 과정을 한 번에 감싸기도 하지만, 내부적으로는 이런 단계가 존재한다.

---

## 핵심 개념

### 1. 전처리: 컴파일러가 읽을 최종 소스를 만든다

전처리기는 `#include`, `#define`, `#if` 같은 지시문을 처리한다.

예시:
```
    #include <stdio.h>
    #define SIZE 10

    int main(void) {
        int arr[SIZE];
        return 0;
    }
```
전처리 단계에서는 대략 다음이 일어난다.

- `stdio.h` 내용 포함
- `SIZE`를 `10`으로 치환
- 조건부 컴파일이 있으면 해당 코드 선택

즉 이 단계의 출력은  
"사람이 작성한 원본 파일"이 아니라 **컴파일러가 실제로 읽을 확장된 소스**다.

이 단계는 타입 검사나 기계어 생성 단계가 아니다.  
소스 준비 단계다.

---

### 2. 컴파일: C 문법과 의미를 분석해 저수준 코드로 바꾼다

컴파일 단계에서는 전처리된 소스를 바탕으로 아래가 일어난다.

- 토큰화
- 구문 분석
- 타입 검사
- 의미 분석
- 최적화
- 어셈블리 코드 생성

즉 이 단계에서 비로소 컴파일러가 C 언어 의미를 이해한다.

예를 들어 다음 오류는 이 단계에서 잡힌다.
```
    int x = "hello";
```
이런 코드는 타입 불일치이므로 컴파일 단계에서 문제가 된다.

컴파일 결과는 보통 어셈블리 코드라고 생각해도 된다.  
즉 아직 최종 기계어 실행 파일은 아니다.

---

### 3. 어셈블: 어셈블리 코드를 오브젝트 파일로 바꾼다

어셈블 단계에서는 사람이 읽기 쉬운 어셈블리 표현을  
CPU가 이해할 수 있는 기계어 형태로 바꾸고, 이를 오브젝트 파일에 담는다.

오브젝트 파일에는 보통 다음 같은 정보가 들어 있다.

- 기계어 코드
- 데이터 영역 정보
- 심볼 정보
- 재배치 정보

즉 오브젝트 파일은 "거의 실행 가능해 보이지만 아직 완성되지 않은 조각"이다.

왜 아직 완성되지 않았는가?  
다른 파일에 있는 함수나 전역 변수 주소가 아직 최종 확정되지 않았을 수 있기 때문이다.

---

### 4. 링크: 여러 조각을 하나의 실행물로 연결한다

링크 단계에서는 여러 오브젝트 파일과 라이브러리를 모아  
심볼을 해석하고 주소를 확정한 뒤 최종 실행 파일을 만든다.

예를 들어:
```
`main.c`

    extern int add(int, int);

    int main(void) {
        return add(1, 2);
    }
```
```
`add.c`

    int add(int a, int b) {
        return a + b;
    }
```
이 둘은 각각 따로 컴파일되어 오브젝트 파일이 될 수 있다.  
하지만 `main.o`만으로는 `add`의 실제 코드 위치를 모른다.

링커가 해야 할 일은 다음이다.

- `main.o`가 필요로 하는 `add` 심볼을 찾는다
- `add.o`에서 그 정의를 찾는다
- 호출 주소를 연결한다
- 최종 실행 파일을 만든다

즉 링크는 단순 파일 합치기가 아니라,  
**심볼 해석과 주소 결정을 포함한 결합 단계**다.

---

## 왜 `.c` 파일 하나만으로는 끝나지 않는가

실무 C 프로젝트는 거의 항상 여러 파일로 나뉜다.

- `main.c`
- `util.c`
- `parser.c`
- `net.c`

각 파일은 독립적으로 컴파일될 수 있다.  
이렇게 하면 변경된 파일만 다시 빌드할 수 있고, 모듈화도 가능하다.

하지만 최종 실행 시에는 이 조각들이 하나로 이어져야 한다.  
그래서 링크 단계가 필요하다.

즉 C 빌드 구조는 처음부터 **분리 컴파일 + 최종 결합**을 전제로 한다.

---

## translation unit은 어디에 해당하는가

전처리가 끝난 뒤의 각 `.c` 파일은 대체로 하나의 translation unit이 된다.

예를 들어 `main.c`가 여러 헤더를 포함하고 있으면,  
전처리 후에는 그것이 하나의 큰 소스 단위처럼 컴파일된다.

즉 translation unit은 대략 다음이다.

- 하나의 `.c` 파일
- 그 파일이 포함한 헤더 내용
- 전처리 결과까지 반영된 컴파일 단위

이 translation unit이 각각 컴파일되어 오브젝트 파일이 된다.

즉 흐름은 이렇게 이해하면 된다.

- source file
- preprocessed translation unit
- object file
- linked executable

---

## 각 단계의 입력과 출력을 구분해서 보기

### 1. 전처리

입력:
- `.c` 파일
- 헤더 파일
- 매크로 정의
- 조건부 컴파일 설정

출력:
- 전처리된 소스

### 2. 컴파일

입력:
- 전처리된 소스

출력:
- 어셈블리 코드

### 3. 어셈블

입력:
- 어셈블리 코드

출력:
- 오브젝트 파일 (`.o`, `.obj`)

### 4. 링크

입력:
- 오브젝트 파일들
- 정적/동적 라이브러리

출력:
- 실행 파일
- 또는 라이브러리

이렇게 구분하면 에러가 어디서 나는지도 훨씬 선명해진다.

---

## 에러는 단계마다 성격이 다르다

### 1. 전처리 단계 문제

예시:

- 헤더 파일을 찾을 수 없음
- 매크로 조건 분기 문제
- 중복 포함 구조 문제

즉 아직 타입 검사 이전 단계의 문제다.

---

### 2. 컴파일 단계 문제

예시:

- 문법 오류
- 타입 불일치
- 선언되지 않은 식별자 사용
- 잘못된 표현식

즉 C 언어 규칙 위반이 여기서 많이 드러난다.

---

### 3. 어셈블 단계 문제

일반적인 고수준 코드에서는 상대적으로 덜 의식되지만,  
인라인 어셈블리나 특정 어셈블 지시어를 쓸 때 문제될 수 있다.

---

### 4. 링크 단계 문제

예시:

- undefined reference
- multiple definition
- 라이브러리 누락
- 심볼 충돌

즉 "소스는 각각 잘 컴파일되는데 최종 결합이 안 되는 문제"는 보통 링크 문제다.

이 구분은 실무에서 매우 중요하다.

---

## 왜 링크 에러는 컴파일 에러와 다른가

이 부분이 특히 중요하다.

예를 들어 아래 코드는 선언만 있으면 각 파일 단위에서는 컴파일이 될 수 있다.
```
    int add(int, int);

    int main(void) {
        return add(1, 2);
    }
```
컴파일러는 "어딘가에 `add`가 정의되어 있겠지"라고 보고  
타입만 맞으면 일단 통과시킬 수 있다.

하지만 링크 시점에 실제 정의가 없으면 최종 실행 파일을 만들 수 없다.

즉 컴파일 성공은 "각 번역 단위가 자기 관점에서 문법과 타입이 맞다"는 뜻이고,  
링크 성공은 "프로그램 전체의 조각들이 실제로 연결 가능하다"는 뜻이다.

둘은 전혀 다른 검사다.

---

## 라이브러리는 어디서 들어오는가

라이브러리는 보통 링크 단계에서 함께 들어온다.

예를 들어 `printf` 같은 표준 라이브러리 함수도  
최종적으로는 관련 라이브러리 코드와 연결되어야 한다.

즉 오브젝트 파일끼리만 링크하는 것이 아니라,  
필요한 라이브러리도 함께 결합된다.

이 때문에 아래 같은 문제가 생길 수 있다.

- 라이브러리 경로 문제
- 링크 순서 문제
- 정적/동적 링크 차이
- 실행 시 공유 라이브러리 로딩 문제

즉 C 빌드 이해는 소스 파일만 보는 것으로 끝나지 않는다.

---

## 왜 이 과정을 알아야 하는가

### 1. 헤더와 소스 분리를 이해하게 된다

왜 선언은 헤더에 두고 정의는 `.c`에 두는가,  
왜 같은 정의를 헤더에 넣으면 문제가 되는가를 이해하려면  
컴파일/링킹 구조를 알아야 한다.

---

### 2. 에러 해석이 쉬워진다

예를 들어 다음을 구분할 수 있게 된다.

- 문법 에러인가
- 타입 에러인가
- 심볼 미해결인가
- 중복 정의인가

즉 빌드 파이프라인을 알면 에러를 훨씬 빠르게 읽는다.

---

### 3. 대규모 프로젝트 구조를 이해하게 된다

실무 프로젝트는 거의 항상 수십, 수백 개 파일로 구성된다.  
이를 효율적으로 빌드하려면 분리 컴파일과 링크 개념이 필수다.

즉 이 흐름은 작은 예제용 지식이 아니라,  
실제 프로젝트 운영 지식이다.

---

## 자주 하는 오해

### 오해 1. 컴파일은 한 번에 다 끝나는 단일 과정이다

아니다.  
실제로는 전처리, 컴파일, 어셈블, 링크라는 서로 다른 단계가 있다.

---

### 오해 2. `.c` 파일이 컴파일되면 바로 실행 파일이 된다

항상 아니다.  
대부분은 먼저 오브젝트 파일이 되고, 여러 파일과 함께 링크되어야 한다.

---

### 오해 3. 컴파일 성공이면 프로그램 전체가 완성된 것이다

아니다.  
각 파일이 개별적으로 맞을 뿐일 수 있다.  
최종 정의 연결은 링크 단계에서 다시 검증된다.

---

### 오해 4. 링크는 단순히 파일을 이어 붙이는 것이다

아니다.  
링크는 심볼 해석, 주소 결정, 재배치 처리까지 포함하는 단계다.

---

### 오해 5. 헤더는 그냥 참고용 파일이다

아니다.  
헤더는 전처리 단계에서 실제로 포함되어 translation unit 형성에 참여한다.

---

## 실무 관점

### 1. 빌드 시스템 이해의 출발점이다

Make, CMake, Ninja 같은 도구는 결국 이 파이프라인을 관리한다.

- 어떤 파일을 다시 컴파일할지
- 어떤 오브젝트를 링크할지
- 어떤 라이브러리를 붙일지

즉 빌드 도구를 제대로 이해하려면 내부 단계부터 알아야 한다.

---

### 2. 증분 빌드와 모듈화가 왜 가능한지 이해하게 된다

파일 하나를 수정했다고 전체 소스를 처음부터 전부 다시 이해할 필요는 없다.  
변경된 translation unit만 다시 컴파일하고, 이후 다시 링크하면 된다.

이것이 대규모 프로젝트가 유지되는 핵심 중 하나다.

---

### 3. 링크 에러를 읽을 수 있게 된다

실무에서 매우 흔한 문제는 아래다.

- 선언은 있는데 정의가 없음
- 같은 전역 심볼이 여러 파일에 중복 정의됨
- `static`/`extern` 사용이 잘못됨
- 라이브러리를 링크하지 않음

이 문제들은 문법보다 빌드 구조를 알아야 해결된다.

---

### 4. 다음 문서들과 직접 연결된다

이 문서는 아래 주제의 입구다.

- 오브젝트 파일과 심볼이란 무엇인가
- extern과 static은 무엇이 다르고 왜 중요한가
- 정적 라이브러리와 동적 라이브러리는 무엇이 다른가
- 헤더 파일과 선언/정의 분리는 왜 필요한가

즉 이 문서는 빌드/링킹 축의 전체 지도를 제공한다.

---

## 정리

C 프로그램은 소스를 바로 실행하지 않는다.  
전처리, 컴파일, 어셈블, 링크를 거쳐 최종 실행물로 만들어진다.

핵심은 다음과 같다.

- 전처리는 소스를 준비한다
- 컴파일은 C 의미를 분석하고 저수준 코드로 바꾼다
- 어셈블은 오브젝트 파일을 만든다
- 링크는 여러 조각과 라이브러리를 연결한다

즉 "컴파일"이라는 한 단어 뒤에는 실제로 여러 단계가 숨어 있다.

이 구조를 이해하면 C 코드가 어떻게 실행 파일이 되는지,  
왜 헤더와 소스가 분리되는지,  
왜 어떤 에러는 컴파일 때가 아니라 링크 때 나는지가 선명해진다.
