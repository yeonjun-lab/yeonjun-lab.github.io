## 정의

버퍼 오버플로우(buffer overflow)는 어떤 버퍼가 수용할 수 있는 경계를 넘어 데이터를 읽거나 쓰는 상황을 뜻한다.  
즉 프로그램이 "이만큼만 들어갈 수 있는 메모리 구역"에 그보다 더 많은 데이터를 밀어 넣거나, 범위 밖까지 접근하는 것이다.

핵심은 다음이다.

1. 버퍼는 크기가 정해진 메모리 구역이다
2. 오버플로우는 그 크기 경계를 넘는 접근이다
3. C는 많은 경우 자동 경계 검사를 해 주지 않는다
4. 따라서 작은 실수도 메모리 손상, 크래시, 보안 취약점으로 이어질 수 있다

즉 버퍼 오버플로우를 이해한다는 것은  
단순히 "길이를 잘못 썼다"를 넘어서 **C에서 메모리 경계가 얼마나 직접적으로 프로그래머 책임인가**를 이해하는 것이다.

---

## 먼저 큰 그림부터

예를 들어 길이가 8인 문자 배열이 있다고 하자.
```c
    char buf[8];
```
이 배열은 인덱스로 보면 다음 범위만 유효하다.

- `buf[0]`
- `buf[1]`
- `buf[2]`
- `buf[3]`
- `buf[4]`
- `buf[5]`
- `buf[6]`
- `buf[7]`

그런데 여기에 8칸을 넘는 쓰기를 하면 경계를 넘는다.

예시:
```c
    buf[8] = 'X';
```
이 한 줄은 겉보기엔 사소하지만, 실제로는 `buf` 바깥 메모리를 건드리는 것이다.

즉 버퍼 오버플로우의 본질은  
**"의도한 객체 밖의 메모리를 침범한다"**는 데 있다.

---

## 핵심 개념

### 1. 버퍼는 연속된 저장 공간이다

버퍼는 보통 배열, 동적 할당 블록, 구조체 안의 배열 필드처럼 나타난다.

예시:
```c
    char name[32];
    int numbers[100];
    char *data = malloc(256);
```
이들은 모두 "정해진 크기의 연속 메모리 구역"이라는 점에서 버퍼로 볼 수 있다.

즉 버퍼 오버플로우는 문자열 함수에서만 생기는 문제가 아니라,  
**고정 크기 또는 길이 제한이 있는 모든 메모리 구역**에서 생길 수 있다.

---

### 2. 오버플로우는 쓰기만이 아니라 읽기에서도 생긴다

많은 사람이 오버플로우를 "너무 많이 써서 넘치는 것"으로만 이해한다.  
하지만 범위 밖 읽기도 중요한 문제다.

예시:
```c
    int arr[3] = {1, 2, 3};
    int x = arr[5];
```
이 코드는 쓰지는 않지만, 여전히 유효 범위 밖을 읽고 있다.

즉 경계 위반은 두 가지 모두를 포함한다.

- overflow write: 너무 많이 씀
- overflow read / out-of-bounds read: 너무 멀리 읽음

쓰기 오버플로우는 메모리 손상을 만들고,  
읽기 오버플로우는 잘못된 값 노출이나 정보 유출, 이후 잘못된 제어 흐름으로 이어질 수 있다.

---

### 3. C는 자동 경계 검사를 거의 해 주지 않는다

예시:
```c
    int arr[3] = {1, 2, 3};
    arr[10] = 99;
```
이 코드는 문법적으로는 컴파일될 수 있다.  
즉 C는 많은 배열 접근에 대해 런타임 경계 검사를 자동 삽입하지 않는다.

왜냐하면 C는 성능과 저수준 제어를 중시하는 언어이기 때문이다.  
그 대가로 경계 관리 책임이 프로그래머에게 넘어온다.

즉 C에서 버퍼 오버플로우는 예외적인 런타임 오류가 아니라,  
**언어 설계상 프로그래머가 직접 막아야 하는 위험**이다.

---

## 버퍼 오버플로우는 실제로 어떻게 생기는가

### 1. 배열 인덱스를 잘못 계산할 때

예시:
```c
    int arr[5];
    for (int i = 0; i <= 5; i++) {
        arr[i] = 0;
    }
```
이 코드는 매우 흔한 실수다.  
유효 인덱스는 `0`부터 `4`까지인데, `i <= 5`로 돌면서 `arr[5]`까지 접근한다.

즉 버퍼 오버플로우는 복잡한 해킹 코드가 아니라,  
단순한 경계 조건 실수 하나로도 쉽게 생긴다.

---

### 2. 문자열 길이를 잘못 다룰 때

예시:
```c
    char buf[8];
    strcpy(buf, "abcdefghijk");
```
이 경우 문자열 길이가 버퍼보다 길 수 있다.  
그러면 널 문자까지 포함한 복사가 버퍼 범위를 넘어간다.

문자열은 특히 위험하다.  
왜냐하면 C 문자열은 길이 필드가 아니라 널 종료 규칙에 의존하므로,  
복사 전에 길이를 명시적으로 확인하지 않으면 쉽게 넘친다.

---

### 3. 널 종료 공간을 빼먹을 때

예시:
```c
    char buf[5];
    strcpy(buf, "hello");
```
겉보기엔 글자 수가 5개라 맞아 보일 수 있다.  
하지만 `"hello"`는 실제로 다음 6칸이 필요하다.

- `'h'`
- `'e'`
- `'l'`
- `'l'`
- `'o'`
- `'\0'`

즉 문자열 버퍼 계산에서는  
**보이는 문자 수 + 널 종료 공간**을 같이 봐야 한다.

---

### 4. 길이 인자를 신뢰할 수 없을 때

예시:
```c
    void copy_data(char *dst, const char *src, size_t len) {
        memcpy(dst, src, len);
    }
```
이 함수는 겉보기엔 평범하지만, `dst`가 실제로 `len`만큼 쓸 수 있는지 보장하지 않는다.  
즉 호출자가 잘못된 길이를 넘기면 쉽게 오버플로우가 발생한다.

실무에서는 이런 문제가 매우 흔하다.

- 길이 계산 실수
- 단위 혼동(원소 수 vs 바이트 수)
- 외부 입력 신뢰
- 프로토콜 길이 필드 검증 누락

즉 버퍼 오버플로우는 함수 하나보다 **계약 검증 실패**의 문제이기도 하다.

---

### 5. `sizeof`를 잘못 쓸 때

예시:
```c
    void clear(int *arr) {
        memset(arr, 0, sizeof(arr));
    }
```
이 코드는 배열 전체를 0으로 만드는 것처럼 보일 수 있다.  
하지만 `arr`는 포인터이므로 `sizeof(arr)`는 포인터 크기만 준다.

반대로 다른 상황에서는 필요한 크기보다 더 크게 계산하는 실수도 생길 수 있다.  
즉 `sizeof` 오해는 버퍼 관련 버그의 중요한 원인이다.

---

### 6. 동적 메모리 크기를 잘못 계산할 때

예시:
```c
    int *p = malloc(10);
    for (int i = 0; i < 10; i++) {
        p[i] = i;
    }
```
이 코드는 `int` 10개를 할당한 것처럼 보일 수 있다.  
하지만 실제로는 10바이트만 할당했을 수 있다.

올바른 의도는 보통 다음과 같아야 한다.
```c
    int *p = malloc(sizeof(*p) * 10);
```
즉 오버플로우는 스택 배열만의 문제가 아니라,  
힙에서도 아주 흔하게 생긴다.

---

## 메모리에서 실제로 무슨 일이 일어나는가

예를 들어:
```c
    char buf[8];
```
바로 뒤에 다른 지역변수, 저장된 상태, 패딩, 스택 프레임 정보가 있을 수 있다.  
그런데 `buf` 바깥까지 쓰면 그 인접 메모리를 덮어쓸 수 있다.

즉 오버플로우는 단순히 "내 데이터가 조금 잘못된다"가 아니라,  
다른 객체의 값을 망가뜨릴 수 있다.

영향 예시:

- 옆 변수 값 손상
- 포인터 값 파손
- 문자열 종료 깨짐
- 제어 정보 손상
- 나중의 free/return 시점 문제 유발

즉 경계 밖 쓰기는 **객체 경계를 침범하는 메모리 손상**이다.

---

## 왜 문자열 함수에서 특히 자주 터지는가

C 문자열은 다음 특성을 가진다.

- 길이 정보가 자동 저장되지 않음
- `'\0'`를 만날 때까지 계속 읽음
- 목적 버퍼 크기를 함수가 항상 아는 것이 아님

예시:
```c
    char dst[16];
    char *src = user_input;
    strcpy(dst, src);
```
여기서 `src` 길이가 16 이상이면 위험하다.  
즉 문자열 함수는 "입력 길이"와 "목적지 용량"을 동시에 관리해야 하므로 특히 사고가 많이 난다.

이 때문에 실무에서는 문자열 관련 코드를 볼 때 항상 다음을 확인해야 한다.

- 목적 버퍼 크기
- 실제 입력 길이
- 널 종료 보장 여부
- 길이 인자의 신뢰성

---

## 스택 오버플로우와 힙 오버플로우

버퍼 오버플로우는 위치에 따라 다르게 나타날 수 있다.

### 1. 스택 기반 버퍼 오버플로우

예시:
```c
    void f(const char *s) {
        char buf[16];
        strcpy(buf, s);
    }
```
지역 배열을 넘쳐 쓰면 같은 스택 프레임 안의 다른 데이터가 손상될 수 있다.

---

### 2. 힙 기반 버퍼 오버플로우

예시:
```c
    char *buf = malloc(16);
    memcpy(buf, src, 64);
```
동적 할당 블록 범위를 넘으면 힙 메타데이터나 인접 블록이 손상될 수 있다.

즉 둘 다 위험하지만, 손상되는 대상과 증상은 다르게 보일 수 있다.

---

## 왜 버퍼 오버플로우가 UB인가

버퍼 경계를 넘는 접근은 C가 그 의미를 보장하지 않는 경우가 많다.  
즉 문제는 단순히 "값이 틀린다"가 아니라 **언어가 그 이후를 책임지지 않는다**는 점이다.

그래서 다음이 모두 가능하다.

- 바로 크래시
- 조용히 잘못된 값 저장
- 전혀 다른 위치에서 나중에 터짐
- 현재 환경에서는 멀쩡해 보임
- 최적화에 따라 증상 변화

즉 버퍼 오버플로우는 UB의 대표 사례다.

---

## 왜 보안 취약점으로 연결되는가

버퍼 오버플로우는 단순 실수로 끝나지 않을 수 있다.  
메모리 레이아웃과 제어 흐름에 영향을 줄 수 있기 때문이다.

예를 들어 오버플로우가 다음을 건드릴 수 있다.

- 함수 포인터
- 반환 관련 제어 정보
- 길이 필드
- 객체 상태 플래그
- 권한 관련 변수

즉 메모리 손상은 프로그램 논리 자체를 뒤틀 수 있다.  
그래서 버퍼 오버플로우는 오래전부터 대표적인 보안 취약점 범주다.

## 자주 하는 오해

### 오해 1. 버퍼 오버플로우는 문자열 복사에서만 생긴다

아니다.  
배열 인덱스 실수, 길이 계산 오류, 힙 크기 오계산, 구조체 필드 접근 실수 등에서도 생긴다.

---

### 오해 2. 많이 넘쳐야만 위험하다

아니다.  
한 바이트만 넘쳐도 널 종료를 깨뜨리거나 인접 상태를 손상시킬 수 있다.

---

### 오해 3. 크래시가 안 나면 괜찮다

아니다.  
오버플로우는 조용히 데이터만 망가뜨리고 훨씬 뒤에서 증상이 나타날 수 있다.

---

### 오해 4. `strncpy`류를 쓰면 자동으로 안전하다

항상 아니다.  
길이 인자 의미, 널 종료 보장 여부, 실제 버퍼 용량과의 관계를 여전히 정확히 이해해야 한다.

---

### 오해 5. 힙은 동적이니까 좀 넘쳐도 괜찮다

아니다.  
힙도 경계가 있는 할당 블록이며, 범위를 넘으면 다른 블록이나 allocator 상태를 손상시킬 수 있다.

---

## 실무 관점

### 1. 경계 검사는 선택이 아니라 기본 계약이다

배열, 버퍼, 동적 블록을 다룰 때는 항상 다음을 같이 봐야 한다.

- 용량(capacity)
- 실제 사용 길이(length)
- 복사/읽기/쓰기 요청량
- 널 종료 필요 여부

즉 "데이터"만 보면 안 되고 **길이 계약**을 같이 봐야 한다.

---

### 2. 단위 혼동이 매우 흔하다

실무에서 자주 나오는 실수:

- 원소 개수를 바이트 수로 착각
- 바이트 수를 원소 개수로 착각
- 문자열 길이에 널 종료를 빼먹음
- `sizeof(pointer)`를 `sizeof(buffer)`로 오해

즉 버퍼 오버플로우는 복잡한 기법보다 **단위 해석 실패**에서 자주 시작된다.

---

### 3. 인터페이스 설계가 중요하다

버퍼를 받는 함수는 보통 아래 중 하나를 같이 받아야 한다.

- 목적 버퍼 크기
- 현재 길이
- 최대 허용 길이

예시:
```c
    int copy_name(char *dst, size_t dst_size, const char *src);
```
이런 형태가 더 안전한 이유는, 함수가 최소한 경계를 판단할 정보가 있기 때문이다.

---

### 4. 도구와 방어 기법을 같이 써야 한다

실무에서는 다음이 중요하다.

- 경고 옵션
- sanitizer
- fuzzing
- 코드 리뷰에서 길이 계약 확인
- API에서 크기 인자 강제

즉 버퍼 오버플로우는 "조심하자"만으로 막기 어렵고,  
도구와 설계 규칙이 함께 필요하다.

---

### 5. 다음 문서와 직접 연결된다

이 문서는 아래와 직접 이어진다.

- use-after-free는 왜 치명적인가
- 문자열은 왜 특별 취급되는가
- sizeof는 무엇을 알려주고 무엇을 숨기는가
- 직렬화와 역직렬화는 왜 단순 memcpy가 아닌가
- length-prefixed format은 왜 필요한가

즉 버퍼 오버플로우는 메모리 안전성 축과 직렬화 축을 이어 주는 핵심 주제다.

## 정리

버퍼 오버플로우는 버퍼가 허용하는 메모리 경계를 넘어 읽거나 쓰는 상황이다.

핵심은 다음과 같다.

- 버퍼는 크기가 정해진 연속 메모리 구역이다
- 경계를 넘는 접근은 다른 객체 메모리를 침범할 수 있다
- C는 자동 경계 검사를 거의 하지 않는다
- 문자열, 배열, 힙 할당, 길이 계산 오류에서 자주 발생한다
- 작은 오버플로우도 메모리 손상과 보안 취약점으로 이어질 수 있다

즉 버퍼 오버플로우를 막는 핵심은  
"복사 함수를 바꾸는 것"만이 아니라 **모든 메모리 구역에 대해 용량과 길이를 분리해서 사고하는 습관**이다.

이 관점이 잡히면 C의 문자열 처리, 힙 사용, 파서 구현, 직렬화 코드가 훨씬 더 안전한 방향으로 정리되기 시작한다.
