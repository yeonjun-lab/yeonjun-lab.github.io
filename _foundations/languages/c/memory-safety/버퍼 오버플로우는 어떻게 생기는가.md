## 1. 왜 이 주제를 깊게 봐야 하는가

C를 배우면서 배열, 포인터, 문자열, 스택과 힙, 파일 입출력까지 오면  
이제 자연스럽게 반드시 만나야 하는 주제가 있다.  
바로 **버퍼 오버플로우(buffer overflow)** 다.

입문 단계에서는 버퍼 오버플로우를 보통 이렇게 설명한다.

- 배열 크기를 넘어서 쓰는 것
- 메모리를 초과해서 접근하는 것
- 문자열 복사를 잘못해서 넘치는 것

이 설명은 맞다.  
하지만 여기서 멈추면 왜 이 문제가 단순한 실수가 아니라  
C에서 가장 중요한 안전성과 보안 문제 중 하나인지까지는 보이지 않는다.

예를 들어 다음 코드를 보자.

```c
char buf[8];
strcpy(buf, "this is too long");
```

겉으로 보면 그냥 문자열 복사 한 줄이다.  
하지만 실제로는 매우 위험할 수 있다.  
왜냐하면 `buf`가 가진 유효한 저장 공간보다 더 많은 데이터를 써 버릴 수 있기 때문이다.

이 문제를 얕게 보면 단순히 “값이 조금 깨질 수 있다”고 생각하기 쉽다.  
하지만 실제로는 훨씬 심각할 수 있다.

- 인접한 변수 값이 덮어써질 수 있다
- 스택 프레임 구조가 깨질 수 있다
- 프로그램 흐름이 비정상적으로 바뀔 수 있다
- 힙 메타데이터가 손상될 수 있다
- 보안 취약점으로 이어질 수 있다
- 겉으로는 한동안 멀쩡히 동작하다가 나중에 터질 수 있다

즉, 버퍼 오버플로우는 “조금 범위를 넘긴다” 정도의 문제가 아니라  
**메모리 객체 경계를 깨고 다른 데이터나 제어 구조를 침범하는 문제**다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- 배열과 포인터를 실제 안전성 관점으로 연결해 준다
- 문자열 함수 사용법이 왜 중요한지 드러난다
- undefined behavior가 실제로 어떤 식으로 현실 문제로 나타나는지 보여 준다
- 스택/힙 구조와 보안 문제를 연결해 준다
- “작동하는 것처럼 보이는 위험한 코드”를 구분하게 해 준다

이 문서는 버퍼 오버플로우가 정확히 무엇인지, 왜 생기는지,  
스택과 힙에서 각각 어떤 형태로 나타날 수 있는지,  
문자열 처리와 배열 처리에서 왜 자주 발생하는지,  
그리고 실무에서 어떤 기준으로 예방해야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

버퍼 오버플로우를 큰 구조로 보면 다음과 같다.

### 2.1 버퍼는 유한한 크기를 가진 메모리 객체다

버퍼(buffer)는 데이터를 임시로 담기 위한 메모리 공간이다.  
C에서는 보통 배열이나 동적 할당 메모리로 버퍼를 만든다.

예를 들어:

```c
char buf[8];
```

이건 8개의 `char`를 담을 수 있는 버퍼다.

즉, 버퍼는 “아무거나 담는 무한 공간”이 아니라  
**정해진 크기를 가진 메모리 객체**다.

### 2.2 오버플로우는 그 경계를 넘어서는 쓰기다

버퍼 오버플로우는 그 객체가 보장하는 유효 범위를 넘어 데이터를 쓰는 상황이다.

예를 들어 8바이트 버퍼에 20바이트를 쓰려고 하면  
나머지 12바이트는 원래 그 버퍼에 속하지 않는 다른 메모리 영역을 침범하게 된다.

즉, 핵심은 “더 썼다”가 아니라  
**자기 소유가 아닌 메모리까지 덮어쓴다**는 것이다.

### 2.3 읽기 초과와 쓰기 초과는 다르지만 둘 다 위험하다

실무에서는 읽기 초과(out-of-bounds read)와 쓰기 초과(out-of-bounds write)를 함께 보게 된다.  
하지만 버퍼 오버플로우라고 할 때 특히 위험한 것은 보통 쓰기 초과다.

왜냐하면 쓰기 초과는 다른 데이터 자체를 바꿔 버릴 수 있기 때문이다.

즉, 정보 유출과 데이터 손상 중 특히 후자가 더 직접적 위험이다.

### 2.4 스택 버퍼 오버플로우와 힙 버퍼 오버플로우는 양상이 다를 수 있다

버퍼가 어디에 있느냐에 따라 문제가 퍼지는 방식이 다를 수 있다.

- 스택 버퍼 오버플로우
- 힙 버퍼 오버플로우
- 전역/정적 버퍼 오버플로우

즉, “범위 초과”라는 본질은 같지만  
어떤 영역을 침범하는지에 따라 영향 양상이 달라질 수 있다.

## 3. 내부 동작 원리

### 3.1 배열은 자기 범위 안에서만 유효하다

다음 코드를 보자.

```c
int arr[3] = {1, 2, 3};
```

이 배열이 유효하게 제공하는 원소는 다음뿐이다.

- `arr[0]`
- `arr[1]`
- `arr[2]`

즉, 이 배열 객체의 유효 범위는 3개의 `int` 원소다.

이제 다음을 보자.

```c
arr[3] = 10;
```

겉으로는 “다음 칸”처럼 보일 수 있다.  
하지만 `arr[3]`은 이 배열 객체가 제공하는 범위를 벗어난다.

즉, 이 쓰기는 배열 객체 바깥 메모리를 덮어쓰는 시도가 된다.

### 3.2 C는 자동 경계 검사를 하지 않는다

많은 고수준 언어는 배열 접근 시 범위를 검사하거나 예외를 던진다.  
하지만 C는 일반적으로 자동 경계 검사를 제공하지 않는다.

즉, 다음 코드가 있다고 해서:

```c
arr[100] = 1;
```

컴파일러나 런타임이 항상 안전하게 막아 주는 것이 아니다.

이건 C가 빠르고 저수준 제어를 제공하는 대신,  
경계 검사의 책임을 프로그래머에게 넘기기 때문이다.

### 3.3 문자열 버퍼 오버플로우는 종료 문자까지 포함해 생각해야 한다

다음 코드를 보자.

```c
char buf[4];
```

여기서 초보자는 `"abc"`가 길이 3이니까 충분하다고 생각할 수 있다.  
하지만 C 문자열은 보통 마지막에 `'\0'` 종료 문자가 필요하다.

즉:

```
"abc"
```

는 실제로 다음 4개를 필요로 한다.

- `'a'`
- `'b'`
- `'c'`
- `'\0'`

따라서 길이 계산을 잘못하면 아주 쉽게 오버플로우가 생긴다.

### 3.4 복사 함수는 버퍼 크기를 자동으로 알아주지 않는다

다음 코드를 보자.

```c
char dst[8];
char *src = "this is too long";
strcpy(dst, src);
```

여기서 `strcpy`는 `dst`의 실제 크기를 자동으로 알지 못한다.  
그저 `src` 문자열을 종료 문자까지 계속 복사하려고 할 뿐이다.

즉, “목적지 버퍼가 충분한가”는 프로그래머가 보장해야 한다.

이게 문자열 관련 버퍼 오버플로우가 자주 생기는 핵심 이유다.

### 3.5 스택 버퍼 오버플로우는 인접한 스택 데이터까지 덮을 수 있다

다음과 같은 구조를 생각해보자.

```c
void f(void) {
    char buf[8];
    int x = 10;
}
```

겉보기에는 `buf`와 `x`가 서로 다른 변수다.  
하지만 둘 다 같은 함수 스택 프레임 안에 존재할 수 있다.

만약 `buf`에 범위를 넘는 쓰기를 하면,  
그 주변에 놓인 다른 지역 변수나 더 민감한 프레임 정보까지 침범할 수 있다.

즉, 스택 버퍼 오버플로우는 단순 데이터 손상을 넘어서  
함수 실행 구조 자체를 깨뜨릴 수 있다.

### 3.6 힙 버퍼 오버플로우는 다른 힙 객체나 메타데이터를 침범할 수 있다

다음 코드를 보자.

```c
char *buf = malloc(8);
```

이 버퍼에 8바이트를 넘는 데이터를 쓰면,  
바로 인접한 다른 힙 객체나 힙 관리 정보까지 침범할 수 있다.

즉, 힙 오버플로우는 단순히 한 버퍼만의 문제가 아니라  
힙 전체 상태를 오염시킬 수 있다.

### 3.7 버퍼 오버플로우는 undefined behavior로 이어진다

버퍼 경계를 넘어 쓰는 순간,  
그 코드는 더 이상 정상적인 객체 경계 안에서 동작하지 않는다.

즉, 버퍼 오버플로우는 단순 실수가 아니라  
언어 차원에서 정의되지 않은 동작으로 이어지는 대표 사례다.

## 4. 핵심 구성 요소

### 4.1 buffer

버퍼는 데이터를 담기 위한 유한한 메모리 객체다.  
배열일 수도 있고, 동적 할당 메모리일 수도 있다.

즉, 버퍼는 항상 크기와 경계를 가진다.

### 4.2 boundary

경계(boundary)는 그 버퍼가 유효하게 제공하는 메모리 범위다.  
이 경계 안에서만 읽고 쓸 수 있다.

즉, 버퍼 문제의 핵심은 항상 “경계를 넘었는가”다.

### 4.3 overflow

overflow는 정해진 저장 용량을 초과하는 쓰기다.  
버퍼 오버플로우에서는 그 초과분이 인접 메모리를 침범한다.

### 4.4 out-of-bounds write

유효 범위를 벗어난 위치에 쓰는 것이다.  
버퍼 오버플로우의 핵심 행위다.

### 4.5 null terminator

문자열 관련 버퍼 처리에서는 종료 문자까지 고려해야 한다.  
이걸 빼먹으면 길이 계산이 자주 틀어진다.

### 4.6 stack / heap corruption

오버플로우는 버퍼 하나만 망치는 것이 아니라  
스택이나 힙 전체 상태를 손상시킬 수 있다.

## 5. 실행 흐름 또는 처리 순서

다음 코드를 기준으로 생각해보자.

```c
char buf[8];
strcpy(buf, "abcdefghijkl");
```

### 5.1 버퍼 준비

먼저 `buf`는 8바이트를 담을 수 있는 버퍼다.  
즉, 유효하게 쓸 수 있는 공간은 8개의 `char`뿐이다.

### 5.2 복사 시작

`strcpy`는 원본 문자열을 문자 단위로 복사한다.  
그리고 종료 문자 `'\0'`까지 포함해 계속 복사하려 한다.

### 5.3 버퍼 경계 도달

`buf`는 8바이트만 수용할 수 있다.  
하지만 원본 문자열은 그보다 더 길다.

즉, 어느 시점부터는 `buf` 내부가 아니라  
그 뒤의 메모리에 계속 쓰게 된다.

### 5.4 인접 메모리 침범

이제 원래 `buf`에 속하지 않는 다른 메모리 내용이 덮어써질 수 있다.  
무엇이 덮이는지는 배치에 따라 다를 수 있다.

- 다른 지역 변수
- 스택 프레임 정보
- 다른 힙 객체
- 관리 메타데이터

### 5.5 결과는 보장되지 않음

겉보기에는 당장 안 터질 수도 있다.  
혹은 즉시 비정상 종료할 수도 있다.  
혹은 나중에 전혀 다른 위치에서 문제가 터질 수도 있다.

즉, 버퍼 오버플로우는 발생 시점과 오류가 드러나는 시점이 다를 수 있다.

## 6. 성능 / 최적화 관점

### 6.1 경계 검사를 안 하는 것이 C의 성능 모델 일부다

C가 빠른 이유 중 하나는 많은 자동 검사 비용을 기본적으로 생략하기 때문이다.  
배열 경계 검사도 대표적인 예다.

즉, 버퍼 오버플로우 위험은  
저수준 제어와 성능의 대가로 프로그래머가 떠안는 책임 중 하나다.

### 6.2 “빠르게 보이는 코드”가 실제로는 더 위험하고 비싸질 수 있다

경계 검사를 생략하고, 길이 검증을 건너뛰고,  
빠른 문자열 복사를 한다고 생각할 수 있다.  
하지만 버퍼 오버플로우가 한 번 생기면 디버깅 비용, 장애 비용, 보안 비용이 훨씬 커진다.

즉, 무방비 저수준 코드는 장기적으로는 비싼 코드다.

### 6.3 안전한 길이 관리가 곧 성능 친화적이기도 하다

버퍼 크기와 실제 데이터 길이를 명확히 관리하면  
불필요한 복사, 불필요한 재할당, 과도한 검사도 줄이기 쉽다.

즉, 안전성과 성능은 완전히 반대 방향만은 아니다.  
명확한 길이 모델은 둘 다에 도움이 된다.

### 6.4 검사 비용보다 오류 비용이 훨씬 큰 경우가 많다

특히 파일 입력, 네트워크 입력, 사용자 입력처럼 외부 데이터가 들어오는 경우,  
경계 검사를 아끼는 것은 보통 전혀 좋은 최적화가 아니다.

즉, 신뢰할 수 없는 입력에서는 안전 검사가 사실상 필수다.

## 7. 실무에서 중요한 판단 기준

### 7.1 “버퍼 크기”와 “현재 데이터 길이”를 항상 분리해서 본다

다음 두 개념은 항상 다르다.

- 이 버퍼가 담을 수 있는 최대 크기
- 지금 실제로 들어 있는 데이터 길이

이 둘을 혼동하면 거의 반드시 오버플로우나 잘못된 길이 계산이 생긴다.

### 7.2 문자열은 종료 문자까지 포함해 계산해야 한다

C 문자열은 문자 수만 세면 안 된다.  
종료 문자 `'\0'`를 포함한 저장 공간을 항상 계산해야 한다.

즉, 길이 8 문자열을 담으려면 최소 9바이트가 필요할 수 있다.

### 7.3 외부 입력은 절대 신뢰하지 않는다

파일, 네트워크, 사용자 입력, 환경 변수, 외부 라이브러리 입력은  
항상 버퍼 경계를 넘을 가능성을 염두에 두고 다뤄야 한다.

즉, 입력 길이 검증은 선택이 아니라 기본이다.

### 7.4 복사와 연결 함수는 목적지 크기를 먼저 점검해야 한다

문자열 복사, 메모리 복사, 연결, 포맷 출력은  
항상 목적지 버퍼 크기와 함께 생각해야 한다.

즉, “무엇을 복사할 것인가”보다 먼저  
“어디까지 안전하게 복사할 수 있는가”를 봐야 한다.

### 7.5 스택 버퍼와 힙 버퍼 모두 안전하지 않다

초보자는 종종 스택 버퍼만 위험하다고 생각한다.  
하지만 힙 버퍼도 충분히 위험하다.

즉, 저장 위치보다 먼저 중요한 것은  
**객체 경계를 넘는가**다.

---

## 8. 더 깊게 볼 포인트

### 8.1 use-after-free와의 차이

둘 다 메모리 안전성 문제지만,  
버퍼 오버플로우는 경계 위반이고 use-after-free는 lifetime 위반이라는 점에서 구분해 볼 수 있다.

### 8.2 안전한 문자열 처리 함수 설계

길이 인자를 함께 받는 API, 버퍼 크기를 명시적으로 전달하는 API 설계로 확장할 수 있다.

### 8.3 stack smashing과 보안 보호 기법

스택 보호 기법, canary 같은 개념은 버퍼 오버플로우와 직접 연결된다.

### 8.4 힙 메타데이터 손상

힙 버퍼 오버플로우가 왜 단순 배열 초과를 넘어서 더 큰 allocator 손상으로 이어질 수 있는지로 확장할 수 있다.

### 8.5 정적 분석 및 sanitizer

AddressSanitizer, 경계 검사 도구, 정적 분석기를 통해 버퍼 오버플로우를 실전에서 잡는 방법으로 이어질 수 있다.

## 정리

버퍼 오버플로우는 C에서 가장 대표적인 메모리 안전성 문제 중 하나다.

핵심은 다음과 같다.

- 버퍼는 유한한 크기를 가진 메모리 객체다.
- 버퍼 오버플로우는 그 유효 경계를 넘어 데이터를 쓰는 것이다.
- 문자열 처리에서는 종료 문자까지 포함한 길이 계산이 중요하다.
- C는 자동 경계 검사를 기본 제공하지 않는다.
- 스택 버퍼 오버플로우는 인접한 지역 데이터나 실행 구조를 침범할 수 있다.
- 힙 버퍼 오버플로우는 다른 힙 객체나 관리 정보를 손상시킬 수 있다.
- 버퍼 오버플로우는 undefined behavior로 이어진다.
- 겉으로 당장 정상처럼 보여도 전혀 안전하지 않다.

즉, 버퍼 오버플로우를 이해한다는 것은  
단순히 “배열 넘치면 안 된다”를 아는 것이 아니라  
**메모리 객체의 경계가 왜 중요한지, 그 경계를 넘는 순간 어떤 안전성·정확성·보안 문제가 생기는지 구조적으로 이해하는 것**을 뜻한다.

이 관점이 잡혀야 문자열 처리, 배열 처리, 외부 입력 검증, 안전한 API 설계, C 보안 문제를 훨씬 더 정확하게 볼 수 있다.
