## 1. 왜 이 주제를 깊게 봐야 하는가

C를 처음 배울 때는 보통 `.c` 파일을 하나 만들고 컴파일해서 실행해 본다.  
이 단계에서는 “컴파일한다”는 말을 거의 마법처럼 사용해도 큰 문제가 없다.

하지만 프로그램이 커지고 파일이 여러 개로 나뉘기 시작하면 곧바로 다음 질문이 생긴다.

- 왜 `.c` 파일 여러 개가 하나의 실행 파일이 되는가?
- 선언만 있고 정의가 없으면 왜 마지막에 에러가 나는가?
- `#include`는 정확히 언제 반영되는가?
- 오브젝트 파일은 무엇인가?
- 링크 에러는 왜 컴파일 에러와 다른가?
- 라이브러리는 어디에서 연결되는가?

이 질문들은 단순 문법 문제가 아니다.  
프로그램이 **소스 코드에서 실행 파일이 되기까지 어떤 단계를 거치는가**와 직접 연결된다.

입문 단계에서는 보통 이렇게 말한다.

- 컴파일한다
- 실행 파일이 나온다

하지만 실제로는 그 사이에 여러 단계가 있다.

- 전처리
- 컴파일
- 어셈블
- 링크

이 단계들을 구분해서 이해해야 비로소 다음이 명확해진다.

- 헤더 파일은 왜 필요한가
- 선언과 정의는 왜 분리되는가
- 오브젝트 파일은 왜 존재하는가
- 여러 파일이 어떻게 하나로 합쳐지는가
- undefined reference 같은 에러는 왜 생기는가
- 정적 라이브러리와 동적 라이브러리는 어디서 연결되는가

예를 들어 다음 구조를 보자.

    // math_utils.h
    int add(int a, int b);

    // math_utils.c
    int add(int a, int b) {
        return a + b;
    }

    // main.c
    #include "math_utils.h"

    int main(void) {
        return add(1, 2);
    }

이 프로그램은 단순히 “파일 2개를 합친다” 수준으로 이해하면 부족하다.  
실제로는 각 파일이 먼저 개별적으로 처리되고, 나중에 심볼 기준으로 연결된다.

즉, 빌드 과정은 단일 덩어리가 아니라 **단계별 변환 파이프라인**이다.

이 문서는 C 소스 코드가 실행 파일이 되기까지의 전체 흐름을 전처리, 컴파일, 어셈블, 링크로 나누어 설명하고, 각 단계가 어떤 입력과 출력을 가지며, 왜 이 구조가 C 프로그램 이해에 중요한지 정리하기 위해 작성한다.

## 2. 전체 구조

C 프로그램이 실행 파일이 되기까지의 흐름을 큰 구조로 보면 다음과 같다.

### 2.1 전처리(preprocessing)

가장 먼저 소스 코드는 전처리기를 거친다.

이 단계에서는 주로 다음이 처리된다.

- `#include`
- `#define`
- 조건부 컴파일(`#if`, `#ifdef` 등)
- 매크로 확장

즉, 전처리는 “진짜 C 문법 해석 전”에 텍스트 수준 정리를 하는 단계다.

### 2.2 컴파일(compilation)

전처리 결과를 바탕으로 C 문법을 해석하고,  
의미를 분석하고, 더 낮은 수준의 코드 표현으로 변환하는 단계다.

즉, 이 단계에서 컴파일러는 다음을 본다.

- 문법이 맞는가
- 타입이 맞는가
- 식과 문장이 의미적으로 타당한가

이 단계의 결과는 보통 어셈블리 코드 수준으로 생각할 수 있다.

### 2.3 어셈블(assembly)

컴파일러가 만든 어셈블리 표현을 실제 기계어에 가까운 오브젝트 파일로 바꾸는 단계다.

이 결과물은 보통 바로 실행 가능한 완성 파일은 아니다.  
대신 아직 연결되지 않은 기계 코드 조각과 심볼 정보를 가진 중간 산출물이다.

즉, 이 단계의 핵심 결과는 오브젝트 파일이다.

### 2.4 링크(linking)

여러 오브젝트 파일과 라이브러리를 연결해서 최종 실행 파일을 만드는 단계다.

이 단계에서는 다음이 중요하다.

- 외부 함수 참조 해결
- 전역 심볼 연결
- 라이브러리 코드 결합
- 최종 실행 파일 생성

즉, 링크는 “따로 만든 조각들을 실제 하나의 프로그램으로 결합하는 단계”다.

## 3. 내부 동작 원리

### 3.1 전처리는 텍스트를 먼저 바꾼다

전처리기는 C 의미를 깊게 이해하기 전에 먼저 텍스트 수준 작업을 한다.

예를 들어 다음 코드가 있다고 하자.

    #include "math_utils.h"

이건 개념적으로 보면 `math_utils.h` 내용을 이 자리에 펼쳐 넣는 것과 비슷하다.

즉, 전처리 단계에서는 헤더 파일이 “연결”되는 것이 아니라  
현재 파일 안에 내용이 삽입되는 효과가 난다.

또한 다음 같은 매크로도 처리된다.

    #define SIZE 10

이후 소스 안의 `SIZE`는 전처리 과정에서 치환될 수 있다.

즉, 전처리는 텍스트 조작 단계다.

### 3.2 컴파일은 C 코드를 해석한다

전처리 후에는 순수한 C 코드 형태가 된다.  
이제 컴파일러는 이 코드를 실제 언어 규칙에 따라 해석한다.

예를 들어 다음을 본다.

    int add(int a, int b) {
        return a + b;
    }

컴파일러는 여기서 다음을 판단한다.

- 함수 선언과 정의가 문법적으로 맞는가
- `a`, `b`는 `int`인가
- `a + b` 결과를 `int`로 반환해도 되는가
- 함수 몸체가 올바른가

즉, 컴파일 단계는 단순 문자열 처리 단계가 아니라  
**문법과 의미 분석 단계**다.

### 3.3 컴파일 결과는 보통 어셈블리 수준 표현으로 이어진다

컴파일러는 C 코드를 내부적으로 분석한 뒤 더 낮은 수준의 코드 형태로 바꾼다.  
이를 사람이 읽을 수 있는 어셈블리로 생각하면 이해하기 쉽다.

즉, 이 단계에서는 다음이 일어난다.

- 변수 배치 전략 결정
- 함수 호출 구조 반영
- 연산을 CPU 명령 수준으로 낮춤
- 제어 흐름 변환

즉, C의 추상 문장이 실제 기계 동작에 가까운 표현으로 바뀐다.

### 3.4 어셈블은 실제 오브젝트 파일을 만든다

어셈블러는 어셈블리 표현을 기계 코드 조각으로 바꾸고,  
그 결과를 오브젝트 파일에 담는다.

오브젝트 파일은 보통 다음을 포함한다.

- 기계 코드 조각
- 데이터 영역 정보
- 심볼 정보
- 아직 해결되지 않은 외부 참조 정보

즉, 오브젝트 파일은 “거의 실행 가능해 보이지만 아직 혼자서는 완전하지 않은 코드 조각”이다.

### 3.5 링크는 오브젝트 파일 사이의 참조를 맞춘다

예를 들어 `main.c`가 `add`를 호출한다고 하자.

    int main(void) {
        return add(1, 2);
    }

컴파일 시점에 `main.c`는 `add`의 선언만 보고도 코드를 만들 수 있다.  
하지만 실제 `add` 구현은 `math_utils.c` 쪽에 있을 수 있다.

링커는 이때 다음을 한다.

- `main.o` 안의 “add를 찾는다”는 참조 확인
- `math_utils.o` 안의 실제 `add` 정의 확인
- 둘을 연결
- 최종 실행 파일 생성

즉, 링크 단계는 **이름 기반 참조를 실제 정의와 결합하는 단계**다.

### 3.6 컴파일 에러와 링크 에러는 다르다

이 차이는 매우 중요하다.

컴파일 에러는 보통 다음과 관련된다.

- 문법 오류
- 타입 오류
- 선언 불일치
- 의미 분석 오류

반면 링크 에러는 보통 다음과 관련된다.

- 선언은 있는데 정의가 없음
- 심볼을 찾지 못함
- 중복 정의
- 라이브러리 연결 실패

즉, 컴파일이 통과했다고 해서 프로그램이 완성된 것은 아니다.  
링크까지 성공해야 진짜 실행 가능한 프로그램이 된다.

## 4. 핵심 구성 요소

### 4.1 preprocessing

전처리는 빌드 흐름의 첫 단계다.  
헤더 포함, 매크로 치환, 조건부 컴파일 같은 텍스트 기반 처리를 담당한다.

즉, 전처리는 C의 의미 분석보다 먼저 일어나는 “준비 작업”이다.

### 4.2 compilation

컴파일은 C 소스의 문법과 의미를 해석하고,  
더 낮은 수준의 표현으로 바꾸는 단계다.

즉, 이 단계가 C 언어 규칙을 본격적으로 적용하는 핵심 해석 단계다.

### 4.3 assembly

어셈블은 저수준 코드 표현을 실제 기계 코드 조각으로 바꾸는 단계다.  
이 결과로 오브젝트 파일이 생성된다.

### 4.4 object file

오브젝트 파일은 아직 최종 실행 파일은 아니지만,  
기계 코드와 심볼 정보를 담고 있는 중간 산출물이다.

즉, 여러 파일을 개별적으로 컴파일한 뒤 나중에 합치기 위한 핵심 조각이다.

### 4.5 symbol

심볼(symbol)은 함수 이름, 전역 변수 이름 등  
링커가 서로 다른 파일 사이의 참조를 맞추는 데 사용하는 식별자다.

예를 들어 `add`라는 함수 이름도 심볼 관점에서 매우 중요하다.

### 4.6 linker

링커는 여러 오브젝트 파일과 라이브러리를 받아  
심볼 참조를 해결하고 최종 실행 파일을 만든다.

즉, 링커는 프로그램 조각들을 실제 하나의 프로그램으로 완성하는 단계의 중심이다.

## 5. 실행 흐름 또는 처리 순서

다음 구조를 기준으로 전체 흐름을 보자.

    // math_utils.h
    int add(int a, int b);

    // math_utils.c
    #include "math_utils.h"

    int add(int a, int b) {
        return a + b;
    }

    // main.c
    #include "math_utils.h"

    int main(void) {
        return add(1, 2);
    }

### 5.1 전처리

먼저 `main.c`와 `math_utils.c` 각각에서 `#include "math_utils.h"`가 처리된다.

개념적으로는 헤더 내용이 각 파일 안에 삽입되는 효과가 난다.  
즉, 각 파일은 자기 안에서 `add` 선언을 알게 된다.

### 5.2 개별 컴파일

그 다음 각 `.c` 파일은 독립적으로 컴파일된다.

- `main.c`는 `main` 함수 코드를 만든다.
- `math_utils.c`는 `add` 함수 코드를 만든다.

이 단계에서 `main.c`는 `add`의 실제 구현이 다른 파일에 있어도 괜찮다.  
왜냐하면 선언을 통해 함수 형태를 알고 있기 때문이다.

### 5.3 오브젝트 파일 생성

각 소스 파일은 각각의 오브젝트 파일이 된다.

예를 들어 개념적으로:

- `main.o`
- `math_utils.o`

이 둘은 아직 완전한 프로그램이 아니다.  
대신 각자 기계 코드와 심볼 정보를 담고 있다.

### 5.4 링크

링커는 `main.o`와 `math_utils.o`를 본다.

- `main.o`는 `add`를 호출하려 한다.
- `math_utils.o`는 `add`를 실제로 정의하고 있다.

링커는 이 둘을 연결해 하나의 실행 파일을 만든다.

### 5.5 최종 실행 파일 생성

모든 외부 참조가 해결되면 최종 실행 파일이 만들어진다.  
이제 비로소 운영체제가 실제로 로드해서 실행할 수 있는 프로그램이 된다.

## 6. 성능 / 최적화 관점

### 6.1 파일 분리는 단순 구조화 이상의 의미가 있다

여러 파일로 나누면 코드 구조가 좋아지는 것뿐 아니라,  
변경된 파일만 다시 컴파일하는 식의 빌드 효율에도 도움이 된다.

즉, 프로그램 구조와 빌드 효율은 연결된다.

### 6.2 컴파일 단위 분리는 대규모 프로젝트에서 중요하다

작은 예제에서는 한 파일이어도 큰 문제가 없지만,  
실무에서는 파일이 수십 개, 수백 개가 될 수 있다.

이때 각 파일이 독립적으로 오브젝트 파일이 되고,  
나중에 링크로 결합되는 구조는 확장성과 유지보수에 중요하다.

### 6.3 링크 시점 최적화나 인라인 관련 이해로 이어질 수 있다

초기 단계에서는 깊게 다루지 않아도 되지만,  
컴파일과 링크가 분리된 구조를 이해하면 나중에 다음 주제도 자연스럽게 이어진다.

- 링크 타임 최적화
- 심볼 가시성
- 정적/동적 라이브러리 차이
- 인라인 함수와 헤더 설계

즉, 이 문서는 단순 빌드 절차를 넘어서 성능 최적화 입구이기도 하다.

### 6.4 잘못된 구조는 빌드 비용과 디버깅 비용을 높인다

헤더에 불필요한 내용을 많이 넣거나,  
인터페이스/구현 경계를 흐리거나,  
전역 심볼을 무분별하게 노출하면 다음 문제가 생기기 쉽다.

- 불필요한 재컴파일
- 중복 의존
- 이름 충돌
- 링크 에러 증가
- 디버깅 복잡도 증가

즉, 빌드 구조 이해는 코드 품질과도 직접 연결된다.

## 7. 실무에서 중요한 판단 기준

### 7.1 컴파일과 링크를 분리해서 생각해야 한다

실무에서 에러를 볼 때는 먼저 구분해야 한다.

- 이건 컴파일 단계 문제인가
- 아니면 링크 단계 문제인가

예를 들어 문법 오류는 링크가 아니라 컴파일 문제다.  
반면 undefined reference는 대개 링크 단계 문제다.

즉, 오류를 단계별로 분류하는 능력이 매우 중요하다.

### 7.2 선언만 있다고 끝나는 것이 아니다

함수 선언이 있다고 컴파일이 될 수는 있다.  
하지만 실제 정의가 없으면 결국 최종 프로그램은 완성되지 않는다.

즉, declaration과 definition의 역할 차이를 빌드 단계와 함께 연결해서 봐야 한다.

### 7.3 헤더는 공유 인터페이스만 담아야 한다

헤더가 커질수록 전처리와 컴파일 영향 범위도 커진다.  
또한 구현 세부가 헤더에 들어가면 구조가 흐려진다.

즉, 헤더는 필요한 선언만 담고, 구현은 `.c`로 분리하는 것이 좋다.

### 7.4 심볼 충돌과 중복 정의를 조심해야 한다

여러 파일이 결합되는 구조에서는 이름 충돌과 중복 정의가 실제 문제로 이어진다.

즉, 파일 단위로만 사고하면 부족하고,  
최종 프로그램 전체에서 심볼이 어떻게 보이는지 생각해야 한다.

### 7.5 라이브러리와 모듈 설계로 확장해서 봐야 한다

이 흐름을 이해하면, 나중에 다음도 자연스럽게 이해할 수 있다.

- 정적 라이브러리
- 동적 라이브러리
- 공개 API
- 내부 전용 심볼
- ABI 안정성

즉, 컴파일-링크 모델은 라이브러리 설계의 바닥이 된다.

## 8. 더 깊게 볼 포인트

### 8.1 extern과 static

심볼을 외부에 공개할지, 파일 내부에만 둘지를 제어하는 핵심 키워드다.  
링크 가시성과 직접 연결된다.

### 8.2 오브젝트 파일 내부 구조

섹션, 심볼 테이블, 재배치 정보 같은 주제로 더 깊게 들어갈 수 있다.  
툴체인 이해에 중요하다.

### 8.3 정적 라이브러리와 동적 라이브러리

링크가 단순히 `.o` 파일끼리만 일어나는 것이 아니라  
라이브러리와도 어떻게 연결되는지 확장해서 볼 수 있다.

### 8.4 전처리기 상세 동작

매크로, include guard, 조건부 컴파일, 토큰 결합 같은 전처리 세부로 더 들어갈 수 있다.

### 8.5 링크 에러 디버깅

undefined reference, multiple definition, unresolved symbol 같은 실전 오류 분석으로 확장할 수 있다.

## 정리

C 프로그램이 실행 파일이 되기까지는 하나의 마법 단계가 아니라 여러 단계가 있다.

핵심은 다음과 같다.

- 전처리는 헤더 포함과 매크로 치환 같은 텍스트 처리를 한다.
- 컴파일은 C 코드를 문법과 의미에 따라 해석한다.
- 어셈블은 저수준 표현을 오브젝트 파일로 만든다.
- 오브젝트 파일은 기계 코드와 심볼 정보를 담는 중간 산출물이다.
- 링크는 여러 오브젝트 파일과 라이브러리를 연결해 최종 실행 파일을 만든다.
- 컴파일 에러와 링크 에러는 다르다.
- 선언과 정의 분리, 헤더 구조, 심볼 관리 모두 이 빌드 흐름 위에 서 있다.

즉, 컴파일, 어셈블, 링크의 전체 흐름을 이해한다는 것은  
단순히 “컴파일하면 실행 파일이 나온다” 수준을 넘어서  
**여러 파일이 어떻게 개별적으로 처리되고, 심볼 기준으로 결합되어 하나의 실행 가능한 프로그램이 되는지 이해한다**는 뜻이다.

이 관점이 잡혀야 C 프로젝트 구조, 라이브러리 설계, 링크 에러 디버깅, 빌드 시스템 이해로 자연스럽게 확장할 수 있다.
