## 1. 왜 이 주제를 공부했는가

C를 배우다 보면 문자열은 생각보다 빨리 등장한다.  
그런데 C의 문자열은 다른 언어처럼 독립적인 고수준 문자열 객체로 다뤄지지 않는다.  
대신 보통 다음 두 형태로 자주 보인다.

```c
char *p = "abc";
```

또는

```c
char a[] = "abc";
```

겉으로 보면 둘 다 문자열 `"abc"`를 나타내는 것처럼 보인다.  
실제로 출력도 비슷하게 될 수 있다.  
그래서 초보자는 자연스럽게 이렇게 생각하기 쉽다.

- 둘은 같은 것 아닌가?
- 둘 다 문자열인데 왜 따로 구분하지?
- 왜 어떤 것은 수정하면 되고 어떤 것은 위험한가?
- 왜 `sizeof` 결과가 다르게 나오기도 하는가?
- 왜 문자열 함수에서는 둘 다 잘 들어가는 것처럼 보이는가?

이 질문들은 매우 중요하다.  
왜냐하면 C 문자열 관련 실수의 상당수가 여기서 시작되기 때문이다.

실제로 다음 같은 코드는 자주 등장한다.

```c
char *p = "abc";
p[0] = 'z';
```

문법만 보면 그냥 첫 글자를 바꾸는 것처럼 보인다.  
하지만 이 코드는 매우 위험하다.

반면 다음은 다른 의미를 가진다.

```c
char a[] = "abc";
a[0] = 'z';
```

이 경우는 보통 수정 가능한 배열 객체를 다루는 것으로 이해할 수 있다.

즉, 둘 다 문자열처럼 보여도 **저장 방식**, **수정 가능성**, **객체의 본질**이 다르다.

이 차이를 정확히 이해해야 다음을 안정적으로 볼 수 있다.

- 문자열 처리 함수
- 버퍼 복사
- API 인자 타입
- 스택/힙 문자열 관리
- 포인터와 배열의 관계
- 안전한 C 문자열 코드 작성

이 문서는 문자열 리터럴과 `char` 배열이 무엇인지, 왜 둘이 비슷해 보이면서도 다르게 다뤄야 하는지, 그리고 실무에서 이 차이가 왜 중요한지 정리하기 위해 작성한다.

## 2. 정의

### 2.1 문자열 리터럴은 프로그램 안에 고정된 문자 시퀀스다

문자열 리터럴(string literal)은 코드 안에 직접 적는 문자열 상수다.

예를 들어:

```
"abc"
```

이 자체가 문자열 리터럴이다.

즉, 문자열 리터럴은 “문자열 값”을 코드에 직접 적어 둔 형태다.  
그리고 이 문자열은 메모리 어딘가에 배치되어 프로그램이 참조할 수 있게 된다.

### 2.2 `char *p = "abc";`는 포인터가 문자열 리터럴 위치를 가리키는 형태다

다음 코드를 보자.

```c
char *p = "abc";
```

여기서 중요한 것은 `p`가 문자열 데이터를 직접 소유하는 배열 객체가 아니라는 점이다.  
`p`는 단지 문자열 리터럴이 놓인 위치를 가리키는 포인터다.

즉,

- `"abc"`는 문자열 리터럴
- `p`는 그 위치를 가리키는 포인터 변수

라는 구조다.

### 2.3 `char a[] = "abc";`는 배열 객체를 만드는 형태다

다음 코드를 보자.

```c
char a[] = "abc";
```

이 경우 `a`는 `char` 배열 객체다.  
즉, `"abc"`의 내용이 배열에 복사되어 들어간다고 이해할 수 있다.

보통은 다음 문자들이 배열 안에 저장된다.

- `'a'`
- `'b'`
- `'c'`
- `'\0'`

즉, 이 경우 `a`는 실제로 수정 가능한 문자 배열 객체다.

### 2.4 문자열의 끝에는 보통 null terminator가 있다

C 문자열은 보통 마지막에 `'\0'`라는 종료 문자를 둔다.  
이 문자는 “여기서 문자열이 끝난다”는 표시다.

예를 들어 `"abc"`는 실제로는 다음과 같이 볼 수 있다.

```
'a' 'b' 'c' '\0'
```

즉, 문자열은 단순 문자 3개가 아니라 종료 표시까지 포함한 문자 시퀀스다.

### 2.5 겉보기 출력이 비슷해도 객체의 본질은 다르다

`char *p = "abc";`와 `char a[] = "abc";`는 둘 다 문자열처럼 출력될 수 있다.  
하지만 본질은 다르다.

- 하나는 문자열 리터럴을 가리키는 포인터다.
- 다른 하나는 문자들이 저장된 배열 객체다.

즉, 출력이 같다고 해서 저장 구조까지 같은 것은 아니다.

## 3. 핵심 개념

### 3.1 문자열 리터럴과 포인터 변수는 서로 다른 존재다

다음 코드를 보자.

```c
char *p = "abc";
```

여기서 초보자가 자주 하는 오해는 `p`가 문자열 그 자체라고 생각하는 것이다.  
하지만 더 정확히 보면 다음처럼 구분해야 한다.

- `"abc"`는 문자열 리터럴이다.
- `p`는 그 문자열 리터럴 위치를 저장하는 포인터 변수다.

즉, `p`는 문자열 데이터 자체가 아니라,  
그 문자열이 있는 위치를 저장하는 별도 객체다.

### 3.2 `char[]`는 실제 문자 배열 객체다

다음은 다르다.

```c
char a[] = "abc";
```

이 경우 `a`는 문자 배열 객체다.  
즉, `a` 안에 문자들이 실제로 들어 있다.

이 차이는 매우 중요하다.  
왜냐하면 배열은 자기 메모리 공간을 가지기 때문이다.

즉,

- `p`는 주소 저장 변수
- `a`는 문자 저장 배열 객체

라는 본질적 차이가 있다.

### 3.3 수정 가능성은 이 차이에서 나온다

다음 코드를 비교해보자.

```c
char *p = "abc";
char a[] = "abc";
```

겉보기는 비슷하다.  
하지만 `a[0] = 'z';`는 배열 원소를 수정하는 개념으로 이해할 수 있다.

반면 `p[0] = 'z';`는 문자열 리터럴을 수정하려는 시도가 될 수 있다.  
이것은 매우 위험하다.

즉, “문자열처럼 보이는 것”과 “수정 가능한 문자 배열 객체”는 구분해야 한다.

### 3.4 `sizeof`는 둘의 차이를 잘 보여준다

다음 코드를 보자.

```c
char *p = "abc";
char a[] = "abc";
```

여기서 `sizeof(p)`는 포인터 변수 크기다.  
즉, 주소 하나를 저장하는 크기다.

반면 `sizeof(a)`는 배열 전체 크기다.  
즉, 문자들과 종료 문자까지 포함한 배열 크기다.

이 차이는 매우 중요하다.  
왜냐하면 포인터와 배열의 본질 차이가 그대로 드러나기 때문이다.

### 3.5 문자열 함수는 둘 다 받을 수 있어 더 헷갈린다

많은 문자열 함수는 `char *` 또는 이에 준하는 포인터 형태로 문자열을 받는다.  
그래서 `char[]`도, 문자열 리터럴을 가리키는 포인터도 비슷하게 들어갈 수 있어 보인다.

이 때문에 초보자는 “둘은 그냥 같은 문자열 표현이구나”라고 생각하기 쉽다.

하지만 함수 인자에서 비슷하게 쓰인다는 것과,  
객체 구조와 수정 가능성이 같다는 것은 전혀 다른 문제다.

### 3.6 문자열은 결국 메모리 위의 문자 시퀀스다

C에서 문자열은 특별한 고수준 객체가 아니다.  
결국은 `char`들이 메모리 위에 순서대로 놓인 구조다.

그래서 문자열 리터럴과 `char[]` 차이는 결국 다음 차이로 귀결된다.

- 어디에 저장되는가
- 누가 그 메모리를 소유하는가
- 수정 가능한가
- lifetime은 어떤가

즉, 문자열 문제도 결국 포인터, 배열, 수명, 메모리 모델 문제다.

## 4. 예시

### 4.1 문자열 리터럴을 가리키는 포인터 예시

다음 코드를 보자.

```c
char *p = "abc";
```

이 코드는 다음처럼 이해할 수 있다.

- `"abc"`라는 문자열 리터럴이 어딘가에 존재한다.
- `p`는 그 위치를 가리킨다.

즉, `p`는 주소를 저장하는 변수다.  
문자열 데이터 자체를 배열처럼 직접 담고 있는 것은 아니다.

### 4.2 문자 배열 예시

다음 코드를 보자.

```c
char a[] = "abc";
```

이 경우 `a`는 문자 배열 객체다.  
배열 안에는 문자들이 들어 있고, 끝에는 `'\0'`가 포함된다.

즉, 이 코드는 문자열 리터럴을 그대로 가리키는 것이 아니라,  
배열 객체를 하나 만들고 그 안에 문자열 내용을 넣는 형태다.

### 4.3 수정 가능성 차이 예시

다음 두 코드를 비교하자.

```c
char *p = "abc";
char a[] = "abc";
```

그리고 각각에 대해 다음을 시도한다고 하자.

```c
p[0] = 'z';
```

또는

```c
a[0] = 'z';
```

둘은 같은 의미가 아니다.

- `a[0] = 'z';`는 배열 원소 수정이다.
- `p[0] = 'z';`는 문자열 리터럴 위치를 수정하려는 시도가 될 수 있다.

즉, 둘은 겉보기 인덱싱 문법은 같아도 다르게 봐야 한다.

### 4.4 `sizeof` 비교 예시

다음 코드를 보자.

```c
char *p = "abc";
char a[] = "abc";
```

```c
printf("%zu\n", sizeof(p));
printf("%zu\n", sizeof(a));
```

여기서 `sizeof(p)`는 포인터 크기다.  
반면 `sizeof(a)`는 배열 전체 크기다.

즉, 이 차이는 “포인터 vs 배열”이라는 본질 차이를 그대로 드러낸다.

### 4.5 포인터 재대입과 배열 불변 형태 차이 예시

다음도 중요한 차이다.

```c
char *p = "abc";
p = "xyz";
```

이건 포인터 `p`가 다른 문자열 리터럴을 가리키게 바꾸는 것이다.  
즉, 포인터 변수 값(주소)을 바꾸는 행위다.

반면 배열 이름은 일반 변수처럼 다른 대상을 가리키도록 바꾸는 식으로 다루지 않는다.  
즉, 배열은 객체고 포인터는 주소 저장 변수라는 차이가 여기서도 드러난다.

## 5. 헷갈리기 쉬운 부분

### 5.1 둘 다 문자열처럼 출력된다고 같은 것이라 생각하는 문제

다음 둘은 모두 출력하면 `"abc"`처럼 보일 수 있다.

```c
char *p = "abc";
char a[] = "abc";
```

하지만 출력 결과가 비슷하다는 것은 겉보기 사용성이 비슷하다는 뜻일 뿐이다.  
객체 구조까지 같다는 뜻은 아니다.

즉, “겉보기 결과가 같다”와 “메모리 구조가 같다”를 구분해야 한다.

### 5.2 `char *`면 항상 수정 가능한 문자열이라고 생각하는 문제

`char *`는 문자 데이터를 가리킬 수 있는 포인터다.  
하지만 그것이 항상 수정 가능한 문자 배열을 가리킨다는 뜻은 아니다.

예를 들어 문자열 리터럴을 가리키는 `char *`는 수정 가능 버퍼와 같은 의미로 쓰면 안 된다.

즉, 포인터 타입만 보고 수정 가능성을 섣불리 단정하면 위험하다.

### 5.3 배열과 포인터 차이를 잊고 `sizeof`를 잘못 쓰는 문제

다음은 전혀 다를 수 있다.

```c
sizeof(p)
sizeof(a)
```

문자열 길이를 구한다고 착각하고 포인터에 `sizeof`를 적용하면 틀린 판단을 하게 된다.  
즉, 포인터인지 배열인지 먼저 확인해야 한다.

### 5.4 문자열 리터럴을 “그냥 상수값” 정도로 가볍게 보는 문제

문자열 리터럴도 결국 메모리에 놓이는 문자 시퀀스다.  
다만 그것을 수정 가능한 배열 객체처럼 다루면 안 된다.

즉, 문자열 리터럴은 “문자열 값”이면서 동시에 **메모리 위에 존재하는 특별한 성격의 데이터**다.

### 5.5 함수 인자에서 둘이 비슷하게 보이는 문제

문자열 함수에 `p`도, `a`도 잘 들어가는 것처럼 보이기 때문에  
둘이 같다고 오해하기 쉽다.

하지만 함수 인자로 전달될 때 포인터처럼 보인다는 것과,  
실제 선언 시 객체 구조가 같다는 것은 다른 문제다.

즉, “전달 방식”과 “원래 객체 성질”을 구분해야 한다.

## 6. 실무 관점

### 6.1 수정 가능한 버퍼와 읽기 전용 성격 문자열을 구분해야 한다

실무에서는 문자열을 다룰 때 먼저 질문해야 한다.

- 이 문자열은 수정할 것인가?
- 아니면 읽기만 할 것인가?

수정 가능한 버퍼라면 `char` 배열이나 동적 할당 버퍼를 생각해야 한다.  
단순 참조용이면 문자열 리터럴을 가리키는 포인터도 가능할 수 있다.

즉, 문자열 표현 선택은 곧 **수정 가능성 설계**다.

### 6.2 API 인자 설계에서 의도가 중요하다

함수가 문자열을 입력으로 받을 때도 의도를 분명히 해야 한다.

- 읽기만 할 문자열인가
- 내부에서 수정할 문자열인가
- 버퍼 길이까지 함께 관리해야 하는가

문자열 리터럴과 배열 차이를 모르면 API 설계가 모호해지고,  
결국 호출자와 구현자 모두가 실수하기 쉬워진다.

### 6.3 버퍼 복사와 길이 계산에서 매우 중요하다

실무에서 문자열 문제는 종종 다음과 연결된다.

- 버퍼 크기 계산 실수
- `sizeof` 오용
- 종료 문자 누락
- 수정 불가능한 문자열에 쓰기
- 길이와 저장 공간 혼동

즉, 문자열 리터럴과 배열 차이는 단순 문법 차이가 아니라 **버퍼 안전성 문제**다.

### 6.4 동적 메모리 문자열 관리와도 연결된다

나중에 `malloc`을 통해 문자열 버퍼를 직접 만들면,  
그 버퍼는 또 다른 형태의 수정 가능한 문자 저장 공간이 된다.

즉, 문자열 리터럴 vs 배열 차이를 이해하는 것은  
다음 단계인 동적 문자열 관리로 넘어가기 위한 중요한 기반이다.

---

## 7. 다음에 볼 것

### 7.1 스택과 힙의 차이

문자 배열이 어디에 만들어지는지, 동적 문자열은 어디에 존재하는지 보려면 스택과 힙 차이를 이해해야 한다.

### 7.2 malloc과 free는 실제로 무엇을 하는가

수정 가능한 문자열 버퍼를 직접 할당하고 해제하려면 동적 메모리 관점이 필요하다.

### 7.3 배열과 포인터는 왜 비슷하면서 다른가

이 문서의 핵심 전제와 연결된다.  
문자열도 결국 배열과 포인터 관계 위에 있다.

### 7.4 문자열 함수의 동작 원리

`strlen`, `strcpy`, `strcmp` 같은 함수들이 왜 종료 문자에 의존하는지, 왜 버퍼 크기를 조심해야 하는지로 자연스럽게 이어진다.

## 정리

문자열 리터럴과 `char` 배열은 겉보기에는 비슷하지만 본질적으로 다르다.

핵심은 다음과 같다.

- 문자열 리터럴은 코드에 적힌 문자 시퀀스다.
- `char *p = "abc";`는 문자열 리터럴 위치를 가리키는 포인터다.
- `char a[] = "abc";`는 문자 배열 객체를 만든다.
- 배열에는 문자들과 종료 문자가 들어간다.
- 수정 가능성은 두 경우에서 다르게 봐야 한다.
- `sizeof(p)`와 `sizeof(a)`는 다르다.
- 둘 다 문자열 함수에 들어갈 수 있어 보여도 객체 구조는 같지 않다.

즉, 문자열 리터럴과 `char` 배열 차이를 이해한다는 것은  
“둘 다 문자열처럼 보인다” 수준을 넘어서  
**메모리 위에 문자열이 어떤 형태로 존재하는지, 그리고 그 형태가 수정 가능성과 안전성에 어떤 차이를 만드는지 이해한다**는 뜻이다.

이 관점이 잡혀야 C 문자열을 훨씬 덜 위험하게 다룰 수 있다.
