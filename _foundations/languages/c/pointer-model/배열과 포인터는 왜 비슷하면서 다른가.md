## 1. 왜 이 주제를 깊게 봐야 하는가

C를 배우는 사람이라면 거의 반드시 한 번은 이런 말을 듣는다.

- 배열은 포인터다.
- 배열 이름은 포인터처럼 동작한다.
- 배열과 포인터는 비슷하다.

이 말들은 완전히 틀린 것도 아니고, 완전히 맞는 것도 아니다.  
문제는 이 문장들이 너무 압축되어 있어서, 제대로 풀어 설명하지 않으면 오히려 더 큰 오해를 만든다는 점이다.

실제로 C에서 배열과 포인터는 매우 밀접하다.  
예를 들어 다음 코드는 겉으로 보면 비슷하게 동작한다.

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

```c
printf("%d\n", arr[1]);
printf("%d\n", p[1]);
```

둘 다 20을 출력할 수 있다.  
그래서 초보자는 쉽게 이렇게 오해한다.

- 배열은 사실 포인터다.
- 배열 이름 자체가 포인터 변수다.
- 배열과 포인터는 그냥 같은 것 아닌가?

하지만 이건 정확하지 않다.  
배열과 포인터는 어떤 상황에서는 비슷하게 보이지만, 본질적으로는 다르다.

이 차이를 정확히 이해하지 못하면 다음에서 계속 흔들린다.

- `sizeof(arr)`와 `sizeof(p)`가 왜 다른가
- 함수 인자에서 배열이 왜 이상하게 보이는가
- 문자열 처리에서 왜 `char *`와 `char[]`가 구분되는가
- 배열은 왜 대입이 안 되는데 포인터는 대입이 되는가
- 다차원 배열과 배열 포인터는 왜 헷갈리는가

즉, 배열과 포인터는 C의 핵심 오해 지점 중 하나다.  
여기를 정확히 정리해야 이후 문자열, 함수 인자, 버퍼 처리, 동적 메모리를 안정적으로 이해할 수 있다.

이 문서는 배열과 포인터가 왜 비슷해 보이는지, 어디까지 비슷한지, 어디서부터는 본질적으로 다른지, 그리고 그 차이가 왜 실무에서 중요한지 깊게 설명하기 위해 작성한다.

## 2. 전체 구조

배열과 포인터의 관계를 큰 구조로 보면 다음과 같다.

### 2.1 배열은 연속된 메모리 객체다

배열은 같은 타입의 여러 원소가 **연속된 메모리 공간**에 놓이는 객체다.

예를 들어:

```c
int arr[3] = {10, 20, 30};
```

이 경우 `arr`는 `int` 3개가 연속해서 저장되는 하나의 배열 객체다.

즉, 배열은 “여러 개의 값을 담는 포인터”가 아니라,  
처음부터 **여러 원소를 포함한 하나의 객체**다.

### 2.2 포인터는 주소를 저장하는 변수다

반면 포인터는 배열이 아니다.  
포인터는 단지 어떤 객체의 주소를 저장하는 변수다.

예를 들어:

```c
int *p = arr;
```

여기서 `p`는 배열 자체가 아니라, 배열 첫 원소의 위치를 가리키는 주소를 저장하는 변수다.

즉,

- 배열은 객체다.
- 포인터는 주소를 담는 변수다.

이 둘은 처음부터 역할이 다르다.

### 2.3 배열 이름은 특정 문맥에서 첫 원소 주소처럼 변환된다

배열과 포인터가 비슷하게 보이는 핵심 이유가 여기 있다.

대부분의 표현식 문맥에서 배열 이름은 **배열 첫 원소를 가리키는 포인터처럼 변환(decay)** 된다.

예를 들어:

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

여기서 `arr`는 “배열 전체 객체”지만, `int *p = arr;` 문맥에서는 배열 첫 원소의 주소처럼 사용된다.

즉, 배열은 포인터가 아니지만, 많은 상황에서 **포인터처럼 변환되어 보인다**.

### 2.4 그래서 둘은 비슷하게 동작하지만 같지는 않다

배열 인덱싱과 포인터 인덱싱이 비슷하게 동작하는 이유는,  
배열 이름이 포인터처럼 변환되는 문맥이 많기 때문이다.

하지만 배열 자체의 성질은 포인터와 다르다.

예를 들어:

- 배열은 자기 크기를 포함하는 객체다.
- 포인터는 주소값 하나를 담는 변수다.
- 배열은 다른 배열에 대입할 수 없다.
- 포인터는 다른 주소로 대입 가능하다.

즉, 배열과 포인터는 “닮은 점이 많다”는 것이지, 같은 것이 아니다.

## 3. 내부 동작 원리

### 3.1 배열은 실제 메모리 공간을 차지하는 객체다

다음 코드를 보자.

```c
int arr[3] = {10, 20, 30};
```

이 배열은 `int` 3개가 들어갈 수 있는 실제 메모리 공간을 가진다.  
즉, `arr`는 단순 이름이 아니라 **세 개의 원소를 포함한 배열 객체 전체**를 의미한다.

메모리 관점으로 보면 다음처럼 이해할 수 있다.

- `arr[0]`
- `arr[1]`
- `arr[2]`

이 세 원소가 연속해서 배치된다.

즉, 배열은 “주소 하나”가 아니라 **원소 여러 개가 묶인 실제 객체**다.

### 3.2 포인터는 주소값 하나를 저장한다

다음 코드를 보자.

```c
int *p = arr;
```

여기서 `p`는 배열이 아니다.  
`p`는 주소 하나를 저장하는 변수다.

즉,

- `arr`는 실제 원소들을 포함한 객체
- `p`는 어떤 위치를 가리키는 주소 저장 변수

다시 말해, 배열은 데이터를 가진다.  
포인터는 데이터의 위치를 가진다.

### 3.3 배열 이름은 왜 포인터처럼 보이는가

다음 코드가 자연스럽게 동작한다.

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

이게 가능한 이유는 배열 이름 `arr`가 이 문맥에서 **첫 번째 원소의 주소**처럼 변환되기 때문이다.

즉, `arr`는 이 문맥에서 사실상 다음과 비슷하게 해석되는 효과를 낸다.

- `&arr[0]`

이 변환 때문에 배열과 포인터가 자꾸 같은 것처럼 보인다.

하지만 중요한 점은,  
배열 이름이 포인터처럼 “변환된다”는 것과,  
배열 자체가 포인터 변수라는 것은 전혀 다른 말이라는 점이다.

### 3.4 인덱싱이 비슷하게 보이는 이유

다음 코드를 보자.

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

```c
printf("%d\n", arr[1]);
printf("%d\n", p[1]);
```

둘 다 20을 읽을 수 있다.

이건 배열과 포인터가 완전히 같기 때문이 아니라,  
포인터 인덱싱이 내부적으로 포인터 연산과 역참조로 해석되기 때문이다.

즉, 개념적으로는 다음처럼 볼 수 있다.

- `arr[1]` → 배열 두 번째 원소 접근
- `p[1]` → `*(p + 1)`

배열 이름이 포인터처럼 변환될 수 있기 때문에 둘이 비슷하게 보이는 것이다.

### 3.5 `sizeof`에서 차이가 드러난다

배열과 포인터의 차이가 가장 잘 드러나는 곳 중 하나가 `sizeof`다.

다음 코드를 보자.

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

```c
printf("%zu\n", sizeof(arr));
printf("%zu\n", sizeof(p));
```

여기서 `sizeof(arr)`는 배열 전체 크기다.  
즉, `int` 3개 크기를 반영한다.

반면 `sizeof(p)`는 포인터 변수 자체의 크기다.  
즉, 주소 하나를 저장하는 크기다.

이 차이는 매우 중요하다.  
왜냐하면 배열은 객체 전체이고, 포인터는 주소 저장 변수라는 본질 차이가 그대로 드러나기 때문이다.

### 3.6 함수 인자에서는 배열이 더 강하게 포인터처럼 보인다

다음 코드를 보자.

```c
void print_first(int arr[]) {
    printf("%d\n", arr[0]);
}
```

겉으로 보면 함수가 배열을 받는 것처럼 보인다.  
하지만 함수 인자 문맥에서는 배열이 포인터 형태로 다뤄진다.

즉, 함수 안에서 `arr`는 배열 전체 객체로 남아 있는 것이 아니라,  
배열 첫 원소를 가리키는 포인터처럼 사용된다.

그래서 함수 안에서 `sizeof(arr)`를 하면 배열 전체 크기가 아니라 포인터 크기처럼 보일 수 있다.

이 지점이 초보자가 가장 많이 헷갈리는 부분 중 하나다.

## 4. 핵심 구성 요소

### 4.1 array object

배열 객체는 같은 타입 원소 여러 개를 연속해서 담는 실제 메모리 객체다.

예를 들어:

```c
int arr[3];
```

이 선언은 `int` 3개를 담는 배열 객체를 만든다.

즉, 배열은 단일 값이 아니라 **원소 집합을 가진 객체**다.

### 4.2 pointer variable

포인터 변수는 주소 하나를 저장하는 변수다.

예를 들어:

```c
int *p;
```

이 선언은 `int` 객체의 주소를 저장할 수 있는 변수를 만든다.

즉, 포인터는 배열과 달리 자기 안에 여러 원소를 포함하지 않는다.

### 4.3 decay

decay는 배열 이름이 대부분의 표현식 문맥에서  
배열 첫 원소를 가리키는 포인터처럼 변환되는 현상이다.

이 개념이 배열과 포인터 관계의 핵심이다.  
배열과 포인터가 닮아 보이는 이유는 바로 이 변환 때문이다.

### 4.4 indexing

배열 인덱싱은 배열 안의 특정 원소에 접근하는 방법이다.

```
arr[1]
```

포인터 인덱싱도 가능하다.

```
p[1]
```

이 둘이 비슷하게 보이는 이유는, 포인터 인덱싱이 결국 포인터 연산과 역참조에 기반하기 때문이다.

### 4.5 contiguous memory

배열은 연속 메모리 구조를 가진다.  
즉, 배열 원소들은 메모리상에서 나란히 놓인다.

이 성질 때문에 다음이 가능하다.

- 인덱싱
- 포인터 순회
- 버퍼 처리
- 문자열 처리
- 캐시 친화적 접근

### 4.6 `sizeof`

`sizeof`는 배열과 포인터 차이를 확인하는 가장 좋은 도구 중 하나다.

- 배열에 적용하면 배열 전체 크기
- 포인터에 적용하면 포인터 변수 크기

즉, 이 연산은 둘의 본질 차이를 드러내 준다.

## 5. 실행 흐름 또는 처리 순서

다음 코드를 기준으로 흐름을 보자.

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

### 5.1 배열 선언

먼저 `arr`라는 배열 객체가 생성된다.

- `int` 원소 3개를 담는 공간 확보
- 연속된 메모리 배치
- 초기값 10, 20, 30 저장

즉, 이 시점에서 `arr`는 실제 원소들을 포함한 객체다.

### 5.2 배열 이름 사용

이제 `arr`를 표현식에 사용한다.

```c
int *p = arr;
```

여기서 `arr`는 배열 전체 그 자체로 남는 것이 아니라,  
첫 원소 주소처럼 변환되어 사용된다.

즉, decay가 일어난다.

### 5.3 포인터에 주소 저장

`p`는 배열 첫 원소를 가리키는 주소를 저장한다.

즉, `p`는 배열이 아니지만, 배열 원소들을 순서대로 따라갈 수 있는 시작 위치를 갖게 된다.

### 5.4 인덱싱 사용

이제 다음이 가능하다.

```
p[1]
```

이 표현은 `p`가 가리키는 시작 위치에서 한 칸 이동한 원소를 읽는 의미다.  
배열 원소들이 연속되어 있기 때문에 자연스럽게 동작한다.

### 5.5 `sizeof` 차이 확인

다음 시점에서 차이가 드러난다.

```c
sizeof(arr)
sizeof(p)
```

- `sizeof(arr)` → 배열 전체 크기
- `sizeof(p)` → 포인터 변수 크기

즉, 같은 듯 보여도 객체와 주소 변수는 분명히 다르다.

## 6. 성능 / 최적화 관점

### 6.1 배열의 연속 메모리 구조는 성능에 유리하다

배열은 원소가 연속해서 배치된다.  
이 성질은 CPU 캐시 친화성에 유리하다.

즉, 배열을 순차적으로 읽는 코드는 메모리 접근 효율이 좋은 편이다.

예를 들어:

```c
for (int i = 0; i < 3; i++) {
    printf("%d\n", arr[i]);
}
```

이런 순차 접근은 하드웨어 관점에서도 자연스럽다.

### 6.2 포인터 순회와 인덱스 순회는 표현만 다를 뿐 연결되어 있다

배열 순회는 인덱스로 할 수도 있고 포인터로 할 수도 있다.

```c
for (int i = 0; i < 3; i++) {
    printf("%d\n", arr[i]);
}
```

또는

```c
int *p = arr;
for (int i = 0; i < 3; i++) {
    printf("%d\n", *(p + i));
}
```

이 둘은 표현 방식은 다르지만 같은 연속 메모리를 접근한다는 점에서 강하게 연결된다.

즉, 배열과 포인터 관계를 이해하면 성능 측면에서도 데이터 접근 방식을 더 잘 볼 수 있다.

### 6.3 함수 인자에서 배열 크기 정보가 사라지는 문제는 성능보다 안전성 문제다

배열이 함수 인자에서 포인터처럼 다뤄지면, 배열 전체 크기 정보는 자동으로 전달되지 않는다.

즉, 보통 길이를 따로 넘겨야 한다.

이걸 놓치면 성능 문제가 아니라 더 큰 문제, 즉 버퍼 경계 오류가 생길 수 있다.

### 6.4 배열과 포인터를 구분해야 API 설계가 안전해진다

실무에서 다음은 전혀 다른 의미다.

- 실제 크기가 고정된 배열 객체
- 어떤 위치만 가리키는 포인터

이 둘을 혼동하면 함수 시그니처, 버퍼 길이 처리, 문자열 처리, 복사 범위 계산에서 실수가 생기기 쉽다.

즉, 배열과 포인터 차이를 정확히 보는 것은 안전한 API 설계와 연결된다.

---

## 7. 실무에서 중요한 판단 기준

### 7.1 배열은 포인터가 아니다

이 문장의 핵심 결론은 분명하다.

- 배열은 객체다.
- 포인터는 주소 저장 변수다.

배열 이름이 자주 포인터처럼 변환될 뿐이지,  
배열 자체가 포인터인 것은 아니다.

이 기준을 놓치면 이후 거의 모든 관련 주제에서 흔들린다.

### 7.2 `sizeof`를 쓸 때는 지금 다루는 것이 배열인지 포인터인지 먼저 확인해야 한다

다음은 다르다.

```c
sizeof(arr)
sizeof(p)
```

배열 길이를 구할 때는 배열 객체 그 자체가 보이는 문맥인지,  
이미 포인터처럼 다뤄진 문맥인지 먼저 판단해야 한다.

특히 함수 인자 안에서는 매우 조심해야 한다.

### 7.3 함수에 배열을 넘길 때는 길이 정보를 별도로 넘기는 습관이 필요하다

다음 같은 함수는 위험하다.

```c
void process(int arr[]) {
}
```

겉으로는 배열을 받는 것 같지만, 함수 안에서는 보통 포인터처럼 다뤄진다.  
따라서 길이 정보를 잃는다.

실무에서는 보통 다음처럼 길이를 함께 넘긴다.

```c
void process(int arr[], size_t len) {
}
```

즉, 배열 처리에서는 데이터 포인터와 길이를 함께 관리하는 습관이 중요하다.

### 7.4 문자열도 결국 배열과 포인터 관계 위에 올라간다

C 문자열은 `char` 배열과 `char *` 포인터가 자주 섞여 보인다.  
배열과 포인터 차이를 정확히 모르면 문자열 처리에서 매우 자주 실수한다.

즉, 이 문서는 문자열 문서의 바로 전제다.

## 8. 더 깊게 볼 포인트

### 8.1 다차원 배열

다차원 배열에서는 배열과 포인터 차이가 더 복잡하게 드러난다.  
특히 `int **`와 `int [][N]`은 같은 것이 아니다.

### 8.2 배열 포인터

포인터가 배열 첫 원소를 가리키는 것과, 배열 전체를 가리키는 포인터는 다르다.  
이 차이는 선언 문법과 함께 더 깊게 볼 가치가 있다.

### 8.3 VLA

가변 길이 배열은 배열의 크기가 런타임에 결정되는 형태다.  
일반 고정 배열과 비교하면 수명과 스택 사용량 관점에서 더 흥미로운 차이가 있다.

### 8.4 구조체 안 배열 필드와 포인터 필드 차이

구조체 안에 배열을 직접 넣는 것과 포인터를 넣는 것은 메모리 배치와 소유권에서 큰 차이를 만든다.  
실무 데이터 구조 설계에서 중요하다.

### 8.5 문자열 리터럴과 `char[]`

배열과 포인터 차이를 실제로 가장 자주 체감하는 사례 중 하나다.  
다음 문서와 자연스럽게 연결된다.

---

## 정리

배열과 포인터는 매우 밀접하지만 같은 것이 아니다.

핵심은 다음과 같다.

- 배열은 연속된 원소들을 가진 실제 메모리 객체다.
- 포인터는 주소를 저장하는 변수다.
- 배열 이름은 많은 문맥에서 첫 원소를 가리키는 포인터처럼 변환된다.
- 이 decay 때문에 둘이 비슷하게 보인다.
- 인덱싱은 비슷하게 동작할 수 있지만 본질은 다르다.
- `sizeof(arr)`와 `sizeof(p)`는 다르다.
- 함수 인자에서 배열은 포인터처럼 다뤄지므로 길이 정보를 잃기 쉽다.

즉, 배열과 포인터를 제대로 이해한다는 것은  
“둘은 같다”는 단순 암기를 버리고,  
**배열 객체와 주소 저장 변수의 차이, 그리고 배열 이름이 포인터처럼 변환되는 문맥을 함께 이해하는 것**을 뜻한다.

이 관점이 잡혀야 문자열, 버퍼 처리, 함수 인자, 다차원 배열, 동적 메모리를 훨씬 안정적으로 이해할 수 있다.
