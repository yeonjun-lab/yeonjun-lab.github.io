## 1. 왜 이 주제를 공부했는가

프로그래밍 입문에서는 변수를 보통 “값을 저장하는 상자”라고 설명한다.  
이 설명은 아주 처음 방향을 잡는 데는 도움이 된다.  
하지만 C를 배우는 단계에서는 이 설명만으로는 부족하다.

왜냐하면 C에서 변수는 단순한 이름표가 아니기 때문이다.  
C에서 변수는 실제로 메모리 안에 존재하는 객체이고, 특정 타입으로 해석되며, 일정한 생명주기를 가지고, 특정 범위 안에서만 접근 가능하다.

즉, C에서 변수를 이해한다는 것은 단순히 다음을 아는 수준이 아니다.

    int x = 10;

이 문장을 보고 “정수 변수 x에 10을 넣었다”라고 말하는 것으로는 충분하지 않다.  
실제로는 더 많은 질문을 던질 수 있어야 한다.

- `x`는 메모리 어디에 존재하는가?
- `x`는 몇 바이트를 차지하는가?
- `x`는 언제 생성되고 언제 사라지는가?
- `x`의 이름과 `x`라는 객체는 같은 것인가?
- `x`의 값과 `x`의 주소는 무엇이 다른가?

이 질문들을 볼 수 있어야 이후에 자연스럽게 다음 주제로 이어진다.

- 함수 호출과 스택
- 지역 변수와 전역 변수
- 포인터
- 값 복사와 주소 전달
- 동적 메모리
- 수명과 유효 범위

즉, 변수는 C의 문법 요소인 동시에 **실행 모델의 입구**다.  
이 문서는 변수를 메모리 관점에서 다시 정의하고, 이름·값·주소·수명·범위라는 핵심 개념을 연결하기 위해 작성한다.

## 2. 정의

### 2.1 변수는 메모리 객체다

C에서 변수는 단순한 “기호”가 아니다.  
변수는 실제로는 **메모리 안에 존재하는 객체(object)** 다.

즉, 변수는 보통 다음을 포함한다.

- 특정 타입
- 특정 크기
- 특정 저장 위치
- 특정 생명주기
- 특정 접근 가능 범위

예를 들어 다음 변수를 보자.

    int x = 10;

여기서 `x`는 단순 문자열이 아니라,  
정수형으로 해석되는 메모리 객체를 가리키는 이름이다.

### 2.2 이름은 객체를 식별하기 위한 수단이다

변수 이름은 사람이 코드를 읽고 객체를 참조하기 쉽게 하기 위한 식별자다.  
즉, 이름은 매우 중요하지만, 이름 자체가 곧 데이터는 아니다.

실제 실행에서 중요한 것은 메모리 안에 존재하는 객체 그 자체다.

즉, `x`라는 이름은 편의를 위한 식별자이고,  
실제로 값이 들어 있는 것은 `x`라는 이름이 아니라 **그 이름이 가리키는 메모리 객체**다.

### 2.3 값과 주소는 다르다

변수에는 값이 있고, 그 변수는 메모리 어딘가에 존재한다.  
따라서 변수에 대해 적어도 두 가지를 구분해야 한다.

- 저장된 값
- 그 값이 놓인 위치

예를 들어:

    int x = 10;

여기서 `x`는 값 10으로 읽을 수 있다.  
하지만 `&x`는 `x`의 주소다.  
즉, 값과 주소는 같은 것이 아니다.

### 2.4 변수는 생명주기를 가진다

변수는 한 번 생성되면 영원히 존재하는 것이 아니다.  
변수는 종류에 따라 언제 생성되고 언제 사라질지가 다르다.

예를 들어:

- 지역 변수는 보통 함수 호출 중에만 존재한다.
- 전역 변수는 프로그램 전체 실행 동안 존재할 수 있다.
- `static`이 붙은 변수는 일반 지역 변수와 다른 수명을 가진다.

즉, 변수는 단순 값 저장 공간이 아니라 **시간 축 위에서 존재하는 객체**다.

### 2.5 변수는 접근 가능한 범위를 가진다

모든 변수는 어디서나 접근 가능한 것이 아니다.  
변수는 코드 구조 안에서 접근 가능한 범위, 즉 scope를 가진다.

즉, 변수는 단지 메모리에 존재한다고 끝나는 것이 아니라,  
**어디서 볼 수 있는가**와 **언제까지 유효한가**를 함께 봐야 한다.

## 3. 핵심 개념

### 3.1 declaration

declaration은 “이런 이름과 타입을 가진 대상을 사용하겠다”는 선언이다.

예를 들어:

    int x;

이 문장은 `x`라는 이름의 정수형 변수를 선언한다.

입문 단계에서는 declaration을 단순 문법으로 보기 쉽지만,  
실행 모델 관점에서는 “이 이름으로 다룰 메모리 객체가 존재한다”는 의미와 연결된다.

### 3.2 initialization

initialization은 선언된 변수에 처음 값을 부여하는 것이다.

예를 들어:

    int x = 10;

이 문장은 `x`라는 정수형 객체를 만들고, 초기값 10을 부여한다.

중요한 것은 declaration과 initialization이 항상 같은 것은 아니라는 점이다.

    int x;
    x = 10;

이 경우 첫 줄은 선언이고, 둘째 줄은 나중에 값을 대입하는 것이다.

즉, “변수가 존재하는 것”과 “값을 가지는 것”은 구분해서 봐야 한다.

### 3.3 object

C에서 변수는 object로 이해하는 것이 중요하다.  
object는 메모리 안에 존재하는 저장 단위이며, 타입과 크기와 수명을 가진다.

즉, 변수는 이름이고, object는 실제 저장 실체라고 보면 된다.

이 구분이 중요한 이유는, 이후 포인터나 함수 인자 전달을 볼 때  
이름이 아니라 **객체 자체가 어디에 있고 어떻게 전달되는가**를 보게 되기 때문이다.

### 3.4 value

value는 변수 객체 안에 저장된 내용이다.  
예를 들어:

    int x = 10;

여기서 value는 10이다.

하지만 값은 변수의 전부가 아니다.  
변수는 값만이 아니라 주소, 타입, 수명도 함께 가진다.

즉, C에서 변수는 “값 하나”가 아니라 **메모리 객체 전체**로 봐야 한다.

### 3.5 address

address는 그 객체가 저장된 위치다.

예를 들어:

    int x = 10;

이때 `&x`는 `x` 객체의 주소다.

이 개념은 이후 포인터와 직접 연결된다.  
즉, 주소를 이해하지 못하면 변수도 절반만 이해한 것이다.

### 3.6 lifetime

lifetime은 그 객체가 언제 생성되고 언제까지 유효한지를 뜻한다.

예를 들어 함수 안에서 선언한 지역 변수는 함수가 시작되고 종료되는 사이에만 유효하다.  
반면 전역 변수는 프로그램 전체 동안 존재할 수 있다.

즉, 변수는 “항상 있는 값”이 아니라 **시간에 따라 유효성이 정해지는 객체**다.

### 3.7 scope

scope는 그 변수 이름에 접근할 수 있는 코드 범위다.

예를 들어 함수 안에서 선언한 변수는 보통 그 함수 블록 안에서만 직접 접근 가능하다.  
즉, lifetime과 scope는 관련은 있지만 같은 개념이 아니다.

- lifetime: 언제 존재하는가
- scope: 어디서 접근 가능한가

이 차이를 구분할 수 있어야 이후 지역 변수, 전역 변수, `static` 등을 더 정확히 이해할 수 있다.

## 4. 예시

### 4.1 가장 기본적인 변수 예시

다음 코드를 보자.

    int x = 10;

이 문장은 단순해 보이지만 실제로는 여러 층을 포함한다.

- `int`는 타입이다.
- `x`는 이름이다.
- `10`은 초기값이다.
- 메모리 안에는 `int` 크기의 객체가 존재한다.
- 그 객체는 값 10을 가진다.
- 그 객체는 특정 주소를 가진다.

즉, 이 한 줄만 봐도 변수는 단순 이름표가 아니라 **타입을 가진 메모리 객체**라는 사실이 드러난다.

### 4.2 값과 주소의 차이 예시

다음 코드를 보자.

    int x = 10;
    printf("%d\n", x);
    printf("%p\n", (void*)&x);

첫 번째 출력은 값 10을 보여준다.  
두 번째 출력은 `x`가 저장된 주소를 보여준다.

이 예시는 매우 중요하다.  
왜냐하면 같은 변수에 대해 “값”과 “주소”라는 서로 다른 층을 동시에 볼 수 있기 때문이다.

### 4.3 선언과 초기화 분리 예시

다음 코드를 보자.

    int x;
    x = 10;

이 경우 첫 번째 줄은 변수의 선언이고, 두 번째 줄은 나중에 값을 대입하는 것이다.

반면 아래 코드는 선언과 초기화가 동시에 일어난다.

    int y = 20;

즉, 선언과 초기화는 자주 함께 등장하지만 개념적으로는 분리할 수 있다.

### 4.4 같은 값을 가져도 다른 객체일 수 있는 예시

다음 코드를 보자.

    int a = 10;
    int b = 10;

`a`와 `b`는 둘 다 값 10을 가진다.  
하지만 보통은 서로 다른 메모리 객체다.  
즉, 값이 같다고 해서 같은 객체인 것은 아니다.

이 차이는 이후 값 복사와 참조 개념을 구분할 때 중요하다.

### 4.5 값 복사 예시

다음 코드를 보자.

    int a = 10;
    int b = a;
    b = 20;

이 경우 `b = a;`는 보통 `a`의 값을 `b`에 복사하는 의미다.  
즉, 이후 `b`를 바꿔도 `a`는 그대로다.

이 예시는 매우 중요하다.  
왜냐하면 초보자는 종종 “b가 a를 가리킨다”고 오해하기 때문이다.  
하지만 일반 변수 대입은 기본적으로 **값 복사**다.

## 5. 헷갈리기 쉬운 부분

### 5.1 변수 이름과 변수 객체를 같은 것으로 보는 문제

코드에서 우리는 늘 이름으로 변수를 다루기 때문에, 이름 자체가 변수의 본질처럼 느껴질 수 있다.  
하지만 실제로 중요한 것은 이름이 아니라 **그 이름이 가리키는 메모리 객체**다.

이 차이를 놓치면 포인터, 배열, 구조체, 함수 인자 전달에서 자주 흔들린다.

### 5.2 선언과 초기화를 같은 것으로 보는 문제

입문 예제에서는 다음처럼 많이 쓴다.

    int x = 10;

그래서 선언과 초기화가 늘 함께 다니는 것처럼 보일 수 있다.  
하지만 이 둘은 다른 개념이다.

- 선언: 객체를 준비한다.
- 초기화: 처음 값을 준다.

이 구분이 명확해야 이후 `const`, 초기화되지 않은 변수, 함수 인자, 구조체 초기화 등을 더 정확히 이해할 수 있다.

### 5.3 값 전달과 주소 전달을 혼동하는 문제

다음 같은 코드를 보면:

    int a = 10;
    int b = a;

많은 초보자가 “b가 a를 참조한다”고 막연히 생각할 수 있다.  
하지만 일반적인 변수 대입은 값을 복사하는 것이다.

즉, `a`와 `b`는 별개의 객체다.

반대로 주소를 넘기려면 포인터가 필요하다.  
이 차이를 놓치면 함수 호출에서도 계속 헷갈리게 된다.

### 5.4 lifetime과 scope를 같은 것으로 보는 문제

두 개념은 관련이 있지만 다르다.

- lifetime은 객체가 존재하는 시간이다.
- scope는 이름이 보이는 코드 범위다.

예를 들어 어떤 객체는 존재는 하지만 해당 이름으로 접근할 수 없는 상황도 있을 수 있다.  
즉, 시간과 범위를 구분해야 한다.

### 5.5 변수는 항상 안전하게 초기화되어 있다고 생각하는 문제

다음처럼 선언만 하고 값을 바로 쓰지 않으면 문제가 생길 수 있다.

    int x;
    printf("%d\n", x);

이 경우 `x`는 원하는 의미 있는 값을 가지고 있지 않을 수 있다.  
즉, “변수가 존재한다”와 “변수가 올바른 값을 가진다”는 다른 문제다.

이 점은 실무에서 매우 중요하다.

## 6. 실무 관점

### 6.1 디버깅에서는 변수의 값만 보지 않는다

실무에서 변수를 볼 때는 단순히 현재 값만 보는 것이 아니다.  
다음도 함께 본다.

- 값이 맞는가
- 주소가 맞는가
- 이 객체는 아직 유효한가
- 예상한 수명을 가지고 있는가
- 어디서 생성되고 어디서 사라지는가

즉, 디버깅에서 변수는 단순 값이 아니라 **실행 중인 객체**다.

### 6.2 함수 설계와 인자 전달을 이해하는 기반이 된다

변수를 메모리 객체로 볼 줄 알면 함수 인자 전달도 더 명확해진다.

예를 들어 값 전달은 “객체의 값이 복사되는 것”이고,  
포인터 전달은 “객체의 주소가 전달되는 것”이다.

이 차이를 정확히 이해해야 함수가 실제로 원본을 바꾸는지, 복사본만 바꾸는지 판단할 수 있다.

### 6.3 메모리 수명 문제를 더 빨리 발견할 수 있다

변수를 생명주기 관점으로 보면 다음 같은 문제를 더 빨리 감지할 수 있다.

- 함수가 끝난 뒤 지역 변수 주소를 쓰는 문제
- 이미 유효 범위가 끝난 객체에 접근하는 문제
- 초기화되지 않은 값을 사용하는 문제

즉, 변수 이해는 곧 **메모리 안전성 감각**으로 이어진다.

### 6.4 다른 언어를 볼 때도 기준점이 된다

C에서 변수를 메모리 객체로 이해하면, 다른 언어를 볼 때도 더 정확한 질문을 던질 수 있다.

- 이 언어의 변수는 값인가, 참조인가?
- 객체의 생명주기는 누가 관리하는가?
- 메모리는 자동으로 관리되는가?
- 이름과 객체의 관계는 어떻게 되는가?

즉, C의 변수 개념은 다른 언어를 더 깊게 해석하는 기준점이 된다.

## 7. 다음에 볼 것

### 7.1 함수 호출 시 스택에서는 무슨 일이 일어나는가

변수를 메모리 객체로 이해했다면, 이제 함수 호출 시 지역 변수가 어디에 놓이고 언제 사라지는지 봐야 한다.  
이 문서를 통해 lifetime과 stack 개념이 더 명확해진다.

### 7.2 포인터란 무엇인가

값과 주소를 구분할 수 있게 되었으면, 다음은 주소를 저장하는 값인 포인터로 들어갈 차례다.  
포인터는 변수 개념의 확장선 위에 있다.

### 7.3 스택과 힙의 차이

변수의 수명을 이해했다면, 자동으로 관리되는 객체와 동적으로 관리되는 객체의 차이를 보는 것이 자연스럽다.

### 7.4 문자열 리터럴과 char 배열의 차이

문자열도 결국 메모리 객체다.  
어떤 것은 수정 가능하고 어떤 것은 수정하면 안 되는 이유를 변수와 저장 위치 관점으로 이어서 볼 수 있다.

## 정리

C에서 변수는 단순한 이름표가 아니다.  
변수는 **타입, 값, 주소, 생명주기, 접근 범위**를 가진 메모리 객체다.

핵심은 다음과 같다.

- 변수는 실제 메모리 안에 존재하는 객체다.
- 이름은 객체를 식별하기 위한 수단이다.
- 값과 주소는 다르다.
- 선언과 초기화는 구분되는 개념이다.
- 변수는 lifetime과 scope를 가진다.
- 일반 대입은 기본적으로 값 복사다.

즉, C에서 변수를 제대로 이해한다는 것은  
“값을 저장하는 상자” 수준을 넘어서  
**실행 중 존재하는 메모리 객체를 읽을 수 있게 되는 것**을 의미한다.

이 관점이 잡혀야 함수 호출, 포인터, 동적 메모리, 프로그램 구조를 훨씬 안정적으로 이해할 수 있다.
