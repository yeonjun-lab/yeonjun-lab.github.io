## 1. 왜 이 주제를 공부했는가

C를 배우다 보면 어느 순간부터 메모리를 단순히 “값이 들어가는 공간”으로만 볼 수 없게 된다.  
특히 함수 호출, 지역 변수, 포인터, 문자열, 배열, 동적 메모리를 함께 보기 시작하면 곧바로 다음 질문이 생긴다.

- 지역 변수는 왜 함수가 끝나면 사라지는가?
- 어떤 데이터는 왜 직접 `free`해야 하는가?
- 포인터가 가리키는 대상은 왜 어떤 경우 안전하고 어떤 경우 위험한가?
- 큰 배열은 왜 지역 변수로 두면 문제를 만들 수 있는가?
- 어떤 객체는 자동으로 정리되고, 어떤 객체는 내가 직접 관리해야 하는가?

이 질문들을 제대로 이해하려면 **스택(stack)** 과 **힙(heap)** 을 구분해야 한다.

입문 단계에서는 보통 이렇게 외운다.

- 스택: 함수 호출과 지역 변수
- 힙: `malloc`으로 할당하는 메모리

이 설명은 방향은 맞지만 충분하지 않다.  
왜냐하면 스택과 힙의 진짜 차이는 단순히 “어디에 있느냐”보다 다음에 있기 때문이다.

- 누가 메모리를 관리하는가
- 객체의 lifetime이 어떻게 결정되는가
- 언제 자동으로 정리되는가
- 언제 직접 해제해야 하는가
- 잘못 사용하면 어떤 종류의 버그가 생기는가

즉, 스택과 힙은 단순 공간 구분이 아니라 **수명 모델과 관리 책임 모델의 차이**다.

예를 들어 다음 코드를 보자.

```c
void f(void) {
    int x = 10;
}
```

여기서 `x`는 함수가 끝나면 더 이상 유효하지 않다.  
반면 다음은 다르다.

```c
int *p = malloc(sizeof(int));
*p = 10;
```

이 경우 `p`가 가리키는 객체는 함수가 끝났다고 자동으로 사라지는 것이 아니다.  
누군가 직접 해제해야 한다.

이 차이를 모르면 C에서는 거의 반드시 수명 관련 버그를 만나게 된다.

- 지역 변수 주소 반환
- dangling pointer
- memory leak
- double free
- use-after-free
- 큰 스택 객체로 인한 문제

이 문서는 스택과 힙이 무엇인지, 왜 둘을 반드시 구분해야 하는지, 그리고 이 차이가 포인터·동적 메모리·문자열·자료구조·실무 안정성과 어떻게 연결되는지를 정리하기 위해 작성한다.

## 2. 정의

### 2.1 스택은 함수 호출과 함께 관리되는 메모리 영역이다

스택은 보통 함수 호출과 반환의 흐름에 맞춰 자동으로 관리되는 메모리 영역으로 이해할 수 있다.

일반적으로 다음이 스택과 강하게 연결된다.

- 함수 호출
- 지역 변수
- 함수 인자
- 스택 프레임
- 짧은 lifetime을 가진 객체

즉, 스택 객체는 보통 **현재 실행 흐름 안에서만 잠깐 존재하는 데이터**다.

### 2.2 힙은 동적으로 할당하고 직접 관리하는 메모리 영역이다

힙은 프로그램 실행 중 필요에 따라 할당받고, 더 이상 필요 없어지면 직접 해제해야 하는 메모리 영역이다.

C에서는 보통 다음 함수들과 연결된다.

- `malloc`
- `calloc`
- `realloc`
- `free`

즉, 힙 객체는 자동으로 생겼다가 자동으로 사라지는 것이 아니라,  
**프로그래머가 생성과 종료를 명시적으로 관리해야 하는 객체**다.

### 2.3 차이의 핵심은 공간보다 수명과 책임이다

스택과 힙을 단순히 “메모리 영역 두 종류”로만 보면 절반만 이해한 것이다.  
핵심은 다음이다.

- 스택 객체는 보통 함수 호출 구조에 따라 자동으로 lifetime이 정해진다.
- 힙 객체는 명시적 할당과 해제를 통해 lifetime이 정해진다.

즉, 차이의 본질은 위치보다 **관리 방식과 유효성**에 있다.

### 2.4 스택 변수와 힙 객체를 가리키는 포인터는 서로 다른 층이다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
```

여기서 `p` 자체는 보통 지역 변수라면 스택에 놓일 수 있다.  
하지만 `p`가 가리키는 대상 객체는 힙에 있다.

즉, 포인터 변수와 그 대상 객체는 같은 곳에 있을 필요가 없다.

이 점은 매우 중요하다.  
왜냐하면 초보자는 종종 “포인터가 힙이다”처럼 오해하기 때문이다.  
하지만 포인터는 변수고, 힙은 그 포인터가 가리키는 대상의 저장 영역일 뿐이다.

## 3. 핵심 개념

### 3.1 스택 객체는 보통 자동 수명을 가진다

다음 코드를 보자.

```c
void f(void) {
    int x = 10;
}
```

여기서 `x`는 `f`가 실행되는 동안만 유효한 지역 변수다.  
즉, `x`의 lifetime은 함수 실행과 함께 시작되고, 함수 종료와 함께 끝난다.

이런 객체는 보통 프로그래머가 직접 “해제”하지 않는다.  
실행 흐름에 따라 자동으로 정리된다.

즉, 스택 객체의 핵심은 **자동 lifetime**이다.

### 3.2 힙 객체는 명시적 lifetime을 가진다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
*p = 10;
```

이 경우 `malloc`은 힙에 정수 하나를 저장할 공간을 만든다.  
그 객체는 함수 호출이 끝났다고 해서 자동으로 사라지지 않는다.

즉, 이 객체는 다음과 같은 성질을 가진다.

- 필요할 때 생성한다.
- 명시적으로 해제하기 전까지 남아 있을 수 있다.
- 누군가 소유권을 가지고 관리해야 한다.

이것이 힙 객체의 핵심이다.

### 3.3 스택은 편하지만 lifetime이 짧다

스택은 함수 호출과 잘 맞기 때문에 매우 자연스럽다.  
지역 변수를 선언하면 바로 사용할 수 있고, 함수가 끝나면 자동으로 정리된다.

하지만 그만큼 lifetime이 짧다.  
즉, 함수 밖까지 살아 있어야 하는 데이터를 지역 변수로 두면 문제가 된다.

예를 들어:

```c
int* bad(void) {
    int x = 10;
    return &x;
}
```

이 코드는 `x`가 스택 객체이기 때문에 위험하다.  
함수가 끝나면 `x`의 lifetime이 끝나기 때문이다.

### 3.4 힙은 유연하지만 책임이 생긴다

힙은 함수의 lifetime을 넘어 살아야 하는 데이터를 만들 수 있게 해준다.  
이건 매우 강력하다.

하지만 그 대가로 책임이 생긴다.

- 언제 해제할 것인가
- 누가 해제할 것인가
- 두 번 해제하지는 않는가
- 잊어버리지는 않는가
- 해제 후 다시 쓰지는 않는가

즉, 힙은 편의가 아니라 **명시적 관리 권한과 위험**을 함께 가져온다.

### 3.5 포인터는 스택과 힙을 연결하는 매개다

포인터는 스택 객체도 가리킬 수 있고, 힙 객체도 가리킬 수 있다.  
즉, 포인터는 저장 영역의 차이를 직접 드러내지 않는다.

예를 들어 다음 두 포인터는 겉으로는 비슷하다.

```c
int x = 10;
int *p1 = &x;
```

```c
int *p2 = malloc(sizeof(int));
*p2 = 10;
```

하지만 `p1`이 가리키는 대상은 스택 객체이고,  
`p2`가 가리키는 대상은 힙 객체다.

즉, 포인터를 이해할 때는 **어디를 가리키느냐**와 **그 대상의 lifetime이 무엇이냐**를 같이 봐야 한다.

### 3.6 스택과 힙의 차이는 결국 ownership 문제로 이어진다

스택 변수는 보통 함수 실행이 끝나면 자동으로 정리되므로 ownership을 강하게 의식하지 않아도 된다.  
반면 힙 객체는 누가 관리하는지 명확하지 않으면 곧 버그로 이어진다.

즉, 힙을 사용하기 시작하는 순간부터는 다음 질문이 중요해진다.

- 이 객체의 소유자는 누구인가?
- 누가 `free`할 책임이 있는가?
- 이 포인터를 복사하면 ownership도 복사되는가?
- 함수가 반환한 힙 객체는 누가 정리하는가?

이 점에서 힙은 단순 메모리 주제가 아니라 설계 주제다.

## 4. 예시

### 4.1 스택 객체 예시

다음 코드를 보자.

```c
void f(void) {
    int x = 10;
    printf("%d\n", x);
}
```

여기서 `x`는 지역 변수다.  
즉, `f`의 실행 동안만 유효하다.

`f`가 끝나면 `x`는 더 이상 유효한 객체가 아니다.  
이것이 스택 기반 객체의 대표적인 예다.

### 4.2 힙 객체 예시

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
*p = 10;
```

이 경우 `p`는 주소를 저장하는 포인터 변수이고,  
`p`가 가리키는 정수 객체는 힙에 존재한다.

이 객체는 자동으로 사라지지 않는다.  
따라서 더 이상 필요 없으면 명시적으로 정리해야 한다.

### 4.3 포인터 변수는 스택에, 대상은 힙에 있는 예시

다음 코드를 보자.

```c
void f(void) {
    int *p = malloc(sizeof(int));
    *p = 10;
}
```

여기서 `p`는 지역 변수이므로 보통 스택 프레임 안에 존재한다.  
하지만 `p`가 가리키는 객체는 힙에 있다.

즉, 함수가 끝나면 `p`라는 변수는 사라질 수 있지만,  
힙 객체는 자동으로 정리되지 않는다.

이 구조를 이해하지 못하면 메모리 누수를 이해할 수 없다.

### 4.4 지역 변수 주소 반환 예시

다음 코드를 보자.

```c
int* bad(void) {
    int x = 10;
    return &x;
}
```

이 코드는 위험하다.  
`x`는 스택 객체이고 함수 종료와 함께 lifetime이 끝난다.  
즉, 반환된 포인터는 더 이상 유효한 객체를 가리키지 않게 된다.

### 4.5 힙 객체 반환 예시

다음 코드는 구조적으로 다르다.

```c
int* good(void) {
    int *p = malloc(sizeof(int));
    *p = 10;
    return p;
}
```

이 경우 반환되는 것은 힙 객체의 주소다.  
함수가 끝나도 힙 객체는 남아 있을 수 있다.

즉, 함수 종료 후에도 살아 있어야 하는 데이터를 만들고 반환하려면  
스택이 아니라 힙이 필요할 수 있다.

단, 이 경우에도 누군가 나중에 `free`해야 한다는 책임이 생긴다.

## 5. 헷갈리기 쉬운 부분

### 5.1 스택이 “좋은 메모리”, 힙이 “나쁜 메모리”라고 단순화하는 문제

초보자는 종종 이렇게 외운다.

- 스택은 빠르다
- 힙은 느리다
- 그러니 스택이 무조건 더 좋다

이건 지나치게 단순화된 이해다.  
중요한 것은 속도보다 **적절한 수명과 관리 방식**이다.

짧게 살아야 하는 작은 데이터는 스택이 자연스럽고,  
함수 밖까지 살아야 하거나 크기가 유연해야 하는 데이터는 힙이 더 적절할 수 있다.

즉, 스택과 힙은 우열이 아니라 용도가 다르다.

### 5.2 포인터 자체가 힙이라고 생각하는 문제

다음 코드에서:

```c
int *p = malloc(sizeof(int));
```

초보자는 `p`가 힙이라고 착각하기 쉽다.  
하지만 `p`는 보통 지역 변수라면 스택에 존재한다.  
힙에 있는 것은 `p`가 가리키는 대상 객체다.

즉, 포인터 변수와 포인터 대상 객체를 구분해야 한다.

### 5.3 힙 객체는 자동으로 정리된다고 오해하는 문제

C에서는 일반적으로 힙 객체를 자동으로 회수해주는 고수준 가비지 컬렉션이 없다.  
따라서 할당한 힙 객체는 명시적으로 해제해야 한다.

이 점을 놓치면 memory leak가 생긴다.

### 5.4 함수가 끝나면 포인터가 가리키는 힙 객체도 사라진다고 생각하는 문제

다음 코드를 보자.

```c
void f(void) {
    int *p = malloc(sizeof(int));
}
```

함수가 끝나면 `p`라는 지역 변수는 사라질 수 있다.  
하지만 힙 객체는 자동으로 사라지지 않는다.

즉, 포인터 변수의 lifetime과 포인터 대상 객체의 lifetime은 다를 수 있다.

### 5.5 큰 지역 배열을 무심코 스택에 두는 문제

다음 같은 코드는 주의가 필요하다.

```c
void f(void) {
    int big[100000];
}
```

이건 지역 배열이므로 보통 스택을 사용한다.  
너무 큰 객체를 스택에 올리면 문제를 만들 수 있다.

즉, 데이터 크기와 lifetime을 같이 보고 저장 위치를 결정해야 한다.

## 6. 어디서부터 저장 위치, 객체 수명, 관리 책임이 갈리는가

### 6.1 스택과 힙 차이의 핵심은 주소 영역이 아니라 수명 모델이다

초보자는 스택과 힙을 메모리 위치 두 개로 외우기 쉽다.  
하지만 실무에서 더 중요한 것은  
객체가 언제 생기고 누가 언제 정리하느냐다.

즉, 스택/힙 구분은 위치 그림보다  
수명과 관리 모델 차이로 이해해야 한다.

### 6.2 포인터 변수 위치와 대상 객체 위치는 서로 다를 수 있다

포인터가 힙 객체를 가리킨다고 해서  
포인터 변수 자신도 힙에 있어야 하는 것은 아니다.  
반대로 스택 객체를 가리키는 포인터가 힙 구조체 안에 있을 수도 있다.

즉, 포인터가 어디에 있느냐와  
포인터가 가리키는 객체가 어디에 있느냐는 별도 문제다.

### 6.3 자동 관리와 수동 관리는 성능 취향이 아니라 책임 차이다

스택 쪽은 보통 호출 구조에 따라 자동으로 정리되지만,  
힙은 명시적 할당/해제와 ownership 설계가 필요하다.

즉, 힙을 쓴다는 것은 단순히 더 유연한 메모리를 쓰는 것이 아니라  
수명 종료 책임을 직접 지는 선택이다.

### 6.4 큰 객체와 긴 수명 객체를 같은 기준으로 판단하면 안 된다

어떤 데이터는 크기 때문에 힙이 필요하고,  
어떤 데이터는 수명 때문에 힙이 필요하다.  
둘은 겹칠 수 있지만 같은 질문은 아니다.

즉, 스택/힙 선택에서는  
크기와 lifetime 요구를 따로 봐야 한다.

### 6.5 스택과 힙 차이는 결국 ownership 설계로 이어진다

힙 객체를 쓰기 시작하면  
누가 만들고, 누가 넘기고, 누가 해제하는지 정해야 한다.  
즉, 이 문서는 메모리 위치 소개에서 끝나지 않고  
API 설계와 ownership 규칙으로 이어져야 한다.

---

## 7. 실무 관점

### 7.1 수명 설계가 먼저다

실무에서 가장 먼저 해야 할 질문은 “이 데이터가 얼마나 오래 살아야 하는가”다.

- 함수 안에서만 잠깐 쓰는가
- 호출한 쪽에 결과를 넘겨야 하는가
- 여러 함수와 구조체가 공유해야 하는가
- 프로그램 전반에서 오래 유지해야 하는가

이 질문에 따라 스택을 쓸지, 힙을 쓸지, 다른 저장 전략을 쓸지가 정해진다.

즉, 저장 위치 결정은 성능보다 먼저 **수명 설계**다.

### 7.2 힙을 쓰면 ownership 규칙이 필요하다

힙 객체는 누군가가 책임지고 해제해야 한다.  
그래서 실무에서는 ownership 규칙이 중요하다.

예를 들어 다음을 명확히 해야 한다.

- 이 함수가 반환한 포인터는 누가 해제하는가
- 이 구조체가 내부 버퍼를 소유하는가
- 복사 시 얕은 복사인지 깊은 복사인지
- 실패 시 중간에 할당한 메모리는 누가 정리하는가

즉, 힙은 코드 몇 줄의 문제가 아니라 API와 설계의 문제다.

### 7.3 문자열과 버퍼 처리에서 매우 자주 등장한다

실무에서 문자열 버퍼, 파일 읽기 버퍼, 네트워크 패킷 버퍼, 동적 배열 같은 것들은 스택과 힙 선택이 매우 중요하다.

- 짧고 고정 크기면 스택
- 크기가 유동적이거나 오래 살아야 하면 힙

이런 판단이 자연스러워져야 안전한 C 코드를 쓸 수 있다.

### 7.4 메모리 오류 대부분은 lifetime 관리 실패와 연결된다

다음 오류들은 대부분 스택/힙과 lifetime 문제에서 출발한다.

- dangling pointer
- use-after-free
- memory leak
- double free
- stack overflow 성격 문제
- 잘못된 버퍼 경계 접근

즉, 스택과 힙 차이를 이해하는 것은 단순 개념 이해가 아니라  
실무 버그를 예방하는 핵심 능력이다.

---

## 8. 판단 체크리스트

- 스택/힙 차이를 단순 메모리 위치가 아니라 수명 모델 차이로 보고 있는가
- 포인터 변수 위치와 대상 객체 위치를 분리해서 설명할 수 있는가
- 힙 사용을 자동 관리가 없는 수명 책임 선택으로 이해하고 있는가
- 큰 객체와 긴 수명 객체 문제를 같은 기준으로 뭉뚱그리지 않고 있는가
- 스택/힙 선택이 결국 ownership 설계 문제로 이어진다는 점을 보고 있는가

## 정리

스택과 힙의 차이는 단순히 메모리 위치 차이가 아니다.  
핵심은 **객체의 lifetime과 관리 책임의 차이**다.

정리하면 다음과 같다.

- 스택 객체는 보통 함수 호출 구조에 따라 자동으로 관리된다.
- 힙 객체는 명시적으로 할당하고 해제해야 한다.
- 스택은 짧은 lifetime의 지역 객체와 잘 맞는다.
- 힙은 유연한 수명과 크기를 제공하지만 관리 책임이 따른다.
- 포인터 변수와 포인터가 가리키는 대상 객체는 서로 다른 저장 영역에 있을 수 있다.
- 스택/힙 구분은 결국 ownership, 수명, 안전성 문제로 이어진다.

즉, 스택과 힙을 제대로 이해한다는 것은  
“메모리 종류 두 개를 외운다”가 아니라  
**객체가 언제 생성되고 언제까지 유효하며 누가 책임지고 관리하는지 이해한다**는 뜻이다.

이 관점이 잡혀야 `malloc`, `free`, dangling pointer, memory leak, 문자열 버퍼, 자료구조 설계를 훨씬 안정적으로 이해할 수 있다.
