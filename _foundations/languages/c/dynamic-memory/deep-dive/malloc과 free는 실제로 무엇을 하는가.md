## 1. 왜 이 주제를 깊게 봐야 하는가

C에서 동적 메모리를 본격적으로 다루기 시작하면 가장 먼저 만나게 되는 함수가 `malloc`과 `free`다.  
입문 단계에서는 보통 이렇게 외운다.

- `malloc`은 메모리를 할당한다.
- `free`는 메모리를 해제한다.

이 설명은 틀리지 않다.  
하지만 이 수준에서 멈추면 실무에서 거의 반드시 문제가 생긴다.  
왜냐하면 `malloc`과 `free`는 단순한 함수 사용법이 아니라 **객체의 lifetime, ownership, 유효성, 책임 분배**와 직결되기 때문이다.

예를 들어 다음과 같은 문제는 대부분 여기서 시작된다.

- memory leak
- dangling pointer
- use-after-free
- double free
- null 반환값 처리 누락
- 해제 책임 불명확
- 얕은 복사로 인한 이중 해제

즉, `malloc`과 `free`를 제대로 이해하지 못하면 C에서 동적 메모리를 “쓸 수는 있어도 안전하게 설계할 수는 없는 상태”가 된다.

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
*p = 10;
free(p);
```

겉보기에는 아주 단순하다.  
하지만 실제로는 다음 질문을 함께 봐야 한다.

- `malloc`이 돌려준 것은 정확히 무엇인가?
- `p`와 `*p`는 각각 무엇인가?
- `free(p)` 이후 `p`는 어떤 상태인가?
- `free`가 포인터 변수 자체를 지우는가?
- `free`를 안 하면 무슨 일이 생기는가?
- `free`를 두 번 하면 왜 위험한가?

즉, `malloc`과 `free`는 “메모리 공간 빌리고 반납하기” 정도로 이해하면 부족하다.  
더 정확히는 **힙 위의 객체를 만들고, 그 객체의 lifetime을 끝내는 행위**로 봐야 한다.

이 문서는 `malloc`과 `free`가 실제로 무엇을 하는지, 포인터와 힙 객체의 관계가 무엇인지, 해제 이후 왜 문제가 생기는지, 그리고 실무에서 어떤 판단 기준을 가져야 하는지 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

`malloc`과 `free`의 전체 구조를 먼저 잡으면 다음과 같다.

### 2.1 `malloc`은 힙에 객체를 둘 공간을 요청한다

`malloc`은 지정한 크기만큼의 메모리 공간을 힙에서 확보하려고 시도한다.

예를 들어:

```c
int *p = malloc(sizeof(int));
```

이 코드는 `int` 하나를 저장할 수 있는 크기의 공간을 힙에서 요청한다.

즉, `malloc`은 값을 만드는 것이 아니라 **저장 공간을 확보하는 것**이다.

### 2.2 `malloc`의 반환값은 힙 객체의 시작 주소다

`malloc`이 성공하면, 확보한 메모리 공간의 시작 위치를 나타내는 주소를 돌려준다.

즉,

- `p`는 포인터 변수다.
- `p` 안에는 힙 객체의 시작 주소가 들어간다.
- `*p`를 통해 그 공간을 `int`처럼 사용할 수 있다.

즉, `malloc`이 돌려주는 것은 “정수 객체”가 아니라 **정수 객체처럼 사용할 수 있는 힙 공간의 주소**다.

### 2.3 `free`는 힙 객체의 lifetime을 종료한다

`free`는 전달된 포인터가 가리키는 힙 메모리 블록을 더 이상 사용하지 않겠다고 해제하는 함수다.

예를 들어:

```c
free(p);
```

이 코드는 `p`가 가리키던 힙 객체를 해제한다.

중요한 점은 `free`가 **포인터 변수 `p`를 없애는 것이 아니라**,  
`p`가 가리키는 힙 객체의 lifetime을 끝낸다는 점이다.

### 2.4 `free` 이후 포인터는 자동으로 안전해지지 않는다

이건 매우 중요하다.

```c
free(p);
```

이후에도 `p`라는 변수 자체는 남아 있을 수 있다.  
하지만 `p`가 가리키던 힙 객체는 더 이상 유효하지 않다.

즉, `p`는 해제된 객체를 가리키는 포인터가 되어 버릴 수 있다.  
이런 포인터를 dangling pointer라고 한다.

### 2.5 핵심은 allocation보다 ownership이다

처음 배우는 사람은 보통 `malloc`의 문법과 `free` 호출 타이밍만 보지만,  
실무에서는 더 중요한 것이 있다.

- 이 객체는 누가 소유하는가?
- 누가 `free`할 책임이 있는가?
- 이 포인터를 다른 곳에 넘기면 ownership도 같이 넘기는가?
- 여러 포인터가 같은 힙 객체를 가리킬 때 해제 책임은 누가 지는가?

즉, `malloc`과 `free`는 단순 함수 호출이 아니라 **메모리 ownership 모델**과 연결된다.

## 3. 내부 동작 원리

### 3.1 `malloc`은 원하는 크기의 바이트 수를 요청한다

`malloc`은 타입을 직접 알지 못한다.  
그냥 몇 바이트가 필요한지만 안다.

예를 들어:

```c
int *p = malloc(sizeof(int));
```

여기서 `malloc`이 보는 것은 “정수 하나”가 아니라 `sizeof(int)` 바이트 수다.

즉, `malloc`은 “타입 기반 할당 함수”가 아니라 **바이트 수 기반 할당 함수**다.

### 3.2 반환된 주소는 아직 값이 정해진 객체가 아닐 수도 있다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
```

이 시점에서 `p`는 `int` 하나를 담을 수 있는 힙 공간의 주소를 가리킬 수 있다.  
하지만 그 안의 값은 자동으로 원하는 값으로 채워지지 않는다.

즉, `malloc`이 해주는 것은 “공간 제공”이지 “의미 있는 초기화”가 아니다.

그래서 보통 다음처럼 값을 넣는다.

```c
*p = 10;
```

즉, 할당과 초기화는 구분해야 한다.

### 3.3 포인터 변수와 힙 객체는 별개다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
```

여기서 `p`는 변수다.  
그리고 `p`가 가리키는 대상은 힙 객체다.

즉, 두 층이 있다.

- `p` 자체: 포인터 변수
- `*p`가 접근하는 대상: 힙 객체

이 구분이 중요한 이유는 `free(p)`가 무엇을 하는지를 정확히 이해하게 해주기 때문이다.

### 3.4 `free(p)`는 힙 객체를 해제하지 포인터 변수 `p`를 없애지 않는다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
*p = 10;
free(p);
```

이때 `free(p)`는 `p` 자체를 없애는 것이 아니다.  
`p`는 여전히 존재할 수 있다.  
다만 `p`가 가리키던 힙 객체가 더 이상 유효하지 않게 된다.

즉, `free` 이후의 핵심 문제는 다음이다.

- 포인터 변수는 남아 있다.
- 하지만 그 포인터가 가리키는 대상은 더 이상 유효하지 않다.

그래서 이후 `*p`를 사용하면 위험하다.

### 3.5 dangling pointer는 “포인터는 남았는데 대상이 사라진 상태”다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
*p = 10;
free(p);
printf("%d\n", *p);
```

이 코드는 위험하다.  
왜냐하면 `free(p)` 이후 `p`는 더 이상 유효한 객체를 가리키지 않기 때문이다.

즉, dangling pointer는 다음 상태다.

- 포인터 변수는 존재
- 주소값도 남아 있을 수 있음
- 하지만 그 주소를 유효한 객체 주소로 보면 안 됨

이게 use-after-free 문제의 핵심이다.

### 3.6 memory leak는 힙 객체를 잃어버린 상태다

다음 코드를 보자.

```c
int *p = malloc(sizeof(int));
p = NULL;
```

이 경우 처음 할당한 힙 객체 주소를 잃어버릴 수 있다.  
즉, 해제할 방법이 사라진다.

이것이 memory leak의 본질이다.

- 힙 객체는 남아 있다.
- 하지만 더 이상 접근할 수 없다.
- 따라서 해제할 수도 없다.

즉, leak는 “메모리가 많이 늘어난다”보다 먼저  
**소유한 힙 객체를 회수할 경로를 잃어버린 상태**다.

### 3.7 double free는 같은 힙 객체를 두 번 종료하려는 시도다

다음은 매우 위험하다.

```c
int *p = malloc(sizeof(int));
free(p);
free(p);
```

이미 lifetime이 끝난 힙 객체에 대해 다시 `free`를 시도하는 것이다.  
이건 단순히 “한 번 더 청소한다”가 아니라 프로그램 상태를 깨뜨릴 수 있는 심각한 오류다.

즉, 해제는 한 객체에 대해 정확히 한 번이어야 한다.

## 4. 핵심 구성 요소

### 4.1 heap allocation

heap allocation은 실행 중 동적으로 힙 메모리를 확보하는 것이다.  
즉, 컴파일 시점이 아니라 런타임에 필요한 크기만큼 공간을 얻는 방식이다.

### 4.2 ownership

ownership은 어떤 코드나 어떤 객체가 해당 힙 메모리를 해제할 책임이 있는지를 뜻한다.

C에서 동적 메모리 문제의 중심은 거의 항상 ownership이다.  
즉, ownership이 흐리면 leak나 double free가 생기기 쉽다.

### 4.3 allocated object

allocated object는 `malloc` 등으로 할당된 힙 객체다.  
포인터는 이 객체의 주소를 저장하고 있을 뿐이다.

즉, 포인터와 allocated object를 구분해서 생각해야 한다.

### 4.4 dangling pointer

dangling pointer는 더 이상 유효하지 않은 객체를 가리키는 포인터다.  
대표적으로 해제된 힙 객체를 계속 가리키는 경우가 여기에 해당한다.

### 4.5 memory leak

memory leak는 할당한 힙 객체를 해제하지 못하고 잃어버리는 상태다.  
즉, lifetime 종료 책임을 이행하지 못한 상태다.

### 4.6 null check

`malloc`은 항상 성공한다고 가정하면 안 된다.  
필요한 메모리를 확보하지 못하면 실패할 수 있다.  
따라서 반환값이 유효한지 확인하는 습관이 중요하다.

예를 들어:

```c
int *p = malloc(sizeof(int));
if (p == NULL) {
    return;
}
```

이런 구조는 매우 기본적이지만 중요하다.

## 5. 실행 흐름 또는 처리 순서

다음 코드를 기준으로 흐름을 보자.

```c
int *p = malloc(sizeof(int));
if (p == NULL) {
    return;
}
*p = 10;
free(p);
p = NULL;
```

### 5.1 할당 요청

먼저 `malloc(sizeof(int))`가 호출된다.

이 의미는 다음과 같다.

- `int` 하나를 저장할 수 있는 바이트 수를 계산한다.
- 그만큼 힙 공간을 요청한다.

### 5.2 반환값 확인

`malloc`이 성공하면 유효한 주소를 반환할 수 있다.  
실패하면 유효한 힙 객체를 얻지 못한다.

그래서 다음을 확인한다.

```c
if (p == NULL) {
    return;
}
```

즉, 포인터가 실제로 유효한 객체를 가리키는지 먼저 본다.

### 5.3 객체 사용

그 다음 공간을 의미 있는 객체처럼 사용한다.

```c
*p = 10;
```

이제 비로소 확보한 힙 공간을 `int` 객체처럼 다루는 것이다.

즉, 할당 후 초기화/사용 단계가 이어진다.

### 5.4 해제

더 이상 필요 없으면 해제한다.

```c
free(p);
```

이 시점에서 `p`가 가리키던 힙 객체의 lifetime은 종료된다.

### 5.5 후처리

해제 후 같은 포인터를 다시 쓰지 않도록 다음처럼 정리하는 습관을 둘 수 있다.

```c
p = NULL;
```

이렇게 하면 적어도 “아무것도 가리키지 않는다”는 상태를 표현할 수 있다.  
즉, dangling pointer를 눈에 보이게 줄이는 데 도움이 된다.

## 6. 성능 / 최적화 관점

### 6.1 동적 할당은 공짜가 아니다

`malloc`과 `free`는 매우 편리하지만 비용이 있다.  
실행 중 힙 관리자를 통해 적절한 공간을 찾고, 상태를 갱신하고, 해제 정보를 관리해야 하기 때문이다.

즉, 아주 짧은 시간에 반복되는 작은 할당/해제는 성능 비용이 될 수 있다.

### 6.2 잦은 할당은 단편화와 관리 비용을 만든다

동적 메모리를 자주 쓰고 자주 해제하면 힙 메모리 사용 패턴이 복잡해질 수 있다.  
이건 단순한 속도 문제를 넘어서 메모리 사용 효율에도 영향을 줄 수 있다.

즉, 필요한 만큼 유연하게 쓴다는 장점 뒤에는 관리 비용이 있다.

### 6.3 가능한 경우 스택과 재사용 전략을 고려할 수 있다

모든 데이터를 무조건 힙에 둘 필요는 없다.  
짧은 lifetime과 작은 크기라면 스택이 더 자연스러울 수 있다.

또는 반복적으로 같은 크기의 버퍼를 자주 만든다면,  
매번 `malloc`/`free`하기보다 재사용 전략이 더 적절할 수 있다.

즉, 동적 할당은 강력하지만 남용하면 비효율적일 수 있다.

### 6.4 안전성이 성능보다 먼저다

C에서 메모리 최적화를 이유로 해제 규칙을 흐리거나,  
null 검사와 ownership 규칙을 생략하면 결국 더 큰 비용으로 돌아온다.

즉, 성능 최적화 이전에 먼저 지켜야 할 것은  
**정확한 lifetime 관리와 안전한 ownership 규칙**이다.

## 7. 어디서부터 포인터 변수, 힙 객체, allocator 정책이 갈리는가

### 7.1 `malloc`이 포인터를 만드는 것이 아니라 힙 객체 시작 위치를 돌려주는 것이다

처음 배우는 사람은 `malloc`을 보면 포인터를 생성한다고 느끼기 쉽다.  
하지만 실제로는 힙 공간을 확보하고 그 시작 주소를 반환할 뿐이다.

즉, 포인터 변수 자체와  
힙 객체 생성은 분리해서 봐야 한다.

### 7.2 할당과 초기화는 전혀 다른 단계다

메모리를 받았다고 해서  
그 안에 이미 의미 있는 값이 들어 있는 것은 아니다.  
`malloc`은 바이트 공간 확보이고, 초기화는 별도 작업이다.

즉, “공간이 생겼다”와 “바로 써도 되는 값 상태다”는 다르다.

### 7.3 `free`는 포인터 값을 지우는 것이 아니라 객체 lifetime을 끝낸다

`free(p)`를 호출해도  
포인터 변수 `p` 자체가 사라지는 것은 아니다.  
끝나는 것은 그 포인터가 가리키던 힙 객체의 lifetime이다.

즉, free를 변수 초기화나 null 대입과 같은 층으로 보면 안 된다.

### 7.4 allocator 내부 정책과 사용자 코드 계약은 분리해서 봐야 한다

할당기가 어떤 free list를 쓰고, 언제 시스템에 메모리를 요청하고,  
어떻게 병합하는지는 구현 정책이다.  
반면 사용자 코드가 지켜야 하는 것은 ownership, 정확한 해제 시점, use-after-free 금지 같은 계약이다.

즉, 내부 메커니즘을 안다고 해서  
사용자 계약이 완화되는 것은 아니다.

### 7.5 동적 메모리 문제의 본질은 API 호출보다 ownership 설계다

`malloc`/`free` 자체는 짧은 함수 호출처럼 보인다.  
하지만 실제 난점은 누가 소유하고, 누가 넘기고, 누가 정확히 종료시키는가에 있다.

즉, 힙 메모리 관리는 호출 문법보다  
객체 lifetime 설계 문제로 읽어야 한다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 누가 free할지 먼저 정해야 한다

힙 객체를 만들었다면 가장 먼저 정해야 하는 것은 “누가 해제하는가”다.  
이게 불명확하면 거의 반드시 문제 생긴다.

예를 들어 함수가 포인터를 반환한다면 다음이 명확해야 한다.

- 호출자가 해제하는가
- 내부에서 계속 소유하는가
- 공유 소유가 허용되는가

즉, `malloc`을 쓴 순간 ownership 문서화가 필요하다.

### 8.2 반환값을 바로 덮어쓰면 안 된다

다음은 매우 위험하다.

```c
p = malloc(sizeof(int));
p = NULL;
```

혹은 다른 주소로 덮어쓰는 경우도 마찬가지다.  
기존 힙 객체 주소를 잃어버릴 수 있기 때문이다.

즉, 힙 포인터를 덮어쓸 때는 항상 기존 ownership이 어떻게 끝나는지 먼저 생각해야 한다.

### 8.3 free 이후에는 해당 객체를 다시 쓰면 안 된다

다음은 금지해야 할 패턴이다.

```c
free(p);
*p = 20;
```

혹은:

```c
free(p);
printf("%d\n", *p);
```

즉, 해제 후에는 포인터가 남아 있더라도 그 대상은 더 이상 유효하지 않다.

### 8.4 해제는 정확히 한 번이어야 한다

해제 누락도 문제고, 두 번 해제도 문제다.  
즉, 힙 객체 lifetime 종료는 정확히 한 번 이루어져야 한다.

이를 위해 다음이 필요하다.

- 소유자 명확화
- 포인터 복사 시 ownership 정책 명확화
- 해제 후 상태 정리

### 8.5 null 처리는 단순 예의가 아니라 방어 코드다

`malloc` 반환값을 무조건 믿는 습관은 위험하다.  
특히 시스템 코드, 대용량 처리, 라이브러리 코드에서는 실패 가능성을 고려해야 한다.

즉, null 체크는 불필요한 장식이 아니라 실제 방어 코드다.

---

## 9. 판단 체크리스트

- `malloc`이 포인터 변수를 만드는 것이 아니라 힙 객체 시작 주소를 돌려준다는 점을 이해하고 있는가
- 할당과 초기화를 서로 다른 단계로 구분하고 있는가
- `free`를 포인터 값 제거가 아니라 객체 lifetime 종료로 보고 있는가
- allocator 내부 정책과 사용자 코드의 ownership 계약을 섞어 보고 있지 않은가
- 동적 메모리 문제를 함수 사용법보다 ownership 설계 문제로 읽고 있는가

---

## 10. 더 깊게 볼 포인트

### 10.1 `calloc`

메모리 공간을 확보하면서 초기화까지 함께 다루는 경우와 연결된다.  
`malloc`과의 차이를 보면 “할당”과 “초기화”를 더 명확히 구분할 수 있다.

### 10.2 `realloc`

기존 힙 객체 크기를 조정하는 주제다.  
포인터 갱신과 실패 처리 때문에 단순해 보이지만 매우 중요한 주제다.

### 10.3 arena / pool allocation

잦은 `malloc`/`free` 비용을 줄이기 위한 설계 전략과 연결된다.  
성능 최적화와 메모리 패턴 제어에 중요하다.

### 10.4 reference counting, RAII와의 비교

C 자체는 ownership을 자동으로 관리하지 않지만,  
다른 언어/기법과 비교해 보면 C의 동적 메모리 설계 부담이 왜 큰지 더 명확해진다.

### 10.5 구조체와 동적 메모리

구조체 필드 안에 포인터가 들어가기 시작하면 얕은 복사, 깊은 복사, 소유권 이동이 더 중요해진다.  
실무에서는 매우 자주 만나는 확장 주제다.

---

## 정리

`malloc`과 `free`는 단순한 함수 쌍이 아니다.  
이들은 **힙 객체의 생성과 lifetime 종료를 관리하는 핵심 도구**다.

핵심은 다음과 같다.

- `malloc`은 바이트 수만큼 힙 공간을 요청한다.
- 반환값은 그 공간의 시작 주소다.
- 할당과 초기화는 다른 단계다.
- `free`는 포인터 변수 자체가 아니라 포인터가 가리키는 힙 객체를 해제한다.
- 해제 후 포인터는 dangling pointer가 될 수 있다.
- 해제하지 못하면 memory leak가 생긴다.
- 두 번 해제하면 double free 문제가 생긴다.
- 결국 가장 중요한 것은 ownership과 lifetime 규칙이다.

즉, `malloc`과 `free`를 제대로 이해한다는 것은  
“메모리 할당 함수를 쓸 줄 안다”가 아니라  
**힙 객체를 만들고, 안전하게 사용하고, 정확한 시점에 올바른 책임 주체가 lifetime을 종료시킬 수 있다**는 뜻이다.

이 관점이 잡혀야 동적 버퍼, 문자열 복사, 자료구조 구현, 시스템 코드, 실무 C API 설계를 훨씬 안정적으로 다룰 수 있다.
