## 1. 왜 이 주제를 깊게 봐야 하는가

C를 배우며 struct layout, padding, packed struct, endianness까지 오면  
아주 흔한 유혹 하나가 생긴다.

- 구조체를 그대로 파일에 쓰면 되지 않나
- 메모리 덩어리를 그대로 네트워크로 보내면 되지 않나
- `memcpy` 한 번이면 직렬화 끝 아닌가

겉보기에는 매우 자연스럽다.  
예를 들어 다음처럼 생각하기 쉽다.

```c
typedef struct {
    int id;
    short level;
    char flag;
} Record;
```

그리고 이렇게 하고 싶어진다.

```c
fwrite(&record, sizeof(record), 1, fp);
```

혹은:

```c
send(sock, &record, sizeof(record), 0);
```

하지만 이 방식은 아주 자주 틀린 전제 위에 서 있다.  
왜냐하면 **메모리 안의 표현**과 **외부 포맷으로서의 표현**은 같은 것이 아니기 때문이다.

문제는 여러 층에서 생긴다.

- padding 때문에 구조체 안에 의미 없는 빈 공간이 들어갈 수 있다
- endianness 때문에 다른 시스템에서 바이트 순서가 다르게 해석될 수 있다
- 타입 크기가 플랫폼마다 다를 수 있다
- 포인터 필드는 값이 아니라 주소이므로 외부에 그대로 내보낼 수 없다
- packed struct를 써도 endian, ABI, 의미 해석 문제는 남는다
- 미래 버전 호환성까지 고려하면 raw dump는 더 위험해진다

즉, 직렬화(serialization)는 단순 메모리 복사가 아니라  
**내부 표현을 외부 교환용 표현으로 변환하는 과정**이다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- “메모리에 있다”와 “외부로 저장/전송 가능하다”를 구분하게 해 준다
- struct layout, padding, endianness를 하나의 실전 문제로 묶어 준다
- 파일 포맷과 네트워크 프로토콜 설계의 핵심 감각을 준다
- raw memory dump가 왜 자주 비이식적이고 위험한지 이해하게 해 준다
- 이후 바이너리 포맷 설계, FFI, RPC, 프로토콜 구현으로 자연스럽게 이어진다

이 문서는 직렬화와 역직렬화가 정확히 무엇인지,  
왜 단순 `memcpy`나 raw dump로 끝나지 않는지,  
어떤 정보가 외부 포맷에 포함되어야 하는지,  
그리고 실무에서 어떻게 사고해야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

직렬화/역직렬화를 큰 구조로 보면 다음과 같다.

### 2.1 직렬화는 내부 객체를 외부 표현으로 바꾸는 과정이다

직렬화는 메모리 안 객체를 파일, 네트워크, 공유 메모리, IPC 메시지 같은  
외부 교환 형태로 변환하는 과정이다.

즉, 핵심은 **내부 객체 → 외부 바이트 표현**이다.

### 2.2 역직렬화는 외부 표현을 내부 객체로 해석하는 과정이다

역직렬화는 반대다.

- 파일에서 읽은 바이트
- 네트워크에서 받은 바이트
- 다른 프로세스가 보낸 메시지

이런 외부 데이터를 내부 구조체나 값으로 다시 구성한다.

즉, 핵심은 **외부 바이트 표현 → 내부 객체**다.

### 2.3 내부 표현과 외부 표현은 목적이 다르다

내부 표현은 보통 현재 프로세스에서 빠르고 편하게 다루기 위한 것이다.  
외부 표현은 시스템 간 교환과 장기 저장을 위해 안정적이고 명시적이어야 한다.

즉, 내부 구조체와 외부 포맷은 같을 수도 있지만,  
같다고 가정하면 위험하다.

### 2.4 raw memory copy는 내부 표현을 그대로 내보내는 방식이다

`memcpy`, `fwrite(&obj, sizeof obj, 1, fp)` 같은 방식은  
현재 메모리 배치를 그대로 외부에 복사한다.

문제는 이 배치가 다음에 의존한다는 점이다.

- padding
- alignment
- endianness
- 타입 크기
- ABI
- 컴파일러/플랫폼

즉, raw copy는 현재 환경에는 맞아도  
외부 포맷으로는 부적절할 수 있다.

## 3. 내부 동작 원리

### 3.1 메모리 안 구조체는 외부 포맷 명세가 아니다

다음 구조체를 보자.

```c
typedef struct {
    int id;
    short level;
    char flag;
} Record;
```

이 구조체는 내부 계산과 로직에 쓰기 좋게 정의한 것이다.  
하지만 이것만으로는 외부 포맷 명세가 되지 않는다.

왜냐하면 외부 포맷은 보통 다음을 명확히 해야 하기 때문이다.

- `id`는 몇 바이트인가
- byte order는 무엇인가
- padding 바이트는 있는가
- 버전이 바뀌면 어떻게 할 것인가

즉, C struct 선언은 곧바로 안정적인 wire format 정의가 아니다.

### 3.2 raw copy는 padding까지 함께 복사한다

예를 들어 구조체에 padding이 있으면:

- 의미 있는 필드 값뿐 아니라
- 중간/끝 padding 바이트도 함께 복사될 수 있다

즉, 외부 파일에 저장된 바이트가 필드 의미만 담는다고 보장할 수 없다.

padding은 포맷 관점에서는 잡음이 될 수 있다.

### 3.3 endianness 때문에 같은 값도 바이트열이 달라질 수 있다

예를 들어 `id = 0x12345678`이라고 하자.  
내부 메모리 바이트 순서는 host system의 endianness를 따른다.

즉, raw copy로 파일에 쓴 바이트를 다른 시스템이 읽으면  
같은 정수로 해석되지 않을 수 있다.

그래서 직렬화는 값의 의미를 보존해야지,  
현재 머신 메모리 모양을 그대로 복사하는 것으로 끝나면 안 된다.

### 3.4 포인터 필드는 직렬화 대상이 아니다

다음 구조체를 보자.

```c
typedef struct {
    int id;
    char *name;
} User;
```

이 구조체를 raw copy하면 `name` 포인터 값도 함께 나간다.  
하지만 그 값은 현재 프로세스 주소 공간 안의 주소일 뿐이다.

즉, 파일이나 네트워크로 내보내도 다른 쪽에서는 아무 의미가 없다.

이건 직렬화가 “구조체 바이트 복사”가 아니라  
**의미 있는 데이터 추출**이어야 한다는 점을 아주 명확히 보여 준다.

### 3.5 역직렬화는 단순 덮어쓰기보다 검증이 필요하다

외부 데이터를 읽을 때도 단순히 구조체 메모리에 덮어쓰면 위험할 수 있다.

예를 들어:

- 길이 검증이 없을 수 있다
- 버전 불일치가 있을 수 있다
- 잘못된 endian일 수 있다
- 공격자가 조작한 데이터일 수 있다

즉, 역직렬화는 단순 읽기가 아니라  
**유효성 검사 + 의미 해석 + 내부 표현 구성** 과정이다.

### 3.6 명시적 포맷이 있어야 안정적인 교환이 가능하다

좋은 직렬화는 보통 외부 포맷을 명시한다.

예를 들어:

- 32비트 unsigned integer, little-endian
- 문자열 길이 16비트 unsigned, big-endian
- flag는 1바이트
- reserved bytes는 0으로 채움

즉, 외부 표현은 “현재 C struct가 어쩌다 그렇게 생겼다”가 아니라  
**독립적인 데이터 계약**이어야 한다.

## 4. 핵심 구성 요소

### 4.1 serialization

내부 객체를 외부 교환용 바이트 표현으로 바꾸는 과정이다.

### 4.2 deserialization

외부 바이트 표현을 읽어 내부 객체로 재구성하는 과정이다.

### 4.3 internal representation

현재 프로그램 안에서 사용하기 위한 표현이다.  
성능과 로직 편의에 최적화될 수 있다.

### 4.4 external representation

파일, 네트워크, IPC 등 외부 경계에서 쓰는 표현이다.  
명시적이고 이식 가능해야 한다.

### 4.5 wire format / file format

외부 표현의 구체 명세다.  
필드 크기, 순서, endian, 버전, 길이 규칙 등을 포함한다.

### 4.6 schema / contract

데이터 교환 규약이다.  
직렬화는 결국 이 계약을 따르는 과정이다.

## 5. 실행 흐름 또는 처리 순서

다음 같은 내부 구조체가 있다고 하자.

```c
typedef struct {
    int id;
    short level;
    char flag;
} Record;
```

이를 외부 포맷으로 내보낸다고 생각해보자.

### 5.1 내부 객체 준비

프로그램 안에서는 `Record`가 편한 형태로 존재한다.  
padding이 있든, host endian이 무엇이든 상관없이 내부 로직은 잘 돌아갈 수 있다.

### 5.2 외부 포맷 규칙 결정

이제 외부 포맷을 정해야 한다. 예를 들어:

- `id`: 4바이트 little-endian
- `level`: 2바이트 little-endian
- `flag`: 1바이트
- padding 없음

즉, 외부 포맷을 먼저 계약으로 정한다.

### 5.3 직렬화

직렬화 시에는 각 필드를 하나씩 포맷 규칙에 맞춰 써야 한다.

- `id`를 지정한 byte order로 변환
- `level`을 지정한 크기와 순서로 저장
- `flag`를 1바이트로 기록

즉, 구조체 전체 raw copy가 아니라 필드별 기록이다.

### 5.4 외부 저장/전송

이제 만들어진 바이트열을 파일에 쓰거나 네트워크로 보낸다.  
이 바이트열은 현재 머신 내부 배치와 독립적이어야 한다.

### 5.5 역직렬화

반대쪽에서는 다음을 수행한다.

- 필요한 바이트 수 읽기
- 길이/버전/유효성 검사
- byte order 변환
- 내부 필드에 값 채우기

즉, 외부 표현을 내부 표현으로 재구성한다.

## 6. 성능 / 최적화 관점

### 6.1 raw copy는 빠를 수 있지만 portable하지 않을 수 있다

메모리 덩어리를 그대로 복사하는 것은 빠를 수 있다.  
하지만 플랫폼, ABI, endian 차이를 넘지 못하면  
성능 이전에 정확성이 깨진다.

즉, 직렬화에서는 빠르기보다 먼저 **포맷 안정성**이 중요하다.

### 6.2 명시적 직렬화는 약간의 비용을 내고 호환성을 얻는다

필드별로 변환하고 쓰는 것은 raw copy보다 코드가 길고 비용도 조금 든다.  
하지만 그 대가로 다음을 얻는다.

- 이식성
- 버전 호환성
- 디버깅 가능성
- 보안 검증 가능성

즉, 직렬화 비용은 대개 정당한 비용이다.

### 6.3 내부 표현과 외부 표현을 분리하면 성능과 안정성을 둘 다 잡기 쉽다

실무에서는 종종 이렇게 한다.

- 내부는 정렬 친화적이고 계산 친화적인 구조체
- 외부는 명시적 wire format
- 경계에서만 변환

즉, packed/raw dump에 집착하기보다  
representation separation이 더 좋은 설계일 수 있다.

### 6.4 대량 데이터에서는 배치 처리와 포맷 설계가 중요하다

직렬화 성능이 중요할 때는 단순 raw copy 대신 다음을 본다.

- 고정 길이 필드인지
- 가변 길이 필드인지
- 변환을 배치 처리할 수 있는지
- 포맷이 캐시/벡터화에 유리한지

즉, 성능 문제도 결국 포맷 설계와 연결된다.

## 7. 어디서부터 내부 메모리 표현과 외부 포맷 계약이 갈리는가

### 7.1 내부 객체를 그대로 복사할 수 있다는 것과 외부 포맷으로 안전하다는 것은 다르다

C에서는 메모리 바이트에 직접 접근할 수 있으니  
내부 객체를 그대로 dump하고 싶어지기 쉽다.  
하지만 내부 메모리 표현은 외부 교환 계약이 아니라  
현재 구현 결과일 뿐이다.

즉, raw copy 가능성과 포맷 안정성은 같은 말이 아니다.

### 7.2 직렬화는 값 보존 문제이지 바이트 복사 문제가 아니다

중요한 것은 현재 메모리 모양을 옮기는 일이 아니라  
값과 의미를 외부 규칙에 맞게 보존하는 일이다.

즉, 직렬화는 메모리 복사보다  
의미 보존과 계약 명시가 중심이다.

### 7.3 역직렬화는 읽기보다 해석과 검증이 더 큰 문제다

외부 바이트를 받아 내부 객체로 만들 때는  
길이, 버전, 범위, byte order, 필드 존재 여부를 해석해야 한다.

즉, 역직렬화는 단순 로딩이 아니라  
포맷 계약을 검증하면서 내부 표현으로 재구성하는 과정이다.

### 7.4 boundary representation과 internal representation은 분리될수록 설명 가능성이 높아진다

내부 계산용 구조는 정렬 친화적이고 단순할 수 있고,  
외부 포맷은 호환성과 안정성을 우선할 수 있다.

즉, 두 표현을 나누는 설계가  
성능과 안정성을 동시에 높이는 경우가 많다.

### 7.5 이 주제의 핵심은 속도보다 계약 유지다

raw copy가 빠를 수 있다는 사실은 중요하지만,  
포맷이 바뀌거나 플랫폼이 달라질 때 깨지는 순간 전체 시스템 비용이 더 커진다.

즉, 직렬화 문제는 미세 성능보다  
장기 계약 유지 문제로 보는 편이 정확하다.

## 8. 실무에서 중요한 판단 기준

### 8.1 struct를 곧바로 포맷 명세로 생각하지 않는다

이건 가장 중요한 원칙이다.  
C struct는 내부 표현일 뿐이며, 외부 포맷은 별도 계약으로 봐야 한다.

### 8.2 외부 포맷에는 크기와 byte order를 명시한다

다음은 반드시 명확해야 한다.

- 필드 크기
- signed/unsigned 여부
- byte order
- 길이 규칙
- 버전 정보

즉, “int 하나” 같은 표현만으로는 부족하다.

### 8.3 포인터, padding, 구현 의존 요소는 직렬화 대상에서 분리한다

포인터 값, padding 바이트, 구현 의존적 bit-field 배치 같은 것은  
그대로 외부로 내보내면 안 된다.

즉, 의미 있는 데이터만 추출해서 내보내야 한다.

### 8.4 역직렬화는 항상 검증 단계가 포함되어야 한다

외부 데이터는 신뢰하지 않는 것이 기본이다.  
길이, 범위, 버전, 포맷 일관성을 먼저 확인해야 한다.

즉, 역직렬화는 단순 읽기가 아니라 파싱이다.

### 8.5 boundary representation과 internal representation을 분리하는 습관이 중요하다

내부에서는 계산 친화적 구조를 쓰고,  
외부 경계에서는 안정적이고 명시적인 포맷을 쓰는 편이 좋다.

즉, 두 표현을 분리하면 packed/raw dump 유혹에서 벗어나기 쉽다.

---

## 9. 판단 체크리스트

- 내부 객체 표현과 외부 포맷 계약을 같은 것으로 보고 있지 않은가
- 직렬화를 바이트 복사보다 값 보존과 계약 명시 문제로 보고 있는가
- 역직렬화를 단순 읽기가 아니라 해석/검증 과정으로 이해하고 있는가
- boundary representation과 internal representation을 분리하고 있는가
- raw copy의 단기 속도보다 장기 포맷 안정성을 먼저 보고 있는가

## 10. 더 깊게 볼 포인트

### 10.1 TLV / length-prefixed formats

가변 길이 데이터를 안전하게 표현하는 직렬화 포맷 설계로 확장할 수 있다.

### 10.2 versioning

파일 포맷과 프로토콜이 시간이 지나며 바뀔 때 어떻게 호환성을 유지할지로 이어질 수 있다.

### 10.3 text vs binary serialization

텍스트 포맷과 바이너리 포맷의 trade-off를 비교할 수 있다.

### 10.4 protocol buffer / flatbuffer 같은 설계 철학 비교

현대 직렬화 시스템이 왜 단순 raw struct dump를 쓰지 않는지 이해하는 데 도움이 된다.

### 10.5 FFI boundary marshalling

다른 언어와 데이터를 주고받을 때도 결국 직렬화/마샬링 문제가 발생한다는 점으로 확장할 수 있다.

## 정리

직렬화와 역직렬화는 단순 `memcpy`가 아니다.

핵심은 다음과 같다.

- 직렬화는 내부 객체를 외부 표현으로 바꾸는 과정이다.
- 역직렬화는 외부 표현을 내부 객체로 재구성하는 과정이다.
- 내부 struct layout은 외부 포맷 명세와 같지 않다.
- raw memory copy는 padding, endianness, ABI, 타입 크기 문제를 그대로 품는다.
- 포인터 필드는 raw serialization 대상이 아니다.
- 외부 포맷은 필드 크기, 순서, byte order, 버전 규칙을 명시해야 한다.
- 역직렬화는 검증과 해석을 포함하는 파싱 과정이어야 한다.

즉, 직렬화와 역직렬화를 이해한다는 것은  
단순히 “구조체를 저장하고 읽는다”가 아니라  
**내부 메모리 표현과 외부 교환 표현이 다르다는 점을 이해하고, 그 경계에서 명시적이고 안정적인 데이터 계약을 설계할 수 있게 되는 것**을 뜻한다.

이 관점이 잡혀야 파일 포맷, 네트워크 프로토콜, 시스템 인터페이스, FFI, 바이너리 호환성을 훨씬 더 정확하게 다룰 수 있다.
