## 1. 왜 이 주제를 깊게 봐야 하는가

C를 어느 정도 배우고 나면 메모리와 값의 차이, 정수 저장 방식, 구조체 레이아웃, packed struct 같은 주제를 보게 된다.  
그 다음 단계에서 반드시 만나게 되는 것이 **endianness**다.

초보자는 보통 숫자를 이렇게 생각한다.

- 32비트 정수는 그냥 4바이트 값이다
- 메모리에 저장되든 레지스터에 있든 같은 숫자다
- `0x12345678`은 어디서 봐도 그대로다

이 생각은 절반만 맞다.  
숫자 자체의 **논리적 값**은 같지만, 그 숫자가 메모리의 여러 바이트로 저장될 때 **바이트 순서**는 시스템에 따라 달라질 수 있다.

예를 들어 다음 32비트 값을 생각해보자.

```
0x12345678
```

이 값은 1바이트씩 나누면 다음 네 조각으로 볼 수 있다.

- 0x12
- 0x34
- 0x56
- 0x78

그런데 메모리의 낮은 주소부터 어떤 순서로 놓일지는 시스템마다 다를 수 있다.

- big-endian: 12 34 56 78
- little-endian: 78 56 34 12

즉, 같은 값이라도 **메모리에 보이는 바이트 배열**은 다를 수 있다.

이 차이를 모르면 다음에서 자주 막힌다.

- 바이너리 파일을 읽고 쓸 때 값이 뒤집혀 보이는 문제
- 네트워크 패킷을 직접 파싱할 때 값이 이상하게 나오는 문제
- packed struct로 바이트 포맷을 바로 읽으려다가 깨지는 문제
- 다른 아키텍처나 다른 언어와의 바이너리 교환에서 값이 달라지는 문제
- 디버거나 hex dump를 봤는데 예상과 다르게 보이는 문제

즉, endianness는 단순 컴퓨터 상식이 아니라  
**값과 메모리 표현을 구분하게 만드는 핵심 개념**이다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- 정수의 “값”과 “메모리 바이트 배열”을 분리해서 보게 해 준다
- 바이너리 파일, 네트워크 프로토콜, FFI, 시스템 인터페이스 이해에 필수다
- packed struct와 직접 매핑의 위험성을 더 정확히 보여 준다
- 직렬화/역직렬화가 왜 단순 `memcpy`가 아닌지 이해하게 해 준다
- C가 왜 시스템 프로그래밍 언어인지 더 선명하게 보여 준다

이 문서는 endianness가 무엇인지, 왜 생기는지,  
값의 의미와 메모리 바이트 순서가 어떻게 다른지,  
실무에서 어떤 곳에 영향을 주는지,  
그리고 언제 반드시 의식해야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

endianness를 큰 구조로 보면 다음과 같다.

### 2.1 endianness는 “바이트 저장 순서” 문제다

endianness는 정수나 다바이트 값이 메모리에 저장될 때  
각 바이트가 어떤 순서로 놓이는지를 말한다.

즉, 핵심은 비트 수준보다 먼저 **바이트 순서**다.

### 2.2 값 자체와 바이트 배열은 다르다

예를 들어 `0x12345678`이라는 값은 논리적으로 동일하다.  
하지만 메모리에 저장하면 시스템에 따라 바이트 배열이 달라질 수 있다.

즉, 숫자의 수학적 의미와 메모리 표현은 구분해야 한다.

### 2.3 대표적으로 little-endian과 big-endian이 있다

가장 대표적인 두 방식은 다음이다.

- little-endian: 가장 덜 중요한 바이트가 낮은 주소에 놓임
- big-endian: 가장 중요한 바이트가 낮은 주소에 놓임

즉, 값은 같아도 메모리에서 보는 순서가 반대일 수 있다.

### 2.4 endianness는 텍스트가 아니라 바이너리에서 중요해진다

문자열 `"1234"` 같은 텍스트는 바이트 순서가 명시적으로 정해져 있다.  
하지만 정수 1234를 바이너리로 저장하면 endianness 문제가 생긴다.

즉, endianness는 특히 다음에서 중요하다.

- 바이너리 파일
- 네트워크 패킷
- 메모리 맵
- 구조체 직렬화
- 다른 시스템과의 이진 데이터 교환

## 3. 내부 동작 원리

### 3.1 1바이트 값에는 endianness 문제가 없다

예를 들어 `unsigned char x = 0x12;` 같은 값은 1바이트다.  
1바이트는 순서를 나눌 수 없으므로 endianness 이슈가 없다.

즉, endianness는 **2바이트 이상 값**에서만 의미가 생긴다.

### 3.2 다바이트 값은 여러 바이트로 분해되어 저장된다

예를 들어 32비트 정수:

```
0x12345678
```

을 메모리에 저장한다고 하자.  
이 값은 바이트 단위로 다음 네 조각으로 나뉜다.

- 0x12
- 0x34
- 0x56
- 0x78

이제 문제는 “이 네 바이트를 낮은 주소부터 어떤 순서로 놓을 것인가”다.

### 3.3 big-endian은 큰 자리 바이트를 먼저 둔다

big-endian에서는 메모리 낮은 주소에 더 중요한 바이트가 먼저 온다.

즉, 대략 이렇게 보일 수 있다.

```
address+0 : 0x12
address+1 : 0x34
address+2 : 0x56
address+3 : 0x78
```

사람이 16진수로 숫자를 읽는 순서와 더 비슷하게 보일 수 있다.

### 3.4 little-endian은 작은 자리 바이트를 먼저 둔다

little-endian에서는 메모리 낮은 주소에 덜 중요한 바이트가 먼저 온다.

즉, 대략 이렇게 보일 수 있다.

```
address+0 : 0x78
address+1 : 0x56
address+2 : 0x34
address+3 : 0x12
```

즉, 숫자를 메모리에서 보면 사람이 기대한 순서와 반대로 보일 수 있다.

### 3.5 포인터로 바이트를 보면 차이가 드러난다

다음 코드를 생각해보자.

```c
unsigned int x = 0x12345678;
unsigned char *p = (unsigned char *)&x;
```

이때 `p[0]`, `p[1]`, `p[2]`, `p[3]`를 보면  
현재 시스템의 바이트 저장 순서가 드러날 수 있다.

즉, endianness는 추상 개념이 아니라  
메모리 바이트를 직접 보면 확인되는 실제 배치 규칙이다.

### 3.6 값 계산 자체와 메모리 표현은 별개다

중요한 점은 CPU가 정수값을 계산할 때 endianness를 “사용자 눈높이에서” 의식하게 만들지는 않는다는 것이다.  
보통 같은 시스템 안에서 값을 읽고 쓰는 한 연산은 정상적으로 동작한다.

문제가 되는 것은 다음 같은 상황이다.

- 메모리를 바이트 단위로 직접 해석할 때
- 파일에 그대로 저장할 때
- 네트워크로 보낼 때
- 다른 시스템과 교환할 때

즉, endianness는 “일반 연산”보다 **표현 외부화**에서 중요해진다.

## 4. 핵심 구성 요소

### 4.1 little-endian

덜 중요한 바이트가 낮은 주소에 저장되는 방식이다.  
메모리 덤프를 보면 사람이 기대한 16진수 표기 순서와 반대로 보일 수 있다.

### 4.2 big-endian

더 중요한 바이트가 낮은 주소에 저장되는 방식이다.  
사람이 읽는 숫자 표기 순서와 메모리 바이트 순서가 더 비슷하게 보일 수 있다.

### 4.3 byte order

바이트 순서 자체를 뜻한다.  
endianness는 결국 byte order 문제다.

### 4.4 host byte order

현재 실행 중인 시스템이 사용하는 바이트 순서다.  
즉, 메모리 안에서 현재 머신이 값들을 저장하는 방식이다.

### 4.5 network byte order

네트워크 프로토콜에서 정한 표준 바이트 순서를 뜻한다.  
일반적으로 별도 규약을 통해 시스템 차이를 넘는다.

### 4.6 serialization boundary

값을 메모리 밖으로 내보내거나 다시 읽어들이는 경계다.  
endianness는 이 경계에서 특히 중요하다.

## 5. 실행 흐름 또는 처리 순서

다음 값을 예로 보자.

```c
unsigned int x = 0x12345678;
```

### 5.1 값 생성

소스 코드 수준에서는 `x`는 그냥 하나의 32비트 정수값이다.  
이 단계에서는 사람이 `0x12345678`이라는 논리값을 이해한다.

### 5.2 메모리 저장

이제 이 값을 메모리에 저장하려면 4바이트로 나눠야 한다.  
이때 시스템은 자기 byte order 규칙에 따라 저장한다.

### 5.3 바이트 단위 관찰

다음처럼 바이트 포인터로 보면:

```c
unsigned char *p = (unsigned char *)&x;
```

`p[0]`부터 `p[3]`까지의 값이 현재 시스템 endianness를 반영한다.

### 5.4 같은 시스템 안 연산

같은 시스템 안에서 그냥 `x + 1`, `x * 2` 같은 연산을 하면  
보통 endianness를 신경 쓰지 않아도 된다.  
값 해석과 저장이 같은 규칙 안에서 일관되기 때문이다.

### 5.5 외부로 내보낼 때 문제 발생 가능

하지만 이 값을 파일에 그대로 쓰거나 네트워크로 보내거나  
다른 시스템이 읽게 하면 상황이 달라진다.

즉, 이 시점에서 “현재 시스템 byte order”와 “외부 포맷이 기대하는 byte order”가 다르면 변환이 필요하다.

## 6. 성능 / 최적화 관점

### 6.1 일반 계산 성능보다 호환성과 정확성 문제가 더 크다

endianness는 대개 성능 최적화 주제라기보다  
정확한 데이터 교환과 해석 문제로 더 중요하다.

즉, 같은 시스템 내부 계산에서는 큰 이슈가 아닐 수 있지만  
시스템 경계에서는 치명적일 수 있다.

### 6.2 무비판적 raw memory copy는 위험하다

구조체나 정수를 그대로 바이트 덩어리로 저장/전송하면  
endianness 차이 때문에 다른 시스템에서 다르게 읽힐 수 있다.

즉, `memcpy`나 raw dump는 빠를 수 있어도  
항상 portable하거나 안전한 직렬화 방식은 아니다.

### 6.3 명시적 직렬화는 비용이 있지만 안전성을 준다

필요한 바이트 순서를 명시적으로 정해 변환해서 저장하면  
시스템 간 호환성과 디버깅 가능성이 훨씬 좋아진다.

즉, 약간의 변환 비용과 얻는 안정성 사이에서 trade-off가 있다.

### 6.4 packed struct와 결합하면 더 위험해질 수 있다

packed struct로 외부 포맷을 그대로 읽는 방식은  
padding 문제만이 아니라 endianness 문제까지 함께 품는다.

즉, “레이아웃이 맞는다”와 “값 의미가 맞는다”는 다른 문제다.

## 7. 어디서부터 논리값과 바이트 표현 순서가 갈리는가

### 7.1 숫자 값이 같다는 것과 메모리 바이트 배열이 같다는 것은 다르다

프로그래머는 보통 정수 값을 숫자 하나로 본다.  
하지만 메모리 안에서는 그 값이 여러 바이트로 나뉘어 놓인다.

즉, 논리값 동일성과 바이트 배열 동일성은 다른 층의 정보다.

### 7.2 같은 시스템 내부 계산과 시스템 경계 밖 교환은 다른 문제다

한 머신 안에서 정수 연산만 할 때는  
endianness가 거의 드러나지 않을 수 있다.  
하지만 파일, 네트워크, FFI, 장치 경계로 나가면 바이트 순서가 직접 문제가 된다.

즉, endianness는 계산 규칙보다  
시스템 경계 문제로 보는 편이 정확하다.

### 7.3 레이아웃이 맞는 것과 값 의미가 맞는 것은 다르다

packed struct나 raw dump로 바이트 배치를 얼추 맞출 수는 있어도  
멀티바이트 필드의 바이트 순서까지 자동으로 해결되지는 않는다.

즉, struct layout과 value interpretation은 분리해서 봐야 한다.

### 7.4 raw copy가 가능한 것과 portable한 것은 다르다

`memcpy`나 raw dump는 현재 환경에서는 자연스러워 보일 수 있다.  
하지만 다른 endian 환경으로 넘어가는 순간 값 의미가 뒤집힐 수 있다.

즉, 물리적 복사 가능성과 의미 보존 가능성은 같은 말이 아니다.

### 7.5 이 주제는 속도보다 정확한 계약 문제다

endianness는 보통 성능보다  
어떤 바이트 순서를 외부 계약으로 삼을 것인가의 문제로 더 중요하다.

즉, endian 문서는 미세 최적화보다  
정확한 표현 계약 문서로 읽어야 한다.

## 8. 실무에서 중요한 판단 기준

### 8.1 메모리 안 값과 외부 포맷 값을 구분한다

실무에서는 먼저 다음을 구분해야 한다.

- 이 값은 지금 내 머신 메모리 안에서만 쓰는가
- 아니면 파일/네트워크/다른 언어와 교환되는가

후자라면 endianness를 반드시 의식해야 한다.

### 8.2 바이너리 포맷에는 byte order를 명시해야 한다

파일 포맷이나 프로토콜을 설계할 때  
“그냥 int를 쓴다” 정도로 정의하면 안 된다.

대신 다음이 필요하다.

- 몇 바이트 정수인가
- signed/unsigned인가
- byte order는 무엇인가

즉, 포맷 설계에는 byte order 명세가 필수다.

### 8.3 구조체를 그대로 직렬화하면 안심하면 안 된다

다음이 맞는지 항상 의심해야 한다.

- struct layout이 같다고 값 의미도 같은가
- packed라고 해서 외부 포맷과 완전히 일치하는가
- 다른 플랫폼에서 같은 결과가 나오는가

즉, struct raw dump는 매우 신중해야 한다.

### 8.4 네트워크와 파일은 host byte order와 다를 수 있다고 가정한다

네트워크 프로토콜이나 표준 파일 포맷은  
현재 머신 메모리 순서와 다를 수 있다고 먼저 생각하는 편이 안전하다.

즉, “내 머신 기준으로 정상”은 외부 호환성 근거가 아니다.

### 8.5 디버깅할 때 hex dump를 읽는 눈이 필요하다

endianness를 이해하면 hex dump를 볼 때  
값이 왜 뒤집혀 보이는지, 어떤 바이트가 어느 의미인지 읽을 수 있다.

즉, 시스템 디버깅 능력과 직접 연결된다.

## 9. 판단 체크리스트

- 논리값과 바이트 배열을 같은 층으로 보고 있지 않은가
- 같은 시스템 내부 계산과 시스템 경계 밖 교환을 구분하고 있는가
- struct layout과 값 의미 해석을 분리해서 보고 있는가
- raw copy 가능성과 portable한 의미 보존을 같은 것으로 보고 있지 않은가
- byte order를 외부 포맷 계약의 일부로 명시하고 있는가

## 정리

endianness는 값이 메모리에 저장될 때의 바이트 순서 문제다.

핵심은 다음과 같다.

- 1바이트 값에는 endianness 문제가 없다.
- 다바이트 값은 여러 바이트로 나뉘어 저장된다.
- little-endian과 big-endian은 그 바이트를 낮은 주소부터 어떤 순서로 놓는지가 다르다.
- 값 자체의 의미와 메모리 바이트 배열은 구분해야 한다.
- 같은 시스템 안 일반 정수 연산에서는 보통 크게 드러나지 않는다.
- 파일, 네트워크, 바이너리 포맷, FFI, packed struct, raw serialization에서는 매우 중요하다.
- struct layout이 맞는 것과 값 의미가 맞는 것은 다른 문제다.

즉, endianness를 이해한다는 것은  
단순히 big/little 정의를 외우는 것이 아니라  
**정수와 구조체의 논리값이 메모리에서 실제 어떤 바이트 배열로 표현되고, 그 표현이 시스템 경계를 넘을 때 왜 별도 해석과 변환이 필요한지 이해하는 것**을 뜻한다.

이 관점이 잡혀야 바이너리 포맷 처리, 네트워크 프로그래밍, 시스템 디버깅, 직렬화 설계를 훨씬 더 정확하게 다룰 수 있다.
