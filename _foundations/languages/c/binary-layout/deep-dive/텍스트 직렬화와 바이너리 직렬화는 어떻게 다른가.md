## 1. 왜 이 주제를 깊게 봐야 하는가

앞 문서에서 직렬화와 역직렬화가 왜 단순 `memcpy`가 아닌지 봤다면,  
다음에는 자연스럽게 이런 질문이 생긴다.

- 외부 포맷을 만들 때 텍스트로 저장할까, 바이너리로 저장할까
- JSON 같은 텍스트 형식이 왜 많이 쓰이는가
- 바이너리 포맷은 왜 빠르다고 하는가
- 사람이 읽을 수 있는 포맷과 기계가 처리하기 좋은 포맷은 왜 다를 수 있는가
- 로그, 설정 파일, 네트워크 프로토콜, 대용량 데이터 파일에서 왜 선택이 달라지는가

처음 직렬화를 배우면 이렇게 생각하기 쉽다.

- 텍스트는 느리고 바이너리는 빠르다
- 바이너리가 항상 더 좋다
- 텍스트는 사람 보기 좋은 대신 비효율적이다
- 바이너리는 구조체를 그냥 저장하면 된다

이런 생각은 일부만 맞다.  
실제로 텍스트와 바이너리의 차이는 단순 속도 하나로 정리되지 않는다.  
다음이 함께 얽힌다.

- 크기
- 파싱 비용
- 디버깅 가능성
- 이식성
- 버전 호환성
- 장애 분석 편의성
- 구현 복잡도
- 보안 검증 난이도

예를 들어 숫자 123456을 외부 포맷으로 저장한다고 하자.

텍스트라면:

```
"123456"
```

처럼 저장할 수 있다.

바이너리라면:

- 4바이트 정수
- little-endian 혹은 big-endian
- signed/unsigned 규칙

같은 식으로 저장해야 한다.

즉, 텍스트는 사람이 직접 의미를 읽기 쉽고,  
바이너리는 기계가 고정된 규칙으로 다루기 쉽다.  
하지만 양쪽 다 장단점이 있다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- 직렬화 포맷 설계가 단순 저장 방식이 아니라 시스템 설계 문제임을 보여 준다
- 텍스트와 바이너리의 trade-off를 성능·운영·호환성 관점에서 볼 수 있게 해 준다
- “빠른 포맷”과 “디버깅 쉬운 포맷”이 왜 다를 수 있는지 이해하게 해 준다
- 파일 포맷과 네트워크 프로토콜 설계의 사고 기준을 준다
- 이후 length-prefixed format, schema evolution, protocol design으로 자연스럽게 이어진다

이 문서는 텍스트 직렬화와 바이너리 직렬화가 각각 무엇인지,  
왜 둘의 장단점이 다른지,  
실무에서 어떤 기준으로 선택해야 하는지 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

텍스트 직렬화와 바이너리 직렬화를 큰 구조로 보면 다음과 같다.

### 2.1 텍스트 직렬화는 사람이 읽을 수 있는 문자열 형태의 외부 표현이다

텍스트 직렬화는 데이터를 문자 기반 표현으로 저장하거나 전송하는 방식이다.

예를 들어:

- CSV
- JSON
- XML
- line-based log format
- 단순 key=value 형식

즉, 사람이 열어 보고 의미를 직접 파악할 수 있는 포맷이 여기에 가깝다.

### 2.2 바이너리 직렬화는 바이트 단위 규칙에 따라 저장하는 방식이다

바이너리 직렬화는 데이터를 문자로 바꾸지 않고  
정해진 바이트 규칙에 따라 저장/전송하는 방식이다.

예를 들어:

- 고정 크기 정수 필드
- 길이 + 데이터 구조
- 비트 플래그
- packed header
- 프로토콜 전용 binary frame

즉, 사람이 텍스트 에디터로 봐서는 바로 의미를 읽기 어려운 대신,  
기계가 명확한 규칙으로 빠르게 해석하기 좋다.

### 2.3 핵심 차이는 “표현 방식”과 “운영 목적”이다

텍스트와 바이너리의 차이는 단순 파일 종류 차이가 아니다.  
핵심은 다음이다.

- 사람이 직접 볼 것인가
- 기계가 빠르게 처리할 것인가
- 디버깅/로그 용도인가
- 장기 저장/교환 포맷인가
- 네트워크 대역폭이 중요한가

즉, 어떤 포맷이 좋은지는 목적에 따라 달라진다.

### 2.4 직렬화 포맷은 데이터 계약이다

텍스트든 바이너리든 결국 중요한 것은 계약이다.

- 필드가 어떻게 표현되는가
- 구분자는 무엇인가
- 길이는 어떻게 알 수 있는가
- 숫자, 문자열, 배열을 어떻게 표기하는가
- 버전 차이는 어떻게 처리하는가

즉, 포맷의 품질은 텍스트냐 바이너리냐보다  
**규칙이 얼마나 명확한가**에 크게 달려 있다.

## 3. 내부 동작 원리

### 3.1 텍스트 직렬화는 값을 문자 표현으로 바꾼다

예를 들어 정수 123을 텍스트로 저장한다면 보통 다음 문자들로 바뀐다.

- `'1'`
- `'2'`
- `'3'`

즉, 숫자 그 자체를 바이트 정수로 넣는 것이 아니라  
숫자를 나타내는 문자 시퀀스로 바꾸는 것이다.

이 덕분에 사람이 읽고 이해하기 쉽다.

### 3.2 바이너리 직렬화는 값을 포맷 규칙에 맞는 바이트열로 바꾼다

같은 정수 123을 바이너리로 저장하면,  
예를 들어 32비트 little-endian 규칙을 정했다면  
그 규칙에 맞는 4바이트 표현으로 저장할 수 있다.

즉, 텍스트처럼 “1”, “2”, “3” 문자로 바꾸지 않는다.  
바로 바이트 표현을 쓴다.

### 3.3 텍스트는 구분과 파싱 규칙이 필요하다

텍스트 포맷은 사람이 보기 좋지만,  
기계가 읽으려면 다음 같은 규칙이 필요하다.

- 필드 구분자
- 레코드 구분자
- escape 규칙
- 숫자/문자열 표현 규칙
- 공백 처리
- 인코딩

예를 들어 CSV는 쉼표와 줄바꿈, 따옴표 escape 규칙이 중요하다.  
JSON도 구조는 단순해 보이지만 실제로는 명확한 문법 규칙이 있다.

즉, 텍스트라고 해서 규칙이 단순한 것은 아니다.

### 3.4 바이너리는 길이와 경계를 명시해야 한다

바이너리 포맷은 문자 구분자가 없으므로,  
필드 길이나 구조 경계를 더 명확히 정의해야 한다.

예를 들어:

- 이 필드는 4바이트 정수
- 다음은 2바이트 길이
- 그 다음은 length 바이트 문자열

즉, 바이너리는 compact할 수 있지만  
경계 규칙이 더 엄격해야 한다.

### 3.5 텍스트는 endian 문제를 많이 숨겨 준다

텍스트 `"123456"`은 어떤 시스템에서 읽어도 문자 그대로다.  
즉, 숫자를 문자열로 바꿔 버리면 endianness 문제가 사실상 사라진다.

반면 바이너리 정수는 byte order를 명시해야 한다.

즉, 텍스트는 성능 비용 대신 portability와 가시성을 얻는 경우가 많다.

### 3.6 바이너리는 더 compact하지만 해석이 더 엄격하다

예를 들어 정수 하나를 텍스트로 쓰면 자리수만큼 여러 바이트가 필요할 수 있다.  
바이너리는 고정 4바이트 같은 식으로 compact할 수 있다.

하지만 그 대신 다음을 명확히 해야 한다.

- signed/unsigned
- byte order
- field boundary
- alignment를 무시할지 여부
- version compatibility

즉, 바이너리는 효율적인 대신 설계 실수 여지가 더 많다.

## 4. 핵심 구성 요소

### 4.1 text serialization

사람이 읽을 수 있는 문자 기반 포맷으로 데이터를 표현하는 방식이다.  
가독성과 디버깅 편의성이 크다.

### 4.2 binary serialization

데이터를 문자로 바꾸지 않고 바이트 규칙에 따라 표현하는 방식이다.  
공간 효율과 처리 효율에 유리할 수 있다.

### 4.3 human readability

사람이 파일을 열어 의미를 직접 파악할 수 있는 성질이다.  
텍스트 포맷의 큰 장점이다.

### 4.4 parse cost

외부 표현을 내부 값으로 바꾸는 비용이다.  
텍스트는 숫자/문자열 파싱 비용이 있고, 바이너리는 바이트 해석 비용이 있다.

### 4.5 portability

다른 플랫폼과 환경에서도 같은 의미로 읽히는 정도다.  
텍스트는 byte order 문제를 덜 드러내는 경우가 많다.

### 4.6 schema clarity

필드 구조와 규칙이 얼마나 명확히 정의되어 있는지다.  
좋은 포맷은 텍스트/바이너리 여부와 별개로 schema가 명확하다.

## 5. 실행 흐름 또는 처리 순서

정수 `id = 12345`, 문자열 `name = "aki"`를 저장한다고 생각해보자.

### 5.1 텍스트 직렬화

텍스트 포맷이라면 예를 들어 다음처럼 표현할 수 있다.

```
id=12345
name=aki
```

또는 JSON 스타일이라면:

```c
{"id":12345,"name":"aki"}
```

직렬화 시에는:

- 정수를 문자열로 변환
- 필드 이름과 구분자 추가
- 줄바꿈/괄호/따옴표 같은 문법 요소 추가
- 전체를 문자 바이트열로 기록

즉, 사람이 읽기 좋은 형태를 만드는 과정이다.

### 5.2 텍스트 역직렬화

역직렬화 시에는:

- 문자열 읽기
- 문법 단위 분리
- 숫자 문자열을 다시 정수로 변환
- 이름 필드 추출

즉, parsing 단계가 꽤 중요하다.

### 5.3 바이너리 직렬화

바이너리 포맷이라면 예를 들어 다음처럼 규칙을 정할 수 있다.

- `id`: 4바이트 unsigned little-endian
- `name_len`: 1바이트
- `name`: 길이만큼 raw bytes

그러면 직렬화는:

- `id`를 4바이트로 저장
- 문자열 길이 저장
- 문자열 바이트 저장

즉, 훨씬 compact하게 표현될 수 있다.

### 5.4 바이너리 역직렬화

역직렬화 시에는:

- 먼저 4바이트 읽어 `id` 해석
- 1바이트 읽어 이름 길이 해석
- 그 길이만큼 문자열 읽기

즉, 문법 parsing보다 경계 해석과 길이 검증이 더 중요해진다.

## 6. 성능 / 최적화 관점

### 6.1 텍스트는 보통 더 크고 파싱 비용이 더 클 수 있다

숫자를 문자열로 바꾸고 다시 읽는 데는 비용이 든다.  
또 구분자와 문법 정보 때문에 크기도 더 커질 수 있다.

즉, 대량 데이터 처리에서는 텍스트가 비효율적일 수 있다.

### 6.2 바이너리는 compact하고 처리 효율이 좋을 수 있다

고정 크기 필드와 명확한 규칙을 가지는 바이너리 포맷은  
읽기/쓰기 속도와 저장 공간 측면에서 유리할 수 있다.

특히 다음 상황에서 강하다.

- 대용량 데이터 파일
- 고빈도 네트워크 메시지
- 실시간 처리
- 메모리 대역폭이 중요한 상황

### 6.3 하지만 운영 비용은 텍스트가 더 낮을 수 있다

텍스트 포맷은 사람이 직접 열어 볼 수 있고,  
장애 분석, 로깅, 운영 중 임시 수정, 테스트 입력 작성이 쉽다.

즉, CPU 비용만 보면 텍스트가 불리할 수 있어도  
운영 비용과 개발 편의까지 보면 오히려 더 유리할 수 있다.

### 6.4 성능만 보고 바이너리를 선택하면 디버깅 비용이 커질 수 있다

바이너리 포맷은 compact하지만  
문제가 생겼을 때 사람이 바로 읽기 어렵다.

즉, 시스템 전체 비용은 다음을 함께 봐야 한다.

- 처리 속도
- 저장 크기
- 디버깅 시간
- 장애 대응 난이도
- 버전 호환성 관리 비용

## 7. 어디서부터 사람 친화적 표현과 기계 친화적 표현이 갈리는가

### 7.1 텍스트와 바이너리는 가독성 차이만이 아니라 계약 방식이 다르다

텍스트는 구분자와 문자 표현을 통해 의미를 드러내고,  
바이너리는 필드 경계와 규칙을 별도로 더 엄격하게 정해야 한다.

즉, 둘의 차이는 보기 좋음 여부보다  
의미를 표현하는 방식 차이다.

### 7.2 텍스트가 느슨해 보인다고 규칙이 약한 것은 아니다

사람이 읽을 수 있다는 이유로  
텍스트 포맷을 느슨한 데이터쯤으로 보면 안 된다.  
텍스트도 스키마, 필드 의미, 오류 처리 정책이 있어야 안정적이다.

즉, 사람 친화성과 계약 엄격성은 다른 축이다.

### 7.3 바이너리가 compact하다고 해서 자동으로 설계가 좋은 것은 아니다

크기가 작고 파싱이 빠를 수는 있다.  
하지만 field boundary, byte order, evolution 정책이 불분명하면  
오히려 장기 운영 비용이 커질 수 있다.

즉, compact함과 좋은 포맷 설계는 같은 말이 아니다.

### 7.4 운영 비용과 실행 비용은 서로 다른 최적화 대상이다

텍스트는 CPU/대역폭 측면에서 불리할 수 있지만  
디버깅, 운영, 장애 분석, 임시 입력 작성 측면에서는 유리하다.

즉, 포맷 선택은 실행 비용과 운영 비용을 따로 봐야 한다.

### 7.5 이 선택은 형식 취향이 아니라 시스템 경계 성격 문제다

사람이 직접 보고 편집하는 경계인지,  
기계끼리 고빈도 교환하는 경계인지에 따라 정답이 달라진다.

즉, 텍스트/바이너리 선택은 포맷 미학보다  
경계 성격과 수명 주기 문제다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 사람이 직접 보고 다뤄야 하는가부터 묻는다

설정 파일, 로그, 운영 중 수동 검사 대상이라면  
텍스트 포맷이 훨씬 유리할 수 있다.

반면 기계끼리 대량 교환하는 데이터라면  
바이너리가 더 적합할 수 있다.

### 8.2 대역폭과 저장 공간이 중요한가를 본다

네트워크 트래픽이 크거나 저장 공간이 민감하면  
바이너리 포맷의 compact함이 큰 장점이 될 수 있다.

### 8.3 버전 호환성과 schema evolution을 고려한다

포맷은 한 번 정하면 끝이 아니다.  
나중에 필드가 추가되거나 바뀔 수 있다.

텍스트는 느슨하게 확장하기 쉬운 경우가 많고,  
바이너리는 명시적 버전 설계가 더 중요해진다.

### 8.4 raw struct dump를 바이너리 포맷으로 착각하지 않는다

바이너리 포맷을 쓴다고 해서 struct를 그냥 써도 된다는 뜻은 아니다.  
padding, endian, pointer, ABI 문제가 여전히 있다.

즉, 바이너리 직렬화도 “명시적 포맷 설계”가 필요하다.

### 8.5 하이브리드 전략도 가능하다

실무에서는 종종 이렇게 쓴다.

- 로그와 설정은 텍스트
- 고빈도 내부 프로토콜은 바이너리
- 디버그 모드는 텍스트, 운영 모드는 바이너리

즉, 선택은 하나로 고정되지 않아도 된다.

---

## 9. 판단 체크리스트

- 텍스트/바이너리 차이를 가독성보다 계약 방식 차이로 보고 있는가
- 텍스트가 사람 친화적이라고 해서 규칙이 약해도 된다고 생각하고 있지 않은가
- 바이너리의 compact함과 좋은 포맷 설계를 같은 것으로 보고 있지 않은가
- 실행 비용과 운영 비용을 분리해서 보고 있는가
- 이 경계가 사람용인지 기계용인지 먼저 정하고 포맷을 고르고 있는가

---

## 10. 더 깊게 볼 포인트

### 10.1 length-prefixed binary format

가변 길이 데이터를 바이너리로 안전하게 표현하는 방식으로 이어질 수 있다.

### 10.2 JSON/CSV vs custom binary protocol

범용 텍스트 포맷과 전용 바이너리 포맷의 trade-off를 더 구체적으로 비교할 수 있다.

### 10.3 schema evolution

필드 추가/삭제/이름 변경이 포맷 호환성에 어떤 영향을 주는지 확장할 수 있다.

### 10.4 compression과의 관계

텍스트가 더 커 보여도 압축 후에는 차이가 줄어드는 경우 등 더 현실적인 비교로 이어질 수 있다.

### 10.5 debugging tooling

텍스트와 바이너리 포맷 각각에서 어떤 도구가 필요한지로 확장할 수 있다.

## 정리

텍스트 직렬화와 바이너리 직렬화는 각각 다른 장단점을 가진다.

핵심은 다음과 같다.

- 텍스트 직렬화는 사람이 읽기 쉽고 디버깅과 운영이 편하다.
- 바이너리 직렬화는 compact하고 처리 효율이 좋을 수 있다.
- 텍스트는 숫자를 문자열로 바꾸므로 파싱 비용이 크지만 endian 문제를 많이 숨겨 준다.
- 바이너리는 field boundary, byte order, schema를 더 엄격하게 정의해야 한다.
- 좋은 포맷은 텍스트냐 바이너리냐보다 규칙이 명확한지가 더 중요하다.
- raw struct dump는 안전한 바이너리 포맷과 다르다.
- 실제 선택은 성능, 운영, 호환성, 디버깅, 대역폭 요구를 함께 보고 해야 한다.

즉, 텍스트 직렬화와 바이너리 직렬화를 이해한다는 것은  
단순히 “사람이 읽는 포맷 vs 기계가 읽는 포맷”을 아는 것이 아니라  
**데이터 교환 포맷 설계에서 성능, 가시성, 호환성, 운영 비용이 어떻게 trade-off를 이루는지 이해하고 목적에 맞는 표현을 선택할 수 있게 되는 것**을 뜻한다.

이 관점이 잡혀야 파일 포맷, 로그 설계, 네트워크 프로토콜, 데이터 교환 인터페이스를 훨씬 더 안정적으로 설계할 수 있다.
