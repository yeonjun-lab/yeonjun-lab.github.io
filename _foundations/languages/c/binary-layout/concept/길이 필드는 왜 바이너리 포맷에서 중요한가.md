## 1. 왜 이 주제를 공부했는가

바이너리 포맷을 설계하거나 읽다 보면  
어디까지가 하나의 메시지이고 어디부터 다음 데이터인지 구분해야 한다.  
이때 길이 필드가 없으면 파싱과 검증이 훨씬 어려워진다.

즉, 길이 필드는 단순 보조 정보가 아니라 포맷 경계를 정의하는 핵심 장치다.

## 2. 정의

길이 필드는 뒤따르는 데이터가 몇 바이트인지,  
혹은 특정 블록이 어디까지 이어지는지 나타내는 메타데이터다.

## 3. 핵심 개념

### 3.1 길이 필드는 경계를 명시한다

가변 길이 데이터는 종료 규칙이나 외부 문맥만으로는 읽기 어렵다.

### 3.2 길이 정보는 검증과도 연결된다

읽을 바이트 수가 명시되면  
입력 검증과 bounds 체크가 가능해진다.

### 3.3 길이 필드는 확장성에도 도움이 된다

필드가 늘어나거나 일부가 선택적으로 붙는 포맷에서  
블록 경계를 유지하기 쉬워진다.

## 4. 예시

문자열이나 바이트 블록을  
널 종료나 파일 끝에만 의존해 읽는 방식은  
중간에 임의 바이트가 들어가는 데이터에서 취약할 수 있다.

반면 길이 필드가 있으면  
정해진 바이트 수만큼 읽고 다음 필드로 넘어갈 수 있다.

## 5. 헷갈리기 쉬운 부분

### 5.1 길이 필드는 중복 정보라서 필요 없다고 생각하는 문제

오히려 명시적 경계가 파싱과 검증을 쉽게 만든다.

### 5.2 종료 문자가 있으면 충분하다고 생각하는 문제

텍스트 문자열과 일반 바이너리 블록은 요구가 다르다.

### 5.3 길이 값만 믿으면 안전하다고 생각하는 문제

길이 필드도 입력 검증 대상이다.

## 6. 어디서부터 직관과 실제 규칙이 갈리는가

### 6.1 데이터 내용과 데이터 경계는 다른 층의 정보다

내용만 보고는 블록 끝을 안정적으로 알 수 없는 경우가 많다.

### 6.2 길이 필드는 파싱 편의가 아니라 안전성 정보이기도 하다

경계를 명시해야 bounds 검사와 malformed input 대응이 가능하다.

### 6.3 길이 정보가 있어도 무조건 신뢰하면 안 된다

외부 입력이라면 값 자체를 검증해야 한다.

## 7. 실무 관점

### 7.1 가변 길이 데이터에는 경계 정보를 명시한다

문자열, 바이트 배열, 반복 필드는 특히 중요하다.

### 7.2 길이와 실제 버퍼 크기를 함께 검증한다

선언된 길이가 입력 크기 안에 들어오는지 확인해야 한다.

### 7.3 버전 확장성을 같이 고려한다

길이 기반 블록은 일부 필드를 건너뛰는 전략에도 유리하다.

## 8. 판단 체크리스트

- 이 데이터 블록의 경계를 무엇으로 판단하는가
- 종료 규칙과 길이 규칙을 혼동하고 있지 않은가
- 길이 값을 입력 검증 없이 신뢰하고 있지 않은가
- 길이 필드가 bounds 검사와 오류 처리에 어떻게 연결되는지 알고 있는가
- 이후 포맷 확장 시 길이 기반 경계가 도움이 되는지 고려했는가

## 9. 다음에 볼 것

이 문서 다음에는 pagination 순서상 아래 흐름으로 가는 것이 가장 자연스럽다.

### 9.1 직렬화와 역직렬화는 왜 단순 memcpy가 아닌가
가장 바로 이어지는 문서다. 방금 본 내용을 같은 `binary-layout` 축 안에서 바로 확장한다.

### 9.2 텍스트 직렬화와 바이너리 직렬화는 어떻게 다른가
한 단계 더 읽으면 방금 이어진 설명이 같은 주제 안에서 어떻게 깊어지는지 더 분명해진다.

### 9.3 endianness는 왜 중요한가
세 번째 문서까지 보면 현재 문서가 전체 C 시리즈의 어디에 놓여 있는지 읽기 흐름이 더 안정된다.

## 정리

길이 필드는 바이너리 포맷에서 단순 메타데이터가 아니다.  
핵심은 **데이터 경계를 명시하고 검증 가능하게 만드는 장치**라는 점이다.
