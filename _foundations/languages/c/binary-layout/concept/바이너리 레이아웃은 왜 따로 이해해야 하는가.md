## 1. 왜 이 주제를 공부했는가

C에서는 메모리 바이트를 직접 다루기 쉽기 때문에  
종종 "메모리에 있는 그대로 파일이나 네트워크로 보내면 되지 않을까"라는 생각을 하게 된다.

하지만 실제로는 정렬, 패딩, endianness, 버전 차이 때문에  
메모리 배치와 외부 바이너리 표현을 같은 것으로 보면 위험하다.

즉, 바이너리 레이아웃은 단순 바이트 복사 문제가 아니라  
**어떤 표현을 외부 계약으로 삼을 것인가**의 문제다.

## 2. 정의

바이너리 레이아웃은 데이터가 바이트 단위로  
어떤 순서와 구조로 배치되는지를 뜻한다.

문제는 이 레이아웃이 항상 한 가지가 아니라는 점이다.

- 메모리 안의 객체 배치
- 파일 포맷 배치
- 네트워크 프로토콜 배치
- 다른 버전 간 호환을 고려한 배치

## 3. 핵심 개념

### 3.1 메모리 객체 배치와 외부 포맷 배치는 다를 수 있다

구조체 메모리 레이아웃은 컴파일러, 정렬 규칙, 패딩 영향을 받는다.  
반면 파일/네트워크 포맷은 더 엄격한 외부 계약이어야 한다.

### 3.2 바이트 순서와 필드 경계는 명시되어야 한다

멀티바이트 정수, 길이 필드, 가변 데이터는  
어떤 순서와 규칙으로 읽을지 합의가 있어야 한다.

### 3.3 버전과 확장성도 레이아웃 문제다

오늘 맞는 구조가 내일도 그대로 읽힐 수 있는지는  
포맷 설계와 호환 전략에 달려 있다.

## 4. 예시

`struct`를 그대로 `write()` 하는 코드는  
현재 머신에서는 되는 것처럼 보여도  
다른 플랫폼, 다른 컴파일러, 다른 버전에서는 깨질 수 있다.

즉, 같은 바이트 복사라도  
"메모리 객체 저장"과 "외부 포맷 직렬화"는 같은 일이 아니다.

## 5. 헷갈리기 쉬운 부분

### 5.1 바이트는 바이트니까 어디서나 같다고 생각하는 문제

바이트 자체는 같아도 해석 규칙과 배치 계약이 다를 수 있다.

### 5.2 struct layout이 곧 포맷 정의라고 생각하는 문제

메모리 배치는 구현체 영향이 있고, 외부 포맷은 계약을 명시해야 한다.

### 5.3 직렬화를 단순 복사로 보는 문제

직렬화는 값 표현을 외부 규칙에 맞게 변환하는 작업일 수 있다.

## 6. 어디서부터 직관과 실제 규칙이 갈리는가

### 6.1 메모리에 있는 모양과 외부에 저장해야 할 모양은 다를 수 있다

현재 객체 배치는 내부 구현 결과이고,  
외부 포맷은 장기 호환 계약일 수 있다.

### 6.2 바이너리 포맷은 값 복사보다 해석 규칙을 먼저 정해야 한다

정수 크기, 바이트 순서, 길이 정보, 확장 필드 규칙이 먼저다.

### 6.3 "지금 읽힌다"와 "넓게 호환된다"는 다른 말이다

현재 환경에서 round-trip이 된다고 해서  
포맷 설계가 견고하다는 뜻은 아니다.

## 7. 실무 관점

### 7.1 외부 계약은 메모리 레이아웃과 분리해서 정의한다

파일, 네트워크, IPC 포맷은 명시적 규칙으로 정의하는 편이 안전하다.

### 7.2 길이, 버전, 확장 가능성을 미리 고려한다

고정 구조만 생각하면 이후 호환성이 약해진다.

### 7.3 직렬화와 검증을 함께 설계한다

읽고 쓰는 코드뿐 아니라 malformed input 처리까지 같이 봐야 한다.

## 8. 판단 체크리스트

- 메모리 객체 배치와 외부 포맷 배치를 같은 것으로 보고 있지 않은가
- 바이트 순서, 길이, 필드 경계를 명시하고 있는가
- 현재 환경에서 읽힌다는 사실과 장기 호환성을 같은 뜻으로 보고 있지 않은가
- struct layout을 그대로 외부 계약으로 내보내려 하고 있지 않은가
- 버전과 확장 전략을 레이아웃 설계에 포함하고 있는가

## 9. 다음에 볼 것

이 문서 다음에는 pagination 순서상 아래 흐름으로 가는 것이 가장 자연스럽다.

### 9.1 길이 필드는 왜 바이너리 포맷에서 중요한가
가장 바로 이어지는 문서다. 방금 본 내용을 같은 `binary-layout` 축 안에서 바로 확장한다.

### 9.2 직렬화와 역직렬화는 왜 단순 memcpy가 아닌가
한 단계 더 읽으면 방금 이어진 설명이 같은 주제 안에서 어떻게 깊어지는지 더 분명해진다.

### 9.3 텍스트 직렬화와 바이너리 직렬화는 어떻게 다른가
세 번째 문서까지 보면 현재 문서가 전체 C 시리즈의 어디에 놓여 있는지 읽기 흐름이 더 안정된다.

## 정리

바이너리 레이아웃은 바이트를 다루는 기술이지만, 본질은 바이트 자체보다  
**그 바이트에 어떤 계약과 해석 규칙을 부여할 것인가**에 있다.

이 기준이 잡혀야 직렬화, 호환성, 포맷 진화, 바이너리 안전성을 제대로 다룰 수 있다.
