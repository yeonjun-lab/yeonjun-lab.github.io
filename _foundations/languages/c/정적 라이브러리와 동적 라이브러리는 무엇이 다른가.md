## 1. 왜 이 주제를 깊게 봐야 하는가

C 프로그램이 여러 파일로 나뉘고, 오브젝트 파일과 심볼, 링크 과정을 이해하게 되면 다음 단계에서 자연스럽게 만나게 되는 것이 라이브러리다.  
실무에서는 거의 모든 프로그램이 어떤 형태로든 라이브러리를 사용한다.

예를 들어 다음과 같은 질문이 곧 생긴다.

- 표준 라이브러리는 어떻게 연결되는가?
- 내가 만든 여러 오브젝트 파일을 재사용 가능한 묶음으로 만들 수 있는가?
- 왜 어떤 실행 파일은 더 크고 어떤 것은 더 작은가?
- 왜 어떤 프로그램은 실행 시 특정 `.so`나 `.dll`이 필요하다고 하는가?
- 정적 링크와 동적 링크는 무엇이 다른가?
- 배포와 업데이트 관점에서 어떤 방식이 더 유리한가?

이 질문들의 중심에는 **정적 라이브러리(static library)** 와 **동적 라이브러리(dynamic library)** 의 차이가 있다.

입문 단계에서는 보통 이렇게 요약한다.

- 정적 라이브러리: 실행 파일에 포함된다
- 동적 라이브러리: 실행 시 따로 불러온다

이 설명은 방향은 맞다.  
하지만 여기서 멈추면 링크 단계에서 실제로 무슨 일이 일어나는지, 심볼 해결이 언제 어떻게 이루어지는지, 배포와 성능과 ABI가 왜 중요해지는지까지는 연결되지 않는다.

예를 들어 어떤 함수를 사용하는 프로그램이 있다고 하자.

    int main(void) {
        return add(1, 2);
    }

이 `add`가 라이브러리 안에 있다면, 다음이 달라질 수 있다.

- 링크 시점에 코드가 실행 파일 안으로 복사되듯 포함되는가
- 아니면 실행 파일은 참조 정보만 가지고, 실행 시 외부 라이브러리와 연결되는가

즉, 라이브러리는 단순 편의 기능 묶음이 아니라 **링크 전략과 배포 전략의 핵심 단위**다.

이 문서는 정적 라이브러리와 동적 라이브러리가 각각 무엇인지, 링크 관점에서 어떻게 다른지, 실행 파일 크기와 배포와 업데이트에 어떤 차이를 만드는지, 그리고 실무에서 어떤 기준으로 선택해야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

정적 라이브러리와 동적 라이브러리를 큰 구조로 보면 다음과 같다.

### 2.1 라이브러리는 재사용 가능한 코드 묶음이다

라이브러리는 여러 함수와 데이터, 때로는 타입과 인터페이스를 묶어서  
다른 프로그램이 재사용할 수 있게 만든 코드 집합이다.

즉, 라이브러리는 “한 번 구현한 기능을 여러 프로그램에서 다시 쓰기 위한 단위”다.

### 2.2 정적 라이브러리는 링크 시점에 프로그램 안으로 들어간다

정적 라이브러리는 링크 단계에서 필요한 코드가 실행 파일 안으로 결합되는 방식과 연결된다.

즉, 최종 실행 파일을 만들 때 라이브러리의 필요한 오브젝트 코드가 함께 들어간다.

그래서 실행 시에는 외부 라이브러리 파일이 따로 없어도 동작할 수 있다.

### 2.3 동적 라이브러리는 실행 파일과 분리된 채로 남는다

동적 라이브러리는 최종 실행 파일과 완전히 하나로 합쳐지는 것이 아니라,  
실행 시점에 외부 라이브러리 파일과 연결되는 방식을 가진다.

즉, 실행 파일은 라이브러리 전체 코드를 다 품고 있는 것이 아니라  
필요한 심볼을 외부 공유 라이브러리에서 찾도록 설계될 수 있다.

### 2.4 핵심 차이는 “언제 결합되느냐”와 “무엇이 최종 실행 파일 안에 들어가느냐”다

정리하면 다음이 핵심이다.

- 정적 라이브러리 → 링크 시점에 결합
- 동적 라이브러리 → 실행 파일과 분리된 상태 유지, 실행 시 연결

즉, 둘의 차이는 단순 파일 확장자 차이가 아니라 **결합 시점과 배포 모델 차이**다.

## 3. 내부 동작 원리

### 3.1 정적 라이브러리는 오브젝트 파일 묶음에 가깝다

정적 라이브러리는 개념적으로 여러 오브젝트 파일을 하나로 묶은 보관소처럼 이해할 수 있다.

즉, 정적 라이브러리는 “소스 코드”가 들어 있는 것이 아니라  
이미 컴파일된 코드 조각들을 재사용 가능한 형태로 묶은 것이다.

링커는 실행 파일을 만들 때 이 묶음 안에서 필요한 오브젝트 코드만 선택해 가져올 수 있다.

즉, 정적 라이브러리는 **링크 시 재료로 쓰이는 오브젝트 파일 모음**에 가깝다.

### 3.2 정적 링크에서는 필요한 코드가 실행 파일 안으로 들어간다

예를 들어 `add`가 정적 라이브러리 안에 있고, `main`이 이를 사용한다고 하자.

링커는 실행 파일을 만들면서 다음을 할 수 있다.

- `main.o`가 요구하는 `add` 심볼 확인
- 정적 라이브러리 안에서 `add` 정의를 찾음
- 필요한 코드 조각을 실행 파일로 포함

즉, 최종 실행 파일 안에 라이브러리 코드 일부가 직접 들어간다.

### 3.3 동적 라이브러리는 공유 가능한 외부 코드 파일이다

동적 라이브러리는 실행 파일과 별개로 존재하는 공유 코드 파일이다.

즉, 프로그램을 링크할 때 다음과 같은 구조가 가능하다.

- 실행 파일은 “이 심볼은 이 공유 라이브러리에서 해결할 수 있다”는 정보를 가진다
- 실제 코드 본체는 외부 라이브러리 파일에 남아 있다
- 프로그램 실행 시 운영체제 로더와 동적 링커가 해당 라이브러리를 적재하고 심볼을 연결한다

즉, 동적 라이브러리는 실행 파일 밖에 있는 **공유 가능한 실행 코드 자원**이다.

### 3.4 동적 링크는 실행 시점의 결합 요소를 가진다

정적 링크는 빌드가 끝나면 대부분 결합이 완료되어 있다.  
반면 동적 링크는 프로그램 시작 시점 또는 더 늦은 시점에 라이브러리를 적재하고 연결할 수 있다.

즉, 동적 라이브러리는 “링크를 완전히 미루는 것”은 아니더라도  
최종 결합의 일부가 실행 환경에 의존할 수 있다.

### 3.5 심볼 해결 시점이 달라진다

정적 라이브러리에서는 링크 시점에 필요한 심볼 코드가 실행 파일 안으로 들어간다.  
즉, 실행 시점에는 외부 라이브러리 파일을 찾을 필요가 없을 수 있다.

동적 라이브러리에서는 실행 파일이 외부 공유 라이브러리에 의존할 수 있으므로,  
실행 시점에 해당 라이브러리가 존재하고 심볼이 적절히 연결되어야 한다.

즉, 심볼 해결 모델도 완전히 같지 않다.

### 3.6 같은 라이브러리를 여러 프로그램이 공유할 수 있다

동적 라이브러리의 큰 장점 중 하나는 여러 프로그램이 같은 라이브러리 코드를 공유할 수 있다는 점이다.

즉, 각 프로그램이 라이브러리 코드를 실행 파일 안에 중복해서 품지 않고,  
하나의 공유 라이브러리를 함께 사용할 수 있다.

이건 메모리 사용량, 배포, 업데이트 측면에서 큰 차이를 만든다.

## 4. 핵심 구성 요소

### 4.1 static library

정적 라이브러리는 링크 시점에 실행 파일 안으로 필요한 코드가 결합되는 라이브러리다.

즉, 정적 라이브러리는 “빌드 시 흡수되는 코드 묶음”으로 이해할 수 있다.

### 4.2 dynamic library

동적 라이브러리는 실행 파일과 별개로 존재하며,  
실행 시점에 적재되고 심볼이 연결될 수 있는 공유 라이브러리다.

즉, 동적 라이브러리는 “실행 시 외부에서 결합되는 코드 자원”이다.

### 4.3 linker

링커는 정적 라이브러리를 사용할 때 라이브러리 안의 필요한 오브젝트 코드를 실행 파일에 결합한다.  
동적 라이브러리에서는 실행 파일이 해당 공유 라이브러리에 의존하도록 정보와 참조를 구성한다.

### 4.4 shared library

공유 라이브러리는 보통 동적 라이브러리와 강하게 연결된다.  
즉, 여러 프로그램이 같은 라이브러리를 공유해서 사용할 수 있게 하는 모델이다.

### 4.5 symbol resolution

라이브러리 사용의 핵심은 결국 심볼 해결이다.

- 정적 라이브러리 → 링크 시 해결
- 동적 라이브러리 → 실행 시점 의존 요소 포함

즉, 라이브러리 이해도 결국 심볼 모델 이해와 연결된다.

### 4.6 ABI

동적 라이브러리는 특히 ABI 안정성과 매우 밀접하다.  
실행 파일과 외부 라이브러리가 나중에 연결되므로, 인터페이스뿐 아니라 바이너리 호환성도 중요해진다.

## 5. 실행 흐름 또는 처리 순서

다음 구조를 기준으로 생각해보자.

    // main.c
    int add(int a, int b);

    int main(void) {
        return add(1, 2);
    }

그리고 `add`는 라이브러리 안에 있다고 하자.

### 5.1 정적 라이브러리 사용 시

정적 라이브러리의 경우 대략 다음 흐름으로 이해할 수 있다.

1. `main.c`를 컴파일해 `main.o` 생성
2. 정적 라이브러리 안에 있는 오브젝트 코드 준비
3. 링커가 `main.o`가 요구하는 `add` 심볼 확인
4. 정적 라이브러리에서 `add` 정의를 찾음
5. 필요한 코드가 최종 실행 파일 안으로 들어감
6. 실행 시 외부 라이브러리 없이도 동작 가능

즉, 정적 라이브러리는 링크가 끝나면 프로그램 안으로 흡수된 형태가 된다.

### 5.2 동적 라이브러리 사용 시

동적 라이브러리의 경우는 다르다.

1. `main.c`를 컴파일해 `main.o` 생성
2. 링커가 `add`가 외부 공유 라이브러리에서 해결될 수 있음을 반영
3. 최종 실행 파일은 외부 공유 라이브러리에 대한 의존 정보를 가짐
4. 프로그램 실행 시 운영체제가 해당 공유 라이브러리를 적재
5. 동적 로더/링커가 `add` 심볼을 연결
6. 실행 중 라이브러리 코드 사용

즉, 실행 파일과 라이브러리가 완전히 하나가 되지 않는다.

### 5.3 배포 관점 차이

정적 라이브러리라면 최종 실행 파일 하나만 배포해도 충분할 수 있다.  
반면 동적 라이브러리라면 실행 파일 외에 적절한 공유 라이브러리도 함께 존재해야 한다.

즉, 라이브러리 방식은 빌드 결과물뿐 아니라 배포 방식에도 직접 영향을 준다.

## 6. 성능 / 최적화 관점

### 6.1 정적 라이브러리는 실행 파일이 커질 수 있다

정적 링크를 하면 필요한 라이브러리 코드가 실행 파일 안으로 들어간다.  
즉, 프로그램마다 같은 라이브러리 코드가 중복 포함될 수 있다.

따라서 실행 파일 크기가 커질 수 있다.

### 6.2 동적 라이브러리는 코드 공유에 유리하다

동적 라이브러리는 여러 프로그램이 같은 라이브러리를 공유할 수 있다.  
즉, 각 프로그램이 같은 라이브러리 코드를 중복해서 품지 않아도 된다.

이건 메모리 사용량과 디스크 공간 효율 측면에서 장점이 될 수 있다.

### 6.3 정적 링크는 배포 단순성이 장점일 수 있다

외부 공유 라이브러리에 대한 의존이 적으면 배포가 단순해질 수 있다.  
실행 파일 하나만 있으면 되는 상황이 생기기 때문이다.

즉, 정적 링크는 실행 환경 의존성을 줄이는 방향에서 유리할 수 있다.

### 6.4 동적 링크는 업데이트 유연성이 장점일 수 있다

라이브러리가 별도 파일로 분리되어 있으면,  
실행 파일을 다시 빌드하지 않고 라이브러리만 교체하는 식의 업데이트 전략이 가능할 수 있다.

즉, 동적 라이브러리는 재배포와 유지보수 측면에서 유연성을 줄 수 있다.

### 6.5 ABI 불일치가 동적 라이브러리에서 더 민감해진다

동적 라이브러리는 실행 파일과 외부 라이브러리가 나중에 결합되므로,  
인터페이스가 같아 보여도 실제 바이너리 호환성이 깨지면 문제가 생길 수 있다.

즉, 동적 라이브러리는 유연한 대신 ABI 관리 부담이 더 커질 수 있다.

## 7. 실무에서 중요한 판단 기준

### 7.1 배포 단순성이 중요하면 정적 라이브러리를 고려할 수 있다

실행 환경을 최대한 단순하게 만들고 싶다면,  
필요한 코드가 실행 파일 안에 포함되는 정적 링크가 더 편할 수 있다.

특히 외부 설치 환경을 통제하기 어려운 경우 이런 판단이 중요해진다.

### 7.2 여러 프로그램이 같은 기능을 공유해야 하면 동적 라이브러리가 유리할 수 있다

공통 기능을 여러 프로그램이 함께 쓰고,  
업데이트와 교체를 더 유연하게 하고 싶다면 동적 라이브러리가 유리할 수 있다.

즉, 공유성과 유지보수성을 중요하게 보면 동적 모델이 강하다.

### 7.3 라이브러리 업데이트와 ABI 안정성을 함께 봐야 한다

동적 라이브러리는 교체가 쉬운 대신,  
실행 파일이 기대하는 ABI와 실제 라이브러리 ABI가 어긋나면 문제가 생긴다.

즉, 동적 라이브러리를 설계할 때는 단순 함수 시그니처뿐 아니라  
바이너리 호환성까지 의식해야 한다.

### 7.4 디버깅과 운영 환경 차이도 고려해야 한다

정적 링크는 실행 환경 차이가 줄어드는 장점이 있다.  
반면 동적 링크는 실제 운영 환경에 설치된 라이브러리 버전에 따라 동작 차이가 생길 수 있다.

즉, 운영 일관성이 중요한지, 공유성과 유연성이 중요한지도 선택 기준이 된다.

### 7.5 “무조건 하나가 더 좋다”는 식으로 보면 안 된다

정적 라이브러리와 동적 라이브러리는 우열 관계라기보다 trade-off 관계다.

- 배포 단순성
- 실행 파일 크기
- 메모리 공유
- 업데이트 유연성
- ABI 관리 비용
- 환경 의존성

이 요소를 함께 봐야 한다.

## 8. 더 깊게 볼 포인트

### 8.1 ABI 안정성

동적 라이브러리를 진지하게 다루려면 ABI가 왜 중요한지 더 깊게 봐야 한다.

### 8.2 loader와 dynamic linker

프로그램 실행 시 공유 라이브러리가 어떻게 적재되고 심볼이 어떻게 해결되는지를 더 깊게 볼 수 있다.

### 8.3 PIC와 공유 라이브러리 코드 생성

공유 라이브러리용 코드가 왜 별도 고려가 필요한지, 위치 독립 코드 개념으로 확장할 수 있다.

### 8.4 라이브러리 버전 관리

심볼 버전, 라이브러리 교체 전략, 배포 정책 같은 실무 주제로 이어질 수 있다.

### 8.5 플러그인 구조와 런타임 로딩

동적 라이브러리 개념은 단순 공유 코드뿐 아니라 플러그인 시스템과 런타임 확장 구조로도 이어질 수 있다.

## 정리

정적 라이브러리와 동적 라이브러리는 둘 다 재사용 가능한 코드 묶음이지만, 결합 방식과 운영 방식이 다르다.

핵심은 다음과 같다.

- 정적 라이브러리는 링크 시점에 필요한 코드가 실행 파일 안으로 들어간다.
- 동적 라이브러리는 실행 파일과 분리된 채로 남고, 실행 시점에 연결된다.
- 정적 링크는 배포가 단순해질 수 있지만 실행 파일이 커질 수 있다.
- 동적 라이브러리는 코드 공유와 업데이트 유연성에 유리할 수 있다.
- 동적 라이브러리는 ABI와 실행 환경 의존성을 더 신경 써야 한다.
- 둘 중 하나가 무조건 더 좋은 것이 아니라, 배포/성능/유지보수/환경 제약에 따라 선택해야 한다.

즉, 정적 라이브러리와 동적 라이브러리를 이해한다는 것은  
단순히 “파일 형태 두 개를 안다”가 아니라  
**코드 재사용이 링크 구조, 배포 방식, 실행 환경, ABI 관리와 어떻게 연결되는지 이해한다**는 뜻이다.

이 관점이 잡혀야 C 프로젝트를 단일 프로그램 수준을 넘어 라이브러리 기반 구조와 배포 가능한 시스템 수준으로 확장할 수 있다.
