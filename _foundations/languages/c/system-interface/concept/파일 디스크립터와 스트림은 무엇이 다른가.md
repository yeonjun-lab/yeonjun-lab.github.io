## 1. 왜 이 주제를 공부했는가

C에서 파일을 다루다 보면 `FILE *`와 `int fd`가 모두 나온다.  
둘 다 파일을 다루는 것처럼 보여도 추상화 수준과 책임이 다르다.

즉, 파일 디스크립터와 스트림 차이를 이해해야  
버퍼링, I/O 비용, API 선택이 선명해진다.

## 2. 정의

파일 디스크립터는 운영체제 쪽 저수준 파일 핸들에 가깝고,  
스트림은 표준 라이브러리가 제공하는 고수준 입출력 객체다.

## 3. 핵심 개념

### 3.1 파일 디스크립터는 더 직접적인 자원 핸들이다

정수 값으로 표현되는 경우가 많고,  
운영체제 인터페이스와 더 가깝게 연결된다.

### 3.2 스트림은 버퍼링과 상태 관리를 포함한다

`FILE *`는 단순 파일 주소가 아니라  
라이브러리 상태와 버퍼를 함께 관리하는 객체다.

### 3.3 같은 파일도 서로 다른 층에서 볼 수 있다

같은 대상 파일이라도  
고수준 스트림과 저수준 핸들로 다룰 수 있다.

## 4. 예시

```c
FILE *fp = fopen("data.txt", "r");
```

```c
int fd = open("data.txt", ...);
```

둘 다 파일을 열지만,  
하나는 스트림 모델이고 다른 하나는 저수준 핸들 모델이다.

## 5. 헷갈리기 쉬운 부분

### 5.1 둘을 같은 파일 객체라고 생각하는 문제

대상은 같아도 계층이 다르다.

### 5.2 고수준 API가 항상 더 느리다고 생각하는 문제

버퍼링 덕분에 오히려 유리한 경우도 많다.

### 5.3 섞어 써도 언제나 안전하다고 생각하는 문제

버퍼와 위치 상태가 어긋날 수 있어 주의가 필요하다.

## 6. 어디서부터 직관과 실제 규칙이 갈리는가

### 6.1 스트림은 파일 그 자체가 아니라 라이브러리 객체다

`FILE *`는 파일 내용 포인터가 아니다.

### 6.2 파일 디스크립터는 더 직접적이지만 더 많은 책임도 준다

버퍼링과 편의 기능이 줄어드는 대신 제어가 늘어난다.

### 6.3 같은 "읽기/쓰기"라도 비용 모델은 다를 수 있다

스트림은 중간 버퍼가 있고, 저수준 I/O는 더 직접적이다.

## 7. 실무 관점

### 7.1 일반 텍스트/파일 처리라면 스트림이 먼저다

편의성과 이식성이 좋다.

### 7.2 더 직접적인 제어가 필요할 때 파일 디스크립터를 본다

polling, nonblocking, 소켓, 저수준 I/O에서는 이쪽이 더 자연스럽다.

### 7.3 계층 혼용은 신중히 한다

같은 자원을 두 층에서 함께 다룰 때 상태 차이를 의식해야 한다.

## 8. 판단 체크리스트

- 지금 필요한 것이 스트림 편의인지 저수준 제어인지 정했는가
- `FILE *`를 파일 자체가 아니라 라이브러리 상태 객체로 이해하고 있는가
- 파일 디스크립터가 더 직접적인 자원 핸들이라는 점을 의식하고 있는가
- 버퍼링과 상태 관리 차이를 비용 모델에 포함하고 있는가
- 두 계층을 섞어 쓸 때 상태 어긋남을 고려하고 있는가

## 9. 다음에 볼 것

이 문서 다음에는 pagination 순서상 아래 흐름으로 가는 것이 가장 자연스럽다.

### 9.1 표준 라이브러리와 시스템 콜의 차이
가장 바로 이어지는 문서다. 방금 본 내용을 같은 `system-interface` 축 안에서 바로 확장한다.

### 9.2 파일 입출력은 내부적으로 어떻게 동작하는가
한 단계 더 읽으면 방금 이어진 설명이 같은 주제 안에서 어떻게 깊어지는지 더 분명해진다.

### 9.3 왜 같은 c 코드도 성능이 다르게 나오는가
세 번째 문서까지 보면 현재 문서가 전체 C 시리즈의 어디에 놓여 있는지 읽기 흐름이 더 안정된다.

## 정리

파일 디스크립터와 스트림은 둘 다 파일을 다루지만 같은 층의 도구가 아니다.  
핵심은 **저수준 자원 핸들**과 **고수준 라이브러리 추상화**를 구분하는 것이다.
