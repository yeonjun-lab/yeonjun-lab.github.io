## 1. 왜 이 주제를 공부했는가

C를 배우면 `printf`, `fopen`, `malloc`, `read` 같은 이름들이 한 덩어리처럼 보인다.  
하지만 실제로는 사용자 코드, 표준 라이브러리, 시스템 콜, 커널이 서로 다른 층에 있다.

즉, 사용자 프로그램이 운영체제와 어떻게 연결되는지 이해해야  
입출력, 메모리, 성능, 이식성을 제대로 읽을 수 있다.

## 2. 정의

사용자 프로그램이 운영체제와 연결된다는 것은  
프로그램이 직접 하드웨어를 제어하는 것이 아니라  
운영체제가 제공하는 인터페이스와 라이브러리 계층을 거쳐 자원을 사용한다는 뜻이다.

보통 큰 구조는 다음과 같다.

- 사용자 코드
- C 표준 라이브러리 또는 런타임
- 시스템 콜 인터페이스
- 커널
- 하드웨어 / 파일 시스템 / 네트워크 장치

## 3. 핵심 개념

### 3.1 사용자 프로그램은 보통 사용자 공간에서 먼저 동작한다

`printf`를 호출했다고 곧바로 커널이 실행되는 것은 아니다.  
먼저 사용자 공간 라이브러리 로직이 돌 수 있다.

### 3.2 운영체제 자원은 시스템 콜 경계 뒤에 있다

파일, 소켓, 프로세스, 가상 메모리 같은 자원은  
사용자 코드가 임의로 직접 만지는 것이 아니라  
정해진 인터페이스를 통해 요청한다.

### 3.3 라이브러리는 추상화와 이식성 계층이다

표준 라이브러리는 운영체제 차이를 일부 감추고  
더 편한 API와 버퍼링, 포맷 처리, 상태 관리를 제공한다.

## 4. 예시

```c
printf("hello\n");
```

겉보기에는 단순 출력 함수지만,  
실제로는 포맷 처리와 버퍼링이 먼저 일어나고  
필요한 시점에 실제 쓰기 요청이 이어질 수 있다.

```c
int fd = open("data.txt", ...);
read(fd, buf, n);
```

이쪽은 더 직접적으로 운영체제 인터페이스와 맞닿아 있다.

즉, 둘 다 “파일/출력”처럼 보이지만  
추상화 수준이 다르다.

## 5. 헷갈리기 쉬운 부분

### 5.1 C 함수가 곧 운영체제 기능이라고 생각하는 문제

표준 라이브러리 함수와 시스템 콜은 같은 것이 아니다.

### 5.2 라이브러리가 있으면 운영체제가 안 중요하다고 생각하는 문제

라이브러리는 추상화 계층일 뿐이고,  
그 아래 자원 모델과 비용 구조는 여전히 운영체제에 연결된다.

### 5.3 함수 호출과 커널 진입을 같은 사건으로 보는 문제

사용자 공간 함수 호출과 시스템 콜 경계 통과는 다르다.

## 6. 어디서부터 직관과 실제 규칙이 갈리는가

### 6.1 "함수를 호출했다"와 "커널에 요청했다"는 다른 말이다

표면적으로는 둘 다 함수 호출처럼 보이지만,  
실제로는 사용자 공간 로직이 먼저 개입할 수 있다.

### 6.2 표준 라이브러리와 시스템 콜은 역할이 다르다

하나는 편한 사용자 공간 인터페이스이고,  
다른 하나는 커널 자원 접근 경계다.

### 6.3 이식성과 직접 제어는 같은 방향이 아닐 수 있다

더 높은 추상화는 보통 이식성을 높이고,  
더 낮은 추상화는 제어력을 높인다.

## 7. 실무 관점

### 7.1 먼저 추상화 수준을 정한다

스트림 API로 충분한지, 파일 디스크립터 수준 제어가 필요한지 먼저 판단한다.

### 7.2 버퍼링과 시스템 콜 비용을 함께 본다

작은 입출력을 반복할 때는 라이브러리 버퍼링이 유리할 수 있다.

### 7.3 운영체제 의존성은 설계 비용이다

직접 제어가 필요하면 그만큼 휴대성과 복잡성을 감수해야 한다.

## 8. 판단 체크리스트

- 지금 다루는 것이 사용자 공간 라이브러리 API인가, 시스템 콜 경계인가
- 함수 호출과 커널 진입을 같은 것으로 보고 있지 않은가
- 버퍼링, 포맷 처리, 상태 관리 같은 중간 계층을 의식하고 있는가
- 이식성과 직접 제어 중 무엇이 더 중요한지 먼저 정했는가
- 운영체제 자원 모델과 라이브러리 추상화를 섞어 읽고 있지 않은가

## 9. 다음에 볼 것

이 문서 다음에는 pagination 순서상 아래 흐름으로 가는 것이 가장 자연스럽다.

### 9.1 파일 디스크립터와 스트림은 무엇이 다른가
가장 바로 이어지는 문서다. 방금 본 내용을 같은 `system-interface` 축 안에서 바로 확장한다.

### 9.2 표준 라이브러리와 시스템 콜의 차이
한 단계 더 읽으면 방금 이어진 설명이 같은 주제 안에서 어떻게 깊어지는지 더 분명해진다.

### 9.3 파일 입출력은 내부적으로 어떻게 동작하는가
세 번째 문서까지 보면 현재 문서가 전체 C 시리즈의 어디에 놓여 있는지 읽기 흐름이 더 안정된다.

## 정리

사용자 프로그램은 운영체제와 직접 맞닿아 있는 것처럼 보여도,  
실제로는 **사용자 코드 -> 라이브러리 -> 시스템 콜 -> 커널**의 층을 거친다.

이 구조를 잡아야 입출력, 성능, 이식성, 시스템 프로그래밍 문서가 한 축으로 연결된다.
