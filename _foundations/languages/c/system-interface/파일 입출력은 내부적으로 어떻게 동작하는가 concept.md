## 정의

C에서 파일 입출력은 단순히 "파일에서 읽고 파일에 쓰는 것"이 아니다.  
표면적으로는 `fopen`, `fread`, `fwrite`, `fprintf`, `fclose` 같은 표준 라이브러리 함수로 다루지만, 내부적으로는 **사용자 공간의 스트림 객체, 버퍼링, 운영체제 파일 디스크립터, 시스템 콜**이 함께 작동하는 계층 구조다.

핵심은 다음이다.

1. 프로그램은 보통 파일 자체를 직접 다루지 않고 스트림(stream) 인터페이스를 다룬다
2. 표준 라이브러리는 내부 버퍼를 두어 I/O 호출을 모은다
3. 실제 장치/파일 접근은 결국 운영체제의 시스템 인터페이스를 통해 일어난다
4. 읽기/쓰기 위치, EOF, 에러 상태, 버퍼 flush 같은 개념이 함께 움직인다

즉 파일 입출력을 이해한다는 것은  
단순한 함수 사용법을 넘어서 **텍스트/버퍼/스트림/커널 I/O가 어떻게 연결되는가**를 이해하는 것이다.

---

## 먼저 큰 그림부터

보통 C에서 파일을 다룰 때는 이렇게 시작한다.

    FILE *fp = fopen("data.txt", "r");

겉보기에는 단순히 파일을 연 것처럼 보인다.  
하지만 내부적으로는 보통 다음 같은 일이 이어진다.

- 경로와 모드를 해석
- 운영체제에 실제 파일 열기 요청
- 라이브러리 내부 `FILE` 구조체 준비
- 읽기/쓰기 버퍼 준비
- 현재 파일 위치와 상태 정보 관리 시작

즉 `FILE *`는 그냥 "파일 포인터"가 아니라,  
**표준 라이브러리가 파일 I/O를 관리하기 위해 사용하는 스트림 객체 핸들**이다.

---

## 핵심 개념

### 1. 표준 I/O는 스트림 인터페이스다

C 표준 라이브러리의 파일 I/O는 스트림(stream) 개념을 중심으로 한다.

예시:

    FILE *fp = fopen("log.txt", "w");
    fprintf(fp, "hello\n");
    fclose(fp);

여기서 프로그램은 디스크 블록이나 커널 내부 파일 객체를 직접 다루지 않는다.  
대신 `FILE *`를 통해 "순차적으로 읽고 쓰는 데이터 흐름"을 다룬다.

즉 표준 I/O의 관점은 다음과 같다.

- 입력 스트림
- 출력 스트림
- 현재 읽기/쓰기 위치
- 내부 버퍼
- EOF / error 상태

이것은 저수준 파일 디스크립터 인터페이스보다 더 고수준의 추상화다.

---

### 2. `fopen`은 단순히 포인터 하나를 만드는 것이 아니다

예시:

    FILE *fp = fopen("data.bin", "rb");

이 호출은 보통 아래 같은 의미를 가진다.

- 운영체제에 해당 경로 파일을 열 수 있는지 요청
- 성공하면 OS가 파일 핸들 또는 디스크립터를 준비
- 라이브러리는 이를 감싸는 `FILE` 객체를 초기화
- 버퍼링 모드와 내부 상태를 준비
- 반환값으로 `FILE *`를 준다

즉 `FILE *`는 파일 내용 그 자체를 가리키는 포인터가 아니라,  
**파일 입출력 상태를 담는 라이브러리 객체에 대한 핸들**로 이해해야 한다.

---

### 3. 실제 읽기/쓰기는 버퍼를 사이에 둔다

예시:

    char buf[128];
    size_t n = fread(buf, 1, sizeof(buf), fp);

이 호출이 항상 "즉시 커널에 한 번 read 시스템 콜"과 일대일 대응되는 것은 아니다.  
표준 라이브러리는 보통 내부 버퍼를 둔다.

예를 들어 읽기에서는:

- 라이브러리 버퍼가 비어 있으면 더 큰 덩어리를 시스템에서 읽어 옴
- 그 중 일부를 사용자 `buf`에 복사
- 나머지는 내부 버퍼에 보관

쓰기에서도:

- 사용자가 쓴 데이터를 내부 버퍼에 모아 둠
- 버퍼가 차거나 flush 시점이 되면 실제 시스템 쓰기 호출 수행

즉 표준 I/O는 보통 **즉시 I/O**가 아니라 **버퍼링된 I/O**다.

---

## 왜 버퍼링을 하는가

### 1. 시스템 콜 횟수를 줄이기 위해서다

시스템 콜은 사용자 공간과 커널 공간 경계를 넘는 작업이다.  
따라서 단순 메모리 복사보다 비용이 큰 편이다.

만약 한 글자 쓸 때마다 즉시 커널에 요청하면 매우 비효율적일 수 있다.

예시:

    fputc('H', fp);
    fputc('e', fp);
    fputc('l', fp);
    fputc('l', fp);
    fputc('o', fp);

이 다섯 번이 곧바로 다섯 번 시스템 콜로 내려가면 비효율적이다.  
그래서 라이브러리는 내부 버퍼에 모았다가 한 번에 쓸 수 있다.

즉 버퍼링은 성능 최적화를 위한 핵심 장치다.

---

### 2. 텍스트 I/O를 더 편하게 하기 위해서다

`printf`, `fprintf`, `fgets`, `fputs` 같은 함수는 사람이 읽는 텍스트 단위 작업과 잘 맞는다.  
이때 버퍼링이 있으면 포맷팅 결과를 모았다가 효율적으로 처리할 수 있다.

즉 표준 I/O는 단순 바이트 전송보다  
**사람 친화적 텍스트 처리 + 효율적 전송**을 함께 겨냥한다.

---

## 파일 디스크립터와 `FILE *`는 왜 다른가

이 차이를 꼭 구분해야 한다.

### 1. 파일 디스크립터

운영체제가 프로세스에게 제공하는 저수준 파일 핸들이다.  
보통 "정수 번호" 형태로 다뤄진다.

이 수준에서는 보통 다음을 직접 생각한다.

- 몇 번 핸들인가
- 현재 오프셋
- 읽기/쓰기 권한
- OS 자원 테이블과 연결

### 2. `FILE *`

표준 라이브러리의 고수준 스트림 객체다.

여기에는 보통 다음이 추가된다.

- 버퍼 포인터
- 버퍼 상태
- EOF 플래그
- 에러 플래그
- 텍스트/바이너리 모드 처리
- 포맷팅 I/O 인터페이스

즉 둘의 관계는 대략 다음처럼 이해하면 된다.

- 파일 디스크립터: OS 레벨 핸들
- `FILE *`: 라이브러리 레벨 스트림 관리 객체

즉 `FILE *`는 파일 디스크립터 위에 얹힌 더 큰 추상화다.

---

## 읽기는 내부적으로 어떻게 진행되는가

예시:

    char buf[10];
    size_t n = fread(buf, 1, 10, fp);

개념적으로는 다음 같은 과정으로 이해하면 좋다.

1. 스트림이 읽기 가능한 상태인지 확인
2. 내부 버퍼에 남은 데이터가 충분한지 확인
3. 부족하면 실제 시스템 읽기 호출로 데이터를 더 가져옴
4. 내부 버퍼에서 사용자 버퍼 `buf`로 복사
5. 읽은 바이트 수를 반환
6. EOF 또는 에러 상태 갱신

즉 사용자는 `fread` 한 번만 보지만, 내부적으로는  
버퍼 검사, 시스템 호출, 상태 갱신이 함께 일어난다.

---

## 쓰기는 내부적으로 어떻게 진행되는가

예시:

    const char *msg = "hello";
    fwrite(msg, 1, 5, fp);

개념적으로는 다음처럼 이해할 수 있다.

1. 스트림이 쓰기 가능한 상태인지 확인
2. 사용자 버퍼 내용을 라이브러리 내부 버퍼에 복사
3. 버퍼가 가득 찼거나 특정 조건이면 실제 시스템 쓰기 호출
4. 기록된 바이트 수를 반환하거나 에러 표시
5. 필요 시 파일 위치 갱신

즉 `fwrite`는 보통 곧바로 디스크에 "확정 기록"된다는 보장이 아니라,  
먼저 라이브러리 버퍼에 들어갈 수 있다.

이 때문에 flush와 close가 중요해진다.

---

## `fflush`와 `fclose`가 왜 중요한가

### 1. `fflush`

예시:

    fflush(fp);

이 함수는 출력 버퍼에 남아 있는 데이터를 가능한 한 실제 시스템 쓰기로 내보내도록 한다.

즉 `fflush`는 "라이브러리 버퍼 → OS 쪽"으로 밀어내는 역할과 연결된다.

다만 이것이 곧 디스크 물리 매체에 영구 저장까지 보장하는지 여부는 더 낮은 계층 문제와 연결될 수 있다.  
기초 단계에서는 우선 "표준 I/O 버퍼를 비운다"로 이해하면 된다.

---

### 2. `fclose`

예시:

    fclose(fp);

이 함수는 단순 포인터 해제가 아니다.  
보통 다음을 포함한다.

- 남은 출력 버퍼 flush 시도
- 스트림 상태 정리
- 내부 자원 해제
- OS 파일 핸들 닫기

즉 `fclose`는 "이 스트림 사용 종료"의 완결 단계다.

이걸 빼먹으면 버퍼에 남은 데이터가 기대대로 기록되지 않거나 자원 누수가 생길 수 있다.

---

## EOF와 에러는 왜 구분해야 하는가

파일 읽기에서 흔한 혼동이다.

예시:

    int ch = fgetc(fp);

읽기 실패처럼 보여도 원인은 두 가지일 수 있다.

- 파일 끝에 도달함
- 실제 에러 발생

즉 "더 이상 읽을 수 없다"는 결과만 보고는  
EOF인지 에러인지 자동으로 구분되지 않을 수 있다.

그래서 표준 I/O는 보통 아래 같은 상태를 따로 관리한다.

- EOF 상태
- error 상태

즉 파일 끝과 입출력 실패는 같은 것이 아니다.

---

## 파일 위치는 어떻게 관리되는가

스트림은 보통 현재 읽기/쓰기 위치를 가진다.

예시:

    fseek(fp, 0, SEEK_SET);

이런 함수는 스트림의 현재 위치를 바꾼다.

즉 파일 I/O는 단순히 "열고 읽고 닫기"가 아니라,  
현재 어디를 보고 있는지라는 위치 상태를 포함한다.

읽기/쓰기 호출 후에는 이 위치가 앞으로 이동할 수 있다.  
그리고 버퍼 상태와 실제 OS 오프셋 관리가 함께 얽힐 수 있다.

즉 파일 I/O는 **데이터 + 상태 + 위치**를 함께 다루는 모델이다.

---

## 텍스트 모드와 바이너리 모드는 왜 나뉘는가

예시:

    fopen("a.txt", "r");
    fopen("a.bin", "rb");

플랫폼에 따라 텍스트 모드와 바이너리 모드는 다르게 취급될 수 있다.  
특히 줄바꿈 표현, EOF 처리 관례 등이 얽힐 수 있다.

기초적으로는 아래처럼 잡으면 된다.

- 텍스트 모드: 사람이 읽는 텍스트 관점 처리 가능
- 바이너리 모드: 바이트를 있는 그대로 다룸

즉 바이너리 데이터, 직렬화 포맷, 이미지/압축 데이터 등은  
일반적으로 바이너리 모드 관점으로 다루는 것이 중요하다.

---

## 왜 표준 I/O와 저수준 I/O를 섞으면 조심해야 하는가

`FILE *` 기반 스트림과 파일 디스크립터 기반 저수준 I/O는 같은 OS 자원을 다른 계층에서 다룰 수 있다.  
그런데 표준 I/O는 내부 버퍼를 갖고 있으므로, 외부에서 저수준 I/O를 섞으면 상태가 어긋날 수 있다.

즉 다음이 문제가 될 수 있다.

- 라이브러리 버퍼가 아직 반영되지 않은 상태
- 실제 OS 오프셋과 라이브러리 인식 상태 차이
- 읽기/쓰기 순서 혼동

기초 단계에서는 원칙적으로  
**하나의 자원은 한 계층 방식으로 일관되게 다루는 편이 안전하다**고 이해하면 좋다.

---

## 자주 하는 오해

### 오해 1. `FILE *`는 파일 내용 메모리를 가리키는 포인터다

아니다.  
`FILE *`는 보통 스트림 상태를 담는 라이브러리 객체 핸들이다.

---

### 오해 2. `fwrite`를 호출하면 곧바로 디스크에 기록된다

항상 아니다.  
표준 I/O는 버퍼링을 할 수 있으므로, 먼저 라이브러리 버퍼에 들어갈 수 있다.

---

### 오해 3. EOF는 에러다

아니다.  
EOF는 정상적인 파일 끝 도달일 수 있다.  
에러는 별도의 상태다.

---

### 오해 4. `fclose`는 그냥 포인터만 정리한다

아니다.  
보통 flush, 자원 정리, 핸들 닫기까지 포함한다.

---

### 오해 5. 텍스트 파일과 바이너리 파일은 내부적으로 완전히 같다

논리적으로는 둘 다 바이트 집합이지만,  
라이브러리와 플랫폼이 텍스트 모드에서 별도 처리 규약을 둘 수 있다.

즉 프로그램 관점에서는 다르게 다뤄야 할 수 있다.

---

## 실무 관점

### 1. 왜 로그가 바로 안 찍히는지 이해하게 된다

출력 버퍼링을 이해하지 못하면  
"분명 `printf` 했는데 왜 바로 안 보이지?" 같은 현상이 헷갈린다.

즉 파일과 표준 출력 모두 버퍼링 정책을 의식해야 한다.

---

### 2. 데이터 유실과 flush 타이밍을 이해하게 된다

버퍼에만 있고 아직 실제 기록이 안 된 상태에서 프로그램이 비정상 종료되면  
기대한 출력이 남지 않을 수 있다.

즉 중요한 출력에서는 flush 전략이 중요하다.

---

### 3. 성능 분석이 가능해진다

작은 단위로 너무 자주 읽고 쓰면  
실제 시스템 호출 수와 메모리 복사 비용이 커질 수 있다.

즉 다음을 판단할 수 있게 된다.

- 버퍼 크기를 키워야 하는가
- 텍스트 I/O 대신 바이너리 I/O가 맞는가
- 표준 I/O로 충분한가
- 더 낮은 계층 제어가 필요한가

---

### 4. 직렬화/파서/로그 시스템과 직접 연결된다

파일 I/O는 단순 예제가 아니라 아래 전부의 기반이다.

- 설정 파일 파서
- 바이너리 포맷 읽기
- 로그 기록기
- 데이터 덤프
- 네트워크와 유사한 스트림 처리 사고

즉 파일 I/O를 이해하면 "바이트 흐름을 다루는 사고"가 생긴다.

---

### 5. 다음 문서들과 직접 연결된다

이 문서는 아래 주제와 자연스럽게 이어진다.

- 버퍼 오버플로우는 어떻게 생기는가
- 텍스트 직렬화와 바이너리 직렬화는 어떻게 다른가
- length-prefixed format은 왜 필요한가
- 표준 라이브러리와 시스템 콜의 차이
- 직렬화와 역직렬화는 왜 단순 memcpy가 아닌가

즉 시스템 인터페이스 축에서 직렬화/안전성 축으로 넘어가는 다리 역할을 한다.

---

## 정리

C의 파일 입출력은 단순히 파일에서 읽고 쓰는 함수 호출이 아니다.  
그것은 스트림 객체, 버퍼링, 파일 위치, EOF/에러 상태, 그리고 운영체제 파일 인터페이스가 함께 작동하는 구조다.

핵심은 다음과 같다.

- `FILE *`는 스트림 관리 객체에 대한 핸들이다
- 표준 I/O는 내부 버퍼를 둔다
- 실제 장치 접근은 결국 운영체제 인터페이스로 이어진다
- 읽기와 쓰기는 상태와 위치를 함께 관리한다
- `fflush`와 `fclose`는 출력 완료와 자원 정리에 중요하다

즉 파일 입출력을 이해한다는 것은  
"함수 이름을 외우는 것"이 아니라 **사용자 공간 버퍼링과 운영체제 I/O가 어떻게 맞물리는지 이해하는 것**이다.

이 관점이 잡히면 이후의 문자열 처리, 버퍼 안전성, 직렬화, 파서 구현도 훨씬 현실적으로 보이기 시작한다.
