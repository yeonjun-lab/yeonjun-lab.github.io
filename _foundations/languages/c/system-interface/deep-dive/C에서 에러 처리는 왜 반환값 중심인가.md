## 1. 왜 이 주제를 깊게 봐야 하는가

C를 처음 배울 때 많은 사람은 에러 처리를 “실패하면 `-1`이나 `NULL`을 반환한다” 정도로 기억한다.  
이 설명은 틀리지 않지만, 너무 얕다. 실무에서는 바로 다음 질문이 따라온다.

- 왜 반환값으로 실패를 표현하는가
- 왜 예외(exception) 같은 구조가 기본이 아닌가
- 왜 상세 에러는 `errno` 같은 별도 채널로 나누는가
- 왜 out-parameter, 상태 코드, sentinel value가 섞여 있는가
- 왜 자원 정리와 에러 전파가 코드 구조의 중심이 되는가

이 질문에 답하지 못하면 C의 시스템 인터페이스를 계속 피상적으로만 보게 된다.  
`open`, `read`, `write`, `malloc`, `fopen`, `strtol`, `pthread_*` 계열 함수가 각각 왜 그렇게 설계되어 있는지, 왜 어떤 함수는 `errno`를 쓰고 어떤 함수는 직접 상태 코드를 반환하는지, 왜 `goto cleanup`이 실무에서 자연스럽게 나오는지 모두 흐려진다.

이 주제를 깊게 봐야 하는 이유는 세 가지다.

첫째, C의 반환값 중심 에러 처리는 우연한 습관이 아니라 **언어의 제약과 시스템 인터페이스의 요구가 맞물려 형성된 설계 방식**이다.  
즉, 단순히 “예외가 없어서”라고만 설명하면 절반만 본 것이다.

둘째, 반환값 중심이라는 말은 “에러도 반환값으로 준다”가 아니라, **함수 호출의 주 결과와 실패 판단을 하나의 명시적 제어 흐름 위에 놓는다**는 뜻에 가깝다.  
이것이 왜 저수준 시스템 코드와 잘 맞는지 이해해야 한다.

셋째, 반환값 중심 모델은 장점만 있는 것이 아니다.  
성공 값과 실패 신호가 같은 채널에 실리기 때문에 구분 규약이 필요하고, 상세 원인은 별도 채널로 분리되어야 하며, 자원 해제와 실패 전파를 호출자가 직접 구조화해야 한다. 즉, 이 모델은 강력하지만 비용이 있다.

이 문서의 목표는 “C에는 예외가 없으므로 반환값을 쓴다”는 단선적인 설명을 넘는 데 있다.  
핵심은 다음을 분리하는 것이다.

- 언어 차원에서 반환값 중심이 자연스러운 이유
- 시스템 콜/라이브러리 인터페이스가 이 모델을 선호하는 이유
- 반환값과 `errno` 또는 out-parameter가 어떻게 역할을 분담하는가
- 왜 같은 “실패”라도 어떤 함수는 `-1`, 어떤 함수는 `NULL`, 어떤 함수는 `0/비0`, 어떤 함수는 직접 에러 코드를 반환하는가
- 이 모델이 자원 관리, cleanup, API 문서화와 어떻게 연결되는가

즉, 이 문서는 C의 에러 처리를 “낡은 방식”으로 소개하지 않는다.  
그 대신 C의 반환값 중심 모델을 **명시적 제어 흐름, 저비용 호출 규약, 시스템 친화적 실패 전달 모델**로 설명한다.

## 2. 전체 구조

C에서 에러 처리가 왜 반환값 중심으로 굳어졌는지 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 언어 구조 층위

C 함수 호출은 기본적으로 다음 구조를 가진다.

- 인자를 넣는다
- 함수가 실행된다
- 하나의 반환값을 돌려받는다

언어 자체는 예외 전파, 자동 stack unwinding, 실패 시 자동 cleanup 같은 고수준 메커니즘을 기본 추상화로 제공하지 않는다.  
즉, 실패는 호출자에게 **명시적으로 되돌려 보내야 할 결과**가 된다.

### 2.2 시스템 인터페이스 층위

시스템 콜과 저수준 라이브러리 함수는 실패 가능성이 높고, 실패 원인이 다양하며, 부분 성공 상태가 중요하다.  
이런 환경에서는 실패가 “숨겨진 제어 흐름”보다 **호출자가 즉시 확인하고 분기할 수 있는 명시적 값**으로 드러나는 것이 유리하다.

### 2.3 결과 표현 층위

C 함수는 자주 다음 문제를 동시에 다뤄야 한다.

- 성공했는가 실패했는가
- 성공했다면 주 결과는 무엇인가
- 실패했다면 상세 원인은 무엇인가
- 호출 도중 일부 자원이 획득되었는가
- 정리 책임은 누구에게 있는가

이 때문에 반환값, `errno`, out-parameter, sentinel value, 별도 상태 코드가 함께 등장한다.

### 2.4 자원 관리 층위

C에서는 자원 해제가 자동으로 따라오지 않는 경우가 많다.  
파일 디스크립터, 힙 메모리, 락, 매핑, 소켓, 임시 버퍼 같은 자원을 직접 정리해야 한다. 이 점 때문에 에러 처리는 단순 분기 문제가 아니라 **실패 경로 설계 문제**가 된다.

### 2.5 실무 API 설계 층위

반환값 중심 모델은 모든 함수에 똑같이 적용되지 않는다.  
주 결과가 무엇인지, 실패가 얼마나 자주 일어나는지, 상세 오류가 필요한지, 상위 계층에 어떤 표현이 더 자연스러운지에 따라 설계가 달라진다.

즉, “반환값 중심”은 단일 패턴이 아니라, **실패를 1차적으로 호출자가 즉시 해석 가능한 값으로 드러내는 공통 철학**에 가깝다.

## 3. 내부 동작 원리

### 3.1 C에서는 실패가 숨겨진 제어 흐름보다 명시적 분기로 드러난다

가장 먼저 봐야 할 것은 C의 기본 호출 모델이다.  
C 함수는 호출자에게 자동으로 예외를 던져 stack을 거슬러 올라가며 cleanup을 수행하는 식의 구조를 기본 제공하지 않는다.

즉, 호출자는 보통 다음을 직접 해야 한다.

- 반환값을 확인한다
- 실패인지 성공인지 판단한다
- 필요하면 `errno`나 다른 상세 상태를 읽는다
- 이미 획득한 자원을 정리한다
- 상위 호출자로 에러를 전파한다

이 구조의 핵심은 실패가 “숨은 제어 흐름”이 아니라 **보이는 분기문**으로 나타난다는 점이다.
```c
    fd = open(path, O_RDONLY);
    if (fd == -1) {
        return -1;
    }
```
이 코드는 단순해 보이지만, 실제로는 중요한 성질을 가진다.

- 실패 경로가 소스 코드에 드러난다
- 호출자가 어떤 자원을 이미 잡았는지 문맥 안에서 바로 판단할 수 있다
- 상위 계층으로 무엇을 어떻게 전파할지 명시적으로 결정할 수 있다

즉, 반환값 중심 에러 처리는 언어의 낮은 추상화 수준과 직접 맞물린다.

### 3.2 왜 “성공 값”과 “실패 신호”가 같은 채널에 놓이는가

반환값 중심 모델의 핵심 특징은 성공 값과 실패 신호가 같은 채널을 공유한다는 점이다.  
이것이 장점이자 제약이다.

예를 들어:

- `open()`은 성공 시 fd, 실패 시 `-1`
- `malloc()`은 성공 시 포인터, 실패 시 `NULL`
- `read()`는 성공 시 바이트 수, 실패 시 `-1`

즉, 같은 반환 채널 안에서 일부 값이 “실패를 뜻하는 특별한 값(sentinel)”로 예약된다.

왜 이런 구조가 자연스러운가?

- 함수 호출의 기본 결과 채널이 이미 하나 존재한다.
- 그 채널로 성공/실패를 우선 표현하면 호출자가 즉시 분기할 수 있다.
- 상세 실패 원인은 별도 채널로 분리할 수 있다.
- 저수준 인터페이스에서는 이 방식이 단순하고 비용이 낮다.

하지만 이 모델은 제약도 만든다.

- 어떤 값이 유효 결과이고 어떤 값이 실패 값인지 계약이 필요하다.
- 주 결과의 전체 값 영역을 다 못 쓸 수 있다.
- 상세 실패 이유는 별도 채널이 필요하다.

즉, 반환값 중심 모델은 “모든 것을 반환값 하나로 해결한다”가 아니라,  
**실패 판정은 반환값으로, 실패 설명은 별도 채널로** 분리하는 구조로 발전한다.

### 3.3 `errno`는 반환값 중심 모델의 확장이지 대체물이 아니다

이 지점에서 `errno`가 등장한다.  
처음 배우는 사람은 `errno`를 에러 처리의 중심으로 보기 쉽지만, 실제로는 그렇지 않다.

정확한 구조는 다음과 같다.

- 반환값이 먼저 실패를 알린다.
- 그 실패가 확인된 뒤에만 `errno`가 상세 이유를 제공한다.

즉, `errno`는 반환값 중심 모델을 보완하는 2차 채널이다.  
반환값 중심을 무너뜨리는 대체 모델이 아니다.

이 차이는 매우 중요하다.

- `fd == -1`이 실패 판정이다.
- `errno == ENOENT`는 실패 이유 설명이다.

이 둘을 섞으면 거의 반드시 API 설계가 흐려진다.

### 3.4 왜 일부 함수는 out-parameter와 상태 코드를 분리하는가

모든 함수가 sentinel value 모델을 쓰기 쉬운 것은 아니다.  
예를 들어 주 결과가 정수 전체 범위를 모두 쓸 수 있거나, 실패와 성공 결과를 같은 채널에 실으면 모호해지는 경우가 있다.

그래서 다음 형태가 등장한다.
```c
    int parse_u32(const char *s, uint32_t *out);
```
이 구조에서는:

- 반환값: 성공/실패 상태 코드
- out-parameter: 주 결과

이 방식이 반환값 중심과 모순되는가? 아니다.  
오히려 같은 철학의 변형이다.

핵심은 다음과 같다.

- 함수 호출의 1차 제어 흐름은 여전히 반환값으로 분기한다.
- 주 계산 결과는 별도 저장 공간으로 전달한다.
- 성공 여부와 주 결과를 분리해 모호성을 줄인다.

즉, 반환값 중심 모델은 “항상 주 결과가 반환값”이라는 뜻이 아니라,  
**호출 제어의 1차 판정을 반환값이 담당한다**는 뜻으로 보는 편이 정확하다.

### 3.5 왜 어떤 함수는 직접 에러 코드를 반환하는가

POSIX thread API 같은 곳에서는 종종 이런 형태를 본다.
```c
    int rc = pthread_mutex_lock(&m);
    if (rc != 0) {
        /* rc 자체가 오류 코드 */
    }
```
이 방식은 `errno` 스타일과 다르다.  
그렇다고 반환값 중심 철학을 벗어나는 것도 아니다.

오히려 더 직접적이다.

- 반환값이 곧 상태 코드다.
- 별도 `errno`를 보지 않아도 된다.
- 실패 이유를 즉시 지역 변수에 보존할 수 있다.
- 호출 직후 다른 함수 때문에 `errno`가 훼손되는 문제도 줄어든다.

즉, 반환값 중심 모델 안에는 적어도 세 가지 하위 패턴이 있다.

- 반환값으로 주 결과, sentinel로 실패, 상세 원인은 `errno`
- 반환값으로 상태 코드, 주 결과는 out-parameter
- 반환값으로 상태 코드 자체를 직접 전달

이들은 서로 다른 패턴처럼 보이지만, 모두 **호출자가 즉시 반환값을 해석해 분기한다**는 공통점을 가진다.

### 3.6 반환값 중심 모델은 cleanup 설계를 호출자에게 밀어 넣는다

이 모델의 가장 중요한 비용은 여기서 드러난다.  
에러가 숨은 제어 흐름으로 전파되지 않으므로, 호출자는 부분 성공 상태를 직접 정리해야 한다.

예를 들어:
```c
    p = malloc(...);
    if (p == NULL) {
        return -1;
    }

    fd = open(...);
    if (fd == -1) {
        free(p);
        return -1;
    }
```
여기서는 `open` 실패 시 이미 획득한 `p`를 호출자가 직접 정리해야 한다.  
즉, 반환값 중심 모델은 실패를 명시적으로 드러내는 대신, 실패 경로 설계를 호출자에게 요구한다.

이것이 바로 다음 문서인 `goto cleanup` 패턴과 자연스럽게 연결되는 이유다.

## 4. 핵심 구성 요소

### 4.1 반환값(return value)

반환값은 C 에러 처리의 1차 채널이다.  
가장 먼저 해석해야 하고, 제어 흐름 분기를 직접 결정한다.

대표 역할은 다음과 같다.

- 성공/실패 판정
- 주 결과 전달
- 상태 코드 전달
- 부분 성공 정도 표현

즉, 반환값은 단순 데이터가 아니라 호출 계약의 중심이다.

### 4.2 sentinel value

반환값 중심 모델에서는 일부 값이 실패를 뜻하는 특별한 값으로 예약된다.

예:

- `NULL`
- `-1`
- 음수 값
- `EOF`
- 0/비0

sentinel value는 단순하지만, 값 공간과 의미 공간이 섞인다는 특징이 있다.  
따라서 문서화가 중요하다.

### 4.3 상세 오류 채널

반환값만으로는 왜 실패했는지 알 수 없는 경우가 많다.  
이때 다음이 보조 채널이 된다.

- `errno`
- 직접 반환되는 에러 코드
- 별도 상태 객체
- 로그/진단 메시지
- 라이브러리별 last-error API

중요한 것은 이것이 반환값의 대체물이 아니라 **보조 채널**이라는 점이다.

### 4.4 out-parameter

주 결과와 상태 분리를 위해 많이 쓰인다.

예:
```c
    int get_config_value(const char *key, int *out_value);
```
이 방식의 장점은 다음과 같다.

- 반환값을 순수 상태 코드로 쓸 수 있다
- 주 결과가 전체 값 영역을 자유롭게 쓸 수 있다
- 실패 시 out-parameter를 갱신하지 않는 규약을 둘 수 있다

즉, out-parameter는 반환값 중심 모델을 보완하는 구조적 도구다.

### 4.5 cleanup responsibility

반환값 중심 에러 처리에서 빠질 수 없는 구성 요소다.  
실패가 자동 unwind를 일으키지 않으므로, 호출자는 다음을 직접 관리해야 한다.

- 어떤 자원을 이미 획득했는가
- 실패 시 무엇을 어떤 순서로 해제해야 하는가
- 원래 실패 원인을 보존해야 하는가
- cleanup 중 추가 실패는 어떻게 다룰 것인가

즉, 반환값 중심 모델은 에러 분기와 자원 정리를 분리할 수 없다.

## 5. 실행 흐름 또는 처리 순서

### 5.1 가장 단순한 형태: sentinel 반환
```c
    FILE *fp = fopen(path, "r");
    if (fp == NULL) {
        return -1;
    }
```
이 패턴은 단순하다.

- 반환값으로 실패 여부를 즉시 확인
- 실패 시 상위 호출자에게 전파 또는 처리

주 결과가 포인터이므로 `NULL`을 실패 sentinel로 쓰기 쉽다.

### 5.2 상세 이유가 필요한 경우: 반환값 + `errno`
```c
    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        int saved_errno = errno;
        report_open_failure(saved_errno);
        return -1;
    }
```
여기서는 반환값이 실패 판정을 맡고, `errno`가 상세 이유를 맡는다.  
즉, 제어 흐름과 원인 설명이 분리된다.

### 5.3 상태 코드 + out-parameter
```c
    int parse_port(const char *s, int *out_port) {
        char *end;
        long v;

        errno = 0;
        v = strtol(s, &end, 10);
        if (errno != 0) {
            return -1;
        }
        if (*end != '\0') {
            return -1;
        }
        if (v < 1 || v > 65535) {
            return -1;
        }

        *out_port = (int)v;
        return 0;
    }
```
이 구조에서는 호출자가 `if (parse_port(...) != 0)`로 즉시 분기한다.  
즉, 여전히 반환값 중심이다.

### 5.4 직접 에러 코드 반환
```c
    int rc = pthread_mutex_lock(&m);
    if (rc != 0) {
        return rc;
    }
```
이 방식은 `errno`를 별도로 읽을 필요가 없다.  
에러 코드 자체가 반환값으로 오므로, 실패 정보 보존이 더 명시적이다.

### 5.5 부분 성공 상태가 있는 경우
```c
    p = malloc(...);
    if (p == NULL) {
        return -1;
    }

    fd = open(...);
    if (fd == -1) {
        free(p);
        return -1;
    }

    buf = malloc(...);
    if (buf == NULL) {
        close(fd);
        free(p);
        return -1;
    }
```
이 코드는 반환값 중심 모델의 비용을 보여 준다.  
실패 분기마다 부분 성공 상태를 되감아야 한다.  
이 때문에 실무에서는 중복 정리를 줄이기 위해 cleanup 구조화가 필요해진다.

## 6. 성능 / 최적화 관점

### 6.1 반환값 중심은 호출 규약이 단순하다

저수준 시스템 코드에서 반환값 중심 모델이 선호되는 이유 중 하나는 호출 규약이 단순하기 때문이다.

- 함수가 값을 하나 돌려준다
- 호출자가 즉시 비교/분기한다
- 실패가 숨겨진 런타임 메커니즘을 거치지 않는다

이 단순성은 저수준 인터페이스와 잘 맞는다.  
특히 시스템 콜 래퍼, 작은 libc 함수, 자주 호출되는 경계에서 예측 가능한 제어 흐름을 유지하기 쉽다.

### 6.2 비용은 성공 경로보다 실패 경로 구조화에 있다

반환값 중심 에러 처리의 비용은 “if 문 하나 더 있다”에 있지 않다.  
진짜 비용은 실패 경로가 많아질수록 호출자가 직접 해야 하는 작업에 있다.

- 획득 자원 추적
- cleanup 순서 관리
- 상세 오류 보존
- 상위 계층용 오류 번역
- 부분 성공 상태 롤백

즉, 이 모델은 런타임 비용보다 **코드 구조 비용**이 더 크다.

### 6.3 숨은 제어 흐름이 적다는 점은 디버깅에 유리할 수 있다

반환값 중심 모델은 verbose할 수 있지만, 제어 흐름은 명시적이다.

- 실패가 어디서 발생했는지 코드에 드러난다
- 어떤 분기에서 어떤 cleanup이 수행되는지 보인다
- 반환값 검사 누락을 리뷰로 식별할 수 있다

즉, 비용이 크지만 추론 가능성이 높다.  
이는 시스템 프로그래밍에서 중요한 장점이다.

### 6.4 그러나 검사 누락 비용도 크다

반환값 중심 모델의 반대편 비용은 명확하다.

- 호출자가 검사하지 않으면 실패가 조용히 흘러간다
- 부분 실패 상태를 그냥 넘기기 쉽다
- `errno` 보존을 놓치기 쉽다
- cleanup 누락이 메모리 누수/FD 누수/락 누수로 이어진다

즉, 이 모델은 단순하지만, 호출자가 규율을 지키지 않으면 매우 취약하다.

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
C 에러 처리에 대한 오해는 주로 “반환값 중심”이라는 말을 너무 좁게 또는 너무 넓게 해석할 때 생긴다.

### 7.1 “반환값 중심”과 “반환값만 사용한다”가 갈리는 지점

반환값 중심은 곧바로 “모든 실패 정보를 반환값 하나에만 넣는다”는 뜻이 아니다.

실제로는 다음이 가능하다.

- 반환값 + `errno`
- 반환값 + out-parameter
- 반환값 자체가 상태 코드
- 반환값 + 별도 last-error state

즉, 반환값 중심의 핵심은 **1차 제어 분기를 반환값이 담당한다**는 데 있다.

### 7.2 주 결과와 실패 판정이 갈리는 지점

어떤 함수는 반환값이 주 결과다.

- `open()`, `malloc()`, `read()`

어떤 함수는 반환값이 상태 코드고 주 결과는 out-parameter다.

- `int parse_xxx(..., out *)`

이 둘은 다른 패턴처럼 보이지만, 에러 처리 철학은 같다.  
둘 다 호출자가 즉시 반환값을 검사해 제어 흐름을 결정한다.

### 7.3 실패 판정과 실패 설명이 갈리는 지점

이 경계는 매우 중요하다.

- 실패 판정: 반환값
- 실패 설명: `errno` 또는 에러 코드

즉, `errno`는 실패를 “알리는” 신호가 아니라 실패를 “설명하는” 보조 채널이다.  
이 구분이 흐려지면 에러 처리 구조가 금방 무너진다.

### 7.4 시스템 계층과 상위 추상화 계층이 갈리는 지점

시스템 계층에서는 `-1 + errno`, `NULL + errno` 같은 패턴이 자연스럽다.  
하지만 상위 비즈니스 로직이나 도메인 계층까지 그대로 끌고 가는 것이 항상 최선은 아니다.

상위 계층에서는 종종 다음이 더 낫다.

- 명시적 enum 상태 코드
- 도메인별 에러 범주
- 래핑된 구조체 결과
- 오류 번역 계층

즉, 반환값 중심 철학은 유지하되, 구체적 표현은 계층에 따라 바뀔 수 있다.

### 7.5 언어 제약과 라이브러리 관례가 갈리는 지점

C가 예외를 기본 제공하지 않는다는 사실은 출발점이다.  
하지만 반환값 중심이 굳어진 이유를 거기에만 돌리면 부족하다.

더 중요한 점은 다음이다.

- 시스템 호출은 원래 명시적 상태 점검과 잘 맞는다
- 실패가 흔하고 부분 성공 상태가 중요하다
- cleanup을 호출자가 정확히 통제해야 한다
- 런타임 숨은 비용보다 예측 가능한 제어 흐름이 선호된다

즉, 반환값 중심은 단순한 언어의 결핍이 아니라 **저수준 시스템 인터페이스와 잘 맞는 설계 선택**이기도 하다.

## 8. 실무에서 중요한 판단 기준

### 8.1 반환값이 무엇을 뜻하는지 먼저 문서화하라

가장 먼저 명확해야 할 것은 반환값 계약이다.

- 성공 시 무엇을 반환하는가
- 실패 시 어떤 sentinel 또는 상태 코드를 반환하는가
- out-parameter는 언제 유효한가
- `errno` 또는 다른 보조 채널은 언제 의미가 있는가

반환값 중심 모델에서는 이 문서화가 API의 절반이다.

### 8.2 주 결과와 상태 코드를 한 채널에 넣을지 분리할지 결정하라

다음 기준이 중요하다.

- 실패 sentinel을 잡기 쉬운가
- 주 결과의 값 영역이 넓어 sentinel이 어색한가
- 상세 오류가 필요한가
- 호출자가 분기하기 쉬운가
- 상위 계층에서 래핑하기 쉬운가

주 결과와 상태를 섞으면 단순해질 수 있지만, 모호성도 늘어난다.  
분리하면 명확해지지만 호출 규약이 길어진다.

### 8.3 시스템 인터페이스 가까울수록 반환값 + `errno`가 자연스럽다

파일, 소켓, 프로세스, 메모리 매핑, 디렉터리, 장치 접근처럼 OS와 맞닿은 계층에서는 이 패턴이 매우 자연스럽다.

- 실패가 흔하다
- 실패 원인이 다양하다
- 상세 코드가 운영체제 의미와 직접 연결된다

즉, 이 계층에서는 `errno` 스타일이 여전히 강력하다.

### 8.4 상위 계층으로 올라갈수록 명시적 상태 코드가 나을 수 있다

도메인 로직, 파서, 비즈니스 규칙, 라이브러리 추상화 계층에서는 `errno`를 그대로 흘리기보다 다음이 더 나을 수 있다.

- enum error
- 상태 코드 반환
- 결과 구조체
- 에러 번역 함수

즉, 하위 시스템 실패를 상위 의미로 번역하는 계층이 필요할 수 있다.

### 8.5 반환값 중심 모델에서는 cleanup 구조화가 필수다

에러 분기마다 자원 해제를 흩뿌리면 코드가 곧 무너진다.  
따라서 다음이 중요해진다.

- 일관된 cleanup 경로
- `goto cleanup` 패턴
- `errno` 보존
- 부분 획득 자원 추적
- 해제 순서 설계

즉, 반환값 중심 에러 처리의 완성은 단순 `if (rc != 0)`가 아니라 **실패 경로를 구조화하는 것**이다.

### 8.6 검사 누락을 구조적으로 줄여야 한다

반환값 중심 모델의 가장 큰 위험은 검사 누락이다.  
실무에서는 이를 줄이기 위해 다음을 활용할 수 있다.

- 즉시 반환 패턴
- wrapper 함수
- 일관된 상태 코드 규약
- 코드 리뷰 규칙
- 정적 분석기
- 경계 API 최소화

즉, 이 모델은 호출자의 규율을 요구하므로, 그 규율을 코드베이스 차원에서 보조해야 한다.

---

## 9. 판단 체크리스트

다음 질문에 명확히 답하지 못하면, 그 API의 에러 처리는 아직 불안정하다.

- 이 함수의 반환값은 주 결과인가, 상태 코드인가
- 실패 sentinel이 명확한가
- 상세 실패 이유가 필요한가
- 필요하다면 `errno`, 직접 상태 코드, out-parameter 중 무엇이 맞는가
- 성공 시 out-parameter가 언제 유효한가
- 실패 시 부분 획득 자원은 누가 정리하는가
- 상위 계층에 `errno`를 그대로 노출할지 번역할지 결정했는가
- 반환값 검사 누락을 줄일 구조가 있는가
- 실패 직후 원인 보존이 필요한가
- cleanup 경로가 중복 없이 구조화되어 있는가

## 정리

C에서 에러 처리가 반환값 중심인 이유는 단순히 예외가 없어서가 아니다.  
더 정확히 말하면, C의 에러 처리는 **언어의 명시적 호출 모델, 저수준 시스템 인터페이스의 요구, 그리고 호출자가 cleanup을 직접 통제해야 하는 현실**이 결합된 결과다.

반드시 기억해야 할 핵심은 다음이다.

- 반환값은 에러 처리의 1차 채널이다.
- 실패 판정과 실패 설명은 분리될 수 있다.
- `errno`는 반환값 중심 모델의 보조 채널이지 대체물이 아니다.
- out-parameter와 직접 에러 코드 반환도 반환값 중심 철학의 변형이다.
- 이 모델의 강점은 명시적 제어 흐름과 낮은 추상화 비용에 있다.
- 이 모델의 비용은 cleanup 구조화와 검사 누락 위험에 있다.
- 따라서 반환값 중심 에러 처리는 단순 분기 기술이 아니라 실패 경로 설계 기술이다.

즉, C의 에러 처리를 이해한다는 것은  
“실패를 값으로 돌려받는다”는 사실을 아는 것이 아니라,  
**그 값을 중심으로 제어 흐름, 상세 원인, 자원 정리, 상위 계층 번역을 어떻게 설계하는지 이해하는 것**에 가깝다.

---

## 10. 더 깊게 볼 포인트

### 10.1 `goto cleanup` 패턴은 왜 실무에서 자주 쓰이는가

다음 문서로 가장 자연스럽다.  
반환값 중심 에러 처리의 비용이 실패 경로 구조화에 있다면, `goto cleanup`은 그 비용을 관리하기 위해 등장한 대표 패턴이다.

### 10.2 자원 해제 순서는 왜 설계 대상인가

부분 성공 상태와 cleanup 순서를 어떻게 정하느냐에 따라 에러 처리의 정확성이 달라진다. 반환값 중심 모델과 직접 연결되는 다음 주제다.

### 10.3 `errno` 스타일과 직접 에러 코드 반환은 각각 어디에 더 잘 맞는가
