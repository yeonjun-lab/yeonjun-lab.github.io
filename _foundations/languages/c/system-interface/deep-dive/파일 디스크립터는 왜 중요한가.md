## 1. 왜 이 주제를 깊게 봐야 하는가

파일 디스크립터(file descriptor)는 UNIX/POSIX 계열 시스템 프로그래밍의 가장 핵심적인 개념 중 하나다.  
그런데 초급 단계에서는 자주 너무 얕게 소개된다.

- 파일을 열면 정수가 하나 나온다
- 그 정수로 `read`, `write`, `close`를 한다
- 파일 디스크립터는 파일 핸들 같은 것이다

이 설명은 시작점으로는 맞지만, 실무 판단에는 거의 도움이 되지 않는다.  
왜냐하면 파일 디스크립터는 단순 “파일 번호”가 아니라, **프로세스가 커널 자원에 접근하기 위해 사용하는 가장 기본적인 참조 슬롯**이기 때문이다.

이 개념을 얕게 이해하면 다음과 같은 문제를 자주 오해한다.

- 왜 파일뿐 아니라 소켓, 파이프, 터미널, 이벤트 객체도 같은 API로 다뤄지는가
- 왜 `dup`, `fork`, `exec`, `close-on-exec`가 중요한가
- 왜 같은 fd를 공유해도 파일 오프셋이 같이 움직일 수 있는가
- 왜 fd leak이 메모리 leak만큼 치명적일 수 있는가
- 왜 “정수 하나”를 복사했다고 해서 독립 자원이 생긴 것이 아닌가
- 왜 blocking, nonblocking, event loop, pipe, socket이 한 모델 안에서 연결되는가

즉, 파일 디스크립터는 단순한 I/O 입문 개념이 아니라, 이후의 다음 주제들을 모두 연결하는 중심 모델이다.

- blocking / nonblocking
- pipe / socket
- event loop
- redirection
- subprocess
- daemonization
- resource leak
- inheritance across exec
- readiness multiplexing

이 주제를 깊게 봐야 하는 이유는 세 가지다.

첫째, 파일 디스크립터는 “파일 입출력”보다 더 넓은 운영체제 인터페이스 모델이다.  
즉, 이름 때문에 파일 전용처럼 보이지만 실제로는 **프로세스가 커널이 관리하는 I/O 자원과 연결되는 공통 핸들 모델**에 가깝다.

둘째, 파일 디스크립터는 값 자체보다 관계가 중요하다.  
정수 `3`, `4`, `5` 자체가 중요한 것이 아니라, 그 정수가 프로세스의 descriptor table에서 무엇을 가리키고, 그 뒤에 어떤 open file description 또는 커널 객체가 있고, 그 상태를 누구와 공유하는지가 중요하다.

셋째, 실무 버그는 대부분 “fd는 그냥 숫자”라는 오해에서 나온다.  
예를 들어:

- `close(fd)` 후 같은 번호가 재사용될 수 있다는 점을 잊는다
- `dup`가 독립 파일을 여는 것이라고 착각한다
- `fork` 후 부모와 자식이 같은 open state를 공유한다는 점을 놓친다
- `FD_CLOEXEC` 누락으로 민감한 fd가 자식 프로세스에 새어 나간다
- stdio `FILE *`와 fd를 같은 층위로 섞어 사용한다

즉, 파일 디스크립터는 시스템 프로그래밍에서 매우 작게 보이는 정수 하나이지만, 실제로는 **프로세스, 커널 객체, I/O 상태, 생애주기, 상속, 동기화가 만나는 경계**다.

이 문서의 핵심 목표는 파일 디스크립터를 “정수형 파일 핸들”로만 설명하는 데 있지 않다.  
핵심은 다음을 분리하는 것이다.

- 파일 디스크립터 자체와 그 뒤의 커널 상태는 무엇이 다른가
- fd 테이블, open file description, inode/소켓/파이프 객체는 어떤 관계인가
- 왜 같은 fd 번호와 같은 열린 파일 상태를 혼동하면 안 되는가
- 왜 fd 생애주기 관리가 설계 대상인가
- 실무에서 fd를 어떻게 다뤄야 leak, aliasing, inheritance 버그를 줄일 수 있는가

즉, 이 문서는 파일 디스크립터를 “파일을 읽기 위한 번호”로 설명하지 않는다.  
그 대신 **프로세스가 커널 I/O 자원에 접근하기 위해 사용하는 가장 기본적인 참조 모델**로 설명한다.

## 2. 전체 구조

파일 디스크립터를 정확히 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 프로세스 관점 층위

프로세스는 파일 디스크립터를 정수로 본다.  
즉, 사용자 코드 입장에서는 `int fd` 하나가 전부처럼 보인다.

하지만 이 정수는 그냥 값이 아니라, **현재 프로세스의 descriptor table에서 어떤 항목을 가리키는 인덱스**다.

### 2.2 커널 참조 층위

fd가 직접 파일 그 자체를 가리키는 것은 아니다.  
보통 더 정확한 모델은 다음과 같다.

- 프로세스의 fd table 엔트리
- open file description(열린 파일 상태)
- 실제 커널 객체(파일, 소켓, 파이프, 장치 등)

즉, fd는 “자원 자체”가 아니라 **자원 접근 경로의 첫 번째 손잡이**다.

### 2.3 상태 공유 층위

중요한 것은 fd 번호만이 아니라, 그 fd가 어떤 open state를 공유하는지다.

- 파일 오프셋
- 상태 플래그
- 접근 모드
- 소켓/파이프 상태
- 참조 수명

즉, 같은 번호가 같은 상태를 의미하는 것도 아니고, 다른 번호가 다른 상태를 의미하는 것도 아니다.

### 2.4 생애주기 층위

파일 디스크립터는 생성되고, 복제되고, 상속되고, 닫히고, 재사용된다.

- `open`
- `socket`
- `pipe`
- `dup`
- `fork`
- `exec`
- `close`

이 생애주기를 정확히 이해하지 못하면 leak, aliasing, inheritance 문제가 생긴다.

### 2.5 실무 I/O 모델 층위

파일 디스크립터는 이후의 거의 모든 시스템 I/O 모델과 연결된다.

- blocking vs nonblocking
- `select`, `poll`, `epoll`, `kqueue`
- 표준 입출력 리다이렉션
- pipe 기반 프로세스 연결
- socket 서버/클라이언트
- event source 통합

즉, fd를 이해하는 것은 단일 API를 배우는 것이 아니라 **UNIX I/O 모델 전체의 공통 인터페이스를 이해하는 것**이다.

## 3. 내부 동작 원리

### 3.1 파일 디스크립터는 자원 자체가 아니라 프로세스 테이블의 인덱스다

가장 먼저 깨야 할 오해는 이것이다.

- fd는 커널 파일 객체 그 자체다
- `fd == 3`이면 “파일 3번” 같은 것이다

실제로는 그렇지 않다.  
프로세스는 보통 “파일 디스크립터 테이블” 같은 구조를 가진다고 이해하면 좋다. `fd`는 그 테이블의 엔트리를 가리키는 정수다.

즉:

- 사용자 코드가 가진 것은 `int fd`
- 그 정수는 현재 프로세스의 descriptor entry를 고른다
- 그 엔트리는 다시 커널의 열린 파일 상태(open file description)에 연결된다
- 그 뒤에 실제 파일/소켓/파이프 객체가 있다

왜 이 차이가 중요한가?

많은 사람이 `fd`를 “파일 객체의 ID”처럼 상상한다.  
하지만 실제로는 `fd`는 **프로세스 로컬한 참조 슬롯 번호**에 가깝다.  
그래서 같은 실제 자원을 다른 프로세스가 같은 번호로 볼 필요도 없고, 같은 프로세스 안에서도 다른 번호들이 같은 열린 상태를 가리킬 수 있다.

### 3.2 fd와 open file description을 구분해야 한다

이 구분이 가장 중요하다.

실무적으로 단순화하면 다음처럼 볼 수 있다.

- 파일 디스크립터: 프로세스 로컬 참조
- open file description: 열린 상태 자체
- inode/socket/pipe endpoint: 실제 커널 객체

예를 들어 `open()`을 호출하면 보통 새 open file description이 생기고, 프로세스의 fd table 한 칸이 그것을 가리키게 된다.

반면 `dup(fd)`는 다르다.

- 새로운 파일을 다시 여는 것이 아니다
- 같은 open file description을 가리키는 새 fd 엔트리를 만든다

즉:

- fd는 다를 수 있다
- 하지만 그 뒤의 열린 상태는 같을 수 있다

이 차이가 왜 중요한가?

같은 open file description을 공유하면 다음도 공유될 수 있다.

- 현재 파일 오프셋
- 일부 상태 플래그
- 파일 상태의 일부 의미

즉, `dup`는 “새 파일”이 아니라 **같은 열린 상태에 대한 새 참조**다.

### 3.3 왜 같은 파일을 다시 `open`한 것과 `dup`한 것이 다른가

이 반례는 매우 중요하다.

#### 경우 1: `dup`
```c
    int fd1 = open(path, O_RDONLY);
    int fd2 = dup(fd1);
```
이 경우 `fd1`과 `fd2`는 같은 open file description을 가리킬 수 있다.  
즉, 한쪽에서 `read`를 하면 파일 오프셋이 함께 이동할 수 있다.

#### 경우 2: 다시 `open`
```c
    int fd1 = open(path, O_RDONLY);
    int fd2 = open(path, O_RDONLY);
```
이 경우는 보통 서로 다른 open file description이 생성된다.  
같은 실제 파일을 가리켜도, 파일 오프셋은 독립적일 수 있다.

왜 이 직관이 틀리기 쉬운가?

많은 사람은 “같은 파일을 열었으니 상태도 같겠지” 혹은 “fd를 복사했으니 독립적이겠지”라고 직관적으로 생각한다.  
하지만 실제로는 반대다.

- 같은 파일을 다시 열면 상태는 독립적일 수 있다
- 같은 fd를 복제하면 상태를 공유할 수 있다

즉, 중요한 것은 경로명이 아니라 **어떤 open state를 공유하느냐**다.

### 3.4 파일 디스크립터 모델이 파일에만 국한되지 않는 이유

이름 때문에 가장 흔한 오해는 다음이다.

- 파일 디스크립터는 파일 전용이다

하지만 POSIX/UNIX 계열에서는 다음도 모두 fd로 다뤄질 수 있다.

- regular file
- pipe
- socket
- terminal
- device
- FIFO
- 일부 이벤트 소스

이것이 중요한 이유는, 커널이 I/O 자원을 **공통 read/write/close/poll 모델** 아래에 두려 하기 때문이다.

즉, fd는 “파일 API”가 아니라  
더 정확히는 **프로세스가 다양한 I/O 객체를 비슷한 방식으로 다루게 하는 공통 손잡이**다.

이 점 때문에 이후 주제들이 하나로 이어진다.

- pipe와 socket이 fd를 쓴다
- `select`/`poll`/`epoll`도 fd를 다룬다
- stdin/stdout/stderr도 fd다
- redirection도 fd 교체로 설명된다

즉, fd를 이해하면 시스템 I/O 모델의 공통 언어가 생긴다.

### 3.5 `close(fd)`는 숫자를 없애는 것이 아니라 참조를 끊는 것이다

초보자는 흔히 이렇게 생각한다.

- `close(fd)`는 파일을 닫는다

이 설명은 결과적으로 맞을 수 있지만, 더 정확한 설명은 다음이다.

- 현재 프로세스의 descriptor entry 하나를 닫는다
- 그 엔트리가 어떤 open file description을 더 이상 참조하지 않게 만든다
- 그 열린 상태를 더 이상 아무도 참조하지 않을 때 비로소 커널 객체 측 정리가 진행될 수 있다

즉, `close`는 항상 “자원 자체 파괴”와 1:1이 아니다.  
참조가 여러 개일 수 있기 때문이다.

예:
```c
    int fd1 = open(path, O_RDONLY);
    int fd2 = dup(fd1);

    close(fd1);
```
이 시점에도 `fd2`가 같은 열린 상태를 참조하고 있으면, 실제로 그 자원 접근은 계속 가능하다.

즉, `close`는 **이 프로세스의 이 참조 슬롯을 닫는 행위**로 이해하는 편이 더 정확하다.

### 3.6 fd 번호는 닫힌 뒤 재사용될 수 있다

이 점은 실무 버그와 직접 연결된다.

예를 들어:
```c
    close(fd);
```
이후 시스템은 다음 `open`이나 `socket` 호출에서 같은 작은 정수 번호를 다시 줄 수 있다.

즉, `fd == 4`라는 숫자 자체는 장기적인 identity가 아니다.  
그것은 현재 시점에만 유효한 로컬 참조 슬롯 번호다.

왜 이 직관이 중요할까?

다음 같은 버그가 생기기 쉽다.

- 이미 닫은 fd 값을 오래 보관
- 다른 코드가 새로 연 자원에 같은 번호가 재사용
- 늦게 도착한 cleanup이 그 새 자원을 잘못 닫음
- stale fd 비교로 잘못된 동작

즉, fd는 pointer처럼 보이는 정수지만, **수명 종료 후 재사용되는 slot number**라는 점을 항상 기억해야 한다.

### 3.7 `fork`와 `exec`를 거치면 fd의 의미가 크게 달라진다

파일 디스크립터가 중요한 진짜 이유 중 하나는 프로세스 생애주기와 직접 연결되기 때문이다.

#### `fork`

`fork` 후 자식은 부모의 fd table 상태를 복사받을 수 있다.  
이때 부모와 자식은 같은 open file description을 공유할 수 있다.

즉:

- 같은 fd 번호를 가질 수 있다
- 같은 열린 상태를 공유할 수 있다
- 파일 오프셋 등 일부 상태가 연결될 수 있다

#### `exec`

`exec`는 프로세스 이미지 자체를 바꾸지만, `FD_CLOEXEC`가 설정되지 않은 fd는 새 프로그램으로 넘어갈 수 있다.

이것이 왜 중요한가?

- 의도하지 않은 fd가 자식 프로그램에 노출될 수 있다
- 민감한 소켓/파이프/파일이 새 프로세스에 새어 나간다
- 파이프 EOF가 오지 않는 버그가 생긴다
- 보안 문제로 이어질 수 있다

즉, fd는 단순 I/O 핸들이 아니라 **프로세스 경계와 자원 상속 경계**의 일부다.

## 4. 핵심 구성 요소

### 4.1 file descriptor table entry

프로세스가 보는 첫 번째 층이다.

- 정수 fd 값으로 선택됨
- 프로세스 로컬
- 닫히면 슬롯이 비워질 수 있음
- 같은 번호가 나중에 재사용될 수 있음

즉, fd 번호는 자원 identity가 아니라 프로세스 로컬 슬롯 번호다.

### 4.2 open file description

열린 상태 자체를 나타내는 더 깊은 층이다.

- 현재 파일 오프셋
- 상태 플래그
- 커널이 유지하는 열린 인스턴스 상태

`dup`, `fork` 등에서 이 상태를 공유할 수 있다는 점이 핵심이다.

### 4.3 underlying kernel object

실제 커널 자원이다.

- regular file inode/entry
- socket endpoint
- pipe endpoint
- terminal/device

즉, fd는 이 실제 객체에 직접 닿는 것이 아니라 중간 열린 상태를 거쳐 접근한다.

### 4.4 descriptor flags vs file status flags

실무적으로는 이 구분도 중요하다.

- fd 자체 엔트리에 붙는 성질
- 열린 상태(open file description)에 붙는 성질

이 구분을 혼동하면 `dup` 이후 어떤 플래그가 공유되고 어떤 것은 독립적인지 헷갈리게 된다.

### 4.5 lifetime and ownership

fd는 다음 생애주기를 가진다.

- 획득
- 전달/복제
- 상속
- 사용
- 닫힘
- 번호 재사용 가능

즉, fd 관리의 핵심은 단순 `close` 호출이 아니라 **누가 이 참조를 소유하고 언제 종료할 책임이 있는가**를 명확히 하는 것이다.

## 5. 실행 흐름 또는 처리 순서

### 5.1 일반 파일 열기와 닫기
```c
    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        return -1;
    }

    /* read/write */

    close(fd);
```
가장 단순한 경우다.  
하지만 여기서도 본질은 “정수 하나를 얻었다”가 아니라 “현재 프로세스에 커널 열린 상태를 참조하는 슬롯이 생겼다”는 점이다.

### 5.2 `dup`로 같은 열린 상태 공유
```c
    int fd1 = open(path, O_RDONLY);
    int fd2 = dup(fd1);

    read(fd1, buf, 10);
    read(fd2, buf, 10);
```
이 경우 `fd2`가 독립적으로 파일 시작점부터 읽는다고 기대하면 틀릴 수 있다.  
왜냐하면 두 fd가 같은 열린 상태를 공유해 오프셋이 연동될 수 있기 때문이다.

### 5.3 같은 파일을 독립적으로 다시 열기
```c
    int fd1 = open(path, O_RDONLY);
    int fd2 = open(path, O_RDONLY);
```
이 경우는 각각 별도 열린 상태를 가질 수 있으므로, 오프셋도 독립적일 수 있다.  
즉, 경로명이 같다고 상태 공유가 생기는 것은 아니다.

### 5.4 `fork` 후 부모/자식 공유
```c
    int fd = open(path, O_RDONLY);
    pid_t pid = fork();

    if (pid == 0) {
        /* child */
        read(fd, ...);
    } else {
        /* parent */
        read(fd, ...);
    }
```
이 경우 부모와 자식이 같은 열린 상태를 공유할 수 있다.  
따라서 오프셋 경쟁이나 예상치 못한 상호 영향이 생길 수 있다.

즉, `fork`는 단순 프로세스 분기이면서 동시에 **fd 상태 공유 모델의 확장**이기도 하다.

### 5.5 `exec` 전 close-on-exec 필요성
```c
    int fd = open(secret_path, O_RDONLY);
    /* FD_CLOEXEC 미설정 */
    execve(...);
```
이 경우 새 프로그램에 이 fd가 그대로 남아 있을 수 있다.  
민감한 파일, 파이프, 소켓, 리스닝 소켓이라면 치명적일 수 있다.

즉, fd는 현재 프로세스 안에서만 끝나는 자원이 아니라, **프로세스 전환 시점의 공개 범위**까지 고려해야 한다.

## 6. 성능 / 최적화 관점

### 6.1 파일 디스크립터는 작은 정수지만 관리 비용은 작지 않다

`int fd`는 작아 보인다.  
그래서 초보자는 종종 다음처럼 생각한다.

- 그냥 정수 하나니까 비용도 작다
- 복사해도 상관없다
- 많이 열어도 큰 문제는 아니겠지

하지만 실제 비용은 숫자 자체가 아니라 그 뒤의 커널 자원에 있다.

- 열린 파일 상태 유지
- 커널 객체 참조 유지
- 버퍼/캐시/소켓 상태 점유
- 프로세스별 fd 한도 소비
- 이벤트 루프 감시 대상 증가
- leak 시 장기 점유

즉, fd는 값은 작지만 **커널 자원 점유 손잡이**라는 점에서 관리 비용이 크다.

### 6.2 fd leak은 메모리 leak과 다른 방식으로 치명적이다

fd leak이 심각한 이유는 다음과 같다.

- 프로세스의 open fd 한도를 소모한다
- 새 파일/소켓/파이프를 열 수 없게 된다
- 네트워크 서버는 accept 실패로 이어질 수 있다
- 파이프 write end가 남아 EOF가 안 온다
- 자식 프로세스 상속과 결합하면 디버깅이 매우 어려워진다

즉, fd leak은 단순 “조금 남아 있는 자원”이 아니라 **I/O 시스템 전체를 막아 버릴 수 있는 누수**다.

### 6.3 `dup`와 `fork` 기반 공유는 편리하지만 상태 결합 비용이 있다

같은 열린 상태를 공유하면 다음 장점이 있다.

- 리다이렉션 구현 용이
- 파이프 구성 간단
- 표준 입출력 교체 쉬움

하지만 비용도 있다.

- 오프셋 공유
- 닫힘 타이밍 추적 어려움
- 어느 참조가 실제 생존을 유지하는지 파악 어려움
- EOF/close 시점 추론 난이도 상승

즉, 상태 공유는 공짜 편의가 아니라 **생애주기 결합 비용**을 만든다.

### 6.4 파일 디스크립터 모델을 이해하면 이후의 nonblocking / event loop 성능 모델도 보인다

fd가 중요한 이유는 성능 논의의 출발점이기도 하기 때문이다.

- blocking fd는 호출 스레드를 멈출 수 있다
- nonblocking fd는 readiness 기반 처리와 연결된다
- `poll`/`epoll`/`kqueue`는 fd 집합을 다룬다
- 서버 성능은 종종 fd 수와 readiness 처리 모델에 의해 좌우된다

즉, fd는 단순 자원 식별자가 아니라 **I/O 병행성과 확장성 모델의 기초 단위**다.

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
파일 디스크립터에 대한 오해는 대부분 “정수 값”, “열린 상태”, “실제 자원”을 같은 것으로 볼 때 생긴다.

### 7.1 fd 번호와 열린 상태가 갈리는 지점

- fd 번호: 프로세스 로컬 슬롯 인덱스
- 열린 상태: 오프셋/플래그를 가진 커널 측 열린 인스턴스

이 둘은 다르다.  
즉, 다른 번호가 같은 열린 상태를 가리킬 수 있고, 같은 실제 파일을 가리켜도 서로 다른 열린 상태일 수 있다.

### 7.2 같은 경로 파일과 같은 open state가 갈리는 지점

- 같은 경로를 두 번 `open`  
  상태가 독립적일 수 있음
- `dup` 또는 `fork` 공유  
  상태가 연결될 수 있음

즉, “같은 파일인가”보다 “같은 열린 인스턴스를 공유하는가”가 더 중요하다.

### 7.3 참조 종료와 자원 종료가 갈리는 지점

- `close(fd)`는 현재 참조 하나를 끊는다
- 모든 참조가 사라져야 실제 열린 상태 정리가 완료될 수 있다

즉, `close`는 항상 즉시 “파일 그 자체 종료”와 같지 않다.

### 7.4 thread/process 로컬 숫자와 시스템 전역 자원이 갈리는 지점

fd 번호는 프로세스 로컬이다.  
하지만 그 뒤의 자원은 시스템 전체 커널 객체일 수 있다.

즉, 숫자의 지역성과 자원의 전역성을 혼동하면 안 된다.

### 7.5 thread-safe와 lifecycle-safe가 갈리는 지점

파일 디스크립터를 여러 코드 경로가 공유할 때 중요한 것은 단지 동시 접근만이 아니다.

- 누가 닫을 것인가
- 닫힌 뒤 누가 stale fd를 쓸 수 있는가
- 번호 재사용으로 다른 자원을 잘못 닫는가

즉, fd 문제는 race만이 아니라 **lifecycle safety** 문제다.

### 7.6 `FILE *`와 fd가 갈리는 지점

표준 라이브러리의 `FILE *`는 스트림 계층이고, fd는 더 아래의 시스템 콜 계층이다.  
둘이 연결될 수는 있지만 같은 것은 아니다.

즉, 버퍼링, flush, sync, 소유권을 섞어 생각하면 위험하다.  
이 구분을 놓치면 stdio와 시스템 콜을 섞는 코드에서 버그가 생기기 쉽다.

## 8. 실무에서 중요한 판단 기준

### 8.1 fd를 “정수”가 아니라 “소유권 있는 커널 자원 참조”로 다뤄라

실무에서 가장 중요한 태도는 이것이다.

- 누가 이 fd를 닫는가
- 언제까지 유효한가
- 복제/상속되었는가
- 같은 열린 상태를 다른 코드도 참조하는가

즉, fd는 숫자 저장이 아니라 자원 소유권 추적 문제다.

### 8.2 `dup`와 `open`을 같은 의미로 쓰지 마라

다음 질문을 분명히 해야 한다.

- 독립 오프셋/상태가 필요한가  
  다시 `open`
- 같은 열린 상태를 공유하며 다른 번호가 필요한가  
  `dup`

이 구분을 흐리면 I/O 위치 공유 버그가 생긴다.

### 8.3 `close` 이후 fd 값을 계속 신뢰하지 마라

닫힌 fd는 더 이상 유효 identity가 아니다.  
특히 다음을 경계해야 한다.

- cleanup이 늦게 다시 호출됨
- stale fd 값을 다른 스레드가 참조
- 재사용된 번호를 잘못 닫음

즉, close 후에는 보통 sentinel 값(`-1`)로 갱신하는 습관이 중요하다.

### 8.4 subprocess / exec 경계에서는 close-on-exec를 기본값으로 보라

민감한 fd가 자식 프로세스로 새어 나가면 보안과 동작 모두 위험해진다.  
따라서 다음 질문을 반드시 해야 한다.

- 이 fd는 exec 이후에도 남아야 하는가
- 아니라면 `FD_CLOEXEC` 또는 동등한 안전한 생성 플래그를 썼는가

즉, fd 설계는 현재 프로세스만이 아니라 **자식 프로세스 공개 범위**까지 포함한다.

### 8.5 pipe / socket / file을 따로 외우지 말고 fd 공통 모델로 묶어라

이 주제를 제대로 이해하면 이후의 시스템 인터페이스가 훨씬 명확해진다.

- pipe도 fd
- socket도 fd
- terminal도 fd
- event loop 감시 대상도 fd 중심
- redirection도 fd 중심

즉, 종류별 예외를 외우기보다 공통 핸들 모델을 먼저 잡는 편이 훨씬 강하다.

### 8.6 stdio와 시스템 콜 계층을 섞을 때는 버퍼링과 소유권을 먼저 확인하라

`FILE *`와 fd를 함께 다루는 순간 다음이 중요하다.

- 누가 닫는가
- flush는 언제 되는가
- 동일한 underlying fd를 둘이 공유하는가
- close 순서가 어떻게 되는가

즉, 스트림 계층과 fd 계층은 연결될 수 있어도 같은 층위가 아니므로, 섞어 쓸 때는 더 명시적이어야 한다.

## 9. 판단 체크리스트

다음 질문에 명확히 답하지 못하면, 그 fd 코드는 아직 불안정하다.

- 이 fd는 현재 누가 소유하는가
- `dup`된 참조인가, 독립 `open`인가
- 같은 열린 상태를 다른 코드/프로세스가 공유하는가
- `fork` 후 부모/자식이 어떻게 관리하는가
- `exec` 후에도 남아야 하는가
- `close` 이후 stale fd 재사용 가능성을 막고 있는가
- fd leak이 발생하면 어떤 시스템 자원이 계속 남는가
- `FILE *`와 fd를 혼용하고 있다면 소유권과 버퍼링을 명확히 했는가
- 이 자원은 blocking인가 nonblocking인가
- 이 fd를 단순 숫자가 아니라 lifecycle 있는 참조로 다루고 있는가

## 정리

파일 디스크립터는 단순한 파일 번호가 아니다.  
더 정확히 말하면, 파일 디스크립터는 **프로세스가 커널이 관리하는 I/O 자원에 접근하기 위해 사용하는 로컬 참조 슬롯**이다.

반드시 기억해야 할 핵심은 다음이다.

- fd는 자원 자체가 아니라 프로세스 로컬한 참조 번호다.
- fd와 open file description은 다르다.
- `dup`는 같은 열린 상태를 공유하는 새 참조를 만든다.
- 같은 파일을 다시 `open`하는 것은 보통 독립 열린 상태를 만든다.
- `close`는 참조 하나를 끊는 것이지 항상 자원 전체를 즉시 없애는 것은 아니다.
- fd 번호는 닫힌 뒤 재사용될 수 있다.
- `fork`와 `exec`는 fd 상속과 누출 문제를 만든다.
- pipe, socket, file, terminal을 하나의 fd 모델로 이해해야 이후 시스템 I/O가 연결된다.

즉, 파일 디스크립터를 이해한다는 것은  
“정수 하나로 파일을 읽는다”를 아는 것이 아니라,  
**프로세스-커널-I/O 자원 사이의 참조 모델과 그 생애주기를 이해하는 것**에 가깝다.

## 10. 더 깊게 볼 포인트

### 10.1 blocking과 nonblocking은 무엇이 다른가

다음 문서로 가장 자연스럽다.  
파일 디스크립터가 프로세스의 I/O 자원 참조 모델이라면, 그 다음은 그 참조가 호출 스레드를 어떻게 멈추게 하거나 멈추지 않게 하는지, 즉 blocking / nonblocking 의미를 보는 것이 맞다.

### 10.2 `dup`, `dup2`, `dup3`는 각각 어떤 문제를 해결하는가

fd aliasing, redirection, close-on-exec 안전성을 더 세밀하게 볼 수 있다.

### 10.3 `FILE *`와 fd는 어떤 계층 차이를 가지며 언제 섞으면 위험한가

stdio 버퍼링과 시스템 콜 계층의 차이를 더 깊게 연결할 수 있다.

### 10.4 fd leak은 왜 EOF, pipe, subprocess에서 특히 교묘한 버그를 만드는가

참조가 남아 있는 한 커널이 end-of-stream을 판단하지 못하는 사례들과 연결된다.

### 10.5 프로세스당 fd 한도와 서버 확장성은 어떤 관계를 가지는가

대량 연결 서버, event loop, connection fan-out과 fd 자원 관리가 어떻게 연결되는지로 확장할 수 있다.
