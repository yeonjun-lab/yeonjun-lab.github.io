## 1. 왜 이 주제를 깊게 봐야 하는가

`errno`는 C와 POSIX 계열 시스템 프로그래밍에서 가장 자주 보이는 이름 중 하나다.  
문제는 자주 보인다는 사실과 정확히 이해된다는 사실이 전혀 같지 않다는 점이다.

실무에서는 흔히 다음과 같이 이해한다.

- 함수가 실패하면 `errno`에 에러 코드가 들어간다.
- `errno`는 전역 변수다.
- 성공하면 `errno`는 0일 것이다.
- `errno`만 보면 무슨 일이 있었는지 항상 알 수 있다.
- 아무 함수 뒤에서나 `errno`를 찍으면 직전 실패 원인을 알 수 있다.

이 설명들은 부분적으로만 맞다.  
그리고 바로 그 “부분적으로 맞다”는 점이 위험하다.

`errno`를 얕게 이해하면 다음과 같은 코드가 생긴다.

- 성공한 호출 뒤의 `errno`를 검사해서 잘못된 에러 처리를 한다.
- 실패 직후 저장하지 않고 다른 함수들을 호출한 뒤 `errno`를 읽어 원인을 잃어버린다.
- `errno`를 단순 전역 변수처럼 생각해 thread safety 경계를 오해한다.
- 시스템 콜, 라이브러리 함수, 사용자 정의 함수의 에러 표현 방식을 같은 층위로 섞는다.
- 반환값과 `errno`의 역할 분리를 하지 못해 API 설계가 흐려진다.

`errno`를 깊게 봐야 하는 이유는 세 가지다.

첫째, `errno`는 “에러 코드 저장소”가 아니라, **반환값이 실패를 먼저 표현하고 그 실패의 상세 이유를 보조적으로 싣는 관례적 채널**이라는 점을 이해해야 한다.  
즉, `errno`를 먼저 보면 거의 반드시 구조를 잘못 잡게 된다.

둘째, `errno`는 언어 규칙, 표준 라이브러리 구현, 운영체제 인터페이스, thread-local storage 관례가 만나는 경계 사례다.  
즉, 이것은 단순 문법 요소가 아니라 시스템 인터페이스의 설계 관례를 보여 주는 사례다.

셋째, `errno`는 에러 처리 문화를 이해하는 출발점이다.  
왜 C 계열 API가 반환값 중심으로 설계되었는지, 왜 상세 오류는 별도 채널로 분리되는지, 왜 `goto cleanup` 같은 패턴이 등장하는지, 왜 스레드별 상태 저장이 필요한지 모두 여기와 연결된다.

이 문서의 핵심 목표는 `errno`를 “에러 번호”로만 소개하는 데 있지 않다.  
핵심은 다음을 분리하는 데 있다.

- 반환값과 `errno`의 역할은 어떻게 다르고 왜 분리되는가
- `errno`는 언제 의미가 있고 언제 의미가 없는가
- `errno`를 전역 변수처럼 이해하면 왜 틀리는가
- `errno`를 읽는 타이밍이 왜 중요한가
- thread safety, reentrancy, signal-safe 맥락에서 `errno`는 어떻게 달라지는가
- 실무 API 설계에서 `errno` 스타일을 어디까지 따르고 어디서 멈춰야 하는가

즉, 이 문서는 `errno`를 “에러를 담는 변수”로 설명하지 않는다.  
그 대신 `errno`를 **실패의 상세 이유를 운반하는 보조 채널**로 설명한다.

## 2. 전체 구조

`errno`를 정확히 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 호출 계약 층위

가장 먼저 봐야 할 것은 함수가 실패를 어떻게 표현하는가다.  
C/POSIX 계열 함수는 대체로 다음 두 단계를 가진다.

- 반환값으로 성공/실패 또는 1차 결과를 전달
- 실패한 경우에 한해 `errno`로 상세 이유를 보조 전달

즉, `errno`는 단독 채널이 아니라 호출 계약의 2차 채널이다.

### 2.2 라이브러리 구현 층위

소스 코드에서는 `errno`가 변수처럼 보이지만, 실제 구현에서는 단순 전역 정수 하나로 끝나지 않을 수 있다.  
특히 멀티스레드 환경에서는 thread-local storage와 연결되는 경우가 많다.

즉, `errno`는 “겉보기 인터페이스”와 “실제 저장 방식”을 분리해서 봐야 하는 대상이다.

### 2.3 운영체제 / 시스템 콜 층위

시스템 콜은 종종 커널 수준의 실패 사유를 사용자 공간에 전달해야 한다.  
이때 반환값과 함께 에러 번호 관례가 형성된다. libc는 이 경계를 감싸면서 호출자에게 일관된 인터페이스를 제공한다.

즉, `errno`는 단순히 libc 내부 취향이 아니라, 커널 실패 원인을 사용자 코드가 받아들이는 관문 중 하나다.

### 2.4 동시성 층위

`errno`는 오래된 코드에서는 전역 변수처럼 보이기 쉽다.  
그러나 실제 멀티스레드 프로그램에서는 “누구의 실패 상태인가”를 분리할 필요가 있다. 이 지점에서 thread-local storage가 중요해진다.

즉, `errno`는 thread safety와 reentrancy를 구분해 보기 좋은 사례다.

### 2.5 실무 설계 층위

모든 사용자 정의 C API가 `errno` 스타일을 그대로 따라야 하는 것은 아니다.  
어떤 경우는 반환값만으로 충분하고, 어떤 경우는 out-parameter + enum error가 낫고, 어떤 경우는 `errno` 스타일이 시스템 인터페이스와 잘 맞는다.

즉, `errno`는 따라야 할 교리가 아니라, 특정 계열의 인터페이스 관례다.

## 3. 내부 동작 원리

### 3.1 `errno`는 “실패를 알리는 신호”가 아니라 “실패 이유의 보조 저장소”다

이 점이 가장 중요하다.

처음 `errno`를 배우면 이렇게 이해하기 쉽다.

- 함수를 호출한다.
- `errno`를 본다.
- 에러가 있으면 실패다.

이 순서는 틀렸다.  
올바른 순서는 대체로 다음과 같다.

- 함수의 반환값을 본다.
- 그 반환값이 계약상 실패를 의미하면
- 그때 `errno`를 읽어 상세 이유를 본다.

예를 들어:

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        int saved_errno = errno;
        /* 실패 처리 */
    }

핵심은 `fd == -1`이 먼저고, `errno`는 그다음이라는 점이다.

왜 이 직관이 틀리는가?

- `errno`는 일반적으로 성공 시 자동 초기화되는 성공 플래그가 아니다.
- 어떤 함수는 성공해도 `errno`를 건드리지 않을 수 있다.
- 어떤 함수는 내부적으로 다른 호출을 하면서 `errno`를 바꿀 수 있다.
- 즉, `errno != 0`만으로 직전 호출 실패를 결론 내리면 안 된다.

### 3.2 성공 시 `errno == 0`을 기대하면 안 된다

이 오해는 매우 흔하다.

잘못된 예:

    errno = 0;
    p = some_function();
    if (errno != 0) {
        /* 실패라고 판단 */
    }

이 구조는 매우 위험하다.  
왜냐하면 많은 함수는 “성공 시 `errno`를 반드시 0으로 만든다”는 계약을 갖지 않기 때문이다.

즉, `errno`는 이런 의미가 아니다.

- 항상 현재 상태를 나타내는 전역 상태등

더 정확히는 이렇다.

- 어떤 함수가 실패를 반환한 경우, 그 실패 이유를 담기 위해 설정될 수 있는 값

따라서 `errno`는 성공 여부를 결정하는 1차 신호가 아니라, 실패한 호출의 부가 정보다.

### 3.3 왜 `errno`는 즉시 저장해야 하는가

다음 코드가 왜 위험한지 보자.

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        fprintf(stderr, "open failed: %s\n", strerror(errno));
        return -1;
    }

겉보기에는 괜찮아 보인다.  
하지만 더 복잡한 코드에서는 실패 직후 다른 함수 호출들이 `errno`를 바꿀 수 있다.

예:

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        log_path(path);
        cleanup_temp_files();
        fprintf(stderr, "open failed: %s\n", strerror(errno));
        return -1;
    }

여기서는 `log_path`, `cleanup_temp_files`, `fprintf`, 심지어 `strerror`를 부르는 과정 자체가 `errno`에 영향을 줄 여지가 있다.

따라서 실무에서는 실패 직후 바로 저장하는 패턴이 중요하다.

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        int saved_errno = errno;
        log_path(path);
        cleanup_temp_files();
        fprintf(stderr, "open failed: %s\n", strerror(saved_errno));
        return -1;
    }

즉, `errno`는 읽고 끝나는 정보가 아니라 **보존 타이밍이 중요한 휘발성 맥락 정보**다.

### 3.4 `errno`는 “전역처럼 보이지만 호출자 맥락에 귀속된 상태”다

표면적으로는 `errno`가 전역 변수처럼 보인다.

    extern int errno;

같은 오래된 설명을 떠올리기 쉽다.  
하지만 멀티스레드 환경에서 진짜 전역 정수 하나를 공유하면 문제가 생긴다.

- 스레드 A의 실패 원인이
- 스레드 B의 실패로 덮어써질 수 있다.

이런 이유로 실제 구현은 대개 thread-local storage를 사용해 스레드마다 별도 `errno` 상태를 가지게 만든다.

즉, `errno`는 “프로그램 전체의 마지막 에러”가 아니다.  
더 정확히는 “현재 실행 맥락, 특히 현재 스레드의 최근 실패 맥락”에 더 가깝다.

이 점은 매우 중요하다.

- `errno`를 전역 변수라고만 이해하면 thread safety를 오해한다.
- 반대로 TLS라고 해서 reentrancy까지 해결된다고 이해하면 또 틀린다.

### 3.5 TLS는 thread safety를 돕지만 `errno`를 마법처럼 안전하게 만들지 않는다

스레드별 `errno`는 여러 스레드가 서로의 실패 상태를 덮어쓰지 않게 해 준다.  
이것은 분명 thread safety 측면의 개선이다.

하지만 이것이 곧바로 모든 안전성을 뜻하지는 않는다.

- 같은 스레드 안에서 signal handler가 끼어들어 `errno`를 바꿀 수 있다.
- 중간에 호출한 라이브러리 함수가 같은 스레드의 `errno`를 바꿀 수 있다.
- callback 재진입에서도 현재 스레드의 `errno`는 같은 저장소를 공유할 수 있다.

즉, TLS는 “스레드별 분리”를 주지만 “호출별 분리”를 주지 않는다.  
이 점은 reentrancy 문서에서 보았던 TLS의 한계와 정확히 같은 구조다.

### 3.6 시스템 콜과 libc wrapper 경계에서 `errno`가 등장한다

시스템 콜은 보통 커널 내부 실패를 어떤 형식으로 사용자 공간에 전달해야 한다.  
libc는 이 커널 결과를 사용자 친화적인 함수 호출 계약으로 감싼다.

호출자는 보통 다음처럼 본다.

- 반환값이 실패를 나타낸다.
- `errno`가 실패 이유를 담는다.

여기서 중요한 점은 `errno`가 함수의 실제 주 결과가 아니라는 점이다.  
예를 들어 `read()`는 성공 시 읽은 바이트 수가 본질적 결과고, 실패 시에만 `errno`가 의미를 가진다.

즉, `errno`는 “주 결과가 실패한 경우에만 해석되는 보조 채널”이다.

### 3.7 모든 함수가 `errno`를 의미 있게 쓰는 것은 아니다

이 점도 반드시 분리해야 한다.

- POSIX 시스템 콜 wrapper 계열은 `errno`와 잘 맞는다.
- 순수 계산 함수는 보통 `errno`가 핵심 채널이 아니다.
- 라이브러리 함수에 따라 실패 표현 방식이 다르다.
- 사용자 정의 함수는 `errno`를 아예 쓰지 않는 편이 더 나을 수 있다.

즉, `errno`는 C 전체의 보편 에러 모델이 아니다.  
그것은 특히 시스템 인터페이스 계열에서 널리 쓰이는 관례다.

## 4. 핵심 구성 요소

### 4.1 반환값(return value)

에러 처리의 1차 채널이다.  
호출자는 먼저 반환값으로 성공/실패를 구분해야 한다.

대표 예:

- `open()`은 성공 시 fd, 실패 시 `-1`
- `read()`는 성공 시 읽은 바이트 수, 실패 시 `-1`
- `malloc()`은 성공 시 포인터, 실패 시 `NULL`

즉, `errno`는 항상 반환값 뒤에 온다.

### 4.2 실패 상세 코드(error number)

`errno`는 왜 실패했는지에 대한 상세 이유를 싣는다.

예:

- 권한 부족
- 파일 없음
- 인터럽트로 중단
- 리소스 부족
- 잘못된 인자
- 논블로킹 상태에서 지금은 읽을 수 없음

중요한 것은, 이것이 “실패 여부”가 아니라 “실패 종류”라는 점이다.

### 4.3 호출 직후 보존(saved errno)

실패 직후의 `errno`를 보존하는 변수는 실무적으로 매우 중요하다.

    int saved_errno = errno;

이 패턴이 필요한 이유는 다음과 같다.

- 이후 함수 호출이 `errno`를 바꿀 수 있다.
- 로그 출력도 안전하다고 가정하면 안 된다.
- 정리 코드가 길수록 원인 보존이 중요하다.

즉, `saved_errno`는 보조적인 편의가 아니라 에러 맥락 보존 장치다.

### 4.4 thread-local storage

현대적인 libc 구현에서 `errno`는 보통 스레드별로 분리된다.  
이것은 여러 스레드의 에러 상태 충돌을 피하기 위한 것이다.

그러나 이 점을 “`errno`는 완전히 안전하다”로 일반화하면 안 된다.  
TLS는 thread safety 일부를 돕지만, call-local 맥락 분리를 제공하지는 않는다.

### 4.5 문자열 변환 계층

`strerror(errno)` 같은 형태는 사람이 읽을 메시지를 만들기 위해 자주 쓰인다.  
하지만 문자열 변환은 본질이 아니다.

본질은 다음이다.

- 프로그램 논리는 숫자 코드 또는 명시적 분기 위에서 돌아간다.
- 문자열 메시지는 주로 진단, 로그, 사용자 표시를 위한 것이다.

즉, `errno`를 처리할 때는 “사람이 읽는 메시지”와 “프로그램 제어 흐름”을 분리해야 한다.

---

## 5. 실행 흐름 또는 처리 순서

### 5.1 올바른 기본 패턴

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        int saved_errno = errno;
        /* 실패 처리 */
    }

핵심 순서는 다음과 같다.

- 반환값 검사
- 실패 확인
- 즉시 `errno` 저장
- 이후 로그/정리/전파

### 5.2 잘못된 패턴: `errno`를 먼저 본다

    errno = 0;
    fd = open(path, O_RDONLY);
    if (errno != 0) {
        /* 실패라고 간주 */
    }

이 구조가 틀린 이유는 다음과 같다.

- 성공 시 `errno`가 0으로 초기화된다는 계약이 없다.
- 직전 다른 호출 흔적이 남아 있을 수 있다.
- 실패 여부를 `errno`가 아니라 반환값이 먼저 결정해야 한다.

### 5.3 잘못된 패턴: 너무 늦게 읽는다

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        cleanup();
        report_failure();
        return errno;
    }

이 구조는 `cleanup()`나 `report_failure()`가 `errno`를 바꿀 수 있으므로 위험하다.  
실패 원인을 잃어버릴 수 있다.

### 5.4 올바른 패턴: 늦게 쓸 일이 있으면 먼저 저장한다

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        int saved_errno = errno;
        cleanup();
        report_failure(saved_errno);
        return -1;
    }

이 구조는 실패 원인을 호출 직후 고정한다.  
즉, 에러 처리가 길어질수록 `saved_errno`의 중요성은 커진다.

### 5.5 `goto cleanup`과의 결합

실무 C 코드에서는 다음 패턴이 자주 나온다.

    int rc = -1;
    int fd = -1;

    fd = open(path, O_RDONLY);
    if (fd == -1) {
        goto cleanup;
    }

    /* 중간 작업 */

    rc = 0;

cleanup:
    if (rc != 0) {
        int saved_errno = errno;
        if (fd != -1) {
            close(fd);
        }
        errno = saved_errno;
    } else {
        if (fd != -1) {
            close(fd);
        }
    }
    return rc;

여기서 핵심은 자원 정리 과정이 원래 실패 원인을 덮어쓰지 않게 하는 것이다.  
즉, `goto cleanup` 패턴에서 `errno` 보존은 단순 선택이 아니라 설계 요소다.

### 5.6 `errno`를 반환값으로 복제해 전파하는 패턴

다음과 같은 wrapper도 흔하다.

    int my_open_readonly(const char *path, int *out_fd) {
        int fd = open(path, O_RDONLY);
        if (fd == -1) {
            return errno;
        }
        *out_fd = fd;
        return 0;
    }

이 구조는 내부적으로 `errno`를 사용하되, 외부 API는 명시적 정수 에러 코드를 반환하는 방식이다.  
즉, 시스템 계층과 상위 계층의 에러 모델을 분리하는 한 방법이다.

## 6. 성능 / 최적화 관점

### 6.1 `errno`는 단순 정수처럼 보여도 호출 경계 비용을 가진다

`errno` 자체는 정수 하나처럼 보이지만, 실제 비용은 그것을 읽고 쓰는 규약에 있다.

- 실패 시점 직후 저장해야 한다.
- TLS 접근 비용이 있을 수 있다.
- 문자열 변환은 추가 비용을 가진다.
- 로그/진단과 제어 흐름을 분리해야 한다.

즉, `errno`는 “공짜 전역 변수”가 아니라 호출 계약 비용이 있는 상태 채널이다.

### 6.2 에러 처리 경로를 길게 만들수록 `errno` 관리 비용이 커진다

성공 경로에서는 `errno`를 거의 신경 쓰지 않아도 된다.  
하지만 실패 경로가 복잡할수록 다음 비용이 커진다.

- 원인 보존
- 정리 코드 중 `errno` 훼손 방지
- 하위 계층의 실패를 상위 계층 의미로 번역
- 사용자 메시지 생성
- 부분 성공 상태 롤백

즉, `errno`는 성공 경로 성능 문제가 아니라 **실패 경로 구조화 문제**와 더 직접적으로 연결된다.

### 6.3 `errno`를 직접 노출하는 API가 항상 좋은 것은 아니다

하위 시스템 계층에서는 `errno` 스타일이 자연스럽다.  
하지만 상위 계층까지 그대로 끌고 오면 문제가 생길 수 있다.

- 호출 계약이 불명확해짐
- 반환값과 `errno`를 둘 다 해석해야 함
- 사용자 정의 에러 범주와 OS 에러 범주가 섞임
- 테스트가 어려워짐
- 인터페이스가 시스템 계층에 과도하게 종속됨

즉, `errno`는 시스템 인터페이스 가까운 곳에서는 유용하지만, 더 높은 추상화 계층에서는 명시적 에러 모델로 감싸는 편이 나을 수 있다.

### 6.4 문자열화는 진단용이지 제어 흐름용이 아니다

`strerror(errno)`를 남용하면 코드가 다음처럼 흐려진다.

- 문자열 비교로 에러 분기
- 로깅과 제어 흐름 혼합
- locale 의존 메시지를 프로그램 로직에 사용
- 테스트 안정성 저하

즉, 성능뿐 아니라 설계 측면에서도, 에러 코드는 코드로 다루고 문자열은 진단에만 쓰는 편이 낫다.

---

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
`errno`에 대한 오해는 주로 “전역 변수처럼 보이는 것”과 “실제로 보장되는 것”을 같은 층위로 다룰 때 생긴다.

### 7.1 반환값과 `errno`가 갈리는 지점

가장 먼저 분리해야 할 것은 이것이다.

- 반환값은 성공/실패 또는 주 결과를 전달한다.
- `errno`는 실패 시 상세 원인을 보조 전달한다.

즉, `errno`는 실패 판정 채널이 아니라 실패 해설 채널이다.  
이 순서를 뒤집으면 거의 반드시 잘못된 코드를 쓴다.

### 7.2 “전역처럼 보이는 인터페이스”와 “스레드별 저장”이 갈리는 지점

`errno`는 소스 수준에서는 전역 이름처럼 보인다.  
하지만 실구현에서는 thread-local일 수 있다. 이 둘은 같은 말이 아니다.

- 인터페이스: 어디서나 `errno`라고 쓴다.
- 구현: 현재 스레드에 귀속된 저장소를 가리킨다.

즉, `errno`를 단순 전역 정수라고 이해하면 thread safety를 과소평가하게 된다.

### 7.3 thread-local과 call-local이 갈리는 지점

이 경계도 매우 중요하다.

- TLS는 스레드 간 분리를 준다.
- 하지만 같은 스레드 안에서 호출 간 분리를 주지는 않는다.

즉, 함수 호출 중 signal handler나 callback이 들어와 `errno`를 바꾸면 같은 스레드 저장소를 공유한다.  
따라서 `errno`는 thread-safe에 가까운 일부 성질을 가질 수 있어도, reentrant-safe 상태 채널로 일반화하면 안 된다.

### 7.4 libc wrapper와 kernel 실패 이유가 갈리는 지점

호출자는 보통 `errno`를 libc 관점에서 본다.  
하지만 그 배경에는 커널 또는 시스템 계층의 실패 이유가 있다.

즉, `errno`는 운영체제 에러 의미를 사용자 코드에 옮기는 번역층 일부다.  
이 때문에 상위 비즈니스 로직에서는 직접 노출보다 번역이 필요할 수 있다.

### 7.5 진단 메시지와 프로그램 분기가 갈리는 지점

에러 번호 자체와 사람이 읽는 문자열은 같은 층위가 아니다.

- 에러 번호: 제어 흐름과 분기
- 에러 문자열: 로그, 진단, 사용자 메시지

이 둘을 섞으면 API 경계도, 테스트도, locale 의존성도 흐려진다.

### 7.6 성공 직후의 `errno`와 실패 직후의 `errno`가 갈리는 지점

이것도 매우 자주 틀린다.

- 실패 직후의 `errno`: 의미 있을 수 있음
- 성공 직후의 `errno`: 일반적으로 판단 근거가 아님

즉, `errno`는 항상 해석 가능한 현재 상태가 아니라, **실패라는 문맥이 먼저 확정될 때만 해석 가능한 상태**다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 `errno`는 실패 직후 즉시 저장하라

이 규칙은 거의 항상 맞다.

- 실패 반환값을 확인했다.
- 곧바로 `saved_errno = errno;`
- 그 후 로그, 정리, 상위 계층 번역

이 순서를 지키지 않으면 원인을 잃기 쉽다.

### 8.2 성공 여부를 `errno`로 판단하지 마라

반드시 반환값 계약을 먼저 보라.

- `-1`인가
- `NULL`인가
- 음수 에러 코드를 직접 반환하는가
- 0/비0 규약인가

성공을 `errno == 0`으로 판정하는 코드는 거의 항상 잘못 설계된 신호다.

### 8.3 사용자 정의 함수는 `errno` 의존을 신중히 선택하라

모든 내부 함수가 `errno`를 쓰면 API가 흐려진다.  
특히 시스템 계층 밖에서는 다음이 더 나을 수 있다.

- 명시적 enum 에러 반환
- `int` 상태 코드 반환 + out-parameter
- 구조화된 에러 객체
- 상위 추상화에 맞춘 에러 번역

즉, `errno`는 시스템 경계 가까이에서 특히 자연스럽다.

### 8.4 cleanup 코드가 원래 실패 원인을 덮어쓰지 않게 설계하라

`close`, `free`, 로그 함수, 보조 함수 호출이 원래 `errno`를 망가뜨릴 수 있다.  
따라서 실패 경로에서는 `errno` 보존을 cleanup 설계 일부로 봐야 한다.

### 8.5 `errno`를 문서화할 때는 “언제 의미가 있는지”까지 써야 한다

좋은 문서는 이렇게 말해야 한다.

- 이 함수는 실패 시 `-1`을 반환하고 `errno`를 설정할 수 있다.
- 성공 시 `errno` 값은 의미가 없다.
- 호출자는 실패 직후 `errno`를 보존해야 한다.

즉, 단순히 “에러 시 `errno` 설정”만 쓰면 부족하다.  
해석 가능한 문맥까지 써야 한다.

### 8.6 thread-safe와 reentrancy를 같이 보라

`errno`는 좋은 경계 사례다.

- 스레드별 저장 구현이라면 thread-safe 성질 일부를 가질 수 있다.
- 하지만 signal/callback 재진입에서는 같은 스레드의 `errno`가 덮일 수 있다.

즉, `errno`가 TLS라는 이유만으로 “재진입에도 안전하다”고 일반화하면 안 된다.

---

## 9. 판단 체크리스트

다음 질문에 명확히 답하지 못하면, 그 코드의 `errno` 처리는 아직 불안정하다.

- 성공/실패 판정을 반환값으로 먼저 하고 있는가
- 실패 직후 `errno`를 즉시 저장하는가
- cleanup/logging이 원래 `errno`를 덮어쓸 수 있다는 점을 고려했는가
- 성공 시 `errno`를 읽어 의미를 부여하고 있지 않은가
- 이 API는 정말 `errno` 스타일이 적합한가
- 상위 계층에서 `errno`를 직접 노출할지, 명시적 에러 코드로 번역할지 결정했는가
- thread-local과 call-local을 혼동하고 있지 않은가
- signal/callback 재진입에서 `errno`가 같은 스레드 맥락을 공유한다는 점을 고려했는가
- 에러 문자열과 프로그램 제어 흐름을 분리했는가
- 문서에서 `errno`가 언제 의미 있는지 명시했는가

## 정리

`errno`는 단순 전역 에러 변수가 아니다.  
더 정확히 말하면, `errno`는 **실패를 반환값으로 먼저 표현한 뒤, 그 실패의 상세 이유를 보조적으로 운반하는 상태 채널**이다.

반드시 기억해야 할 핵심은 다음이다.

- 반환값이 먼저고 `errno`는 그다음이다.
- `errno`는 성공 여부 판정용 플래그가 아니다.
- 성공 시 `errno` 값은 일반적으로 의미가 없다.
- 실패 직후 바로 저장하지 않으면 원인을 잃을 수 있다.
- `errno`는 전역처럼 보일 수 있지만 실제로는 스레드별 저장일 수 있다.
- TLS는 thread safety 일부를 돕지만 call-local 분리를 제공하지 않는다.
- `errno`는 시스템 인터페이스 계열에서 특히 자연스럽고, 상위 계층에서는 번역이 더 나을 수 있다.
- 문자열 메시지는 진단용이고, 프로그램 로직은 반환값과 코드 분기 위에 있어야 한다.

즉, `errno`를 잘 이해한다는 것은 단순히 에러 번호 이름을 외우는 것이 아니라,  
**C 계열 시스템 인터페이스가 실패를 어떻게 두 단계로 표현하는지 이해하는 것**에 가깝다.

## 10. 더 깊게 볼 포인트

### 10.1 C에서 에러 처리는 왜 반환값 중심인가

다음 문서로 가장 자연스럽다.  
`errno`를 이해했다면, 왜 C 계열 API가 실패를 먼저 반환값으로 표현하고 상세 사유를 별도 채널로 나누는지로 바로 이어지는 흐름이 맞다.

### 10.2 `goto cleanup` 패턴은 왜 실무에서 자주 쓰이는가

`errno` 보존, 자원 해제, 부분 실패 롤백이 어떻게 하나의 패턴으로 모이는지 연결해서 볼 수 있다.

### 10.3 자원 해제 순서는 왜 설계 대상인가

실패 경로가 길어질수록 `errno` 보존과 cleanup 순서가 얽힌다.  
즉, 자원 해제 순서와 에러 원인 보존은 따로가 아니라 함께 설계해야 한다.

### 10.4 thread-local storage는 어떤 문제를 해결하고 어떤 문제를 해결하지 않는가

`errno`는 TLS의 장점과 한계를 함께 보여 주는 좋은 사례다.  
thread-local과 call-local의 경계를 더 깊게 볼 수 있다.

### 10.5 시스템 에러 코드를 상위 계층 에러 모델로 번역할 때 무엇을 기준으로 나눠야 하는가

POSIX 계층의 상세 오류와 비즈니스 로직 계층의 오류를 분리하는 기준을 더 정교하게 다룰 수 있다.
