## 1. 왜 이 주제를 깊게 봐야 하는가

초급 단계에서는 자원 해제를 흔히 이렇게 배운다.

- `malloc` 했으면 `free`
- `open` 했으면 `close`
- 락을 잡았으면 `unlock`
- 파일을 열었으면 `fclose`

이 설명 자체는 틀리지 않는다.  
하지만 실무에서는 곧바로 다음 문제가 등장한다.

- 자원이 여러 개면 무엇부터 해제해야 하는가
- 어떤 자원은 다른 자원 위에 성립하는데, 해제 순서를 바꾸면 왜 문제가 되는가
- partially initialized object는 어떻게 되감아야 하는가
- cleanup 중 실패는 어떻게 다뤄야 하는가
- 소유권이 이미 다른 곳으로 넘어간 자원을 누가 해제해야 하는가
- thread, callback, file descriptor, buffer, mutex, mapping이 얽혀 있을 때 무엇이 먼저 사라져야 하는가

즉, 자원 해제는 단순히 “마지막에 치우는 일”이 아니다.  
그것은 프로그램이 만들었던 상태와 의존 관계를 **안전한 순서로 해체하는 과정**이다.

이 주제를 깊게 봐야 하는 이유는 세 가지다.

첫째, 자원 해제 순서는 correctness의 일부다.  
많은 초보자는 생성은 설계하고 해제는 뒤에 붙이는 후처리로 생각한다. 하지만 실제로는 생성과 해제가 하나의 짝을 이룬다. 어떤 순서로 획득했는지, 누가 누구를 참조하는지, 무엇이 무엇의 유효성을 전제하는지를 모르면 올바른 teardown을 만들 수 없다.

둘째, 잘못된 해제 순서는 단순 누수만 만드는 것이 아니다.  
다음과 같은 심각한 문제로 이어질 수 있다.

- use-after-free
- double close / double free
- unlock of unheld mutex
- flush 되지 않은 데이터 유실
- partially destroyed state 노출
- callback이 이미 파괴된 객체를 참조
- 자식 객체가 부모 자원에 접근하다 붕괴

즉, 자원 해제 순서는 메모리 안전성과 시스템 인터페이스 안정성을 동시에 건드린다.

셋째, C에서는 자동 unwinding이 기본 제공되지 않기 때문에, teardown의 순서와 범위를 호출자가 직접 설계해야 한다.  
이 때문에 자원 해제는 문법 문제가 아니라 API, ownership, lifetime, cleanup path 설계 문제다.

이 문서의 핵심 목표는 “역순으로 해제하면 된다”는 단순한 직관을 넘는 데 있다.  
핵심은 다음을 분리하는 것이다.

- 왜 역순 해제가 자주 맞는가
- 언제 단순 역순만으로는 부족한가
- ownership, lifetime, dependency, publication이 teardown 순서를 어떻게 결정하는가
- cleanup 중 실패는 무엇을 의미하는가
- 실무에서는 해제 순서를 어떤 기준으로 문서화하고 검증해야 하는가

즉, 이 문서는 자원 해제를 “끝나고 치우는 코드”로 설명하지 않는다.  
그 대신 자원 해제를 **생성 시 세운 의존 관계를 안전하게 해체하는 역방향 설계**로 설명한다.

## 2. 전체 구조

자원 해제 순서를 정확히 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 소유권(ownership) 층위

가장 먼저 봐야 할 것은 “누가 이 자원을 해제할 책임을 가지는가”다.  
소유권이 분명하지 않으면 해제 순서 논의 자체가 흔들린다.

- 현재 함수가 소유하는가
- 호출자에게 넘겼는가
- 공유 참조인가
- 참조 카운트 기반인가
- 부모 객체가 자식 자원을 포함하는가

즉, 해제 순서는 ownership model 위에서만 의미를 가진다.

### 2.2 의존 관계(dependency) 층위

자원은 독립적이지 않은 경우가 많다.

- 버퍼는 파일 디스크립터에 연결된 상태일 수 있다
- 파서 컨텍스트는 입력 버퍼와 테이블을 전제로 한다
- 워커 스레드는 큐와 종료 플래그를 참조한다
- 매핑은 fd와 별개일 수 있지만, 상위 상태 머신과 결합될 수 있다
- 락 아래 보호되는 객체는 unlock 전까지 불변식을 유지해야 한다

즉, 해제 순서는 자원의 획득 순서가 아니라 **실제 의존 순서**를 따라야 한다.

### 2.3 생애주기(lifetime) 층위

해제는 단순 함수 호출이 아니라 어떤 객체나 자원의 lifetime을 끝내는 행위다.  
따라서 해제 시점과 순서는 “누가 아직 접근 가능한가”와 직접 연결된다.

- 아직 다른 스레드가 보고 있는가
- callback이 늦게 도착할 수 있는가
- 상위 객체가 자식 포인터를 여전히 노출 중인가
- publication을 먼저 끊어야 하는가

즉, teardown은 값 삭제가 아니라 **접근 가능성 종료 절차**다.

### 2.4 실패 경로(error path) 층위

정상 종료와 실패 종료는 같은 순서를 따르지 않을 수 있다.  
초기화 도중 3단계까지만 성공했다면, cleanup은 그 3단계만 되감아야 한다.

즉, 해제 순서는 정상 종료용 one-shot destructor와 실패 rollback path를 분리해서 봐야 한다.

### 2.5 실무 운영 층위

실제 시스템에서는 자원 해제 순서가 다음과 연결된다.

- 로그 flush
- 네트워크 shutdown handshake
- 파일 flush/sync
- worker thread join
- event source deregistration
- signal/callback unhook
- metrics exporter 종료
- process exit 시점 자원 정리

즉, teardown 순서는 시스템 동작 의미를 바꿀 수 있다.

## 3. 내부 동작 원리

### 3.1 왜 “대체로 역순 해제”가 자주 맞는가

가장 널리 알려진 직관은 이것이다.

- 획득한 순서의 반대로 해제하라

이 직관은 자주 맞다.  
왜냐하면 많은 경우 자원 획득이 이미 의존 순서를 반영하기 때문이다.

예를 들어:
```c
    buf = malloc(...);
    fd = open(...);
    lock();
    init_parser(buf, fd);
```
이 구조라면 흔히 다음이 맞다.

- parser 정리
- unlock
- close(fd)
- free(buf)

왜 이런 역순이 자연스러운가?

- 나중에 만든 상태가 앞서 만든 상태에 기대는 경우가 많다
- 마지막에 획득한 자원이 가장 안쪽 문맥에 있다
- 따라서 먼저 만들어진 기반 자원을 먼저 없애면 상위 자원이 dangling state가 되기 쉽다

즉, 역순 해제는 단순 암기 규칙이 아니라, **중첩된 의존 관계를 바깥으로부터가 아니라 안쪽부터 해체하는 방식**이라서 자주 맞는다.

하지만 중요한 단서가 있다.  
역순이 맞는 것은 획득 순서가 실제 dependency order를 잘 반영할 때다.  
그 둘이 어긋나면 단순 역순만 외우면 틀릴 수 있다.

### 3.2 해제 순서의 본질은 “의존 그래프를 역방향으로 끊는 것”이다

실무적으로 더 정확한 표현은 이것이다.

- 자원 해제 순서는 acquisition order가 아니라 dependency graph의 reverse order를 따라야 한다.

예를 들어 어떤 객체가 다음을 가진다고 하자.

- worker thread
- message queue
- socket
- parser buffer

worker thread는 queue와 socket과 buffer를 참조할 수 있다.  
이때 단순히 “나중에 만든 것부터 free”만 보면 위험하다.

잘못된 해제:

- queue free
- buffer free
- socket close
- thread join

왜 위험한가?

- worker thread가 아직 실행 중일 수 있다
- queue/buffer/socket을 여전히 참조할 수 있다
- 즉, publication이 살아 있는 동안 기반 자원을 먼저 없애 버린다

더 올바른 해제는 보통 이렇다.

- 외부에서 더 이상 새 작업이 들어오지 않게 publication 차단
- worker에 종료 신호 전달
- thread join
- queue 정리
- socket close
- buffer free

즉, teardown은 “무엇을 먼저 지우느냐”보다 “누가 아직 그것을 볼 수 있는가”를 먼저 따져야 한다.

### 3.3 publication을 끊기 전에 파괴하면 왜 위험한가

자원 해제에서 매우 자주 놓치는 개념이 publication이다.

- 전역 포인터에 등록됨
- callback 대상 목록에 들어 있음
- event loop에 핸들러로 연결됨
- 다른 스레드가 큐나 레지스트리로 참조 가능
- 외부 API에 핸들 노출됨

이런 상태의 객체는 “아직 관찰 가능한 상태”다.  
즉, 단순히 더 이상 이 함수에서 안 쓴다고 해서 lifetime이 끝난 것이 아니다.

왜 이 직관이 틀리는가?

많은 사람은 이렇게 생각한다.

- 더 이상 필요 없으니 free한다
- 그 다음 레지스트리에서 뺀다

하지만 실제로는 순서가 반대여야 한다.

- 먼저 새 참조가 생기지 않게 publication을 끊고
- 기존 사용자가 끝났는지 확인하고
- 그다음 자원을 해제해야 한다

즉, teardown의 핵심은 “지우기”보다 **도달 가능성과 접근 가능성을 먼저 끊는 것**이다.

### 3.4 cleanup과 rollback은 같은 듯 보이지만 완전히 같지 않다

초기화 도중 실패한 경우와 정상 종료 시 destructor를 호출하는 경우를 분리해야 한다.

예를 들어:

- 1단계: buf 할당
- 2단계: fd open
- 3단계: parser init
- 4단계: thread start

3단계에서 실패했다면 rollback은 다음만 하면 된다.

- parser partial state 정리
- close(fd)
- free(buf)

하지만 정상 종료 시에는 thread 종료, queue drain, flush, callback deregistration 같은 더 큰 절차가 필요할 수 있다.

즉, cleanup 순서는 하나의 고정 레시피가 아니다.

- initialization failure rollback
- normal shutdown teardown
- abort path emergency cleanup

이들은 겹치지만 완전히 같지 않을 수 있다.  
이 차이를 무시하면 partially initialized object에 full destructor를 호출해 버리거나, 정상 종료 절차를 실패 cleanup에 무리하게 섞어 버릴 수 있다.

### 3.5 해제 중 실패는 “정리 실패”와 “원래 실패”를 분리해서 봐야 한다

자원 해제 함수도 실패할 수 있다.

- `close()`가 실패할 수 있다
- `fflush()`가 실패할 수 있다
- `pthread_mutex_unlock()`도 문맥에 따라 문제 신호를 줄 수 있다
- network shutdown이 완전히 끝나지 않을 수 있다

이때 초보자는 종종 하나로 뭉뚱그린다.

- 실패했으니 어쨌든 실패다

하지만 실무적으로는 두 종류를 분리해야 한다.

- 원래 작업 실패
- 정리 도중 추가 실패

예를 들어 파일 쓰기 실패 뒤 `close()`도 실패할 수 있다.  
이때 어떤 오류를 호출자에게 돌려줄지, 로그에는 둘 다 남길지, cleanup 실패는 진단용으로만 취급할지 정책이 필요하다.

즉, teardown은 성공/실패 이분법이 아니라 **원래 실패와 정리 실패가 겹칠 수 있는 다단계 에러 모델**을 가진다.

### 3.6 자원 해제 순서는 불변식(invariant)을 마지막까지 유지해야 한다

자원을 해제할 때도 객체 불변식이 중요하다.

예를 들어 락이 보호하는 구조체가 있다고 하자.

- 리스트 head/tail
- size
- 내부 버퍼
- 상태 플래그

이 상태를 파괴하는 동안 락을 너무 일찍 풀면, 다른 실행 흐름이 partially destroyed object를 볼 수 있다.  
즉, teardown에서도 중요한 것은 “해제해도 되는가”가 아니라 “해제 과정 중에도 외부가 일관되지 않은 상태를 볼 수 없는가”다.

이 점에서 해제 순서는 메모리 free 순서만의 문제가 아니다.  
그것은 **객체 불변식을 언제까지 유지하고 언제 관찰 불가능하게 만들 것인가**의 문제다.

## 4. 핵심 구성 요소

### 4.1 ownership

자원 해제의 출발점이다.

- 누가 해제 책임을 가지는가
- 성공 시 소유권이 caller로 넘어가는가
- 실패 시 현재 함수가 rollback하는가
- 공유 자원인가 독점 자원인가

소유권이 모호하면 해제 순서도 모호해진다.

### 4.2 dependency

자원끼리의 의존 관계를 명시해야 한다.

- worker → queue
- parser → buffer
- stream → fd
- object field → backing allocation
- callback registration → object lifetime

이 dependency를 적지 않으면 teardown 순서는 추측에 맡겨진다.

### 4.3 publication state

객체가 외부에 노출되었는가를 나타내는 상태다.

- 전역 레지스트리에 등록됨
- 다른 스레드가 볼 수 있음
- callback target으로 등록됨
- 이벤트 루프 감시 대상
- handle 반환 완료

이 publication이 살아 있으면 free 이전에 먼저 끊어야 할 수 있다.

### 4.4 lifetime markers

실무에서는 다음과 같은 마커가 필요할 수 있다.

- initialized
- started
- registered
- joined
- closed
- destroyed

이 마커는 단순 상태 플래그가 아니라 teardown 순서를 안전하게 결정하기 위한 판단 근거다.

### 4.5 rollback / teardown split

같은 cleanup이라도 종류가 다르다.

- 단계별 initialization rollback
- 정상 종료 teardown
- 비정상 종료 emergency cleanup

이 구분이 없으면 partially initialized state에 잘못된 destructor를 호출하거나, 정상 종료가 요구하는 flush/join/deregister 절차를 건너뛰기 쉽다.

## 5. 실행 흐름 또는 처리 순서

### 5.1 단순 자원 두 개의 역순 해제
```c
    char *buf = malloc(1024);
    if (buf == NULL) {
        return -1;
    }

    int fd = open(path, O_RDONLY);
    if (fd == -1) {
        free(buf);
        return -1;
    }

    /* 사용 */

    close(fd);
    free(buf);
```
이 경우 역순 해제가 자연스럽다.  
`fd` 사용은 `buf`에 의존하지 않을 수 있지만, 최소한 생성 경로와 cleanup 경로가 단순하게 대응한다.

### 5.2 thread가 참조하는 자원은 먼저 free하면 안 된다

잘못된 예:
```c
    free(ctx->queue);
    close(ctx->fd);
    pthread_join(ctx->worker, NULL);
```
왜 틀릴 수 있는가?

- worker가 아직 queue나 fd를 참조할 수 있다
- join 전까지 worker lifetime은 끝나지 않았다

더 나은 구조:
```c
    signal_stop(ctx);
    pthread_join(ctx->worker, NULL);
    free(ctx->queue);
    close(ctx->fd);
```
핵심은 “누가 아직 참조 중인가”를 먼저 끊는 것이다.

### 5.3 callback 등록 객체의 해제 순서

잘못된 예:
```c
    free(obj);
    unregister_callback(loop, obj);
```
이 순서는 위험하다.  
event loop나 다른 스레드가 이미 `obj`를 callback target으로 들고 있을 수 있다.

더 나은 구조:
```c
    unregister_callback(loop, obj);
    drain_pending_callbacks_if_needed();
    free(obj);
```
즉, publication 해제 → 기존 실행 종료 확인 → 객체 해제가 더 자연스럽다.

### 5.4 partially initialized object rollback
```c
    obj = malloc(sizeof(*obj));
    if (obj == NULL) {
        return -1;
    }

    obj->buf = malloc(1024);
    if (obj->buf == NULL) {
        free(obj);
        return -1;
    }

    obj->fd = open(path, O_RDONLY);
    if (obj->fd == -1) {
        free(obj->buf);
        free(obj);
        return -1;
    }
```
이 경우 full destructor가 아직 없거나, object publication이 일어나지 않았을 수 있다.  
즉, rollback은 현재까지 획득한 필드만 되감는 구조여야 한다.

### 5.5 lock 보호 객체 teardown

잘못된 예:
```c
    pthread_mutex_unlock(&obj->lock);
    free(obj->buf);
    obj->buf = NULL;
    obj->state = DESTROYED;
```
이 순서는 위험하다.  
다른 스레드가 unlock 이후 `obj`를 보고 partially destroyed state를 관찰할 수 있다.

더 나은 방향은 보통 다음 중 하나다.

- destruction 전체를 외부에서 이미 exclusive하게 보장
- publication을 먼저 끊고 참조 종료를 보장
- lock 아래에서 상태 전이를 완료하고 외부 접근 경로를 차단
- 그다음 실제 자원 해제

즉, teardown에서도 invariant 유지 경계가 중요하다.

## 6. 성능 / 최적화 관점

### 6.1 잘못된 해제 순서는 correctness를 깨뜨리고 성능 문제처럼 보일 수 있다

실무에서는 teardown bug가 종종 다음처럼 보인다.

- 가끔 크래시 난다
- shutdown이 느리다
- fd leak이 생긴다
- 메모리가 회수되지 않는다
- 종료 시점에만 race가 난다

이때 사람들은 처음에는 성능 문제나 드문 운영 이슈로 보기도 한다.  
하지만 실제로는 해제 순서가 틀려 lifetime과 ownership이 깨진 경우가 많다.

즉, teardown bug는 성능 이슈로 위장된 correctness bug일 수 있다.

### 6.2 너무 일찍 해제하면 빠른 것이 아니라 위험한 것이다

다음과 같은 잘못된 최적화 직관이 있다.

- 안 쓰는 것 같으니 빨리 free하자
- fd는 어차피 곧 안 쓰니 먼저 닫자
- lock은 오래 잡으면 느리니 빨리 풀자

이 직관은 단일 흐름에서는 맞아 보일 수 있다.  
하지만 다른 스레드, callback, delayed use, buffered I/O, pending work가 있으면 치명적이다.

즉, 자원 해제는 “최대한 빨리”가 아니라 **더 이상 누구도 안전하게 참조하지 않게 된 뒤에** 해야 한다.

### 6.3 teardown 설계는 정상 경로 성능에도 영향을 준다

좋은 teardown 구조는 생성 경로 설계까지 바꾼다.

- ownership이 명확해야 한다
- partially initialized state를 식별할 수 있어야 한다
- join/shutdown point가 명확해야 한다
- publication/deregistration 경계가 분명해야 한다

즉, cleanup을 뒤에 붙이는 코드베이스는 생성 경로도 흐려지기 쉽다.  
반대로 해제 순서를 먼저 설계하면 생성 순서와 상태 전이도 더 명확해진다.

### 6.4 참조 종료 확인 비용과 자원 회수 속도는 trade-off일 수 있다

특히 동시성 환경에서는 다음 선택이 생긴다.

- 즉시 free하고 싶다
- 하지만 아직 다른 실행 흐름이 볼 수 있다
- 그래서 join, reference drain, grace period, refcount zero wait가 필요하다

이것은 단순 낭비가 아니라 correctness를 위한 비용이다.  
즉, teardown latency를 줄이기 위한 최적화는 종종 lifetime 안전성과 직접 충돌한다.

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 핵심이다.  
자원 해제 순서에 대한 오해는 보통 “획득의 역순”이라는 기억법을 만능 규칙처럼 사용할 때 생긴다.

### 7.1 acquisition order와 dependency order가 갈리는 지점

많은 경우 둘은 비슷하지만, 항상 같지는 않다.

- 단순 경우: 나중에 획득한 자원이 앞선 자원에 의존  
  역순 해제가 맞음
- 복잡한 경우: 별도 publication, worker lifetime, callback registry, refcount가 얽힘  
  단순 역순만으로는 부족

즉, 진짜 기준은 acquisition order가 아니라 dependency order다.

### 7.2 소유권 종료와 접근 종료가 갈리는 지점

어떤 함수가 “이제 내 소유가 아니다”라고 생각해도, 외부에서 아직 접근할 수 있으면 lifetime은 끝난 것이 아니다.

- 소유권 회수
- publication 해제
- 기존 참조 종료
- 최종 free

이 단계들은 종종 다르다.  
즉, free 시점 판단은 ownership만으로 충분하지 않고 reachability까지 봐야 한다.

### 7.3 rollback과 destructor가 갈리는 지점

초기화 실패 cleanup과 정상 종료 destructor는 겹칠 수 있지만 완전히 같은 것은 아니다.

- rollback: 현재까지 획득한 것만 되감기
- destructor: 완전 초기화 + publication + runtime state를 종료

이 둘을 혼동하면 partial state에 full teardown을 적용해 버리거나, 정상 종료에 필요한 절차를 누락할 수 있다.

### 7.4 cleanup 함수 호출과 시스템 의미 종료가 갈리는 지점

`close(fd)` 같은 한 줄이 시스템 의미 전체 종료를 뜻하지 않을 수 있다.

- buffered stream이면 flush 의미가 따로 있다
- network socket이면 shutdown handshake가 따로 있다
- thread면 join이 필요하다
- callback source면 unregister가 먼저다

즉, “해제 함수 하나 호출했다”와 “시스템 의미상 안전하게 종료했다”는 다를 수 있다.

### 7.5 메모리 해제와 객체 파괴가 갈리는 지점

`free(obj)`는 메모리 회수다.  
하지만 객체 파괴는 더 넓은 과정일 수 있다.

- 내부 필드 정리
- 자식 자원 해제
- publication 해제
- worker 종료
- pending work drain
- state transition to destroyed

즉, 메모리 free는 destructor의 마지막 단계일 수는 있어도 전체가 아니다.

## 8. 실무에서 중요한 판단 기준

### 8.1 먼저 “누가 아직 이것을 볼 수 있는가”를 물어라

자원 해제 순서를 결정할 때 가장 먼저 물어야 할 질문은 이것이다.

- 이 자원을 아직 참조할 수 있는 스레드가 있는가
- callback이 들어올 수 있는가
- 전역/레지스트리/큐를 통해 도달 가능한가
- 자식 객체가 부모 자원을 아직 사용 중인가

이 질문에 답하지 못하면 free 순서를 정할 수 없다.

### 8.2 역순 해제를 기본값으로 두되, dependency를 먼저 검증하라

“대체로 역순”은 좋은 출발점이다.  
하지만 다음을 반드시 검증해야 한다.

- 획득 순서가 실제 의존 순서를 반영하는가
- publication은 별도 단계인가
- thread/callback lifetime은 자원 획득 순서와 분리되는가
- rollback과 full teardown을 같은 코드로 처리해도 되는가

즉, 역순은 기억법이지 면책 규칙이 아니다.

### 8.3 소유권 이전 지점은 코드와 문서에 명시하라

다음 순간부터 cleanup 책임이 바뀔 수 있다.

- 함수가 포인터를 caller에 반환
- 객체 필드에 fd 저장 완료
- refcount 증가 후 공유 레지스트리에 등록
- thread 시작 후 ctx를 worker가 소유

이 지점을 명시하지 않으면 double free와 leak이 모두 생긴다.

### 8.4 teardown에서 publication 해제를 먼저 고려하라

복잡한 객체라면 free보다 먼저 봐야 할 것이 있다.

- unregister
- stop accepting new work
- mark as closing/destroying
- detach from registry
- cancel timers
- disable callbacks

즉, destruction은 종종 “먼저 보이지 않게 하고, 그다음 안전하게 없애는” 2단계 과정이다.

### 8.5 cleanup 실패 정책을 분리하라

다음 질문을 미리 정해야 한다.

- 원래 실패와 cleanup 실패 중 무엇을 반환할 것인가
- cleanup 실패는 로그만 남길 것인가
- destructor는 실패를 무시 가능한가
- shutdown path에서 일부 실패를 best-effort로 둘 것인가

정책이 없으면 현장 코드가 제각각 된다.

### 8.6 destructor와 rollback helper를 분리할지 판단하라

다음 경우 분리가 유리하다.

- partial initialization 단계가 많음
- 정상 종료 절차가 rollback보다 훨씬 큼
- publication 이후와 이전의 상태가 크게 다름
- join/unregister/drain 절차가 있음

즉, 모든 정리를 하나의 함수에 몰면 오히려 복잡성이 커질 수 있다.

## 9. 판단 체크리스트

다음 질문에 명확히 답하지 못하면, 그 teardown 설계는 아직 불안정하다.

- 각 자원의 소유권은 누구에게 있는가
- 각 자원은 무엇에 의존하고 무엇이 그것에 의존하는가
- 아직 이 자원을 볼 수 있는 다른 실행 흐름이 있는가
- publication 해제를 free보다 먼저 해야 하는가
- initialization rollback과 normal teardown은 같은가 다른가
- 자원 해제 순서는 dependency graph의 역방향과 일치하는가
- cleanup 중 실패를 어떻게 처리할 정책이 있는가
- partially destroyed state가 외부에 보일 가능성은 없는가
- lock, callback, thread, fd, memory 중 무엇이 lifetime 경계를 결정하는가
- 이 해제 순서를 코드와 문서에서 추론 가능하게 표현하고 있는가

## 정리

자원 해제 순서는 “마지막에 치우는 순서”가 아니다.  
더 정확히 말하면, 자원 해제 순서는 **생성 과정에서 세운 ownership, dependency, publication, lifetime 관계를 안전하게 역방향으로 해체하는 설계**다.

반드시 기억해야 할 핵심은 다음이다.

- 해제 순서는 ownership 없이는 정의되지 않는다.
- 자주 맞는 규칙은 역순 해제지만, 진짜 기준은 dependency order의 역방향이다.
- publication을 먼저 끊지 않으면 free는 너무 이르다.
- rollback과 normal teardown은 같아 보이지만 다를 수 있다.
- cleanup 중 실패는 원래 실패와 분리해서 다뤄야 한다.
- free 한 줄이 곧 객체 파괴 전체를 뜻하지는 않는다.
- teardown에서도 invariant와 접근 가능성 제어가 중요하다.

즉, 자원 해제 순서를 설계한다는 것은  
“뭘 먼저 free할까”를 정하는 것이 아니라,  
**어떤 상태가 누구에게 언제까지 보일 수 있는지와 어떤 의존 관계를 어떤 순서로 끊을지 설계하는 것**에 가깝다.

## 10. 더 깊게 볼 포인트

### 10.1 함수형 매크로는 왜 위험한가

다음 우선순위 축으로 자연스럽게 넘어간다.  
시스템 인터페이스와 자원 관리 축을 마쳤다면, 이제 전처리기 심화로 넘어가며 “겉보기 함수 호출”과 “실제 언어 규칙”이 갈리는 지점을 다루는 흐름이 자연스럽다.

### 10.2 partial initialization과 full destruction은 왜 같은 코드로 합치기 어려운가

객체 생성/파괴 설계에서 rollback helper와 destructor split을 더 정교하게 볼 수 있다.

### 10.3 reference counting은 teardown 순서를 어떻게 바꾸는가

독점 소유권이 아니라 공유 소유권일 때 free 시점과 publication 해제 순서가 더 복잡해진다.

### 10.4 thread shutdown, callback drain, event loop deregistration은 왜 free보다 먼저 고려해야 하는가

동시성과 lifetime 문제가 얽힌 teardown의 대표 사례들이다.

### 10.5 cleanup 실패를 사용자에게 어떻게 드러낼지, 내부 로그로만 남길지는 어떤 기준으로 정해야 하는가

원래 실패와 정리 실패가 겹칠 때의 정책 설계를 더 깊게 다룰 수 있다.
