## 1. 왜 이 주제를 깊게 봐야 하는가

캐시 친화적 접근이 왜 중요한지 이해했다면, 다음으로 자연스럽게 보게 되는 주제가 있다.  
바로 **데이터를 메모리에 어떤 모양으로 배치할 것인가**다.

겉으로 보기에는 같은 정보를 담는 구조라도,  
배치 방식에 따라 실제 성능은 크게 달라질 수 있다.  
대표적인 두 방식이 바로 다음이다.

- AoS: Array of Structures
- SoA: Structure of Arrays

초보자는 종종 이 차이를 단순 문법 차이 정도로 본다.

- 구조체 배열이냐
- 배열 여러 개냐

하지만 실제로는 이 차이가 다음에 직접 영향을 준다.

- 캐시 적중률
- 메모리 대역폭 사용량
- 벡터화 가능성
- 특정 필드만 반복 처리할 때의 효율
- 데이터 복사 비용
- API 설계 방식

예를 들어 3차원 좌표 데이터를 처리한다고 하자.  
다음 두 표현은 논리적으로 같은 정보를 담을 수 있다.

AoS 방식:

```c
typedef struct {
    float x;
    float y;
    float z;
} Point;
```

```c
Point points[N];
```

SoA 방식:

```c
typedef struct {
    float x[N];
    float y[N];
    float z[N];
} Points;
```

둘 다 `N`개의 점을 표현할 수 있다.  
하지만 실제 처리 패턴이 다음과 같다면 어떨까?

```c
for (size_t i = 0; i < N; i++) {
    sum += points[i].x;
}
```

또는 SoA에서는:

```c
for (size_t i = 0; i < N; i++) {
    sum += points.x[i];
}
```

겉보기엔 비슷해 보이지만, 실제 메모리 접근 패턴은 다르다.

- AoS에서는 `x` 사이에 `y`, `z`가 끼어 있다
- SoA에서는 `x` 값들만 연속으로 모여 있다

즉, “어떤 필드만 집중적으로 처리하는가”에 따라 더 좋은 배치가 달라질 수 있다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- 데이터 배치가 캐시 친화성과 어떻게 연결되는지 구체적으로 보여 준다
- 구조체 설계가 단순 코드 스타일이 아니라 성능 설계임을 이해하게 해 준다
- 고성능 루프, 수치 계산, 그래픽스, AI/시뮬레이션 코드에서 왜 SoA가 자주 나오는지 설명해 준다
- 반대로 일반 비즈니스 로직에서 왜 AoS가 더 자연스러운 경우가 많은지 이해하게 해 준다
- “같은 정보라도 배치가 다르면 비용 모델이 달라진다”는 것을 분명히 보여 준다

이 문서는 AoS와 SoA가 각각 무엇인지, 메모리 배치가 어떻게 다른지,  
어떤 접근 패턴에서 어떤 방식이 유리한지,  
그리고 실무에서 어떤 기준으로 선택해야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

AoS와 SoA를 큰 구조로 보면 다음과 같다.

### 2.1 AoS는 “객체 단위로 묶는 배치”다

AoS(Array of Structures)는 구조체 하나가 하나의 완전한 객체를 표현하고,  
그 구조체들이 배열로 나열되는 방식이다.

예를 들어:

```c
typedef struct {
    float x;
    float y;
    float z;
} Point;
```

```c
Point points[N];
```

이 경우 메모리에는 대략 다음처럼 놓인다.

- x0, y0, z0
- x1, y1, z1
- x2, y2, z2

즉, 객체 하나의 필드들이 서로 가까이 있다.

### 2.2 SoA는 “필드 단위로 묶는 배치”다

SoA(Structure of Arrays)는 각 필드가 따로 배열이 되는 방식이다.

예를 들어:

```c
typedef struct {
    float x[N];
    float y[N];
    float z[N];
} Points;
```

이 경우 메모리에는 대략 다음처럼 놓인다.

- x0, x1, x2, ...
- y0, y1, y2, ...
- z0, z1, z2, ...

즉, 같은 필드 값들이 연속 메모리에 모여 있다.

### 2.3 둘은 논리 구조보다 접근 패턴에서 차이가 난다

AoS와 SoA는 담는 정보 자체는 같을 수 있다.  
핵심 차이는 정보의 의미가 아니라 **어떻게 접근할 것이냐**다.

- 객체 단위로 자주 접근하는가
- 특정 필드만 반복적으로 처리하는가
- 여러 필드를 한 번에 같이 쓰는가
- 벡터 연산이 중요한가

즉, 어떤 배치가 더 나은지는 데이터 자체보다 사용 패턴에 달려 있다.

### 2.4 핵심은 locality와 불필요한 데이터 이동이다

AoS와 SoA 차이는 결국 다음으로 귀결된다.

- 내가 실제로 필요한 데이터가 연속되어 있는가
- 필요 없는 필드까지 같이 끌고 오고 있지는 않은가
- 캐시 라인을 얼마나 효율적으로 쓰는가
- 벡터화와 병렬 처리가 쉬운가

즉, 두 방식의 차이는 캐시 친화성과 메모리 대역폭 활용 차이로 이어진다.

## 3. 내부 동작 원리

### 3.1 AoS는 객체 중심 접근에 자연스럽다

AoS에서는 한 객체의 모든 필드가 가까이 있다.

예를 들어:

```c
Point p = points[i];
```

처럼 하나의 점을 읽고, 바로 `x`, `y`, `z`를 모두 사용하는 경우를 생각해보자.

```c
sum += points[i].x + points[i].y + points[i].z;
```

이런 경우에는 AoS가 자연스럽다.  
왜냐하면 한 객체를 구성하는 데이터가 한 덩어리처럼 모여 있기 때문이다.

즉, 객체 단위 처리에서는 AoS가 직관적이고 캐시 효율도 나쁘지 않을 수 있다.

### 3.2 SoA는 필드 중심 접근에 강하다

반대로 다음처럼 `x` 값만 반복 처리한다고 하자.

```c
for (size_t i = 0; i < N; i++) {
    sum += points[i].x;
}
```

AoS에서는 각 `x` 사이에 `y`, `z` 데이터가 끼어 있다.  
즉, 실제로는 `x`만 필요해도 `y`, `z`가 함께 메모리에 섞여 있다.

반면 SoA에서는:

```c
for (size_t i = 0; i < N; i++) {
    sum += points.x[i];
}
```

`x`들만 연속으로 붙어 있다.  
즉, 필요한 데이터만 밀집해 있다.

이게 SoA가 필드 단위 연산에서 강한 이유다.

### 3.3 AoS는 필요 없는 필드를 함께 끌고 올 수 있다

캐시는 보통 작은 블록 단위로 데이터를 가져온다.  
AoS에서 특정 필드만 반복 접근하면, 그 캐시 블록 안에 필요 없는 다른 필드도 같이 포함되기 쉽다.

즉, `x`만 계산하고 싶은데 `y`, `z`도 함께 따라오는 셈이다.

이건 작은 차이처럼 보이지만 대규모 데이터에서는 메모리 대역폭 낭비로 이어질 수 있다.

### 3.4 SoA는 벡터화와 SIMD에 유리한 경우가 많다

같은 타입 데이터가 연속적으로 놓여 있으면,  
컴파일러나 하드웨어가 여러 값을 한 번에 처리하기 쉬워질 수 있다.

예를 들어 `x` 배열만 쭉 더하는 작업은 벡터화가 더 자연스러울 수 있다.

반면 AoS에서는 중간에 다른 필드가 끼어 있어  
필드 하나만 모아서 처리하기가 더 불리할 수 있다.

즉, SoA는 데이터 병렬 처리에 유리한 배치를 제공하는 경우가 많다.

### 3.5 AoS는 API와 코드 가독성에서 유리할 수 있다

성능만이 전부는 아니다.  
AoS는 사람이 읽기에 자연스럽다.

```
points[i].x
points[i].y
points[i].z
```

이런 구조는 “하나의 객체”라는 사고 모델과 잘 맞는다.

반면 SoA는 같은 개체의 논리적 필드가 물리적으로 흩어져 있다.  
즉, 성능은 유리할 수 있지만 코드 모델은 덜 직관적일 수 있다.

### 3.6 결국 중요한 것은 실제 연산 패턴이다

정리하면 다음과 같다.

- 객체 단위로 자주 쓰면 AoS가 자연스럽다
- 특정 필드만 대량 처리하면 SoA가 유리할 수 있다

즉, 둘의 우열은 절대적이지 않고 **접근 패턴 의존적**이다.

## 4. 핵심 구성 요소

### 4.1 AoS

Array of Structures는 구조체 배열 방식이다.  
하나의 인덱스가 하나의 완전한 객체를 나타낸다.

### 4.2 SoA

Structure of Arrays는 필드별 배열 방식이다.  
같은 필드들이 연속적으로 모여 있다.

### 4.3 field-wise access

특정 필드만 반복해서 처리하는 접근 패턴이다.  
SoA가 강한 경우가 많다.

### 4.4 object-wise access

하나의 객체에 속한 여러 필드를 함께 읽고 쓰는 접근 패턴이다.  
AoS가 더 자연스러운 경우가 많다.

### 4.5 cache locality

필요한 데이터가 얼마나 밀집해 있는지가 중요하다.  
AoS와 SoA의 차이는 결국 locality 차이로 이어진다.

### 4.6 vectorization friendliness

같은 타입 데이터가 연속되어 있는 구조는 벡터화에 유리한 경우가 많다.  
SoA가 여기서 강점을 가지기 쉽다.

## 5. 실행 흐름 또는 처리 순서

좌표 데이터에서 `x` 값만 합산하는 예를 보자.

AoS:

```c
typedef struct {
    float x;
    float y;
    float z;
} Point;
```

```c
Point points[N];
```

```c
for (size_t i = 0; i < N; i++) {
    sum += points[i].x;
}
```

### 5.1 AoS에서의 접근

`points[i].x`를 읽기 위해서는  
`Point` 객체 단위 메모리 배치를 따라가게 된다.

즉, 실제 메모리에는 `x`, `y`, `z`가 함께 놓여 있고,  
다음 `x`를 읽으려면 다음 `Point`로 건너뛰어야 한다.

### 5.2 AoS에서의 비용 특성

`x`만 필요해도 `y`, `z`가 같은 객체 안에 끼어 있다.  
즉, 캐시 입장에서는 “필요한 것과 불필요한 것”이 섞여 올라오기 쉽다.

이제 SoA를 보자.

```c
typedef struct {
    float x[N];
    float y[N];
    float z[N];
} Points;
```

```c
for (size_t i = 0; i < N; i++) {
    sum += points.x[i];
}
```

### 5.3 SoA에서의 접근

이번에는 `x` 배열만 연속적으로 읽는다.  
즉, 다음 원소도 바로 옆에 있고, 그 다음 원소도 바로 옆에 있다.

### 5.4 SoA에서의 비용 특성

필요한 데이터만 연속적으로 있으므로  
캐시와 프리페치가 더 잘 작동할 가능성이 높다.  
또한 같은 타입 연속 데이터라 벡터화에도 유리할 수 있다.

이제 반대로 객체 전체를 갱신하는 경우를 보자.

AoS:

```c
for (size_t i = 0; i < N; i++) {
    points[i].x *= scale;
    points[i].y *= scale;
    points[i].z *= scale;
}
```

### 5.5 객체 단위 작업에서는 AoS가 자연스럽다

한 점의 모든 필드를 같이 읽고 같이 쓰므로  
AoS의 객체 단위 묶음이 잘 맞는다.

반면 SoA에서는 `x`, `y`, `z` 배열을 각각 따로 다뤄야 하므로  
코드 구조가 더 분산될 수 있다.

즉, 접근 패턴에 따라 유불리가 바뀐다.

## 6. 성능 / 최적화 관점

### 6.1 SoA는 필드 중심 수치 계산에서 강할 수 있다

특정 필드만 반복적으로 처리하는 고성능 루프에서는  
SoA가 캐시와 SIMD 관점에서 더 유리한 경우가 많다.

예를 들어:

- 물리 시뮬레이션
- 그래픽스
- 벡터 수치 계산
- AI/ML 전처리 일부
- 대규모 센서 데이터 처리

이런 경우 SoA를 자주 고려하게 된다.

### 6.2 AoS는 일반 객체 지향적 사고와 잘 맞는다

일반적인 애플리케이션 로직에서는 “한 객체”를 통째로 다루는 경우가 많다.  
이런 경우 AoS가 읽기 쉽고 유지보수하기 쉽다.

즉, 성능이 절대적으로 중요하지 않다면 AoS가 더 자연스러운 경우도 많다.

### 6.3 성능 차이는 데이터 크기와 반복 패턴에서 커진다

작은 데이터셋에서는 AoS와 SoA 차이가 거의 안 느껴질 수 있다.  
하지만 데이터가 커지고 루프가 반복되면 메모리 접근 패턴 차이가 크게 드러날 수 있다.

즉, 이 주제는 특히 대량 데이터 처리에서 중요하다.

### 6.4 재배치는 성능만이 아니라 코드 복잡도 trade-off다

SoA는 빠를 수 있지만 코드가 덜 직관적일 수 있다.  
AoS는 읽기 쉬울 수 있지만 특정 연산에서 메모리 낭비가 생길 수 있다.

즉, 선택은 성능만이 아니라  
유지보수성, API 자연스러움, 팀 이해도까지 포함한 trade-off다.

## 7. 어디서부터 데이터 의미와 데이터 접근 패턴이 갈리는가

### 7.1 같은 정보를 담아도 메모리 배치에 따라 비용 모델이 달라진다

AoS와 SoA는 같은 데이터를 표현할 수 있다.  
하지만 필드가 메모리에서 어떻게 모이느냐에 따라  
캐시, 벡터화, 대역폭 비용이 달라진다.

즉, 정보 동일성과 비용 동일성은 다른 말이다.

### 7.2 AoS의 자연스러움과 SoA의 효율은 서로 다른 축이다

AoS는 객체 단위 사고와 API 설계에 잘 맞고,  
SoA는 특정 필드 반복 처리와 SIMD에 더 잘 맞는 경우가 많다.

즉, 읽기 쉬움과 처리 효율은 같은 기준으로 결정되지 않는다.

### 7.3 이 선택은 자료구조 취향이 아니라 접근 패턴 선택이다

한 객체를 통째로 보는 코드가 많은지,  
특정 필드 배열을 반복 처리하는 코드가 많은지에 따라 결론이 달라진다.

즉, AoS/SoA 문제는 데이터 "의미"보다 데이터 "사용 방식"에 더 가깝다.

### 7.4 작은 데이터에서는 차이가 흐리고 큰 데이터에서는 구조 차이가 드러난다

작을 때는 단순성 차이만 보일 수 있다.  
하지만 데이터 크기와 반복 수가 커지면  
메모리 배치 차이가 실제 병목으로 드러난다.

즉, 이 선택은 항상 성능 차이를 만들지 않지만  
적절한 조건에서는 매우 크게 작용한다.

### 7.5 흑백 선택이 아니라 계층적 혼합 전략도 가능하다

전체 모델은 AoS로 유지하고,  
핵심 루프만 SoA 버퍼로 바꾸는 식의 절충이 가능하다.

즉, AoS와 SoA는 이념 대결이 아니라  
비용 모델에 맞춘 배치 전략 선택이다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 먼저 접근 패턴을 본다

AoS/SoA 선택에서 가장 먼저 물어야 할 것은 이것이다.

- 한 객체를 통째로 자주 다루는가
- 특정 필드만 대량 반복 처리하는가

즉, 데이터 의미보다 **사용 방식**이 먼저다.

### 8.2 성능 병목이 메모리 접근인지 확인한다

AoS와 SoA는 메모리 접근 패턴 최적화다.  
따라서 실제 병목이 메모리인지, 계산인지, I/O인지 먼저 확인해야 한다.

즉, 이 주제도 측정 없이 감으로 결정하면 안 된다.

### 8.3 작은 구조는 단순성을 우선할 수 있다

데이터 규모가 작고 성능 병목이 아닌 경우에는  
AoS의 단순성과 가독성이 더 중요할 수 있다.

즉, 모든 곳에 SoA를 적용하는 것은 과잉 최적화일 수 있다.

### 8.4 대량 수치 처리에서는 SoA를 적극 검토한다

대규모 루프, 특정 필드 반복 계산, SIMD 활용이 중요하다면  
SoA가 구조적으로 더 적합할 수 있다.

즉, 성능 민감한 코어 루프에서는 데이터 배치부터 다시 볼 가치가 있다.

### 8.5 혼합 전략도 가능하다

전체 시스템은 AoS로 설계하되,  
성능 핵심 구간에만 SoA 변환 버퍼를 쓰는 식의 혼합 전략도 가능하다.

즉, 설계는 흑백이 아니라 계층적으로 할 수 있다.

## 9. 판단 체크리스트

- 같은 정보를 담아도 배치에 따라 비용 모델이 달라질 수 있음을 의식하고 있는가
- AoS의 자연스러움과 SoA의 효율을 서로 다른 축으로 보고 있는가
- 데이터 의미보다 실제 접근 패턴을 먼저 보고 선택하고 있는가
- 입력 크기와 반복 패턴에 따라 차이가 커질 수 있음을 고려하고 있는가
- 흑백 선택만이 아니라 혼합 전략 가능성까지 열어 두고 있는가

## 10. 더 깊게 볼 포인트

### 10.1 struct padding과 alignment

AoS에서 구조체 내부 패딩이 실제 메모리 사용량과 캐시 효율에 어떤 영향을 주는지 더 깊게 볼 수 있다.

### 10.2 SIMD / vectorization

### 10.3 ECS(Entity Component System)

게임/시뮬레이션 쪽에서 SoA 계열 배치를 왜 선호하는지 더 깊게 이어질 수 있다.

AoS와 SoA를 섞는 AoSoA 같은 중간 전략으로 확장할 수 있다.

### 10.5 profiling 기반 데이터 배치 최적화

실제 성능 측정을 통해 데이터 배치를 바꾸는 실전 방법으로 이어질 수 있다.

## 정리

AoS와 SoA는 같은 정보를 다른 메모리 배치 방식으로 표현하는 전략이다.

핵심은 다음과 같다.

- AoS는 객체 단위로 묶는 배치다.
- SoA는 필드 단위로 묶는 배치다.
- AoS는 객체 전체를 함께 다루는 코드에 자연스럽다.
- SoA는 특정 필드만 반복 처리하는 코드에서 유리할 수 있다.
- SoA는 캐시 친화성과 벡터화 측면에서 강점을 가지는 경우가 많다.
- AoS는 가독성과 API 자연스러움에서 강점을 가지는 경우가 많다.
- 어떤 방식이 더 좋은지는 데이터 의미보다 접근 패턴에 달려 있다.

즉, AoS와 SoA를 이해한다는 것은  
단순히 “구조체 배열 vs 배열 구조체”를 외우는 것이 아니라  
**데이터 배치가 캐시, 메모리 대역폭, 벡터화, 코드 구조에 어떤 영향을 주는지 이해하고, 실제 접근 패턴에 맞는 배치를 선택할 수 있게 되는 것**을 뜻한다.

이 관점이 잡혀야 고성능 루프, 수치 계산, 그래픽스, 시뮬레이션, 시스템 수준 데이터 설계를 더 깊게 다룰 수 있다.
