## 1. 왜 이 주제를 깊게 봐야 하는가

C를 배우는 많은 이유 중 하나는 성능 때문이다.  
실제로 C는 운영체제, 런타임, 데이터베이스, 네트워크 소프트웨어, 임베디드 시스템, 고성능 라이브러리 같은 영역에서 오래도록 중요한 위치를 차지해 왔다.

하지만 초보자는 성능을 종종 너무 단순하게 이해한다.

- C는 빠르다
- 포인터를 쓰면 빠르다
- 어셈블리와 가까우니 무조건 빠르다
- 반복문만 줄이면 빨라진다
- 동적 메모리를 안 쓰면 빨라진다

이런 생각은 일부만 맞고, 대부분은 너무 단순하다.  
실제로 C 코드의 성능은 한 가지 요소로 결정되지 않는다.  
다음이 함께 영향을 준다.

- 알고리즘 복잡도
- 데이터 구조 선택
- 메모리 접근 패턴
- 캐시 친화성
- 분기 예측
- 함수 호출 비용
- 동적 할당 빈도
- 컴파일러 최적화 가능성
- 데이터 배치와 정렬
- 시스템 콜 빈도

즉, C 성능은 단순히 “언어가 저수준이다”로 설명되지 않는다.  
오히려 C의 진짜 장점은 **이런 성능 요인을 직접 볼 수 있게 해준다는 점**에 있다.

예를 들어 다음 두 코드를 생각해보자.

```c
for (int i = 0; i < n; i++) {
    sum += arr[i];
}
```

그리고

```c
for (int i = 0; i < n; i++) {
    sum += linked_list_get(head, i);
}
```

둘 다 숫자를 더하는 코드처럼 보일 수 있다.  
하지만 실제 성능은 매우 다를 수 있다.

- 첫 번째는 연속 메모리 접근일 가능성이 높다
- 두 번째는 포인터를 따라다니는 비연속 접근일 수 있다
- CPU 캐시 친화성이 다르다
- 함수 호출과 분기 비용도 다르다

즉, C 성능을 제대로 이해하려면 “문법”이 아니라  
**실행 비용 모델**을 봐야 한다.

이 문서를 깊게 봐야 하는 이유는 다음과 같다.

- C가 빠르다는 말이 정확히 무슨 뜻인지 분해해 준다
- 메모리와 캐시를 성능 관점에서 다시 보게 해 준다
- 알고리즘과 구현 디테일을 함께 보게 해 준다
- 성능 최적화와 안전성/유지보수성의 trade-off를 판단하게 해 준다
- 이후 캐시 친화적 접근, 데이터 배치, 시스템 수준 최적화로 자연스럽게 이어진다

이 문서는 C 코드 성능이 어떤 요인들로 결정되는지,  
왜 단순히 “C라서 빠른 것”이 아닌지,  
그리고 실무에서 어떤 기준으로 병목을 바라봐야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

C 코드 성능을 큰 구조로 보면 다음과 같다.

### 2.1 가장 큰 것은 알고리즘과 자료구조다

아무리 C로 작성해도 알고리즘이 비효율적이면 느리다.  
즉, 언어보다 먼저 보는 것은 보통 다음이다.

- 시간 복잡도
- 공간 복잡도
- 자료구조 선택
- 탐색/정렬/업데이트 패턴

즉, C 성능의 첫 번째 층은 언제나 알고리즘이다.

### 2.2 그 다음은 메모리 접근 패턴이다

현대 시스템에서 성능은 CPU 연산 자체보다  
메모리 접근 패턴에 더 크게 좌우되는 경우가 많다.

즉, 중요한 것은 다음이다.

- 데이터가 연속적인가
- 캐시에 잘 맞는가
- 포인터를 많이 따라다니는가
- 예측 가능한 순차 접근인가

즉, C 성능의 두 번째 큰 층은 메모리다.

### 2.3 그 다음은 호출, 분기, 시스템 경계 같은 실행 구조다

성능은 단지 데이터만의 문제가 아니다.  
다음도 영향을 준다.

- 함수 호출 빈도
- 간접 호출 여부
- 분기 예측 실패
- 동적 메모리 할당 빈도
- 시스템 콜 빈도

즉, 실행 구조의 미세 비용도 누적되면 커질 수 있다.

### 2.4 마지막으로 컴파일러 최적화가 영향을 준다

C는 컴파일러가 매우 적극적으로 최적화할 수 있는 언어다.  
하지만 컴파일러가 잘 최적화할 수 있는 코드와 그렇지 못한 코드가 있다.

즉, 성능은 코드를 “어떻게 썼는가”와  
컴파일러가 “어떻게 해석할 수 있는가”의 상호작용이기도 하다.

## 3. 내부 동작 원리

### 3.1 알고리즘이 가장 큰 차이를 만든다

예를 들어 `O(n)`과 `O(n^2)` 차이는  
미세한 최적화 몇 개보다 훨씬 큰 영향을 줄 수 있다.

즉, 다음 두 접근을 비교할 때:

- 배열 한 번 순회
- 배열 안에서 또 배열 전체를 반복 탐색

후자는 C로 써도 느릴 수 있다.

즉, 성능을 볼 때는 먼저 “이 구현 방식 자체가 큰 비용 구조를 가지고 있는가”를 봐야 한다.

### 3.2 데이터 구조 선택이 메모리 접근 패턴을 만든다

배열, 연결 리스트, 해시 테이블, 트리 같은 자료구조는  
단순 API 차이가 아니라 실제 메모리 접근 패턴 차이를 만든다.

예를 들어:

- 배열은 연속 메모리
- 연결 리스트는 포인터 기반 비연속 메모리
- 트리는 분기와 포인터 추적이 많음

즉, 자료구조 선택은 알고리즘이면서 동시에 하드웨어 친화성 문제다.

### 3.3 연속 메모리 접근은 보통 유리하다

다음 코드를 보자.

```c
for (int i = 0; i < n; i++) {
    sum += arr[i];
}
```

이건 연속 메모리 접근이다.  
CPU 입장에서는 예측이 쉽고, 캐시와 프리페치가 잘 작동할 가능성이 높다.

반면 다음은 다르다.

```c
node = head;
while (node != NULL) {
    sum += node->value;
    node = node->next;
}
```

이건 각 노드가 메모리 여기저기에 흩어져 있을 수 있다.  
즉, 포인터를 따라가며 비연속 접근이 발생할 수 있다.

따라서 단순 문법보다 더 중요한 것은 **데이터 배치와 접근 패턴**이다.

### 3.4 캐시 미스는 계산보다 훨씬 비쌀 수 있다

현대 CPU는 산술 연산 자체는 매우 빠르다.  
문제는 필요한 데이터가 CPU 가까이에 없을 때다.

즉, 캐시에 없는 데이터를 메인 메모리에서 가져와야 하면  
연산 몇 개보다 훨씬 큰 비용이 들 수 있다.

이 때문에 C 성능을 보려면 다음 질문이 중요하다.

- 이 데이터는 캐시에 잘 실릴까
- 접근 순서가 예측 가능한가
- 같은 데이터를 재사용하는가

즉, 성능의 큰 부분은 “몇 번 계산하느냐”보다  
“얼마나 비싼 메모리 접근을 하느냐”에 달려 있다.

### 3.5 함수 호출과 간접 호출도 비용을 만들 수 있다

일반 함수 호출은 공짜가 아니다.  
호출 준비, 분기, 반환, 인자 전달 같은 비용이 있다.

보통은 이 비용이 매우 크진 않지만,  
짧은 함수가 매우 자주 호출되면 누적될 수 있다.

특히 다음은 더 주의할 수 있다.

- 아주 짧은 함수의 과도한 호출
- 함수 포인터를 통한 간접 호출
- 분기 예측이 어려운 호출 패턴

즉, 호출 구조도 성능 일부다.

### 3.6 동적 메모리 할당은 편리하지만 비용이 있다

`malloc`, `free`는 유연한 메모리 사용을 가능하게 하지만  
힙 관리 비용도 함께 가져온다.

즉, 다음은 성능에 영향을 줄 수 있다.

- 매우 잦은 `malloc/free`
- 작은 객체의 반복 할당/해제
- 힙 단편화
- allocator lock/metadata 비용

이 때문에 실무에서는 종종 다음을 고민한다.

- 스택으로 둘 수 없는가
- 풀 할당이 더 적절한가
- 재사용 버퍼가 가능한가

### 3.7 시스템 콜은 사용자 공간 함수보다 더 비쌀 수 있다

파일 읽기, 쓰기, 네트워크 I/O 등은 시스템 콜과 연결될 수 있다.  
이건 커널 경계 비용이 있기 때문에 일반 함수 호출보다 무겁다.

즉, 다음은 성능 병목이 되기 쉽다.

- 작은 write를 아주 많이 호출
- 매우 자주 flush
- 작은 read를 반복

이 때문에 버퍼링과 배치가 중요해진다.

### 3.8 컴파일러 최적화는 코드 구조에 의존한다

컴파일러는 반복문, 상수 전파, 인라인, 벡터화 같은 여러 최적화를 할 수 있다.  
하지만 코드가 너무 복잡하거나 alias 가능성이 높거나 side effect가 많으면  
최적화 여지가 줄어들 수 있다.

즉, 성능은 “좋은 컴파일러”만의 문제가 아니라  
**컴파일러가 이해하기 쉬운 코드 구조**와도 연결된다.

## 4. 핵심 구성 요소

### 4.1 algorithmic complexity

복잡도는 성능의 가장 큰 층이다.  
O(1), O(log n), O(n), O(n^2) 같은 차이는 미세 최적화보다 훨씬 크다.

### 4.2 data layout

데이터가 메모리에 어떻게 배치되는지가 성능에 큰 영향을 준다.  
연속성, 정렬, 구조체 크기, 배열 배치 같은 요소가 여기에 들어간다.

### 4.3 cache behavior

캐시 적중률과 캐시 미스는 현대 성능 모델의 핵심이다.  
같은 계산이라도 캐시에 잘 맞는 코드가 훨씬 빠를 수 있다.

### 4.4 branch behavior

조건문, switch, 간접 점프 같은 분기 패턴은 분기 예측과 연결된다.  
예측이 어려운 코드일수록 비용이 커질 수 있다.

### 4.5 allocation behavior

메모리 할당/해제 패턴도 성능 요소다.  
특히 빈번한 힙 할당은 성능과 안정성 모두에 영향을 줄 수 있다.

### 4.6 syscall frequency

시스템 콜은 상대적으로 무거운 경계다.  
입출력 중심 코드에서는 호출 횟수가 성능의 큰 부분이 된다.

### 4.7 compiler optimization friendliness

코드가 컴파일러 최적화와 잘 맞는지,  
즉 단순하고 예측 가능한 구조인지도 성능에 영향을 준다.

## 5. 실행 흐름 또는 처리 순서

다음과 같은 데이터를 처리한다고 생각해보자.

```c
int sum_array(const int *arr, size_t n) {
    int sum = 0;
    for (size_t i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
}
```

이 코드의 성능을 결정하는 흐름을 개념적으로 보면 다음과 같다.

### 5.1 알고리즘 선택

먼저 배열 전체를 한 번 순회하는 O(n) 알고리즘이다.  
즉, 큰 구조는 비교적 단순하고 예측 가능하다.

### 5.2 데이터 구조 특성

입력은 배열이다.  
즉, 데이터는 연속 메모리에 있을 가능성이 높다.

### 5.3 메모리 접근 패턴

`arr[i]` 접근은 순차적이다.  
따라서 캐시와 프리페치에 유리할 수 있다.

### 5.4 연산 비용

각 반복은 덧셈 하나와 인덱스 증가 정도다.  
즉, 순수 계산 자체는 매우 단순하다.

### 5.5 컴파일러 최적화 가능성

루프 구조가 단순하므로 인라인, 벡터화, 루프 최적화 가능성이 있을 수 있다.

즉, 이 코드는 여러 층에서 비교적 성능 친화적인 구조를 가진다.

반대로 다음 같은 구조를 보자.

```c
int sum_list(Node *head) {
    int sum = 0;
    Node *cur = head;
    while (cur != NULL) {
        sum += cur->value;
        cur = cur->next;
    }
    return sum;
}
```

이 코드는 논리적으로는 여전히 O(n)일 수 있다.  
하지만 데이터 구조와 접근 패턴은 완전히 다르다.

- 포인터 추적
- 비연속 접근 가능성
- 캐시 미스 증가 가능성
- 예측 어려운 메모리 패턴

즉, 성능은 복잡도만으로 다 설명되지 않는다.

## 6. 성능 / 최적화 관점

### 6.1 큰 병목부터 봐야 한다

실무에서 가장 흔한 실수는 작은 최적화에 먼저 매달리는 것이다.  
하지만 보통 순서는 이렇다.

1. 알고리즘 병목
2. 데이터 구조 병목
3. 메모리 병목
4. I/O 병목
5. 미세 최적화

즉, 성능은 아래에서부터 조금씩 깎는 것이 아니라  
큰 병목부터 제거해야 한다.

### 6.2 연속 메모리와 단순 루프는 강력하다

C에서는 복잡한 마법보다  
연속 메모리 + 단순 루프 + 명확한 데이터 흐름이 매우 강력한 경우가 많다.

즉, 성능 최적화의 시작은 종종  
“더 저수준 문법”이 아니라 “더 단순한 데이터 흐름”이다.

### 6.3 안전하지 않은 코드는 최적화 친화적이지 않을 수도 있다

UB에 기대거나, alias가 과도하거나, lifetime이 불명확한 코드는  
겉보기엔 빠를 것 같아도 장기적으로는 더 위험하고 비효율적일 수 있다.

즉, 안전성과 성능은 완전히 반대만은 아니다.  
명확한 객체 모델과 경계 모델이 오히려 최적화와 디버깅 모두에 유리한 경우가 많다.

### 6.4 측정 없이 감으로 최적화하면 위험하다

어떤 코드가 느린지, 실제 병목이 무엇인지 측정하지 않으면  
헛된 최적화에 시간을 쓰기 쉽다.

즉, 실무 성능 최적화의 핵심은 다음이다.

- 추측하지 말고 측정한다
- 병목을 찾고
- 가장 큰 비용 구조부터 바꾼다

---

## 7. 어디서부터 코드 표면과 실제 비용 모델이 갈리는가

### 7.1 "연산이 적다"와 "실제로 빠르다"는 같은 뜻이 아니다

프로그래머는 소스에서 눈에 보이는 계산식 개수에 먼저 주목하기 쉽다.  
하지만 실제 비용은 메모리 대기, 캐시 미스, 할당, 분기, 시스템 경계가 함께 만든다.

즉, 코드 표면의 단순함과 실제 실행 비용은 분리해서 봐야 한다.

### 7.2 알고리즘, 자료구조, 데이터 배치가 서로 다른 층의 병목을 만든다

같은 함수 안에서도 병목 원인은 여러 층으로 갈린다.

- 알고리즘: 처리량 자체를 결정
- 자료구조: 접근 방식과 연산 패턴을 결정
- 데이터 배치: 캐시/대역폭 비용을 결정

즉, 성능 문제를 한 줄 최적화로만 보면  
어느 층에서 병목이 생기는지 놓치게 된다.

### 7.3 안전하지 않은 코드가 항상 빠른 것은 아니다

UB에 기대거나 aliasing/lifetime이 흐린 코드는  
겉보기에는 과감한 최적화처럼 보여도  
장기적으로는 최적화 안정성과 디버깅 가능성을 같이 잃을 수 있다.

즉, 성능과 명확성은 항상 반대만은 아니다.

### 7.4 성능은 언어 특성보다 입력 크기와 패턴에 더 민감할 수 있다

작은 데이터에서는 아무 차이 없어 보이던 코드가  
데이터 크기와 반복 수가 커지면서 전혀 다른 병목을 드러낼 수 있다.

즉, 성능 판단은 코드 조각 하나가 아니라  
실행 조건과 입력 모델까지 포함해야 한다.

### 7.5 측정은 마지막 단계가 아니라 병목 분류 단계다

많은 사람이 최적화한 뒤 측정하려고 한다.  
하지만 실제로는 측정이 먼저여야  
문제가 계산인지, 메모리인지, I/O인지 분류할 수 있다.

즉, profiling은 증명 도구이기 전에  
병목 종류를 가르는 분류 도구다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 먼저 “무엇이 비싼가”를 구분한다

코드를 볼 때 먼저 질문해야 한다.

- 계산이 비싼가
- 메모리 접근이 비싼가
- 할당이 비싼가
- I/O가 비싼가
- 시스템 콜이 많은가

즉, 병목 종류를 먼저 구분해야 올바른 최적화가 가능하다.

### 8.2 자료구조 선택이 문법 미세 튜닝보다 중요하다

배열을 쓸지, 연결 리스트를 쓸지, 해시 구조를 쓸지 같은 결정은  
루프 한 줄 최적화보다 훨씬 큰 차이를 만들 수 있다.

즉, 성능 설계는 문법보다 자료구조 수준에서 먼저 이뤄져야 한다.

### 8.3 데이터 길이와 접근 순서를 함께 본다

실무에서는 단순히 데이터 양만이 아니라  
“어떤 순서로 어떻게 접근하는가”가 중요하다.

즉, 같은 양의 데이터라도 순차 접근과 무작위 접근의 비용은 크게 다를 수 있다.

### 8.4 동적 할당과 시스템 콜은 묶어서 줄일 수 있는지 본다

작은 `malloc/free`, 작은 `read/write`를 반복하는 구조는 병목이 되기 쉽다.  
배치 처리, 버퍼링, 재사용 전략이 가능한지 먼저 보는 것이 좋다.

### 8.5 컴파일러가 잘 최적화할 수 있는 단순 구조를 선호한다

복잡한 트릭보다, 명확한 데이터 흐름과 단순 반복 구조가  
실제로는 더 빠르고 더 안전한 경우가 많다.

즉, 성능 좋은 C 코드는 종종 “영리한 코드”보다 “단순한 코드”에 가깝다.

---

## 9. 판단 체크리스트

- 연산 수만이 아니라 메모리, 할당, 시스템 경계 비용을 함께 보고 있는가
- 알고리즘, 자료구조, 데이터 배치를 서로 다른 병목 층으로 구분하고 있는가
- 안전하지 않은 트릭이 항상 빠르다고 가정하고 있지 않은가
- 입력 크기와 반복 패턴에 따라 병목이 달라질 수 있음을 의식하고 있는가
- profiling을 결과 확인이 아니라 병목 분류 단계로 사용하고 있는가

---

## 10. 더 깊게 볼 포인트

### 10.1 캐시 친화적 접근

성능에서 메모리 접근 패턴이 왜 그렇게 중요한지 더 구체적으로 볼 수 있다.

### 10.2 데이터 지향 설계

구조체 배열과 배열 구조체 차이, 데이터 배치 최적화 같은 주제로 확장할 수 있다.

### 10.3 branch prediction

분기 예측 실패가 실제로 어떤 비용을 만드는지 더 깊게 볼 수 있다.

### 10.4 allocator 전략

일반 `malloc/free` 대신 pool, arena, slab 같은 전략으로 확장할 수 있다.

### 10.5 profiling

성능 측정 도구를 사용해 실제 병목을 찾는 실전 방법으로 이어질 수 있다.

---

## 정리

C 코드의 성능은 단순히 “C라서 빠르다”로 결정되지 않는다.

핵심은 다음과 같다.

- 가장 큰 것은 알고리즘과 자료구조다.
- 그 다음은 메모리 접근 패턴과 캐시 친화성이다.
- 함수 호출, 분기, 동적 할당, 시스템 콜 빈도도 누적되면 중요하다.
- 컴파일러 최적화는 코드 구조에 따라 효과가 달라진다.
- 연속 메모리와 단순 루프는 매우 강력한 성능 특성을 가질 수 있다.
- 안전성과 명확성은 종종 성능 최적화와도 잘 맞는다.
- 감으로 최적화하지 말고 병목을 측정해야 한다.

즉, C 성능을 이해한다는 것은  
단순히 “저수준 언어라 빠르다”를 믿는 것이 아니라  
**알고리즘, 데이터 배치, 메모리 접근, 실행 구조, 컴파일러 최적화가 함께 만들어내는 비용 모델을 이해하는 것**을 뜻한다.

이 관점이 잡혀야 이후 캐시 친화적 코드, 데이터 배치 최적화, 시스템 수준 성능 분석으로 자연스럽게 확장할 수 있다.
