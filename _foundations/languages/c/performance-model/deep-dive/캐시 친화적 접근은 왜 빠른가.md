## 1. 왜 이 주제를 깊게 봐야 하는가

C 성능을 이야기할 때 많은 사람이 처음에는 CPU 연산만 떠올린다.

- 덧셈이 몇 번 일어나는가
- 함수 호출이 몇 번 있는가
- 조건문이 얼마나 많은가

물론 이런 요소도 중요하다.  
하지만 현대 시스템에서 실제 성능 병목은 종종 연산 그 자체보다 **메모리 접근**에서 더 크게 생긴다.

즉, 다음 같은 코드가 있다고 해보자.

```c
for (size_t i = 0; i < n; i++) {
    sum += arr[i];
}
```

겉보기에는 단순한 루프다.  
하지만 이 코드가 빠를 수 있는 핵심 이유는 덧셈이 단순해서만이 아니다.  
오히려 더 중요한 것은 다음이다.

- `arr`가 연속 메모리에 놓여 있다
- 접근 순서가 예측 가능하다
- CPU가 필요한 데이터를 미리 가져오기 쉽다
- 캐시에 이미 올라온 데이터를 재사용하기 쉽다

반대로 논리적으로 비슷해 보이는 다음 코드를 생각해보자.

```c
Node *cur = head;
while (cur != NULL) {
    sum += cur->value;
    cur = cur->next;
}
```

이 코드도 결국 값을 하나씩 더한다.  
복잡도도 `O(n)`일 수 있다.  
하지만 실제 성능은 배열 순회보다 훨씬 느릴 수 있다.

왜냐하면:

- 각 노드가 메모리 여기저기에 흩어져 있을 수 있고
- CPU가 다음 데이터를 미리 예측하기 어렵고
- 캐시에 없는 데이터를 자꾸 가져와야 하고
- 포인터 추적 때문에 파이프라인과 캐시 효율이 나빠질 수 있기 때문이다

즉, 같은 `O(n)`이라도 **메모리 locality**, 특히 **cache locality**가 다르면 실제 시간은 크게 달라질 수 있다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- “왜 배열이 빠른가”를 단순 문법이 아니라 하드웨어 관점에서 이해하게 해 준다
- 자료구조 선택이 성능에 미치는 영향을 더 정확히 보게 해 준다
- C에서 데이터 배치가 왜 중요한지 감각을 준다
- 이후 struct layout, AoS/SoA, 루프 최적화, 시스템 성능 분석으로 이어지는 기반이 된다
- “연산 수는 같아도 성능은 다를 수 있다”는 점을 명확히 보여 준다

이 문서는 캐시가 왜 필요한지, 캐시 친화적 접근이 왜 빠른지,  
공간 지역성과 시간 지역성이 무엇인지,  
배열과 연결 리스트가 왜 다른 성능 특성을 가지는지,  
그리고 실무에서 어떻게 캐시 친화적 코드를 설계해야 하는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

캐시 친화적 접근을 큰 구조로 보면 다음과 같다.

### 2.1 CPU는 메모리보다 훨씬 빠르다

현대 CPU는 산술 연산 자체는 매우 빠르게 처리할 수 있다.  
문제는 필요한 데이터가 CPU 가까이에 항상 있는 것이 아니라는 점이다.

즉, CPU가 계산할 준비가 되어 있어도  
데이터가 멀리 있는 메모리 계층에 있으면 기다려야 할 수 있다.

이 차이를 줄이기 위해 캐시가 존재한다.

### 2.2 캐시는 자주 쓰거나 곧 쓸 데이터를 가까이 두는 계층이다

캐시는 메인 메모리보다 작지만 훨씬 빠른 저장 공간이다.  
CPU는 자주 접근할 데이터나 인접한 데이터를 캐시에 올려 두고 사용하려 한다.

즉, 캐시는 “데이터를 계산 장치 가까이에 임시로 두는 고속 계층”이다.

### 2.3 캐시 친화적 접근은 캐시가 잘 작동하도록 만드는 접근 방식이다

캐시 친화적 접근은 다음 성질을 가진다.

- 접근 순서가 예측 가능하다
- 인접한 데이터를 순서대로 읽는다
- 같은 데이터를 짧은 시간 안에 다시 쓴다
- 불필요하게 흩어진 메모리를 따라다니지 않는다

즉, 캐시가 미리 가져오고 오래 유지하기 좋은 접근 패턴이다.

### 2.4 핵심은 locality다

캐시 친화성의 중심에는 locality가 있다.

- **spatial locality**: 가까운 주소를 함께 접근하는 성질
- **temporal locality**: 최근 사용한 데이터를 곧 다시 쓰는 성질

즉, locality가 좋을수록 캐시가 효율적으로 동작하기 쉽다.

## 3. 내부 동작 원리

### 3.1 캐시는 보통 한 바이트씩 가져오지 않는다

초보자는 종종 메모리를 “필요한 값 하나씩 가져오는 것”처럼 상상한다.  
하지만 캐시는 보통 더 큰 덩어리 단위로 데이터를 가져온다.

즉, `arr[i]` 하나를 읽더라도 실제로는 그 주변 인접 데이터까지 함께 캐시에 들어올 수 있다.

이게 중요한 이유는 배열처럼 연속된 데이터를 순차 접근할 때  
다음 원소들이 이미 캐시에 들어와 있을 가능성이 높아지기 때문이다.

### 3.2 배열 순회는 spatial locality가 좋다

다음 코드를 보자.

```c
for (size_t i = 0; i < n; i++) {
    sum += arr[i];
}
```

배열은 연속 메모리에 놓이기 때문에 `arr[i]`, `arr[i+1]`, `arr[i+2]`는 서로 가까운 주소에 있다.

즉, 하나를 읽으면서 인접 데이터도 함께 캐시에 들어오기 쉽다.  
그리고 다음 반복에서 바로 그 데이터를 쓰게 된다.

이게 바로 spatial locality다.

### 3.3 같은 데이터를 곧 다시 쓰면 temporal locality가 좋다

예를 들어 어떤 작은 배열이나 구조체 필드를 반복해서 여러 번 쓰는 경우를 생각해보자.

```c
for (int iter = 0; iter < 1000; iter++) {
    sum += arr[0];
}
```

이런 경우 `arr[0]`은 최근에 사용된 데이터이므로  
캐시에 남아 있을 가능성이 높다.

즉, 같은 데이터를 짧은 시간 안에 다시 쓰는 것이 temporal locality다.

### 3.4 포인터 추적은 locality를 깨기 쉽다

다음 코드를 보자.

```c
Node *cur = head;
while (cur != NULL) {
    sum += cur->value;
    cur = cur->next;
}
```

연결 리스트 노드가 메모리에 연속적으로 놓여 있다는 보장은 없다.  
따라서 `cur->next`를 따라갈 때마다 CPU는 전혀 다른 위치의 데이터를 새로 가져와야 할 수 있다.

즉, 포인터 추적은 spatial locality를 나쁘게 만들기 쉽다.

### 3.5 캐시 미스는 비싼 대기 시간을 만든다

필요한 데이터가 캐시에 없으면 CPU는 더 아래 메모리 계층으로 내려가 데이터를 가져와야 한다.  
이때 연산 몇 번보다 훨씬 큰 비용이 들 수 있다.

즉, 프로그램이 느린 이유가 “계산이 많아서”가 아니라  
“기다리는 시간이 많아서”일 수 있다.

### 3.6 프리페치와 예측 가능한 접근 패턴

배열처럼 순차 접근이 명확한 경우 CPU나 하드웨어는 다음 데이터를 미리 가져오려 시도할 수 있다.  
이건 캐시 친화성을 더 높인다.

반면 포인터를 따라가는 구조는 다음 위치를 예측하기 어렵다.  
즉, 예측 가능한 순차성도 캐시 친화성의 중요한 일부다.

## 4. 핵심 구성 요소

### 4.1 cache

캐시는 CPU 가까이에 있는 고속 메모리 계층이다.  
메인 메모리보다 작지만 훨씬 빠르다.

### 4.2 cache line

캐시는 보통 작은 고정 크기 블록 단위로 데이터를 다룬다.  
즉, 하나의 값만이 아니라 주변 데이터도 함께 올라올 수 있다.

이 때문에 인접 데이터 접근이 유리하다.

### 4.3 spatial locality

공간 지역성은 가까운 주소를 함께 접근하는 성질이다.  
배열 순회가 대표적이다.

### 4.4 temporal locality

시간 지역성은 최근 사용한 데이터를 곧 다시 사용하는 성질이다.  
작은 작업 집합을 반복해서 도는 코드에서 좋게 나타날 수 있다.

### 4.5 cache miss

필요한 데이터가 캐시에 없어서 더 아래 메모리 계층으로 가야 하는 상태다.  
성능에 큰 비용을 만든다.

### 4.6 prefetch-friendly pattern

미리 다음 데이터를 가져오기 쉬운 접근 패턴이다.  
주로 순차 배열 접근이 여기에 가깝다.

## 5. 실행 흐름 또는 처리 순서

다음 배열 순회를 기준으로 보자.

```c
for (size_t i = 0; i < n; i++) {
    sum += arr[i];
}
```

### 5.1 첫 원소 접근

먼저 `arr[0]`에 접근한다.  
이때 `arr[0]`이 포함된 주변 데이터 블록이 캐시에 올라올 수 있다.

### 5.2 인접 원소 접근

이제 `arr[1]`, `arr[2]` 등으로 진행한다.  
이 값들은 이전에 함께 들어온 캐시 라인 안에 있을 가능성이 높다.

즉, 다음 접근 비용이 작아질 수 있다.

### 5.3 순차 접근 반복

이 패턴이 계속 반복되면 CPU 입장에서는 매우 예측 가능하다.

- 다음 접근 위치를 예상할 수 있고
- 인접 데이터가 함께 올라오고
- 캐시 적중률이 높아질 수 있다

### 5.4 누적 효과

루프가 길어질수록 “한 번 가져온 덩어리에서 여러 값을 소비하는 효과”가 누적된다.  
즉, 메모리 접근 비용이 상대적으로 줄어든다.

이제 연결 리스트를 보자.

```c
Node *cur = head;
while (cur != NULL) {
    sum += cur->value;
    cur = cur->next;
}
```

### 5.5 현재 노드 접근

현재 노드 데이터를 읽는다.  
하지만 다음 노드가 메모리 어디에 있는지는 멀리 떨어져 있을 수 있다.

### 5.6 다음 포인터 추적

`cur->next`를 따라간다.  
다음 노드가 다른 캐시 라인, 심지어 훨씬 먼 메모리 위치에 있을 수 있다.

### 5.7 반복 시 캐시 효율 저하 가능성

각 단계가 서로 멀리 떨어진 데이터를 요구하면  
캐시 적중률이 낮아지고, CPU가 계속 기다리는 상황이 생길 수 있다.

즉, 같은 `O(n)`이라도 실행 시간은 크게 달라질 수 있다.

## 6. 성능 / 최적화 관점

### 6.1 연속 메모리 자료구조가 종종 더 빠르다

배열, 연속 버퍼, 구조체 배열은 캐시 친화적일 가능성이 높다.  
반면 포인터 기반 분산 구조는 접근 유연성은 있지만 캐시 효율이 나쁠 수 있다.

즉, 자료구조 선택은 알고리즘뿐 아니라 하드웨어 비용 모델도 반영해야 한다.

### 6.2 작은 연산 최적화보다 데이터 배치가 더 중요할 수 있다

루프 안에서 덧셈 하나 줄이는 것보다  
데이터 배치를 바꿔 캐시 미스를 줄이는 쪽이 더 큰 차이를 만들 수 있다.

즉, 성능 최적화는 종종 “수식을 줄이는 일”보다 “데이터를 잘 놓는 일”이다.

### 6.3 구조체 설계도 캐시 친화성과 연결된다

구조체 안 필드 순서, 크기, padding, 자주 같이 쓰는 필드 배치도  
캐시 효율에 영향을 줄 수 있다.

즉, 단순 타입 선언도 성능 모델의 일부다.

### 6.4 큰 working set은 캐시를 넘어서면 급격히 느려질 수 있다

작은 데이터는 캐시 안에 잘 머물 수 있다.  
하지만 데이터 집합이 커지고 접근 패턴이 나쁘면 캐시 혜택이 급격히 줄어든다.

즉, 입력 크기 증가에 따라 같은 코드도 성능 특성이 크게 달라질 수 있다.

### 6.5 캐시 친화성은 알고리즘 이후의 두 번째 큰 층이다

정리하면 성능에서 보통 순서는 이렇다.

1. 알고리즘
2. 데이터 배치 / 캐시 친화성
3. 분기 / 호출 / 할당 / 시스템 경계
4. 미세 최적화

즉, 캐시 친화성은 부가 디테일이 아니라 매우 큰 층이다.

## 7. 어디서부터 알고리즘 복잡도와 하드웨어 비용 모델이 갈리는가

### 7.1 같은 `O(n)`이라도 메모리 접근 방식은 전혀 다를 수 있다

복잡도 표기만 보면 두 루프가 비슷해 보여도  
하나는 연속 메모리를 순서대로 읽고, 다른 하나는 포인터를 계속 따라갈 수 있다.

즉, 알고리즘 복잡도와 실제 메모리 비용은 같은 층의 설명이 아니다.

### 7.2 캐시는 "운 좋으면 빨라지는 요소"가 아니라 비용 모델의 핵심 일부다

초보자는 캐시를 하드웨어 내부 디테일로만 보기 쉽다.  
하지만 실제로는 자료구조와 접근 순서가  
캐시 적중률을 통해 직접 성능을 바꾼다.

즉, 캐시는 부가 요소가 아니라  
실행 비용을 결정하는 중심 층 중 하나다.

### 7.3 포인터 기반 유연성과 지역성은 같은 방향이 아닐 수 있다

동적 구조와 간접 참조는 유연하지만  
연속 배치를 깨뜨려 locality를 약하게 만들 수 있다.

즉, 추상화와 캐시 효율은 trade-off가 될 수 있다.

### 7.4 데이터 크기가 커질수록 같은 코드도 다른 비용 체제로 들어간다

작을 때는 캐시 안에 머물던 working set이  
커지면서 전혀 다른 성능 특성을 보일 수 있다.

즉, 캐시 친화성 판단은 코드만이 아니라  
입력 규모까지 같이 봐야 한다.

### 7.5 캐시 친화성은 계산 최적화보다 먼저 봐야 할 때가 많다

덧셈 몇 번 줄이는 것보다  
데이터를 더 잘 배치하고 순차적으로 읽는 편이 더 큰 차이를 만들 수 있다.

즉, 이 주제는 미세 튜닝이 아니라  
상위 구조 최적화 주제다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 “몇 번 계산하느냐”보다 “어떻게 읽느냐”를 함께 본다

실무에서는 다음 질문이 중요하다.

- 이 데이터는 연속적인가
- 순차 접근인가
- 포인터를 여러 번 따라가야 하는가
- 같은 데이터를 다시 쓸 가능성이 높은가

즉, 계산량뿐 아니라 메모리 접근량과 패턴을 같이 봐야 한다.

### 8.2 가능한 경우 연속 메모리 구조를 선호한다

무조건 배열이 정답은 아니지만,  
성능이 중요하고 순차 처리 비중이 높다면 연속 메모리 구조가 유리한 경우가 많다.

즉, linked structure의 유연성과 cache-friendly layout의 성능 사이에서 목적에 맞게 선택해야 한다.

### 8.3 자주 함께 쓰는 데이터는 함께 배치하려고 한다

같이 접근되는 필드나 값들은 가능하면 메모리에서도 가깝게 두는 것이 유리할 수 있다.

즉, 데이터 관계를 코드 논리뿐 아니라 배치 관점에서도 볼 필요가 있다.

### 8.4 지나친 포인터 기반 추상화는 비용을 만들 수 있다

포인터 추적이 많아지면 코드가 유연해질 수는 있다.  
하지만 캐시, 분기, 간접 접근 비용이 함께 올라갈 수 있다.

즉, 추상화와 성능의 균형을 봐야 한다.

### 8.5 성능 문제는 실제 측정과 함께 봐야 한다

캐시 친화성이 중요하다고 해서 감으로만 구조를 바꾸면 안 된다.  
프로파일링과 측정을 통해 실제 병목이 메모리 접근인지 확인해야 한다.

즉, 원리는 중요하지만 최종 판단은 측정과 함께 해야 한다.

## 9. 판단 체크리스트

- 같은 복잡도라도 메모리 접근 방식이 다를 수 있음을 의식하고 있는가
- 캐시를 하드웨어 부가 디테일이 아니라 비용 모델의 핵심 일부로 보고 있는가
- 포인터 기반 유연성과 locality 사이의 trade-off를 생각하고 있는가
- 입력 크기에 따라 같은 코드의 캐시 특성이 달라질 수 있음을 고려하고 있는가
- 계산 미세 튜닝보다 데이터 배치와 접근 순서를 먼저 보고 있는가

## 10. 더 깊게 볼 포인트

### 10.1 AoS와 SoA

Array of Structures와 Structure of Arrays 차이는 캐시 친화적 데이터 배치의 대표 사례다.

### 10.2 struct padding과 alignment

구조체 배치가 캐시 사용량과 접근 효율에 어떤 영향을 주는지 더 깊게 볼 수 있다.

### 10.3 blocking / tiling

큰 데이터 처리에서 캐시에 맞게 작업 단위를 쪼개는 최적화로 확장할 수 있다.

### 10.4 false sharing

멀티스레드 환경에서 캐시 라인 공유가 성능을 떨어뜨리는 문제로 이어질 수 있다.

### 10.5 prefetching과 hardware behavior

CPU가 실제로 어떤 방식으로 다음 데이터를 예측하는지 더 깊게 볼 수 있다.

## 정리

캐시 친화적 접근이 빠른 이유는 단순히 문법이 단순해서가 아니다.

핵심은 다음과 같다.

- CPU는 메모리보다 훨씬 빠르다.
- 캐시는 자주 쓰는 데이터와 인접 데이터를 CPU 가까이에 둔다.
- 연속 메모리 순차 접근은 spatial locality가 좋다.
- 최근 사용한 데이터를 다시 쓰면 temporal locality가 좋다.
- 포인터 추적과 비연속 접근은 캐시 효율을 나쁘게 만들 수 있다.
- 캐시 미스는 연산 몇 번보다 훨씬 큰 비용을 만들 수 있다.
- 따라서 자료구조와 데이터 배치는 실제 성능에 매우 큰 영향을 준다.

즉, 캐시 친화적 접근을 이해한다는 것은  
단순히 “배열이 빠르다”를 외우는 것이 아니라  
**데이터가 메모리에 어떻게 놓이고, CPU가 그것을 어떤 단위로 가져오며, 접근 순서가 왜 실제 성능을 크게 좌우하는지 하드웨어 비용 모델 관점에서 이해하는 것**을 뜻한다.

이 관점이 잡혀야 이후 데이터 배치 최적화, 구조체 설계, AoS/SoA, 고성능 시스템 코드 설계로 자연스럽게 확장할 수 있다.
