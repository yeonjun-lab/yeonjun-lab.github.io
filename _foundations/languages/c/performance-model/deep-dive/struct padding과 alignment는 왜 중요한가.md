## 1. 왜 이 주제를 깊게 봐야 하는가

C에서 구조체를 처음 배울 때는 보통 이렇게 이해한다.

- 여러 타입의 값을 하나로 묶는 사용자 정의 자료형
- 관련 있는 데이터를 한 객체로 다루기 위한 방법
- 함수 인자, 반환값, 데이터 모델링에 유용한 도구

이 설명은 맞다.  
하지만 실제로 C 구조체를 깊게 다루기 시작하면 곧 이런 질문이 생긴다.

- 왜 구조체 크기가 필드 크기 합과 다를 수 있는가
- 왜 필드 순서를 바꾸면 `sizeof` 결과가 달라지는가
- 왜 어떤 구조체는 생각보다 메모리를 많이 쓰는가
- 왜 구조체 배열 성능이 필드 순서에 영향을 받을 수 있는가
- 왜 바이너리 파일 저장, 네트워크 전송, FFI 연동에서 구조체 배치를 조심해야 하는가

이 질문들의 중심에는 **padding**과 **alignment**가 있다.

예를 들어 다음 구조체를 보자.

```c
typedef struct {
    char a;
    int b;
    char c;
} Example;
```

초보자는 종종 이렇게 예상한다.

- `char` 1바이트
- `int` 4바이트
- `char` 1바이트
- 총합 6바이트

하지만 실제 `sizeof(Example)`는 그보다 더 클 수 있다.  
이건 컴파일러가 실수한 것이 아니라, 메모리 배치와 정렬 규칙이 작동한 결과다.

즉, 구조체는 단순히 필드를 순서대로 이어 붙인 덩어리가 아니다.  
구조체는 **CPU가 더 효율적으로 접근할 수 있도록 정렬 규칙을 반영한 메모리 객체**다.

이 주제를 깊게 봐야 하는 이유는 다음과 같다.

- 구조체를 진짜 메모리 배치 관점에서 이해하게 해 준다
- AoS/SoA, 캐시 친화성, 성능 최적화와 바로 연결된다
- 바이너리 호환성과 ABI 이해에 필수다
- 네트워크 패킷, 파일 포맷, 하드웨어 레지스터 매핑 같은 저수준 작업에서 매우 중요하다
- “겉보기 논리 구조”와 “실제 물리 배치”가 다를 수 있다는 점을 분명히 보여 준다

이 문서는 struct padding과 alignment가 각각 무엇인지, 왜 생기는지,  
구조체 크기와 필드 배치를 어떻게 바꾸는지,  
그리고 실무에서 왜 중요한지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

padding과 alignment를 큰 구조로 보면 다음과 같다.

### 2.1 alignment는 데이터가 놓이기 좋은 경계 규칙이다

어떤 타입은 메모리 어디에나 놓일 수 있는 것이 아니라,  
특정 배수 경계에 놓일 때 더 자연스럽거나 효율적일 수 있다.

예를 들어 어떤 환경에서 `int`가 4바이트이고 4바이트 경계 정렬을 선호한다고 하자.  
그러면 컴파일러는 `int`를 가능한 한 4의 배수 주소에 두려고 한다.

즉, alignment는 “이 타입을 어떤 경계에 배치하는 것이 적절한가”에 대한 규칙이다.

### 2.2 padding은 그 정렬을 맞추기 위해 삽입되는 빈 공간이다

구조체 안에서 필드들이 단순히 딱 붙어 있으면,  
다음 필드가 원하는 정렬 경계에 놓이지 않을 수 있다.

이때 컴파일러는 중간에 padding 바이트를 넣어 정렬을 맞춘다.

즉, padding은 낭비처럼 보일 수 있지만  
사실은 alignment를 만족시키기 위한 배치 조정이다.

### 2.3 구조체 끝에도 padding이 들어갈 수 있다

padding은 필드 사이에만 생기는 것이 아니다.  
구조체 전체 크기도 특정 정렬 조건을 만족해야 할 수 있어서,  
구조체 마지막에도 tail padding이 들어갈 수 있다.

즉, `sizeof(struct)`가 필드 합보다 더 커질 수 있는 이유는 중간 padding뿐 아니라 끝 padding도 있기 때문이다.

### 2.4 핵심은 “논리 필드 순서”와 “실제 메모리 배치”는 같지만 간격은 달라질 수 있다는 점이다

필드 순서는 보존된다.  
하지만 필드들 사이의 간격은 타입별 정렬 규칙 때문에 달라질 수 있다.

즉, 구조체는 순서만 있는 것이 아니라  
**정렬이 반영된 실제 배치 객체**다.

## 3. 내부 동작 원리

### 3.1 컴파일러는 각 필드를 정렬 규칙에 맞춰 배치한다

다음 구조체를 보자.

```c
typedef struct {
    char a;
    int b;
} Example;
```

논리적으로만 보면 `a` 뒤에 바로 `b`를 두고 싶을 수 있다.  
하지만 `b`가 4바이트 정렬을 원한다면, `a` 뒤 바로 다음 1바이트 위치는 적합하지 않을 수 있다.

그래서 컴파일러는 `a` 뒤에 padding을 넣고,  
`b`를 더 적절한 경계에 배치할 수 있다.

즉, 필드 배치는 단순 선언 순서 복사가 아니라  
정렬 규칙을 만족시키는 배치 과정이다.

### 3.2 가장 큰 정렬 요구가 구조체 전체 정렬에도 영향을 줄 수 있다

구조체는 내부 필드만 정렬하는 것이 아니라,  
구조체 객체 자체도 특정 정렬을 만족해야 할 수 있다.

예를 들어 구조체 안에 `int`나 `double` 같은 정렬 요구가 큰 필드가 있으면,  
그 구조체 자체도 더 큰 정렬 단위를 따를 수 있다.

이게 중요한 이유는 구조체 배열 때문이다.

```c
Example arr[10];
```

구조체 배열에서는 각 원소가 같은 정렬 조건을 만족해야 하므로,  
구조체 하나의 크기 자체도 적절한 배수로 맞춰지는 경우가 있다.

### 3.3 tail padding은 다음 구조체 원소 정렬을 위해 중요하다

다음 구조체를 생각하자.

```c
typedef struct {
    char a;
    int b;
    char c;
} Example;
```

마지막 필드 `c`까지 넣고 나면 논리적 내용은 끝났다.  
하지만 이 구조체가 배열 원소로 사용된다면, 다음 원소도 다시 정렬 조건을 만족해야 한다.

즉, 구조체 끝에도 tail padding을 넣어  
다음 원소 시작 위치가 적절한 정렬 경계에 오도록 만들 수 있다.

### 3.4 필드 순서를 바꾸면 padding 양이 달라질 수 있다

다음 두 구조체를 비교하자.

```c
typedef struct {
    char a;
    int b;
    char c;
} A;
```

```c
typedef struct {
    int b;
    char a;
    char c;
} B;
```

둘은 논리적으로 비슷한 정보를 담는다.  
하지만 실제 메모리 배치는 다를 수 있다.  
필드 순서를 잘 배치하면 중간 padding을 줄일 수 있다.

즉, 구조체 설계에서 필드 순서는 단순 미관 문제가 아니라 메모리 사용량과 성능에 영향을 줄 수 있다.

### 3.5 잘못 정렬된 접근은 하드웨어에 따라 더 비싸거나 위험할 수 있다

현대 CPU 중 일부는 misaligned access를 어느 정도 처리할 수 있다.  
하지만 여전히 더 비싸거나, 특정 환경에서는 문제를 일으킬 수 있다.

즉, alignment는 컴파일러 취향이 아니라  
하드웨어 접근 비용 모델과도 연결된다.

### 3.6 padding은 “쓸모없는 낭비”가 아니라 성능과 호환성 trade-off다

초보자는 padding을 보면 “메모리 낭비”로만 생각하기 쉽다.  
하지만 padding을 없애는 것이 항상 좋은 것은 아니다.

정렬을 무시하면:

- 접근 비용 증가
- ABI 문제
- 플랫폼 호환성 문제
- 하드웨어 제약 문제

가 생길 수 있다.

즉, padding은 단순 낭비가 아니라  
정렬, 성능, 호환성을 위한 설계 결과다.

## 4. 핵심 구성 요소

### 4.1 alignment

특정 타입이 놓이기 원하는 메모리 경계 규칙이다.  
예를 들어 4바이트 정렬, 8바이트 정렬 같은 개념이 여기에 해당한다.

### 4.2 padding

정렬을 맞추기 위해 구조체 안이나 끝에 삽입되는 빈 공간이다.  
실제 유효 필드는 아니지만 구조체 크기에 포함될 수 있다.

### 4.3 member offset

각 필드가 구조체 시작점으로부터 얼마나 떨어져 있는지를 나타내는 위치 정보다.  
padding이 있으면 오프셋이 단순 누적 합과 다를 수 있다.

### 4.4 tail padding

구조체 마지막에 들어가는 padding이다.  
특히 구조체 배열에서 다음 원소 정렬을 맞추는 데 중요하다.

### 4.5 ABI

Application Binary Interface는 함수 호출, 데이터 배치, 구조체 전달 방식과 연결된다.  
struct layout은 ABI와 직접 관련된다.

### 4.6 packed layout

정렬을 줄이거나 padding을 억제하는 배치 전략이 존재할 수 있다.  
하지만 이는 성능과 호환성 trade-off를 가져온다.

## 5. 실행 흐름 또는 처리 순서

다음 구조체를 기준으로 생각해보자.

```c
typedef struct {
    char a;
    int b;
    char c;
} Example;
```

### 5.1 첫 필드 배치

먼저 `a`가 구조체 시작 위치에 놓인다.  
`char`는 보통 정렬 요구가 작으므로 바로 시작할 수 있다.

### 5.2 다음 필드 정렬 확인

이제 `b`를 배치해야 한다.  
`b`가 `int`이고 더 큰 정렬을 원한다면, 현재 위치가 적절한지 확인한다.

만약 적절하지 않으면 그 사이에 padding을 넣는다.

### 5.3 나머지 필드 배치

`b` 다음에 `c`를 놓는다.  
`c`는 `char`이므로 보통 바로 뒤에 놓일 수 있다.

### 5.4 구조체 전체 크기 정렬

모든 필드를 놓은 뒤에도 구조체 전체 크기를 적절한 배수로 맞춰야 할 수 있다.  
그래서 구조체 끝에 tail padding이 들어갈 수 있다.

### 5.5 배열 관점 검토

이 구조체를 배열로 만들면:

```c
Example arr[10];
```

각 원소가 같은 정렬을 유지해야 한다.  
따라서 구조체 크기가 단순 필드 합이 아니라 “다음 원소도 정렬되게 하는 크기”가 된다.

즉, struct layout은 단일 객체뿐 아니라 배열 사용까지 고려한 결과다.

## 6. 성능 / 최적화 관점

### 6.1 잘 정렬된 접근은 보통 더 효율적이다

정렬이 잘 맞으면 CPU가 데이터를 더 자연스럽게 읽고 쓸 수 있다.  
즉, alignment는 메모리 접근 비용과 직접 연결된다.

### 6.2 padding을 줄이기 위한 필드 재배치는 메모리 절약에 도움이 될 수 있다

큰 구조체 배열을 많이 사용할 때는  
필드 순서를 바꿔 padding을 줄이면 전체 메모리 사용량이 크게 줄 수 있다.

예를 들어 구조체가 16바이트냐 24바이트냐 차이는  
대규모 배열에서는 매우 커질 수 있다.

### 6.3 하지만 무조건 padding 최소화만 추구하면 안 된다

padding을 줄이는 것이 곧 최적 성능이라는 뜻은 아니다.  
자주 함께 쓰는 필드를 가깝게 두는 것이 더 중요할 수도 있고,  
ABI 호환성과 코드 명확성이 더 중요할 수도 있다.

즉, 메모리 절약과 접근 패턴은 함께 봐야 한다.

### 6.4 구조체 배열 성능은 layout에 크게 좌우될 수 있다

AoS 구조에서 구조체 크기가 커지고 불필요한 패딩이 많으면  
캐시 효율이 나빠질 수 있다.

즉, struct padding은 단순 크기 문제가 아니라  
캐시 라인 활용과도 직접 연결된다.

### 6.5 packed 구조는 조심해서 써야 한다

padding을 강제로 없애는 packed 구조는  
외부 바이너리 포맷 해석에는 필요할 수 있다.  
하지만 일반 계산용 데이터 구조로는 접근 비용을 높일 수 있다.

즉, packed는 만능 최적화가 아니라 특수 목적 도구다.

## 7. 어디서부터 논리 구조와 실제 메모리 배치 결과가 갈리는가

### 7.1 선언에 적힌 필드 목록과 실제 메모리 크기는 다를 수 있다

프로그래머는 구조체를 필드들의 논리 묶음으로 본다.  
하지만 실제 메모리에서는 alignment와 padding이 개입해  
필드 합과 다른 배치 결과가 나온다.

즉, 구조체 선언서와 실제 객체 표현은 같은 층이 아니다.

### 7.2 padding은 낭비가 아니라 정렬 규칙의 결과다

빈 바이트가 보이면 무조건 쓸모없는 공간처럼 느껴질 수 있다.  
하지만 padding은 CPU 접근 효율, 배열 원소 정렬 유지, ABI 규칙 때문에 생긴다.

즉, padding은 우연한 쓰레기 공간이 아니라  
정렬 규칙이 만든 구조적 결과다.

### 7.3 필드 재배치는 메모리 절약과 접근 패턴 사이의 균형 문제다

padding을 줄이기 위해 필드를 재배치할 수 있다.  
하지만 자주 함께 쓰는 필드 근접성, ABI 안정성, 코드 명확성과 충돌할 수도 있다.

즉, 최소 크기가 항상 최적 구조라는 뜻은 아니다.

### 7.4 packed는 "더 타이트한 구조"가 아니라 다른 비용 모델을 여는 선택이다

padding을 제거하면 메모리 절약처럼 보이지만  
정렬이 깨진 접근 비용과 이식성 부담이 생길 수 있다.

즉, packed는 기본 최적화가 아니라  
특수 목적에 맞춘 다른 비용 체계 선택이다.

### 7.5 구조체 layout은 단일 객체가 아니라 배열/ABI/FFI까지 영향을 준다

구조체 하나의 크기 차이는 사소해 보여도  
대규모 배열, 함수 호출 규약, 다른 언어와의 경계에서 큰 차이를 만든다.

즉, layout 문제는 로컬 구현 디테일이 아니라  
경계와 호환성 문제로 확장된다.

---

## 8. 실무에서 중요한 판단 기준

### 8.1 구조체 크기는 감으로 판단하지 않고 `sizeof`로 확인한다

구조체는 필드 크기 합과 실제 크기가 다를 수 있다.  
따라서 성능이나 메모리 설계에서 중요한 구조체는 반드시 실제 크기를 확인해야 한다.

즉, 추측보다 측정이 우선이다.

### 8.2 필드 순서는 의미와 배치를 함께 고려해 정한다

필드 순서를 정할 때는 다음을 함께 봐야 한다.

- 가독성과 도메인 의미
- padding 최소화
- 자주 함께 쓰는 필드의 근접성
- ABI/직렬화 요구

즉, 구조체 설계는 단순 선언 나열이 아니다.

### 8.3 외부 바이너리 포맷과 직접 대응할 때는 특히 조심한다

네트워크 패킷, 파일 포맷, 하드웨어 레지스터 매핑과 같이  
정확한 바이트 배치가 중요한 경우에는 struct layout을 더욱 신중히 다뤄야 한다.

즉, “내 머신에서 sizeof가 이렇게 나왔다”만으로 안심하면 안 된다.

### 8.4 성능 핵심 구조체는 캐시 관점으로도 본다

대규모 배열에 담기거나 루프에서 자주 접근되는 구조체는  
padding뿐 아니라 캐시 라인 활용까지 함께 고려해야 한다.

즉, 구조체는 논리 모델이면서 동시에 성능 모델이다.

### 8.5 packed 사용은 목적이 명확할 때만 한다

단순히 메모리를 아끼고 싶어서 무조건 packed를 쓰는 것은 위험할 수 있다.  
정렬, 성능, 이식성 비용을 함께 봐야 한다.

즉, packed는 기본값이 아니라 예외적 선택이다.

---

## 9. 판단 체크리스트

- 구조체 선언서와 실제 메모리 배치 결과를 같은 것으로 보고 있지 않은가
- padding을 단순 낭비가 아니라 정렬 규칙 결과로 이해하고 있는가
- 필드 재배치를 최소 크기 문제와 접근 패턴/ABI 문제로 함께 보고 있는가
- packed를 기본 최적화가 아니라 별도 비용 모델 선택으로 보고 있는가
- layout 문제가 배열, ABI, FFI, 바이너리 호환성까지 이어짐을 의식하고 있는가

---

## 10. 더 깊게 볼 포인트

### 10.1 offsetof

각 필드 오프셋을 명시적으로 확인하는 도구와 함께 struct layout을 더 정밀하게 볼 수 있다.

### 10.2 ABI 문서와 구조체 전달 규칙

플랫폼 ABI가 구조체를 함수 인자로 어떻게 전달하는지까지 확장할 수 있다.

### 10.3 packed attribute / pragma

컴파일러별 packed 제어 방식과 그 부작용으로 더 깊게 들어갈 수 있다.

### 10.4 AoS/SoA와 struct layout 결합

구조체 내부 정렬 문제와 AoS/SoA 데이터 배치를 함께 최적화하는 문제로 이어질 수 있다.

### 10.5 FFI와 바이너리 호환성

다른 언어, 다른 컴파일러, 다른 플랫폼과 구조체를 주고받을 때 왜 layout이 중요한지로 확장할 수 있다.

## 정리

struct padding과 alignment는 C 구조체 메모리 배치의 핵심이다.

핵심은 다음과 같다.

- alignment는 타입이 놓이기 원하는 메모리 경계 규칙이다.
- padding은 그 정렬을 맞추기 위해 삽입되는 빈 공간이다.
- 구조체 크기는 필드 크기 합과 다를 수 있다.
- 구조체 끝에도 tail padding이 들어갈 수 있다.
- 필드 순서를 바꾸면 padding 양과 구조체 크기가 달라질 수 있다.
- 이 차이는 메모리 사용량, 캐시 효율, ABI, 바이너리 호환성에 영향을 준다.
- packed layout은 특수 목적에 유용할 수 있지만 일반적인 최적 해법은 아니다.

즉, struct padding과 alignment를 이해한다는 것은  
단순히 “왜 sizeof가 다르지?”를 넘어서  
**구조체가 실제 메모리에 어떻게 배치되고, 그 배치가 성능·호환성·안전성에 어떤 영향을 주는지 이해하는 것**을 뜻한다.

이 관점이 잡혀야 구조체 배열 최적화, 바이너리 포맷 처리, FFI, ABI, 고성능 데이터 구조 설계를 더 깊게 다룰 수 있다.
