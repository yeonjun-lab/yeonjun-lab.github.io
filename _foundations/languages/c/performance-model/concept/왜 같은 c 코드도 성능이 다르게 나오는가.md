## 1. 왜 이 주제를 공부했는가

C는 빠르다고 자주 말한다.  
하지만 실제로는 같은 C 코드라도 데이터 크기, 접근 순서, 할당 방식, 시스템 경계 때문에 성능이 크게 달라질 수 있다.

즉, 성능은 언어 이름보다  
**코드가 어떤 비용 모델 위에서 실행되는가**를 이해해야 읽을 수 있다.

## 2. 정의

C 코드의 성능은 프로그램이 실행되면서 소비하는  
연산, 메모리 접근, 분기, 할당, 시스템 호출 비용이 합쳐져 결정된다.

중요한 큰 축은 다음과 같다.

- 알고리즘 / 자료구조
- 메모리 접근 패턴
- 캐시 친화성
- 동적 할당과 시스템 경계
- 컴파일러 최적화 가능성

## 3. 핵심 개념

### 3.1 가장 큰 비용은 종종 계산이 아니라 메모리 접근이다

CPU는 빠르지만 메모리는 상대적으로 느리다.  
따라서 데이터를 어떻게 읽느냐가 실제 성능에 큰 영향을 준다.

### 3.2 단순한 코드가 더 빠를 수 있다

복잡한 트릭보다  
연속 메모리, 단순 루프, 명확한 데이터 흐름이 더 유리한 경우가 많다.

### 3.3 성능은 측정 대상이지 감으로 결정할 수 있는 것이 아니다

병목이 계산인지, 메모리인지, I/O인지 먼저 측정해야 한다.

## 4. 예시

같은 원소 수를 처리하더라도  
순차 접근 배열 루프와 포인터를 여러 번 따라가는 비연속 구조는 비용이 다를 수 있다.

즉, “같은 기능”과 “같은 비용”은 같은 말이 아니다.

## 5. 헷갈리기 쉬운 부분

### 5.1 C면 자동으로 빠르다고 생각하는 문제

언어보다 알고리즘과 메모리 접근 구조가 더 크게 작용할 수 있다.

### 5.2 미세 최적화가 가장 중요하다고 생각하는 문제

실제로는 자료구조와 접근 패턴이 더 큰 차이를 만드는 경우가 많다.

### 5.3 측정 없이 병목을 짐작하는 문제

성능 문제는 겉모습과 실제 원인이 다를 수 있다.

## 6. 어디서부터 직관과 실제 규칙이 갈리는가

### 6.1 "연산 수가 적다"와 "실제로 빠르다"는 다른 말이다

메모리 대기, 캐시 미스, 시스템 경계 비용이 크면  
연산 몇 번 줄이는 것은 큰 의미가 없을 수 있다.

### 6.2 저수준 문법과 좋은 성능은 같은 뜻이 아니다

포인터 트릭이나 복잡한 매크로보다  
컴파일러와 하드웨어가 읽기 쉬운 단순 구조가 더 유리할 수 있다.

### 6.3 성능은 코드 한 줄이 아니라 전체 비용 구조에서 결정된다

알고리즘, 데이터 배치, 캐시, 할당, I/O를 함께 봐야 한다.

## 7. 실무 관점

### 7.1 먼저 병목 종류를 구분한다

계산, 메모리, 할당, 시스템 콜, I/O 중 무엇이 비싼지 먼저 본다.

### 7.2 자료구조와 접근 순서를 우선 본다

문법 미세 조정보다 구조적 병목 제거가 먼저다.

### 7.3 측정 후 바꾼다

프로파일링 없이 감으로 최적화하면 헛된 작업이 되기 쉽다.

## 8. 판단 체크리스트

- 지금 병목이 계산인지 메모리 접근인지 구분했는가
- 연산 수만이 아니라 접근 순서와 데이터 배치를 보고 있는가
- 저수준 트릭과 실제 성능 향상을 같은 것으로 보고 있지 않은가
- 자료구조 선택이 미세 최적화보다 먼저라는 점을 의식하고 있는가
- 측정 없이 감으로 결론내리고 있지 않은가

### 9.3 캐시 친화적 접근은 왜 빠른가
세 번째 문서까지 보면 현재 문서가 전체 C 시리즈의 어디에 놓여 있는지 읽기 흐름이 더 안정된다.

## 정리

같은 C 코드도 성능이 다르게 나오는 이유는 간단하다.  
핵심은 **언어 이름이 아니라 비용 모델**이다.

알고리즘, 데이터 배치, 메모리 접근, 할당, 시스템 경계가 어떻게 섞이는지 봐야 C 성능을 제대로 이해할 수 있다.
