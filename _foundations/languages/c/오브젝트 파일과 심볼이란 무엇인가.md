## 1. 왜 이 주제를 깊게 봐야 하는가

C를 여러 파일로 나누어 작성하기 시작하면 컴파일과 링크를 따로 생각해야 한다는 점까지는 비교적 빨리 이해하게 된다.  
하지만 그 다음에는 곧 더 구체적인 질문이 생긴다.

- `.c` 파일을 컴파일하면 정확히 무엇이 만들어지는가?
- 왜 `.o` 파일은 바로 실행되지 않는가?
- 링커는 무엇을 기준으로 여러 파일을 연결하는가?
- 함수 이름과 전역 변수 이름은 어디에 저장되어 있는가?
- undefined reference는 왜 생기는가?
- multiple definition은 왜 생기는가?

이 질문들의 중심에는 **오브젝트 파일(object file)** 과 **심볼(symbol)** 이 있다.

입문 단계에서는 보통 이렇게 요약한다.

- `.c`를 컴파일하면 `.o`가 나온다
- 링커가 `.o`를 합쳐 실행 파일을 만든다

이 설명은 맞다.  
하지만 여기서 멈추면 링크 에러를 제대로 해석하기 어렵고,  
헤더/선언/정의/extern/static이 왜 필요한지도 충분히 연결되지 않는다.

예를 들어 다음 구조를 생각해보자.

    // math_utils.c
    int add(int a, int b) {
        return a + b;
    }

    // main.c
    int add(int a, int b);

    int main(void) {
        return add(1, 2);
    }

이 프로그램에서 `main.c`는 `add`를 호출하지만, 실제 구현은 `math_utils.c`에 있다.  
그렇다면 링커는 어떻게 `main.c`의 호출과 `math_utils.c`의 정의를 연결하는가?

핵심은 이름이다.  
하지만 단순 문자열 비교 수준이 아니라, 컴파일 결과물 안에 들어 있는 **심볼 정보**를 통해 연결된다.

즉, 오브젝트 파일과 심볼을 이해한다는 것은 단순 빌드 절차를 넘어서  
**코드 조각들이 이름 단위로 어떻게 결합되는지 이해하는 것**이다.

이 문서는 오브젝트 파일이 무엇인지, 심볼이 어떤 역할을 하는지, 재배치(relocation)는 왜 필요한지, 그리고 이 모든 것이 링크 에러와 프로그램 구조 이해에 어떻게 연결되는지를 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

오브젝트 파일과 심볼을 큰 구조로 보면 다음과 같다.

### 2.1 오브젝트 파일은 컴파일 결과로 나온 중간 산출물이다

`.c` 파일 하나를 컴파일하면 보통 `.o` 같은 오브젝트 파일이 만들어진다.

예를 들어:

- `main.c` → `main.o`
- `math_utils.c` → `math_utils.o`

이 오브젝트 파일은 기계 코드 조각을 담고 있지만,  
보통 아직 혼자서는 완전한 실행 파일이 아니다.

즉, 오브젝트 파일은 “거의 실행 가능한 조각”이지 최종 프로그램은 아니다.

### 2.2 오브젝트 파일에는 코드만 있는 것이 아니다

오브젝트 파일 안에는 단순히 기계 코드만 들어 있는 것이 아니다.  
보통 다음과 같은 정보가 함께 들어 있다.

- 기계 코드
- 데이터 영역 정보
- 심볼 정보
- 아직 해결되지 않은 외부 참조 정보
- 재배치 정보

즉, 오브젝트 파일은 단순 바이너리 덩어리가 아니라  
**나중에 링크할 수 있도록 구조화된 중간 결과물**이다.

### 2.3 심볼은 이름 기반 연결의 기준이다

심볼은 함수 이름, 전역 변수 이름 같은 식별자를 의미한다.  
링커는 이 심볼 정보를 바탕으로 “이 파일이 찾는 이름”과 “저 파일이 제공하는 이름”을 연결한다.

예를 들어:

- `main.o`는 `add`를 찾고 있다
- `math_utils.o`는 `add`를 정의하고 있다

링커는 이 둘을 심볼 기준으로 연결한다.

### 2.4 재배치는 “아직 주소가 확정되지 않은 참조를 나중에 맞추는 작업”이다

오브젝트 파일 단계에서는 최종 실행 파일 안에서 코드와 데이터가 정확히 어디에 놓일지 아직 확정되지 않았을 수 있다.  
따라서 외부 함수나 전역 변수 참조는 나중에 다시 맞춰야 한다.

이 작업이 재배치와 연결된다.

즉, 오브젝트 파일은 “아직 최종 배치가 끝나지 않은 상태”다.

## 3. 내부 동작 원리

### 3.1 각 번역 단위는 독립적으로 오브젝트 파일이 된다

다음 두 파일이 있다고 하자.

    // math_utils.c
    int add(int a, int b) {
        return a + b;
    }

    // main.c
    int add(int a, int b);

    int main(void) {
        return add(1, 2);
    }

컴파일러는 이 둘을 각각 독립적으로 처리한다.  
즉, 먼저 `math_utils.c` 하나만 보고 `math_utils.o`를 만들고,  
`main.c` 하나만 보고 `main.o`를 만든다.

이 단계에서는 아직 두 파일이 하나의 실행 파일로 합쳐진 것이 아니다.

즉, 오브젝트 파일은 **번역 단위별 개별 결과물**이다.

### 3.2 각 오브젝트 파일은 자기 안에서 아는 것과 모르는 것을 함께 가진다

`math_utils.o`는 `add`의 정의를 알고 있다.  
즉, `add`를 제공하는 쪽이다.

반면 `main.o`는 `add`를 호출하지만, 그 구현이 현재 파일 안에는 없다.  
즉, `add`가 어딘가에 있다고만 알고 있다.

따라서 `main.o`는 다음과 같은 상태를 가진다.

- `main` 함수 코드는 있다
- `add`를 호출해야 한다는 정보도 있다
- 하지만 `add`가 정확히 어디 있는지는 아직 모른다

이 “아직 해결되지 않은 참조”가 나중에 링크에서 해결된다.

### 3.3 심볼은 제공하는 이름과 요구하는 이름으로 나뉠 수 있다

개념적으로 보면 각 오브젝트 파일은 다음 두 종류의 이름 정보를 가질 수 있다.

- 내가 **정의해서 제공하는 이름**
- 내가 **외부에서 찾고 있는 이름**

예를 들어:

- `math_utils.o`는 `add`를 제공한다
- `main.o`는 `main`을 제공하고 `add`를 요구한다

링커는 이 관계를 보고 서로 맞춘다.

즉, 심볼은 단순 이름 목록이 아니라 **제공/요구 관계의 핵심 정보**다.

### 3.4 링커는 심볼을 기준으로 참조를 해결한다

링커는 여러 오브젝트 파일을 모아 다음을 수행한다.

1. 각 파일이 제공하는 심볼 확인
2. 각 파일이 요구하는 심볼 확인
3. 요구와 제공을 맞춤
4. 최종 실행 파일 내 배치 반영
5. 기계 코드 안의 미해결 참조를 수정

즉, 링크는 단순 합치기가 아니라 **심볼 해석과 주소 확정 작업**이다.

### 3.5 undefined reference는 요구 심볼을 못 찾았다는 뜻이다

예를 들어 `main.o`가 `add`를 요구하는데,  
어떤 오브젝트 파일이나 라이브러리에서도 `add` 정의를 제공하지 않는다면 어떻게 될까?

이 경우 링커는 다음을 할 수 없다.

- `add` 호출 위치를 실제 코드 위치와 연결

즉, undefined reference는 “이 이름이 어딘가 있을 거라 생각했는데 최종적으로 못 찾았다”는 뜻이다.

이건 문법 문제가 아니라 심볼 해결 실패다.

### 3.6 multiple definition은 같은 제공 심볼이 여러 번 나왔다는 뜻이다

반대로 어떤 이름을 여러 오브젝트 파일이 동시에 “내가 제공한다”고 주장하면 문제가 생긴다.

예를 들어 두 파일이 모두 `add`를 실제 정의하고 있다면,  
링커는 어떤 것을 최종 정의로 써야 할지 결정할 수 없다.

즉, multiple definition은 “제공 심볼 충돌” 문제다.

### 3.7 `static`은 심볼 공개 범위에도 영향을 준다

파일 수준 `static` 함수나 변수는 보통 현재 번역 단위 내부 linkage를 가진다.  
즉, 외부로 공개되는 심볼 목록에 영향을 줄 수 있다.

예를 들어:

    static void helper(void) {
    }

이 함수는 다른 파일이 참조할 외부 심볼로 다뤄지지 않는다.

즉, `static`은 단순 코드 스타일이 아니라 **심볼 가시성 제어 도구**이기도 하다.

## 4. 핵심 구성 요소

### 4.1 object file

오브젝트 파일은 각 번역 단위를 컴파일/어셈블한 결과로 나오는 중간 산출물이다.  
기계 코드 조각과 링크용 메타정보를 함께 담고 있다.

즉, 오브젝트 파일은 “링크 가능한 코드 단위”다.

### 4.2 symbol

심볼은 함수 이름, 전역 변수 이름 등 링크 과정에서 기준이 되는 식별자다.  
링커는 심볼을 통해 파일 간 참조를 해결한다.

즉, 심볼은 프로그램 조각들을 연결하는 이름 기반 접점이다.

### 4.3 defined symbol

defined symbol은 현재 오브젝트 파일이 실제 정의를 제공하는 이름이다.

예를 들어:

    int add(int a, int b) {
        return a + b;
    }

여기서 `add`는 정의된 심볼이 된다.

### 4.4 undefined symbol

undefined symbol은 현재 파일이 필요로 하지만, 현재 파일 안에서는 정의하지 않은 이름이다.

예를 들어:

    int add(int a, int b);

    int main(void) {
        return add(1, 2);
    }

여기서 `main.o` 입장에서는 `add`가 undefined symbol처럼 취급될 수 있다.  
즉, 다른 곳에서 해결해야 할 이름이다.

### 4.5 relocation

재배치는 아직 확정되지 않은 참조 위치를 나중에 맞춰 넣는 과정이다.  
링커는 최종 배치가 정해진 뒤 이 정보를 이용해 참조를 수정한다.

즉, relocation은 “나중에 진짜 주소를 반영하기 위한 후처리 정보”다.

### 4.6 translation unit

번역 단위는 전처리 후 하나의 `.c` 파일 기준으로 컴파일되는 독립 단위다.  
오브젝트 파일은 보통 번역 단위 하나당 하나씩 대응된다.

즉, 번역 단위를 이해해야 오브젝트 파일 구조도 이해하기 쉽다.

## 5. 실행 흐름 또는 처리 순서

다음 구조를 기준으로 보자.

    // math_utils.c
    int add(int a, int b) {
        return a + b;
    }

    // main.c
    int add(int a, int b);

    int main(void) {
        return add(1, 2);
    }

### 5.1 각 파일 개별 컴파일

먼저 `math_utils.c`는 `math_utils.o`가 되고,  
`main.c`는 `main.o`가 된다.

이 시점에서 각 오브젝트 파일은 독립적이다.

### 5.2 `math_utils.o`는 `add` 정의를 포함

`math_utils.o`는 `add`의 실제 코드와,  
이 파일이 `add`를 제공한다는 심볼 정보를 가진다.

즉, `add`는 이 파일의 defined symbol이다.

### 5.3 `main.o`는 `main` 정의와 `add` 참조를 포함

`main.o`는 `main` 함수 코드를 가진다.  
그리고 `add`를 호출해야 한다는 정보도 가진다.

하지만 `add`의 구현은 현재 파일 안에 없으므로,  
`main.o` 입장에서는 `add`가 아직 해결되지 않은 외부 참조다.

### 5.4 링크 단계에서 심볼 매칭

링커는 여러 오브젝트 파일을 보며 다음을 맞춘다.

- `main.o`가 요구하는 `add`
- `math_utils.o`가 제공하는 `add`

이 매칭이 성공하면 `main`의 `add` 호출은 실제 `add` 코드 위치와 연결된다.

### 5.5 최종 배치와 재배치 반영

링커는 코드와 데이터의 최종 위치를 결정하고,  
오브젝트 파일 안의 미해결 참조를 실제 주소에 맞게 수정한다.

즉, 이때 재배치 정보가 사용된다.

### 5.6 최종 실행 파일 생성

모든 심볼 참조가 해결되고 재배치가 끝나면 최종 실행 파일이 만들어진다.  
이제 운영체제가 로드하고 실행할 수 있는 형태가 된다.

## 6. 성능 / 최적화 관점

### 6.1 모듈 분리가 빌드 구조를 더 잘게 나눈다

오브젝트 파일 단위로 나뉘면 전체 프로젝트를 더 작은 조각으로 관리할 수 있다.  
이건 단순 구조화뿐 아니라 빌드 효율에도 도움이 된다.

예를 들어 일부 파일만 바뀌면 해당 파일만 다시 컴파일하고 나중에 링크하면 된다.

즉, 오브젝트 파일 구조는 대규모 빌드 효율의 기반이다.

### 6.2 심볼 공개 범위를 줄이면 구조가 더 안정된다

필요 없는 전역 심볼을 줄이고, 내부 구현은 `static`으로 숨기면  
링크 관점에서도 프로그램 표면적이 더 작아진다.

즉, 심볼 설계는 단순 네이밍 문제가 아니라 모듈 경계와 연결된다.

### 6.3 링크 단계 이해는 라이브러리 최적화로도 이어진다

현재 단계에서는 깊게 다루지 않아도 되지만,  
오브젝트 파일과 심볼 구조를 이해하면 이후 다음 주제로 자연스럽게 연결된다.

- 정적 라이브러리
- 동적 라이브러리
- 심볼 가시성 최적화
- 링크 타임 최적화

즉, 이 문서는 빌드 최적화의 입구이기도 하다.

### 6.4 링크 에러를 빠르게 해석할 수 있게 된다

컴파일 에러보다 링크 에러는 처음엔 더 추상적으로 느껴질 수 있다.  
하지만 오브젝트 파일과 심볼 구조를 알면 에러 메시지를 훨씬 구조적으로 읽을 수 있다.

예를 들어:

- undefined reference → 요구 심볼을 못 찾음
- multiple definition → 제공 심볼 충돌

즉, 링크 에러가 “이상한 툴 오류”가 아니라 명확한 구조 문제로 보이게 된다.

---

## 7. 실무에서 중요한 판단 기준

### 7.1 선언과 정의를 구분할 때 심볼 관점으로 생각한다

함수 선언만 있는 상태는 “이 이름을 외부에서 찾겠다”는 뜻으로 이어질 수 있다.  
반면 정의는 “이 이름의 실제 구현을 제공한다”는 뜻이다.

즉, 선언/정의 차이를 심볼 제공/요구 관점으로 보면 훨씬 명확하다.

### 7.2 링크 에러는 심볼 제공/요구 불일치로 먼저 해석한다

실무에서 링크 에러를 보면 먼저 이렇게 질문하는 것이 좋다.

- 이 파일은 어떤 심볼을 요구하는가?
- 그 심볼을 실제로 제공하는 파일이 있는가?
- 같은 심볼을 여러 파일이 동시에 제공하고 있지는 않은가?
- 라이브러리 연결이 빠진 것은 아닌가?

즉, 링크 에러는 텍스트 문제가 아니라 심볼 관계 문제로 보는 습관이 중요하다.

### 7.3 헤더 설계는 곧 심볼 설계다

헤더에 어떤 선언을 공개하느냐는  
결국 어떤 심볼을 외부에서 쓰게 할 것이냐와 연결된다.

즉, 헤더를 설계할 때는 단순히 “편한가”보다  
“외부에 어떤 이름을 공개할 것인가”를 생각해야 한다.

### 7.4 내부 전용 함수와 상태는 심볼 노출을 최소화하는 편이 좋다

모든 helper 함수를 외부 심볼로 노출할 필요는 없다.  
파일 내부에서만 쓰는 함수와 변수는 내부 linkage로 제한하는 것이 구조적으로 더 안전하다.

즉, 심볼은 적게 공개할수록 모듈 경계가 명확해지는 경향이 있다.

### 7.5 오브젝트 파일은 중간 산출물이지 최종 목적지가 아니다

`.o` 파일을 이해하면 빌드 과정이 더 잘 보인다.  
하지만 `.o`는 보통 그 자체로 목적이 아니라, 최종 실행 파일이나 라이브러리를 만들기 위한 연결 가능한 조각이다.

즉, 오브젝트 파일은 중간 산출물로서의 역할을 분명히 인식하는 것이 중요하다.

---

## 8. 더 깊게 볼 포인트

### 8.1 섹션(section)

오브젝트 파일 안에는 코드, 초기화된 데이터, 초기화되지 않은 데이터 등  
여러 섹션 개념이 들어간다. 더 깊게 보면 매우 중요하다.

### 8.2 심볼 테이블(symbol table)

링커와 디버거가 이름 정보를 어떻게 보는지 이해하려면 심볼 테이블 개념으로 더 들어갈 수 있다.

### 8.3 relocation entry

재배치 정보가 실제로 어떤 참조를 어떻게 수정할지 기록하는 방식으로 더 깊게 확장할 수 있다.

### 8.4 정적 라이브러리와 동적 라이브러리

오브젝트 파일이 어떻게 묶여 라이브러리가 되는지,  
그리고 심볼 해석이 라이브러리 수준에서 어떻게 달라지는지로 확장할 수 있다.

### 8.5 심볼 가시성과 ABI

공개 심볼 설계는 라이브러리 인터페이스, 버전 호환성, ABI 안정성과 연결된다.

---

## 정리

오브젝트 파일과 심볼은 C 빌드 구조의 핵심이다.

핵심은 다음과 같다.

- 오브젝트 파일은 각 번역 단위를 컴파일한 중간 산출물이다.
- 오브젝트 파일에는 기계 코드만이 아니라 심볼과 재배치 정보도 들어 있다.
- 심볼은 함수 이름, 전역 변수 이름 등 링크 기준이 되는 식별자다.
- 어떤 오브젝트 파일은 심볼을 제공하고, 어떤 오브젝트 파일은 그 심볼을 요구한다.
- 링커는 이 제공/요구 관계를 맞춰 최종 실행 파일을 만든다.
- undefined reference는 요구 심볼을 못 찾은 경우다.
- multiple definition은 같은 심볼을 여러 번 제공한 경우다.
- `static` 같은 키워드는 외부에 공개되는 심볼 범위를 제어하는 데 중요하다.

즉, 오브젝트 파일과 심볼을 이해한다는 것은  
단순히 “컴파일하면 `.o`가 나온다” 수준을 넘어서  
**프로그램의 코드 조각들이 이름 기반으로 어떻게 연결되고, 어떤 참조가 언제 해결되며, 링크 에러가 왜 발생하는지 구조적으로 이해한다**는 뜻이다.

이 관점이 잡혀야 링크 에러 디버깅, 라이브러리 설계, 심볼 가시성 제어, 더 큰 C 프로젝트 구조를 훨씬 안정적으로 다룰 수 있다.
