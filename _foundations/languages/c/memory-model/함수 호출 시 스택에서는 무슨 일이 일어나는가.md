## 1. 왜 이 주제를 깊게 봐야 하는가

프로그래밍 입문에서는 함수를 보통 “재사용 가능한 기능 묶음”이라고 배운다.  
이 설명은 틀리지 않다.  
하지만 C를 이해하는 데에는 그 설명만으로는 부족하다.

C에서 함수 호출은 단순히 어떤 기능을 실행하는 문법이 아니다.  
함수 호출은 실제로 다음과 연결된다.

- 인자 전달
- 지역 변수 생성
- 반환값 처리
- 함수 종료 시 정리
- 객체의 수명 시작과 종료
- 스택 메모리 사용

즉, 함수 호출은 실행 흐름의 변화이면서 동시에 **메모리 상태 변화**다.

이 점을 정확히 보지 못하면 다음과 같은 문제에서 자주 막힌다.

- 왜 지역 변수는 함수가 끝나면 사라지는가?
- 왜 지역 변수의 주소를 반환하면 위험한가?
- 왜 재귀 호출은 스택을 많이 쓰는가?
- 함수 인자를 넘길 때 실제로 복사가 일어나는가?
- 왜 어떤 함수는 안전하고 어떤 함수는 수명 문제가 생기는가?

예를 들어 다음 코드를 보자.

```c
int add(int a, int b) {
    int result = a + b;
    return result;
}
```

겉으로 보면 단순한 덧셈 함수다.  
하지만 실행 관점에서는 더 많은 일이 일어난다.

- `a`, `b`를 위한 공간이 필요하다.
- `result`를 위한 지역 객체가 생성된다.
- 함수가 끝나면 이 지역 객체들은 정리된다.
- 반환값은 호출한 쪽으로 전달된다.

즉, 함수는 단순한 수학식 포장이 아니라 **스택 프레임을 만들고 정리하는 실행 단위**다.

이 문서는 함수 호출 시 스택에서 어떤 일이 일어나는지, 지역 변수의 수명은 어떻게 결정되는지, 그리고 왜 이것이 C에서 매우 중요한지 깊게 설명하기 위해 작성한다.

## 2. 전체 구조

함수 호출을 큰 구조로 보면 다음과 같이 이해할 수 있다.

### 2.1 호출자와 피호출자가 있다

함수를 호출하는 쪽을 caller, 호출당하는 함수를 callee라고 생각할 수 있다.

예를 들어:

```c
int add(int a, int b) {
    int result = a + b;
    return result;
}
```

```c
int main(void) {
    int x = add(10, 20);
    return 0;
}
```

여기서 `main`은 caller이고, `add`는 callee다.

즉, 함수 호출은 “한 코드 덩어리에서 다른 코드 덩어리로 제어가 이동하는 것”이다.

### 2.2 함수 호출 시 스택 프레임이 만들어진다

함수가 호출되면 보통 스택 위에 그 함수 실행을 위한 새로운 영역이 만들어진다.  
이 영역을 흔히 **스택 프레임(stack frame)** 이라고 부른다.

이 프레임 안에는 보통 다음과 관련된 정보가 들어간다.

- 함수 인자
- 지역 변수
- 복귀 정보
- 실행 문맥 일부

즉, 함수 호출은 스택 프레임 생성과 매우 밀접하다.

### 2.3 지역 변수는 보통 현재 함수의 스택 프레임 안에 존재한다

함수 안에서 선언한 지역 변수는 일반적으로 그 함수의 실행 동안만 유효하다.

예를 들어:

```c
void f(void) {
    int x = 10;
}
```

여기서 `x`는 `f`가 실행되는 동안만 존재하는 지역 객체다.  
즉, `f`가 끝나면 `x`를 위한 스택 공간도 더 이상 유효한 객체로 간주되지 않는다.

### 2.4 함수가 끝나면 프레임도 정리된다

함수 실행이 끝나면 반환값이 있으면 전달하고, 스택 프레임은 정리된다.  
즉, 그 프레임 안에 있던 지역 변수들도 유효성을 잃는다.

이 점이 매우 중요하다.  
왜냐하면 지역 변수의 주소를 바깥으로 넘기는 코드가 왜 위험한지를 설명하는 핵심이기 때문이다.

## 3. 내부 동작 원리

### 3.1 함수가 호출되면 제어가 이동한다

프로그램이 어떤 함수를 호출하면, 단순히 그 함수 코드가 “복사되어 실행되는 것”이 아니다.  
실제로는 현재 실행 흐름이 피호출 함수 쪽으로 이동한다.

예를 들어:

```c
int x = add(10, 20);
```

이 코드는 다음과 같은 흐름으로 이해할 수 있다.

1. `add(10, 20)`을 호출할 준비를 한다.
2. 인자 전달을 준비한다.
3. `add` 함수로 실행 흐름이 이동한다.
4. `add` 내부가 실행된다.
5. 반환값이 caller로 돌아온다.
6. caller는 그 값을 받아 다음 문장을 계속 실행한다.

즉, 함수 호출은 실행 흐름의 점프이면서 복귀를 전제로 한 이동이다.

### 3.2 호출마다 독립적인 실행 공간이 생긴다

같은 함수라도 호출될 때마다 별도의 실행 공간이 필요하다.  
그래야 각 호출이 서로 다른 인자와 지역 변수를 독립적으로 가질 수 있다.

예를 들어:

```c
int add(int a, int b) {
    int result = a + b;
    return result;
}
```

이 함수가 한 번 호출될 때와 여러 번 호출될 때를 생각해보자.  
각 호출은 `a`, `b`, `result`를 자기만의 실행 맥락 안에서 가져야 한다.

이 독립적인 맥락이 바로 스택 프레임으로 이해된다.

### 3.3 인자는 함수 실행용 객체로 다뤄진다

함수 인자는 단순히 “문법상 괄호 안 값”이 아니다.  
함수가 실행되는 동안 참조할 수 있는 객체처럼 다뤄진다.

예를 들어:

```c
int add(int a, int b) {
    return a + b;
}
```

```c
int main(void) {
    int x = 10;
    int y = 20;
    int z = add(x, y);
    return 0;
}
```

`add(x, y)` 호출 시 `a`, `b`는 `main`의 `x`, `y`와 같은 이름도 아니고 같은 객체도 아니다.  
핵심은 다음이다.

- `x`, `y`의 값이 전달된다.
- `add`는 자기 실행 맥락 안에서 `a`, `b`를 사용한다.

즉, 기본적인 함수 인자 전달은 **값 전달** 관점으로 보는 것이 맞다.

### 3.4 지역 변수는 함수 실행 동안만 유효하다

다음 코드를 보자.

```c
void f(void) {
    int x = 10;
    printf("%d\n", x);
}
```

여기서 `x`는 `f` 내부에서만 쓰이는 지역 변수다.  
`f`가 실행되는 동안에는 유효하다.  
하지만 `f`가 끝나면 `x`라는 지역 객체의 lifetime은 종료된다.

중요한 것은 “메모리 칸이 물리적으로 곧바로 0이 된다” 같은 식으로 이해하면 안 된다는 점이다.  
핵심은 그 공간이 더 이상 `x`라는 유효한 객체로 보장되지 않는다는 것이다.

즉, 함수 종료 후에는 그 지역 변수에 접근하는 것이 논리적으로 잘못된 접근이 된다.

### 3.5 함수 종료 시 복귀와 정리가 일어난다

함수가 끝나면 보통 다음이 일어난다.

1. 반환값이 있으면 준비된다.
2. caller가 다시 실행될 위치로 제어가 돌아간다.
3. 현재 함수의 스택 프레임은 정리된다.
4. 그 프레임에 속한 지역 객체들은 lifetime이 끝난다.

즉, 함수 호출은 “들어갔다가 돌아오는 구조”이며, 들어갈 때 만든 실행 맥락은 돌아올 때 함께 정리된다.

### 3.6 지역 변수 주소 반환이 왜 위험한가

다음 코드는 C 입문에서 매우 중요한 위험 사례다.

```c
int* bad(void) {
    int x = 10;
    return &x;
}
```

겉으로 보면 `x`의 주소를 반환했으니 유용해 보일 수 있다.  
하지만 실행 구조를 보면 즉시 문제가 보인다.

- `x`는 `bad`의 지역 변수다.
- `x`는 `bad`의 스택 프레임 안에 존재한다.
- `bad`가 끝나면 그 프레임은 정리된다.
- 따라서 `&x`는 함수 종료 후 유효한 객체를 가리키지 않는다.

즉, 이 코드는 문법 문제가 아니라 **스택 수명 문제**다.

### 3.7 재귀는 스택 프레임을 계속 쌓는다

재귀 호출은 함수가 자기 자신을 다시 호출하는 구조다.  
이 경우 호출할 때마다 새로운 스택 프레임이 생긴다.

예를 들어:

```c
int sum_to(int n) {
    if (n == 0) return 0;
    return n + sum_to(n - 1);
}
```

이 함수는 `sum_to(3)` 호출 시 대략 다음처럼 프레임이 쌓인다.

- `sum_to(3)`용 프레임
- `sum_to(2)`용 프레임
- `sum_to(1)`용 프레임
- `sum_to(0)`용 프레임

즉, 재귀는 함수 호출 비용과 스택 사용량을 함께 늘린다.

## 4. 핵심 구성 요소

### 4.1 stack

스택은 함수 호출과 함께 사용되는 메모리 영역으로 이해할 수 있다.  
지역 변수, 인자, 실행 맥락 일부가 여기에 놓인다.

중요한 점은 스택이 “그냥 임시 메모리”가 아니라 **호출 구조와 수명 모델을 반영하는 메모리**라는 것이다.

### 4.2 stack frame

스택 프레임은 개별 함수 호출을 위해 스택 위에 잡히는 실행 단위다.  
즉, 함수가 실행되는 동안 필요한 지역적 맥락이다.

한 번의 호출마다 하나의 프레임이 생긴다고 생각하면 이해하기 쉽다.

### 4.3 parameter

parameter는 함수가 받는 입력이다.  
실행 관점에서는 함수가 자기 안에서 사용할 수 있도록 준비된 값 또는 객체다.

즉, 인자는 단순 문법 조각이 아니라 함수 프레임 안에서 다뤄지는 실행 데이터다.

### 4.4 local variable

local variable은 함수나 블록 안에서 선언되는 변수다.  
일반적으로 그 함수나 블록의 실행 범위 안에서만 유효하다.

즉, 지역 변수는 “작은 변수”가 아니라 **짧은 lifetime을 가지는 스택 기반 객체**로 이해하는 것이 중요하다.

### 4.5 return value

return value는 callee가 caller에게 돌려주는 결과다.  
이 값은 caller 쪽에서 이어지는 계산에 사용될 수 있다.

중요한 것은 반환 자체가 “지역 변수 그 자체를 넘기는 것”과 같은 것이 아니라는 점이다.  
지역 변수를 return하더라도 보통은 그 **값**을 반환하는 것이지, 지역 객체 자체를 외부에 영원히 남기는 것이 아니다.

### 4.6 return address

함수 호출 후에는 호출한 쪽으로 다시 돌아가야 한다.  
이 복귀 지점과 관련된 정보가 필요하다.

구현 세부 사항까지 깊게 들어가지 않아도, 함수 호출이 가능한 이유 중 하나는 “끝난 뒤 어디로 돌아갈지”를 추적하기 때문이라고 이해하면 된다.

## 5. 실행 흐름 또는 처리 순서

다음 코드를 기준으로 흐름을 보자.

```c
int add(int a, int b) {
    int result = a + b;
    return result;
}
```

```c
int main(void) {
    int x = add(10, 20);
    return 0;
}
```

### 5.1 호출 전 상태

`main`이 실행 중이다.  
현재 caller는 `main`이다.

- `main`의 실행 맥락이 존재한다.
- `x`는 아직 초기화되지 않았거나 아직 값이 정해지지 않았다.
- `add(10, 20)` 호출 직전이다.

### 5.2 호출 직후

`add` 호출이 시작되면 `add`를 위한 새로운 실행 맥락이 필요하다.

- `a = 10`
- `b = 20`
- `add`용 스택 프레임 생성

즉, caller의 실행 위에 callee의 실행 맥락이 하나 더 올라간다.

### 5.3 함수 본문 실행

이제 `add` 내부가 실행된다.

```c
int result = a + b;
```

여기서 `result`라는 지역 변수가 `add` 프레임 안에서 생성된다.  
그리고 `a + b` 계산 결과가 저장된다.

즉, 이 시점에는 `a`, `b`, `result`가 `add`의 지역 맥락 안에 존재한다.

### 5.4 반환 직전

다음 줄이 실행된다.

```c
return result;
```

여기서 caller에게 돌려줄 값이 준비된다.  
중요한 점은 `result` 자체를 바깥으로 “살려서 내보내는 것”이 아니라,  
그 **값**이 반환 흐름에 사용된다는 점이다.

### 5.5 반환 후

`add` 실행이 끝난다.

- `add` 프레임 정리
- `a`, `b`, `result` lifetime 종료
- caller인 `main`으로 복귀
- 반환값을 받아 `x`에 저장

즉, `x`는 `main`의 객체이고, `result`는 `add`의 객체다.  
값은 이어질 수 있지만 객체 자체는 각자의 수명을 가진다.

## 6. 성능 / 최적화 관점

### 6.1 함수 호출에는 비용이 있다

함수 호출은 공짜가 아니다.  
호출 준비, 인자 처리, 실행 흐름 이동, 복귀 등의 비용이 있다.

보통 이 비용은 아주 크지는 않지만, 매우 빈번한 짧은 함수 호출이 누적되면 의미가 생길 수 있다.

즉, 함수는 가독성과 구조화에 큰 장점이 있지만, 성능 관점에서는 호출 비용도 존재한다.

### 6.2 재귀는 스택 사용량을 빠르게 늘릴 수 있다

재귀는 구조적으로 우아할 수 있다.  
하지만 호출마다 프레임이 하나씩 더 생긴다는 점을 잊으면 안 된다.

즉, 깊은 재귀는 다음 문제를 만들 수 있다.

- 스택 사용량 증가
- 성능 저하
- 너무 깊으면 스택 고갈 위험

그래서 재귀는 논리적 아름다움만이 아니라 **프레임 누적 비용**까지 함께 봐야 한다.

### 6.3 큰 지역 객체는 스택 부담을 만든다

다음 같은 코드는 주의가 필요하다.

```c
void f(void) {
    int big[100000];
}
```

이 경우 큰 배열이 지역 변수로 스택에 잡힐 수 있다.  
즉, 지역 변수는 편하지만 무한정 안전한 것이 아니다.

큰 데이터는 다음을 따져봐야 한다.

- 정말 스택에 두어도 되는가
- 수명이 함수 내부에 한정되는가
- 힙 할당이 더 적절한가

### 6.4 인라인과 호출 구조 최적화의 입구가 된다

컴파일러는 경우에 따라 작은 함수를 인라인 처리할 수 있다.  
즉, 호출 비용을 줄이기 위한 최적화가 가능하다.

이 주제를 깊게 다루지 않더라도, 함수 호출 구조를 이해하면 왜 컴파일러가 이런 최적화를 시도하는지 자연스럽게 연결된다.

## 7. 실무에서 중요한 판단 기준

### 7.1 지역 변수 주소를 반환하면 안 된다

이건 C에서 매우 중요한 기본 규칙이다.

다음은 위험하다.

```c
int* bad(void) {
    int x = 10;
    return &x;
}
```

이유는 간단하다.

- `x`는 스택 프레임 안의 지역 객체다.
- 함수가 끝나면 그 lifetime은 끝난다.
- 그 주소를 바깥에서 쓰는 것은 유효하지 않다.

즉, 함수 수명과 객체 수명을 연결해서 판단해야 한다.

### 7.2 값 반환과 주소 반환을 구분해야 한다

다음은 안전하다.

```c
int good(void) {
    int x = 10;
    return x;
}
```

여기서는 `x`의 값을 반환할 뿐이다.  
반환 후 `x` 자체는 사라져도 문제없다.

반면 주소를 반환하면 지역 객체의 생존을 바깥에 기대하게 되므로 문제가 된다.

즉, “값을 넘기는 것”과 “객체의 위치를 넘기는 것”은 전혀 다르다.

### 7.3 스택과 힙을 구분해서 설계해야 한다

함수 안에서 만든 모든 데이터가 지역 변수여야 하는 것은 아니다.  
수명이 길거나 크기가 큰 데이터는 힙에 두는 것이 더 적절할 수 있다.

즉, 함수 설계에서는 다음을 같이 봐야 한다.

- 이 데이터는 얼마나 오래 살아야 하는가
- 크기는 얼마나 되는가
- 누가 소유하고 해제할 것인가

### 7.4 재귀는 멋있어 보여도 비용을 계산해야 한다

재귀가 논리적으로 자연스러울 수 있다.  
하지만 실무에서는 다음을 같이 봐야 한다.

- 입력 크기
- 최대 재귀 깊이
- 스택 사용량
- 반복문으로 대체 가능한지 여부

즉, 재귀는 수학적으로 예쁘다는 이유만으로 선택하면 안 된다.

## 8. 더 깊게 볼 포인트

### 8.1 calling convention

함수 인자를 어떤 순서와 방식으로 전달하는지, 반환값은 어떻게 전달하는지와 관련된 규칙이다.  
ABI와 함께 보면 더 명확해진다.

### 8.2 ABI

Application Binary Interface는 함수 호출, 바이너리 호환성, 데이터 배치 등과 연결된다.  
다중 언어 연동, 라이브러리 인터페이스, 시스템 수준 이해에 중요하다.

### 8.3 tail call

특정 조건에서 함수 호출이 추가 프레임 없이 최적화될 수 있는 경우와 연결된다.  
재귀 최적화와 관련된 중요한 주제다.

### 8.4 call stack 디버깅

디버거나 크래시 로그에서 call stack을 읽는 능력은 실무에서 매우 중요하다.  
어느 함수가 누구를 불렀는지, 어디서 문제가 시작되었는지를 추적할 수 있다.

### 8.5 static 지역 변수

일반 지역 변수와 달리 블록 안에 선언되더라도 수명이 함수 호출 전체를 넘어가는 경우가 있다.  
scope와 lifetime을 구분하는 좋은 예시다.

## 정리

함수 호출은 단순히 기능을 실행하는 문법이 아니다.  
함수 호출은 **스택 프레임을 만들고, 인자와 지역 변수를 위한 실행 맥락을 만들고, 끝나면 그것을 정리하는 과정**이다.

핵심은 다음과 같다.

- 함수 호출 시 caller에서 callee로 실행 흐름이 이동한다.
- 호출마다 독립적인 스택 프레임이 생긴다.
- 인자와 지역 변수는 그 프레임 안에서 다뤄진다.
- 함수가 끝나면 프레임은 정리되고 지역 변수 lifetime도 끝난다.
- 지역 변수 주소 반환은 위험하다.
- 재귀는 프레임을 계속 쌓으므로 비용과 한계를 고려해야 한다.

즉, C에서 함수를 제대로 이해한다는 것은  
“기능 묶음”이라는 표면 설명을 넘어서  
**실행과 수명과 메모리 관점으로 함수 호출을 볼 수 있게 되는 것**을 뜻한다.

이 관점이 잡혀야 포인터, 스택과 힙, 동적 메모리, 재귀, 디버깅을 훨씬 안정적으로 이해할 수 있다.
