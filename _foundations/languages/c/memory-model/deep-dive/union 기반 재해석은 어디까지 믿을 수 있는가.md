## 1. 왜 이 주제를 깊게 봐야 하는가

C를 어느 정도 공부하면 곧 이런 코드를 보게 된다.
```c
    union U {
        float f;
        unsigned int u;
    };

    union U x;
    x.f = 1.0f;
    printf("%u\n", x.u);
```
겉보기에는 매우 자연스럽다.

- 같은 메모리 공간을
- 여러 타입으로 겹쳐 두고
- 한쪽으로 쓰고 다른 쪽으로 읽는다

그래서 많은 사람이 이렇게 생각한다.

- union은 원래 이런 용도 아닌가
- C에서 비트 패턴 재해석은 union으로 하면 되는 것 아닌가
- 포인터 캐스팅보다 union이 더 안전한 것 아닌가

이 생각은 반은 맞고 반은 위험하다.

union은 실제로 **같은 저장공간을 여러 멤버 타입이 공유하게 만드는 구조**다.  
따라서 표현 수준에서 보면 type punning, 즉 비트 패턴 재해석에 매우 잘 어울려 보인다.

하지만 문제는 C에서 중요한 것이 단순 메모리 모양만이 아니라는 점이다.

- 현재 어떤 멤버가 유효한가
- 다른 멤버를 읽는 것이 언어적으로 어디까지 보장되는가
- 구현마다 어떤 동작 차이가 있을 수 있는가
- strict aliasing, effective type, object representation과 어떻게 연결되는가

즉 union 기반 재해석은  
"자주 쓰이는 기법"인 것은 맞지만  
"항상 완전히 믿을 수 있는 정석"이라고 말하기는 어렵다.

이 문서를 깊게 봐야 하는 이유는 다음과 같다.

- union이 왜 type punning에 자주 등장하는지 이해하게 해 준다
- 포인터 캐스팅 기반 재해석과 union 기반 재해석의 차이를 분리해 준다
- 어떤 경우는 관행적으로 잘 동작하고, 어떤 경우는 언어적으로 보장이 약한지 구분하게 해 준다
- 왜 실무에서 `memcpy`가 더 보수적이고 이식성 높은 기준점으로 자주 추천되는지 이해하게 해 준다
- 이후 strict aliasing, endianness, 직렬화, 바이너리 포맷 처리와 자연스럽게 연결된다

즉 이 문서는  
union을 “재해석 마법 도구”로 보는 단계에서 벗어나,  
**어디까지는 실용적이고 어디부터는 기대를 줄여야 하는가**를 정리하기 위해 필요하다.

## 2. 전체 구조

union 기반 재해석을 이해할 때는 아래 구조로 보면 가장 명확하다.

### 2.1 union은 저장공간을 공유한다

union의 모든 멤버는 같은 메모리 영역을 겹쳐서 사용한다.  
즉 어느 한 멤버에 값을 쓰면, 그 바이트 패턴은 다른 멤버와도 같은 저장공간에 존재한다.

### 2.2 하지만 "같은 저장공간"과 "언어적으로 같은 값으로 읽어도 된다"는 다르다

메모리가 겹친다는 사실은 물리적 현상이다.  
반면 어떤 멤버를 읽는 것이 정의된 동작인지 여부는 언어 규칙의 문제다.

### 2.3 union type punning은 구현 관행과 언어 보장 사이에 긴장이 있다

일부 컴파일러와 환경에서는 널리 사용되고 잘 동작한다.  
하지만 이것이 언제나 가장 이식성 높은 경로라는 뜻은 아니다.

### 2.4 그래서 실무에서는 "무엇을 하려는가"에 따라 전략이 갈린다

- 디버그용 비트 확인
- 특정 컴파일러/플랫폼 고정 코드
- 고이식성 라이브러리 코드
- 파일/네트워크 포맷 파싱

이 목적들에 대해 union을 같은 강도로 추천하면 안 된다.

---

## 3. 내부 동작 원리

### 3.1 union은 멤버들이 하나의 저장공간을 공유한다

예를 들어:
```c
    union U {
        int i;
        float f;
    };
```
이 union은 `int`와 `float`를 동시에 따로 저장하는 것이 아니라,  
하나의 저장공간을 `i`와 `f`가 겹쳐서 사용한다.

즉 개념적으로는 다음처럼 보면 된다.

- 저장공간은 하나
- 멤버 이름은 여러 개
- 각 멤버는 그 저장공간을 자기 타입으로 해석하는 창구

따라서:
```c
    union U u;
    u.i = 42;
```
라고 쓰면 그 저장공간에는 `int` 42의 객체 표현이 들어간다.

그 직후:
```c
    u.f
```
를 읽으면 같은 바이트를 `float` 해석 규칙으로 보는 셈이다.

바로 이 지점 때문에 union이 type punning에 자주 쓰인다.

---

### 3.2 union은 "겹친 메모리"를 제공하지만, 의미 보장은 별도 문제다

많은 사람이 여기서 착각한다.

- 같은 저장공간이니까
- 다른 멤버로 읽어도 당연히 허용될 것 같다

하지만 C는 단순한 바이트 언어가 아니라  
객체 타입, 객체 표현, 유효 값, aliasing 규칙이 함께 있는 언어다.

즉 다음 두 질문을 분리해야 한다.

1. 물리적으로 같은 바이트를 보는가  
2. 언어적으로 그 읽기를 보장하는가

union은 1번에는 분명히 "예"라고 답하게 해 준다.  
하지만 2번은 상황과 구현 기대를 더 조심해서 봐야 한다.

---

### 3.3 멤버 하나에 쓰고 다른 멤버를 읽는 것은 "표현 재해석"이다

예를 들어:
```c
    union Bits {
        float f;
        unsigned int u;
    };

    union Bits x;
    x.f = 1.0f;
```
이 상태에서 `x.u`를 읽는 행위는  
부동소수점 값 1.0을 정수로 "변환"하는 것이 아니다.

그것은 다음이다.

- `x.f`의 바이트 표현을
- 같은 저장공간에서
- `unsigned int` 객체처럼 해석하는 것

즉 union 기반 재해석은 산술 변환이 아니라  
**표현 해석 방식 변경**이다.

이 차이를 놓치면 type punning이 왜 위험할 수 있는지 이해하기 어렵다.

---

### 3.4 union은 포인터 캐스팅과는 미묘하게 다른 경로다

포인터 캐스팅 방식은 보통 이렇다.
```c
    float f = 1.0f;
    unsigned int u = *(unsigned int *)&f;
```
이건 `float` 객체 저장공간을  
`unsigned int` lvalue로 직접 읽는 것이다.

반면 union은 보통 이렇게 쓴다.
```c
    union Bits {
        float f;
        unsigned int u;
    };

    union Bits x;
    x.f = 1.0f;
    unsigned int u = x.u;
```
이건 "같은 union 저장공간의 다른 멤버를 읽는다"는 형태다.

즉 둘 다 재해석처럼 보이지만, 언어 구조는 다르다.

- 포인터 캐스팅: 기존 객체를 다른 타입 lvalue로 직접 접근
- union: 같은 union 객체의 다른 멤버를 통해 해석

그래서 많은 구현에서 union 방식이 포인터 캐스팅보다 덜 공격적으로 취급되거나, 관행적으로 더 받아들여지는 경우가 있다.

하지만 그걸 곧바로 "완전 이식 가능"으로 일반화하면 안 된다.

---

## 4. 핵심 구성 요소

### 4.1 shared storage

union 멤버들은 같은 저장공간을 공유한다.  
이것이 union의 가장 본질적인 특징이다.

### 4.2 active member라는 사고

실무적으로는 "지금 마지막으로 의미 있게 쓴 멤버가 무엇인가"를 따지는 습관이 중요하다.  
왜냐하면 현재 어떤 표현이 저장되어 있는지와 연결되기 때문이다.

### 4.3 object representation

union 기반 재해석은 값 변환이 아니라 객체 표현을 다른 타입 규칙으로 읽는 문제다.

### 4.4 effective type / aliasing tension

같은 저장공간을 공유한다는 점과, 어떤 타입 접근이 허용되는가 사이에는 긴장이 있다.  
union은 이 긴장 지점 한가운데 있다.

### 4.5 portability boundary

특정 컴파일러/특정 ABI/특정 하드웨어에서 잘 되더라도,  
그것이 넓은 이식성을 의미하지는 않는다.

즉 union 문서는 항상 이식성 경계를 함께 봐야 한다.

## 5. 실행 흐름 또는 처리 순서

다음 예시를 기준으로 생각해 보자.
```c
    union Bits {
        float f;
        unsigned int u;
    };

    union Bits x;
    x.f = 1.0f;
    unsigned int bits = x.u;
```
이 코드의 개념적 흐름은 다음과 같다.

### 5.1 union 객체 생성

`x`라는 union 객체가 만들어진다.  
이 객체는 `float`와 `unsigned int`가 겹쳐서 사용할 수 있는 하나의 저장공간을 가진다.

### 5.2 `f` 멤버에 쓰기
```c
    x.f = 1.0f;
```
이 시점에 그 저장공간에는 `float` 1.0의 객체 표현이 기록된다.

### 5.3 다른 멤버 `u`로 읽기
```c
    bits = x.u;
```
이제 같은 바이트를 `unsigned int` 해석 규칙으로 읽는다.

즉 여기서는 값 변환이 아니라  
"방금 저장한 표현을 다른 타입 창으로 바라본다"는 일이 일어난다.

### 5.4 결과 해석

현재 플랫폼의 float 표현, integer 크기, endianness, 구현 동작에 따라  
`bits`에는 특정 비트 패턴이 들어올 수 있다.

즉 union은 결과를 산술적으로 바꾸는 것이 아니라  
**표현을 드러내는 도구처럼 동작**한다.

---

## 6. 무엇이 실용적으로 믿을 만하고, 무엇이 아닌가

### 6.1 구현 고정, 플랫폼 고정, 컴파일러 고정 코드에서는 자주 사용된다

예를 들어:

- 특정 임베디드 타깃
- 특정 컴파일러 전용 코드
- 디버깅/분석 목적 코드
- 내부 도구 코드

이런 곳에서는 union type punning이 관행적으로 쓰이기도 한다.

이유는 다음과 같다.

- 짧다
- 의도가 직관적이다
- 실제 환경이 고정되어 있다
- 표현 확인이 목적이다

즉 통제된 환경에서는 실용적일 수 있다.

---

### 6.2 고이식성 라이브러리 코드에서는 보수적으로 보는 편이 낫다

반면 다음 같은 경우는 다르다.

- 여러 컴파일러 지원
- 여러 아키텍처 지원
- 장기 유지보수
- strict warning / aggressive optimization 환경
- 바이너리 포맷/직렬화 핵심 코드

이런 곳에서는 union보다 `memcpy` 경로가 더 안전한 기준점이 된다.

이유는 union이 언제나 틀려서가 아니라,  
**의미를 둘러싼 기대 차이를 줄이기 위해서**다.

즉 고이식성 코드는 "관행적으로 되더라"보다  
"보수적으로도 설명 가능한가"를 더 중시해야 한다.

---

### 6.3 union은 endianness, padding, 표현 차이를 해결하지 않는다

예를 들어 float와 int가 같은 크기라고 해도  
다음 문제는 여전히 남는다.

- float 표현 방식
- 정수 크기
- endianness
- trap representation 가능성
- 플랫폼별 ABI 차이

즉 union은 "겹친 저장공간 제공" 도구일 뿐,  
바이너리 해석의 모든 문제를 해결하는 도구가 아니다.

이 점은 `memcpy`와 마찬가지다.

---

## 7. `memcpy`와 비교하면 무엇이 다른가

### 7.1 union은 같은 저장공간을 다른 멤버 창으로 본다

즉 재해석의 성격이 더 직접적이다.

### 7.2 `memcpy`는 별도 객체로 바이트를 옮긴다

즉 같은 표현을 다루더라도 객체 모델을 더 보수적으로 유지한다.

### 7.3 union은 짧고 직관적이지만, 의미 해석 경계가 더 민감하다

### 7.4 `memcpy`는 다소 장황하지만, 이식성과 설명 가능성이 더 높다

그래서 실무 기준으로는 보통 이렇게 정리할 수 있다.

- 표현 확인/통제된 환경: union 가능
- 범용 라이브러리/엄격 이식성: `memcpy` 우선

즉 둘 중 누가 절대적으로 우월하냐보다  
**요구되는 보수성 수준이 다르다**고 보는 편이 맞다.

---

## 8. 자주 하는 오해

### 오해 1. union은 type punning을 위해 만들어진 공식 정답 도구다

반은 맞고 반은 위험하다.  
union이 저장공간 공유를 제공하는 것은 맞지만, 모든 재해석 상황에 대한 만능 정답으로 보면 곤란하다.

### 오해 2. union이면 strict aliasing 문제가 완전히 사라진다

그렇게 단순화하면 안 된다.  
union은 포인터 캐스팅과 구조가 다르지만, 그렇다고 모든 의미 문제가 증발하는 것은 아니다.

### 오해 3. union이 잘 동작하면 이식성도 확보된 것이다

아니다.  
현재 컴파일러/현재 플랫폼에서 원하는 결과가 나온다는 것과, 넓은 이식성이 있다는 것은 다르다.

### 오해 4. union은 값 변환을 수행한다

아니다.  
산술 변환이 아니라 객체 표현을 다른 타입 창으로 해석하는 것이다.

### 오해 5. union을 쓰면 endianness 문제도 함께 해결된다

아니다.  
union은 바이트 배열을 어떻게 볼지에 관여할 수는 있어도, 바이트 순서 차이 자체를 자동 교정하지는 않는다.

## 9. 실무에서 중요한 판단 기준

### 9.1 목적이 "표현 관찰"인지 "이식성 높은 변환"인지 먼저 구분한다

- 디버깅용 비트 관찰
- 플랫폼 고정 내부 코드
- 범용 라이브러리
- 네트워크/파일 포맷 처리

이 목적들은 같은 재해석처럼 보여도 요구 조건이 다르다.

### 9.2 범용 코드일수록 `memcpy` 기준으로 생각하는 편이 안전하다

이건 과하게 보수적인 습관이 아니라,  
설명 가능성과 유지보수성을 높이는 습관이다.

### 9.3 union은 "표현을 겹쳐 보는 구조"이지, 이식성 마법이 아니다

즉 union을 쓸 때는 항상 함께 물어야 한다.

- 이 코드가 어느 컴파일러를 전제로 하나
- 어느 아키텍처를 전제로 하나
- endianness/표현 차이까지 고려했나
- 정말 union이 필요한가, 아니면 `memcpy`가 더 나은가

### 9.4 디버그 편의성과 라이브러리 품질 기준은 다를 수 있다

내부 실험 코드와 공개 라이브러리 코드는 같은 기준으로 쓰지 않는 편이 좋다.

## 정리

union 기반 재해석은 같은 저장공간을 여러 타입 멤버가 공유한다는 union의 성질을 이용해,  
한 멤버에 쓴 바이트 표현을 다른 멤버 타입으로 읽는 기법이다.

핵심은 다음과 같다.

- union은 저장공간 공유를 제공한다
- 그래서 type punning에 자주 쓰인다
- 하지만 "겹친 메모리"와 "언어적으로 넓게 보장된다"는 같은 말이 아니다
- 통제된 환경에서는 실용적일 수 있다
- 고이식성 코드에서는 `memcpy`가 더 보수적이고 설명 가능한 기준점이 된다
- union은 endianness, 표현 차이, padding 문제를 자동 해결하지 않는다

즉 union 기반 재해석은  
"절대 금지"도 아니고 "무조건 정답"도 아니다.  
정확히는 **어느 수준의 이식성과 어떤 환경 가정을 두는가에 따라 신뢰 범위를 조절해야 하는 기법**이다.

이 관점이 잡혀야 다음의 strict aliasing, endianness, 직렬화 문서들이 더 정확하게 연결된다.
