## 1. 왜 이 주제를 깊게 봐야 하는가

C를 오래 사용하다 보면 “atomic이 없던 시절에도 멀티스레드 코드는 있었는데, 그럼 그 코드는 도대체 어떤 의미 위에서 돌아간 것인가”라는 질문을 피할 수 없다.  
이 질문은 단순한 역사 이야기가 아니다. 오늘날에도 많은 레거시 C 코드, 드라이버 코드, 임베디드 코드, 오래된 서버 코드, POSIX thread 기반 코드가 사실상 “atomic 이전의 사고방식” 위에서 유지되고 있기 때문이다.

문제는 이런 코드가 흔히 다음 직관 위에서 작성되었다는 점이다.

- 공유 변수는 그냥 메모리에 있으니 서로 볼 수 있을 것이다.
- `volatile`만 붙이면 다른 실행 흐름이 바꾼 값을 읽을 수 있을 것이다.
- x86에서는 잘 돌아가니 대체로 안전할 것이다.
- 락 없이도 int 하나 정도는 대충 안전할 것이다.
- 경쟁이 드물면 구조적으로도 문제 없다고 봐도 된다.

이 직관들은 일부 환경에서는 겉보기로 맞아 보인다.  
하지만 그것을 “언어가 보장하는 의미”로 오해하면, 이후의 `race condition`, `thread-safe`, `reentrant`, `signal-safe`, `memory ordering`, `atomics`를 모두 잘못 이해하게 된다.

이 주제를 깊게 봐야 하는 이유는 세 가지다.

첫째, “동시성 문제”를 언어 차원과 시스템 차원으로 분리하는 훈련이 필요하다.  
레거시 C 코드는 자주 운영체제 스레드, 컴파일러 관행, CPU ordering, 우연히 강한 하드웨어 모델 위에서 돌아간다. 하지만 그것이 곧바로 C 언어 차원의 명확한 동시성 의미론을 뜻하지는 않는다.

둘째, `volatile`의 한계를 정확히 이해하려면 그 앞뒤 문맥이 필요하다.  
왜 과거에는 `volatile`이 동시성 문제 해결책처럼 오용되었는지, 왜 그것이 접근 보존과 동기화를 혼동한 것인지, 어떤 부분이 우연한 구현 의존인지 봐야 한다.

셋째, 실무 판단 기준은 “옛날에도 이렇게 했다”가 아니라 “지금 이 코드가 무엇을 보장하고 무엇에 기대고 있는가”여야 한다.  
레거시 코드를 유지보수할 때 가장 위험한 순간은, 우연히 동작하던 코드를 “원래 안전한 코드”로 착각하는 순간이다.

이 문서의 핵심 목표는 atomic 도입 이전의 C 동시성 코드를 미화하는 데 있지 않다.  
핵심은 다음을 분리해서 보는 데 있다.

- 언어가 명시적으로 다루던 것과 그렇지 않던 것
- 운영체제 스레드 라이브러리가 사실상 제공하던 규율
- CPU와 컴파일러가 우연히 만들어 준 안정성 착시
- 실무 관례로 정착했지만 이식성과 엄밀성이 약한 패턴
- 현대 기준에서 반드시 atomic 또는 synchronization primitive로 바꿔야 하는 부분

즉, 이 문서는 “예전 코드는 다 틀렸다”라고 단순화하지 않는다.  
대신 “예전 코드는 무엇에 기대어 성립했는가”를 해부한다.

## 2. 전체 구조

atomic 이전의 C 동시성 문제를 이해하려면 먼저 문제를 다섯 층위로 나눠야 한다.

### 2.1 언어 규칙 층위

초기의 C는 오늘날 우리가 기대하는 형태의 정교한 멀티스레드 메모리 모델을 중심에 둔 언어가 아니었다.  
즉, 동시성은 원래부터 언어의 핵심 추상화라기보다, 운영체제와 런타임이 더 크게 관여하는 영역이었다.

이 말은 “옛날 C에는 동시성이 없었다”는 뜻이 아니라,  
동시성 correctness를 언어 차원에서 지금처럼 표준화해 설명하지 않았다는 뜻에 가깝다.

### 2.2 컴파일러 층위

컴파일러는 본질적으로 단일 실행 흐름 의미론을 바탕으로 최적화한다.  
공유 변수에 대한 재읽기, store ordering, load hoisting, dead store 제거 문제는, 동시성 의미론이 약할수록 “컴파일러가 왜 그런 최적화를 하면 안 되는가”를 소스만 보고 알기 어려워진다.

그래서 과거에는 `volatile`, 함수 호출, 락 함수, 외부 함수 호출, inline assembly 등이 사실상의 경계 역할을 하는 경우가 많았다.

### 2.3 운영체제 / 스레드 라이브러리 층위

실제 동시성은 대개 pthread 같은 스레드 라이브러리를 통해 들어왔다.  
즉, 프로그램은 “언어가 제공하는 동시성 모델”보다 “OS 스레드 API가 제공하는 synchronization contract” 위에서 동작했다.

실무적으로는 이 층위가 매우 중요했다.

- mutex lock/unlock
- condition variable wait/signal
- semaphore
- thread creation/join

이들은 단지 API가 아니라, 사실상 동시성 질서를 만드는 경계였다.

### 2.4 CPU / 하드웨어 층위

어떤 CPU는 상대적으로 강한 memory ordering을 보이고, 어떤 CPU는 더 약한 ordering을 보인다.  
과거 코드가 어떤 플랫폼에서는 “문제 없이” 돌아간 이유 중 상당수는, CPU가 충분히 보수적이었기 때문이다.

즉, 많은 레거시 동시성 코드는 “언어 정의”가 아니라 “하드웨어가 생각보다 얌전했다”는 사실에 기대어 성립했다.

### 2.5 실무 관례 층위

실무에서는 아래와 같은 패턴이 자주 사용되었다.

- 단순 플래그를 `volatile`로 선언
- 공유 int를 락 없이 읽고 쓰기
- 락을 잡으면 그 안의 모든 접근은 안전하다고 가정
- OS synchronization API가 컴파일러와 CPU 재주문까지 모두 막아 줄 것이라 기대
- 특정 플랫폼에서 관찰된 동작을 일반 규칙처럼 취급

이 관례들 중 일부는 지금도 맞는 방향의 근사치이지만, 일부는 착시다.  
따라서 atomic 이전의 코드를 볼 때는 “틀렸나/맞았나”보다 “무엇에 기대고 있나”를 먼저 봐야 한다.

## 3. 내부 동작 원리

### 3.1 atomic 이전의 핵심 문제는 “공유 상태를 언어가 얼마나 직접 설명하느냐”였다

단일 스레드 C에서는 다음과 같은 직관이 자연스럽다.

- 변수는 메모리에 저장된다.
- 쓰면 다음에 읽을 때 그 값이 나온다.
- 소스 순서대로 쓰면 대체로 그 순서대로 의미가 전달된다.

하지만 동시성이 들어오면 바로 질문이 바뀐다.

- 어떤 쓰기를 다른 실행 흐름이 언제 보는가
- 두 흐름이 동시에 같은 객체를 만지면 무엇이 의미상 허용되는가
- load/store가 컴파일러나 CPU에 의해 재배치되면 어떤 경계에서 막히는가
- read-modify-write가 쪼개져도 되는가
- 한 변수의 쓰기가 다른 변수의 관찰 순서와 어떤 관계를 가지는가

atomic 이전의 C 코드는 이런 질문을 언어 차원에서 엄밀하게 표현하기보다, 외부 synchronization mechanism에 의존해 처리하는 경우가 많았다.  
즉, 동시성 correctness는 종종 “언어 안”이 아니라 “언어 바깥 API 계약”으로 밀려났다.

### 3.2 왜 `volatile`이 동시성 도구처럼 오해되었는가

다음 코드를 보자.
```c
    volatile int ready = 0;

    void worker(void) {
        while (!ready) {
        }
    }
```
이 코드는 겉보기에는 그럴듯하다.  
`ready`를 반복해서 다시 읽으니, 다른 흐름이 `ready = 1;`을 쓰면 루프가 풀릴 것처럼 보인다.

이런 패턴이 과거에 널리 쓰인 이유는 단순하다.

- 컴파일러가 값을 레지스터에 고정하지 못하게 만들고 싶었다.
- “외부에서 값이 바뀔 수 있다”는 사실을 표현하고 싶었다.
- 테스트에서 실제로 잘 동작하는 경우가 많았다.

하지만 이 패턴은 접근 보존과 동기화를 혼동한다.

`volatile`이 주는 것은 대략 다음과 같은 성질이다.

- 이 접근을 지우지 마라
- 이 접근을 일반 객체 접근처럼 함부로 합치지 마라

하지만 그것이 다음을 자동으로 주는 것은 아니다.

- 원자성
- publish/consume ordering
- 다른 공유 데이터의 가시성 보장
- happens-before 관계

즉, 과거 `volatile`이 동시성 도구처럼 쓰였던 이유는, 언어 차원의 동기화 추상화가 약한 상황에서 컴파일러 최적화 억제 장치가 “동기화 비슷한 것”처럼 보였기 때문이다.

### 3.3 락은 왜 atomic 이전 코드의 핵심 경계였는가

atomic 이전에도 많은 코드가 제대로 동작했다.  
그 이유는 락이 있었기 때문이다.

예를 들어 다음을 보자.
```c
    pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
    int shared = 0;

    void set_value(void) {
        pthread_mutex_lock(&m);
        shared = 42;
        pthread_mutex_unlock(&m);
    }

    int get_value(void) {
        int v;
        pthread_mutex_lock(&m);
        v = shared;
        pthread_mutex_unlock(&m);
        return v;
    }
```
이 코드의 핵심은 `shared`가 특별한 타입이기 때문이 아니다.  
핵심은 모든 접근이 동일한 synchronization discipline 아래에 있다는 점이다.

과거 C 동시성 코드의 중요한 원리는 이것이었다.

- 공유 상태 그 자체가 안전한 것이 아니라
- 공유 상태를 둘러싼 접근 규율이 안전성을 만든다.

즉, atomic 이전 코드의 correctness는 자주 “변수의 성질”이 아니라 “프로토콜의 성질”에 달려 있었다.

### 3.4 왜 “락 없는 단순 읽기/쓰기 정도는 괜찮다”는 착시가 생겼는가

다음 코드는 레거시 코드에서 매우 흔하다.
```c
    int flag = 0;

    void producer(void) {
        flag = 1;
    }

    void consumer(void) {
        while (flag == 0) {
        }
    }
```
어떤 플랫폼에서는 이것이 잘 동작한다.  
특히 디버그 빌드, 낮은 최적화, x86 계열, 단순 테스트 환경에서는 “문제 없다”는 인상을 준다.

왜 이런 착시가 생기는가?

- 컴파일러가 우연히 매번 메모리에서 다시 읽는다.
- CPU ordering이 상대적으로 강하다.
- 경쟁 타이밍이 단순하다.
- 테스트 시간이 짧고 부하가 낮다.
- 캐시 coherence가 기대한 대로 보인다.

하지만 이 코드는 구조적으로 취약하다.  
왜냐하면 correctness가 명시적 synchronization이 아니라 우연한 환경 성질에 기대고 있기 때문이다.

### 3.5 과거의 동시성 코드는 흔히 “언어 + OS + CPU + 관례”의 합성물이었다

atomic 이전의 C 동시성 코드를 한 문장으로 요약하면 이렇다.

- 언어는 충분히 엄밀한 동시성 의미론을 제공하지 않았고
- 운영체제는 스레드와 락 API를 제공했으며
- 컴파일러는 대체로 그 경계에서 보수적으로 동작했고
- 하드웨어는 특정 ordering 모델을 제공했고
- 실무자는 그 조합 위에 경험적으로 안전한 패턴을 쌓았다

즉, 과거 동시성 코드는 “순수한 C 의미론”만으로 읽으면 잘 안 보이고,  
반대로 “그냥 시스템이 알아서 맞춰 준다”라고 보면 과신하게 된다.

## 4. 핵심 구성 요소

### 4.1 공유 상태

동시성의 출발점은 공유 상태다.  
한 실행 흐름만 접근하는 값은 동시성 문제가 아니다.  
문제가 되는 것은 둘 이상의 실행 흐름이 같은 논리적 객체에 접근하는 순간이다.

공유 상태는 다음처럼 다양하다.

- 단순 플래그
- 카운터
- 큐 인덱스
- 연결 리스트 포인터
- 참조 카운트
- 상태 머신 필드
- 버퍼 내용과 길이 쌍

중요한 점은 “int 하나”냐 “큰 구조체”냐가 본질이 아니라는 것이다.  
작은 값도 synchronization 없이 공유되면 동시성 문제가 된다.

### 4.2 synchronization primitive

atomic 이전 코드에서 핵심은 synchronization primitive였다.

- mutex
- rwlock
- semaphore
- condition variable
- thread join
- signal mask 또는 특정 OS 이벤트 메커니즘

이들은 단순히 “경쟁을 줄이는 도구”가 아니라,  
공유 상태에 대해 어떤 접근이 언제 허용되는지 정하는 질서 생성 장치다.

### 4.3 compiler barrier 역할을 하던 경계들

과거 코드에서 다음 요소들은 자주 “컴파일러가 함부로 넘지 못하는 경계”처럼 사용되었다.

- 외부 함수 호출
- 락/언락 함수 호출
- inline assembly
- `volatile` 접근
- 시스템 콜

물론 이것들이 모두 동일한 의미를 갖는 것은 아니다.  
하지만 실무자는 종종 이런 요소들을 이용해 “여기서는 컴파일러가 대충 조심할 것”이라고 기대했다.

여기서 특히 조심해야 할 점은, 이 목록이 모두 같은 보장 수준을 주는 것은 아니라는 사실이다.

- 락/언락 함수는 단순 함수 호출이 아니라 synchronization contract의 일부일 수 있다.
- 외부 함수 호출이나 시스템 콜은 구현이 보수적으로 다룰 수는 있어도, 그것만으로 공유 상태 전체의 ordering이 설명되는 것은 아니다.
- `volatile` 접근은 access preservation에만 관여하지, synchronization 전체를 만들지 않는다.
- inline assembly도 barrier 역할을 하도록 어떻게 작성했는지에 따라 의미가 달라진다.

문제는 이 기대가 언어 차원에서 엄밀하지 않을 수 있다는 점이다.
즉, 과거 코드에서 “경계처럼 보였던 것들”은 같은 종류의 경계가 아니라, 서로 다른 층위의 제약이 우연히 비슷하게 쓰였던 경우가 많다.

### 4.4 CPU memory ordering

과거 코드는 하드웨어 ordering에 크게 기대었다.  
특히 강한 ordering을 보이는 플랫폼에서는 일부 잘못된 코드도 겉보기로 안정적이었다.

하지만 약한 ordering 아키텍처로 가면 같은 코드가 깨질 수 있다.  
즉, 과거 동시성 코드의 portability 문제는 종종 여기서 드러난다.

### 4.5 운영체제 API 계약

많은 레거시 C 코드는 사실상 “pthread contract 문서” 위에서 읽어야 한다.  
예를 들어 mutex lock/unlock은 단지 mutual exclusion만이 아니라, 공유 상태를 관찰하는 경계를 제공한다.

더 정확히 말하면, mutex는 “한 번에 한 실행 흐름만 들어간다”는 성질만 중요한 것이 아니다.  
실무적으로 더 중요한 것은 lock으로 들어가기 전과 unlock 이후 사이에 어떤 공유 상태를 읽고 쓸 수 있는지, 그리고 다른 실행 흐름이 그 상태를 어떤 질서로 관찰하는지를 프로토콜로 만든다는 점이다.

즉, atomic 이전 동시성의 실제 의미는 종종 C 표준서보다 OS thread library 문서에 더 많이 있었다.

## 5. 실행 흐름 또는 처리 순서

### 5.1 가장 단순한 오해: busy-wait flag

잘못된 기대:
```c
    int ready = 0;

    void producer(void) {
        ready = 1;
    }

    void consumer(void) {
        while (!ready) {
        }
    }
```
이 코드는 가장 흔한 레거시 패턴 중 하나다.  
문제는 다음과 같다.

- compiler가 `ready`를 루프 밖에서 한 번만 읽어도 이상하지 않다.
- 다른 데이터와의 ordering은 전혀 설명되지 않는다.
- 플랫폼에 따라 겉보기 동작이 달라질 수 있다.

보수적으로 보이게 만든 버전:
```c
    volatile int ready = 0;

    void producer(void) {
        ready = 1;
    }

    void consumer(void) {
        while (!ready) {
        }
    }
```
이 버전은 접근 보존 측면에서는 더 강해 보인다.  
하지만 이것도 “동기화 완성”은 아니다.  
그저 컴파일러가 `ready` 접근을 없애는 문제를 일부 제어할 뿐이다.

### 5.2 락으로 보호된 공유 상태

권장되는 구조:
```c
    pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
    int ready = 0;
    int payload = 0;

    void producer(void) {
        pthread_mutex_lock(&m);
        payload = 123;
        ready = 1;
        pthread_mutex_unlock(&m);
    }

    void consumer(void) {
        int local_payload = 0;

        for (;;) {
            pthread_mutex_lock(&m);
            if (ready) {
                local_payload = payload;
                pthread_mutex_unlock(&m);
                break;
            }
            pthread_mutex_unlock(&m);
        }

        use(local_payload);
    }
```
이 구조의 핵심은 단순하다.

- `ready`와 `payload`를 따로따로 믿지 않는다.
- 둘을 같은 synchronization discipline 아래 둔다.
- correctness를 변수 성질이 아니라 접근 프로토콜에 맡긴다.

즉, mutex가 만드는 것은 단순 상호배제가 아니라 “이 공유 상태는 lock을 기준으로 읽고 쓴다”는 관찰 질서다.  
atomic 이전 코드에서 안전한 부분은 대개 변수 자체가 아니라 이런 질서가 명시된 부분이었다.

### 5.3 락 없이 카운터 증가

잘못된 구조:
```c
    int counter = 0;

    void worker(void) {
        counter++;
    }
```
겉보기에는 단순하다.  
하지만 `counter++`는 read-modify-write이고, 여러 흐름이 동시에 실행하면 lost update가 발생할 수 있다.

`volatile`로 고친 척하는 구조:
```c
    volatile int counter = 0;

    void worker(void) {
        counter++;
    }
```
이 코드는 여전히 본질 문제가 남아 있다.  
접근은 남아도 연산은 원자적이지 않다.

### 5.4 double-checked 같은 최적화 패턴의 위험

atomic 이전 코드에서 자주 등장한 패턴은, 락 비용을 줄이기 위해 바깥에서 한 번 검사하고 필요할 때만 락을 잡는 식의 구조다.
```c
    if (!initialized) {
        lock();
        if (!initialized) {
            setup_object();
            initialized = 1;
        }
        unlock();
    }
```
이 패턴은 겉보기에는 효율적이지만, ordering과 publication의 의미가 엄밀하지 않으면 구조적으로 위험하다.  
특히 “객체 내부 필드 초기화”와 “initialized 플래그 공개” 사이의 관계를 명시적으로 만들지 않으면, 일부 환경에서는 깨질 수 있다.

과거에는 이런 코드가 특정 플랫폼에서 잘 돌았기 때문에 널리 퍼졌지만,  
현대 기준에서는 매우 조심해서 봐야 하는 패턴이다.

## 6. 성능 / 최적화 관점

### 6.1 atomic 이전의 성능 최적화는 자주 정확성과 충돌했다

과거 C 동시성 코드에서는 “락은 비싸다”는 직관이 매우 강했다.  
그 결과 다음과 같은 선택이 자주 나왔다.

- 가능하면 락을 줄이자
- 플래그 하나 정도는 그냥 읽자
- int 카운터쯤은 락 없이 하자
- 특정 CPU에서는 안전하니 괜찮다

문제는 이런 최적화가 correctness contract를 약하게 만들었다는 점이다.  
즉, 성능을 얻는 대신 보장 범위를 잃었다.

### 6.2 왜 특정 플랫폼에서는 틀린 코드가 더 빨라 보였는가

락 없는 코드는 당연히 빠르게 보일 수 있다.  
또 `volatile`만 붙인 busy-wait는 락보다 간단해 보인다.

하지만 이것은 자주 다음 비용을 숨긴다.

- 구조적 취약성
- 이식성 상실
- rare timing bug
- 고부하에서만 터지는 비정상 동작
- CPU 아키텍처 변경 시 붕괴
- 컴파일러 버전 변경 시 재현 양상 변화

즉, atomic 이전의 “빠른 동시성 코드” 중 상당수는  
빠른 것이 아니라 “보장 비용을 외부로 미룬 코드”였다.

### 6.3 락이 느리다는 직관도 단순화다

과거에는 락 자체를 지나치게 두려워해 잘못된 최적화를 하는 경우가 많았다.  
하지만 많은 경우 진짜 병목은 다음에 있었다.

- 너무 넓은 critical section
- 잘못된 자료구조
- 캐시 친화성 부족
- 과도한 contention
- wake-up / sleep 패턴 설계 실패
- false sharing

즉, atomic 이전 코드의 성능 문제를 “락 때문에 느리다”로만 보면 틀릴 수 있다.  
정확성 경계를 먼저 세우고, 그다음 병목을 측정해야 한다.

### 6.4 `volatile`은 성능 해법이 아니라 최적화 제한 장치다

과거에는 종종 `volatile`이 “가벼운 동기화”처럼 오해되었다.  
하지만 `volatile`은 일반적으로 다음을 제한한다.

- 레지스터 캐싱
- load/store 제거
- 일부 재배치

즉, 성능을 위해 쓰는 도구가 아니라, 외부 의미가 있는 접근을 보존하기 위해 쓰는 도구다.  
동시성 correctness를 위해 `volatile`을 쓰기 시작하면 성능도, 정확성도 함께 흐려진다.

## 7. 어디서부터 보장 범위와 구현 경계가 갈리는가

이 섹션이 이 문서의 핵심이다.  
atomic 이전의 C 동시성 문제는 “어디까지가 언어이고 어디부터가 시스템인가”를 분리하지 않으면 거의 반드시 오해하게 된다.

### 7.1 언어 규칙과 시스템 관례가 갈리는 지점

과거 C 코드가 스레드를 사용했다고 해서, 그 동시성 의미가 전부 언어 규칙으로 설명되는 것은 아니다.  
많은 경우 코드는 다음의 합성물이었다.

- C 언어의 일반 객체 접근 규칙
- OS 스레드 API의 synchronization contract
- 컴파일러의 관행적 보수성
- CPU ordering
- 실무자의 경험적 패턴

즉, “C 코드”로 보인다고 해서 “C 언어만으로 보장되는 코드”는 아니다.

### 7.2 `volatile`과 synchronization이 갈리는 지점

`volatile`은 접근 보존을 다루고, synchronization은 경쟁 환경의 정합성을 다룬다.  
과거 코드가 `volatile`에 기대었다면, 먼저 다음을 물어야 한다.

- 이 코드는 단지 재읽기를 원한 것인가
- 아니면 ordering과 visibility까지 기대한 것인가
- 공유 데이터 전체의 질서를 만들고 싶은 것인가
- 단지 플래그 하나의 접근 제거만 막고 싶은 것인가

이 질문에 뒤쪽이 포함되면, `volatile`은 거의 항상 불충분하다.

### 7.3 락 API와 언어 semantics가 갈리는 지점

mutex lock/unlock은 C 문법 요소가 아니다.  
그렇지만 실무 동시성에서는 바로 그 API 호출이 핵심 경계였다.

즉, 많은 레거시 코드의 의미는 “이 C 문장이 무엇을 뜻하나”보다  
“이 lock/unlock 구간이 어떤 프로토콜을 만들고 있나”로 읽어야 한다.

### 7.4 CPU ordering과 portability가 갈리는 지점

특정 CPU에서 잘 동작하는 lock-free 비슷한 코드가, 다른 아키텍처에서는 깨질 수 있다.  
이것은 매우 중요한 경계다.

- x86에서 잘 됨
- ARM에서 가끔 깨짐
- 최적화 레벨 바꾸면 깨짐
- 컴파일러 버전 바꾸면 재현 양상 변화

이런 현상이 보이면, 코드는 대개 언어 차원의 명시적 계약보다 하드웨어/구현 관례에 기대고 있을 가능성이 높다.

### 7.5 “작동했다”와 “보장된다”가 갈리는 지점

atomic 이전 코드에서 가장 위험한 문장은 이것이다.

- 우리 환경에서는 잘 돌아간다.

이 문장은 관찰 사실일 수는 있다.  
하지만 보장 문장은 아니다.

실무에서는 반드시 다음처럼 바꿔 읽어야 한다.

- 왜 돌아가는가
- 무엇이 그 동작을 떠받치고 있는가
- 그 전제가 바뀌면 어떻게 깨지는가
- 이 코드를 다른 플랫폼이나 다른 컴파일러로 옮겨도 같은가

즉, atomic 이전 동시성 코드는 “관찰된 정상 동작”과 “명시적 보장”을 분리해서 읽어야 한다.

## 8. 실무에서 중요한 판단 기준

### 8.1 레거시 코드를 볼 때 먼저 확인할 것

atomic 이전 스타일의 코드를 보면 먼저 다음을 확인해야 한다.

- 공유 상태가 무엇인가
- 누가 그 상태를 읽고 쓰는가
- 같은 락이 모든 접근을 감싸는가
- 일부 접근만 락 밖으로 빠져나가 있는가
- `volatile`이 붙어 있다면 그 목적이 무엇인가
- correctness가 락에 기대는가, 하드웨어 ordering에 기대는가, 우연한 테스트 환경에 기대는가

### 8.2 안전한 레거시 코드의 특징

레거시 코드라도 다음 특징을 가지면 상대적으로 읽기 쉽다.

- 공유 상태마다 명확한 owner 또는 보호 락이 있다
- 락 규율이 문서화되어 있다
- 플래그와 payload가 같은 synchronization discipline 안에 있다
- `volatile`은 MMIO나 signal 같은 좁은 경계에서만 쓰인다
- 락 없는 접근은 이유와 범위가 명확하다
- portability 전제가 문서화되어 있다

### 8.3 위험한 레거시 코드의 특징

다음 특징이 보이면 구조적으로 위험할 가능성이 높다.

- `volatile`이 동기화 대용으로 넓게 쓰인다
- shared int를 락 없이 읽고 쓰는데 “대부분 괜찮다”라고 설명한다
- 특정 CPU에서 잘 되는 것을 일반 규칙처럼 말한다
- publish flag 패턴이 ordering 설명 없이 퍼져 있다
- initialization / publication / consumption 경계가 모호하다
- 데이터 구조 일부만 락으로 보호된다
- “rare bug”가 있는데 재현이 어렵다고만 말한다

### 8.4 현대 기준으로 어떻게 해석해야 하는가

레거시 atomic 이전 코드를 지금 읽을 때의 원칙은 단순하다.

- 원래 작성자의 의도를 추측한다.
- 현재 코드가 어떤 전제에 기대는지 분해한다.
- 우연한 구현 의존을 명시적 synchronization으로 치환할 수 있는지 본다.
- `volatile`이 synchronization 대용이면 거의 반드시 재검토한다.
- 락 기반 프로토콜이라면 보호 범위와 invariants를 다시 문서화한다.

즉, 과거 코드를 지금의 눈으로 무조건 비난하는 것이 아니라,  
현재 보장 언어로 다시 번역하는 것이 중요하다.

---

## 9. 판단 체크리스트

다음 질문에 답하지 못하면, 그 동시성 코드는 아직 제대로 이해한 것이 아니다.

- 이 코드의 공유 상태는 정확히 무엇인가
- 각 공유 상태는 어떤 synchronization discipline 아래 있는가
- 락이 보호하는 범위와 락 밖 접근의 범위는 어디서 갈리는가
- `volatile`이 있다면 access preservation 때문인가, synchronization 기대 때문인가
- flag와 payload가 분리되어 있다면 ordering은 어떻게 설명되는가
- correctness가 OS API 계약에 기대는가, CPU ordering에 기대는가, 관찰된 우연에 기대는가
- 다른 컴파일러, 다른 최적화, 다른 CPU에서도 같은가
- read-modify-write가 원자적이라고 암묵적으로 가정하고 있지 않은가
- “테스트에서는 된다”를 “보장된다”로 착각하고 있지 않은가
- 현대 기준에서 `_Atomic`, mutex, condition variable, semaphore 등으로 더 명확히 표현할 수 있는가

## 정리

atomic 이전의 C 동시성 코드는 대체로 “언어만으로 완결된 동시성 모델” 위에 있지 않았다.  
많은 코드는 다음의 조합 위에서 동작했다.

- 운영체제 스레드 API
- 락과 condition variable 같은 synchronization primitive
- 컴파일러의 관행적 보수성
- 특정 CPU의 memory ordering
- 실무 관례와 경험적 패턴

따라서 과거 코드를 이해할 때 핵심은 “왜 그 코드가 그 환경에서 돌아갔는가”를 분해하는 것이다.

특히 반드시 기억해야 할 점은 다음이다.

- `volatile`은 synchronization 자체가 아니다.
- 락 없는 공유 상태는 우연히 동작해도 구조적으로 취약할 수 있다.
- correctness는 변수 타입보다 접근 프로토콜에 더 크게 달려 있다.
- “작동했다”는 관찰 사실이지, 곧바로 보장 문장은 아니다.
- 현대 기준에서는 atomic 또는 명시적 synchronization으로 의도를 드러내는 편이 훨씬 안전하다.

즉, atomic 이전의 C 동시성은 “없었다”가 아니라,  
“언어 바깥 계약과 구현 관례에 크게 기대고 있었다”라고 이해하는 편이 정확하다.

## 10. 더 깊게 볼 포인트

### 10.1 race condition은 왜 결과를 예측 불가능하게 만드는가

이 문서 다음 주제로 가장 자연스럽다.  
이 문서가 “atomic 이전의 코드가 무엇에 기대고 있었는가”를 다뤘다면, 다음 문서는 그 기대가 깨질 때 왜 결과가 비결정적으로 변하는지를 더 직접적으로 다룬다.

### 10.2 reentrant와 thread-safe는 무엇이 다른가

과거 코드에서는 signal-safe, reentrant, thread-safe가 자주 섞여 이해되었다.  
동시성 경계를 더 명확히 하려면 이 셋을 분리해서 봐야 한다.

### 10.3 `volatile`과 `_Atomic`은 어떤 문제를 각각 다루는가

`volatile`의 경계를 본 뒤에는, `_Atomic`이 어떤 문제를 해결 대상으로 삼는지 비교해 보는 것이 좋다.  
둘은 비슷한 “특별한 타입 한정”처럼 보이지만 해결하는 문제가 전혀 다르다.

### 10.4 mutex가 만드는 질서는 정확히 무엇인가

실무적으로는 lock/unlock이 공유 상태의 의미를 만드는 핵심 경계다.  
mutex를 단순한 “상호배제 장치”가 아니라 “공유 상태 관찰 질서 생성기”로 이해하면 코드 해석력이 올라간다.

### 10.5 특정 플랫폼에서만 잘 도는 lock-free 비슷한 코드는 왜 위험한가

이 주제는 CPU ordering, compiler reordering, portability 문제와 직접 연결된다.  
과거 코드 유지보수에서 특히 중요하다.
