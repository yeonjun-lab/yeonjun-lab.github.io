## 1. 왜 이 주제를 깊게 봐야 하는가

C에서 정수형은 단순히 “숫자를 담는 타입”이 아니다.  
정수형은 실제로는 **메모리 안에 저장되는 비트 패턴을 어떤 방식으로 해석할 것인가**를 결정하는 규칙이다.

초보 단계에서는 보통 이렇게 배운다.

- `char`
- `short`
- `int`
- `long`

그리고 각각이 숫자를 담는다고 이해한다.  
이 설명 자체가 틀린 것은 아니다.  
하지만 이 수준에서 멈추면 C에서 자주 등장하는 중요한 현상을 설명하지 못한다.

예를 들면 다음과 같은 문제다.

- 왜 signed와 unsigned를 섞으면 이상한 결과가 나오는가?
- 왜 음수는 비트로 저장할 수 있는가?
- 왜 overflow는 위험한가?
- 왜 같은 메모리 값인데 해석 결과가 달라질 수 있는가?
- 왜 작은 타입과 큰 타입을 섞으면 예상과 다른 결과가 나오는가?

이 문제들은 문법 문제가 아니라 **표현 방식** 문제다.  
즉, 정수를 제대로 이해하려면 “숫자”라는 추상 개념이 아니라 **비트, 바이트, 저장, 해석**이라는 실행 관점으로 내려가야 한다.

특히 C는 이런 표현 차이가 코드 의미와 성능, 버그에 직접 연결되는 언어다.  
그래서 정수 메모리 표현은 선택 과목이 아니라 필수 기반이다.

이 문서는 정수가 메모리에 어떻게 저장되는지, signed와 unsigned는 실제로 무엇이 다른지, 그리고 이 차이가 왜 실무에서 중요한지 깊게 정리하기 위해 작성한다.

## 2. 전체 구조

정수를 메모리에 저장하는 과정을 큰 구조로 보면 다음과 같다.

### 2.1 정수는 결국 비트 패턴이다

메모리에는 숫자 자체가 저장되지 않는다.  
메모리에는 오직 비트들이 저장된다.

예를 들어 1바이트 공간이라면 8개의 비트가 있고, 그 안에는 다음 같은 패턴이 들어갈 수 있다.

```
00000000
00000001
01111111
11111111
```

이 비트 패턴 자체는 그냥 저장된 상태일 뿐이다.  
이것을 **정수로 어떻게 해석할 것인지**는 타입과 규칙이 정한다.

### 2.2 타입은 크기와 해석 방식을 함께 결정한다

C에서 정수 타입은 보통 다음 두 가지를 결정한다.

- 얼마나 많은 바이트를 사용할 것인가
- 그 비트 패턴을 signed로 볼 것인가, unsigned로 볼 것인가

즉, 타입은 단순한 문법 라벨이 아니라 저장과 해석의 규칙이다.

### 2.3 signed와 unsigned는 “다른 해석 체계”다

같은 비트 패턴이라도 signed로 볼지 unsigned로 볼지에 따라 의미가 달라질 수 있다.

예를 들어 8비트 패턴 `11111111`을 생각해보자.

- unsigned 8비트 정수로 보면 255다.
- signed 8비트 정수로 보면 보통 -1이다.

즉, 메모리에 저장된 패턴은 같아도, 타입이 다르면 결과가 다르다.

### 2.4 음수 표현에는 보통 2의 보수가 사용된다

현대 시스템 대부분에서는 signed 정수를 2의 보수(two’s complement) 방식으로 표현한다.  
이 방식 덕분에 덧셈과 뺄셈을 하드웨어 수준에서 효율적으로 처리할 수 있다.

즉, 음수를 저장하는 방식은 별도의 특수 저장이 아니라, **같은 비트 공간을 다른 규칙으로 해석하는 것**에 가깝다.

## 3. 내부 동작 원리

### 3.1 양수는 비교적 직관적으로 저장된다

양수는 보통 우리가 생각하는 이진수 형태로 저장된다고 보면 된다.

예를 들어 8비트에서 정수 5는 다음처럼 표현할 수 있다.

```
00000101
```

정수 10은 다음처럼 표현된다.

```
00001010
```

즉, 양수는 비트 단위로 값을 분해한 표현이라고 이해하면 비교적 쉽다.

### 3.2 unsigned 정수는 모든 비트를 크기 표현에 사용한다

unsigned 정수는 부호를 위한 특별한 비트를 두지 않는다.  
즉, 가진 비트를 모두 숫자 크기 표현에 사용한다.

예를 들어 8비트 unsigned 정수는 0부터 255까지 표현 가능하다.

- 최소값: 0
- 최대값: 255

왜냐하면 8비트로 표현 가능한 경우의 수는 256개이기 때문이다.

즉, n비트 unsigned 정수의 범위는 일반적으로 다음과 같다.

- 0 ~ 2^n - 1

### 3.3 signed 정수는 음수와 양수를 함께 표현해야 한다

signed 정수는 양수만이 아니라 음수도 표현해야 하므로, 전체 비트 공간을 unsigned와 똑같이 사용할 수 없다.

현대 시스템 대부분은 2의 보수 방식을 쓴다.  
이 방식에서는 최상위 비트가 결과적으로 부호와 관련된 역할을 하게 된다.

예를 들어 8비트 signed 정수의 범위는 보통 다음과 같다.

- -128 ~ 127

즉, unsigned보다 양수 최대값은 줄어들지만 음수도 표현할 수 있다.

### 3.4 2의 보수는 음수를 효율적으로 표현한다

2의 보수 방식은 음수를 표현하는 대표적인 방법이다.

예를 들어 8비트에서 `1`은 다음과 같다.

```
00000001
```

이때 `-1`은 다음과 같이 표현된다.

```
11111111
```

그리고 `-2`는 다음과 같이 표현된다.

```
11111110
```

이 방식의 중요한 장점은, 하드웨어가 양수와 음수 연산을 통일된 회로로 처리하기 쉽다는 점이다.

예를 들어 `1 + (-1)`을 8비트로 보면,

```
00000001
11111111
----------
00000000
```

가 된다.

즉, 2의 보수는 단순히 음수를 억지로 저장하는 방식이 아니라, **연산을 자연스럽게 통합하기 위한 표현 방식**이다.

### 3.5 같은 비트 패턴도 해석 방식에 따라 결과가 달라진다

다음 비트 패턴을 보자.

```
11111111
```

이 값을 unsigned 8비트 정수로 해석하면 255다.  
하지만 signed 8비트 정수로 해석하면 -1이다.

즉, 중요한 것은 저장된 비트 자체만이 아니라, **그 비트를 어떤 타입으로 읽느냐**다.

이 점은 C에서 특히 중요하다.  
왜냐하면 C는 캐스팅, 형변환, 비트 연산, 포인터 재해석이 가능한 언어이기 때문이다.

### 3.6 오버플로우는 표현 범위를 넘는 현상이다

정수는 무한한 수를 저장하지 못한다.  
정해진 비트 수 안에 들어갈 수 있는 범위만 표현한다.

예를 들어 8비트 unsigned 정수에서 최대값은 255다.

만약 여기에 1을 더하면 어떻게 될까?

```
11111111   // 255
00000001   // 1
----------
00000000
```

즉, 다시 0으로 돌아간다.

이것이 unsigned에서의 wrap-around 현상이다.

signed에서는 문제가 더 복잡하다.  
signed overflow는 단순히 “한 바퀴 돌아간다”고만 생각하면 안 된다.  
C에서는 signed overflow가 위험한 의미를 가질 수 있으며, 안전한 코드 작성 시 반드시 피해야 한다.

## 4. 핵심 구성 요소

### 4.1 `char`, `short`, `int`, `long`

이들은 서로 다른 크기를 가질 수 있는 정수형들이다.  
정확한 바이트 수는 구현과 플랫폼에 따라 달라질 수 있다.

중요한 것은 이름 자체보다 다음이다.

- 각 타입은 정해진 크기 범위를 가진다.
- 크기가 다르면 표현 가능한 숫자 범위도 달라진다.
- 타입이 다르면 연산과 형변환 결과도 달라질 수 있다.

### 4.2 signed

signed 정수형은 양수와 음수를 함께 표현한다.  
보통 2의 보수 표현을 사용한다.

즉, signed는 “음수를 표현할 수 있는 정수 해석 방식”이라고 이해하면 된다.

### 4.3 unsigned

unsigned 정수형은 음수를 표현하지 않는다.  
대신 모든 비트를 양수 범위 표현에 사용한다.

즉, unsigned는 “부호 없이 크기만 표현하는 정수 해석 방식”이다.

### 4.4 binary representation

binary representation은 정수를 실제 비트 패턴으로 표현하는 방식이다.  
C에서 숫자를 이해한다는 것은 결국 이 비트 표현을 의식한다는 뜻이다.

### 4.5 overflow

overflow는 표현 가능한 범위를 넘는 연산 결과가 발생하는 현상이다.  
이것은 단순한 계산 실수가 아니라, **저장 가능한 비트 공간의 한계** 때문에 생긴다.

### 4.6 integer conversion

서로 다른 정수 타입끼리 연산하면 내부적으로 변환 규칙이 작동한다.  
이 규칙을 모르고 있으면 “왜 결과가 이렇게 나오지?”라는 상황을 자주 만나게 된다.

## 5. 실행 흐름 또는 처리 순서

정수 저장과 해석을 실행 흐름 관점으로 보면 다음처럼 볼 수 있다.

### 5.1 선언

먼저 정수형 변수를 선언한다.

```c
int x;
```

이 시점에는 “정수형 데이터를 담을 수 있는 공간을 확보한다”는 의미가 있다.

### 5.2 초기화 또는 대입

그 다음 값을 넣는다.

```c
int x = 10;
```

이 값 10은 메모리에 비트 패턴으로 저장된다.  
즉, 사람이 보는 “10”이라는 기호 그대로 저장되는 것이 아니다.

### 5.3 타입에 따른 해석

프로그램이 `x`를 읽을 때는, `x`의 타입 규칙에 따라 그 비트 패턴을 정수값으로 해석한다.

즉, “메모리 → 비트 패턴 → 타입 규칙 → 정수값” 순서로 이해할 수 있다.

### 5.4 연산 수행

정수끼리 연산하면 CPU와 컴파일러는 해당 타입 규칙을 고려해 계산을 수행한다.

```c
int a = 100;
int b = 20;
int c = a + b;
```

이때 중요한 것은 단순히 수학적으로 맞는가가 아니라,  
그 결과가 해당 타입의 표현 범위 안에 들어가는가도 함께 봐야 한다.

### 5.5 범위 초과 여부

만약 표현 가능한 범위를 초과하면 overflow 문제가 생긴다.

예를 들어 작은 범위 타입, signed/unsigned 혼합, 큰 수 연산 등에서는 여기서 오류나 예상치 못한 결과가 나올 수 있다.

## 6. 성능 / 최적화 관점

### 6.1 타입 크기는 메모리 사용량에 직접 영향을 준다

정수 타입이 클수록 더 많은 메모리를 사용한다.  
데이터가 아주 많을 때는 이 차이가 전체 메모리 사용량에 큰 영향을 줄 수 있다.

예를 들어 수천만 개의 값을 저장할 때는 `int`와 `long long`의 차이가 작지 않다.

즉, 타입 선택은 단순 취향이 아니라 **데이터 규모와 범위 요구사항**을 같이 봐야 한다.

### 6.2 작은 타입이 항상 더 빠른 것은 아니다

처음 배우는 사람은 “작은 타입이 무조건 더 빠르다”고 생각하기 쉽다.  
하지만 실제로는 꼭 그렇지 않다.

현대 CPU는 특정 기본 정수 크기를 기준으로 더 효율적으로 동작하는 경우가 많다.  
즉, 메모리를 아끼는 것과 연산이 빠른 것은 항상 같은 방향이 아니다.

따라서 타입은 다음을 함께 고려해야 한다.

- 표현 범위
- 메모리 사용량
- 정렬과 패딩
- CPU의 기본 처리 크기

### 6.3 signed/unsigned 혼합은 버그 비용이 크다

성능보다 더 큰 문제는 버그다.  
signed와 unsigned를 무심코 섞으면 경계 조건, 루프, 비교 연산에서 예상과 다른 결과가 나올 수 있다.

예를 들어 길이나 크기와 관련된 값은 unsigned를 많이 쓰지만, 일반 산술 연산과 섞이면 조심해야 한다.

즉, 정수형 선택은 성능 문제이기도 하지만 **정확성 문제**이기도 하다.

### 6.4 overflow를 허용하는 설계는 위험하다

정수 overflow를 당연하게 허용하는 코드는 실무에서 위험하다.  
특히 배열 크기 계산, 버퍼 길이 계산, 인덱스 계산 같은 곳에서는 overflow가 보안 문제로 이어질 수도 있다.

즉, 정수 연산은 단순한 계산이 아니라 **범위 검증이 필요한 연산**으로 봐야 한다.

## 7. 어디서부터 비트 패턴, 수학적 값, 연산 규칙이 갈리는가

### 7.1 저장된 비트 패턴과 우리가 읽는 수학적 값은 같은 층이 아니다

메모리에는 결국 비트 패턴이 놓여 있고,  
우리가 보는 정수 값은 그 패턴을 특정 타입 규칙으로 해석한 결과다.

즉, 메모리에 “숫자 자체”가 저장된다고 생각하면  
signed/unsigned 해석 차이와 표현 범위 문제를 놓치기 쉽다.

### 7.2 같은 비트열도 타입에 따라 다른 값이 될 수 있다

이 사실은 처음 배우는 사람의 직관을 자주 깨뜨린다.  
같은 바이트를 봐도 signed인지 unsigned인지, 폭이 몇 비트인지에 따라 값 의미가 달라진다.

즉, 정수 이해의 핵심은 비트열 그 자체보다  
그 비트열에 어떤 해석 규칙을 얹느냐다.

### 7.3 저장 표현과 연산 중 변환 규칙을 분리해서 봐야 한다

메모리에 어떻게 저장되는가와  
연산 중 어떤 타입으로 승격되고 비교되는가는 별도 문제다.

즉, 정수 버그는 저장 표현에서만 생기지 않는다.  
promotion, conversion, overflow 규칙이 섞이면서 생기는 경우가 매우 많다.

### 7.4 overflow는 단순히 큰 값이 나오는 현상이 아니라 표현/규칙 경계 문제다

표현 가능한 범위를 넘는 순간,  
그 이후 동작은 타입 종류와 언어 규칙에 따라 완전히 다르게 다뤄질 수 있다.

즉, overflow를 “숫자가 한 바퀴 돈다” 정도로만 이해하면 부족하다.  
어떤 타입인지, 어떤 규칙이 적용되는지까지 함께 봐야 한다.

### 7.5 구현 관행과 언어 규칙을 같이 보되 섞지 않아야 한다

현대 시스템에서 흔한 표현 방식이 있다고 해도  
그것만으로 언어 규칙 전체를 설명할 수는 없다.  
반대로 언어 규칙만 보고 실제 플랫폼 비용과 관행을 무시해도 실무 감각이 약해진다.

즉, 정수 문서는  
언어 보장과 구현 현실을 분리해서 읽는 연습이 필요하다.

## 8. 실무에서 중요한 판단 기준

### 8.1 unsigned를 무조건 안전하다고 생각하면 안 된다

처음 배우는 사람은 “음수가 필요 없으면 unsigned가 더 좋다”고 생각하기 쉽다.  
하지만 실무에서는 unsigned가 오히려 비교와 형변환 버그를 만들기도 한다.

즉, unsigned는 “좋은 타입”이 아니라 **특정 상황에 맞는 타입**이다.

### 8.2 범위와 의미를 먼저 정해야 한다

타입을 고를 때는 먼저 질문해야 한다.

- 이 값은 음수가 될 수 있는가?
- 최대값은 어느 정도인가?
- 이 값은 길이인가, 인덱스인가, 일반 산술값인가?
- 외부 입력과 연산 결과를 함께 담을 것인가?

즉, 타입 선택은 문법이 아니라 **도메인 설계**의 일부다.

### 8.3 overflow 가능성을 코드 단계에서 의식해야 한다

특히 다음 연산은 늘 조심해야 한다.

- 크기 계산
- 배열 길이 계산
- 곱셈
- 인덱스 이동
- 버퍼 오프셋 계산

겉으로는 단순한 정수 계산처럼 보여도, 실제로는 메모리 안전성과 직접 연결된다.

### 8.4 표현과 연산을 분리해서 봐야 한다

정수형을 볼 때는 다음 두 가지를 따로 봐야 한다.

- 이 값이 어떻게 저장되는가
- 이 값이 연산될 때 어떤 규칙이 적용되는가

저장은 표현 문제고, 연산은 변환과 규칙 문제다.  
C에서는 두 층을 모두 봐야 한다.

---

## 9. 판단 체크리스트

- 비트 패턴과 수학적 정수 값을 같은 층으로 보고 있지 않은가
- 같은 비트열도 타입에 따라 다른 값 의미를 가질 수 있음을 설명할 수 있는가
- 저장 표현과 연산 중 변환 규칙을 분리해서 보고 있는가
- overflow를 단순 wraparound 직관 하나로 일반화하고 있지 않은가
- 구현 관행과 언어 규칙을 함께 보되 서로 섞어 결론내리지 않고 있는가

---

## 10. 더 깊게 볼 포인트

### 10.1 integer promotion

작은 정수형들이 연산 전에 더 큰 정수형으로 변환되는 규칙이다.  
`char`나 `short` 연산 결과가 왜 `int`처럼 보이는지를 설명할 때 중요하다.

### 10.2 usual arithmetic conversions

서로 다른 타입의 정수를 같이 연산할 때 어떤 타입으로 맞춰지는지 설명하는 규칙이다.  
signed/unsigned 혼합 문제를 이해하려면 반드시 봐야 한다.

### 10.3 endianness

비트 패턴이 아니라 **바이트 배열 순서**와 관련된 주제다.  
정수 표현을 더 깊게 볼 때 자연스럽게 연결된다.

### 10.4 비트 연산

정수 표현을 직접 다루는 가장 강력한 도구다.  
플래그, 마스크, 저수준 제어, 성능 최적화에서 자주 등장한다.

### 10.5 고정 폭 정수형

실무에서는 `int32_t`, `uint64_t`처럼 크기가 명확한 타입을 쓰는 경우가 많다.  
플랫폼 간 일관성을 확보할 때 중요하다.

---

## 정리

정수는 메모리에 “숫자” 형태로 저장되는 것이 아니다.  
정수는 **비트 패턴**으로 저장되고, 타입 규칙에 따라 해석된다.

핵심은 다음과 같다.

- unsigned는 모든 비트를 크기 표현에 사용한다.
- signed는 음수와 양수를 함께 표현한다.
- 현대 시스템에서는 보통 2의 보수 방식을 사용한다.
- 같은 비트 패턴도 signed와 unsigned에 따라 다르게 해석될 수 있다.
- overflow는 표현 가능한 범위를 넘는 현상이다.
- 타입 선택은 표현 범위, 정확성, 메모리, 연산 규칙을 함께 고려해야 한다.

즉, C에서 정수를 제대로 이해한다는 것은  
단순히 “숫자 타입을 안다”가 아니라,  
**비트 수준의 저장과 해석 규칙을 함께 본다**는 뜻이다.

이 관점이 잡혀야 이후 포인터, 배열, 구조체, 비트 연산, 성능 문제를 훨씬 안정적으로 이해할 수 있다.
