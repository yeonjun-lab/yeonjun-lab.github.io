## 정의

C에서 배열은 많은 문맥에서 첫 번째 원소를 가리키는 포인터처럼 변환된다.  
이 변환을 보통 배열의 decay라고 부른다.

즉 `int arr[10]`가 있을 때, 많은 표현식 문맥에서 `arr`는 `int *`처럼 취급된다.  
하지만 이것은 "배열과 포인터가 같은 것"을 뜻하지 않는다.  
정확히는 **배열 표현식이 특정 문맥에서 포인터 값으로 변환되는 것**이다.

핵심은 다음이다.

1. 배열은 원래 포인터가 아니다
2. 하지만 많은 표현식에서 첫 원소의 주소로 변환된다
3. 모든 문맥에서 그렇게 변환되는 것은 아니다
4. 이 차이를 이해하지 못하면 `sizeof`, 함수 인자, 문자열, 다차원 배열에서 계속 헷갈리게 된다

즉 decay는 배열의 정체가 아니라,  
**배열 표현식에 적용되는 문맥 의존적 변환 규칙**이다.

---

## 핵심 개념

### 1. 배열은 원래 "같은 타입의 객체 여러 개가 연속 배치된 것"이다

예시:
```
    int arr[4] = {10, 20, 30, 40};
```
이 선언에서 `arr`의 본질은 포인터가 아니다.  
`arr`는 `int` 4개가 연속해서 배치된 배열 객체다.

즉 메모리 관점에서 `arr`는 다음과 같다.

- `arr[0]`
- `arr[1]`
- `arr[2]`
- `arr[3]`

이 네 개의 `int` 객체가 연속된 하나의 배열 객체를 이룬다.

배열의 본질은 "주소를 담는 변수"가 아니라,  
**원소들이 연속 저장된 집합 객체**다.

---

### 2. 하지만 많은 표현식에서 배열 이름은 첫 원소의 주소로 변환된다

예시:
```
    int arr[4] = {10, 20, 30, 40};
    int *p = arr;
```
여기서 `arr`는 포인터 변수처럼 선언된 적이 없지만, `int *`에 대입될 수 있다.  
이유는 이 문맥에서 `arr`가 `&arr[0]`으로 decay되기 때문이다.

즉 위 코드는 개념적으로 다음과 비슷하게 이해할 수 있다.
```
    int *p = &arr[0];
```
이때 중요한 점은 아래다.

- `p`는 포인터 변수다
- `arr`는 배열 객체다
- 둘은 같은 것이 아니다
- 다만 이 문맥에서 `arr`가 포인터 값으로 변환되었을 뿐이다

---

### 3. decay는 배열 자체가 사라진다는 뜻이 아니다

처음 배우는 사람은 `arr`가 decay되면 배열이 포인터로 "바뀐다"고 생각하기 쉽다.  
그렇게 이해하면 거의 반드시 헷갈린다.

정확한 이해는 다음이다.

- 배열 객체는 여전히 배열이다
- 다만 어떤 표현식에서 배열 표현식의 값이 첫 원소 주소로 변환된다

즉 decay는 타입의 영구적 변화가 아니라,  
**표현식 평가 시 적용되는 일시적 변환 규칙**이다.

---

## 언제 decay가 일어나는가

### 1. 일반적인 표현식 문맥에서

예시:
```
    int arr[3] = {1, 2, 3};
    int *p = arr;
```
여기서 decay가 일어난다.

또 예시:
```
    printf("%d\n", arr[1]);
    int x = *(arr + 1);
```
`arr + 1`이 가능하다는 것 자체가 `arr`가 포인터처럼 변환되었다는 뜻이다.

즉 배열 이름으로 산술 연산을 하거나, 포인터 인자에 넘기거나,  
대부분의 일반 표현식에 넣으면 배열은 decay한다고 보면 된다.

---

### 2. 함수 인자로 전달될 때

예시:
```
    void f(int *p) {
    }

    int main(void) {
        int arr[5];
        f(arr);
    }
```
여기서 `arr`는 함수 호출 시 `int *`로 decay된다.

즉 함수는 배열 전체를 전달받는 것이 아니라,  
배열 첫 원소의 주소를 전달받는다.

이 때문에 함수 안에서는 배열 길이를 자동으로 알 수 없다.

---

### 3. 문자열 리터럴과 문자 배열에서도

예시:
```
    char s[] = "abc";
    char *p = s;
```
여기서도 `s`는 배열이지만, 포인터 문맥에서 decay되어 `char *`처럼 쓰인다.

이 규칙 때문에 문자열 처리에서 배열과 포인터가 자주 헷갈린다.  
하지만 본질은 같지 않다.

- `s`는 배열 객체
- `p`는 포인터 변수

---

## 언제 decay가 일어나지 않는가

이 부분이 가장 중요하다.  
배열이 "항상" 포인터가 되는 것은 아니다.

### 1. `sizeof`의 피연산자일 때

예시:
```
    int arr[10];
    sizeof(arr);
```
여기서는 decay가 일어나지 않는다.  
`sizeof(arr)`는 배열 전체 크기를 구한다.

예를 들어 `int`가 4바이트라면:
```
    sizeof(arr) == 40
```
만약 decay가 일어났다면 포인터 크기만 나왔어야 한다.  
하지만 실제로는 배열 전체 크기가 나온다.

이 사실 하나만으로도 배열과 포인터가 같지 않다는 점이 분명해진다.

---

### 2. unary `&`의 피연산자일 때

예시:
```
    int arr[10];
    &arr
```
여기서도 decay가 일어나지 않는다.

- `arr`가 decay되면 `int *`
- 하지만 `&arr`의 타입은 `int (*)[10]`

즉 `&arr`는 "배열 전체의 주소"다.  
이것은 첫 원소의 주소 `&arr[0]`와 다르다.

주소값이 숫자로는 같아 보일 수 있어도, 타입과 의미는 다르다.

- `arr` → 많은 문맥에서 `int *`
- `&arr[0]` → `int *`
- `&arr` → `int (*)[10]`

이 차이는 매우 중요하다.

---

### 3. 문자열/배열 초기화 문맥에서

예시:
```
    char s[] = "abc";
```
여기서는 `"abc"`라는 문자열 리터럴이 초기화에 사용되며,  
보통의 decay 문맥처럼 단순히 포인터만 전달되는 상황으로 보면 안 된다.

또한 배열 선언 자체는 배열 타입을 유지한다.

즉 선언과 초기화 문맥은 일반 표현식 문맥과 다르게 봐야 한다.

---

## `arr`, `&arr[0]`, `&arr`는 왜 다른가

이 셋은 매우 자주 헷갈린다.

예시:
```
    int arr[4];
```
### `arr`

많은 표현식 문맥에서 `arr`는 `&arr[0]`으로 decay된다.  
즉 첫 원소의 주소처럼 동작한다.

### `&arr[0]`

이것은 첫 번째 원소의 주소다. 타입은 `int *`다.

### `&arr`

이것은 배열 전체의 주소다. 타입은 `int (*)[4]`다.

즉 숫자 주소가 우연히 같게 보일 수 있어도 타입이 다르다.

그리고 포인터 연산 결과도 다르다.

예시:
```
    arr + 1        // 다음 int 원소
    &arr + 1       // 다음 "배열 전체" 위치
```
즉 `arr + 1`은 `sizeof(int)`만큼 이동하고,  
`&arr + 1`은 `sizeof(arr)`만큼 이동한다.

---

## 함수 매개변수에서 왜 특히 많이 헷갈리는가

예시:
```
    void f(int arr[10]) {
    }
```
이 선언을 보면 많은 사람이 "배열 10개를 받는 함수"처럼 느낀다.  
하지만 함수 매개변수에서 이런 배열 표기는 실제로 포인터로 조정된다.

즉 위 선언은 사실상 다음과 같다.
```
    void f(int *arr) {
    }
```
따라서 함수 안에서:
```
    sizeof(arr)
```
를 하면 배열 전체 크기가 아니라 포인터 크기가 나온다.

이 때문에 함수 매개변수의 `int arr[10]` 표기는  
"길이 10 배열 전체 전달"이 아니라 **읽기 쉬운 표기 관습**에 가깝다.

---

## decay 때문에 가능한 것과 불가능한 것

### 가능한 것

예시:
```
    int arr[3] = {1, 2, 3};
    int *p = arr;
    int x = *(arr + 1);
```
배열이 decay되므로 포인터 문맥에서 사용할 수 있다.

### 불가능한 것

예시:
```
    int arr[3];
    arr = p;    // 불가능
```
배열은 포인터로 decay될 수는 있지만,  
배열 자체가 포인터 변수인 것은 아니므로 대입 대상이 될 수 없다.

즉 decay는 "배열을 포인터처럼 사용할 수 있게 하는 규칙"이지,  
"배열이 포인터 변수라는 뜻"이 아니다.

---

## 다차원 배열에서 왜 더 중요해지는가

예시:
```
    int arr[3][4];
```
이 배열에서 `arr`는 많은 문맥에서 첫 번째 원소로 decay된다.  
그런데 이때 첫 번째 원소는 `int[4]`다.

즉 `arr`가 decay되면 타입은 `int (*)[4]`다.  
`int **`가 아니다.

이 지점이 매우 중요하다.

- 1차원 배열 `int arr[4]` → decay하면 `int *`
- 2차원 배열 `int arr[3][4]` → decay하면 `int (*)[4]`

즉 "배열이 decay한다"는 규칙은 맞지만,  
**무엇으로 decay하는지는 첫 원소의 타입에 따라 달라진다.**

이 때문에 `int **`와 2차원 배열은 같은 것이 아니다.

---

## 자주 하는 오해

### 오해 1. 배열은 포인터다

아니다.  
배열은 연속된 원소들의 집합 객체다.  
다만 많은 문맥에서 첫 원소 주소로 decay할 뿐이다.

---

### 오해 2. 배열 이름은 항상 첫 원소 주소다

아니다.  
`sizeof(arr)`, `&arr` 같은 문맥에서는 decay가 일어나지 않는다.

즉 배열 이름은 항상 포인터가 아니다.

---

### 오해 3. `&arr`와 `arr`는 같다

아니다.  
숫자 주소가 같아 보일 수는 있지만 타입이 다르다.

- `arr` → 보통 `int *`로 decay
- `&arr` → `int (*)[N]`

포인터 연산 의미도 다르다.

---

### 오해 4. 함수가 배열을 받으면 배열 전체가 복사된다

아니다.  
배열은 함수 호출 시 첫 원소 주소로 decay되어 전달된다.

즉 함수는 배열 전체 객체를 직접 받는 것이 아니라,  
배열 시작 위치를 가리키는 포인터를 받는다.

---

### 오해 5. `int **`는 2차원 배열 인자와 같다

아니다.  
2차원 배열 인자는 보통 열 크기를 포함한 타입이 필요하다.

예시:
```
    void f(int (*p)[4]);
```
이것은 `int **`와 다르다.

---

## 어디서부터 배열 객체와 표현식 변환이 갈리는가

### 1. 배열은 타입이고 decay는 문맥 의존 변환이다

처음 배우는 사람이 가장 많이 하는 단순화는 “배열은 포인터다”다.  
하지만 정확히는 배열은 배열 타입의 객체이고,  
많은 표현식 문맥에서만 첫 원소 주소로 변환된다.

즉, 배열 자체와 배열이 특정 문맥에서 보이는 모양을 분리해야 한다.

### 2. `arr`, `&arr[0]`, `&arr`는 비슷해 보여도 같은 층의 값이 아니다

세 표현식은 출력해 보면 비슷한 주소처럼 보일 수 있다.  
하지만 타입과 의미가 다르다.

- `arr`: 많은 문맥에서 첫 원소 포인터로 decay
- `&arr[0]`: 첫 원소의 주소
- `&arr`: 배열 전체 객체의 주소

즉, 값이 우연히 비슷해 보여도 객체 경계와 타입 정보는 다를 수 있다.

### 3. 함수 인자에서의 “배열” 표기는 실제 계약을 가린다

`void f(int arr[])`를 보면 배열을 받는다고 느끼기 쉽다.  
하지만 함수 매개변수 문맥에서는 실제로 포인터 조정이 일어난다.

즉, 호출부가 가진 배열 객체와  
함수 내부가 받는 포인터 표현을 같은 것으로 보면 안 된다.  
이 지점이 `sizeof`, 길이 인자, 버퍼 처리 버그의 출발점이다.

### 4. 다차원 배열에서는 “무엇으로 decay하는가”가 더 중요하다

2차원 배열은 첫 원소가 다시 배열이다.  
그래서 decay 결과도 “원소 배열에 대한 포인터”가 된다.

즉, 다차원 배열에서 중요한 것은 포인터가 되느냐가 아니라  
정확히 어느 타입의 포인터가 되느냐다.  
이 점을 놓치면 `int **` 같은 잘못된 직관이 생긴다.

### 5. decay 규칙은 편의 기능이지만 길이와 경계를 숨긴다

배열을 포인터처럼 다룰 수 있다는 점은 편리하다.  
하지만 그 대가로 배열 길이, 전체 객체 경계, 배열 자체 타입 정보가  
표현식 수준에서 덜 드러나게 된다.

즉, decay는 문법 편의이면서 동시에  
정보 손실을 동반하는 변환으로 봐야 한다.

---

## 실무 관점

### 1. `sizeof` 버그의 핵심 원인 중 하나다

예시:
```
    void f(int arr[]) {
        size_t n = sizeof(arr) / sizeof(arr[0]);
    }
```
이 코드는 함수 안에서 잘못된 결과를 만든다.  
왜냐하면 `arr`는 이미 포인터로 조정되었기 때문이다.

즉 배열 길이 계산은 배열이 실제 배열 객체로 보이는 문맥에서만 가능하다.

---

### 2. API 설계에서 길이 인자를 따로 받는 이유다

C 함수가 배열을 받을 때 보통 길이를 함께 받는 이유는,  
배열이 decay되면 길이 정보가 사라지기 때문이다.

예시:
```
    void process(int *arr, size_t len);
```
이 설계는 우연이 아니라 C 배열 전달 모델의 직접적 결과다.

---

### 3. 문자열 처리 버그와도 연결된다

문자열 버그의 상당수는 아래를 혼동하면서 생긴다.

- 배열인지 포인터인지
- 실제 저장공간을 가진 객체인지
- 단순히 어떤 위치를 가리키는 포인터인지

예시:
```
    char s[] = "abc";
    char *p = "abc";
```
겉보기엔 비슷하지만 메모리 모델은 다르다.

- `s`는 수정 가능한 배열 객체
- `p`는 문자열 리터럴 위치를 가리키는 포인터

이 차이는 매우 중요하다.

---

### 4. 다차원 데이터 처리에서 필수다

영상, 행렬, 버퍼, 네트워크 패킷 파싱 등에서  
배열 decay를 잘못 이해하면 타입 선언이 전부 꼬이기 쉽다.

특히 아래를 구분해야 한다.

- 연속된 2차원 배열
- 포인터 배열
- 이중 포인터
- 배열 전체 주소

이것은 단순 문법 문제가 아니라 메모리 레이아웃 문제다.

---

## 판단 체크리스트

- 배열 자체와 배열이 문맥상 포인터처럼 보이는 현상을 구분하고 있는가
- `arr`, `&arr[0]`, `&arr`의 타입과 의미 차이를 설명할 수 있는가
- 함수 매개변수의 `int arr[]` 표기가 실제로는 포인터 조정된다는 점을 이해하고 있는가
- 다차원 배열에서 decay 결과가 정확히 어떤 포인터 타입인지 따지고 있는가
- decay가 편의이면서 동시에 길이와 경계를 숨긴다는 점을 의식하고 있는가

---

## 정리

배열은 포인터가 아니다.  
배열은 같은 타입의 객체들이 연속 배치된 배열 객체다.

하지만 많은 표현식 문맥에서 배열은 첫 번째 원소의 주소로 decay된다.  
이 규칙 때문에 배열은 자주 포인터처럼 보인다.

핵심은 다음과 같다.

- 배열은 본질적으로 포인터가 아니다
- decay는 문맥에 따라 적용되는 변환 규칙이다
- `sizeof`와 `&arr`에서는 decay가 일어나지 않는다
- 함수 인자에서는 배열이 포인터처럼 전달된다
- 다차원 배열에서는 무엇으로 decay하는지 더 신중히 봐야 한다

즉 C에서 배열을 제대로 이해하려면  
"배열과 포인터가 같다"라고 외우는 것이 아니라  
**배열이 언제, 어떤 문맥에서, 무엇으로 변환되는가**를 이해해야 한다.

이 관점이 잡히면 다음 주제들도 자연스럽게 연결된다.

- 문자와 문자열은 왜 구분해서 이해해야 하는가
- 함수 인자는 왜 값으로 전달되는가
- `int **`와 2차원 배열은 왜 다른가
- `sizeof`는 무엇을 알려주고 무엇을 숨기는가
