## 정의

구조체(struct)는 여러 멤버를 하나의 객체로 묶는 타입이지만, C에서 그것은 단순한 논리적 묶음이 아니라 **실제로 메모리에 배치되는 객체 레이아웃**이기도 하다.  
즉 구조체를 이해한다는 것은 "필드 여러 개를 묶는다"는 수준을 넘어서, **각 멤버가 메모리 안에서 어떤 순서와 간격으로 놓이는가**를 이해하는 것이다.

핵심은 다음이다.

1. 구조체 멤버는 선언 순서를 기준으로 배치된다
2. 하지만 멤버 크기만 단순히 이어 붙이는 것과는 다를 수 있다
3. 정렬(alignment) 때문에 멤버 사이에 padding이 들어갈 수 있다
4. 따라서 구조체 크기와 메모리 모습은 눈에 보이는 선언보다 더 복잡할 수 있다

즉 구조체는  
**논리적 필드 집합이면서 동시에 물리적 메모리 레이아웃을 가진 타입**이다.

---

## 먼저 큰 그림부터

많은 초보자는 구조체를 이렇게 이해한다.

- 여러 변수를 하나로 묶는 문법
- 자바의 class보다 단순한 버전
- 필드를 순서대로 저장하는 자료형

이 설명은 반쯤만 맞다.

예를 들어:
```c
    struct Point {
        int x;
        int y;
    };
```
이 구조체는 직관적으로 다음처럼 보인다.

- `x` 4바이트
- `y` 4바이트
- 합쳐서 8바이트

이 경우는 실제로도 대체로 그렇게 보일 수 있다.

하지만 다음은 어떨까.
```c
    struct Example {
        char c;
        int x;
    };
```
겉보기엔 1바이트 + 4바이트라서 5바이트 같아 보인다.  
하지만 실제 크기는 8바이트처럼 더 클 수 있다.

즉 구조체는  
"필드 합 = 구조체 크기"라고 단순화하면 거의 반드시 막히게 된다.

---

## 핵심 개념

### 1. 구조체는 멤버들을 하나의 객체 안에 순서대로 둔다

예시:
```c
    struct Point {
        int x;
        int y;
    };
```
이 구조체 객체가 하나 만들어지면, 그 안에는 `x`와 `y`라는 두 멤버 객체가 들어 있다.

즉 구조체는 추상적인 레코드가 아니라,  
**하나의 큰 객체 안에 여러 하위 객체가 들어 있는 형태**로 볼 수 있다.

그리고 중요한 점은:

- 멤버 순서는 선언 순서를 따른다
- 즉 `y`가 `x`보다 앞에 가도록 재배열되지는 않는다

이 순서 보장은 구조체 레이아웃을 이해하는 출발점이다.

---

### 2. 하지만 멤버는 딱 붙지 않을 수 있다

예시:
```c
    struct Example {
        char c;
        int x;
    };
```
직관적으로는:

- `c` : 1바이트
- `x` : 4바이트

이므로 5바이트 같아 보인다.

하지만 실제로는 `x`가 특정 정렬 조건을 요구할 수 있어서,  
`c` 다음에 바로 `x`가 오지 않고 중간에 빈 공간이 들어갈 수 있다.

즉 구조체 레이아웃은 단순 선언 나열이 아니라,  
**정렬 제약을 만족시키는 배치 결과**다.

---

### 3. padding은 멤버 사이 또는 끝에 들어갈 수 있다

구조체에서 padding은 보통 두 군데에 들어간다.

#### 멤버 사이 padding

앞 멤버와 다음 멤버의 정렬 요구가 맞지 않을 때 들어간다.

#### 끝 padding

구조체 배열을 만들 때 각 원소가 적절히 정렬되도록 구조체 끝에 추가될 수 있다.

즉 padding은 이상한 예외가 아니라,  
구조체를 실제 머신 친화적으로 배치하기 위한 매우 일반적인 요소다.

---

### 4. 구조체 크기는 가장 큰 정렬 요구와도 관련된다

예시:
```c
    struct Example {
        char c;
        int x;
    };
```
이 구조체 하나만 보면 `x` 앞의 padding만 생각하기 쉽다.  
하지만 구조체 배열을 생각하면 구조체 전체 크기도 중요해진다.
```c
    struct Example arr[10];
```
이때 `arr[1]`의 `x`도 여전히 올바르게 정렬되어야 한다.  
그래서 구조체 끝에도 padding이 붙어 전체 크기가 정렬 조건에 맞춰질 수 있다.

즉 구조체 크기는  
**멤버 크기 합**이 아니라 **멤버 배치 + 정렬 제약 + 배열 친화성**의 결과다.

---

## 왜 정렬이 필요한가

정렬(alignment)은 어떤 타입의 객체가 메모리에서 특정 배수 주소에 놓이는 것이 더 유리하거나 필요하다는 제약이다.

예를 들어 어떤 시스템에서는:

- `char`는 거의 어디든 가능
- `int`는 4바이트 경계 선호
- `double`은 더 큰 경계 선호

이럴 수 있다.

즉 정렬은 다음과 연결된다.

- CPU가 데이터를 더 효율적으로 읽기 위해
- 어떤 아키텍처에서는 잘못 정렬된 접근이 느리거나 불가능할 수 있어서
- ABI와 호출 규약을 맞추기 위해

즉 padding은 낭비처럼 보일 수 있지만,  
실제로는 정렬 요구를 만족시키기 위한 구조적 비용이다.

---

## 예시로 보는 구조체 배치 감각

### 1. 단순한 구조체
```c
    struct A {
        int x;
        int y;
    };
```
이 경우는 보통 비교적 단순하다.

- `x`
- `y`

즉 대체로 연속 8바이트처럼 보이기 쉽다.

---

### 2. 작은 타입 뒤에 큰 타입
```c
    struct B {
        char c;
        int x;
    };
```
이 경우는 `c` 뒤에 padding이 들어가 `x` 정렬을 맞출 수 있다.

즉 눈으로 보이는 1 + 4가 아니라,  
실제로는 더 큰 크기가 나올 수 있다.

---

### 3. 필드 순서를 바꾸면 레이아웃이 달라질 수 있다
```c
    struct C1 {
        char c;
        int x;
        char d;
    };

    struct C2 {
        int x;
        char c;
        char d;
    };
```
이 둘은 같은 멤버를 가진 것처럼 보여도,  
padding 위치와 전체 크기는 달라질 수 있다.

즉 구조체 설계에서는  
**멤버 종류뿐 아니라 멤버 순서도 성능과 크기에 영향**을 준다.

---

## 구조체는 왜 배열과 함께 생각해야 하는가

구조체 레이아웃을 이해할 때 구조체 하나만 보면 감이 덜 온다.  
배열로 보면 더 분명해진다.

예시:
```c
    struct Point pts[100];
```
배열 원소 `pts[i]`는 같은 크기 간격으로 놓여야 한다.  
즉 구조체 크기는 "다음 구조체가 시작되는 간격"이 된다.

그래서 구조체 끝 padding은 단일 객체만의 문제가 아니라,  
**구조체 배열에서 각 원소의 정렬을 유지하기 위한 규칙**과 연결된다.

즉 구조체는 혼자 있는 객체보다  
배열 원소 관점에서 보면 레이아웃 의미가 더 선명해진다.

---

## 구조체 멤버 접근은 왜 그냥 오프셋 계산처럼 보이는가

예시:
```c
    struct Point p;
    p.x = 10;
```
겉보기에는 `p.x`라는 고수준 문법이지만,  
실제 머신 수준으로 내려가면 구조체 시작 주소에서 특정 오프셋만큼 이동한 위치를 읽고 쓰는 형태로 연결된다.

즉 구조체는 "객체 안의 객체들"이고,  
각 멤버 접근은 그 구조체 레이아웃 위에서 정해진 위치 접근으로 생각할 수 있다.

이 점 때문에 `offsetof`, 패딩, 정렬 문서가 바로 뒤에 이어진다.

---

## 왜 직렬화와 바로 연결되는가

많은 초보자는 구조체를 보면 이런 유혹을 느낀다.

- 구조체가 메모리에 배치돼 있으니까
- 그냥 그 바이트를 파일이나 네트워크로 보내면 되지 않나

하지만 여기서 바로 문제가 생긴다.

- padding이 들어갈 수 있음
- endianness 차이 가능
- ABI/컴파일러 차이 가능
- 포인터 멤버는 의미가 없음

즉 구조체 메모리 레이아웃은 분명 중요하지만,  
그걸 곧바로 외부 포맷으로 동일시하면 위험하다.

이 때문에 구조체 레이아웃 문서는 직렬화 문서와도 연결된다.

---

## 왜 성능과도 연결되는가

구조체 레이아웃은 단순 크기 문제를 넘어서 성능에도 영향을 준다.

예를 들어:

- 불필요한 padding이 많으면 캐시에 덜 들어감
- 자주 쓰는 필드가 멀리 떨어져 있으면 접근 효율 저하
- 멤버 순서에 따라 캐시 친화성이 달라질 수 있음

즉 구조체는 "필드 묶기 문법"이면서 동시에  
**데이터 지향 설계의 기본 단위**다.

그래서 이후의 `struct padding과 alignment`, `AoS와 SoA`와 자연스럽게 연결된다.

## 자주 하는 오해

### 오해 1. 구조체 크기는 멤버 크기 합이다

아니다.  
padding과 alignment 때문에 더 커질 수 있다.

---

### 오해 2. 컴파일러가 멤버 순서를 바꿀 수 있다

아니다.  
C 구조체는 멤버 선언 순서가 중요하다.  
그 순서를 기준으로 배치가 결정된다.

---

### 오해 3. padding은 쓸모없는 낭비일 뿐이다

아니다.  
정렬 요구를 만족하고, 배열 원소 배치를 맞추기 위한 구조적 장치다.

---

### 오해 4. 구조체 메모리 모습은 모든 플랫폼에서 같다

아니다.  
크기, 정렬, padding, ABI에 따라 달라질 수 있다.

---

### 오해 5. 구조체를 그대로 저장/전송하면 직렬화가 된다

아니다.  
padding, endianness, 포인터, 표현 차이 때문에 그렇게 단순하지 않다.

---

## 어디서부터 언어 보장과 ABI 계약이 갈리는가

### 1. 선언 순서는 보장되지만 전체 레이아웃 세부는 환경 의존일 수 있다

구조체를 설명할 때 흔히 두 가지가 섞인다.

- 언어 차원의 기본 보장
- 플랫폼/ABI/컴파일러가 정하는 세부 배치

멤버 선언 순서는 중요한 출발점이지만,
정렬, padding, 전체 크기 같은 구체 값은 구현 환경에 따라 달라질 수 있다.

즉 구조체를 볼 때는  
"순서는 의미를 갖는다"와 "세부 배치는 환경 의존일 수 있다"를 함께 잡아야 한다.

### 2. 구조체는 논리 모델과 물리 모델이 겹쳐지는 드문 타입이다

문자열이나 정수는 보통 논리 의미가 먼저 보이지만,  
구조체는 설계 순간부터 레이아웃 비용, 캐시, ABI, 직렬화까지 함께 따라온다.

즉 구조체 문서는 단순 문법 설명이 아니라  
**객체 모델과 메모리 모델이 만나는 접점**을 설명하는 문서다.

---

## 실무 관점

### 1. 구조체는 논리 모델이면서 물리 모델이다

실무에서 구조체를 설계할 때는 다음 둘을 같이 봐야 한다.

- 논리적으로 어떤 필드를 묶는가
- 물리적으로 메모리에 어떻게 놓이는가

즉 구조체 설계는 단순 API 설계가 아니라 데이터 레이아웃 설계이기도 하다.

---

### 2. 멤버 순서가 실제 비용을 바꿀 수 있다

특히 구조체가 많고, 배열로 대량 저장되고, 반복 접근되는 경우  
멤버 순서가 전체 메모리 사용량과 캐시 효율에 영향을 줄 수 있다.

즉 필드 순서는 단순 미관 문제가 아니다.

---

### 3. 바이너리 호환성과도 연결된다

라이브러리 경계, ABI, 파일 포맷, 네트워크 메시지 등에서는  
구조체 레이아웃 차이가 호환성 문제를 만들 수 있다.

즉 구조체는 내부 구현에만 머무르지 않고,  
바이너리 인터페이스 문제와도 이어진다.

---

### 4. 다음 문서들과 직접 연결된다

이 문서는 아래와 바로 이어진다.

- struct padding과 alignment는 왜 중요한가
- offsetof와 struct layout을 어떻게 확인하는가
- packed struct는 왜 위험할 수 있는가
- AoS와 SoA는 무엇이 다르고 언제 중요한가

즉 구조체 레이아웃 축의 입구 문서다.

---

## 판단 체크리스트

구조체를 설계하거나 읽을 때는 아래 순서로 보면 좋다.

1. 논리적으로 어떤 필드를 묶는가
2. 각 필드의 정렬 요구가 배치에 어떤 영향을 주는가
3. 배열로 놓였을 때 구조체 전체 크기가 어떤 간격을 만들까
4. 이 구조를 메모리 내부 용도로만 쓸지, 바이너리 경계 밖으로 내보낼지

즉 구조체는 "필드 모음"이 아니라  
**논리 구조와 물리 레이아웃을 동시에 설계해야 하는 타입**으로 읽어야 한다.

---

## 정리

구조체는 여러 멤버를 하나로 묶는 타입이지만, C에서는 동시에 실제 메모리 레이아웃을 가진 객체다.

핵심은 다음과 같다.

- 멤버는 선언 순서를 기준으로 배치된다
- 하지만 멤버 크기 합만으로 구조체 크기를 판단하면 안 된다
- 정렬 때문에 멤버 사이와 끝에 padding이 들어갈 수 있다
- 구조체 배열을 생각하면 전체 크기와 정렬의 이유가 더 분명해진다
- 구조체 레이아웃은 성능, ABI, 직렬화와 모두 연결된다

즉 구조체를 제대로 이해한다는 것은  
필드를 묶는 문법으로만 보는 것이 아니라 **하나의 큰 객체 안에 여러 하위 객체가 실제로 어떻게 놓이는가**를 이해하는 것이다.

이 관점이 잡히면 이후의 padding, alignment, offsetof, packed struct 문서가 훨씬 자연스럽게 이어집니다.
