## 정의

C에서 함수 인자는 값으로 전달된다.  
즉 함수 호출 시 호출자가 가진 객체 자체가 함수 안으로 들어가는 것이 아니라, 그 객체의 값이 매개변수로 복사되어 전달된다.

이 규칙은 정수형에서만 적용되는 것이 아니다.  
포인터도 값으로 전달되고, 구조체도 값으로 전달된다.  
다만 전달되는 값의 성격이 다를 뿐이다.

핵심은 다음이다.

1. 함수는 호출자의 변수를 직접 받지 않는다
2. 함수는 호출자가 넘긴 값을 복사한 자기 매개변수를 가진다
3. 따라서 함수 안에서 매개변수 자체를 바꿔도 원래 변수는 바뀌지 않는다
4. 원래 객체를 간접적으로 바꾸려면 그 객체의 주소를 값으로 넘겨야 한다

즉 C의 함수 호출 모델은 "참조를 자동으로 공유하는 모델"이 아니라,  
**값 복사를 기반으로 한 모델**이다.

---

## 핵심 개념

### 1. 정수형 인자를 보면 가장 쉽게 드러난다

예시:
```
    void set_value(int x) {
        x = 100;
    }

    int main(void) {
        int a = 10;
        set_value(a);
    }
```
많은 초보자는 `set_value(a);` 호출 뒤에 `a`가 100이 될 것처럼 느낄 수 있다.  
하지만 실제로는 아니다.

이유는 함수가 `a` 자체를 받은 것이 아니라, `a`의 값 10을 복사해서 자기 매개변수 `x`에 넣었기 때문이다.

즉 함수 안의 `x`는 `a`와 다른 객체다.

- `a`는 `main`의 지역변수
- `x`는 `set_value`의 지역 매개변수
- 값은 처음에 같을 수 있지만, 같은 객체는 아니다

따라서 `x = 100;`은 복사본 수정일 뿐, `a`를 수정하지 않는다.

---

### 2. "값 전달"은 "복사된 별도 객체"를 만든다는 뜻이다

함수 인자가 값으로 전달된다는 말은 단순 구호가 아니다.  
실제로는 호출 시점에 다음이 일어난다고 이해하면 된다.

- 호출자가 인자 값을 준비한다
- 함수의 매개변수가 그 값을 받아 자기 쪽 저장공간에 놓는다
- 함수는 자기 매개변수를 사용해 계산한다

즉 매개변수는 호출자의 변수와 이름만 다른 것이 아니라,  
**수명과 저장공간이 분리된 별도 객체**다.

이 관점을 잡아야 포인터, 배열, 구조체 전달도 전부 정리된다.

---

### 3. 포인터도 값으로 전달된다

이 지점에서 많은 사람이 혼란을 느낀다.

예시:
```
    void set_value(int *p) {
        *p = 100;
    }

    int main(void) {
        int a = 10;
        set_value(&a);
    }
```
이 코드는 호출 후 `a`를 실제로 바꾼다.  
그러면 많은 사람이 "포인터는 참조 전달 아닌가?"라고 생각한다.

하지만 아니다.  
여기서도 전달된 것은 여전히 값이다.  
단지 그 값이 `a`의 주소일 뿐이다.

즉 함수는 다음을 받는다.

- `a` 자체가 아니라 `&a`라는 주소 값
- 그 주소 값을 자기 매개변수 `p`에 복사
- `*p`를 통해 원래 객체 `a`에 간접 접근

즉 포인터 전달은 참조 전달이 아니라,  
**주소라는 값을 전달하고 그 주소를 따라가서 원래 객체를 수정하는 것**이다.

---

### 4. 포인터 매개변수 자체를 바꾸면 원래 포인터는 안 바뀐다

예시:
```
    void reset_ptr(int *p) {
        p = NULL;
    }

    int main(void) {
        int x = 10;
        int *p = &x;
        reset_ptr(p);
    }
```
이 코드를 보고 `p`가 `NULL`이 될 것 같다고 생각하면 안 된다.  
왜냐하면 함수 안의 `p`도 포인터 값의 복사본이기 때문이다.

즉 아래 두 경우를 구분해야 한다.

- `*p = ...` → p가 가리키는 대상 수정
- `p = ...` → 함수 안의 포인터 복사본 수정

원래 포인터 변수 자체를 바꾸려면 `int **`가 필요하다.

이것이 바로 이중 포인터 문서와 연결되는 지점이다.

---

## 왜 배열이 특히 헷갈리게 만드는가

배열은 함수 인자 전달에서 가장 자주 오해를 만든다.

예시:
```
    void f(int arr[]) {
    }
```
많은 사람이 이 코드를 "배열 전체를 받는 함수"라고 이해한다.  
하지만 실제로 함수 매개변수에서 배열 표기는 포인터로 조정된다.

즉 위 선언은 사실상 아래와 같다.
```
    void f(int *arr) {
    }
```
따라서 함수는 배열 전체 객체를 직접 받지 않는다.  
배열 첫 원소의 주소를 값으로 받는다.

즉 여기서도 값 전달 원칙은 그대로다.

- 배열 자체가 복사되는 것이 아님
- 배열이 decay한 포인터 값이 전달됨
- 함수 안에서는 포인터 매개변수만 존재함

이 때문에 함수 안에서는 배열 길이를 자동으로 알 수 없다.

---

## 구조체도 값으로 전달된다

예시:
```
    struct Point {
        int x;
        int y;
    };

    void move(struct Point p) {
        p.x += 1;
        p.y += 1;
    }

    int main(void) {
        struct Point a = {10, 20};
        move(a);
    }
```
여기서도 `a` 자체가 함수 안으로 들어간 것이 아니다.  
`a`의 값이 복사되어 매개변수 `p`가 된 것이다.

따라서 함수 안에서 `p.x`, `p.y`를 바꿔도 원래의 `a`는 바뀌지 않는다.

즉 구조체는 포인터처럼 간접 수정이 자동으로 일어나는 것이 아니다.  
구조체 전달도 기본적으로 값 전달이다.

이 점은 시스템 코드에서 큰 구조체를 자주 복사하면 비용이 커질 수 있다는 실무 문제와도 연결된다.

---

## 스택 프레임 관점에서 보면 왜 그런가

함수 호출 시 스택 프레임 관점으로 생각하면 더 명확하다.

예시:
```
    void f(int x) {
        x = 20;
    }

    int main(void) {
        int a = 10;
        f(a);
    }
```
개념적으로는 다음처럼 볼 수 있다.

- `main`의 스택 프레임 안에 `a`가 있음
- `f(a)`를 호출하면 `f`의 스택 프레임이 만들어짐
- `f` 안에 매개변수 `x`가 별도로 놓임
- `x`는 `a`의 값을 복사해서 시작함

즉 `a`와 `x`는 서로 다른 스택 프레임의 다른 객체다.

이 때문에 함수 안에서 `x`를 바꿔도 원래의 `a`는 바뀌지 않는다.

포인터도 마찬가지다.

- `main`의 포인터 변수 `p`
- `f`의 포인터 매개변수 `p`
- 둘은 같은 주소값으로 시작할 수는 있어도, 다른 변수다

이 그림이 머릿속에 잡히면 값 전달 모델이 훨씬 쉬워진다.

---

## "값 전달"과 "원본 수정 가능"은 모순이 아니다

이 부분이 핵심 오해 지점이다.

많은 사람이 아래 두 문장을 충돌한다고 느낀다.

- C는 값 전달이다
- 포인터를 넘기면 원본을 바꿀 수 있다

하지만 둘은 모순이 아니다.

정확한 구조는 다음이다.

1. 함수는 값만 받는다
2. 그 값이 정수일 수도 있고, 구조체일 수도 있고, 주소일 수도 있다
3. 주소를 받으면 그 주소가 가리키는 원래 객체에 간접 접근할 수 있다

즉 원본 수정이 가능한 이유는 "참조 전달"이어서가 아니라,  
**주소라는 값을 통해 원본 위치를 찾을 수 있기 때문**이다.

---

## 왜 이 규칙이 중요한가

### 1. 함수의 경계를 명확하게 만든다

함수는 호출자의 변수를 직접 소유하지 않는다.  
항상 값을 받아 자기 매개변수로 다룬다.

이 규칙 덕분에 다음이 분명해진다.

- 무엇이 입력인가
- 무엇이 복사되는가
- 무엇을 바꿀 수 있는가
- 원래 객체를 바꾸려면 어떤 경로가 필요한가

즉 값 전달 모델은 함수 경계를 명확하게 만든다.

---

### 2. 부작용을 의식하게 만든다

원래 객체를 바꾸려면 주소를 넘기고, 함수 안에서 역참조해야 한다.  
즉 수정은 의도적으로 드러난다.

예시:
```
    void update(int *p)
```
이 시그니처만 봐도 "호출자 객체를 바꿀 수 있겠구나"를 알 수 있다.

반면:
```
    void update(int x)
```
이 함수는 호출자 정수 자체를 바꿀 수 없다.

즉 값 전달 모델은 함수의 부작용 범위를 읽기 쉽게 만든다.

---

## 자주 하는 오해

### 오해 1. 포인터를 넘기면 참조 전달이다

아니다.  
포인터도 값으로 전달된다.  
다만 그 값이 주소이기 때문에 원래 객체에 간접 접근할 수 있을 뿐이다.

---

### 오해 2. 함수 안의 매개변수 이름이 같으면 원래 변수와 같은 것이다

아니다.  
이름이 같든 다르든 매개변수는 함수의 별도 객체다.

---

### 오해 3. 배열 인자를 넘기면 배열 전체가 복사된다

아니다.  
배열은 함수 호출 문맥에서 decay되어 첫 원소 주소가 전달된다.

---

### 오해 4. 구조체는 특별해서 자동으로 참조처럼 전달된다

아니다.  
구조체도 기본적으로 값으로 전달된다.  
원본 수정을 원하면 포인터를 넘겨야 한다.

---

### 오해 5. 값 전달이면 절대 원본을 못 바꾼다

아니다.  
원본의 주소를 값으로 넘기면, 그 주소를 통해 간접 수정할 수 있다.

즉 "값 전달"과 "간접 원본 수정 가능"은 동시에 성립한다.

---

## 실무 관점

### 1. API 시그니처를 읽는 기준이 된다

C 함수 선언을 볼 때 가장 먼저 봐야 하는 것은  
"이 함수가 값을 받는가, 주소를 받는가"다.

예시:
```
    void sort(int *arr, size_t n);
```
이 함수는 배열 내용을 바꿀 수 있다.  
왜냐하면 배열의 시작 주소를 받기 때문이다.

반면:
```
    int abs_value(int x);
```
이 함수는 입력 정수 자체를 바꾸지 않는다.  
복사된 값을 이용해 계산만 한다.

즉 함수 인자 타입을 보면 수정 가능성과 부작용이 읽힌다.

---

### 2. 불필요한 복사와 성능 문제를 이해하게 해 준다

작은 스칼라 값은 값 전달이 자연스럽다.  
하지만 큰 구조체를 계속 값으로 넘기면 복사 비용이 커질 수 있다.

그래서 실무에서는 아래를 비교하게 된다.

- 작은 값: 값 전달
- 큰 구조체: `const T *`로 읽기 전용 전달
- 수정 필요: `T *`로 전달

즉 값 전달 모델 이해는 단순 개념이 아니라 성능 설계와도 연결된다.

---

### 3. 버그를 줄인다

다음 같은 실수는 매우 흔하다.

- 함수 안에서 매개변수만 바꿔 놓고 원래 변수가 바뀌길 기대함
- 포인터와 이중 포인터를 혼동함
- 배열 길이가 같이 전달된다고 착각함
- 구조체가 자동으로 참조처럼 공유될 것이라 생각함

이런 실수는 전부 "C 함수는 값 전달"이라는 원칙을 정확히 이해하면 줄어든다.

---

### 4. 이후 주제와 직접 연결된다

이 문서는 다음 문서들과 직접 연결된다.

- 포인터의 포인터는 왜 필요한가
- 배열은 언제 포인터로 decay되는가
- 문자열은 왜 특별 취급되는가
- `char **argv`는 무엇을 의미하는가
- `sizeof`는 무엇을 알려주고 무엇을 숨기는가

즉 함수 인자 전달은 포인터, 배열, 문자열을 연결하는 중심축이다.

---

## 정리

C에서 함수 인자는 값으로 전달된다.  
즉 함수는 호출자의 객체 자체를 받는 것이 아니라, 그 값의 복사본을 매개변수로 받는다.

핵심은 다음이다.

- 정수도 값으로 전달된다
- 포인터도 값으로 전달된다
- 구조체도 값으로 전달된다
- 배열은 함수 호출 시 decay한 포인터 값이 전달된다
- 원래 객체를 바꾸려면 그 객체의 주소를 값으로 넘겨야 한다

즉 C의 함수 호출 모델은 "자동 공유"가 아니라  
**값 복사 + 필요 시 주소를 통한 간접 접근** 모델이다.

이 관점이 잡히면 포인터, 배열, 이중 포인터, 문자열 인자, 구조체 전달이 모두 하나의 원리로 정리되기 시작한다.
