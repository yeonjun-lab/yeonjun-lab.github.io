## 정의

C의 타입 시스템은 단순히 "변수의 종류를 구분하는 문법"이 아니다.  
C에서 타입은 어떤 값이 메모리에서 몇 바이트를 차지하는지, 그 바이트들을 어떤 의미로 해석하는지, 어떤 연산을 적용할 수 있는지, 그리고 컴파일러가 어떤 최적화를 해도 되는지를 결정하는 규칙의 집합이다.

즉 C의 타입 시스템은 다음 네 가지를 함께 결정한다.

1. 이 객체가 메모리에서 어떤 형태로 존재하는가
2. 이 값을 어떤 연산으로 다룰 수 있는가
3. 이 값을 다른 타입으로 바꿀 때 어떤 변환 규칙이 적용되는가
4. 컴파일러가 이 값을 어느 범위까지 같은 객체라고 가정할 수 있는가

다른 고수준 언어에서는 타입이 주로 "추상적 자료의 성격"을 나타내는 경우가 많지만, C에서는 타입이 훨씬 더 낮은 수준에서 동작한다.  
C의 타입은 값의 의미와 메모리 표현을 동시에 다룬다.

---

## 핵심 개념

### 1. C의 타입은 값의 분류이면서 메모리 해석 규칙이다

예를 들어 `int`와 `float`는 둘 다 4바이트일 수 있다.  
하지만 같은 4바이트라도 `int`는 정수 비트 패턴으로, `float`는 부동소수점 규칙으로 해석한다.

즉 타입이 다르면 단순히 "사용 가능한 연산자"만 달라지는 것이 아니라, 같은 바이트열을 읽는 방식 자체가 달라진다.

예시:
```c
    int a = 1065353216;
    float b = 1.0f;
```
둘 다 4바이트일 수 있지만, 메모리에서 같은 패턴을 가진다고 해서 같은 의미가 되지는 않는다.  
C에서는 "무슨 타입으로 읽느냐"가 곧 "무슨 값으로 해석하느냐"와 연결된다.

---

### 2. C의 기본 타입은 크게 객체 타입과 함수 타입으로 나뉜다

C에서 타입은 크게 보면 다음처럼 볼 수 있다.

- 객체 타입(object type)
- 함수 타입(function type)
- 불완전 타입(incomplete type)

실무에서 먼저 중요한 것은 객체 타입이다. 객체 타입은 메모리에 저장될 수 있는 타입이다.

대표적인 객체 타입은 다음과 같다.

- 정수형: `char`, `short`, `int`, `long`, `long long`
- 부호 없는 정수형: `unsigned char`, `unsigned int` 등
- 부동소수점형: `float`, `double`, `long double`
- 파생 타입: 포인터, 배열, 구조체, 공용체, 열거형

즉 C 타입 시스템은 몇 개의 원시 타입만 있는 것이 아니라,  
기본 타입 위에 포인터, 배열, 구조체 같은 파생 구조를 얹어 가는 방식이다.

---

### 3. C는 "크기"보다 "범주"가 먼저다

초보자는 흔히 `int는 4바이트`, `long은 8바이트`처럼 외운다.  
하지만 C 표준은 모든 타입의 크기를 고정하지 않는다. 구현마다 다를 수 있다.

중요한 것은 "몇 바이트냐"보다 먼저 다음이다.

- 정수형인가
- 부호가 있는가
- 문자형인가
- 부동소수점형인가
- 포인터인가
- 배열인가
- 구조체인가

즉 타입 시스템의 출발점은 절대 크기 암기가 아니라 **분류 체계**다.

예를 들어 `char`는 1바이트이지만, 그 1바이트가 곧 "8비트"라고 단정할 수는 없다.  
C 표준에서 `sizeof(char)`는 항상 1이지만, 그 1이 몇 비트인지는 구현이 정한다.

---

### 4. 정수형은 부호와 승격 규칙까지 포함해서 이해해야 한다

C의 정수형은 단순하지 않다.  
아래 요소가 함께 작동한다.

- 타입 크기
- signed / unsigned 여부
- integer promotion
- usual arithmetic conversions

예를 들어 작은 정수형끼리 연산해도 실제 연산은 `int`로 승격되어 일어나는 경우가 많다.

예시:
```c
    unsigned char a = 200;
    unsigned char b = 100;
    int c = a + b;
```
겉으로는 `unsigned char + unsigned char`처럼 보이지만, 실제 계산은 보통 `int`로 승격된 뒤 수행된다.

즉 C의 타입은 선언된 모습만 보면 안 되고,  
연산 순간에 어떤 타입으로 바뀌는지도 함께 봐야 한다.

이 지점이 나중의 `integer promotion`, `signed/unsigned 변환`, `bitwise 연산` 문서로 이어진다.

---

### 5. 포인터도 하나의 타입이다

C에서 포인터는 "주소를 담는 특수 문법"이 아니라 엄연한 타입이다.

예시:
```c
    int *p;
    char *s;
    void *vp;
```
모두 포인터이지만 서로 다른 타입이다.  
이 차이는 단순 형식 차이가 아니라 다음을 결정한다.

- 역참조했을 때 몇 바이트를 읽는가
- 포인터 연산에서 한 칸 이동이 몇 바이트인가
- 어떤 객체를 가리키는 것으로 해석하는가

예를 들어 `int *p`에서 `p + 1`은 보통 4바이트 또는 그 구현의 `int` 크기만큼 이동한다.  
반면 `char *p`에서 `p + 1`은 항상 1바이트 단위다.

즉 포인터의 타입은 주소 그 자체보다, **그 주소를 어떻게 해석할지**를 결정한다.

---

### 6. 배열과 함수는 표현식에서 자주 다른 형태로 바뀐다

C의 타입 시스템은 선언 시 타입과 표현식에서의 동작이 다를 수 있다.

대표적으로:

- 배열은 많은 문맥에서 포인터로 decay한다
- 함수는 함수 포인터처럼 다뤄질 수 있다

예시:
```c
    int arr[10];
```
`arr`는 선언상으로는 "int 10개짜리 배열"이다.  
하지만 많은 표현식 문맥에서는 첫 번째 원소를 가리키는 `int *`처럼 동작한다.

이 때문에 C 타입 시스템은 "선언된 타입"과 "표현식에서의 변환된 타입"을 분리해서 이해해야 한다.

---

### 7. 구조체와 공용체는 사용자 정의 메모리 배치 도구다

구조체와 공용체는 단순히 여러 값을 묶는 문법이 아니다.

- `struct`는 여러 멤버를 순서대로 배치한다
- `union`은 같은 메모리를 여러 타입으로 겹쳐서 본다

예시:
```c
    struct Point {
        int x;
        int y;
    };
```
이 타입은 단순한 "객체 묶음"이 아니라,  
메모리에 `x`, `y`가 어떤 순서와 정렬 규칙으로 놓일지를 결정하는 타입이다.

즉 C의 타입 시스템은 사용자 정의 추상화 도구이면서 동시에 메모리 레이아웃 기술이기도 하다.

---

### 8. 타입은 컴파일러 최적화와도 연결된다

C에서 타입은 단지 사람을 위한 정보가 아니다.  
컴파일러는 타입 정보를 바탕으로 다음을 추론한다.

- 어떤 값이 같은 객체를 가리킬 가능성이 있는가
- 어떤 메모리 접근을 재배치해도 되는가
- 어떤 레지스터 보관이 안전한가
- 어떤 연산을 단순화할 수 있는가

즉 타입 시스템을 잘못 이해하면 단순한 문법 실수로 끝나지 않고,  
UB나 aliasing 문제를 통해 최적화 단계에서 예상 못 한 동작으로 이어질 수 있다.

이 때문에 C의 타입은 "컴파일을 통과하는지"보다 "컴파일러가 어떤 세계를 가정하는지"까지 포함해서 봐야 한다.

---

## 타입 분류를 한 번에 보는 큰 지도

C 타입 시스템은 아래처럼 잡아두면 좋다.

### 1. 기본 스칼라 타입

- 문자형
- 정수형
- 부동소수점형

### 2. 파생 타입

- 포인터
- 배열
- 함수

### 3. 집계 타입

- 구조체
- 공용체

### 4. 기타 분류

- 열거형
- 불완전 타입
- 한정자 추가 타입(`const`, `volatile`, `restrict`)

이 지도를 먼저 머릿속에 넣어 두면 이후 문서들이 정리된다.

- 포인터 문서는 파생 타입 이해로 연결된다
- 배열 decay 문서는 선언 타입과 표현식 변환의 차이로 연결된다
- 구조체 문서는 메모리 레이아웃으로 연결된다
- const 문서는 타입 한정자로 연결된다
- strict aliasing과 effective type 문서는 타입 해석 규칙으로 연결된다

## 자주 하는 오해

### 오해 1. 타입은 변수에 붙는 이름표일 뿐이다

아니다.  
타입은 메모리 해석 규칙, 연산 규칙, 변환 규칙, 최적화 가정까지 함께 결정한다.

---

### 오해 2. 같은 크기면 같은 타입처럼 다뤄도 된다

아니다.  
같은 4바이트라도 `int`, `float`, 포인터는 의미가 전혀 다르다.

---

### 오해 3. 선언만 보면 타입 동작을 다 알 수 있다

아니다.  
C에서는 배열 decay, integer promotion, usual arithmetic conversions 때문에  
표현식에서 실제로 사용되는 타입이 달라질 수 있다.

---

### 오해 4. 포인터는 전부 같은 종류다

아니다.  
`int *`, `char *`, `void *`, `int **`, `int (*)[10]`는 전혀 다른 타입이다.  
가리키는 대상, 연산 단위, 역참조 결과가 다르다.

## 실무 관점

실무에서 C 타입 시스템을 제대로 이해해야 하는 이유는 아래와 같다.

### 1. 메모리 버그의 상당수가 타입 해석 오류에서 시작된다

- 잘못된 캐스팅
- 잘못된 포인터 타입 사용
- 배열과 포인터 혼동
- signed/unsigned 혼합 연산
- 구조체 레이아웃 오해

이런 문제는 겉보기에는 작은 문법 실수처럼 보여도 실제로는 메모리 해석 모델을 잘못 잡은 경우가 많다.

---

### 2. 바이너리 포맷, 네트워크, 시스템 프로그래밍에서 필수다

C를 실제로 쓰는 영역은 보통 아래와 겹친다.

- OS / 커널
- 임베디드
- 드라이버
- 네트워크 프로토콜
- 파일 포맷 파서
- 고성능 런타임

이 영역들은 전부 "바이트를 어떤 타입의 값으로 볼 것인가"가 핵심이다.  
즉 타입 시스템 이해가 곧 시스템 이해다.

---

### 3. 이후 심화 주제의 기반이 된다

이 문서를 기준으로 다음 문서들이 자연스럽게 이어진다.

- 초기화와 대입은 왜 다른가
- 포인터의 포인터는 왜 필요한가
- 배열은 언제 포인터로 decay되는가
- sizeof는 무엇을 알려주고 무엇을 숨기는가
- strict aliasing은 왜 어려운가
- effective type은 무엇인가
- signed/unsigned 변환은 왜 위험한가
- integer promotion은 왜 헷갈리는가

즉 이 문서는 단독 주제가 아니라,  
앞으로의 C 전체 학습 축을 지탱하는 기반 문서다.

---

## 정리

C의 타입 시스템은 단순한 문법 체계가 아니다.  
그것은 메모리 위의 바이트를 값으로 해석하는 규칙이며, 연산과 변환의 기준이고, 컴파일러 최적화의 전제다.

C를 제대로 이해하려면 타입을 "변수의 종류"가 아니라 아래처럼 이해해야 한다.

- 메모리 해석 규칙
- 연산 가능 범위
- 암묵적 변환 규칙
- 객체와 포인터의 관계
- 컴파일러가 믿는 가정

이 관점을 잡고 나면 포인터, 배열, 구조체, promotion, aliasing 같은 주제들이 따로 떨어진 문법이 아니라 하나의 시스템으로 연결되기 시작한다.
