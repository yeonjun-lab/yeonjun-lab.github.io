## 정의

C에서 lvalue와 rvalue는 단순히 "왼쪽 값 / 오른쪽 값"이라는 위치 이름이 아니다.  
이 둘은 표현식이 **객체 자체를 가리키는가**, 아니면 **그 객체에서 얻어진 값으로 계산되는가**를 구분하는 개념이다.

핵심은 다음이다.

1. lvalue는 보통 어떤 객체를 식 안에서 가리키는 표현이다
2. rvalue는 보통 계산에 사용되는 값 자체를 뜻한다
3. 대입 가능성은 lvalue와 연결되지만, 모든 lvalue가 수정 가능한 것은 아니다
4. 배열, 함수, const 객체, 포인터 역참조를 이해할 때 이 구분이 매우 중요하다

즉 lvalue/rvalue를 이해한다는 것은  
단순히 대입문의 좌우를 보는 것이 아니라 **표현식이 객체를 나타내는지, 값을 나타내는지**를 구분하는 것이다.

---

## 먼저 큰 그림부터

많은 초보자는 이렇게 외운다.

- lvalue는 대입문 왼쪽
- rvalue는 대입문 오른쪽

이 설명은 입문용으로는 쓸 수 있지만, 정확하지는 않다.

예를 들어:
```c
    x = 10;
```
여기서 `x`는 lvalue처럼 보이고, `10`은 rvalue처럼 보인다.  
이건 맞다.

하지만 다음을 보자.
```c
    *p = 10;
```
여기서 lvalue는 `p`가 아니라 `*p`다.

또:
```c
    arr[2] = 5;
```
여기서 lvalue는 `arr`가 아니라 `arr[2]`다.

또:
```c
    const int x = 10;
```
`x`는 객체를 가리키는 표현이라는 점에서 lvalue처럼 이해해야 하지만, 수정 가능한 lvalue는 아니다.

즉 lvalue/rvalue는  
"왼쪽에 쓸 수 있느냐" 수준으로만 보면 곧바로 막히고,  
**객체와 값의 구분**으로 이해해야 정리된다.

---

## 핵심 개념

### 1. lvalue는 보통 객체를 가리키는 표현이다

예시:
```c
    int x = 10;
```
여기서 `x`는 메모리 어딘가에 있는 `int` 객체다.  
식 안에서 `x`를 쓰면, 그것은 단순 숫자 10이 아니라 **그 객체를 가리키는 표현**으로 볼 수 있다.

즉 `x`는 lvalue다.

이 말은 곧 다음을 뜻한다.

- `x`는 저장공간을 가진 객체와 연결된다
- 필요하면 그 객체의 값을 읽을 수 있다
- 경우에 따라 그 객체에 새 값을 쓸 수도 있다

즉 lvalue는 "값이 있다"보다  
**객체를 식 안에서 가리킨다**는 점이 핵심이다.

---

### 2. rvalue는 계산에 쓰이는 값이다

예시:
```c
    int x = 10;
    int y = x + 1;
```
여기서 `x + 1`은 하나의 계산 결과 값이다.  
이 표현식은 어떤 독립 객체를 직접 가리키는 것이 아니라, 계산 결과를 나타낸다.

즉 이런 식은 rvalue로 이해하면 된다.

마찬가지로:
```c
    10
    x + 1
    f()
    a < b
```
이런 것은 보통 값으로 계산되는 표현이다.

즉 rvalue는  
**식 평가 결과로 얻어진 값**이라는 관점이 핵심이다.

---

### 3. lvalue라고 해서 항상 대입 가능한 것은 아니다

이 부분이 매우 중요하다.

예시:
```c
    const int x = 10;
```
`x`는 객체를 가리키는 표현이라는 점에서 lvalue처럼 이해해야 한다.  
하지만:
```c
    x = 20;
```
은 허용되지 않는다.

왜냐하면 `x`는 수정 가능한 lvalue(modifiable lvalue)가 아니기 때문이다.

즉 lvalue와 대입 가능성은 완전히 같은 말이 아니다.

- lvalue: 객체를 가리키는 표현
- modifiable lvalue: 그중 수정 가능한 표현

이 차이를 이해해야 `const`와 함께 정리된다.

---

### 4. 대입문의 왼쪽에는 보통 modifiable lvalue가 와야 한다

예시:
```c
    int x = 10;
    x = 20;      // 가능
```
하지만:
```c
    10 = x;      // 불가
    (x + 1) = 3; // 불가
```
`10`이나 `x + 1`은 값으로 계산되는 rvalue이지,  
수정 가능한 객체를 가리키는 표현이 아니기 때문이다.

즉 대입 연산은 보통  
**수정 가능한 객체를 식으로 가리키는 표현**을 요구한다.

---

## 왜 `*p`가 lvalue인가

예시:
```c
    int x = 10;
    int *p = &x;

    *p = 20;
```
많은 초보자는 `p`가 포인터니까 대입 대상처럼 느낀다.  
하지만 실제로 값을 바꾸는 대상은 `p`가 아니라 `*p`다.

이유는 다음과 같다.

- `p`는 주소 값을 담은 포인터 변수
- `*p`는 그 주소가 가리키는 객체

즉 `*p`는 객체를 가리키는 표현이다.  
그래서 lvalue다.

이것은 포인터의 본질과 직접 연결된다.  
포인터는 값을 담고, 역참조 결과가 객체를 식으로 드러낸다.

---

## 왜 `arr[i]`도 lvalue인가

예시:
```c
    int arr[3] = {1, 2, 3};
    arr[1] = 10;
```
여기서 `arr[1]`은 배열 원소 객체 하나를 가리킨다.  
즉 `arr[1]`은 lvalue다.

이건 매우 중요하다.  
왜냐하면 배열 이름 `arr` 자체는 배열 객체 전체와 연결되지만,  
실제 개별 원소 수정은 `arr[i]`라는 lvalue를 통해 일어나기 때문이다.

즉 배열 접근은 단순 값 읽기가 아니라  
**배열 내부 특정 객체를 가리키는 표현**을 만든다.

---

## 왜 함수 호출 결과는 대체로 rvalue처럼 보이는가

예시:
```c
    int f(void) {
        return 42;
    }

    int x = f();
```
여기서 `f()`는 호출 결과 값이다.  
이건 보통 독립적인 수정 가능한 객체를 직접 가리키는 표현으로 보지 않는다.

따라서:
```c
    f() = 10;
```
같은 것은 허용되지 않는다.

즉 함수 호출 결과는 대체로 rvalue처럼 이해하는 편이 맞다.

---

## 왜 배열 이름은 특이한가

예시:
```c
    int arr[3];
```
배열 이름 `arr`는 단순 변수 이름처럼 보이지만,  
대입 대상이 되지는 않는다.
```c
    arr = something;   // 불가
```
그 이유는 배열은 식 안에서 독특한 규칙을 가지기 때문이다.

- 배열 객체와 연결된 표현이지만
- 많은 문맥에서 decay가 일어나고
- 수정 가능한 lvalue처럼 다뤄지지 않는다

즉 배열은 lvalue/rvalue를 배울 때 가장 좋은 반례 중 하나다.  
"객체와 연결된다"는 것과 "대입 가능하다"는 것이 다르다는 점을 강하게 보여 준다.

---

## 왜 괄호를 쳐도 rvalue는 lvalue가 되지 않는가

예시:
```c
    (x) = 10;      // 가능할 수 있음, x 자체를 감싼 것
    (x + 1) = 10;  // 불가
```
괄호는 보통 표현의 묶음 방식을 바꿀 뿐,  
그 표현이 본질적으로 객체를 가리키는지, 값인지까지 바꾸지는 않는다.

즉 `x + 1`은 괄호를 쳐도 여전히 계산 결과 값이다.  
객체를 가리키는 표현이 되지 않는다.

이 점은 "문법 모양"보다 **표현식 의미**가 중요하다는 것을 보여 준다.

---

## 초기화와 대입과도 어떻게 연결되는가

예시:
```c
    int x = 10;  // 초기화
    x = 20;      // 대입
```
이전 문서에서 초기화와 대입은 다르다고 했다.  
그 차이를 더 정교하게 보면, 대입은 보통 왼쪽에 modifiable lvalue를 요구한다.

즉:

- 초기화는 새 객체의 첫 상태 지정
- 대입은 기존 객체를 lvalue로 가리켜 새 값을 씀

따라서 lvalue 개념은 대입 문서를 더 깊게 이해하는 데 필요하다.

---

## 왜 `const`와 바로 연결되는가

예시:
```c
    const int x = 10;
```
`x`는 식 안에서 객체를 가리키는 표현으로 볼 수 있으므로 lvalue처럼 이해해야 한다.  
하지만 `const` 때문에 수정 가능한 lvalue는 아니다.

즉 `const` 문서를 제대로 이해하려면 다음 구분이 필요하다.

- lvalue인가
- modifiable lvalue인가

이 구분이 없으면  
"const 객체는 lvalue가 아닌가?" 같은 혼란이 생긴다.

---

## 메모리 모델 관점에서 왜 중요한가

이 문서가 memory-model 축에 있는 이유는  
lvalue/rvalue가 단순 문법 분류가 아니라  
**식과 객체 사이의 관계를 설명하는 개념**이기 때문이다.

C는 객체, 저장공간, 값, 타입이 모두 중요하다.  
그리고 식은 그 객체를 직접 나타낼 수도 있고, 그 객체에서 얻어진 값만 나타낼 수도 있다.

즉 lvalue/rvalue는  
**식이 메모리 위 객체와 어떤 관계를 맺는가**를 설명하는 기본 언어다.

이 관점이 잡혀야:

- 대입
- 포인터 역참조
- 배열 원소 접근
- const 제약
- 함수 반환값
- aliasing 논의

가 하나로 연결된다.

---

## 자주 하는 오해

### 오해 1. lvalue는 무조건 대입문 왼쪽이다

아니다.  
lvalue는 본질적으로 객체를 가리키는 표현이다.  
대입문 왼쪽에 자주 오지만, 그게 정의는 아니다.

---

### 오해 2. lvalue면 무조건 수정할 수 있다

아니다.  
`const` 객체처럼 lvalue이지만 수정 가능한 lvalue가 아닌 경우가 있다.

---

### 오해 3. 포인터 변수 `p`가 lvalue면 `*p`는 값일 뿐이다

아니다.  
`*p`는 가리키는 객체를 나타내는 표현이므로 대표적인 lvalue다.

---

### 오해 4. 배열 이름은 일반 변수처럼 대입할 수 있다

아니다.  
배열은 객체와 연결되지만, 수정 가능한 lvalue처럼 자유롭게 대입되는 구조가 아니다.

---

### 오해 5. 괄호를 치면 값 식도 객체 식이 된다

아니다.  
괄호는 그룹화일 뿐, 계산 결과 식을 객체 표현으로 바꾸지 않는다.

---

## 실무 관점

### 1. 포인터/배열/const를 제대로 읽으려면 필수다

실무 C에서 자주 헷갈리는 대부분의 코드가 이 축에 걸린다.

- `*p = ...`
- `arr[i] = ...`
- `const T *`
- 함수 반환값 사용
- 매크로 식 대입 가능성

즉 lvalue/rvalue는 이론 용어가 아니라,  
실제 코드를 정확히 읽기 위한 도구다.

---

### 2. API 설계와도 연결된다

어떤 함수가 값을 돌려주는지,  
기존 객체를 수정하는지,  
포인터를 통해 외부 객체를 바꾸는지 같은 것은 전부 객체/값 구분과 연결된다.

즉 lvalue/rvalue 구분은  
식 평가뿐 아니라 함수 인터페이스 이해에도 도움이 된다.

---

### 3. 컴파일러 에러 메시지를 읽기 쉬워진다

예를 들어 "lvalue required" 같은 에러는  
막연히 문법 오류가 아니라 다음 뜻으로 읽을 수 있다.

- 지금 이 자리에 객체를 가리키는 수정 가능한 표현이 와야 하는데
- 너는 계산 결과 값만 주고 있다

이걸 이해하면 에러가 훨씬 덜 추상적으로 느껴진다.

---

### 4. 다음 문서들과 직접 연결된다

이 문서는 아래와 자연스럽게 이어진다.

- const는 무엇을 제한하는가
- 초기화와 대입은 왜 다른가
- 배열은 언제 포인터로 decay되는가
- 포인터의 포인터는 왜 필요한가

즉 식 의미와 객체 모델을 연결하는 핵심 문서다.

---

## 정리

C에서 lvalue와 rvalue는 단순히 왼쪽 값과 오른쪽 값이 아니다.  
이 둘은 표현식이 객체를 직접 가리키는지, 아니면 계산 결과 값으로 쓰이는지를 구분하는 개념이다.

핵심은 다음과 같다.

- lvalue는 보통 객체를 가리키는 표현이다
- rvalue는 보통 계산 결과 값이다
- 모든 lvalue가 수정 가능한 것은 아니다
- `*p`, `arr[i]`는 대표적인 lvalue다
- `const`, 배열, 함수 호출 결과를 이해할 때 이 구분이 중요하다

즉 lvalue/rvalue를 이해한다는 것은  
문법 위치를 외우는 것이 아니라 **식과 메모리 위 객체의 관계를 읽는 법**을 익히는 것이다.

이 관점이 잡히면 대입, 포인터, 배열, const, 함수 식이 훨씬 더 정확하게 연결된다.
