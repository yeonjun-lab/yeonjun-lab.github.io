## 정의

bitwise 연산과 shift는 정수를 "숫자"로만 보지 않고 **비트 패턴**으로 다루는 연산이다.  
즉 덧셈·뺄셈처럼 값의 산술 의미를 중심으로 계산하는 것이 아니라, 각 비트를 마스크하고, 이동시키고, 조합하고, 분리하는 데 쓰인다.

핵심은 다음이다.

1. bitwise 연산은 정수의 각 비트를 직접 다룬다
2. shift는 비트 위치를 좌우로 이동시킨다
3. C에서는 이 연산들이 integer promotion, signed/unsigned 규칙과 강하게 얽힌다
4. 따라서 겉보기보다 훨씬 저수준적이고, 실무에서는 데이터 표현/포맷/플래그/프로토콜 처리와 직접 연결된다

즉 bitwise와 shift를 이해한다는 것은  
단순히 연산자 사용법을 외우는 것이 아니라 **정수의 비트 표현을 조작하는 사고 방식**을 이해하는 것이다.

---

## 먼저 큰 그림부터

많은 초보자는 bitwise 연산을 이렇게 배운다.

- `&`는 AND
- `|`는 OR
- `^`는 XOR
- `~`는 NOT
- `<<`, `>>`는 왼쪽/오른쪽 시프트

이 설명 자체는 맞다.  
하지만 실무에서는 이걸 "연산자 이름" 수준으로만 알면 거의 도움이 안 된다.

실제로 중요한 것은 다음이다.

- 어떤 비트를 꺼내는가
- 어떤 비트를 켜고 끄는가
- 바이트/필드를 어떻게 조합하는가
- 플래그를 어떻게 저장하는가
- 직렬화된 데이터에서 필요한 비트 영역을 어떻게 추출하는가

즉 bitwise 연산은 C에서  
**메모리 표현과 외부 데이터 포맷을 다루는 핵심 도구**다.

---

## 핵심 개념

### 1. bitwise 연산은 "정수 값"이 아니라 "비트 패턴"을 조작한다

예시:
```c
    unsigned int x = 0b1100;
    unsigned int y = 0b1010;
    unsigned int z = x & y;
```
이 연산은 "`x`와 `y`를 산술적으로 결합"하는 것이 아니라,  
각 비트를 자리별로 비교해서 새 비트 패턴을 만든다.

즉 bitwise 연산은 다음처럼 봐야 한다.

- AND: 특정 비트만 남기기
- OR: 특정 비트 켜기
- XOR: 다름을 표시하거나 토글하기
- NOT: 비트 뒤집기

즉 숫자를 값으로 다루기보다  
**비트들의 집합/상태/플래그**로 다루는 연산이다.

---

### 2. shift는 비트 위치를 움직인다

예시:
```c
    unsigned int x = 1;
    unsigned int y = x << 3;
```
이 연산은 1을 8로 "곱하는 것처럼" 보일 수 있다.  
하지만 bitwise 관점에서는:

- 1의 비트를
- 왼쪽으로 3칸 이동

시키는 것이다.

즉 shift는 본질적으로 **비트 자리 이동**이다.  
산술적 의미는 그 결과로 따라올 수 있을 뿐이다.

---

### 3. bitwise는 정수형에서만 의미가 있다

C에서 bitwise 연산은 보통 정수형에서 사용한다.

- `char`
- `short`
- `int`
- `long`
- `long long`
- 각 unsigned 버전
- 열거형 등 정수 계열

즉 부동소수점 값을 bitwise 연산으로 직접 다루는 것이 아니라,  
정수형 값의 비트 표현을 조작하는 방식으로 사용한다.

부동소수점 비트를 보고 싶다면 보통 `memcpy`나 다른 방식으로 정수형 쪽으로 옮겨서 다룬다.  
이 점은 앞 문서들과 직접 연결된다.

---

## 왜 integer promotion과 같이 봐야 하는가

이 주제는 앞 문서와 강하게 연결된다.

예시:
```c
    unsigned char x = 0x80;
    int y = ~x;
```
겉으로 보면 "8비트 값 뒤집기"처럼 보인다.  
하지만 실제로는 `x`가 먼저 promotion된 뒤 비트 반전이 적용된다.

즉 다음이 중요하다.

- `x`는 저장공간상 1바이트일 수 있다
- 하지만 식에서는 더 큰 정수형으로 승격될 수 있다
- `~`는 그 승격된 타입 전체 비트를 뒤집는다

그래서 결과는 초보자 직관과 쉽게 달라진다.

즉 bitwise 코드는 항상 다음 질문을 먼저 해야 한다.

- 지금 이 값은 저장 타입이 무엇인가
- 연산 시점에는 어떤 타입으로 promotion되는가
- 실제로 몇 비트 폭을 가진 식으로 계산되는가

---

## 왜 signed/unsigned와도 같이 봐야 하는가

bitwise와 shift는 signed/unsigned 차이에 매우 민감하다.

### 1. signed는 값 의미가 우선이고, 부호 비트 해석이 따라온다

signed 정수에 bitwise를 쓰면,  
사람은 종종 "그냥 비트를 만지면 되겠지"라고 생각한다.

하지만 signed 값은 음수/양수 해석이 함께 들어가므로,  
특히 오른쪽 shift 같은 연산에서 기대가 달라질 수 있다.

### 2. unsigned는 비트 조작 대상으로 더 예측 가능하다

실무에서 플래그, 마스크, 바이트 조합, 시프트 연산을 할 때  
unsigned 계열을 더 자주 권장하는 이유가 여기에 있다.

즉 bitwise/shift는  
**값 의미보다 비트 의미를 우선하는 연산**이므로, signed보다 unsigned에서 더 덜 놀랍게 동작하는 경우가 많다.

---

## bitwise 연산은 실무에서 어디에 쓰이는가

### 1. 플래그 저장

예시:
```c
    enum {
        FLAG_READ  = 1 << 0,
        FLAG_WRITE = 1 << 1,
        FLAG_EXEC  = 1 << 2
    };

    unsigned int flags = 0;
    flags |= FLAG_READ;
    flags |= FLAG_WRITE;
```
여기서는 하나의 정수 안에 여러 boolean 상태를 비트로 packed해서 저장한다.

즉 bitwise는 메모리를 아끼는 기술이라기보다,  
**상태를 조합 가능한 플래그 집합으로 표현하는 도구**다.

---

### 2. 특정 비트 검사

예시:
```c
    if (flags & FLAG_WRITE) {
        /* write enabled */
    }
```
이건 "`FLAG_WRITE` 비트가 켜져 있는가"를 확인하는 전형적인 패턴이다.

즉 AND는 "공통 비트만 남긴다"는 성질 때문에  
마스크 검사에 매우 자주 쓰인다.

---

### 3. 특정 비트 켜기 / 끄기 / 토글

예시:
```c
    flags |= FLAG_READ;      // 켜기
    flags &= ~FLAG_READ;     // 끄기
    flags ^= FLAG_READ;      // 토글
```
이 패턴은 시스템 프로그래밍, 드라이버, 게임 입력 상태, 권한 비트 등에서 매우 흔하다.

---

### 4. 바이트 조합과 분해

예시:
```c
    unsigned int x =
        ((unsigned int)buf[0] << 24) |
        ((unsigned int)buf[1] << 16) |
        ((unsigned int)buf[2] << 8)  |
        ((unsigned int)buf[3]);
```
이 코드는 4바이트를 하나의 32비트 정수처럼 조합하는 전형적인 패턴이다.

반대로:
```c
    unsigned char b0 = (x >> 24) & 0xFF;
    unsigned char b1 = (x >> 16) & 0xFF;
    unsigned char b2 = (x >> 8)  & 0xFF;
    unsigned char b3 = x & 0xFF;
```
이렇게 특정 바이트를 추출할 수도 있다.

즉 bitwise와 shift는  
**바이트 스트림과 정수 표현을 오가는 핵심 도구**다.

---

### 5. 프로토콜/파일 포맷 필드 추출

예를 들어 한 바이트 안에:

- 상위 3비트는 타입
- 하위 5비트는 길이

처럼 packed된 필드가 있을 수 있다.

예시:
```c
    unsigned char byte = ...;
    unsigned char type = (byte >> 5) & 0x07;
    unsigned char len  = byte & 0x1F;
```
즉 bitwise는 packed field를 다루는 데 매우 중요하다.

---

## shift에서 특히 조심할 점

### 1. 왼쪽 shift는 비트를 왼쪽으로 민다

예시:
```c
    unsigned int x = 1;
    unsigned int y = x << 3;
```
이건 흔히 8처럼 보인다.  
하지만 본질은 비트 이동이다.

문제는 signed 값이나 범위를 넘는 이동에서 생각보다 위험해질 수 있다는 점이다.

---

### 2. 오른쪽 shift는 부호와 해석 문제를 동반한다

예시:
```c
    int x = -4;
    int y = x >> 1;
```
이 결과를 직관적으로 "반으로 나눈다"라고 보면 위험하다.  
오른쪽 shift는 산술 나눗셈이 아니라 비트 이동이며, signed 음수에서의 의미는 구현 차이를 의식해야 할 수 있다.

즉 실무에서 shift는  
산술 대체 수단으로 보기보다 **비트 조작 도구**로 보는 편이 안전하다.

---

### 3. 너무 많이 shift하면 안 된다

예시:
```c
    unsigned int x = 1;
    unsigned int y = x << 40;
```
타입 비트폭 이상으로 shift하는 것은 매우 위험하다.  
즉 shift 횟수는 항상 대상 타입 비트폭 안에 있어야 한다.

이 부분은 UB와도 연결된다.

---

## 왜 마스크가 항상 같이 다니는가

bitwise 코드에서 shift와 마스크는 거의 함께 나온다.

예시:
```c
    unsigned int field = (x >> 8) & 0xFF;
```
이유는 단순하다.

- shift는 원하는 비트 영역을 특정 위치로 이동
- mask는 그중 필요한 부분만 남김

즉 shift만 하면 주변 비트가 섞여 있을 수 있고,  
mask가 그 결과를 정리한다.

그래서 실무에서 비트 추출 패턴은 보통:

- shift
- mask

순서로 생각하면 된다.

---

## 왜 `1 << n`도 조심해야 하는가

예시:
```c
    int flag = 1 << 31;
```
이건 흔히 보이는 코드지만, signed `int` 문맥에서는 주의가 필요하다.  
왜냐하면 리터럴 `1` 자체가 기본적으로 signed 정수로 시작하기 때문이다.

실무에서는 이런 문제를 줄이기 위해:
```c
    1u << n
```
처럼 unsigned 리터럴을 더 자주 쓰는 습관이 중요하다.

즉 bitwise/shift에서는  
**리터럴 타입까지 같이 의식해야 한다.**

---

## 왜 bitwise 코드는 가독성이 쉽게 나빠지는가

bitwise 코드는 한 줄로 강력한 일을 하지만, 동시에 의도가 숨어들기 쉽다.

예시:
```c
    x = ((x >> 3) & 0x1F) | ((y & 0x07) << 5);
```
이 코드는 정확할 수 있지만, 설명 없이 두면 나중에 읽기 어렵다.

그래서 실무에서는 다음이 중요하다.

- 상수에 이름 붙이기
- 필드 의미를 주석으로 남기기
- 마스크/shift 폭을 명시적으로 표현하기
- 헬퍼 함수로 의도를 분리하기

즉 bitwise는 "짧은 코드"보다 **해석 가능한 코드**가 중요하다.

---

## 자주 하는 오해

### 오해 1. bitwise는 그냥 빠른 산술 연산이다

아니다.  
본질은 비트 조작이다.  
산술처럼 보이는 효과가 따라올 수는 있지만, 그게 핵심이 아니다.

---

### 오해 2. 작은 정수형 bitwise는 그 작은 비트폭 안에서 계산된다

아니다.  
integer promotion 때문에 더 큰 타입으로 올라간 뒤 계산되는 경우가 많다.

---

### 오해 3. signed에서도 unsigned처럼 자연스럽게 shift하면 된다

아니다.  
signed는 부호 해석이 얽히므로 더 조심해야 한다.

---

### 오해 4. `1 << n`은 언제나 안전한 플래그 생성 방식이다

아니다.  
리터럴 타입과 shift 범위를 함께 봐야 한다.  
보통 `1u << n`이 더 낫다.

---

### 오해 5. shift는 곱셈/나눗셈의 빠른 대체다

항상 그렇게 보면 위험하다.  
실무에서는 비트 이동으로 이해하는 편이 맞고, signed 값에서는 더욱 조심해야 한다.

---

## 실무 관점

### 1. 플래그/권한/상태 비트 표현에서 매우 중요하다

운영체제, 네트워크, 드라이버, 런타임, 파서 코드에서는  
여러 상태를 하나의 정수에 넣고 bitwise로 다루는 일이 매우 흔하다.

즉 bitwise는 실무 시스템 코드의 기본 문법 중 하나다.

---

### 2. 직렬화/프로토콜/파일 포맷과 직접 연결된다

패킷 헤더, 바이너리 파일 헤더, 압축 포맷, 이미지 포맷 등은  
필드가 비트 단위로 packed된 경우가 많다.

즉 bitwise를 이해하지 못하면  
바이너리 레벨 parsing/packing 코드가 거의 불가능하다.

---

### 3. unsigned 중심으로 설계하는 편이 더 안전한 경우가 많다

비트 패턴 조작이 핵심인 코드에서는  
값의 부호 의미보다 비트 의미가 우선이므로 unsigned 계열이 더 예측 가능하다.

즉 signed 정수에 bitwise를 쓰는 순간  
"나는 값 조작을 하고 있나, 표현 조작을 하고 있나"를 다시 점검해야 한다.

---

### 4. promotion과 mask를 항상 같이 떠올려야 한다

실무 bitwise 코드를 읽을 때는 다음 순서가 좋다.

1. 이 값은 저장 타입이 무엇인가  
2. 연산 시 어떤 타입으로 promotion되는가  
3. shift는 몇 비트 폭에서 일어나는가  
4. 최종적으로 어떤 mask를 적용하는가

이 사고 순서를 가지면 많은 버그를 줄일 수 있다.

---

### 5. 다음 문서들과 직접 연결된다

이 문서는 아래와 바로 이어진다.

- integer promotion은 왜 헷갈리는가
- signed/unsigned 변환은 왜 위험한가
- endianness는 왜 중요한가
- 직렬화와 역직렬화는 왜 단순 memcpy가 아닌가

즉 정수 규칙, 바이너리 표현, 직렬화 축을 연결하는 핵심 문서다.

---

## 정리

bitwise 연산과 shift는 정수를 숫자라기보다 비트 패턴으로 다루는 연산이다.

핵심은 다음과 같다.

- AND/OR/XOR/NOT은 비트 조작 도구다
- shift는 비트 위치 이동이다
- 실무에서는 플래그, 마스크, 필드 추출, 바이트 조합에 자주 쓰인다
- 하지만 integer promotion과 signed/unsigned 규칙 때문에 겉보기보다 훨씬 조심해야 한다
- 특히 unsigned 중심 사고, mask 사용, shift 범위 확인이 중요하다

즉 bitwise와 shift를 이해한다는 것은  
연산자 이름을 외우는 것이 아니라 **값을 비트 표현으로 보고 조작하는 실무적 사고 방식**을 익히는 것이다.

이 관점이 잡히면 C의 바이너리 포맷 처리, 프로토콜 파싱, 플래그 설계가 하나의 체계로 연결되기 시작한다.
