## 정의

`const`는 C에서 "이 값은 절대 변하지 않는다"는 마법 선언이 아니라,  
**특정 경로를 통해 그 객체를 수정하는 것을 금지하는 타입 한정자(type qualifier)**다.

핵심은 다음이다.

1. `const`는 객체 그 자체보다 **그 객체를 다루는 타입**에 붙는다
2. 따라서 "무엇이 const인가"를 항상 타입 구조 안에서 읽어야 한다
3. `const`는 특히 포인터와 결합할 때 의미가 단계별로 갈린다
4. `const`가 붙었다고 값이 물리적으로 ROM에 들어간다거나, 언어가 모든 변화를 자동 차단하는 것은 아니다

즉 `const`를 이해한다는 것은  
단순히 "읽기 전용"이라고 외우는 것이 아니라 **어떤 접근 경로가 수정 금지인지**를 정확히 읽는 것이다.

---

## 먼저 큰 그림부터

많은 초보자는 `const`를 이렇게 이해한다.

- const면 값이 절대 안 바뀜
- 상수 같은 것
- 그냥 "읽기 전용" 표시

이 설명은 반쯤만 맞다.

예를 들어:
```c
    const int x = 10;
```
이 경우 `x`는 수정할 수 없는 객체처럼 보인다.  
이건 대체로 맞다.

하지만 포인터가 들어오면 바로 헷갈린다.
```c
    const int *p;
    int * const q;
    const int * const r;
```
셋 다 `const`가 들어가지만 의미가 다르다.

즉 `const`는  
"이 선언 전체가 읽기 전용이다"처럼 뭉뚱그리면 안 되고,  
**타입 구조에서 정확히 어디를 수식하는가**를 읽어야 한다.

---

## 핵심 개념

### 1. `const`는 값을 "상수화"하는 것이 아니라 수정 경로를 제한한다

예시:
```c
    const int x = 10;
```
이 코드는 보통 `x`를 직접 수정하지 못하게 만든다.
```c
    x = 20;   // 오류
```
하지만 중요한 것은 `const`가 "우주적으로 값 불변"을 선언하는 것이 아니라,  
해당 선언을 통해 얻은 타입 경로에서 수정이 금지된다는 점이다.

즉 `const`는 메모리 셀에 스티커를 붙이는 개념이라기보다,  
**그 객체를 다루는 타입 계약**에 가깝다.

---

### 2. `const`는 타입 한정자다

C에서 `const`는 `int`, `char`, 포인터, 배열 요소 타입 등과 결합한다.

예시:
```c
    const int x;
    const char *s;
    int * const p;
```
이 셋은 모두 `const`를 쓰지만, 한정되는 대상이 다르다.

즉 `const`를 볼 때는 항상 물어야 한다.

- const가 **무엇을 수식하는가**
- 수정이 금지되는 대상이 값인가, 포인터가 가리키는 대상인가, 포인터 변수 자체인가

이 질문을 하지 않으면 포인터 선언에서 거의 반드시 헷갈린다.

---

### 3. `const`는 포인터와 결합할 때 단계별로 읽어야 한다

#### `const int *p`

이건 보통 이렇게 읽는다.

- `p`는 `const int`를 가리키는 포인터
- 즉 `*p`를 통해 정수를 수정할 수 없음
- 하지만 `p` 자체는 다른 곳을 가리키도록 바꿀 수 있음

예시:
```c
    const int a = 10;
    const int b = 20;
    const int *p = &a;

    p = &b;      // 가능
    *p = 30;     // 불가
```
즉 **가리키는 대상은 수정 불가, 포인터는 이동 가능**이다.

---

#### `int * const p`

이건 다르다.

- `p`는 const 포인터
- 즉 포인터 변수 자체를 바꿀 수 없음
- 하지만 `p`가 가리키는 정수는 수정 가능할 수 있음

예시:
```c
    int x = 10;
    int y = 20;
    int * const p = &x;

    *p = 30;     // 가능
    p = &y;      // 불가
```
즉 **포인터는 고정, 대상은 수정 가능**이다.

---

#### `const int * const p`

이건 둘 다 걸려 있다.

- `p` 자체를 바꿀 수 없음
- `*p`를 통해 대상도 수정할 수 없음

즉 **포인터도 고정, 대상도 수정 불가**다.

---

## 왜 이렇게 헷갈리는가

### 1. `const`가 영어식으로 읽히지 않기 때문이다

초보자는 `const int *p`를 보면  
그냥 "const가 있으니까 전부 const"처럼 느낀다.

하지만 C 선언은 영어 문장처럼 읽는 것보다  
**식 구조처럼 읽는 편이 더 정확하다.**

예를 들어:
```c
    int * const p
```
이 선언에서 `p` 바로 옆의 `const`는  
포인터 자체를 한정한다.

반면:
```c
    const int *p
```
여기서는 `int` 쪽이 한정된다.

즉 `const`는 단어 뜻보다 **선언 구조**로 읽어야 한다.

---

### 2. "값이 안 바뀐다"와 "이 경로로 못 바꾼다"를 혼동하기 때문이다

예를 들어 어떤 객체가 원래 non-const였고,  
그걸 `const int *`로 가리킬 수도 있다.
```c
    int x = 10;
    const int *p = &x;
```
이 경우 `p`를 통해서는 `x`를 수정할 수 없다.  
하지만 `x` 자체가 본질적으로 const 객체가 된 것은 아니다.
```c
    x = 20;   // 가능
```
즉 `const`는 때로는 객체의 본질적 성질처럼 보이지만,  
실제로는 **특정 타입 경로에서의 수정 가능성 제약**으로 읽는 편이 더 정확하다.

---

## 함수 인자에서 왜 중요한가

### 1. 읽기 전용 의도를 명확하게 표현한다

예시:
```c
    void print_array(const int *arr, size_t n);
```
이 시그니처는 다음 의미를 준다.

- 함수는 `arr`가 가리키는 원소들을 읽을 수 있다
- 하지만 수정하지 않겠다는 계약을 표현한다

즉 `const`는 단순 문법이 아니라  
**API의 의도와 부작용 범위**를 드러내는 도구다.

---

### 2. 호출자 입장에서 더 넓은 객체를 안전하게 넘길 수 있게 한다

예시:
```c
    void print_name(const char *s);
```
이런 함수는 문자열을 읽기만 하겠다는 뜻이다.  
따라서 문자열 리터럴, const 데이터, 일반 버퍼 모두를 더 안전하게 받을 수 있다.

즉 `const`는 함수 사용 범위를 넓히고,  
잘못된 수정 가능성을 줄인다.

---

## 문자열과 왜 자주 같이 나오나

예시:
```c
    const char *s = "hello";
```
문자열 리터럴은 수정 가능한 char 배열처럼 다루면 위험하다.  
그래서 실무적으로는 `const char *`로 받는 것이 자연스럽다.

즉 문자열 API에서 `const char *`가 매우 흔한 이유는  
"문자열을 읽기만 할 것"이라는 의도와  
리터럴 수정 금지 성격이 잘 맞기 때문이다.

따라서 `const`는 문자열 처리 문서와도 직접 연결된다.

---

## 배열과는 어떻게 연결되는가

배열 자체에 대한 직접 대입은 원래 불가능하지만,  
배열 원소 타입에 `const`가 붙을 수는 있다.

예시:
```c
    const int arr[3] = {1, 2, 3};
```
이 경우 의미는 다음과 같다.

- `arr` 원소들은 `const int`
- 따라서 `arr[i]`를 통해 수정할 수 없음

즉 `const`는 배열 전체를 "특별한 객체"로 만드는 것이 아니라,  
**배열 원소 타입을 읽기 전용으로 제한**하는 방식으로 작동한다.

---

## `const`가 메모리 배치를 바꾸는가

기초적으로는 이렇게 생각하면 된다.

- `const`는 우선 타입 한정자다
- 즉 핵심은 수정 가능성 제약이다
- 메모리 배치 자체를 바꾸는 개념으로 먼저 이해하면 안 된다

물론 구현에 따라 const 객체가 더 제한적인 영역에 놓일 수는 있다.  
하지만 언어를 이해하는 첫 관점은 그게 아니다.

즉 `const`의 본질은 "어디에 저장되나"보다  
**어떻게 접근 가능한가**다.

---

## `const` 캐스팅은 왜 조심해야 하는가

예시:
```c
    const int x = 10;
    int *p = (int *)&x;
    *p = 20;
```
겉보기에는 캐스팅했으니 수정할 수 있을 것처럼 보인다.  
하지만 이런 코드는 매우 위험하다.

이유는 간단하다.

- `const` 제약을 억지로 벗겨서
- 원래 수정하지 말아야 할 객체를
- 강제로 수정하려는 것이기 때문이다

즉 `const` 캐스팅은  
문법상 가능해 보일 수 있어도 언어 의미와 충돌하기 쉽다.

실무에서는 "왜 이 cast가 필요한가?"를 매우 엄격하게 봐야 한다.

---

## 왜 포인터의 포인터와 만나면 더 어려워지는가

예시:
```c
    const char **pp;
```
이런 선언이 어려운 이유는 `const`가 한 단계가 아니라  
여러 간접 참조 단계에 걸쳐 영향을 줄 수 있기 때문이다.

즉 이중 포인터 이상에서는 항상 단계별로 물어야 한다.

- `pp` 자체를 바꿀 수 있는가
- `*pp`를 바꿀 수 있는가
- `**pp`를 바꿀 수 있는가

이 구조를 제대로 읽지 못하면  
잘못된 타입 변환과 안전하지 않은 대입이 쉽게 나온다.

---

## 자주 하는 오해

### 오해 1. `const`는 값을 물리적으로 절대 못 바꾸게 하는 마법이다

아니다.  
우선은 타입 한정자다.  
즉 특정 접근 경로에서 수정이 금지된다는 계약으로 보는 것이 맞다.

---

### 오해 2. `const int *p`면 `p`도 const다

아니다.  
이 경우 보통 const인 것은 `*p` 쪽이다.  
`p` 자체는 다른 곳을 가리키게 바꿀 수 있다.

---

### 오해 3. `int * const p`와 `const int *p`는 비슷하다

아니다.  
하나는 포인터 자체를 고정하고,  
다른 하나는 가리키는 대상을 읽기 전용으로 만든다.

---

### 오해 4. `const`는 그냥 스타일 문제다

아니다.  
API 계약, 부작용 통제, 잘못된 수정 방지, 가독성, 최적화 힌트까지 연결되는 중요한 도구다.

---

### 오해 5. cast로 const를 없애면 안전하게 수정할 수 있다

아니다.  
원래 const로 다뤄져야 하는 객체를 강제로 수정하는 것은 매우 위험하다.

---

## 실무 관점

### 1. `const`는 문서화가 아니라 타입 시스템에 의한 계약이다

함수에 `const`를 붙이면 단순 주석보다 훨씬 강하다.

예시:
```c
    void log_message(const char *msg);
```
이건 "읽기만 한다"는 의도를 타입 수준에서 명시한다.

즉 `const`는 실무에서  
**부작용 경계를 타입으로 표현하는 도구**다.

---

### 2. 읽기 전용 입력은 적극적으로 const를 붙이는 편이 좋다

입력을 수정하지 않을 함수라면 `const`를 붙이는 것이 좋다.  
그러면 다음 이점이 있다.

- 실수로 수정하는 코드 방지
- 문자열 리터럴 같은 입력도 더 자연스럽게 받음
- API 의도가 명확해짐

즉 `const`는 보수적 장식이 아니라  
인터페이스 품질을 높이는 장치다.

---

### 3. 포인터 선언에서는 "무엇이 const인가"를 소리 내어 읽는 습관이 중요하다

예를 들어:

- `const int *p` → p는 const int를 가리킨다
- `int * const p` → p는 const 포인터다
- `const int * const p` → p는 const int를 가리키는 const 포인터다

이 식으로 읽으면 혼란이 줄어든다.

---

### 4. 다음 문서들과 직접 연결된다

이 문서는 아래와 자연스럽게 이어진다.

- lvalue와 rvalue는 C에서 어떻게 이해해야 하는가
- 포인터의 포인터는 왜 필요한가
- 문자와 문자열은 왜 구분해서 이해해야 하는가
- 초기화와 대입은 왜 다른가

즉 `const`는 타입 시스템, 포인터 모델, API 설계 축을 연결하는 핵심 문서다.

---

## 정리

`const`는 C에서 단순히 "상수"를 뜻하는 것이 아니라,  
특정 타입 경로를 통해 객체를 수정하는 것을 금지하는 타입 한정자다.

핵심은 다음과 같다.

- `const`는 타입에 붙는다
- 무엇이 const인지 선언 구조로 읽어야 한다
- 포인터와 결합하면 대상과 포인터 자체를 구분해야 한다
- 함수 인자에서는 읽기 전용 계약을 표현하는 데 매우 중요하다
- cast로 const를 억지로 벗기는 것은 매우 위험하다

즉 `const`를 제대로 이해한다는 것은  
"읽기 전용"이라는 말만 외우는 것이 아니라 **수정 가능성이 타입 구조 안에서 어떻게 제한되는가**를 읽을 수 있게 되는 것이다.

이 관점이 잡히면 포인터 선언, 문자열 API, 함수 시그니처, 안전한 인터페이스 설계가 훨씬 더 선명해진다.
