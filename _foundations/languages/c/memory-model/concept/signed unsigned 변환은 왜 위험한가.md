## 정의

C에서 signed/unsigned 변환은 단순히 "음수 가능 여부가 바뀌는 것"이 아니다.  
이 변환은 **값의 해석 방식, 비교 결과, 산술 결과, 경계 검사, 루프 조건**까지 바꿀 수 있기 때문에 위험하다.

핵심은 다음이다.

1. signed와 unsigned는 같은 비트폭이어도 의미가 다르다
2. 둘을 섞으면 C의 정수 변환 규칙이 개입한다
3. 그 결과 사람의 직관과 다른 비교·계산이 쉽게 나온다
4. 특히 길이, 인덱스, 에러 코드, 바이트 수를 함께 다룰 때 문제가 자주 생긴다

즉 signed/unsigned 문제를 이해한다는 것은  
단순히 타입 이름 차이를 아는 것이 아니라 **C가 정수를 어떤 규칙으로 변환해서 계산하는가**를 이해하는 것이다.

---

## 먼저 큰 그림부터

처음 이 문제를 보면 이렇게 생각하기 쉽다.

- `int`는 그냥 정수
- `unsigned int`는 음수가 없는 정수
- 필요하면 서로 섞어 써도 되겠지

하지만 C에서는 이 섞임이 매우 중요하다.

예를 들어:
```c
    int a = -1;
    unsigned int b = 1;

    if (a < b) {
        /* ... */
    }
```
사람 직관으로는 `-1 < 1`이므로 참 같아 보인다.  
하지만 C에서는 이 비교가 그렇게 단순하지 않다.  
왜냐하면 비교 전에 정수 변환 규칙이 적용될 수 있기 때문이다.

즉 문제의 핵심은 숫자 자체보다  
**비교 직전에 각 값을 어떤 타입으로 맞춘 뒤 계산하느냐**에 있다.

---

## 핵심 개념

### 1. signed와 unsigned는 같은 비트폭이어도 다른 값 체계다

예를 들어 32비트라고 가정하면:

- `signed int`는 대략 음수와 양수를 함께 표현
- `unsigned int`는 0 이상만 표현

즉 같은 32비트를 써도 값의 해석 범위가 다르다.

이 말은 곧 다음을 뜻한다.

- 같은 비트 패턴도 signed/unsigned에 따라 의미가 달라질 수 있다
- 타입 변환 시 값 해석이 달라질 수 있다
- 산술과 비교 결과도 달라질 수 있다

즉 signed/unsigned 차이는 단순 제약이 아니라  
**값 체계 자체의 차이**다.

---

### 2. 변환이 일어나면 "원래 타입"보다 "계산에 사용되는 타입"이 더 중요해진다

예시:
```c
    int a = -1;
    unsigned int b = 1;
    if (a < b) {
    }
```
이 코드는 `int`와 `unsigned int`를 바로 비교하는 것처럼 보인다.  
하지만 실제로는 비교 전에 공통 타입으로 맞추는 과정이 들어간다.

즉 C에서는 선언 타입만 보면 안 되고,  
**연산 시점에 어떤 타입으로 바뀌는가**를 봐야 한다.

이것이 나중의 `integer promotion은 왜 헷갈리는가`와 직접 연결된다.

---

### 3. 음수가 unsigned로 바뀌면 직관이 쉽게 깨진다

예시:
```c
    int x = -1;
    unsigned int y = (unsigned int)x;
```
이건 "음수를 unsigned에 넣었다" 정도로 보일 수 있다.  
하지만 중요한 것은 결과가 단순히 "음수 불가라 에러"가 아니라는 점이다.

C는 이런 변환을 허용할 수 있고, 그 결과는 보통 그 unsigned 타입의 값 체계 안에서 표현 가능한 값으로 대응된다.

즉 `-1`은 "그냥 유지"되지 않는다.  
**unsigned 값 체계 안의 큰 값**으로 해석될 수 있다.

바로 이 때문에 signed/unsigned 혼합은 직관을 깨뜨린다.

---

## 왜 비교에서 특히 위험한가

### 1. 음수 비교가 이상해질 수 있다

예시:
```c
    int a = -1;
    unsigned int b = 1;

    if (a < b) {
        /* 직관상 참 같음 */
    }
```
하지만 실제로는 `a`가 unsigned 쪽으로 변환되어 매우 큰 값처럼 비교될 수 있다.  
그러면 결과는 거짓이 될 수 있다.

즉 signed/unsigned 문제는  
"값이 틀려진다"보다 **비교의 의미 자체가 바뀐다**는 점이 더 위험하다.

---

### 2. 경계 검사에서 자주 터진다

예시:
```c
    int idx = -1;
    size_t len = 10;

    if (idx < len) {
        /* 안전해 보임 */
    }
```
겉으로 보면 `-1 < 10`이므로 참 같다.  
하지만 `len`이 unsigned 계열이라면 비교 직전에 `idx`가 unsigned로 바뀔 수 있다.

그 결과:

- `idx`는 큰 양수처럼 해석될 수 있고
- 비교는 거짓이 될 수 있다

또는 반대로 다른 형태에서는 방어 검사가 무력화될 수도 있다.

즉 signed/unsigned 혼합은 **경계 검사 코드에서 특히 치명적**이다.

---

### 3. 길이와 에러 코드를 섞을 때 흔하다

실무에서 매우 흔한 패턴은 이것이다.

- 길이/개수: `size_t`
- 인덱스/반환값: `int`
- 에러 코드: 음수
- 실제 바이트 수: unsigned 계열

이런 값들을 섞기 시작하면 signed/unsigned 문제가 바로 나온다.

예를 들어:
```c
    int n = get_length_or_error();
    size_t cap = buffer_size;

    if (n < cap) {
        /* ... */
    }
```
겉으로는 자연스럽지만, `n`이 음수 에러 코드일 때  
비교 의미가 완전히 달라질 수 있다.

즉 signed/unsigned 문제는 실무에서  
**정상 값과 에러 값 체계를 같은 코드에 섞을 때** 특히 많이 터진다.

---

## 왜 산술에서도 위험한가

### 1. 뺄셈에서 쉽게 드러난다

예시:
```c
    unsigned int a = 3;
    unsigned int b = 5;
    unsigned int c = a - b;
```
사람 직관으로는 `-2` 같지만, unsigned에는 음수가 없다.  
즉 결과는 signed 직관과 다르게 해석될 수 있다.

문제는 이것이 단순 산술 차이를 넘어서  
이후 비교, 길이 계산, 루프 조건에 연쇄 영향을 준다는 점이다.

---

### 2. "음수가 될 수도 있는 계산"을 unsigned로 하면 위험하다

예시:
```c
    size_t remaining = end - start;
```
겉보기엔 자연스럽지만, 실제로 `start > end`이면 문제가 된다.  
signed라면 음수라는 신호가 나올 수 있지만, unsigned에서는 큰 값처럼 해석될 수 있다.

즉 unsigned는 "절대 음수가 되면 안 되는 값"에 쓰는 것은 괜찮지만,  
**중간 계산에서 음수가 될 수 있는 식**에 섞이면 매우 위험하다.

---

### 3. 작은 underflow가 큰 양수로 바뀌는 것이 흔하다

예시:
```c
    size_t n = 0;
    n--;
```
이 코드는 문법적으로 별문제 없어 보여도, 의미는 직관과 다를 수 있다.  
즉 unsigned 감소는 "음수로 내려간다"가 아니라 값 체계 안에서 다른 큰 값으로 이어질 수 있다.

그래서 다음 같은 루프는 매우 위험하다.
```c
    for (size_t i = n - 1; i >= 0; i--) {
        /* ... */
    }
```
이건 사람 눈에는 자연스럽지만, unsigned의 성질 때문에 의도대로 끝나지 않을 수 있다.

즉 unsigned는 감소 루프에서도 조심해야 한다.

---

## 왜 `size_t`가 자주 문제의 출발점이 되는가

`size_t`는 크기, 길이, 바이트 수, 원소 수를 표현할 때 널리 쓰인다.  
그 자체는 매우 자연스럽다.

문제는 다음이다.

- `size_t`는 unsigned 계열이다
- 실무 코드는 종종 `int`와 섞인다
- 에러 코드는 종종 음수다
- 차이 계산은 음수가 될 수도 있다

즉 `size_t` 자체가 나쁜 것이 아니라,  
**unsigned 길이 타입과 signed 제어/에러 타입을 무심코 섞는 것**이 위험하다.

---

## 메모리 모델 관점에서 왜 중요한가

이 주제는 단순 API 사용법 문제가 아니다.  
memory-model 축에 있는 이유는 다음과 같다.

C에서 정수형은 단순 "숫자 컨테이너"가 아니라,  
비트폭과 해석 규칙을 가진 객체 표현 체계다.

즉 signed/unsigned 변환은:

- 값 체계 변환
- 비트 해석 방식 차이
- 비교 연산 의미 변화
- 산술 의미 변화

를 함께 만든다.

그래서 이 문제는 단순 문법 함정이 아니라  
**C 정수 모델 자체를 이해하는 핵심 관문**이다.

---

## 왜 보안과도 연결되는가

signed/unsigned 혼합은 길이 검증과 버퍼 처리에서 자주 취약점으로 이어진다.

예시적 패턴:

- 음수 길이값이 검증을 통과
- unsigned 변환 후 매우 큰 길이로 해석
- 복사/할당 크기가 비정상적으로 커짐
- 버퍼 오버플로우 또는 out-of-bounds 접근 발생

즉 이 문제는 "결과가 좀 이상하다" 수준이 아니라,  
메모리 안전성과 직접 연결된다.

그래서 length, count, size, offset, index를 다루는 코드에서는  
signed/unsigned 규칙을 특히 엄격하게 봐야 한다.

---

## 자주 하는 오해

### 오해 1. unsigned는 그냥 더 큰 양수 범위를 주는 좋은 타입이다

아니다.  
unsigned는 음수를 표현하지 않는 대신, signed와 다른 값 체계를 가진다.  
그 차이가 비교와 산술에서 직관을 깨뜨릴 수 있다.

---

### 오해 2. 음수를 unsigned에 넣으면 그냥 0이 되거나 에러다

아니다.  
C는 그런 단순한 규칙으로 처리하지 않는다.  
결과는 unsigned 값 체계 안에서 해석된다.

---

### 오해 3. 비교는 눈에 보이는 숫자대로 한다

아니다.  
비교 전에 공통 타입으로 맞추는 과정이 중요하다.  
즉 비교 결과는 선언 타입보다 **연산 시 변환**에 달려 있다.

---

### 오해 4. 길이 타입은 unsigned니까 아무 때나 int와 비교해도 괜찮다

아니다.  
바로 그 비교가 가장 흔한 문제 지점이다.

---

### 오해 5. unsigned를 쓰면 overflow 문제가 줄어든다

부분적으로만 맞다.  
signed overflow와 다른 양상일 수는 있지만, underflow/비교/길이 계산 문제는 오히려 더 교묘해질 수 있다.

---

## 어디서부터 직관과 실제 규칙이 갈리는가

### 1. 문제는 타입 이름보다 "연산 직전의 공통 타입"이다

많은 버그는 `int`와 `size_t`를 함께 썼다는 사실 자체보다,  
**비교나 산술 직전에 둘이 어떤 타입으로 맞춰지는가**를 놓쳐서 생긴다.

즉 다음을 분리해서 봐야 한다.

- 선언 시 타입
- 연산 직전 승격/변환 결과
- 그 결과가 값 의미를 어떻게 바꾸는가

사람은 원래 값 `-1`과 `1`을 비교한다고 생각하지만,  
언어는 이미 변환된 두 값을 비교할 수 있다.

### 2. "unsigned면 안전하다"는 직관이 가장 위험하다

unsigned는 음수를 막아 주는 타입처럼 보이지만,  
실무에서는 오히려 경계 검사를 더 교묘하게 깨뜨릴 수 있다.

왜냐하면 음수라는 신호가 사라지는 것이 아니라,  
다른 큰 값으로 재해석되어 **버그가 조용히 의미를 바꾸기 때문**이다.

그래서 unsigned는 "안전 타입"이 아니라  
"값 체계가 다른 타입"으로 이해하는 편이 훨씬 정확하다.

---

## 실무 관점

### 1. 길이, 개수, 인덱스, 에러 코드를 한 흐름에 섞지 않는 것이 중요하다

실무에서 가장 위험한 패턴은 이거다.

- 길이는 unsigned
- 반환값은 signed
- 에러는 음수
- 비교는 그냥 섞어서 함

이런 구조는 나중에 반드시 문제를 만든다.

즉 설계 단계에서부터 다음을 분리하는 것이 좋다.

- 정상 길이 값 체계
- 에러 값 체계
- 차이 계산 값 체계

---

### 2. 비교 전에 타입 의미를 먼저 확인해야 한다

코드를 읽을 때 단순히 `<`, `>`, `<=`만 보지 말고 먼저 봐야 한다.

- 왼쪽은 signed인가
- 오른쪽은 unsigned인가
- 비교 전에 어떤 변환이 일어나는가
- 이 값은 음수가 될 수 있는가

즉 비교문을 읽을 때 숫자보다 **타입 체계**를 먼저 봐야 한다.

---

### 3. 음수가 될 수 있는 중간 계산은 signed 쪽에서 처리하는 편이 낫다

예를 들어 차이, offset, 감소 카운트처럼  
중간에 음수가 될 수 있는 계산은 unsigned로 바로 처리하지 않는 편이 낫다.

이유는 음수라는 신호가 unsigned에서 사라지고, 큰 양수처럼 보일 수 있기 때문이다.

즉 unsigned는 "절대 음수가 아닌 최종 크기"에는 적합할 수 있지만,  
**음수가 될 수도 있는 중간 상태 표현**에는 조심해야 한다.

---

### 4. 다음 문서들과 직접 연결된다

이 문서는 아래와 바로 이어진다.

- integer promotion은 왜 헷갈리는가
- bitwise 연산과 shift는 실무에서 어떻게 연결되는가
- undefined behavior는 왜 위험한가
- 버퍼 오버플로우는 어떻게 생기는가

즉 정수 규칙, 메모리 안전성, 비트 연산 축을 연결하는 핵심 문서다.

---

## 판단 체크리스트

signed/unsigned가 섞인 코드를 보면 먼저 아래를 확인하면 된다.

1. 이 값은 음수가 될 수 있는가
2. 비교 또는 산술 직전에 어떤 공통 타입으로 변환되는가
3. 이 변수는 길이/개수 체계인가, 에러/차이/오프셋 체계인가
4. unsigned를 "안전" 때문에 쓴 것인지, 값 도메인 때문에 쓴 것인지

즉 핵심은 타입 이름이 아니라  
**이 값이 어느 값 체계에 속하며, 연산 직전에 어느 체계로 강제되는가**를 보는 것이다.

## 정리

signed/unsigned 변환은 단순히 음수 가능 여부가 바뀌는 것이 아니다.  
그것은 값 체계, 비교 결과, 산술 결과, 경계 검사 의미를 함께 바꾸는 규칙이다.

핵심은 다음과 같다.

- signed와 unsigned는 같은 비트폭이어도 다른 값 체계다
- 둘을 섞으면 연산 전 변환 규칙이 중요해진다
- 비교와 길이 검사에서 직관이 쉽게 깨진다
- underflow와 감소 루프에서 특히 위험하다
- 메모리 안전성 문제와도 직접 연결된다

즉 이 주제를 이해한다는 것은  
"unsigned는 음수 없는 정수"라고 외우는 수준을 넘어서, **C가 정수를 어떤 규칙으로 맞춰서 계산하는가**를 이해하는 것이다.

이 관점이 잡히면 다음의 `integer promotion`, `bitwise 연산`, 길이 계산 관련 실수들이 하나의 체계로 연결되기 시작한다.
