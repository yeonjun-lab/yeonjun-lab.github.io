## 정의

C의 전처리기(preprocessor)는 컴파일 전에 소스 코드를 문자 단위 수준에서 가공하는 단계다.  
즉 전처리기는 C 문법을 이해해서 프로그램 의미를 분석하는 컴파일러 본체가 아니라,  
그 이전에 `#include`, `#define`, `#if` 같은 지시문을 처리해 **컴파일러가 읽을 최종 소스 형태를 만들어 주는 도구**다.

핵심은 다음이다.

1. 전처리기는 컴파일 전에 동작한다
2. 헤더 포함, 매크로 치환, 조건부 컴파일을 처리한다
3. 전처리기는 타입 시스템이나 실행 의미를 이해하지 않는다
4. 전처리 결과가 이후 컴파일 단계를 위한 입력이 된다

즉 전처리기를 이해한다는 것은  
C 소스가 "작성된 모습 그대로" 컴파일되는 것이 아니라,  
**먼저 가공된 뒤 컴파일된다**는 사실을 이해하는 것이다.

---

## 핵심 개념

### 1. C 소스는 작성된 그대로 바로 컴파일되지 않는다

예시:
```
    #include <stdio.h>

    #define SIZE 10

    int main(void) {
        int arr[SIZE];
        return 0;
    }
```
이 코드는 겉보기에는 간단하지만, 컴파일러가 바로 이 텍스트를 해석하는 것은 아니다.  
먼저 전처리기가 개입해서 다음 같은 작업을 수행한다.

- `#include <stdio.h>`를 헤더 내용으로 확장
- `SIZE`를 `10`으로 치환
- 조건부 컴파일 지시문이 있으면 활성/비활성 코드 선택

즉 컴파일러는 "원본 소스"가 아니라  
**전처리된 결과 소스**를 기준으로 본격적인 문법 분석과 컴파일을 시작한다.

---

### 2. 전처리기의 대표 작업은 세 가지다

전처리기가 가장 자주 하는 일은 아래 세 가지다.

#### 1) 헤더 포함
```
    #include "myheader.h"
```
이 지시문은 해당 헤더 파일 내용을 현재 위치에 삽입하는 효과를 낸다.

#### 2) 매크로 치환
```
    #define BUF_SIZE 1024
```
이후 `BUF_SIZE`라는 토큰이 나오면 전처리기가 이를 `1024`로 바꾼다.

#### 3) 조건부 컴파일
```
    #ifdef DEBUG
    printf("debug\n");
    #endif
```
빌드 조건에 따라 특정 코드를 포함하거나 제외한다.

즉 전처리기는 C 프로그램의 구조를 만드는 사전 가공기다.

---

### 3. 전처리기는 "텍스트 치환기"에 가깝다

이것이 매우 중요하다.

전처리기는 기본적으로 타입, 변수, 함수 의미를 이해하지 않는다.  
즉 다음 같은 문맥적 의미를 모른다.

- 이 이름이 변수인지 함수인지
- 이 값이 정수인지 포인터인지
- 이 식이 안전한지
- 이 코드가 UB인지

전처리기는 우선적으로 토큰 수준 치환과 조건 처리에 집중한다.

예시:
```
    #define SQUARE(x) x * x
```
이 매크로는 수학적 제곱 의미를 "이해"해서 동작하는 것이 아니다.  
그냥 주어진 패턴을 텍스트 수준으로 치환한다.

따라서:
```
    SQUARE(1 + 2)
```
는 기대와 다르게 다음처럼 확장될 수 있다.

    1 + 2 * 1 + 2

즉 전처리기는 똑똑한 함수 호출 해석기가 아니라,  
**정교하지만 위험할 수 있는 치환기**다.

---

## `#include`는 실제로 무엇을 하는가

### 1. 파일 내용을 현재 위치에 포함시킨다

예시:
```
`main.c`

    #include "config.h"

    int main(void) {
        return VALUE;
    }

`config.h`

    #define VALUE 42
```
전처리 후 개념적으로는 다음처럼 생각할 수 있다.
```
    #define VALUE 42

    int main(void) {
        return 42;
    }
```
즉 `#include`는 "참조"가 아니라,  
현재 소스 위치에 다른 파일의 내용을 가져와 합치는 동작에 가깝다.

---

### 2. 헤더 가드가 왜 필요한지가 여기서 드러난다

예시:
```
    #ifndef CONFIG_H
    #define CONFIG_H

    #define VALUE 42

    #endif
```
헤더가 여러 번 포함되면 선언/정의 충돌이 날 수 있다.  
왜냐하면 `#include`는 그냥 내용을 반복 삽입하는 동작에 가깝기 때문이다.

따라서 헤더 가드는 중복 포함을 막기 위한 장치다.

즉 헤더 가드는 컴파일러의 특별한 마법이 아니라,  
전처리기의 텍스트 포함 동작 때문에 필요해진다.

---

## `#define`은 실제로 무엇을 하는가

### 1. 상수처럼 보이는 이름도 실제로는 치환일 수 있다

예시:
```
    #define MAX_COUNT 100
```
이후:
```
    int arr[MAX_COUNT];
```
는 전처리 후 개념적으로:
```
    int arr[100];
```
처럼 바뀐다.

즉 `MAX_COUNT`는 타입이 있는 객체가 아니라 매크로 이름일 수 있다.

이 점에서 `const int max_count = 100;`과는 본질적으로 다르다.

- `#define`은 전처리 치환
- `const`는 타입을 가진 객체

즉 둘은 비슷한 용도로 보일 수 있어도 같은 것이 아니다.

---

### 2. 함수형 매크로는 함수가 아니다

예시:
```
    #define DOUBLE(x) ((x) * 2)
```
이것은 함수처럼 보이지만 함수가 아니다.  
호출 시 인자를 평가해서 스택 프레임을 만드는 것이 아니라,  
그 자리에 치환된 표현식을 삽입하는 효과를 낸다.

이 때문에 함수형 매크로는 다음 위험이 있다.

- 괄호 누락 문제
- 인자 중복 평가 문제
- 디버깅 어려움
- 타입 검사 부재

예시:
```
    #define INC(x) ((x) + 1)

    int i = 3;
    int y = INC(i++);
```
이런 코드는 예상보다 위험할 수 있다.  
전처리기는 부작용 의미를 고려하지 않기 때문이다.

---

## 조건부 컴파일은 왜 필요한가

### 1. 플랫폼별 코드 분기

예시:
```
    #ifdef _WIN32
    /* Windows code */
    #else
    /* POSIX code */
    #endif
```
운영체제나 컴파일러에 따라 다른 코드를 선택할 수 있다.

---

### 2. 디버그/릴리즈 분기

예시:
```
    #ifdef DEBUG
    printf("debug: x=%d\n", x);
    #endif
```
디버그 빌드에서는 로그를 포함하고, 릴리즈 빌드에서는 제거할 수 있다.

---

### 3. 기능 플래그

예시:
```
    #if FEATURE_X_ENABLED
    /* feature code */
    #endif
```
빌드 시점에 기능 포함 여부를 제어할 수 있다.

즉 조건부 컴파일은 런타임 `if`와 다르다.  
실행 중에 분기하는 것이 아니라, **아예 컴파일될 코드 자체를 선택**한다.

---

## 전처리기와 컴파일러는 왜 구분해야 하는가

이 차이를 모르면 C 빌드 구조가 흐려진다.

### 전처리기

- `#include` 처리
- `#define` 치환
- `#if`, `#ifdef` 처리
- 컴파일러가 읽을 최종 소스 생성

### 컴파일러 본체

- 토큰/구문 분석
- 타입 검사
- 의미 분석
- 최적화
- 오브젝트 코드 생성

즉 전처리기는 C 언어의 의미를 이해하는 주체가 아니라,  
그 전에 소스를 준비하는 단계다.

이 차이를 이해해야 다음이 분명해진다.

- 왜 매크로는 타입 검사를 못 받는가
- 왜 에러 메시지 위치가 헷갈릴 수 있는가
- 왜 헤더 포함이 translation unit을 만든다고 하는가

---

## translation unit과 어떻게 연결되는가

전처리가 끝나면 소스 파일 하나는 보통 하나의 translation unit으로 이어진다.

예를 들어 `main.c`가 여러 헤더를 포함하고 있으면,  
전처리 후에는 그 헤더 내용이 모두 펼쳐진 하나의 큰 소스 덩어리처럼 볼 수 있다.

즉 translation unit은 대략적으로:

- 원본 `.c` 파일
- 그 파일이 포함한 헤더들
- 전처리 결과까지 반영한 하나의 컴파일 단위

이 개념은 나중의 아래 주제와 직결된다.

- 선언과 정의 분리
- extern과 static
- 오브젝트 파일과 심볼
- 링크 과정

즉 전처리기는 빌드 구조의 출발점이다.

---

## 자주 하는 오해

### 오해 1. `#include`는 링크처럼 외부 파일을 연결하는 것이다

아니다.  
전처리기 수준에서는 파일 내용을 현재 위치에 포함시키는 것에 가깝다.

링크는 훨씬 뒤 단계다.

---

### 오해 2. 매크로는 변수나 함수와 비슷하다

아니다.  
매크로는 타입이 있는 객체도 아니고, 함수 호출 규칙을 따르는 함수도 아니다.  
기본적으로 치환 규칙이다.

---

### 오해 3. 조건부 컴파일은 런타임 분기와 같다

아니다.  
전처리 단계에서 코드 포함 여부를 결정하므로, 컴파일 대상 코드 자체가 바뀐다.

---

### 오해 4. 전처리기가 C 문법을 완전히 이해한다

아니다.  
전처리기는 기본적으로 텍스트/토큰 수준의 가공 단계다.  
타입 의미와 실행 의미는 컴파일러 본체가 처리한다.

---

### 오해 5. `#define` 상수와 `const` 객체는 같다

아니다.

- `#define`은 전처리 치환
- `const`는 타입을 가진 실제 객체 또는 읽기 전용 의미를 가진 선언

둘은 디버깅, 타입 검사, 주소 취급, 스코프에서 다르다.

---

## 왜 매크로는 조심해서 써야 하는가

### 1. 괄호 문제

예시:
```
    #define SQUARE(x) x * x

    int y = SQUARE(1 + 2);
```
이 코드는 기대와 다를 수 있다.

보통은 아래처럼 더 안전하게 작성한다.
```
    #define SQUARE(x) ((x) * (x))
```
---

### 2. 인자 중복 평가

예시:
```
    #define MAX(a, b) ((a) > (b) ? (a) : (b))

    int i = 0;
    int m = MAX(i++, 10);
```
이런 코드는 인자가 여러 번 평가될 수 있어 위험하다.

전처리기는 함수처럼 "한 번 평가한 결과를 안전하게 보관"하지 않는다.

---

### 3. 디버깅과 에러 추적의 어려움

매크로는 소스 원문과 전처리 결과 사이에 차이를 만든다.  
그래서 에러 메시지나 디버깅 시 실제 확장 결과를 의식해야 할 수 있다.

즉 매크로는 강력하지만, 타입 안전성과 가독성 측면에서 비용이 있다.

---

## 어디서부터 텍스트 변환과 의미 해석이 갈리는가

### 1. 전처리기는 프로그램 의미를 이해하지 않고 입력 형태를 바꾼다

초보자는 `#include`, `#define`, `#if`까지 전부 컴파일러가 “이해해서” 처리한다고 느끼기 쉽다.  
하지만 전처리기의 핵심 역할은 C 의미를 해석하는 것이 아니라  
컴파일 전에 소스 입력을 조합하고 치환하는 데 있다.

즉, 전처리기는 의미 분석기라기보다  
컴파일 입력을 만드는 앞단 변환기다.

### 2. `#include`는 모듈 로딩이 아니라 텍스트 포함이다

헤더를 가져온다는 표현 때문에  
런타임 모듈 로딩처럼 상상하기 쉽다.  
하지만 실제로는 현재 소스 위치에 헤더 내용이 들어오는 쪽에 가깝다.

즉, include를 이해할 때는 “불러온다”보다  
“현재 translation unit에 포함시킨다”는 감각이 더 정확하다.

### 3. `#define`은 타입 안전한 상수가 아니라 치환 규칙이다

매크로 이름이 변수나 함수처럼 보여도  
전처리 단계에서는 타입 검사나 단일 평가 보장이 없다.  
그래서 괄호 문제, 중복 평가, 예상 밖 확장이 생길 수 있다.

즉, 매크로는 함수나 객체의 저렴한 복제품이 아니라  
텍스트/토큰 치환 규칙으로 읽어야 한다.

### 4. 조건부 컴파일은 실행 시 분기가 아니라 빌드 시 분기다

`#if`, `#ifdef`는 프로그램이 실행되면서 조건을 고르는 것이 아니다.  
아예 어떤 코드가 translation unit에 들어갈지를 미리 결정한다.

즉, 조건부 컴파일은 제어 흐름이 아니라  
소스 구성 선택 문제다.

### 5. 전처리 단계 이해가 없으면 헤더와 빌드 구조가 계속 흐려진다

헤더 가드, 중복 포함, 선언/정의 분리, 플랫폼 분기, 매크로 비용은  
전부 전처리 단계가 먼저 소스를 어떻게 조립하는지 알아야 선명해진다.

즉, 전처리기는 문법 본론 이전의 부가 기능이 아니라  
translation unit 경계를 결정하는 핵심 단계다.

---

## 실무 관점

### 1. 헤더 구조 이해의 출발점이다

왜 선언은 헤더에 두고 정의는 `.c`에 두는가,  
왜 헤더 가드가 필요한가,  
왜 중복 정의 에러가 나는가를 이해하려면 전처리기부터 알아야 한다.

즉 헤더 설계는 전처리기 이해 없이 제대로 잡기 어렵다.

---

### 2. 플랫폼 추상화와 빌드 제어에 필수다

실무 C 프로젝트는 운영체제, 컴파일러, 아키텍처별 분기를 자주 가진다.  
이때 조건부 컴파일은 핵심 도구다.

예시:

- Windows / Linux 분기
- debug / release 분기
- 기능 on/off 분기
- 컴파일러별 속성 분기

즉 전처리기는 휴대성과 빌드 전략의 핵심 일부다.

---

### 3. 매크로 남용은 장기적으로 비용이 크다

매크로는 간단한 상수 치환이나 컴파일 조건 분기에는 유용하다.  
하지만 복잡한 로직까지 매크로로 밀어 넣으면 다음 문제가 커진다.

- 읽기 어려움
- 디버깅 어려움
- 타입 안전성 부족
- 부작용 추적 어려움

따라서 실무에서는 다음 기준이 중요하다.

- 단순 상수 → 매크로 또는 enum/const 검토
- 짧은 조건부 컴파일 → 허용
- 복잡한 계산 → 함수 또는 inline 함수 우선 검토

---

### 4. 이후 빌드/링킹 축과 직접 연결된다

이 문서는 다음 문서와 직접 이어진다.

- 헤더 파일과 선언/정의 분리는 왜 필요한가
- 컴파일, 어셈블, 링크의 전체 흐름
- extern과 static은 무엇이 다르고 왜 중요한가
- 오브젝트 파일과 심볼이란 무엇인가

즉 전처리기는 기초 축의 마지막이면서,  
빌드/링킹 축의 입구다.

---

## 판단 체크리스트

- 전처리기를 의미 해석기가 아니라 컴파일 입력 변환기로 이해하고 있는가
- `#include`를 모듈 로딩이 아니라 텍스트 포함으로 설명할 수 있는가
- `#define`을 타입 안전한 상수/함수가 아니라 치환 규칙으로 보고 있는가
- 조건부 컴파일을 런타임 분기가 아니라 빌드 시 분기로 구분하고 있는가
- 헤더 구조와 빌드 문제를 전처리 단계와 연결해서 보고 있는가

---

## 정리

전처리기는 C 컴파일 전에 소스를 가공하는 단계다.  
그것은 프로그램 의미를 완전히 이해하는 컴파일러 본체가 아니라,  
`#include`, `#define`, `#if` 같은 지시문을 처리해 최종 컴파일 입력을 만드는 도구다.

핵심은 다음과 같다.

- `#include`는 파일 내용을 포함한다
- `#define`은 치환 규칙을 만든다
- 조건부 컴파일은 컴파일 대상 코드를 선택한다
- 전처리기는 타입 의미를 이해하지 않는다
- 전처리 결과가 translation unit의 기반이 된다

즉 전처리기를 이해하면 C 소스가 "눈에 보이는 그대로" 컴파일되는 것이 아니라,  
**먼저 조립되고 변형된 뒤 컴파일된다**는 사실이 선명해진다.

이 관점이 잡히면 헤더 구조, 매크로, 빌드 분기, 선언/정의 분리, 링크 과정을 훨씬 정확하게 이해할 수 있다.
