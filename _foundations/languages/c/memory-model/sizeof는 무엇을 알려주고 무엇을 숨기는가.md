## 정의

`sizeof`는 C에서 어떤 타입이나 객체 표현이 차지하는 바이트 수를 알려주는 연산자다.  
하지만 `sizeof`는 모든 것을 알려주지는 않는다.  
그것은 객체의 **표현 크기**를 알려줄 뿐이며, 값의 의미, 동적 길이, 실제 사용 중인 데이터 양, 포인터가 가리키는 대상 크기까지 자동으로 알려주지는 않는다.

즉 `sizeof`는 단순한 "크기 측정 도구"가 아니라,  
**C가 객체를 어떻게 보고 있는지 드러내는 연산자**다.

핵심은 다음이다.

1. `sizeof`는 타입 또는 객체 표현 크기를 알려준다
2. 배열에서는 전체 크기를 볼 수 있지만, 포인터에서는 대상 크기를 알 수 없다
3. 함수 인자에서는 배열이 이미 포인터처럼 보이므로 기대와 다른 결과가 나온다
4. `sizeof`는 메모리 모델을 보여주지만, 런타임 논리 길이까지는 보여주지 않는다

즉 `sizeof`를 제대로 이해한다는 것은  
단순히 숫자를 읽는 것이 아니라 **C의 타입 시스템과 객체 모델을 읽는 것**이다.

---

## 핵심 개념

### 1. `sizeof`는 객체의 "값"이 아니라 "표현 크기"를 본다

예시:
```
    int x = 10;
    sizeof(x)
```
여기서 `sizeof(x)`는 `x`의 값이 10이라서 10과 관련된 결과를 주는 것이 아니다.  
`x`라는 객체를 표현하는 데 필요한 바이트 수를 준다.

즉 다음 둘은 전혀 다르다.

- 값의 크기
- 객체 표현 크기

예시:
```
    char a = 1;
    char b = 100;
```
둘 다 값은 다르지만:
```
    sizeof(a)
    sizeof(b)
```
결과는 같다.  
왜냐하면 `sizeof`는 값의 의미가 아니라 객체 타입의 표현 크기를 보기 때문이다.

---

### 2. `sizeof(type)`와 `sizeof(expr)`는 비슷하지만 보는 대상이 다르다

예시:
```
    sizeof(int)
    sizeof(x)
```
첫 번째는 타입 자체의 크기다.  
두 번째는 표현식 `x`의 타입을 기준으로 한 크기다.

실무적으로는 둘 다 자주 쓰인다.

예시:
```
    int *p = malloc(sizeof(int));
    int arr[10];
    size_t n = sizeof(arr);
```
즉 `sizeof`는 타입에 직접 쓸 수도 있고, 객체/표현식에 쓸 수도 있다.

---

### 3. 배열에서는 전체 크기를 알려준다

예시:
```
    int arr[10];
    sizeof(arr)
```
이 결과는 포인터 크기가 아니라 배열 전체 크기다.

예를 들어 `int`가 4바이트라면:
```
    sizeof(arr) == 40
```
이것은 매우 중요하다.  
배열 이름이 많은 문맥에서 decay한다고 해도, `sizeof(arr)`에서는 decay가 일어나지 않는다.

즉 `sizeof`는 배열을 포인터처럼 보지 않고,  
**배열 전체 객체**로 본다.

이 성질 덕분에 같은 스코프 안에서는 배열 원소 개수를 구할 수 있다.

예시:
```
    size_t count = sizeof(arr) / sizeof(arr[0]);
```
이 계산은 배열이 실제 배열 객체로 보이는 문맥에서만 유효하다.

---

### 4. 포인터에서는 "가리키는 대상 크기"가 아니라 "포인터 자체 크기"를 알려준다

예시:
```
    int *p;
    sizeof(p)
```
이 결과는 `int`의 크기가 아니다.  
또 `p`가 가리킬지도 모르는 배열 전체 크기도 아니다.  
그냥 포인터 객체 자체의 크기다.

즉 `sizeof(p)`는 아래를 알려주지 않는다.

- `p`가 가리키는 원소 개수
- 동적 할당된 블록 전체 크기
- 문자열 길이
- 포인터가 가리키는 실제 데이터의 논리 길이

오직 포인터 값 하나를 저장하는 데 필요한 크기만 알려준다.

이 지점이 `sizeof` 관련 버그의 핵심이다.

---

## 배열과 포인터에서 왜 자꾸 헷갈리는가

### 1. 겉보기 코드가 비슷하기 때문이다

예시:
```
    char s1[] = "hello";
    char *s2 = "hello";
```
둘 다 문자열처럼 보이지만:
```
    sizeof(s1)
    sizeof(s2)
```
는 다르다.

- `sizeof(s1)` : 배열 전체 크기, 보통 6
- `sizeof(s2)` : 포인터 크기

즉 `s1`은 저장공간을 가진 배열 객체이고,  
`s2`는 어떤 위치를 가리키는 포인터다.

문자열처럼 보인다고 해서 같은 메모리 모델이 아니다.

---

### 2. 함수 인자로 가면 배열 정보가 사라지기 때문이다

예시:
```
    void f(int arr[]) {
        printf("%zu\n", sizeof(arr));
    }
```
많은 초보자는 `arr`가 배열이므로 전체 크기가 나올 것이라고 생각한다.  
하지만 함수 매개변수에서 `arr[]`는 실제로 포인터로 조정된다.

즉 위 함수 안의 `arr`는 배열 객체가 아니라 포인터다.

따라서:
```
    sizeof(arr)
```
는 배열 전체 크기가 아니라 포인터 크기다.

이것은 `sizeof`가 틀린 것이 아니라,  
함수 안에서 이미 타입이 달라져 있기 때문이다.

---

## `sizeof`가 알려주는 것

`sizeof`는 아래를 잘 알려준다.

### 1. 객체 표현 크기

예시:
```
    sizeof(int)
    sizeof(double)
    sizeof(struct Point)
```
즉 어떤 타입의 객체를 저장하려면 몇 바이트가 필요한지 알 수 있다.

---

### 2. 배열 전체 저장공간 크기

예시:
```
    int arr[100];
    sizeof(arr)
```
배열 전체가 차지하는 바이트 수를 알 수 있다.

---

### 3. 구조체 레이아웃 결과

예시:
```
    struct Sample {
        char c;
        int x;
    };

    sizeof(struct Sample)
```
이 값은 단순히 멤버 크기 합과 같지 않을 수 있다.  
정렬과 padding이 반영되기 때문이다.

즉 `sizeof`는 구조체 메모리 배치 결과를 관찰하는 도구이기도 하다.

---

### 4. 동적 할당 시 필요한 크기 계산 기준

예시:
```
    int *p = malloc(sizeof(*p) * 10);
```
이런 코드는 타입명을 반복하지 않고도 안전하게 크기를 계산하는 데 유용하다.

즉 `sizeof`는 메모리 할당과 강하게 연결된다.

---

## `sizeof`가 숨기는 것

이 부분이 더 중요하다.

### 1. 논리 길이

예시:
```
    char *p = "hello";

    sizeof(p)
```
이 결과는 문자열 길이 5를 알려주지 않는다.  
포인터 크기만 알려준다.

즉 `sizeof`는 "내용의 의미적 길이"를 모른다.

문자열 길이는 널 문자를 찾는 규칙으로 결정되며, 필요하면 `strlen` 같은 방식이 필요하다.

---

### 2. 동적 메모리 블록 전체 크기

예시:
```
    int *p = malloc(100 * sizeof(int));
```
이후:
```
    sizeof(p)
```
를 해도 100개짜리 블록 크기는 나오지 않는다.  
여전히 포인터 크기만 나온다.

즉 C에서 포인터는 "내가 얼마나 큰 메모리를 가리키는지"를 스스로 기억하지 않는다.

이 점은 매우 중요하다.

---

### 3. 사용 중인 실제 데이터 양

예시:
```
    char buf[1024];

    sizeof(buf)
```
는 1024를 줄 수 있지만, 실제로 안에 유효한 문자열이 10바이트만 들어 있을 수도 있다.

즉 `sizeof`는 버퍼 용량을 알려줄 수는 있어도,  
현재 의미 있는 데이터 양을 알려주지는 않는다.

---

### 4. 가리키는 대상의 타입 의미

예시:
```
    void *p;

    sizeof(p)
```
는 `void *` 포인터 크기를 줄 뿐이다.  
그 포인터가 실제로 어떤 객체를 가리키는지, 그 객체가 몇 바이트인지 알려주지 않는다.

즉 `sizeof`는 추적 도구가 아니라, 현재 타입/표현 기준의 크기 도구다.

---

## 왜 `sizeof(arr) / sizeof(arr[0])`는 유명한가

예시:
```
    int arr[10];
    size_t n = sizeof(arr) / sizeof(arr[0]);
```
이 계산은 배열 원소 개수를 구하는 고전적인 패턴이다.

왜 동작하는가?

- `sizeof(arr)` : 배열 전체 크기
- `sizeof(arr[0])` : 원소 하나 크기

따라서 전체 크기를 원소 하나 크기로 나누면 개수가 나온다.

하지만 이 계산이 항상 되는 것은 아니다.

예시:
```
    void f(int arr[]) {
        size_t n = sizeof(arr) / sizeof(arr[0]);   // 잘못된 기대
    }
```
함수 안에서는 `arr`가 포인터이므로 이 계산은 배열 길이를 주지 않는다.

즉 이 패턴은 **실제 배열 객체가 보이는 스코프에서만 안전하다.**

---

## `sizeof`와 구조체 padding

예시:
```
    struct Example {
        char c;
        int x;
    };
```
겉보기에는 `char` 1바이트 + `int` 4바이트라서 5바이트 같아 보일 수 있다.  
하지만 실제:
```
    sizeof(struct Example)
```
는 8이 될 수도 있다.

이유는 정렬과 padding 때문이다.

즉 `sizeof`는 단순 멤버 크기 합이 아니라,  
**실제 메모리 배치 결과**를 보여준다.

이 때문에 `sizeof`는 구조체 레이아웃을 공부할 때 매우 중요한 도구다.

---

## `sizeof(*p)`를 쓰는 이유

예시:
```
    int *p = malloc(sizeof(*p) * 10);
```
이 방식은 다음 장점이 있다.

- 타입명을 반복하지 않아도 된다
- `p`의 타입이 바뀌어도 코드 수정이 줄어든다
- 실수로 다른 타입 크기를 넣을 위험이 줄어든다

예를 들어 아래보다:
```
    int *p = malloc(sizeof(int) * 10);
```
다음이 더 유연할 수 있다.
```
    int *p = malloc(sizeof(*p) * 10);
```
즉 `sizeof`는 단순 조회 도구가 아니라,  
타입 일관성을 유지하는 실무 도구이기도 하다.

---

## 자주 하는 오해

### 오해 1. `sizeof`는 포인터가 가리키는 대상 전체 크기를 알려준다

아니다.  
포인터에서는 포인터 자체 크기만 나온다.

---

### 오해 2. `sizeof`는 문자열 길이를 알려준다

항상 아니다.

- 배열 문자열이면 저장공간 크기를 줄 수 있다
- 포인터 문자열이면 포인터 크기만 준다

즉 문자열 의미 길이와 `sizeof` 결과는 다른 개념이다.

---

### 오해 3. `sizeof`는 런타임 데이터 사용량을 보여준다

아니다.  
`sizeof`는 현재 타입/객체 표현 기준의 크기를 보여줄 뿐이다.

---

### 오해 4. 배열과 포인터는 `sizeof` 결과도 비슷하다

아니다.  
이 차이는 배열과 포인터가 같은 것이 아님을 가장 강하게 보여주는 증거 중 하나다.

---

### 오해 5. 구조체 크기는 멤버 크기 합과 같다

아니다.  
padding과 alignment 때문에 더 커질 수 있다.

---

## 실무 관점

### 1. 버퍼 크기 계산의 기본이다

안전한 C 코드에서는 버퍼 크기 계산이 매우 중요하다.

예시:
```
    char buf[256];
    fread(buf, 1, sizeof(buf), fp);
```
이런 식으로 `sizeof`를 쓰면 하드코딩 없이 배열 전체 크기를 사용할 수 있다.

---

### 2. 동적 할당 크기 계산에서 필수다

예시:
```
    struct Node *nodes = malloc(sizeof(*nodes) * count);
```
이 패턴은 타입 불일치 실수를 줄여 준다.

---

### 3. 함수 인자에서 잘못 쓰면 버그가 생긴다

예시:
```
    void clear(int arr[]) {
        memset(arr, 0, sizeof(arr));   // 잘못된 기대
    }
```
이 코드는 배열 전체를 0으로 초기화하지 못할 수 있다.  
왜냐하면 `sizeof(arr)`는 포인터 크기이기 때문이다.

즉 함수에서는 크기를 별도 인자로 받는 설계가 필요하다.

---

### 4. 구조체 직렬화/패딩 분석에서도 중요하다

`sizeof(struct T)`는 구조체를 파일/네트워크에 그대로 써도 되는가를 검토할 때 중요한 출발점이다.  
padding이 포함될 수 있으므로, 단순히 `sizeof`만 보고 안전하다고 판단하면 안 된다.

즉 `sizeof`는 정보를 주지만, 동시에 더 깊은 검토가 필요하다는 신호이기도 하다.

---

### 5. 이후 심화 주제로 이어진다

이 문서는 아래와 직접 연결된다.

- 구조체는 메모리에 어떻게 배치되는가
- struct padding과 alignment는 왜 중요한가
- 배열과 포인터는 왜 비슷해 보이지만 같지 않은가
- malloc과 free는 실제로 무엇을 하는가
- 직렬화와 역직렬화는 왜 단순 memcpy가 아닌가

즉 `sizeof`는 C의 메모리 모델을 관찰하는 핵심 창구 중 하나다.

---

## 정리

`sizeof`는 타입이나 객체 표현이 차지하는 바이트 수를 알려준다.  
하지만 그것이 모든 크기 정보를 알려주는 것은 아니다.

핵심은 다음과 같다.

- 배열에서는 전체 크기를 볼 수 있다
- 포인터에서는 포인터 자체 크기만 본다
- 함수 인자에서는 배열이 포인터처럼 보이므로 기대가 달라질 수 있다
- 구조체에서는 padding과 alignment 결과를 반영한다
- 논리 길이, 문자열 길이, 동적 블록 크기는 자동으로 알려주지 않는다

즉 `sizeof`는 단순한 숫자 도구가 아니라  
**C가 객체를 어떤 단위로 보고 있는지 드러내는 연산자**다.

이 관점을 잡으면 `sizeof` 관련 실수는 줄어들고,  
배열, 포인터, 구조체, 동적 메모리, 직렬화 문서들이 더 선명하게 연결되기 시작한다.
